## 2. 漏洞链路复核

| 环节 | 位置 | 问题 |
|------|------|------|
| **买入写入点** | `BlackMerchantHandler.lua:HandleBuy` | 服务端直接写入 `saveData.backpack`，无仓库交互，安全 |
| **存仓/取仓未同步点** | `WarehouseSystem.StoreItem/RetrieveItem` | **修复前**：仓库操作仅本地生效，不发 `save_request`，服务端 `saveData` 不知仓库变动 |
| **卖出校验点** | `BlackMerchantHandler.lua:HandleSell` | 服务端通过 `CountItemHeld(saveData.backpack)` 校验，只查背包不查仓库。物品存入仓库后背包仍有"旧数据"（未保存），服务端误认为物品仍在 |
| **卖出后本地扣除失败点** | `BlackMerchantUI_HandleBMResult` sell 分支 | **修复前**：`ConsumeConsumable` 返回 false（物品已在仓库）后仍无条件发 `save_request`，导致仓库副本覆盖服务端已扣数据 |

## 3. 实际修复策略

| 决策点 | 方案 |
|--------|------|
| **未同步状态放在哪里** | `WarehouseSystem._dirty` 布尔标记，模块级变量 |
| **哪些操作会置脏** | `StoreItem`、`StoreItemToSlot`、`RetrieveItem`、`UnlockNextRow` — 4 个变更入口全部覆盖 |
| **哪些事件会清脏** | `game_saved`（存档成功回调）、`game_loaded`（加载存档） |
| **黑市卖出前拦截放在哪一层** | `BlackMerchantUI.lua` 确认弹窗 onClick → `SendSell()` 前调用 `WarehouseSystem.IsDirty()` 守卫 |
| **本地扣除失败时如何处理** | `BlackMerchantUI_HandleBMResult` sell 分支追踪 `localDeductOk`；失败时 **抑制 `save_request`**，显示错误提示"售出成功，但本地同步异常，请重新加载" |

## 7. 风险与遗留

### 7.1 仍然存在的理论攻击面

**改包客户端可绕过客户端前置校验。** 本次修复是纯客户端热修，攻击者若修改客户端代码：

1. 跳过 `IsDirty()` 守卫 → 仓库未同步时仍可发送 `C2S_BM_Sell`
2. 跳过 `localDeductOk` 检查 → 本地扣除失败后仍发 `save_request`

根本原因：服务端 `HandleSell` 仅查 `saveData.backpack`，不查 warehouse。物品从背包移入仓库后，若仓库状态未保存到服务端，服务端背包仍有物品。

### 7.2 是否建议继续单开服务端修复

**建议**。后续应考虑：

- 服务端 HandleSell 增加 warehouse 交叉校验（检查物品是否同时存在于 warehouse）
- 或将仓库操作改为服务端权威（C2S_StoreItem / C2S_RetrieveItem）

此为架构级改动，不属于本次热修范围。

### 7.3 是否允许上线

**允许。** 本次修复已堵住正常客户端的可复现刷收益路径：

- 普通玩家无法在正常操作中触发存仓→卖出→仓库副本保存的刷钱链路
- 客户端前置校验 + 本地扣除失败抑制 save_request 双重防护
- 仅改包客户端（作弊器）可绕过，需后续服务端权威化彻底解决

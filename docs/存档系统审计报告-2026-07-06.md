# 「人狗仙途」存档系统审计报告

> 审计日期：2026-07-06
> 背景：服务器震荡后，疑似大部分玩家保存能力失效
> 性质：**只读审计**——未修改任何代码、未构建、未上传
> 审计范围：客户端存档链（SaveSystem / systems/save/* / CloudStorage / SaveSystemNet）+ 服务端存档链（server_main / SaveWriteService / SaveLoadService / SlotReadService / SaveBackupService / ServerSession）+ 协议层（SaveProtocol / SavePayloadCodec）+ 迁移与序列化（SaveMigrations / SaveSerializer），共约 1.1 万行
> 依据：`docs/架构文档/rules.md`（存档安全红线、§3 存档规范、§12 服务端校验红线）

---

## 一、总体结论（TL;DR）

"服务器震荡后大部分保存能力失效"**不是单点 bug，而是三类缺陷叠加**：

1. **服务端内存态锁泄漏**（slot 写锁无 TTL，震荡期批量泄漏后，每次存档永远返回 `slot_locked`）；
2. **客户端网络状态机无自愈能力**（加载/拉槽位的 pending 永不超时、没有任何断线重连逻辑，玩家进不了游戏 → `loaded=false` → 所有保存被拒）；
3. **备份与恢复链在线上（网络模式）形同虚设**（checkpoint/登录备份是"假成功"，恢复路径三重失效），导致震荡造成的损伤无法自愈，只能人工处理。

最可能的现场机制按可能性排序：**锁泄漏（存不进）> pending 卡死（进不去所以存不了）> 空读当新玩家（索引/存档被覆盖）**。三者的外部表现都是"保存能力失效"。

## 二、事故机制还原（三条根因链）

### 根因链 A：slot 写锁泄漏 → 存档永久 `slot_locked`（最匹配症状）

```
震荡期 serverCloud API 挂起
→ 奖励类 handler（DaoTree/Redeem/Event/Wubao/SealDemon/Trial/TriggeredBattle 共 8 处）
  已 AcquireSlotLock，但释放锁的代码在 serverCloud 异步回调里，回调永远不来
→ 锁留在 slotWriteLocks_ 内存表（ServerSession.lua:78-115，无 TTL、无超时扫描）
→ 玩家断线重连时走"重复登录清理"（server_main.lua:335-342）：
  只删 userIds_，不删 connSlots_，也不释放锁；
  旧连接随后触发 HandleClientDisconnected 时 userId 已为 nil，
  server_main.lua:367 的 pcall(ReleaseSlotLock) 条件不满足 → 释放被跳过
→ 此后该玩家每次 C2S_SaveGame 都命中 SaveWriteService.lua:87 → 回 "slot_locked"
→ 客户端把 slot_locked 当"短暂冲突"（SavePersistence.lua:641-648）：
  每 3 秒无限重试、不累计失败、不触发 save_warning
→ 该玩家保存能力永久失效，直到服务端进程重启
```

震荡引发的"回调挂起 + 批量重连"恰好同时满足两个泄漏条件，所以是**批量**失效。

### 根因链 B：客户端 pending 永久卡死 + 无重连 → 进不了游戏

- `SaveSystemNet.lua` 的 `_pending`（FetchSlots/Load）记录了 `createdAt` 却**没有任何超时清扫**，断线事件也不清理；`rejectPending` 定义了但全工程零调用（SaveSystemNet.lua:53-110/257/421）。
- 服务端所有 C2S handler 在 `GetUserId(connKey)` 为 nil 时**静默 return 不回包**（server_main.lua:718-733 等）——震荡后客户端在服务端 `ClientIdentity` 建立映射前发出的请求全部石沉大海。
- 结果：一次无响应 → pending 永挂 → 玩家点重试被 single-flight 拒绝（"already pending"）→ **无限重试循环，只能杀进程**；卡在加载 → `loaded=false` → 保存全被 `not_loaded` 拒绝。
- 更底层：**客户端完全没有断线重连处理**（全工程无 ServerDisconnected 订阅、无重连逻辑）；`client_main.lua` 的 `InitializeNetwork` 是一次性的（`networkInitDone_=true` 后跳过），即使引擎层重连成功，新连接也不会重做 scene 关联和版本握手。

### 根因链 C："ok 但空"降级读被当正常数据 → 索引/账号数据被覆盖

- `SlotReadService.lua:74-80`：serverCloud 读回 `ok` 但 `slots_index` 缺失时，直接合成 `{version=1, slots={}}` 按新玩家回包，**不做任何交叉校验**（`player_level`、`_code_ver` 等旁路 key 明明可以证明这是老玩家）。
- 客户端把空/残缺索引缓存进 `_cachedSlotsIndex`，之后每次存档**整表写回** `slots_index`；服务端只防"全空"不防"残缺"（SaveWriteService.lua:134-139）→ 其他角色的索引条目丢失。
- 最危险分支：空索引让老玩家的槽位显示为空，玩家点"创建角色"——`server_main.lua:497-546` 的 CreateChar **只查索引占用、不查 `save_{slot}` 数据是否存在、不走 BeforeOverwrite 备份** → 满级存档被 1 级新档直接覆盖，不可恢复。

## 四、优化方案（基于 rules.md 红线，分三层）

所有方案均遵守 rules.md：不删/不改已有存档字段和 key、不删迁移函数、新字段用默认值兼容、服务端校验向"只读+日志"收敛、灰度先 log-only。

### 第一层：止血（小改动、可独立上线、直接对应本次事故）

1. **slot 锁加 TTL**：`slotWriteLocks_` 值从 `true` 改为 `{ acquiredAt }`（内存结构，不涉及存档格式），`IsSlotLocked` 判断超过 30~60s 视为过期自动释放并打 `[LOCK-EXPIRED]` 日志。同时修复重复登录清理：删 `userIds_` 时同步释放该 userId 的所有锁、清 `connSlots_`。
2. **客户端 pending 超时清扫**：给 `_pending` 加 30s 扫描（复用已有 `createdAt` 和 PostUpdate 延迟队列），过期即 `rejectPending`（该函数已写好、零调用）；`connection_lost` 事件同时清 pending。这一条直接修复"重试按钮永远失败"。
3. **服务端 userId=nil 必须回错误包**：SaveGame/LoadGame/FetchSlots 等 handler 在拿不到 userId 时回 `ok=false, message="identity_not_ready"`，客户端收到后延迟重发，替代 15s 盲等。
4. **SlotReadService 空读熔断**：`slots_index` 缺失时交叉检查同批的 `player_level`/`_code_ver`/任一 `save_N`；有任何旁证存在则判定"降级读"，回 `ok=false` 让客户端重试，而不是按新玩家回空索引。
5. **CreateChar 双保险**：创建前额外检查 `save_{slot}` 数据键是否存在；存在则拒绝创建；创建路径接入 `SaveBackupService.BeforeOverwrite`。
6. **`account_atlas` nil 短路**：对齐 `accountCosmetics` 的写法——Atlas 未真正加载过（服务端未回包该字段）时序列化返回 nil，服务端 `if coreData.atlas` 自然短路，杜绝空覆盖。
7. **slot_locked 重试加上限**：连续 N 次（如 10 次 / 30s）slot_locked 后升级为普通失败（累计 failures、触发 save_warning），让玩家和监控能感知，而不是静默 3s 循环。

### 第二层：结构修复（中期，涉及协议/链路，需灰度）

8. **备份链路归一**：二选一——(a) 承认服务端 `save_prewrite_backup_{slot}` 是唯一真实备份，删除客户端 checkpoint/登录备份的假路径（先把假成功日志改为如实 WARN，防误导排障）；或 (b) 为 `prev_`/`login_` 增加专用 C2S 事件走真实落盘。推荐 (a)+服务端增加多代备份（如 prewrite 保留最近 2~3 代），恢复入口统一从服务端读。
9. **恢复链重建**：`SaveLoadService` 数据缺失时的候选序列扩展为 `save_prewrite_backup_{slot}` → `deleted_{slot}` → v10 旧 key，并把恢复结果（用了哪一代）回传客户端展示。
10. **断线重连状态机**：客户端订阅引擎断连事件，断线后进入显式 RECONNECTING 状态（UI 提示 + 指数退避重连）；`InitializeNetwork` 支持对新连接重做 scene 关联 + 版本握手 + `OnNetworkReady`。这是"震荡后不用杀进程"的根本解。
11. **加载按模块隔离 pcall**：`ProcessLoadedData` 对每个子系统单独 pcall，单件物品/单系统失败时降级（跳过该件并记日志）而非整户禁载；配套加"加载完整性摘要"（哪些模块降级了）上报 NetDiag。注意与 P0-7 联动：库存必须区分"没有数据"和"反序列化失败"，失败时禁止置 loaded=true。
12. **库存空表护栏**：对齐 collection 的退化检测模式——`_lastKnownItemCount` 缓存 + 序列化结果骤减 >50% 时用缓存替代并告警。
13. **`data.version` 丢失防御**：version 缺失但存在 v11+ 特征字段（如 `mingge`/`ascension`/`trialTower`）时拒绝按 v1 迁移，改走"存档疑似损坏"恢复流程。
14. **服务端校验对齐 rules §12**：V6 仓库删除、V7d 技能删除改为 log-only 一个版本，确认无误报后再决定是否保留"剔除"；V5 硬编码上限全部改为引用 GameConfig/SealDemonConfig 权威表（部分已做，补齐剩余）。
15. **UpdateCritical 移出 paused 门**：把 `SaveSystem.UpdateCritical(dt)` 移到 main.lua 的 paused 早退之前（与 `CloudStorage.Tick` 同区），兑现 WP-01 承诺。
16. **分片下发容错**：客户端 Load 分片收齐超时（如 10s）后主动重新发起整包 Load 请求（服务端幂等），替代永挂。
17. **1MB 预检**：客户端发送前检查 payload 大小，超限时触发显式 `save_warning`（提示清理背包/仓库），并上报 NetDiag；同时评估把大头字段（S3 审计显示的 top 字段）拆 key 的长期方案。

### 第三层：可观测性与演练（保证下次震荡可定位、可自愈）

18. **补三个关键指标**（NetDiag 体系已在，成本低）：服务端 slot 锁持有时长分布 + 当前泄漏数；客户端 pending age 最大值；备份写成功率（真实成功，不是假成功）。
19. **GM 诊断命令**：查询/强制释放指定 userId 的 slot 锁；查询玩家最近一次 SaveResult 的失败原因分布。
20. **混沌演练用例**：模拟 serverCloud "ok-but-empty"、"回调永不返回"、"回包丢失" 三种降级形态，验证第一、二层修复后全链路能自愈。

### 落地顺序建议

第一层 1~7 全部是小 diff、互相独立、不动存档格式，建议作为一个止血版本尽快评审；其中 **#1（锁 TTL）+ #2（pending 超时）+ #4（空读熔断）** 三条直接覆盖三条根因链。第二层按 8/9（数据安全）→ 10/11（可用性）→ 其余的顺序分版本灰度。

## 五、排障验证建议（确认现场命中哪条根因）

拿到震荡时段的服务端/客户端日志后按以下特征分流：

| 日志特征 | 命中根因 |
|---|---|
| 服务端大量 `SaveGame DEFERRED: slot N is locked`，且同一 userId 持续数分钟以上 | 根因链 A（锁泄漏） |
| 客户端 `REJECT: load already pending` / `FetchSlots already pending` 反复出现 | 根因链 B（pending 卡死） |
| 服务端 `SaveGame: no userId for connKey=...` 密集出现 | 根因链 B 变体（identity 竞态） |
| 客户端 `needMigration=true but migration is DEPRECATED, treating as new player` 出现在老玩家 | 根因链 C（空读当新玩家）——需立即核对该玩家云端 `slots_index` 是否被覆盖 |
| 客户端 `Save FAILED ... backup_failed` 密集 | P1-10/11（备份链放大失败） |
| 客户端 `Save TIMEOUT (no response in 15s)` 密集但服务端无对应错误 | 回包丢失/挂起（叠加 P1-10 的无回包路径） |

# 存档与掉线问题优化方案-执行清单

## 目标

降低两类问题：

1. 高频保存导致的超时、重试、锁竞争、限流
2. 上述问题被玩家感知成“掉线”

- [存档触发情况清单-代码基准.md](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/docs/架构文档/存档触发情况清单-代码基准.md>)

1. 先改系统级调度：`AUTO_SAVE_INTERVAL = 120`、自动存档改成“距上次成功保存 120 秒后触发”、取消重连后立即直存。

## C/S 复核结论

给执行 AI 的结论，直接按这个理解：

1. 没有发现“存档路径主动断开连接”的直接证据。  
服务端存档主路径是返回失败结果，不是主动 `Disconnect`。断线清理在 [server_main.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/server_main.lua:265)。

2. 保存失败最常见的放大点是 `slot_locked`，不是硬断线。  
服务端在 [server_main.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/server_main.lua:916) 发现锁冲突时直接回 `slot_locked`。

3. 奖励类服务端逻辑会和普通保存直接竞争同一槽位锁。  
典型点：
   - [DaoTreeHandler.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/network/DaoTreeHandler.lua:207)
   - [SealDemonHandler.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/network/SealDemonHandler.lua:236)
   - [TrialHandler.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/network/TrialHandler.lua:50)

4. 客户端对保存失败的体感放大，确实有代码证据。  
`SaveSystem` 15 秒超时即判失败并进入警告 / 重试链，[SaveSystem.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/systems/SaveSystem.lua:99)。  
`CloudStorage` 的 pending callback 30 秒才清理，[CloudStorage.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/network/CloudStorage.lua:62)。  
服务端限流后会发 `S2C_RateLimited`，[server_main.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/server_main.lua:136)，客户端会把 saving 解锁并保留脏状态准备重试，[client_main.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/client_main.lua:309)。

5. 当前保存仍是全量序列化、全量上传。  
入口在 [SavePersistence.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/systems/save/SavePersistence.lua:97)；这决定了高频玩法必须降频。

结论：

`当前掉线反馈更像“保存调度 + 锁竞争 + 超时 / 限流放大的体感问题”，不是“保存代码直接踢连接”。`

## 会话式合并保存参数

执行时默认按这组参数实现，除非有新决定：

1. 空闲 `15 秒` 收口保存
2. 关闭当前 UI 时强制收口保存
3. 切章节、`F5`、保存并退出、自动存档到点时强制收口保存
4. 连续 `5 次` 操作后强制收口一次
5. 连续 `20 秒` 未落盘则强制收口一次
6. 收口失败时不能清脏，必须保留待重试状态
7. **限量丹药（`SendBuyPill` 授权类）不参与会话合并，保持即时 `save_request`**。原因：限量丹药的炼制需要服务端 `SendBuyPill` 授权扣额度，如果客户端将落盘延迟到会话收口，期间崩溃会导致服务端额度已扣但客户端存档无丹药。涉及的丹药：虎骨丹（`DoCraftTiger`）、蛇胆丹（`DoCraftSnake`）、钻石丹（`DoCraftDiamond`）、淬体丹（`DoCraftTempering`）、龙血丹（`DoCraftDragonBlood`）、挑战丹（`DoCraftChallengePill`）。非限量丹药（气丹、筑基丹、金丹砂、元婴果、九转金丹、令牌箱）正常参与会话合并。

## 自动存档结论

执行时直接按这个结论，不需要再重新讨论：

1. 自动存档不取消
2. 自动存档改回 `120 秒`
3. 原因不是“120 秒更优雅”，而是：
   - `60 秒` 相比 `120 秒` 的额外收益有限
   - 当前项目已有大量关键点保存
   - 当前保存是全量快照，`60 秒` 会明显增加碰撞、重试、限流和掉线体感

## 验收标准

### 功能正确性

1. 高价值低频掉落仍然即时保护：
   - 特殊装备
   - 神器碎片
   - 福缘果
2. 挑战入场扣门票不会因强退 / 重登轻易回档
3. 镇狱塔退出不再触发单点保存

### 保存频率

1. 连续炼丹 / 洗练 / 附灵 / 拆解时，保存次数显著下降
2. 不再出现“刚事件保存完又立刻 auto-save”的高概率连发
3. 重连后不再马上拉起一次直存

### 掉线体感

1. `slot_locked` / `rate_limited` 造成的连续警告明显下降
2. “存档同步中，请保持网络连接...”提示出现频率下降
3. 网络恢复后的首个保存成功率提升

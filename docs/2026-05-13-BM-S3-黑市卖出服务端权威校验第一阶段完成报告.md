## 2. 当前服务端卖出口径盘点表

`HandleSell()` 中的校验链路（修改后）：

| 步骤 | 校验内容 | 数据源 | 可信度 |
|------|---------|--------|--------|
| 1 | 并发锁 `sellActive_[connKey]` | 服务端内存 | ✅ 可信 |
| 2 | 白名单 `BMConfig.ITEMS[id]` | 配置常量 | ✅ 可信 |
| **2.5** | **SellGuard 安全分类判定** | **配置常量** | **✅ 可信（本次新增）** |
| 3 | 数量 `amount >= 1` | 客户端请求 | ✅ 可信（服务端验证） |
| 4 | 背包持有量 `held >= amount` | `saveData.backpack`（服务端快照） | ⚠️ 快照可能过期 |
| 5 | 库存检查 `stock < MAX_STOCK` | `MoneyCost(SYSTEM_UID, stockKey)` | ✅ 可信 |
| 6 | 扣除+发放 BatchCommit | 服务端原子操作 | ✅ 可信 |

**核心风险点**: 步骤 4 的 `saveData.backpack` 是服务端云存档快照，存在 SaveSession 延迟窗口（最长 20 秒或 5 次操作），客户端可在本地消耗物品后、快照更新前发起卖出请求，造成「幻影卖出」。

## 3. 安全类 / 高风险类判定规则

### 判定原则

**"宁可拒绝，不可误发仙石"** — 默认保守策略

### 分类逻辑

```
商品 ID → BMConfig.ITEMS[id].category
  ├─ category ∈ {rake, bagua, tiandi}  →  ✅ 安全（整分类白名单）
  ├─ category = consumable_mat
  │     ├─ id ∈ SAFE_CONSUMABLE_MAT_IDS  →  ✅ 安全
  │     └─ 其他（如 gold_brick）          →  ❌ 高风险
  └─ 其他分类（lingyu/herb/skill_book/special_equip/未来新分类）
        →  ❌ 高风险（默认保守）
```

### 安全类判定依据

这些商品**没有**「本地先消耗→延迟落盘→卖旧快照」的攻击路径：
- **碎片类**(rake/bagua/tiandi): 消耗路径为合成法宝，合成后碎片变法宝，不存在"持有碎片但被消耗"的状态不一致窗口
- **龙鳞**(dragon_scale_*): 消耗路径为镶嵌法宝，镶嵌后龙鳞直接变为属性加成，同理
- **令牌盒**(token_boxes): 已由 BM-01A 热修补仓库一致性校验

### 高风险类判定依据

这些商品**存在**本地消耗路径，且服务端无法独立证明消耗前的真实持有量：
- **附灵玉**(lingyu): 消耗路径为装备附灵
- **金砖**(gold_brick): 消耗路径为出售换金币
- **草药**(herb): 消耗路径为炼丹
- **技能书**(skill_book): 消耗路径为学习技能
- **特殊装备**(special_equip): 消耗路径为装备/分解

## 4. 高风险商品被本阶段显式拒绝（53 项）

| 分类 | 数量 | 风险等级 | 示例 |
|------|------|---------|------|
| lingyu（附灵玉） | 4 | P0 | lingyu_xuesha_lv1, lingyu_xuesha_lv2, ... |
| consumable_mat（金砖） | 1 | P0 | gold_brick |
| herb（草药） | 10 | P1 | herb_pill, herb_essence, herb_lingzhi, ... |
| skill_book（技能书） | 34 | P1 | skill_book_* (各系各级) |
| special_equip（特殊装备） | 4 | P2 | jiutian_ring, ice_silk_robe, ... |
| **合计** | **53** | | |

**用户可见影响**: 玩家尝试在黑市卖出以上商品时，UI 显示 `"该商品暂不支持出售"`。

**为什么拒卖比继续放行更安全**:
1. 服务端 `saveData.backpack` 快照存在最长 20 秒的延迟窗口
2. 修改过的客户端可以跳过 BM-S2 客户端门禁直接发送 `C2S_BMSell`
3. 高风险商品被卖出后发放仙石（硬通货），造成的经济损害不可逆
4. 拒卖只影响黑市卖出通道，玩家仍可正常使用这些物品的消耗功能

## 6. 自动回收受影响评估

**结论: 不受影响**

自动回收（胤/RecycleTick）调用链:
```
ScheduledUpdate → RecycleTick() → ExecuteRecycle() → MoneyCost(SYSTEM_UID, stockKey)
```

HandleSell 调用链:
```
C2S_BMSell → HandleSell() → SellGuard.CheckSellAllowed() → ...
```

两条路径**完全独立**：
- RecycleTick/ExecuteRecycle 不调用 HandleSell()
- SellGuard 模块没有被自动回收路径引用的接口
- 自动回收操作的是系统库存（SYSTEM_UID），不涉及玩家背包快照

## 10. 未来新增商品扩展性影响（§5）

### 默认保守原则

新增商品到 `BMConfig.ITEMS` 时：
- 如果 `category` 属于 `{rake, bagua, tiandi}` → **自动允许卖出**
- 如果 `category = consumable_mat` 且 ID 不在 `SAFE_CONSUMABLE_MAT_IDS` → **自动拒卖**
- 如果 `category` 是任何其他值（包括未来新增分类）→ **自动拒卖**

### 放行新商品的操作

如需放行新商品，只需修改 `BlackMarketSellGuard.lua`：
- 新分类整体安全 → 添加到 `SAFE_CATEGORIES`
- 新 consumable_mat 单品安全 → 添加到 `SAFE_CONSUMABLE_MAT_IDS`

无需修改 HandleSell 或其他文件。

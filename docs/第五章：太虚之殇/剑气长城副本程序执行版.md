# 剑气长城副本程序执行版

> 项目：人狗仙途  
> 位置：第五章·太虚之殇 / 藏经书阁底部入口 → 剑气长城回廊  
> 当前口径：以现有代码实现为准（2026-07-11）  
> 关联实现：`scripts/config/SwordWallConfig.lua`、`scripts/systems/SwordWallSystem.lua`、`scripts/network/SwordWallHandler.lua`、`scripts/ui/SwordWallUI.lua`

---

## 1. 当前定位

剑气长城是已经落地的**门票型单人多波挑战副本**，不是 `docs/设计文档/多人副本.md` 中的“四兽岛持久世界 Boss 多人副本”。

当前实现特点：

1. 不切换引擎多人模式，不使用 `persistent_world`。
2. 复用现有 background_match C/S 连接，仅通过 RemoteEvent 做开始注册、宝箱结算、积分商店。
3. 战斗发生在第五章现有地图 `ch5_sword_corridor` 区域内，不进入独立异空间。
4. 副本怪由客户端本地生成并战斗；奖励由服务端权威生成和结算。
5. 副本奖励不走普通怪物掉落；全部通过最终宝箱与积分商店。

## 6. 客户端状态机

实现：`scripts/systems/SwordWallSystem.lua`

### 6.1 状态字段

| 字段 | 说明 |
| --- | --- |
| `active` | 是否处于剑气长城副本中 |
| `runId` | 本次副本 ID，本地生成并上报服务端注册 |
| `state` | `idle` / `wave` / `wave_clear_delay` / `chest_ready` / `claiming` / `completed` / `failed` |
| `waveIndex` | 当前波次 |
| `waveMonsters` | 当前波存活怪引用 |
| `chestOpened` | 宝箱是否已开启 |
| `rewardClaimed` | 奖励是否已确认领取 |
| `entryReturn` | 进入前玩家位置，完成或失败后返回 |

### 6.2 进入流程

`SwordWallSystem.Enter(gameMap)`：

1. 检查当前未在剑气长城副本中。
2. 检查声望挑战、试炼塔、镇妖塔均不在进行中。
3. 检查玩家存在且存活。
4. 检查 `immortal_essence_blood >= 1`。
5. 本地扣除门票。
6. 初始化状态与 `runId`。
7. 记录进入前返回点。
8. 将玩家传送到 `PLAYER_ENTER_POS`。
9. 向服务端发送 `C2S_SwordWallStart` 注册 `runId`。
10. 生成退出传送阵。
11. 刷新第 1 波怪物。

### 6.3 波次推进

1. `EventBus.On("monster_death")` 监听副本怪死亡。
2. 只处理 `isSwordWall=true` 且 `challengeRunId` 匹配当前 `runId` 的怪。
3. 当前波全灭后：
   - 若不是最后一波，进入 `wave_clear_delay`，倒计时后刷新下一波。
   - 若是最后一波，进入 `chest_ready`，生成结算宝箱并在小地图标记。

监听器当前已在 `SwordWallSystem.Init()` 中先退订旧监听，再重新订阅，避免切角色 / 重建世界后累加监听。

### 6.4 失败与返回

失败入口：`SwordWallSystem.Fail(gameMap, reason)`

触发场景：

1. 主动使用退出传送阵撤退。
2. 外部死亡 / 副本中断逻辑调用。

失败结果：

1. 清理副本怪、退出传送阵、结算宝箱。
2. 移除小地图宝箱标记。
3. 清理战斗特效。
4. 玩家返回 `entryReturn`，并回满血。
5. 状态重置为 `idle`。

完成返回：`SwordWallSystem.UpdateReturn(dt, gameMap)` 只在 `state="completed"` 且 `_returnTimer` 存在时执行。

## 7. 服务端协议与结算

### 7.1 协议

定义：`scripts/network/SaveProtocol.lua`

| 方向 | 协议 | 字符串 | 用途 |
| --- | --- | --- | --- |
| C2S | `C2S_SwordWallStart` | `C2S_SWStart` | 副本开始注册 |
| C2S | `C2S_SwordWallClaim` | `C2S_SWClaim` | 宝箱结算请求 |
| C2S | `C2S_SwordWallShopQuery` | `C2S_SWShopQuery` | 查询积分与商店 |
| C2S | `C2S_SwordWallShopBuy` | `C2S_SWShopBuy` | 积分商店购买 |
| S2C | `S2C_SwordWallClaimResult` | `S2C_SWClaimResult` | 宝箱结算结果 |
| S2C | `S2C_SwordWallShopData` | `S2C_SWShopData` | 商店数据 |
| S2C | `S2C_SwordWallShopBuyResult` | `S2C_SWShopBuyResult` | 商店购买结果 |

服务端注册：`scripts/server_main.lua`

- `HandleSwordWallStart`
- `HandleSwordWallClaim`
- `HandleSwordWallShopQuery`
- `HandleSwordWallShopBuy`

### 7.2 宝箱结算

服务端实现：`SwordWallHandler.HandleClaim`

当前流程：

1. `claimActive_[connKey]` 防并发。
2. 解析 `runId`。
3. 若 `runId` 已在本地缓存 `claimedRuns_` 中，返回缓存结果，实现短期幂等重试。
4. 检查当前连接是否通过 `HandleStart` 注册过同一 `runId`。
5. 检查通关最短时间：`MIN_CLEAR_TIME = 25` 秒。
6. 服务端随机生成 `10~30` 剑气积分。
7. 服务端生成 T10 青色奖励装备。
8. `BatchCommit("剑气长城宝箱结算")` 中原子增加 `sword_wall_points`。
9. 成功后缓存结果、清理 active run、回包给客户端。
10. 客户端收到成功回包后将装备入库，并显示奖励弹窗。

当前宝箱结算的可靠性边界：

- 剑气积分已经服务端落到 Money 域。
- 装备通过回包交给客户端入库并随本地后续保存落盘。
- 若回包成功但客户端崩溃 / 断线且未保存，当前没有完整 WAL 自动补发闭环。
- `SwordWallConfig.WAL_ITEM_KEY` 当前只是配置预留，主流程未完成 WAL 补偿。

### 7.3 服务端断线清理现状

`SwordWallHandler.OnDisconnect(connKey)` 会清理 `activeRuns_[connKey]`，但当前 `server_main.lua` 的断线路径尚未调用该函数。

因此这是当前已知缺口：断线后 active run 可能残留到进程生命周期结束。文档不得把“断线清理已接线”写成已完成。

注意：当前实际商品数为 9 个。旧测试 `test_sword_wall.lua` 中“商店应有 8 个商品”“四仙剑 2000 积分”的断言已落后于配置。

### 8.2 查询流程

`SwordWallHandler.HandleShopQuery`：

1. 从 `Session.connSlots_[connKey]` 取当前角色 slot。
2. 读取 `serverCloud.money:Get(userId)` 获取 `sword_wall_points` 余额。
3. 对有限购字段的商品，使用 `BatchGet` 读取角色级已购次数。
4. 限购 key 格式：`sw_shop_bought_s{slot}_{itemId}`。
5. 回包包含：`balance`、`shop`、`boughtCounts`。

### 8.3 购买流程

`SwordWallHandler.HandleShopBuy` + `_doBuy`：

1. 校验 userId 和服务端当前 slot。
2. `shopActive_[connKey]` 防并发。
3. 查找商品。
4. 若有限购，读取 `sw_shop_bought_s{slot}_{itemId}` 并判断是否达到上限。
5. 查询积分余额。
6. 服务端生成奖励数据：
   - 装备：`LootSystem.CreateSpecialEquipment(equipId)`
   - 消耗品：`{ type="consumable", consumableId=..., count=... }`
7. `BatchCommit("剑气积分商店购买")`：
   - `MoneyCost(userId, sword_wall_points, price)`
   - 有限购时 `ScoreAdd(userId, limitKey, 1)`
8. 成功后读取新余额，并把奖励 JSON 回包给客户端。
9. 客户端收到成功回包后入包并 `EventBus.Emit("save_request")` 请求保存。

当前购买可靠性边界：

- 已有服务端连接级防并发锁。
- 扣积分与限购计数在同一次 BatchCommit 中完成。
- 限购是角色级，不是账号级。
- 当前请求未携带 `clientTxnId`，服务端没有跨请求幂等结果 key；极端网络重试仍不是完整可靠交易。
- 客户端成功入包后只发 `save_request`，不是阻塞式强保存。

## 11. 运行合同

当前已静态确认的运行链路：

```text
入口 NPC
→ SwordWallUI.Show
→ SwordWallSystem.Enter 扣门票/注册 runId/刷怪
→ EventBus monster_death 推进波次
→ 最终 BOSS 死亡生成宝箱
→ SwordWallSystem.OpenChest
→ C2S_SwordWallClaim
→ SwordWallHandler.HandleClaim 服务端加积分并生成装备
→ S2C_SwordWallClaimResult
→ SwordWallSystem.OnClaimSuccess 入库装备并展示奖励
```

商店链路：

```text
SwordWallUI.ShowShop
→ C2S_SwordWallShopQuery
→ SwordWallHandler.HandleShopQuery 读取余额和限购
→ S2C_SwordWallShopData
→ SwordWallUI._doBuy
→ C2S_SwordWallShopBuy
→ SwordWallHandler.HandleShopBuy/_doBuy 扣积分/写限购/生成奖励
→ S2C_SwordWallShopBuyResult
→ SwordWallUI_HandleShopBuyResult 客户端入包
→ EventBus.Emit("save_request")
```

本次文档更新只做静态对照，未运行真实 serverCloud 交易，也未执行构建。

当前铸造系统已经不是旧文档中的“第五章复用第四章 UI”结构，而是：

```text
地图交互物
  -> NPCDialog 按 interactType 路由
  -> SwordForgeUI / DragonForgeUI 展示
  -> ForgeSystem.Check 验证
  -> ForgeSystem.DispatchGenerator 生成
  -> 扣金币、装备、消耗品
  -> 入包或原地替换装备栏武器
  -> save_request + equipment_changed
```

### 2.2 输入模式

| 字段 | 当前能力 |
|------|----------|
| `inputMode = "bag_consume"` | 从背包和消耗品栏扣材料 |
| `inputMode = "equip_weapon"` | 要求玩家先装备指定武器或武器池内任意武器 |
| `fromBag` | 消耗 1 件背包装备 |
| `fromBag2` | 消耗第 2 件背包装备 |
| `fromBagList` | 消耗多件背包装备；允许重复 `equipId`，会占用不同格子 |
| `materials` | 消耗普通消耗品，可设置数量 |
| `taixu_token` | 第四章龙极令的历史专用字段 |
| `gold` | 金币消耗 |

### 2.3 输出模式

| 模式 | 行为 |
|------|------|
| `add_to_bag` | 生成新物品并放入背包；消耗装备腾出的格子可承接产物 |
| `replace_weapon` | 原地改写当前装备栏武器，不占背包格 |

`replace_weapon` 并不统一保留洗练：

- 龙神圣器、第五章解封古剑：保留 `forgeStat`。
- 第三章黄沙换兵：明确清除 `forgeStat`。
- 第六章真仙剑：清除洗练、旧灵性和旧圣性，重新生成随机圣性。

### 2.4 当前生成器

| generator.type | 用途 |
|----------------|------|
| `fixed_special_equipment` | 固定特殊装备模板入包 |
| `special_equipment` | 第四章龙神圣器原地替换 |
| `random_huangsha_weapon_except_equipped` | 第三章随机换成另一把黄沙武器 |
| `fabao` | 生成龙极令、龙魂令、龙王令 |
| `longhunling` | 旧龙魂令升级生成器，仅保留兼容，现行配方不再使用 |
| `saint_equipment` | 生成固定模板，并处理圣性属性 |
| `jiefeng_sword` | 装备栏封印古剑升级，清除灵性并生成圣性 |
| `true_sword` | 装备栏解封仙剑二次升级，清除洗练/旧灵性/旧圣性并重生成圣性 |
| `random_lingqi_mixed` | 普通灵器 / 套装灵器概率混合生成 |

- 需要为二次解封增加独立生成器或扩展 `jiefeng_sword`，明确清除洗练、灵性和旧圣性，再生成新圣性。

### 4.4 镇界仙甲

| 类型 | 定稿值 |
|------|--------|
| 品级 | `T11 / 仙1 / 红色圣器` |
| 主属性 | `def +210` |

### 4.5 哼哈双刃

| 类型 | 定稿值 |
|------|--------|
| 品级 | `T11 / 仙1 / 红色圣器` |
| 主属性 | `atk +262.5` |
| 打造副属性 | `critRate +7.92%~9.9%`、`critDmg +39.6%~49.5%` 随机；`wisdom +23` 固定 |
| 圣性 | 随机 1 条圣性属性 |
| 特效 | **哼哈震杀**：普攻15%概率追加 `50000` 点固定伤害，冷却1秒 |

固定伤害不读取攻击、防御、暴击伤害或天诛倍率，不可暴击、不可触发天诛，也不再触发其他普攻特效。固定伤害还必须绕过攻击方最终增伤，以及目标的易伤、区域增伤、洗髓增伤、护盾减伤等最终乘区；但仍需保留正常扣血、死亡、仇恨和多人副本伤害上报。运行时统一使用 `HitResolver.FixedDamage`，禁止仅用 `skipCalcDamage/skipCrit` 冒充固定伤害。数值不随角色成长，确保第六章当前版本强势，后续章节自然衰减。

### 4.6 呱呱均灵环

| 类型 | 定稿值 |
|------|--------|
| 品级 | `T11 / 仙1 / 金色仙器` |
| 主属性 | `atk +187.5` |
| 打造副属性 | `def +21.6~28.8`、`maxHp +108~144`、`hpRegen +9~12` 随机 |
| 仙缘/圣性 | 三条打造副属性不配置悟性、福缘、根骨、体魄；额外生成1条随机圣性属性 |
| 特效 | **呱呱均灵**：当前总值最低的仙缘属性额外获得150点加成；并列最低时同时生效；唯一，同类取最高值 |

副属性种类完整参考第五章均灵环的 `防御 / 生命 / 生命回复`，按 T11 金色仙器 `1.2~1.6` 区间打造。特效直接复用 `xianyuan_lowest_boost`，`bonus` 为 `150`。

`xianyuan_lowest_boost` 的“唯一”统一解释为：两个戒指槽同时装备普通均灵环、呱鸣大王戒或呱呱均灵环时，不累加多个加成，只取所有同类特效中最高的 `bonus` 并应用一次；装备槽位及遍历顺序不得影响结果。因此 `100 + 100 = 100`，`100 + 150 = 150`。均灵加成必须先于根骨派生防御、体魄派生生命的结算，确保四种仙缘获得加成时均完整生效。

### 4.7 蚀界仙氅

## 八、经典问题复盘与强制规则

### 8.1 输入字段语义

| 字段 | 唯一语义 | UI 必须显示 | 执行器行为 |
|------|----------|-------------|------------|
| `equipSource` | 当前装备栏必须装备指定 `equipId` | `指定装备名（需要装备）` | 校验装备栏并原地读取该实例 |
| `equipSourcePool` | 当前装备栏必须装备池内任意一件 | `池名称（需要装备）` | 校验装备栏实例属于白名单 |
| `fromBag/fromBag2/fromBagList` | 从背包消耗装备 | 装备名与持有/缺少状态 | 定位背包格并在成功后扣除 |
| `materials` | 消耗品数量 | 名称、持有数、需求数 | 成功后统一扣除 |

`inputMode="equip_weapon"` 时必须且只能配置 `equipSource` 或 `equipSourcePool` 其中一种。禁止用 `fromBag` 表达装备栏输入，也禁止 UI 根据当前装备类型自行猜测要求。

### 8.2 展示与状态分离

材料行包含两类信息，禁止混用：

1. **配方要求**：来自 `FORGE_RECIPES`，始终是主文案。
2. **玩家当前状态**：来自装备栏或背包，只能用于勾叉、数量和“当前装备”诊断。

典型错误：

```lua
-- 错误：当前装备会覆盖配方要求。玩家装备黄沙武器时，
-- 真·诛仙会被错误显示成黄沙武器，且“需要装备”一起丢失。
local name = item and item.name or requiredText
```

正确规则：

```lua
local displayText = requiredName .. "（需要装备）"
if not matched and currentItem then
    displayText = displayText .. "；当前装备：" .. currentItem.name
end
```

### 8.3 已发生的经典问题

| 问题 | 根因 | 永久防线 |
|------|------|----------|
| 黄沙武器重铸后卸下失效 | 原地替换未补齐运行时 `slot/type` | 所有 `replace_weapon` 统一执行 `NormalizeEquipmentRuntimeFields`，流水线验证卸下再装备 |
| 真·诛仙显示成黄沙武器 | UI 用当前装备名称覆盖配方要求名称 | 要求名称固定来自 `equipSource/equipSourcePool`，当前装备只追加显示 |
| 配方未标“需要装备” | 标记由调用方拼接，容易在覆盖或新分支中丢失 | `BuildEquippedWeaponRow` 组件内部统一追加“（需要装备）” |
| PNG 材料显示成资源路径 | 把未知 `icon` 直接作为 Label 文本 | 材料图标统一使用 `AdaptiveIcon/IconUtils.IsImagePath` |
| 材料列表过高、频繁滚动 | 行距与内边距多层叠加 | 材料行固定紧凑高度，列表使用统一紧凑 gap/padding |
| UI 与实际扣除不一致 | 旧配方表与统一配方表双数据源 | UI、`Check`、`Execute` 只读取 `FORGE_RECIPES` |
| 仙器与圣器副属性范围显示相同 | `SwordForgeUI` 将所有品质硬编码成 `1.1~1.5` | 品质区间由 `ForgeStatRules` 唯一提供；红色 `1.2~1.5`，金色 `1.2~1.6` |
| 固定装备被标成随机、随机装备生成固定值 | 只看 UI 标题或模板值，没有核对生成器与 `fixedSubStats` | 预览模式由配方生成器和模板共同判定；实际生成样本反查范围 |
| 圣衣、圣氅显示灵性但产物生成圣性 | UI 直接读取模板历史字段，忽略 `saint_equipment` 生成器语义 | 额外属性类型由统一生成器规则判定 |
| 洗练整数上限少显示 1 点 | 范围显示向下取整，实际生成四舍五入 | `GetRerollRange` 与 `RollRerollValue` 共用最终取整函数 |
| 固定伤害仍被最终乘区修改 | 只跳过防御和暴击，仍进入攻击方增伤及目标易伤/减伤 | 固定伤害统一走 `HitResolver.FixedDamage`，并用真实 `Monster` 覆盖所有最终乘区 |

### 8.4 新增或修改配方的必测项

1. 校验站点、顺序、产物、生成器和输入模式。
2. 装备栏输入校验正确装备、错误装备、空装备三种状态。
3. UI 必须显示配方要求名称和“需要装备”，错误装备不得覆盖要求。
4. `Check` 失败不得扣材料；`Execute` 成功后扣除与产出一致。
5. `replace_weapon` 必须验证实例 ID、`slot/type`、卸下和重新装备。
6. PNG/emoji 材料图标分别验证，不得泄漏资源路径。
7. 长材料列表验证间距、按钮和滚动范围。
8. 配方文档、GM 测试包和自动化合同必须同步更新。
9. 固定伤害必须使用真实怪物验证，在狂暴、易伤、区域增伤、洗髓和目标减伤同时存在时仍保持配置值。
10. 每条有明确产物的配方，必须用实际生成样本验证三条副属性全部落在界面展示范围内。
11. 禁止在 UI 内再次声明品质波动常量；新增品质或修改倍率只能改统一规则模块。
12. 圣性/灵性的界面文案必须由生成器类型决定，不得仅凭模板历史字段判断。

# 2026-05-14 C5-PREP 第五章开发护栏冻结任务卡

## 1. 任务定位

这不是一张“大优化”任务卡。  
这是第五章正式开发前的**开发护栏冻结卡**。

目标只有一个：

- 在**不继续大改运行时代码**的前提下
- 把第五章开发过程中最容易重新引入风险的边界、准入规则、验收要求
- 一次性冻结清楚

本任务的价值不在“新增功能”，而在：

- 防止第五章开发把黑市、存档、网络波动、线上稳定性问题重新带回来

---

## 2. 为什么现在做

截至当前基线：

1. `R3` 已完成并通过
2. 黑市卖出已恢复到可用状态
3. `N1R` 已收口
4. 自动回收代码层没有新的明显 bug，转入观察

这意味着：

- 第五章前已经**不需要继续推进大规模结构优化**
- 但如果没有护栏，第五章新增商品、新 UI、新存档字段、新写请求入口，很容易重新污染当前稳定面

所以这一步现在做，性价比最高。

---

## 3. 本任务目标

### 3.1 冻结黑市新增商品准入规则

必须把第五章涉及黑市的新增商品明确分成两类：

1. `A 类：纯库存型`
   - 无本地先变化路径
   - 可直接按现有黑市配置接入

2. `B 类：受保护接入型`
   - 可直接使用
   - 或会被本地系统消耗
   - 或会被合成 / 激活 / 打造 / 仓库等流程改变状态
   - 这类不得只改 `BlackMerchantConfig.lua` 直接接入
   - 必须同时说明：
     - 是否需要黑市保护锁
     - 是否需要本地操作闸
     - 是否需要新增测试

### 3.2 冻结第五章新增高风险写入口规则

凡是第五章新增 UI 里包含：

- `SendRemoteEvent`
- `SendToServer`
- 兑换 / 购买 / 提交 / 抽取 / 领奖 / 合成确认 / 激活确认

都必须先判断是否属于“高风险写入口”。

若属于高风险写入口，则入口必须接：

```lua
if NetworkStatus.IsDisconnected() then
    return false
end
```

或同语义的阻断逻辑。

### 3.3 冻结第五章新增存档影响点规则

凡是第五章新增系统如果会修改：

- 玩家资源
- 背包 / 仓库
- 装备 / 技能 / 图鉴
- 章节进度 / 挑战进度
- 任意需要持久化的状态

都必须明确：

1. 是否接 `save_request`
2. 是否进入 `SaveSession`
3. 是否新增 DTO / 存档字段
4. 是否影响旧存档兼容

如果执行 AI 说不清这 4 条，则不允许直接上线开发实现。

### 3.4 冻结第五章验收最小门

第五章开发中，任何新系统上线前至少要经过：

1. 黑市准入检查（如涉及交易商品）
2. 存档影响检查
3. 高风险写入口检查
4. 最小脚本测试 / runner 挂接检查

## 二、核心伤害公式

### 2.1 CalcDamage

**位置**: `config/GameConfig.lua:537`

```lua
function GameConfig.CalcDamage(atk, def)
    if atk + def <= 0 then return 1 end
    local dmg = (atk * atk) / (atk + def)
    local variance = 0.9 + math.random() * 0.2  -- ±10% 随机波动
    return math.max(1, math.floor(dmg * variance))
end
```

**公式**: `DMG = ATK² / (ATK + DEF) × rand(0.9~1.1)`，最低 1

**特性**:
- DEF=0 → DMG=ATK（线性）
- DEF=ATK → DMG=ATK/2（50%减伤）
- DEF→∞ → DMG→0（渐近线，永远>0）
- 无穿甲参数，无技能倍率参数（倍率在调用前乘入 ATK）

### 2.2 暴击判定

**位置**: `entities/Player.lua:322`

```lua
function Player:ApplyCrit(damage)
    if math.random() < self:GetTotalCritRate() then
        return math.floor(damage * self:GetTotalCritDmg()), true
    end
    return damage, false
end
```

- 暴击率 = 5%(基础) + equipCritRate + titleCritRate + collectionCritRate
- 暴击伤害 = 1.5×(基础) + equipCritDmg

## 三、伤害管线详解

### 3.1 普攻伤害管线（9层）

**入口**: `CombatSystem.AutoAttack()`

```
第1层: 属性聚合
  ├─ player:GetTotalAtk()
  │   = (base + equip + collection + title + seaPillar + medal + artifactTiandi)
  │     × (1 + titleAtkBonus + buffAtkPercent)
  │
第2层: CalcDamage(totalAtk, monster.def)
  ├─ ATK²/(ATK+DEF) × ±10%
  │
第3层: ApplyCrit(damage)
  ├─ 概率暴击 → damage × critDmg倍率
  │
第4层: 噬魂增伤 buff（一次性消耗乘区）
  ├─ damage × (1 + buffPercent)
  │
第5层: Monster:TakeDamage 内部增伤
  ├─ 5a. 易伤 debuff: × (1 + vulnPercent)
  ├─ 5b. 洗髓增伤: × (1 + washLevel × 0.01)，上限26%
  │
第6层: 重击判定（独立伤害，概率触发，真实伤害）
  ├─ heavyDmg = floor(totalAtk + totalHeavyHit)  ← 不走 CalcDamage，无视防御
  ├─ heavyDmg × (1 + heavyDmgBonus)  // 根骨衍生加成
  ├─ ApplyCrit(heavyDmg)
  ├─ Monster:TakeDamage(heavyDmg, player)  // 享受易伤/洗髓增伤
  │   ※ 重击和裂地(TryHeavyStrike)都是真实伤害，不走 CalcDamage
  │   ※ 龙象功(charge_passive)同样是真实伤害，公式相同
  │
第7层: 装备特效触发 (EFFECT_HANDLERS)
  ├─ bleed_dot / wind_slash / shadow_strike / lifesteal_burst 等
  ├─ 各自独立走 CalcDamage → Monster:TakeDamage
  │
第8层: 神器被动追加打击（天蓬遗威，真实伤害）
  ├─ passiveDmg = floor(totalAtk × dmgMultiplier)  ← 不走 CalcDamage，无视防御
  ├─ Monster:TakeDamage(passiveDmg, player)  // 享受易伤/洗髓增伤
  │
第9层: 套装AOE (SetAoeSystem)
  └─ EventBus 事件触发，attrValue × coeff → CalcDamage
```

### 3.2 技能伤害管线

**入口**: `SkillSystem.CastDamageSkill()`

```
第1层: 技能有效攻击力
  ├─ effectiveAtk = totalAtk × (1 + equipSkillDmg + wisdomDerived)
  │   wisdomDerived = GetSkillDmgPercent() = floor(totalWisdom / 5) × 0.01
  │
第2层: 技能倍率应用
  ├─ scaledAtk = floor(effectiveAtk × skill.damageMultiplier)
  │
第3层: CalcDamage(scaledAtk, monster.def)
  │
第4层: ApplyCrit(damage)
  │
第5层: Monster:TakeDamage(damage, player)
  ├─ 易伤 + 洗髓增伤（同普攻）
  │
第6层: 连击判定（概率触发，重新施放整个技能）
  └─ comboChance = equipComboChance + classCombo + wisdomCombo
```

### 3.3 重击伤害管线（真实伤害，不走 CalcDamage）

**触发点**: `CombatSystem.AutoAttack()` 内联概率重击 + `CombatSystem.TryHeavyStrike()` 装备裂地重击

```
第1层: 重击概率判定
  ├─ 内联重击: 职业天赋基础概率 + 根骨衍生概率(constitution/25 × 1%)
  │   罗汉: 10%(天赋) + constitution/25 × 1%
  ├─ 裂地重击: 每N次攻击必定触发（装备 heavy_strike 特效）
  │
第2层: 重击伤害计算（真实伤害，无视防御）
  ├─ heavyDmg = floor(totalAtk + totalHeavyHit)  ← 不走 CalcDamage
  ├─ heavyDmg × (1 + heavyDmgBonus)  // 根骨衍生加成
  │
第3层: ApplyCrit(heavyDmg)
  │
第4层: Monster:TakeDamage(heavyDmg, player)
  ├─ 享受易伤/洗髓增伤，返回实际伤害用于浮字
```

### 3.4 各伤害来源是否走 CalcDamage

| 伤害来源 | 走 CalcDamage | 走 ApplyCrit | 走 TakeDamage增伤 |
|---------|:---:|:---:|:---:|
| 普攻 | ✅ | ✅ | ✅ |
| 重击（内联概率 + 裂地） | ❌ 真实伤害 | ✅ | ✅ |
| 主动技能（melee_aoe/lifesteal/aoe_dot/ground_zone） | ✅ | ✅ | ✅ |
| 龙象功释放 | ❌ 无视防御 | ✅ | ✅ |
| 奔雷式 DOT（电击） | ❌ 真实伤害 | ✅ 每tick | ✅ |
| 血怒引爆 | ❌ maxHP% 真伤 | ✅ | ✅ |
| 流血 DOT (bleed_dot) | ❌ ATK% 无视防御 | ❌ | ✅ 走 TakeDamage |
| 天蓬遗威（神器被动） | ❌ ATK×倍率 真伤 | ❌ | ✅ |
| 装备特效（wind_slash/shadow_strike） | ✅ | ❌ | ✅ |
| 套装AOE | ✅ | ❌ | ✅ |
| 结界/领域 DOT (BossMechanics) | ✅ | ❌ | — |

## 四、防御/受伤管线详解

### 4.1 Player 受伤管线（6层）

**入口**: `entities/Player.lua:763` `Player:TakeDamage(damage, source)`

```
第0层: 无敌判定
  ├─ invincibleTimer > 0 → 完全免疫，return
  │
第1层: 闪避（evade_damage 装备特效）
  ├─ 概率完全闪避 → 浮字"闪避"，return
  │
第2层: 装备减伤 (equipDmgReduce)
  ├─ damage × (1 - min(dmgReduce, 0.75))
  ├─ 来源: 披风(cape)主属性 + 套装加成
  ├─ 上限 75%
  │
第3层: 洗髓减伤（独立乘区）
  ├─ damage × (1 - washLevel × 0.01)
  ├─ 上限 26%（washLevel 上限 26）
  │
第4层: _preDamageHook（被动技能注入点）
  ├─ 御剑诀等被动注入: 可修改 damage 值或归零
  ├─ ⚠️ 当前只有一个槽位（单一 function），后注册覆盖前一个
  │
第5层: 护盾吸收
  ├─ 5a. 金钟罩护盾 (shieldHp): 优先吸收，完全吸收则 return
  ├─ 5b. 第四章神器护盾 (ArtifactCh4.AbsorbDamage): 二次吸收
  │
第6层: 免死判定 (death_immunity 装备特效)
  ├─ HP归零后概率触发 → HP=1 + 短暂无敌
  └─ 蜃影特效
```

### 4.2 Monster 受伤管线

**入口**: `entities/Monster.lua:594` `Monster:TakeDamage(damage, source)`

```
第1层: 易伤 debuff (vulnerability)
  ├─ damage × (1 + vulnPercent)
  │
第2层: 洗髓增伤（仅玩家本体伤害，宠物不享受）
  ├─ damage × (1 + washLevel × 0.01)
  ├─ 判定: source 拥有 GetWashLevel 方法
  │
第3层: 训练假人特判
  ├─ isTrainingDummy → 不扣血，触发闪烁和事件，HP保持满
  │
第4层: 扣血
  ├─ 副本BOSS: 不本地扣血（服务端控制）
  ├─ 普通怪: self.hp = self.hp - damage
  │
第5层: 死亡/阶段转换
  ├─ hp <= 0 → Die()（副本BOSS由服务端控制）
  └─ hp > 0 → CheckPhaseTransition() + CheckDualStateTransition()
  │
第6层: 返回实际伤害值
  ├─ return damage  -- 经过易伤/洗髓增伤后的实际值
  ├─ 调用方用于: 浮字显示、吸血回复计算、DPS统计
  └─ ⚠️ v1.7.9 新增，之前不返回导致浮字不显示洗髓/易伤加成
```

### 4.3 减伤层级叠算示例

假设 damage=1000, equipDmgReduce=0.20, washLevel=10, 御剑诀剑盾减30%, 护盾500:

```
原始伤害:                     1000
→ 装备减伤 20%:               1000 × 0.80 = 800
→ 洗髓减伤 10%（独立乘区）:    800 × 0.90 = 720
→ 御剑诀减伤 30%:             720 × 0.70 = 504
→ 护盾吸收 500:               504 - 500 = 4
最终扣血:                      4
```

## 五、属性聚合体系

### 5.1 Player 属性缓存机制

**位置**: `entities/Player.lua:217` `Player:_RecalcStatsCache()`

每帧最多重算一次（`_statsCacheFrame` 帧号去重），批量计算 6 个核心属性。

### 5.2 ATK 聚合

```
totalAtk = (base_atk + equipAtk + collectionAtk + titleAtk
            + seaPillarAtk + medalAtkFlat + artifactTiandiAtk)
           × (1 + titleAtkBonus + buffAtkPercent)
```

| 来源 | 字段 | 说明 |
|------|------|------|
| 基础 | `self.atk` | 等级成长 |
| 装备 | `equipAtk` | InventorySystem 聚合 |
| 图录 | `collectionAtk` | CollectionSystem |
| 称号 | `titleAtk` | TitleSystem |
| 海神柱 | `seaPillarAtk` | SeaPillarSystem |
| 勋章 | `medalAtkFlat` | MedalSystem |
| 天帝剑痕 | `artifactTiandiAtk` | ArtifactSystem_tiandi |
| Buff% | `buffAtkPercent` | CombatSystem.GetBuffAtkPercent() |
| 称号% | `titleAtkBonus` | TitleSystem |

### 5.3 DEF 聚合（含根骨衍生）

```
defBase = base_def + equipDef + collectionDef + seaPillarDef + artifactCh4Defense + medalDefFlat
defBonus = floor(totalConstitution / 5) × 0.01 + buffDefPercent
totalDef = floor(defBase × (1 + defBonus))
```

**根骨→DEF 衍生**: 每 5 点根骨 → +1% DEF

### 5.4 MaxHP 聚合（含体魄衍生）

```
hpBase = base_maxHp + equipHp + collectionHp + seaPillarMaxHp + medalHpFlat
hpBonus = floor(totalPhysique / 5) × 0.01
totalMaxHp = floor(hpBase × (1 + hpBonus))
```

**体魄→HP 衍生**: 每 5 点体魄 → +1% MaxHP

### 5.5 仙缘属性聚合

| 属性 | 聚合来源 | 衍生效果 |
|------|---------|---------|
| **根骨** | equip + pill + artifact + class×level + pet + wine + medal + collection | DEF+1%/5pt, 重击概率+1%/25pt, 重击伤害+1%/25pt |
| **体魄** | equip + artifact + pet + pill + wine + medal + collection | MaxHP+1%/5pt, 回血效率+1%/10pt, 血怒概率+1%/25pt |
| **悟性** | equip + artifactCh4 + pet + daoTree + wine + medal + class×level + collection | 技能伤害+1%/5pt, 连击概率+1%/50pt |
| **福缘** | equip + collection + artifactCh4 + fruit + pet + wine + medal | 击杀金币+1/pt, 灵韵+2%/5pt, 宠物同步率+2%/25pt |

### 5.6 装备→属性→技能 聚合流程

**InventorySystem.RecalcEquipStats()** 执行以下完整流程:

```
1. 遍历已装备装备 → 聚合主属性 + 副属性 + 洗练属性 + 灵性属性
2. 计算套装件数 → 应用套装加成 + 收集活跃套装AOE
3. 写入 player.equip* 字段（18+ 个属性字段）
4. 收集装备特效 → player.equipSpecialEffects（数组）
5. 法宝(treasure)槽技能 → SkillSystem.BindEquipSkill → 技能槽5
6. 专属(exclusive)槽技能 → SkillSystem.BindExclusiveSkill → 技能槽4
```

### 6.2 已注册特效

| 特效类型 | 触发时机 | 效果 | 伤害走CalcDamage |
|---------|---------|------|:---:|
| `bleed_dot` | onHit(35%概率) | ATK×8%/s DOT，3s，3s CD | ❌ 无视防御 |

| `death_immunity` | Player:TakeDamage 内 | 致命伤→HP=1+无敌 | — |
| `sacrifice_aura` | BossMechanics | 献祭光环 | — |

### 6.3 触发流程

```
CombatSystem.TriggerEquipEffectsOnHit(player, monster, isCrit)
  └─ 遍历 player.equipSpecialEffects
     └─ 查找 EFFECT_HANDLERS[eff.type]
        └─ 调用 handler.onHit(eff, player, monster, isCrit)

CombatSystem.TriggerEquipEffectsOnKill(player, monster)
  └─ 同上，调用 handler.onKill

CombatSystem.UpdateEquipEffects(dt)
  └─ 遍历所有 EFFECT_HANDLERS，调用 handler.update(dt)
```

### 6.4 扩展方式

```lua
-- 注册新特效（任何模块均可调用）

    onHit = function(eff, player, monster, isCrit)
        -- 攻击命中时触发
    end,
    onKill = function(eff, player, monster)
        -- 击杀时触发
    end,
    update = function(dt)
        -- 每帧更新
    end,
})
```

### 7.4 连击机制

**位置**: `SkillSystem.TryCombo()`

- 连击概率 = equipComboChance + classComboChance(天赋) + wisdomComboChance(悟性/50 × 1%)
- 触发后延迟施放（延迟队列），视觉上有明确反馈
- **连击计入 nth_cast_trigger 计数器**（如一剑开天的 N=8 计数）
- **连击不递归**（连击触发的技能不再判连击）

## 九、法宝技能系统设计

### 9.1 现有机制（当前已实现）

法宝(treasure)槽位目前的工作方式：

1. 玩家装备法宝 → `InventorySystem.RecalcEquipStats()` 触发
2. 读取 `treasureItem.skillId` → 调用 `SkillSystem.BindEquipSkill(skillId)` 绑定到**槽位5**
3. 卸下法宝 → 调用 `SkillSystem.UnbindEquipSkill(skillId)` 解绑
4. **同时只能装备一件法宝** → 槽位5同时只有一个技能

### 9.2 设计约束

基于用户需求明确的设计约束：

| 约束 | 说明 |
|------|------|
| 每件法宝携带 **1个主动技能** | 不是多技能，是精确的 1 个 |
| 同时只能装备 **1件法宝** | treasure 槽位只有一个 |
| 法宝技能在 **技能5号栏位** | 已实现，`SkillSystem.BindEquipSkill → slot 5` |
| 法宝主动技能是 **通用技能** | 走标准 TypeHandlers 管线 |
| 法宝技能可触发 **连击** | 需纳入 `SkillSystem.TryCombo()` 判定 |
| 法宝技能可触发 **其他属性机制** | 如一剑开天计数、龙象功叠层（如果技能类型支持） |

### 9.3 法宝技能与职业技能的关系

```
┌──────────────────────────────────────────────────┐
│                   技能槽布局                       │
│                                                  │
│  [1] 职业技能1  [2] 职业技能2  [3] 职业被动       │
│                                                  │
│  [4] 专属装备技能（未来）  [5] 法宝技能（当前）    │
│                                                  │
│  ※ 槽1~3: 按职业+等级自动解锁                      │
│  ※ 槽4~5: 由装备穿戴动态绑定/解绑                  │
└──────────────────────────────────────────────────┘
```

### 9.4 法宝技能作为"通用技能"的管线接入

法宝主动技能走与职业技能完全相同的管线：

```
SkillSystem.CastSkill(slotIndex=5)
  │
  ├─ 查找 skill = SkillData.Skills[equippedSkills[5]]
  ├─ 查找 handler = TypeHandlers[skill.type]
  ├─ 调用 handler.cast(skill, player) → 执行技能效果
  │
  ├─ EventBus.Emit("skill_cast", skillId, skill)
  │   ├─ → SwordShieldOnSkillCast (御剑诀恢复剑)
  │   ├─ → NthCastTrigger (一剑开天计数)
  │   └─ → DragonElephantAddStacks (如果是攻击事件)
  │
  ├─ TryCombo(skill, player) → 连击判定
  │   └─ 连击成功 → 重新施放 + 计入 nth 计数
  │
  └─ AutoCast 计数（排除被动类型）
```

**关键**: 只要法宝技能的 `skill.type` 在 TypeHandlers 中有对应的 `cast` 函数，即可完整接入所有机制（连击、一剑开天计数、御剑诀恢复等）。

## 十、基于职业设计的优化建议

### 10.1 法宝技能触发连击的条件分析

当前连击触发条件（`SkillSystem.TryCombo`）:

1. 技能必须有 `cast` handler（排除被动）
2. 概率 = equipComboChance + classComboChance + wisdomComboChance
3. 连击不递归

**法宝技能已满足条件**：法宝技能通过 `BindEquipSkill` 绑定到槽5，走 `CastSkill` 标准路径，会执行 `TryCombo`。只要法宝技能的 TypeHandler 有 `cast` 函数，连击自动生效。

**各职业的法宝技能连击收益**:

| 职业 | 基础连击率 | 100pt 悟性连击 | 法宝技能连击价值 |
|------|-----------|---------------|----------------|
| 太虚 | 8%(天赋) | +2%(悟性100) = 10% | 高 — 连击计入一剑开天计数 |
| 罗汉 | 0% | +2% = 2% | 低 — 无连击天赋，概率极低 |
| 天衍 | 0% | +2% = 2% | 低 |
| 镇岳 | 0% | +2% = 2% | 低 |

### 10.2 法宝技能与 nth_cast_trigger 的交互

当前 `TryNthCastTrigger` 的计数规则:
- 所有有 `cast` handler 的技能释放 +1 计数
- 连击触发也 +1 计数
- 被动/事件驱动类型不计数

**法宝技能自动参与计数**。对太虚影响最大:
- 太虚一剑开天 N=8: 每释放8次技能（含连击）触发
- 加入法宝技能后: 破剑式(5s) + 奔雷式(5s) + **法宝技能(CDx)** = 更快触发巨剑
- 如果法宝技能 CD ≤ 8s，60s 内额外增加 ~7 次释放 → 一剑开天触发次数从 3.3 提升到 ~4.5

### 10.3 法宝技能与龙象功的交互

龙象功叠层来源: `player_normal_hit`(+1) 和 `player_heavy_hit`(+2)

**法宝技能不直接叠龙象功层数**（除非法宝技能本身触发普攻事件）。这是合理的 — 龙象功是普攻蓄力被动，不应被技能加速。

如果需要法宝技能也能叠层，可以在 SkillData 中加入 `addsChargeStacks = true` 字段，在 CastDamageSkill 后触发 `DragonElephantAddStacks`。**但不建议这样做**，会破坏罗汉的蓄力节奏设计。

### 10.4 法宝技能与血怒的交互

血怒叠层来源: 普攻概率 + 震山掌必定叠层

**建议**: 法宝技能命中时，按血怒概率（不是必定）叠层。这对镇岳有额外价值（技能频率越高，血怒叠层越快），但不破坏震山掌"确定性叠层"的独特性。

实现方式: 在 `CastDamageSkill` 完成后，检查并触发血怒叠层判定:
```lua
-- 在 CastDamageSkill 末尾
if player.classId == "zhenyue" or (player blood rage chance > 0) then
    -- 走概率判定叠层，非确定叠层
end
```

### 10.5 优化建议总表

| # | 建议 | 优先级 | 影响范围 | 说明 |
|---|------|--------|---------|------|
| 1 | **法宝技能自动参与连击判定** | ✅ 已支持 | 无需改动 | 走 CastSkill 标准路径即可 |
| 2 | **法宝技能自动参与 nth_cast 计数** | ✅ 已支持 | 无需改动 | 有 cast handler 即自动计数 |
| 3 | **法宝技能自动参与御剑诀恢复** | ✅ 已支持 | 无需改动 | skill_cast 事件自动触发 |
| 4 | **法宝技能命中触发血怒概率叠层** | 🟡 建议新增 | SkillSystem | 对镇岳有正向价值 |
| 5 | **`_preDamageHook` 改为列表** | 🔴 需要改 | Player.lua | 多被动共存的前提条件 |

| 7 | **法宝技能不叠龙象功** | ✅ 维持现状 | — | 保护罗汉普攻蓄力的独特性 |

### 10.6 法宝技能设计建议（基于职业平衡）

法宝技能作为通用技能，应避免过度偏向某个职业。建议:

**设计原则**:
1. 法宝技能使用标准伤害类型（`melee_aoe`、`aoe_dot` 等），走 CalcDamage 管线
2. 法宝技能 CD 建议 8~15s（避免太短导致太虚一剑开天过快触发）
3. 法宝技能倍率建议 100%~200% ATK（与职业技能1持平或略低）
4. 法宝技能吃技能伤害% → 悟性越高收益越大 → 太虚自然偏好高倍率法宝
5. 法宝技能走概率血怒叠层 → 镇岳有额外收益但不破坏震山掌独特性

**法宝技能类型推荐**:

| 类型 | 适合法宝 | 职业偏好 |
|------|---------|---------|
| `melee_aoe` | 近战法宝（剑/拳） | 通用，太虚连击收益最高 |
| `aoe_dot` | 法阵法宝（符/阵） | 太虚（真伤DOT叠加）|
| `buff` | 辅助法宝（鼎/炉） | 天衍（BUFF流） |
| `lifesteal` | 吸血法宝（血珠） | 罗汉（续航流） |
| `ground_zone` | 领域法宝（旗/幡） | 通用 |

## 十二、附录：乘区划分

### 12.1 普攻伤害乘区

```
最终伤害 = CalcDamage(ATK, DEF)    ← 加法区1: ATK聚合
         × CritMultiplier           ← 乘区1: 暴击
         × (1 + 噬魂buff%)          ← 乘区2: 一次性buff
         × (1 + vulnPercent)        ← 乘区3: 易伤debuff (Monster内)
         × (1 + washLevel × 0.01)   ← 乘区4: 洗髓增伤 (Monster内)
```

### 12.2 重击伤害乘区（真实伤害）

```
最终伤害 = (ATK + HeavyHit)                ← 加法区1: 攻击力 + 重击值（不走 CalcDamage，无视防御）
         × (1 + heavyDmgBonus)              ← 乘区1: 根骨重击加成
         × CritMultiplier                    ← 乘区2: 暴击
         × (1 + vulnPercent)                 ← 乘区3: 易伤debuff (Monster内)
         × (1 + washLevel × 0.01)            ← 乘区4: 洗髓增伤 (Monster内)
```

### 12.3 技能伤害乘区

```
最终伤害 = CalcDamage(ATK × skillMul × (1+skillDmg%), DEF)  ← 加法区1: ATK × 技能倍率 × 技能伤害%
         × CritMultiplier                                     ← 乘区1: 暴击
         × (1 + vulnPercent)                                  ← 乘区2: 易伤
         × (1 + washLevel × 0.01)                             ← 乘区3: 洗髓增伤
```

### 12.4 受伤减伤乘区

```
最终受伤 = 原始伤害
         × (1 - min(equipDmgReduce, 0.75))    ← 乘区1: 装备减伤
         × (1 - washLevel × 0.01)              ← 乘区2: 洗髓减伤
         × (1 - preDamageHook减伤%)             ← 乘区3: 被动减伤（御剑诀等）
         - shieldAbsorb                         ← 减法区: 护盾吸收
```

## 十三、属性与机制实现层通用性评估

> 评估目标：审查属性系统和机制系统的**实现层**通用性，识别硬编码绑定到单个技能/效果的代码，
> 为法宝系统（多种法宝携带不同技能）的接入提供改造依据。
>
> 评估标准：
> - ✅ 通用 — 基于类型/数据驱动，新技能/效果零代码修改即可接入
> - ⚠️ 部分通用 — 框架通用但存在单例限制或特定分支
> - ❌ 硬编码 — 绑死单个技能/效果 ID，新增同类功能需修改源码

### 13.1 属性系统通用性评估

#### 13.1.1 属性聚合模式

**当前实现**: `Player:_RecalcStatsCache()` 使用**平铺字段累加**模式。

```
totalAtk = base + equipAtk + collectionAtk + titleAtk + seaPillarAtk + medalAtkFlat + artifactTiandiAtk
```

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 聚合正确性 | ✅ | 各来源独立累加，互不干扰 |
| 帧级去重 | ✅ | `_statsCacheFrame` 防止同帧重复计算 |
| 新增来源扩展性 | ⚠️ | 每新增一个属性来源（如法宝属性加成），需手动修改 `_RecalcStatsCache()` 添加字段 |
| 属性来源可追溯性 | ❌ | 无"属性来源"抽象，无法在运行时查询"我的ATK有多少来自法宝" |

**具体问题**:
- 没有 `AttributeSource` 注册表模式。所有来源都是 Player 上的平铺字段（`equipAtk`, `collectionAtk`, `titleAtk` 等共 18+ 个字段）
- 新增"法宝属性加成"需要：(1) 在 Player 上加 `treasureAtk`, `treasureDef` 等字段，(2) 在 `_RecalcStatsCache()` 里每个属性的聚合行都加上对应字段
- 这不是阻断性问题（加几行代码即可），但每新增一个系统都要改 Player 核心代码

**改造建议（可选）**:
```lua
-- 方案：属性来源注册表（仅当来源系统持续增长时才值得）
Player._attrSources = {}  -- { [sourceName] = { atk=N, def=N, ... } }
function Player:SetAttrSource(name, attrs) ... end
function Player:_RecalcStatsCache()
    local sum = { atk=self.atk, def=self.def, ... }
    for _, src in pairs(self._attrSources) do
        sum.atk = sum.atk + (src.atk or 0)
        -- ...
    end
end
```
> **当前评估**: 属性来源数量有限（~7个），平铺字段模式可接受。如果后续新增来源频率高（法宝属性、坐骑属性、阵法属性...），再考虑重构。

|---------|---------|------|------|
| 根骨→DEF% | `GetConstitutionDefBonus()` | ✅ | 纯函数，参数化 |
| 体魄→MaxHP% | `GetPhysiqueHpBonus()` | ✅ | 纯函数，参数化 |

| 触发效果 | ⚠️ | `TriggerPassiveGiantSword()` 函数名绑死"巨剑"，但逻辑本身读 `skill.trigger*` 数据字段，是通用的 |
| 法宝技能兼容 | ✅ | 法宝技能的 `skill_cast` 事件自动被监听 |

**问题**: 函数命名 `TriggerPassiveGiantSword` 暗示绑死一剑开天，但实际逻辑是通用的——读取 `skill.triggerDamageMultiplier`、`skill.triggerIgnoreDef` 等数据字段。如果未来有第二个 `nth_cast_trigger` 类型技能，逻辑可复用，但函数名会造成困惑。

**改造建议**: 将 `TriggerPassiveGiantSword()` 重命名为 `TriggerNthCastEffect()` 或类似通用名称。

#### 13.2.4 charge_passive (龙象功)

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 叠层机制 | ✅ | `ChargePassiveAddStacks()` 基于 `skill.maxStacks` 数据驱动 |
| 多技能支持 | ✅ | `FindUnlockedByType("charge_passive")` 返回列表，遍历处理 |
| 释放效果 | ⚠️ | `TriggerDragonElephantQuake()` 函数名绑死"龙象功"，但逻辑读 `skill.chargeDamageMultiplier` 等数据字段 |
| 事件监听 | ✅ | 通过 `player_normal_hit`/`player_heavy_hit` 事件驱动 |

**问题**: 与 nth_cast_trigger 相同 — 函数名特化，逻辑通用。

**改造建议**: 重命名 `TriggerDragonElephantQuake()` → `TriggerChargePassiveRelease()`。

#### 13.2.5 sword_shield_passive (御剑诀)

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 技能释放恢复 | ✅ | `SwordShieldOnSkillCast()` 监听 `skill_cast` 事件，通用 |
| 过滤逻辑 | ⚠️ | 过滤条件是 `castSkill.classId`，只要释放的技能属于本职业就恢复剑。这是 class-aware 的，不是 skill-ID-hardcoded 的 |
| 减伤注入 | ❌ | 通过 `player._preDamageHook` 注入，**单槽限制**（见 13.2.8） |

#### 13.2.6 重击机制 (TryHeavyStrike)

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 概率来源 | ✅ | 职业天赋 + 根骨衍生 + 装备 `heavy_strike` 特效 |
| 计数器 | ❌ | `heavyStrikeCounter` 是**模块级单例变量** |
| 多来源共存 | ❌ | 只有一个计数器，无法区分"装备A每5次必定重击"和"装备B每8次必定重击" |
| 伤害计算 | ✅ | 走标准 CalcDamage + ApplyCrit |

**问题详解**:
```lua
-- CombatSystem.lua 模块级变量
local heavyStrikeCounter = 0  -- 全局唯一计数器

-- TryHeavyStrike 内部
for _, eff in ipairs(player.equipSpecialEffects) do
    if eff.type == "heavy_strike" then
        heavyStrikeCounter = heavyStrikeCounter + 1
        if heavyStrikeCounter >= eff.everyN then  -- ← 只用最后遍历到的 eff
            heavyStrikeCounter = 0
            forceHeavy = true
        end
    end
end
```
如果有两件装备都有 `heavy_strike` 特效（`everyN=5` 和 `everyN=8`），共用同一个计数器，行为不可预测。

**改造建议**: 将计数器绑定到效果实例而非模块级：
```lua
-- 在效果实例上维护状态
eff._counter = (eff._counter or 0) + 1
if eff._counter >= eff.everyN then
    eff._counter = 0
    forceHeavy = true
end
```

#### 13.2.7 血怒机制 (BloodRage / BossMechanics)

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 叠层触发 | ✅ | 通过 `EventBus.On("player_deal_damage")` 事件驱动 |
| 数据驱动 | ✅ | 体魄衍生概率 `bloodRageChance = physique/25 × 1%` |
| 引爆效果 | ✅ | 真实伤害 = maxHP × percent |
| 技能触发 | ⚠️ | 震山掌通过 `skill_cast` 事件必定叠层，但其他技能不叠层——法宝技能需要额外接入 |

**法宝技能接入**: 需在 `skill_cast` 的血怒叠层逻辑中，允许法宝技能以**概率**（而非确定）叠层。

#### 13.2.8 _preDamageHook（被动减伤注入点）

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 机制设计 | ⚠️ | 允许被动技能注入受伤前逻辑，思路正确 |
| 实现 | ❌ **单槽限制** | `player._preDamageHook = function(...)` 是**单一函数变量**，不是列表 |
| 多被动共存 | ❌ | 后注册的 hook 覆盖前一个。御剑诀 + 法宝被动减伤无法共存 |

**问题详解**:
```lua
-- Player.lua:797 TakeDamage 中
if self._preDamageHook then
    damage = self._preDamageHook(self, damage, source) or damage
end
```

```lua
-- SkillSystem.lua 中御剑诀注册
player._preDamageHook = function(self, damage, source)
    -- 御剑诀减伤逻辑
end
```

如果法宝被动也需要注入 `_preDamageHook`，会覆盖御剑诀的 hook。

**改造建议**:
```lua
-- 改为 hook 列表
player._preDamageHooks = {}

function Player:AddPreDamageHook(id, fn)
    self._preDamageHooks[id] = fn
end

function Player:RemovePreDamageHook(id)
    self._preDamageHooks[id] = nil
end

-- TakeDamage 中依次执行
for id, hook in pairs(self._preDamageHooks) do
    damage = hook(self, damage, source) or damage
end
```

#### 13.2.9 EFFECT_HANDLERS 装备特效注册表

| 评估项 | 评级 | 说明 |
|--------|------|------|
| 注册模式 | ✅ | `RegisterEffectHandler(type, {onHit?, onKill?, update?})` |
| 分发机制 | ✅ | 遍历 `equipSpecialEffects` → 查表分发 |
| 新特效扩展 | ✅ | 注册即生效 |
| 实例状态 | ❌ | 部分特效使用**模块级单例状态** |

**单例状态问题**:
```lua
-- CombatSystem.lua 模块顶部
CombatSystem.lifestealBurstTimer = nil    -- 噬魂增伤计时器（单例）
CombatSystem.lifestealBurstDmgMult = 0    -- 噬魂增伤倍率（单例）
CombatSystem.lifestealBurstCooldown = 0   -- 噬魂增伤CD（单例）
local shadowStrikeCooldown = 0            -- 暗影突袭CD（单例）
local heavyStrikeCounter = 0              -- 重击计数器（单例）
```

这意味着每种特效类型**全局只能有一个实例**。如果两件装备都有 `shadow_strike` 效果，共用同一个 CD。

**改造建议**: 将状态绑定到效果实例 `eff` 上：
```lua
-- shadow_strike handler
onHit = function(eff, player, monster, isCrit)
    eff._cooldown = eff._cooldown or 0
    if eff._cooldown > 0 then return end
    -- 触发效果
    eff._cooldown = 2.0
end,
update = function(eff, dt)  -- 需要改签名，传入 eff
    if eff._cooldown then eff._cooldown = math.max(0, eff._cooldown - dt) end
end,
```

### 13.3 硬编码技能 ID 清单

以下位置存在硬编码到特定技能 ID 的代码，阻碍通用化：

| # | 文件 | 位置 | 硬编码内容 | 严重程度 | 改造方案 |
|---|------|------|-----------|---------|---------|
| 1 | SkillSystem.lua | `Update()` CD 结束分支 | `skill.id == "jade_gourd_heal"` → 调用 `WineSystem.OnGourdSkillReady()` | ❌ 高 | 改为技能数据字段 `skill.onCDReady = "WineSystem.OnGourdSkillReady"` |
| 2 | SkillSystem.lua | `CastBuffSkill()` duration 获取 | `skill.id == "jade_gourd_heal"` → `WineSystem.GetDrinkDuration()` | ❌ 高 | 改为技能数据字段 `skill.getDuration` 回调或 `skill.durationSource` 配置 |
| 3 | SkillSystem.lua | `CastBuffSkill()` 施放回调 | `skill.id == "jade_gourd_heal"` → WineSystem 回调 | ❌ 高 | 改为技能数据字段 `skill.onCast` / `skill.onEnd` 回调 |
| 4 | SkillData.lua | `GetDynamicDescription()` | `skillId == "jade_gourd_heal"` | ⚠️ 中 | 改为通过 `skill.descriptionProvider` 函数字段 |
| 5 | SkillData.lua | `GetDynamicDescription()` | `skillId == "blood_sea_aoe"` | ⚠️ 中 | 同上 |
| 6 | SkillData.lua | `GetDynamicDescription()` | `skillId == "haoqi_seal_zone"` | ⚠️ 中 | 同上 |

**影响**: 如果新增一个法宝技能也是 `buff` 类型且需要外部系统联动（如"灵丹"法宝技能读取炼丹系统的 duration），当前必须在 `CastBuffSkill()` 中加 `if skill.id == "xxx"` 分支。这违反开闭原则。

**统一改造方案**:
```lua
-- SkillData 中配置回调
jade_gourd_heal = {
    type = "buff",
    -- 通用回调字段（替代硬编码 if-else）
    getDuration = function(skill, player)
        local ok, WineSystem = pcall(require, "systems.WineSystem")
        return ok and WineSystem.GetDrinkDuration() or skill.baseDuration
    end,
    onCast = function(skill, player)
        local ok, WineSystem = pcall(require, "systems.WineSystem")
        if ok then WineSystem.OnDrinkStart(player) end
    end,
    onEnd = function(skill, player)
        local ok, WineSystem = pcall(require, "systems.WineSystem")
        if ok then WineSystem.OnDrinkEnd(player) end
    end,
    onCDReady = function(skill, player)
        local ok, WineSystem = pcall(require, "systems.WineSystem")
        if ok then WineSystem.OnGourdSkillReady() end
    end,
}

-- CastBuffSkill() 改为读回调
local duration = skill.getDuration and skill.getDuration(skill, player) or skill.baseDuration
if skill.onCast then skill.onCast(skill, player) end
```

### 13.4 噬魂增伤内联问题

**位置**: `CombatSystem.AutoAttack()` 内部

```lua
-- 直接在普攻管线中内联处理，绕过 EFFECT_HANDLERS 体系
if CombatSystem.lifestealBurstTimer and CombatSystem.lifestealBurstTimer > 0 then
    local bonus = CombatSystem.lifestealBurstDmgMult or 0
    local bonusDmg = math.floor(damage * bonus)
    damage = damage + bonusDmg
    CombatSystem.lifestealBurstTimer = 0  -- 一次性消耗
end
```

**问题**: 
- 此逻辑应该在 `EFFECT_HANDLERS.lifesteal_burst.onHit` 中处理，而不是散落在 `AutoAttack()` 里
- 如果法宝特效也需要"下次攻击增伤"机制，要在 `AutoAttack()` 里再加一段类似代码
- 违反了 EFFECT_HANDLERS 的设计意图

**改造建议**: 将一次性 buff 消耗抽象为通用模式：
```lua
-- 通用"下次攻击增伤"列表
CombatSystem._nextHitBonuses = {}

function CombatSystem.AddNextHitBonus(source, multiplier)
    table.insert(CombatSystem._nextHitBonuses, { source=source, mult=multiplier })
end

-- AutoAttack 中统一消耗
for _, bonus in ipairs(CombatSystem._nextHitBonuses) do
    damage = damage + math.floor(damage * bonus.mult)
end
CombatSystem._nextHitBonuses = {}
```

### 13.5 TakeDamage 内联效果检查

**位置**: `Player:TakeDamage()` 内部

```lua
-- evade_damage — 内联在 TakeDamage 中
for _, eff in ipairs(self.equipSpecialEffects or {}) do
    if eff.type == "evade_damage" and math.random() < (eff.evadeChance or 0) then
        -- 闪避成功
        return
    end
end

-- death_immunity — 内联在 TakeDamage 中
for _, eff in ipairs(self.equipSpecialEffects or {}) do
    if eff.type == "death_immunity" and math.random() < (eff.immuneChance or 0) then
        self.hp = 1
        -- 无敌 + 特效
        return
    end
end
```

**问题**: 这两个效果虽然注册在 `EFFECT_HANDLERS` 中，但实际触发逻辑在 `TakeDamage` 里内联硬编码。如果法宝提供类似"受伤时X%概率格挡"效果，需要在 `TakeDamage` 中再加 if-else。

**改造建议**: 在 EFFECT_HANDLERS 中增加 `onTakeDamage` 钩子：
```lua
EFFECT_HANDLERS["evade_damage"] = {
    onTakeDamage = function(eff, player, damage, source)
        if math.random() < (eff.evadeChance or 0) then
            return 0, true  -- damage=0, absorbed=true
        end
        return damage, false
    end,
}

-- TakeDamage 中统一分发
for _, eff in ipairs(self.equipSpecialEffects or {}) do
    local handler = CombatSystem.GetEffectHandler(eff.type)
    if handler and handler.onTakeDamage then
        damage, absorbed = handler.onTakeDamage(eff, self, damage, source)
        if absorbed then return end
    end
end
```

**为法宝系统接入，建议按以下顺序改造**：

| 优先级 | 改造项 | 工作量 | 收益 | 状态 |
|:------:|--------|:------:|------|:----:|
| **P0** | `_preDamageHook` 改为列表 | 小 | 解除多被动共存阻断，法宝被动减伤才能与御剑诀共存 | ✅ 已完成 |
| **P0** | jade_gourd_heal 3处硬编码改为数据回调 | 中 | 解除 buff 类型技能的通用化阻断，新法宝 buff 技能不再需要改 SkillSystem | ✅ 已完成 |

| **P2** | evade/death_immunity 改为 `onTakeDamage` 钩子 | 小 | TakeDamage 管线可扩展 | 延后 |

### 13.8 结论

**整体评估**: 系统的**架构层通用性良好**（TypeHandlers + EFFECT_HANDLERS 双注册表模式设计正确），但**实现层存在局部退化**——部分机制在实现时退化为硬编码或单例模式，与架构层的通用设计意图不一致。

**法宝系统接入的关键阻断点**只有两个：
1. `_preDamageHook` 单槽 → 法宝被动无法与职业被动共存
2. `jade_gourd_heal` 硬编码 → 新 buff 类法宝技能无法走通用管线

解决这两个 P0 问题后，法宝系统的主动技能可以**零代码修改**接入连击、nth_cast 计数、御剑诀恢复等全部通用机制。P1/P2 的改造是渐进式优化，不阻断法宝系统上线。

## 十四、改造风险评估

> 评估维度：
> - **系统破坏** — 改动是否可能导致战斗逻辑崩溃、技能失效、伤害异常
> - **存档兼容** — 改动是否影响已有存档的加载/迁移
> - **线上影响** — 改动是否影响多人副本的服务端校验、伤害同步、结算正确性

### 14.1 存档系统与改造交叉分析

**关键发现**: 经审查 `SaveSerializer.lua`，**所有改造项均不涉及存档数据结构**。

存档持久化的战斗相关数据：
```
skills → SkillSystem.GetSkillBarSlots() → 仅保存 { [slotIndex] = skillId }
equipment → 装备物品数据（含 specialEffect 字段）
player → 属性字段（atk, def, maxHp, 仙缘属性等）
```

以下**运行时状态不会被持久化**（每次加载后重建）：
- `_preDamageHook` / `_preDamageHooks` → 由 SkillSystem 在解锁技能时重新注入
- `equipSpecialEffects` 数组 → 由 `InventorySystem.RecalcEquipStats()` 在装备加载后重建
- `heavyStrikeCounter` / `lifestealBurstTimer` / `shadowStrikeCooldown` → 模块级运行时变量，不持久化
- `_swordShield` 状态 → 由 SkillSystem 技能解锁时重建

**结论: 所有改造项对存档的风险为零**。无需版本迁移（无需 v18）。

### 14.2 线上多人副本交叉分析

**关键发现**: 副本伤害同步采用**混合架构** — 客户端本地计算伤害 → 上报数值 → 服务端校验上限后累加。

```
客户端: CombatSystem.AutoAttack() → CalcDamage → 本地伤害值
         ↓
网络层: DungeonClient.AttackBoss(damage) → C2S_DungeonAttack
         ↓
服务端: DungeonHandler.HandlePlayerAttack()
         → 距离校验、CD校验、数值上限校验(MAX_DAMAGE_PER_REPORT=50000)
         → 内存扣血 → 累计伤害 → 广播
```

**改造项与线上的关系**：

| 改造项 | 影响客户端计算 | 影响服务端校验 | 影响伤害同步 |
|--------|:------------:|:------------:|:----------:|
| `_preDamageHook` → 列表 | ❌ 这是受伤管线 | ❌ | ❌ |
| jade_gourd_heal 回调化 | ✅ 技能施放逻辑 | ❌ | ❌ |
| 重击计数器实例化 | ✅ 可能改变重击触发频率 | ❌ 服务端只校验上限 | ⚠️ 间接 |
| 效果状态实例化 | ✅ 可能改变特效触发 | ❌ | ⚠️ 间接 |
| 噬魂增伤通用化 | ✅ 伤害计算 | ❌ | ⚠️ 间接 |
| evade/death_immunity 钩子化 | ❌ 这是受伤管线 | ❌ | ❌ |

> **"间接影响"说明**: 这些改造不改变协议或服务端逻辑，但如果改造引入 bug 导致客户端伤害计算异常（如重击每次都触发），异常伤害值会被上报到服务端。服务端有 `MAX_DAMAGE_PER_REPORT=50000` 上限保护，不会造成数据灾难，但会影响副本 DPS 排名。

### 14.3 逐项风险评估

#### P0-1: `_preDamageHook` 改为 `_preDamageHooks` 列表

| 维度 | 风险等级 | 分析 |
|------|:--------:|------|
| 系统破坏 | 🟡 中 | 改动 Player.TakeDamage 核心受伤管线。如果列表遍历顺序错误或 hook 返回值处理不当，可能导致：(1) 减伤完全失效（玩家秒死），(2) 减伤叠加过度（玩家无敌） |
| 存档兼容 | 🟢 无 | `_preDamageHook` 是运行时状态，不持久化 |
| 线上影响 | 🟢 无 | 仅影响客户端受伤管线，不涉及伤害上报或服务端校验 |

    local dmg3 = GameConfig.CalcDamage(0, 100)
    assert_true("CalcDamage(0,100)_min1", dmg3 >= 1)

    -- ② 属性聚合验证
    local totalAtk = player:GetTotalAtk()
    assert_true("totalAtk_positive", totalAtk > 0)
    assert_true("totalAtk_reasonable", totalAtk < 100000)

    local totalDef = player:GetTotalDef()
    assert_true("totalDef_non_negative", totalDef >= 0)

    local totalMaxHp = player:GetTotalMaxHp()
    assert_true("totalMaxHp_positive", totalMaxHp > 0)

    -- ③ 暴击判定验证（统计 100 次）
    local critCount = 0
    for i = 1, 100 do
        local _, isCrit = player:ApplyCrit(100)
        if isCrit then critCount = critCount + 1 end
    end
    local critRate = player:GetTotalCritRate()
    -- 允许 ±15% 偏差（统计波动）
    assert_range("crit_rate_statistical",
        critCount / 100,
        math.max(0, critRate - 0.15),
        math.min(1, critRate + 0.15))

    -- ④ 连击概率验证
    local comboChance = player:GetSkillComboChance()
    assert_true("comboChance_non_negative", comboChance >= 0)
    assert_true("comboChance_not_over_1", comboChance <= 1)

    -- ⑤ 重击概率验证
    local heavyChance = player:GetClassHeavyHitChance()
    assert_true("heavyHitChance_non_negative", heavyChance >= 0)

    -- ⑥ 减伤上限验证
    assert_true("dmgReduce_cap_075",
        (player.equipDmgReduce or 0) <= 0.75)

    -- ⑦ 受伤管线（无敌状态）
    local origHp = player.hp
    player.invincibleTimer = 1.0
    player:TakeDamage(9999, nil)
    assert_true("invincible_no_damage", player.hp == origHp)
    player.invincibleTimer = 0

    print("========= Combat Regression 测试结果 =========")
    for _, r in ipairs(results) do print(r) end
    print(string.format("总计: %d 通过, %d 失败", pass, fail))
    print("===============================================")

    return fail == 0
end

return TestCombatRegression

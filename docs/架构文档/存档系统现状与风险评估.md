# 存档系统现状与风险评估

> 生成日期：2026-07-01  
> 基准范围：`scripts/systems/save/`、`scripts/systems/SaveSystem.lua`、`scripts/network/Save*.lua`、`CloudStorage.lua`、黑市/丹药/属性/仓库/命格等存档相关代码。  
> 当前原则：可以接受保存失败，但不能接受旧档被错误覆盖、残缺档被写回、或恢复逻辑造成事实性丢档。

## 1. 当前结论

当前项目采用“单槽位单大对象”的主存档结构，核心数据集中保存到 `save_{slot}`。该结构本身未达到 serverCloud 理论 1MB 容量上限，但已经触发客户端与服务端之间 RemoteEvent/WebSocket 单消息大小限制。

当前已观测错误：

```text
WebSocketTransport::Send - message size 66004 exceeds maximum 65535
```

对应保存数据日志：

```text
SaveSystem saveData (save_3): 63.6KB (65173 bytes)
```

也就是说，`saveData` JSON 本体约 63.6KB，但加上 VariantMap、字段名、RemoteEvent 包装后，实际发送消息达到 66004 字节，超过 65535 字节限制。当前问题不是云存储 1MB key 容量，而是网络传输层单消息上限。

因此，当前存档系统的第一优先级不是压缩字段，而是把保存/加载改为可分片、可校验、可失败但不覆盖旧档的传输流程。

## 2. 主存档链路

### 2.1 保存链路

```text
SaveSystem.Update / RequestImmediateSave
  -> SavePersistence.DoSave
  -> CloudStorage.BatchSet():Save()
  -> C2S_SaveGame
  -> SaveWriteService.Execute
  -> serverCloud:BatchSet():Save()
  -> S2C_SaveResult
```

关键文件：

| 文件 | 职责 |
| --- | --- |
| `scripts/systems/SaveSystem.lua` | 存档系统入口，驱动自动保存、脏标记、网络恢复、超时重试 |
| `scripts/systems/save/SavePersistence.lua` | 组装完整 `save_{slot}` 数据，发起保存 |
| `scripts/network/CloudStorage.lua` | 客户端云存储适配层，将 BatchSet 转成网络请求 |
| `scripts/network/SaveWriteService.lua` | 服务端保存入口，校验并写入 serverCloud |
| `scripts/network/SaveProtocol.lua` | 保存/加载 RemoteEvent 名称定义 |

### 2.2 加载链路

```text
SaveSystemNet.Load
  -> C2S_LoadGame
  -> SaveLoadService.Execute
  -> serverCloud:BatchGet()
  -> S2C_LoadResult(saveData=json)
  -> SaveSystemNet.HandleLoadResult
  -> SaveLoader.ProcessLoadedData
```

当前加载路径同样一次性返回完整 JSON。如果存档继续增长，保存之外，加载也会触发相同的 64KB 级别单消息风险。

## 4. 当前体积分布

最近一次问题存档的字段体积大致如下：

| 字段 | 体积 |
| --- | ---: |
| `mingge` | 24.5KB |
| `backpack` | 19.0KB |
| `equipment` | 5.5KB |
| `warehouse` | 3.4KB |
| `challenges` | 2.2KB |
| `titles` | 1.5KB |
| `quests` | 1.3KB |
| `player` | 1.3KB |
| `pendingXianyuanRewards` | 1.0KB |
| `atlas` | 0.8KB |

当前最大压力来自 `mingge` 和 `backpack`。命格保存了大量可由配置反推的展示字段；背包/装备/仓库保存的是完整物品快照，字段丰富但冗余较多。

不过，单纯压缩这两个字段只能缓解当前大小，不能解决未来再次超过 RemoteEvent 单消息限制的问题。

## 5. 玩家属性与丹药存档现状

### 5.1 玩家属性

`player` 字段保存核心属性与成长来源，包括但不限于：

- 等级、经验、当前血量、最大血量、攻击、防御、回血。
- 金币、灵韵、境界、章节、坐标。
- 职业、道树、葫芦美酒、血气积累等长期成长字段。
- 海柱、剑池、镇狱塔等系统带来的永久属性加成。

### 5.2 丹药字段

丹药相关数据分布在多个系统中：

| 来源 | 内容 |
| --- | --- |
| `player.pillCounts` | 聚合后的丹药服用数量 |
| `player.pillKillHeal` | 击杀回血类丹药累计效果 |
| `player.pillConstitution` | 体质类丹药累计 |
| `player.pillFortune` / `premiumZongEaten` | 福缘/粽子类成长 |
| `AlchemySystem` | 虎力丹、灵蛇丹、金刚丹、淬体丹、龙血丹、剑意丹、深渊封印丹、暗影神丹等计数 |
| `ChallengeSystem` | 血煞丹、浩气丹等挑战丹药 |
| `shop` 旧字段 | 兼容旧版本商店丹药状态 |

加载时会进行丹药修正：

- 根据挑战完成记录推导血煞丹、浩气丹数量。
- 修正击杀回血和暗影神丹回血字段。
- 对 `pillCounts` 做补全和对齐。
- 服务端保存时会对部分丹药数量和属性上限做校验。

风险点：丹药不是单一真源，属于“多字段互相修正”的结构。后续压缩或迁移时，不能直接删字段，必须先定义事实来源和派生字段。

## 6. 背包、装备、仓库与黑市存档现状

### 6.1 物品保存

背包、装备、仓库使用 `SaveSerializer.SerializeItemFull` 保存完整物品快照。典型字段包括：

- `id`、`name`、`type`、`slot`、`quality`、`tier`、`level`
- 主属性、附加属性、套装、强化/附灵/圣装属性
- 图标、售价、货币、装备 ID、消耗品 ID、数量
- 技能书、宠物经验物品等特殊字段
- 黑市字段：`bmNoResell`、`bmSource`、`bmBuyAt`、`bmBuyItemId`

当前序列化明确不会保存旧的临时黑市锁字段：

- `bmLockUntil`
- `bmLockSource`
- `bmLockBatchId`

### 6.2 黑市安全边界

黑市相关文件包括：

- `scripts/systems/BlackMarketTradeLock.lua`
- `scripts/systems/BlackMarketSyncState.lua`
- `scripts/network/BlackMerchantHandler.lua`
- `scripts/network/BlackMarketSellGuard.lua`

当前黑市的核心持久化边界是 `bmNoResell`。该字段必须在任何背包/仓库压缩方案中保留，否则会破坏防倒卖规则。

黑市还有服务端 WAL 和直接写回 `save_{slot}` 的逻辑。也就是说，黑市购买不是单纯等客户端下一次自动保存，它可能由服务端直接修改云端存档。

风险点：

- 黑市直写存档也需要服务端备份保护。
- 物品压缩时必须保留黑市不可转卖标记。
- 当前出售侧有高风险阻断，但购买写回仍属于需要重点保护的存档写路径。

## 7. 命格存档现状

`mingge` 是当前最大体积字段，包含装备命格和背包命格。

每个命格项保存内容较多：

- `id`
- `minggeId`
- `bossId`
- `bossName`
- `name`
- `element`
- `type`
- `tier`
- `quality`
- `stat`
- `value`
- `roll`
- `setId`
- `sellCurrency`
- `sellPrice`
- `locked`
- `category`

其中不少字段可通过 `minggeId`、`bossId`、`quality`、`tier`、`setId` 反推。命格是后续体积优化的第一候选，但应该在传输分片完成之后再做，避免压缩过程中引入数据兼容风险。

## 8. 任务、封印与章节进度现状

任务和封印状态主要通过 `QuestSystem.Serialize()` 保存。

当前 `sealStates` 有特殊保护逻辑：

- `futureLocked` 封印不会被普通任务同步误解锁。
- `manualOnly` 封印不会被自动任务完成误解锁。
- 加载时会重置不应被存档永久覆盖的封印临时状态。

这对第六章虎王试炼这类未来副本封印很重要。正确做法是沿用现有封印体系，不额外发明新封印状态。

## 9. 账号级数据现状

当前存在角色存档与账号级 key 混合结构。

| 账号级数据 | 主存档字段 | 独立 key |
| --- | --- | --- |
| 图鉴 | `atlas` | `account_atlas` |
| 外观 | `accountCosmetics` | `account_cosmetics` |
| 账号仙体 | `accountImmortalBodies` | `account_immortal_bodies` |
| 公告 | `bulletin` | `account_bulletin` |

服务端保存时会把部分账号级数据同步写入独立 key。加载和拉取角色列表时也会注入部分账号级数据。

风险点：`accountImmortalBodies` 在拉取角色列表时有读取和注入路径，但 `SaveLoadService` 加载单个槽位时的注入路径不如 `atlas`、`accountCosmetics` 完整，需要后续统一。

## 10. 当前已有防护

当前系统已有以下防护能力：

| 防护 | 当前表现 |
| --- | --- |
| 脏标记 | 保存请求只标记 dirty，成功后才清理 |
| 保存中保护 | 保存中再次变更会设置 `_dirtySinceFlush` |
| 断线保护 | 网络断开时暂停自动保存 |
| 超时重试 | 保存超过 15 秒无响应会失败重试 |
| 等级回退保护 | 本地等级明显低于云端等级时阻止保存 |
| 加载失败保护 | 加载失败时不会设置 `loaded=true` |
| 未来封印保护 | `futureLocked`/`manualOnly` 防止误解锁 |
| 黑市同步门 | 背包/仓库/交易未同步时阻断高风险出售 |
| 宠物校验 | 服务端保存时会校验宠物等级、阶数、技能、外观 |
| 仓库校验 | 服务端会按解锁行数清理越界仓库物品 |
| 丹药/属性校验 | 服务端会对部分丹药数量和属性上限做修正 |

这些保护能降低普通异常风险，但不能解决当前的单消息过大问题，也不足以满足“绝不丢档”的底线。

## 11. 当前主要风险

### 11.1 保存请求可能永远到不了服务端

当消息超过 65535 字节时，客户端发送失败，服务端不会收到 `C2S_SaveGame`。表现为保存超时、重试、反复失败。

优点：这种情况下通常不会覆盖旧云端档。

缺点：玩家新进度无法保存，且如果后续恢复逻辑处理不当，可能把问题扩大为丢档。

### 11.2 加载响应未来也会撞线

`S2C_LoadResult` 当前也一次性发送完整 `saveData` JSON。存档继续增长后，玩家可能不只是保存失败，还会加载失败。

### 11.3 客户端备份不可靠

代码里存在 `login_{slot}`、`prev_{slot}`、`deleted_{slot}` 等备份/恢复 key。但在当前网络模式下，客户端 `CloudStorage.BatchSet` 对非主存档保存请求可能跳过并返回成功。

这意味着不能把客户端写入的 `login_`、`prev_` 视为可靠云端备份。

### 11.4 自动最小档恢复不符合当前安全底线

`SaveRecovery` 在找不到真实存档时，会尝试从 `slots_index` 重建最小存档并写回。

这个逻辑对“尽量让玩家进游戏”有帮助，但对“绝不丢档”不安全。它可能把真实存档读取失败、传输失败、或云端异常，变成一个残缺的新档写回。

该逻辑后续应改成：

- 默认不写回。
- 只作为只读候选恢复信息。
- 需要明确人工确认或 GM 恢复指令后才允许落盘。

### 11.5 服务端校验存在 fail-open 路径

`SaveWriteService` 中，如果校验函数自身异常，会记录 CRITICAL 但继续保存。

业务字段修正可以宽容，但以下问题必须 fail-closed：

- JSON 解码失败。
- 槽位不合法。
- 用户归属不匹配。
- 版本结构不可识别。
- 分片不完整。
- 校验和不一致。
- 核心字段缺失。

### 11.6 旧 key 删除时机偏早

当前服务端保存会在同一个 batch 中写入 `save_{slot}` 并删除旧版 key：

- `save_data_{slot}`
- `save_bag1_{slot}`
- `save_bag2_{slot}`

如果底层 batch 原子性完全可靠，风险较低；但从“绝不丢档”的标准看，旧 key 清理应放到确认新档和备份都成功之后的独立清理阶段。

## 12. 改造原则

### 12.1 第一原则：失败不能覆盖旧档

任何保存流程都必须满足：

```text
新数据完整到达
  -> 校验通过
  -> 服务端可靠备份当前旧档
  -> 写入新档
  -> 新档写入成功
  -> 返回保存成功
```

如果任意一步失败，只能返回保存失败，不能覆盖旧档。

### 12.2 第二原则：先修传输，再做压缩

字段压缩有价值，但不能替代分片传输。因为只要游戏继续增长，单消息上限还会再次触发。

推荐顺序：

1. 保存/加载分片传输。
2. 服务端写前备份。
3. 恢复逻辑改为安全模式。
4. 命格压缩。
5. 背包/装备/仓库压缩。
6. 账号级数据与丹药真源整理。

### 12.3 第三原则：兼容旧档读取

任何压缩方案都必须满足：

- 新代码能读旧档。
- 新字段写入前有版本号。
- 老字段迁移后不能立即删除恢复路径。
- 黑市字段、丹药字段、账号字段必须有专项兼容测试。

## 13. 建议方案

### P0：保存/加载分片传输

新增分片协议，不改变云端主存档结构。

建议事件：

```text
C2S_SaveGameBegin
C2S_SaveGameChunk
C2S_SaveGameCommit
S2C_SaveGameChunkAck
S2C_SaveResult

C2S_LoadGame
S2C_LoadGameBegin
S2C_LoadGameChunk
S2C_LoadGameEnd
```

保存流程：

1. 客户端 JSON 编码完整 `saveData`。
2. 计算总长度、分片数、校验和。
3. 按安全大小切片，例如每片 24KB 到 32KB。
4. 服务端按 `requestId` 和 `slot` 暂存分片。
5. 收齐后校验长度和 checksum。
6. 解码 JSON。
7. 做核心校验。
8. 读取当前旧档并写入服务端备份。
9. 写入新 `save_{slot}`。
10. 返回成功。

加载流程：

1. 服务端读取 `save_{slot}`。
2. JSON 编码。
3. 如果小于安全阈值，可保留兼容单包返回。
4. 如果超过阈值，改为分片返回。
5. 客户端收齐后校验并解码。
6. 只有完整解码成功才进入 `SaveLoader.ProcessLoadedData`。

### P1：服务端写前备份

保存覆盖 `save_{slot}` 前，服务端应写入可靠备份，例如：

```text
backup_save_{slot}_latest
backup_save_{slot}_{timestamp}
```

最低要求：

- 备份来源必须是服务端读取到的当前云端旧档。
- 备份成功前不覆盖主档。
- 备份失败则保存失败。
- 备份写入不依赖客户端 `CloudStorage.BatchSet`。

### P2：恢复逻辑安全化

调整 `SaveRecovery`：

- 禁止默认用 `slots_index` 重建最小档并自动写回。
- 真实备份不存在时，只返回恢复失败。
- 最小档重建只能作为 GM/人工恢复工具，不应自动执行。

### P3：命格压缩

命格建议从完整快照改为最小事实字段：

```lua
{
    id = "...",
    minggeId = "...",
    bossId = "...",
    tier = 1,
    quality = 4,
    roll = 0.83,
    locked = true
}
```

可反推字段不再保存：

- `bossName`
- `name`
- `element`
- `type`
- `stat`
- `value`
- `sellCurrency`
- `sellPrice`
- `category`

前提：反序列化必须兼容旧完整格式。

### P4：物品压缩

背包、装备、仓库可逐步从完整物品快照改为“配置 ID + 动态字段”。

必须保留的动态字段：

- 数量。
- 随机词条。
- 强化/附灵/圣装结果。
- 特殊装备标记。
- 消耗品状态。
- 黑市不可转卖字段：`bmNoResell`、`bmSource`、`bmBuyAt`、`bmBuyItemId`。

### P5：账号级数据统一

统一 `atlas`、`accountCosmetics`、`accountImmortalBodies` 的加载注入路径，避免角色存档和账号 key 出现不一致。

## 14. 自测方案

### 14.1 基础保存加载

- 新建角色，保存，重登，确认等级、装备、背包、章节、任务一致。
- 老角色加载后保存，再重登，确认迁移字段无丢失。

### 14.2 大存档压力

- 构造接近 64KB、128KB、256KB 的存档。
- 验证保存分片可成功。
- 验证加载分片可成功。
- 验证分片乱序、缺片、重复片不会写档。

### 14.3 失败不覆盖

模拟以下失败：

- 保存 begin 成功，chunk 丢失。
- chunk 收齐但 checksum 错误。
- JSON 解码失败。
- 服务端备份失败。
- 主档写入失败。

预期：

- `save_{slot}` 保持旧档。
- 客户端收到保存失败。
- `_dirty` 不被清理。
- 下次可继续重试。

### 14.4 黑市专项

- 黑市购买后保存，重登确认物品存在。
- 黑市物品 `bmNoResell` 保存加载后仍存在。
- 仓库转移后 `bmNoResell` 不丢。
- 黑市 WAL 未完成时重登，确认补偿或补发逻辑不重复、不丢失。

### 14.5 丹药专项

- 吃各类炼丹丹药后保存重登，确认属性和服用次数一致。
- 血煞丹、浩气丹、暗影神丹通过挑战/任务推导后不重复增加。
- 旧档缺少 `pillCounts` 时，加载修正后保存不丢属性。

### 14.6 命格专项

- 旧完整命格格式可读取。
- 新压缩命格格式可读取。
- 命格锁定状态、品质、随机 roll、套装效果不丢。
- 命格出售价格、属性值可由配置正确反推。

### 14.7 恢复专项

- 主档存在时，优先主档。
- 主档损坏时，优先真实备份。
- 无真实备份时，不自动写入最小档。
- GM/人工恢复路径可单独验证。

## 16. 安全底线

后续任何存档改造都应满足以下底线：

1. 发送失败，不写档。
2. 分片不完整，不写档。
3. 校验失败，不写档。
4. 解码失败，不写档。
5. 备份失败，不写档。
6. 业务校验失败，可以拒绝保存，不能写残缺档。
7. 恢复失败，可以不让玩家进入，不能自动生成残缺档覆盖旧档。
8. 任何压缩和迁移必须支持旧档读取。
9. 黑市、丹药、账号级数据必须专项验证。
10. 保存成功返回之前，服务端必须已经确认新档落盘成功。

## 17. 建议下一步

P0 已按“先保命、再瘦身”的路线落地：保存/加载分片、提交前完整性校验、写前备份、禁止自动最小档写回均已完成。

1. 先持续验证满背包、满仓库、满命格、切换存档、重登、弱网/断线重试，确认分片链路稳定。
2. 再处理命格压缩，优先减少可由配置反推的展示字段。
3. 最后处理背包/装备/仓库压缩，必须保留随机词条、强化/附灵、黑市不可转卖等动态字段。

这样即使压缩代码存在兼容问题，也有服务端备份和失败不覆盖机制兜底。

## 18. 通信文档复核与本次落地（2026-07-01）

### 18.1 通信文档结论

已复核 `engine-docs/recipes/network-game-guide.md` 与 `engine-docs/api/network.md`：

- C/S 通信使用 `network:RegisterRemoteEvent(eventName)` 注册远程事件。
- 客户端与服务端通过 `Connection:SendRemoteEvent(eventName, true, VariantMap)` 发送事件。
- `VariantMap` 是事件字段承载格式，文档未提供大消息自动分片、断点续传或云存档专用分包 API。
- 因此当前 65535 bytes 报错不是 serverCloud 单 key 容量问题，而是 RemoteEvent/WebSocket 单消息限制。

结论：P0 方案应采用应用层透明分片，保留现有 `save_{slot}` 主存档语义，不立即改成系统级多 key 存档。

### 18.2 已实施内容

- 新增 `scripts/network/SavePayloadCodec.lua`：统一分片大小、1MB 上限、checksum、split/join、元数据校验。
- 新增 `scripts/network/SaveBackupService.lua`：覆盖 `save_{slot}` 前先写 `save_prewrite_backup_{slot}`；备份失败则本次写入失败。
- 保存链路：
  - 小存档继续走 `C2S_SaveGame`。
  - 大存档走 `C2S_SaveGameBegin`、`C2S_SaveGameChunk`、`C2S_SaveGameCommit`。
  - 服务端分片收齐并通过 checksum/长度校验后，仍复用 `SaveWriteService.Execute` 的成熟保存框架。
- 加载链路：
  - 小存档继续走 `S2C_LoadResult`。
  - 大存档走 `S2C_LoadGameBegin`、`S2C_LoadGameChunk`、`S2C_LoadGameEnd`。
  - 客户端组包后统一进入 `ProcessLoadPayload`，再交给 `SaveSystem.ProcessLoadedData`。
- 恢复策略：
  - 禁用从 `slots_index` 自动重建最小存档。
  - 服务端加载主档失败时会尝试读取 `save_prewrite_backup_{slot}` 作为本次加载兜底来源。
- 直接写主档路径：
  - 黑市回写、仙缘宝箱回写、兑换码发奖、登录 pending 兑换补偿、宠物技能修复补偿已纳入写前备份。

### 18.3 当前风险边界

- 分片当前主要拆 `coreData/saveData`，`slotsIndex/rankData/bulletin` 仍在 begin 包内；按当前体量风险低，但未来账号级公告或账号数据膨胀后应改为 envelope 级分片。

### 18.6 2026-07-01 首次登录满背包空状态缓解

用户实测“满背包、满仓库、满命格，来回切换存档”后，分片存档可成功保存且未再触发 65535 报错；随后发现首次登录时背包界面会出现约 1 秒空状态。

本问题判断为体验层/初始化刷新时序问题，不是丢档风险：

- `SaveSystem.loaded=false` 期间，保存入口会拒绝保存，不会把“尚未加载完成的空 UI 状态”写回。
- `SaveLoader.ProcessLoadedData` 完成反序列化后才会进入正常 loaded 状态。
- UI 空状态发生在背包 UI 首次创建/刷新期间，数据源不是空档。

已落地缓解：

- `scripts/systems/save/SaveSerializer.lua`
  - 背包和装备反序列化期间临时暂停 `InventoryManager.onChange`。
  - 反序列化完成后只执行一次 `InventorySystem.RecalcEquipStats()`。
  - 目的：避免满背包恢复时每个格子写入都触发 UI/属性连锁刷新。
- `scripts/ui/InventoryUI.lua`
  - 背包 UI 创建完成后立即执行 `UpdateAllSlots()`、`UpdateStats()`、`UpdateFreeSlots()`。
  - 目的：首次打开前预热格子显示，减少“空面板闪一下”的体感。

该优化不改变存档结构，不改变保存协议，不改变背包数据内容，仅减少加载后的 UI 首刷延迟和恢复期重复刷新。

## P0-1: RedeemHandler MAX_SLOTS=40（应为 60）

**文件**: `scripts/network/RedeemHandler.lua:105`

| 维度 | 内容 |
|------|------|
| **改动** | `local MAX_SLOTS = 40` → 改为 `BackpackUtils.MAX_BACKPACK_SLOTS` 或 `GameConfig.BACKPACK_SIZE`（值=60）。RedeemHandler 是唯一自行实现背包逻辑而非调用 `BackpackUtils.AddToBackpack()` 的 Handler |
| **评估** | 兑换码发物只扫描前 40 格。玩家背包 41-60 格有空位时仍提示"背包已满"，物品**静默丢失不可追回**。注释 `-- GameConfig.BACKPACK_SIZE` 说明本意跟随 60，写成了过时字面量 |
| **修复复杂度** | **低** — 改一个常量引用，或替换整段为 `BackpackUtils.AddToBackpack()` 调用 |
| **修复风险** | **零** — 只扩大扫描范围 40→60，不改存档结构，不改序列化格式 |
| **测试** | **自测**: GM 给物品填满前 40 格 → 确认 41-60 格空 → 兑换码兑换 consumable → 验证物品落入 41 格<br>**脚本**: 构造 `saveData.backpack[1..40]` 全满 → 调用 `ServerApplyRedeemRewards` → 断言 `remaining == 0`<br>**回归**: 背包未满时兑换、背包全满时兑换、堆叠上限溢出时兑换 |

## P0-4: RankHandler TOCTOU 竞态导致道印超发

**文件**: `scripts/network/RankHandler.lua:392-455`

| 维度 | 内容 |
|------|------|
| **改动** | `GetRankList` → check count → `SetInt` 三步非原子操作，改为 `serverCloud:Quota` 原子预占名额或 `BatchCommit` 条件写入 |
| **评估** | 道印限量 `maxSlots=10`/种。两个请求并发读到 `count=9`，都通过 `< maxSlots` 检查，都 `SetInt` 成功，排行榜实际 11 人。典型 TOCTOU 竞态。超发后回收引发运营争议，不回收则限量品稀缺性永久破坏 |

## P0-5: SaveRecovery 境界循环 break 位置错误

**文件**: `scripts/systems/save/SaveRecovery.lua:175-186`

| 维度 | 内容 |
|------|------|
| **改动** | 将 `if rid == rebuildRealm then break end` 从循环体**开头**移到**末尾**（累加之后） |
| **评估** | `break` 在累加之前 → 当前境界的 rewards 永远不被计入。练气初期玩家恢复时少 `maxHp=15, atk=3, def=2, hpRegen=0.2`；境界越高丢失越大。SaveRecovery 是存档损坏的**最后防线**，恢复出错误数据导致二次损失。同源 BUG 曾出现在 SaveMigrations v14，已在 v15 修复，此处为遗漏修复点 |
| **修复复杂度** | **极低** — 移动一行 `break` |
| **修复风险** | **零** — 只改恢复路径的累加逻辑，不触碰正常存档读写。已被错误恢复的存档需运营决定是否重新恢复或手动补偿 |
| **测试** | **单元测试**: 构造 `rebuildRealm = "lianqi_1"` → 跑循环 → 断言 `realmBonusHp == 15`（当前 BUG 得到 0）<br>**边界**: `rebuildRealm = "mortal"`（mortal 无 rewards，累加 0 后 break，结果应为 0）<br>**回归**: 正常存档加载路径不经过此函数，不受影响 |

## P0-6: EventBus.Emit 正向遍历 + Remove 跳过监听器

**文件**: `scripts/core/EventBus.lua`（57 行）

| 维度 | 内容 |
|------|------|
| **改动** | `Emit` 中 `for i = 1, #cbs` 改为遍历快照：`local snapshot = {table.unpack(cbs)}; for i = 1, #snapshot` |
| **评估** | 监听器 C 在回调中调用 `Off` 自我移除 → `table.remove(cbs, 3)` 使后续元素前移 → 循环 `i=4` 跳过了原来的 D → D 永远不执行。`pcall(cbs[5])` 访问 nil 还会产生一条错误日志。**静默性**是最大风险：被跳过的监听器不报错，只是不执行。EventBus 是全项目事件枢纽，`On`/`Off`/`Emit` 调用遍布各系统 |
| **修复复杂度** | **低** — Emit 中加一行快照，或改为反向遍历 |
| **修复风险** | **零** — 纯运行时逻辑，不涉及数据持久化。快照方案有微量 GC 开销，但 EventBus 通常监听器数量 < 20，可忽略 |
| **测试** | **单元测试**: 注册 A、C（回调中 Off 自身）、D → `Emit` → 断言 callLog = `{"A","C","D"}`（当前 BUG 得到 `{"A","C"}`）<br>**边界**: 回调中 On 新增监听器（快照方案下本轮不触发，符合预期）<br>**回归**: grep 全项目 `EventBus.Off`，确认所有 Off 调用场景下后续监听器不再被跳过 |

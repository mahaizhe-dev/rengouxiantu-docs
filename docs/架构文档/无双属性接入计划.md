## 一、审查结论

本稿已对以下代码链路做过交叉审查：

- `scripts/entities/Player.lua`
- `scripts/systems/combat/HitResolver.lua`
- `scripts/systems/CombatSystem.lua`
- `scripts/systems/SkillSystem.lua`
- `scripts/systems/combat/BossMechanics.lua`
- `scripts/systems/InventorySystem.lua`
- `scripts/ui/InventoryUI.lua`
- `scripts/utils/StatNames.lua`
- `scripts/config/EquipmentData.lua`
- `scripts/systems/LootSystem.lua`
- `scripts/systems/save/SaveSerializer.lua`
- `scripts/entities/Pet.lua`
- `scripts/rendering/DungeonHUD.lua`
- `scripts/network/DungeonClient.lua`
- `scripts/systems/combat/BattleContracts.lua`

结论先定如下：

1. 天诛必须挂在 `Player:ApplyCrit(...)` 层，不能只挂 `HitResolver`。当前仍有多处手写 `player:ApplyCrit(...)` 调用。
2. 天诛属性不能直接加入 `EquipmentData.SUB_STATS`。该池同时被普通副属性、灵性生成、洗练复用，直接加入会污染普通词条池和锻造池。
3. `天诛伤害` 的基础值是 `150%`，因此角色面板不能只显示装备增量，否则默认 `150%` 会丢失。
4. `spiritStat` 链路已存在，`saintStat` 还不是完整契约。聚合、Tooltip、挑战预览、龙锻预览、存档都要补。
5. 宠物暴击在 `Pet:CalcPetDamage()` 内独立实现。首版天诛不默认覆盖宠物。

## 二、属性定义

### 2.1 字段与显示名

| 内部字段 | 面板显示 | 存储语义 | 默认总值 |
|------|------|------|------|
| `tianzhuChance` | `天诛` | 概率加成 | `0%` |
| `tianzhuDamage` | `天诛伤害` | `150%` 之外的附加值 | `150%` |

说明：

- `tianzhuChance` 的存储方式与 `critRate` 一致，`0.01 = 1%`。
- `tianzhuDamage` 的存储方式与 `critDmg` 一致，只存附加值，不存基础 `150%`。
- 装备、灵性、圣性、命格给的是 `天诛伤害加成`，角色总值通过 Getter 叠到基础 `150%` 上。

### 2.2 建议 Getter

```lua
Player:GetTotalTianzhuChance()   -- 0 + 所有来源加总
Player:GetTotalTianzhuDamage()   -- 1.5 + 所有来源加总
```

结论：

- `tianzhuChance` 默认 `0`
- `tianzhuDamage` 的增量默认 `0`
- `GetTotalTianzhuDamage()` 的默认显示结果是 `150%`

### 2.3 结算公式

统一公式：

```lua
暴击最终伤害 = 原始伤害 × (150% + 暴击伤害)
天诛最终伤害 = 暴击最终伤害 × (150% + 天诛伤害)
```

实现层可写成：

```lua
critFinalDamage = math.floor(baseDamage * (1.5 + critBonus))
tianzhuFinalDamage = math.floor(critFinalDamage * (1.5 + tianzhuDamageBonus))
```

说明：

- 天诛伤害算法与暴击伤害完全一致。
- `tianzhuDamage = 0` 时，天诛仍然有固定 `150%` 倍率。
- 灵性、圣性提供的是 `tianzhuDamageBonus`，不会覆盖基础 `150%`。

### 2.4 数值示例

- `天诛 0%`，`天诛伤害 150%`
  触发天诛时：`天诛最终伤害 = 暴击最终伤害 × 150%`
- 灵性天诛伤害 `+5%`
  角色面板显示：`天诛伤害 155%`
- 圣性天诛伤害 `+10%`
  角色面板显示：`天诛伤害 160%`

## 三、生效范围

### 3.1 首版覆盖范围

首版只覆盖“玩家来源，且当前走玩家暴击链”的伤害：

- 普攻
- 重击
- `HitResolver.Standard(...)`
- `HitResolver.TrueDamage(...)`
- `HitResolver.Hit(...)` 中 `source = player` 且允许暴击的路径
- `SkillSystem` / `BossMechanics` 中直接调用 `player:ApplyCrit(...)` 的路径
- `conditionalCrit = true` 且 `canCrit = true` 的技能路径

### 3.2 首版明确不覆盖

- `skipCrit = true` 的路径
- `canCrit = false` 的路径
- `HitResolver.NoCrit(...)`
- `HitResolver.Bypass(...)`
- 宠物普攻与宠物技能
- 非伤害事件

### 3.3 未来扩展位

若后续需要让某条“可暴击伤害”排除天诛，单独补：

```lua
canTianzhu = false
```

判定顺序固定为：

1. 先判暴击
2. 暴击成立后再判是否允许天诛
3. 最后再判天诛触发

## 四、战斗接入方案

### 4.0 硬约束

统一硬约束：

```text
所有玩家侧可暴击伤害，必须统一经过 Player:ApplyCrit()。
```

落地含义：

- 普攻必须经过 `Player:ApplyCrit()`
- 重击必须经过 `Player:ApplyCrit()`
- 技能伤害只要允许暴击，就必须经过 `Player:ApplyCrit()`
- 特殊机制伤害只要允许暴击，就必须经过 `Player:ApplyCrit()`
- 不允许在业务代码里手写第二套暴击判定

这样可以保证：

- 天诛只需要挂一次，就能覆盖所有玩家侧可暴击伤害
- 后续新增技能不会因为走了旁路而漏掉天诛
- 暴击、天诛、飘字、统计的判断口径保持一致

### 4.1 必须挂载在 `Player:ApplyCrit`

原因：

- `HitResolver` 已统一大部分伤害链路
- 但 `CombatSystem.lua`、`SkillSystem.lua`、`BossMechanics.lua` 仍有多处直接写 `player:ApplyCrit(...)`
- 如果只改 `HitResolver`，这些路径会漏接

因此首版建议：

```lua
Player:ApplyCrit(damage) -> finalDamage, isCrit, isTianzhu
```

兼容性：

- 老调用方只接前两个返回值，不会崩
- 新调用方可以读取第三值做飘字、统计、联机表现

### 4.2 `Player` 侧新增字段

- `equipTianzhuChance = 0`
- `equipTianzhuDamage = 0`
- `GetTotalTianzhuChance()`
- `GetTotalTianzhuDamage()`

### 4.3 `HitResolver` / `BattleContracts`

建议同步把第三返回值一路往上透：

```lua
damage, isCrit, isTianzhu
```

`BattleContracts.NewHitResult()` 建议增加：

- `isTianzhu = false`

### 4.4 事件兼容

当前 `player_deal_damage` 老签名是：

```lua
player, monster, isDot
```

如果要把天诛状态透给监听方，建议只追加到末尾：

```lua
player, monster, isDot, isTianzhu
```

不要改前三个参数顺序，否则现有 `isDot` 过滤逻辑会被破坏。

### 4.5 飘字与日志

本地战斗表现建议：

- 普通伤害：`1234`
- 暴击伤害：`暴击 1234`
- 天诛：`天诛 1851`

联机 / 地下城若需要单独表现天诛：

- `DungeonHandler` 攻击结果 payload 增加 `isTianzhu`
- `DungeonClient._HandleAttackResult()` 解析 `isTianzhu`
- `DungeonHUD.OnAttackResult(attacker, damage, isCrit, isTianzhu)` 增加第四参

如果首版不做联机表现，也至少要保证数值正确，不阻断以后补字段。

## 五、属性聚合与角色面板

### 5.1 `InventorySystem` 聚合

`InventorySystem.RecalcEquipStats()` 需要补：

- `totals.tianzhuChance`
- `totals.tianzhuDamage`
- `item.saintStat` 的累加逻辑
- `STAT_TO_PLAYER.tianzhuChance = "equipTianzhuChance"`
- `STAT_TO_PLAYER.tianzhuDamage = "equipTianzhuDamage"`
- `GetEquipStatSummary()` 增加这两项增量值

### 5.2 角色面板显示口径

当前 `InventoryUI` 展示的是“装备贡献汇总”，不是角色总战斗面板。

这对 `天诛伤害` 不够用，因为该属性有基础值 `150%`。如果只显示装备增量，那么默认值会错误显示成 `+0%`。

因此需要拆成两种口径：

- 装备 Tooltip / 词条预览：显示增量值，例如 `+5%`
- 角色面板：显示总值
  - `天诛 = 0.0%`
  - `天诛伤害 = 150.0%`

### 5.3 首版最小实现建议

首版可以继续复用 `InventoryUI`，但 `天诛` / `天诛伤害` 两行不要只读 `GetEquipStatSummary()`。

建议直接显示：

```lua
player:GetTotalTianzhuChance()
player:GetTotalTianzhuDamage()
```

这样能正确显示默认值 `150%`。

### 5.4 面板行名

角色面板建议写：

- `天诛`
- `天诛伤害`

说明文案里再解释：

- `天诛`：暴击后再次判定
- `天诛伤害`：天诛触发后的总倍率

### 5.5 `StatNames` 约束

`StatNames.lua` 需要补：

- `NAMES.tianzhuChance = "天诛"`
- `NAMES.tianzhuDamage = "天诛伤害"`
- `SHORT_NAMES`
- `ICONS`
- `COLORS`
- `PCT_STATS`

但要注意：

- `FormatValue("tianzhuDamage", 0.05)` 用于词条时，应显示 `+5.0%`
- 角色面板总值 `150%` 不应直接复用这个格式器

也就是说，`tianzhuDamage` 存在“词条增量显示”和“角色总值显示”两种口径，不能混用。

## 七、属性投放

### 7.0 装备体系硬约束

统一硬约束：

```text
在装备体系内，天诛 / 天诛伤害只能出现在灵性属性、圣性属性。
```

反向约束：

- 不允许出现在主属性
- 不允许出现在普通副属性
- 不允许出现在洗练属性
- 不允许出现在普通随机装备词条池
- 不允许出现在独特装备固定副属性
- 不允许出现在任何复用普通副属性池的随机入口

也就是说：

- 装备系统内，`tianzhuChance` / `tianzhuDamage` 是 **灵性、圣性独有属性**
- 它们不是通用副属性，只是通用战斗属性

### 7.1 固定投放值

| 来源 | 天诛 | 天诛伤害 |
|------|------|------|
| 灵性 | `1%` | `5%` |
| 圣性 | `2%` | `10%` |

### 7.2 关键约束

天诛属性不能加入任何通用随机词条池，尤其不能直接加入 `EquipmentData.SUB_STATS`。

原因：

- `LootSystem.GenerateSubStats()` 使用 `SUB_STATS`
- `LootSystem.GenerateSpiritStat()` 也使用 `SUB_STATS`
- `ForgeUI` 的洗练池同样使用 `SUB_STATS`
- 独特装备模板里的 `subStats` 也是固定副属性入口
- 以后若有其他复用 `SUB_STATS` 的生成器，也会一起被污染

如果直接加入，会导致：

- 普通副属性可能刷出天诛
- 洗练可能洗出天诛
- 独特装备可能被错误配置为固定带天诛副属性
- 灵性专属词条设计失效
- 圣性独有定位失效

### 7.3 建议数据池

新增独立池：

- `EquipmentData.SPIRIT_STATS`
- `EquipmentData.SAINT_STATS`

示例：

```lua
EquipmentData.SPIRIT_STATS = {
    { stat = "tianzhuChance", name = "天诛", value = 0.01 },
    { stat = "tianzhuDamage", name = "天诛伤害", value = 0.05 },
}

EquipmentData.SAINT_STATS = {
    { stat = "tianzhuChance", name = "天诛", value = 0.02 },
    { stat = "tianzhuDamage", name = "天诛伤害", value = 0.10 },
}
```

对应生成器建议：

- `LootSystem.GenerateSpiritStat()` 不再直接遍历 `SUB_STATS`
- 后续若做圣性，新增 `LootSystem.GenerateSaintStat()`

额外约束：

- `GenerateSubStats()` 永远不读取天诛属性
- `ForgeUI` 洗练池永远不读取天诛属性
- 独特装备模板 `subStats` 不允许手填天诛属性
- 若后续增加新的随机词条系统，也必须默认排除天诛属性，除非该系统明确属于灵性或圣性

### 7.4 数据结构

```lua
item.spiritStat = {
    stat = "tianzhuChance",
    name = "天诛",
    value = 0.01,
}

item.saintStat = {
    stat = "tianzhuDamage",
    name = "天诛伤害",
    value = 0.10,
}
```

### 7.5 验证规则

投放验收时，必须额外检查：

1. 全量 `SUB_STATS` 中不存在 `tianzhuChance` / `tianzhuDamage`
2. 全量洗练池中不存在 `tianzhuChance` / `tianzhuDamage`
3. 全量独特装备模板 `subStats` 中不存在 `tianzhuChance` / `tianzhuDamage`
4. 灵性、圣性之外的任何随机装备生成流程，均不会产出天诛属性

## 九、验收项

### 9.1 战斗验收

1. `tianzhuChance = 0` 时，天诛永不触发
2. `tianzhuDamage = 0` 时，天诛倍率固定为 `150%`
3. `tianzhuDamage = 0.05` 时，天诛倍率为 `155%`
4. 直接调用 `player:ApplyCrit(...)` 的路径也能触发天诛
5. `skipCrit = true` 与 `canCrit = false` 的路径不触发天诛
6. 宠物攻击不触发天诛

### 9.2 投放验收

1. 灵性只能出 `1%` / `5%`
2. 圣性只能出 `2%` / `10%`
3. 普通副属性池不出现天诛
4. 洗练池不出现天诛

### 9.3 UI 与存档验收

1. 角色面板显示 `天诛` / `天诛伤害`
2. 无任何来源时，面板显示 `0.0%` / `150.0%`
3. 灵性 `+5%` 时，Tooltip 显示 `+5.0%`，角色面板总值显示 `155.0%`
4. `saintStat` 在 Tooltip / 挑战预览 / 龙锻预览 / 存档读回都正常

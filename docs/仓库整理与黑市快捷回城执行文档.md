> 日期：2026-05-18
> 目标：为执行 AI 和审核 AI 提供一份可直接落地的实现文档。
> 范围：本次仅覆盖两个功能。
> 1. 参照现有背包整理，实现仓库整理功能，并补齐安全护栏。

1. 背包整理不是纯排序，它还会做消耗品堆叠合并。
2. 黑市锁定堆叠与普通堆叠不能合并，不同 `bmLockBatchId` 之间也不能合并。

5. 章节安全传送点与死亡复活点当前统一使用 `ActiveZoneData.Get().SPAWN_POINT`。

## 2. 实现目标

### 2.1 功能 A：仓库整理

新增仓库整理按钮，一键整理当前已解锁仓库格子。

目标行为：

1. 排序规则与背包整理保持一致。
2. 消耗品可堆叠，但必须完整继承黑市锁规则。
3. 未解锁格子不参与整理。
4. 整理成功后更新仓库 UI，并走现有仓库脏标记与存档链路。
5. 整理不能成为绕过黑市保护锁、未同步门禁和存档校验的旁路。

### 2.2 功能 B：黑市快捷回城

在黑市入口图标下方增加一个独立入口按钮，点击后开始 2 秒读条，读条完成将玩家传送至本章节安全点。

目标行为：

1. 入口位置在黑市按钮下方，风格保持同一套小地图功能入口。
2. 读条时长固定 2.0 秒。
3. 读条不会被攻击打断。
4. 玩家主动移动会打断读条。
5. 读条成功后回到本章节 `SPAWN_POINT`。
6. 成功时播放简单脚下法阵/光柱/粒子光效。
7. 传送后同步宠物、移动方向、卡住检测状态，并立即请求存档。

## 3. 功能 A：仓库整理执行方案

### 3.1 排序规则

仓库整理必须与背包整理共享同一套规则，不允许复制一份逻辑后各自演化。

建议规则保持与 `InventorySystem.SortBackpack()` 一致：

1. 先收集 `1..WarehouseSystem.GetUnlockedSlots()` 内全部非空物品。
2. 对消耗品执行堆叠合并。
3. 排序顺序：
   - 装备在前，消耗品在后。
   - 装备：品质降序 -> `tier` 降序 -> `slot` 升序。
   - 消耗品：`consumableId` 升序。
4. 清空已解锁仓库格子，再按排序结果从前往后写回。
5. 未解锁格子保持不动，也不能被写入。

### 3.2 共享算法抽取

不要在 `WarehouseSystem.lua` 再手写一套近似逻辑。建议抽取共享 helper。

建议方案：

1. 新增 `scripts/systems/InventorySortUtil.lua`。
2. 抽出两个能力：
   - `NormalizeAndSortItems(items, opts)`
   - `BuildConsumableMergeKey(item, tradeLock)`
3. `InventorySystem.SortBackpack()` 改为调用 helper。
4. `WarehouseSystem.SortWarehouse()` 调用同一 helper。

这样能保证：

1. 背包与仓库整理行为一致。
2. BM-S4A / BM-S4AR 锁规则只维护一份。
3. 后续若新增排序字段，不会只改一边。

### 3.3 锁定堆叠安全规则

仓库整理必须遵守以下硬规则：

1. `TradeLock.IsLocked(item)` 的堆叠不能与未锁定堆叠合并。
2. 两个锁定堆叠只有在 `bmLockBatchId` 相同的情况下才允许合并。
3. 锁元数据字段必须原样保留：
   - `bmLockUntil`
   - `bmLockSource`
   - `bmLockBatchId`
4. 若拆分堆叠生成新 table，必须复制完整业务字段，并为新表生成新 `id`。
5. 装备类物品不能在整理过程中被浅拷贝重建，避免丢失附灵、洗练、词条、神器信息。

### 3.3.1 装备引用交换规则（防字段丢失）

**装备在排序过程中只能做"槽位引用交换"，禁止值拷贝重建。**

原因：装备 table 包含大量嵌套字段（强化等级、附灵属性、洗练词条、神器信息、套装标记等），使用 `for k,v in pairs` 浅拷贝会丢失嵌套表的引用完整性。

实现要求：

1. 排序阶段只收集引用到临时数组，排序后直接将引用写回目标槽位。
2. 禁止对装备 table 执行 `for k,v in pairs` 拷贝。
3. 消耗品合并时如需创建新 table，必须使用深拷贝工具函数，并验证所有业务字段完整。
4. 排序前后必须断言：每个装备 table 的 `id` 集合完全一致（无丢失、无重复、无新增）。

### 3.3.2 物品数量守恒断言

**排序前后必须做数量守恒校验，不通过则回滚并报错。**

实现要求：

1. 排序前统计：装备总件数 + 每种消耗品（按 mergeKey 分组）的总 count。
2. 排序后统计同样的数据。
3. 两组数据必须完全一致，否则：
   - 不写回仓库/背包。
   - 返回 `false, "数量守恒校验失败"`。
   - 记录错误日志。
4. 堆叠合并时单组 count 不得超过 `GameConfig.MAX_STACK_COUNT`，超出部分保留为独立堆叠。

### 3.3.3 并发保护与防抖

**排序操作必须有互斥保护，防止双击或与存取操作并发。**

实现要求：

1. `SortWarehouse()` 入口处检查互斥标志 `_sorting`，若为 true 直接返回 `false, "整理中"`。
2. 执行期间设置 `_sorting = true`，无论成功或失败最终都重置为 `false`。
3. `StoreItem` / `StoreItemToSlot` / `RetrieveItem` 在 `_sorting == true` 时返回 `false, "整理中，请稍后"`。
4. UI 按钮在点击后短暂禁用（300ms 防抖），防止快速双击。
5. 背包 `SortBackpack()` 同样需要加入互斥标志（若尚未有）。

### 3.4 仓库系统新增接口

建议在 `scripts/systems/WarehouseSystem.lua` 新增：

```lua
function WarehouseSystem.SortWarehouse()
    -- return true/false, errOrNil, changed:boolean
end
```

建议行为：

1. 仅操作已解锁格子。
2. 排序前后做一次结果比较。
3. 如果无变化：
   - 返回 `true, nil, false`
   - 不置脏
   - 不发 `save_request`
4. 如果有变化：
   - 回填仓库
   - `WarehouseSystem._dirty = true`
   - `BlackMarketSyncState.MarkWarehouseOp("sort")`
   - `EventBus.Emit("warehouse_changed")`
   - `EventBus.Emit("save_request")`
   - 返回 `true, nil, true`

### 3.5 仓库 UI 改动

文件：`scripts/ui/WarehouseUI.lua`

改动建议：

1. 在仓库标题行加入“整理”按钮，位置与背包 UI 风格对齐。
2. 点击后调用 `WarehouseSystem.SortWarehouse()`。
3. 根据返回值显示 toast：
   - `仓库已整理`
   - `仓库无需整理`
   - 或错误信息
4. 操作完成后调用 `UpdateAllSlots()`。

### 3.6 仓库存取安全补强

当前仓库存取需要补锁校验，避免锁定物品被塞进仓库或从仓库继续流转。

文件：`scripts/systems/WarehouseSystem.lua`

需要补的点：

1. `StoreItem(bagSlot)` 读取背包物品后，先调用 `TradeLock.IsOperationBlocked(item)`。
2. `StoreItemToSlot(bagSlot, whSlot)` 同样检查。
3. `RetrieveItem(whSlot)` 读取仓库物品后，也执行相同检查。

推荐策略：

1. 被锁定则直接返回 `false, TradeLock.LOCK_MESSAGE`。
2. 不改变任何状态。
3. 不触发 `save_request`。

说明：

1. 从纯规则上看，黑市锁的设计目标是“保护期内阻断一切本地状态变更”。
2. 仓库存取属于状态变更，因此应被阻断。
3. 整理不额外新增阻断，只要它不会合并或洗掉锁即可。

### 3.7 存档与服务端一致性

本功能不能修改现有存档结构。

原因：

1. 仓库序列化已由 `WarehouseSystem.Serialize()` 负责。
2. 服务端已有 V6 仓库校验：
   - `unlockedRows` 范围
   - `items` 槽位范围
3. 本次整理只重排 `items` 内容，不新增字段。

因此：

1. 不需要修改 `SaveSerializer` DTO。
2. 不需要提升存档版本号。
3. 不需要增加 migration。

### 3.8 功能 A 测试清单

建议新增 `scripts/tests/test_warehouse_sort.lua`，并补齐既有仓库/黑市一致性测试。

至少覆盖以下用例：

**基础排序（1-6）：**

1. 装备按品质、阶数、部位排序正确。
2. 普通消耗品整理后合并堆叠。
3. 锁定堆叠与普通堆叠不会合并。
4. 不同 `bmLockBatchId` 的锁定堆叠不会合并。
5. 相同 `bmLockBatchId` 的锁定堆叠允许合并。
6. 未解锁格子不参与整理。

**事件与存档（7-8）：**

7. 整理结果无变化时，不触发 `save_request`。
8. 整理结果有变化时，触发 `warehouse_changed` 和 `save_request`。

**锁定物品存取拦截（9-10）：**

9. 锁定消耗品无法存仓。
10. 锁定消耗品无法取仓。

**装备字段完整性（11-13）：**

11. 装备排序前后，每件装备的所有字段（强化等级、附灵属性、洗练词条、神器信息、套装标记等）完全一致——用 `id` 匹配后逐字段 `assert`。
12. 装备排序前后，装备 `id` 集合完全一致（无丢失、无重复、无新增）。
13. 混合场景：仓库同时存在装备 + 普通消耗品 + 锁定消耗品，整理后三类物品各自规则均正确。

**数量守恒（14-16）：**

14. 排序前后 `sum(count)` 对每种消耗品（按 mergeKey）完全相等。
15. 合并时单组 count 达到 `MAX_STACK_COUNT` 上限，超出部分保留为独立堆叠（如 2 组各 80，上限 99 → 合并为 99 + 61）。
16. 仅有 1 个物品的仓库，排序后仍为 1 个物品，无幽灵产生。

**边界场景（17-20）：**

17. 空仓库执行整理：不崩溃，返回 `true, nil, false`（无变化）。
18. 满仓库（所有已解锁格子都有物品）执行整理：排序正确，无溢出。
19. 排序幂等性：连续执行 2 次 `SortWarehouse()`，第二次返回 `changed = false`。
20. 并发保护：`_sorting == true` 期间调用 `StoreItem` / `RetrieveItem` 返回 `false`。

## 4. 功能 B：黑市快捷回城执行方案

### 4.1 入口形态

建议不要把按钮塞进 `BlackMerchantUI` 面板内部，而是直接做成与黑市入口同层的第二个快捷入口。

原因：

1. 用户需求明确是“黑市下方增加快捷传送功能”。
2. 现有黑市入口本身就挂在 `minimapContainer` 下方，位置稳定。
3. 做成独立入口后，玩家不必先打开黑市面板。
4. 该功能本质是快捷操作，不是黑市交易子功能。

建议新建文件：

`scripts/ui/TownReturnEntry.lua`

挂载策略：

1. 仍然挂到 `minimapContainer`。
2. 添加顺序在 `BlackMerchantEntry.Create(overlay)` 之后。
3. 视觉尺寸与黑市入口保持一致或略小。

### 4.2 新系统模块

建议新建：

`scripts/systems/TownReturnSystem.lua`

推荐内部状态：

```lua
TownReturnSystem.CHANNEL_DURATION = 2.0
TownReturnSystem._channeling = false
TownReturnSystem._channelTimer = 0
TownReturnSystem._fxTimer = 0
TownReturnSystem._justTeleportedFx = 0
```

推荐接口：

### 4.3 读条规则

直接参考福源果系统的客户端读条框架，但改成本功能的交互约束。

启动规则：

1. 第一次点击按钮，开始 2 秒读条。
2. 读条中再次点击按钮，取消读条。

中断规则：

1. 不被攻击打断。
2. 玩家主动移动会打断。
3. 不因攻击输入打断。
4. 仅在以下情况取消：
   - 玩家主动再次点击取消
   - 玩家主动移动
   - 玩家死亡
   - 场景切换
   - 当前模式不允许使用
   - 打开全屏 UI 面板（背包、仓库、商店等）

注意：读条开始后必须锁定玩家攻击输入，防止读条中触发技能动作导致表现异常。

### 4.4 使用限制

建议在系统层硬拦截以下场景：

1. `ChallengeSystem.IsActive()`
2. `TrialTowerSystem.IsActive()`
3. `PrisonTowerSystem.IsActive()`
4. `DungeonClient.IsDungeonMode()`
5. 玩家不存在或已死亡

失败时给出浮字或 toast，例如：

`当前场景无法使用回城`

### 4.5 传送目标

目标点统一使用：

`ActiveZoneData.Get().SPAWN_POINT`

理由：

1. 当前复活逻辑已使用这一落点。
2. 各章节 `ZoneData` 已维护该坐标。
3. 用户需求是“本章节安全传送点 or 复活点”，当前项目里这两者已统一。

### 4.6 传送成功后的状态同步

`PerformTeleport()` 不能只改 `player.x/y`，必须完整同步。

建议至少执行：

1. `player.x = spawn.x`
2. `player.y = spawn.y`
3. `player.lastMoveX = spawn.x`
4. `player.lastMoveY = spawn.y`
5. `player.stuckTimer = 0`
6. `player.dx = 0`
7. `player.dy = 0`
8. `player.moving = false`
9. `player:SetMoveDirection(0, 0)`
10. `player.target = nil`

宠物同步：

1. `pet.x = spawn.x + 0.5`
2. `pet.y = spawn.y + 0.5`
3. `pet.target = nil`
4. `pet.state = "follow"`

怪物仇恨建议同步清理：

1. 遍历 `GameState.monsters`
2. 若目标指向玩家或宠物，则清空 `target`
3. 必要时将状态置回 `idle`

说明：

1. 这一步不是死亡复活，但它属于“安全点重定位”。
2. 若不清理目标，玩家回到安全点后可能立刻被远处仇恨链重新追踪。

### 4.7 成功后的存档策略

传送成功后发一次：

`EventBus.Emit("save_request")`

理由：

1. 玩家坐标是存档字段。
2. 若不落盘，玩家回城后立即退出，重登可能仍出现在危险点。
3. 该行为与“关键位置重定位”一致，值得即时落盘。

### 4.10 简单传送光效

推荐参考：

`scripts/rendering/DecoGeneric.lua` 中 `RenderTeleportArray`

**场景拦截（6）：**

**传送结果（7-10）：**

8. 传送后同步 `lastMoveX/Y`、`stuckTimer`、方向和宠物位置。
9. 传送成功后触发 `save_request`。
10. 光效状态计时器会正确开始和结束。

**边界场景（11-16）：**

11. 读条中打开全屏 UI 面板（背包、仓库等），读条取消。
12. 读条中切换章节/区域，读条取消且无残留定时器。
13. 读条期间攻击输入被锁定，不触发技能动作。
14. 读条取消后立即再次点击，可正常重新开始读条（无状态残留）。
15. 传送后怪物仇恨链正确清理，不会被远处怪物立刻追踪。
16. `CanUse()` 在玩家不存在/已死亡时返回 `false`。

## 7. 验收口径

### 7.1 仓库整理验收

1. 点击仓库整理后，仓库内物品顺序稳定。
2. 同类普通消耗品会自动合堆。
3. 黑市锁定堆叠不会与普通堆叠合并。

5. 整理后重新登录，仓库顺序保持一致。

### 7.2 黑市快捷回城验收

1. 黑市下方出现独立回城图标。
2. 点击后开始 2 秒读条，玩家可正常被打但读条不中断；玩家主动移动会打断读条。
3. 读条完成后玩家和宠物出现在当前章节安全点。
4. 传送时有清晰但不夸张的法阵/光柱效果。
5. 传送后退出重登，位置仍为安全点。

## 9. 审核重点

审核 AI 需要重点看以下风险点：

**仓库整理（1-8）：**

1. 是否真的复用了同一套排序规则，而不是复制粘贴两份。
2. 仓库整理是否保留了黑市锁字段。
3. 仓库存取是否拦截了锁定消耗品。
4. no-op 整理是否错误触发 `save_request`。
5. 🔴 装备是否仅做引用交换，未使用浅拷贝重建（防字段丢失）。
6. 🔴 排序前后是否有物品数量守恒断言（装备件数 + 每种消耗品 count 总和）。
7. 🔴 是否有 `_sorting` 互斥标志，防止排序期间并发存取或双击触发。
8. 堆叠合并是否正确处理 `MAX_STACK_COUNT` 上限溢出。

**快捷回城（9-14）：**

9. 快捷回城是否错误允许在副本/挑战类场景使用。
10. 传送后是否同步了 `lastMoveX/Y` 和 `stuckTimer`。
11. 传送成功后是否遗漏 `save_request`。
12. 读条期间是否锁定了攻击输入。
13. 打开全屏 UI 面板是否正确取消读条。
14. 读条取消后是否有状态残留（定时器、标志位）。

## 10. 最终建议

本任务的正确做法不是“各做一个功能”，而是：

1. 把仓库整理收口到现有背包整理规则体系里。
2. 把快捷回城收口到现有章节安全点体系里。

只要这两个“收口”做对，功能本身实现并不复杂，后续维护成本也最低。

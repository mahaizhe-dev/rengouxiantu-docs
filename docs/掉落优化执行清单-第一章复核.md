## 1. 新增巡逻 Boss 机制

本节定义通用巡逻 Boss 机制，供 `拦路蛛妖`、`二大王` 及后续章节巡逻 Boss 直接复用。

### 机制目标

- 巡逻 Boss 必须是一个可复用机制，不是为 `拦路蛛妖` 单独写死的临时逻辑。
- 机制建立后，后续 Boss 只通过配置即可获得相同能力。
- 适用对象至少包括：
  - 第一章 `拦路蛛妖`
  - 第一章 `二大王`
  - 第二章 `乌大傻`
  - 第三章外围巡逻 Boss
  - 第四章八卦岛随机 / 巡逻类 Boss

### 最终分层

- `行为层`
  - 统一放在 `Monster.lua` / `Spawner.lua`
  - 负责巡逻状态机、锚点 / 路点切换、脱战回路点、远处降频兼容
- `模板层`
  - 建议单独新增 `PatrolPresets.lua`
  - 负责沉淀几类可复用巡逻模板
- `实例层`
  - 具体 Boss 的 `spawn` 只填写 `patrolPreset` 与必要覆盖项
  - 不在 `MonsterTypes` 里硬编码地图巡逻细节

### 推荐模板

- `large_waypoint_loop`
  - 适合狭长通路、寨内绕场、固定巡逻线
- `large_region_roam`
  - 适合开阔野外，在指定区域内多锚点随机巡逻
- `fortress_loop`
  - 适合堡垒 / 山寨 / 院落，围绕核心空地和入口巡逻
- `outer_ring_roam`
  - 适合第三章外围、第四章岛外这种环状或边缘巡逻

### 推荐配置方式

- 出生点支持两层配置：
  - `patrolPreset`
  - `patrol` 覆盖项

```lua
{
    type = "trail_boss_spider",
    x = 66, y = 39,
    patrolPreset = "large_waypoint_loop",
    patrol = {
        regionKey = "narrow_trail",
        nodes = {
            { x = 58, y = 34 },
            { x = 64, y = 35 },
            { x = 71, y = 34 },
            { x = 75, y = 39 },
            { x = 69, y = 45 },
            { x = 60, y = 43 },
        },
        pauseMin = 0.8,
        pauseMax = 1.6,
        returnMode = "resume_patrol",
    }
}
```

- 含义：
  - `preset` 提供默认算法和默认参数
  - `patrol` 只覆盖该 Boss 的区域、路点、停顿时间等实例差异

### 推荐最终形态

- 后续新增巡逻 Boss 时，优先只做三件事：
  - 在 `MonsterTypes` 定义战斗与掉落
  - 在 `zone spawns` 指定出生点
  - 选择一个 `patrolPreset` 并填最少量覆盖参数
- 不建议每新增一只巡逻 Boss，就再往 `Monster.lua` 里加分支。

### 现状限制

- [Monster.lua](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/entities/Monster.lua:89>) 里，巡逻能力只有一个 `patrolRadius`。
- [Monster.lua](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/entities/Monster.lua:517>) 里的现有逻辑，本质上是“以出生点为圆心，随机找一个半径内目标点，再直线走过去”。
- [Spawner.lua](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/world/Spawner.lua:64>) 当前只把 `怪物类型 + 出生坐标` 传入怪物实例，区域级巡逻参数不会进到 Monster。
- [main.lua](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/main.lua:108>) 与 [main.lua](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/main.lua:1298>) 当前对远处 `idle/patrol` 怪物做了 `0.5s` 降频更新。

### 为什么不建议直接把 `patrolRadius` 调大

- 只能表达“围绕出生点画圆”，不能表达“在指定区域内巡逻”。
- 羊肠小径和山贼主寨都不是规则圆形，用大半径会把 Boss 引到边界、障碍堆、门洞等低质量位置。
- 现有巡逻目标点没有区域约束，也没有路线层次，Boss 会显得像随机乱走，不像“巡逻”。
- 大范围随机点如果只靠 `MoveToward + IsWalkable` 硬顶，容易在障碍附近来回抖动。
- 对远处怪物虽然已有降频，但如果巡逻目标本身质量低，降频后只会把“乱走”放大成更明显的卡顿感。

### 推荐做法

- 不全局修改 `boss = 0` 的默认巡逻。
- 不把区域巡逻参数写死在 `MonsterTypes_ch1.lua`。
- 将巡逻配置挂在 `zone spawns` 上，由出生点决定“这只 Boss 在哪一块区域怎么巡逻”。
- 采用“区域巡逻锚点”方案，而不是“超大随机半径”方案。
- 同时在配置层引入 `preset + override` 结构，让巡逻 Boss 成为可复制机制。

### 数据结构设计

- 在出生点上新增可选 `patrol` 配置：

```lua
{
    type = "trail_boss_xxx",
    x = 66, y = 39,
    patrolPreset = "large_region_roam",
    patrol = {
        mode = "region_roam",      -- 或 "waypoint_loop"
        regionKey = "narrow_trail",
        nodeCount = 6,
        anchorMinDist = 4,
        pauseMin = 0.8,
        pauseMax = 1.8,
        arrivalDist = 0.45,
        farThinkInterval = 3.0,
        nearRepickInterval = 0.0,
        returnMode = "resume_patrol",
    }
}
```

- `mode = "region_roam"`：
  - 在指定区域内预生成若干巡逻锚点。
  - Boss 在锚点之间轮转或随机切换。
- `mode = "waypoint_loop"`：
  - 直接写死一串路点。
  - 更适合山贼主寨这类希望 Boss 绕寨门、篝火、中心空地巡逻的区域。
- `regionKey` 直接引用 `ZoneData` 里的区域键，不重复维护坐标矩形。
- `patrolPreset` 负责提供默认参数；实例只覆盖自己真正不同的部分。

### 初始化流程

1. `Spawner` 创建怪物实例。
2. 如果 `sp.patrol` 存在，则把它挂到 `monster` 上。
3. 如果是 `region_roam`：
   - 根据 `regionKey` 取到区域矩形。
   - 在矩形内采样 `nodeCount` 个候选点。
   - 采样时先随机，再用 `Spawner` 现有的“最近可通行点修正”思路做 walkable 校正。
   - 如果新点和现有锚点太近，小于 `anchorMinDist`，则重抽。
4. 如果是 `waypoint_loop`：
   - 直接使用配置里的路点列表。
5. 把最终锚点列表缓存到 `monster.patrolNodes`，后续巡逻不再重新扫描区域。

### 运行时状态机

- `idle`
  - 进入等待。
  - 等到 `pauseMin~pauseMax` 的随机停顿时间后，选择下一个锚点。
- `patrol`
  - 向当前目标锚点移动。
  - 到达后回到 `idle`。
- `chase / attack`
  - 维持现有逻辑，不改仇恨和战斗流。
- `return`
  - 对巡逻 Boss 不建议直接回出生点。
  - 推荐 `returnMode = "resume_patrol"`：
    - 记录开战前或脱战时最近的巡逻锚点。
    - 脱战后先回该锚点。
    - 到达后继续正常巡逻。

### 远处怪物与性能约束

- 不建议给巡逻 Boss 上完整寻路系统。第一章只有极少数巡逻 Boss，做全图寻路性价比低。
- 直接复用当前 [main.lua](</C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/main.lua:1298>) 的“远处 `idle/patrol` 降频更新”即可，不要额外改主循环框架。
- 但在 `Monster` 内部需要再做一层轻量约束：
  - 远处时只保留“目标锚点选择”和“低频位移推进”。
  - 不在远处反复重抽随机点。
  - 不在远处重复扫描区域。
- 推荐参数：
  - 近处巡逻：沿用现有逐帧移动。
  - 远处巡逻：依然由主循环 `0.5s` 触发一次更新，但 Boss 只在 `farThinkInterval` 到点时才切换锚点。
- 这样做的收益是：
  - 不新增全局调度器。
  - 不打破现有性能优化。
  - Boss 靠近玩家时仍是连续移动，远离玩家时则退化为低频路线推进。

### 为什么“区域锚点巡逻”更有价值

- 它能精确控制 Boss 出现在“玩家真的会经过”的高价值位置。
- 它不会把 Boss 送进不可读的边角。
- 它能让同一套 AI 同时服务于：
  - 羊肠小径首个引导 Boss
  - 山贼主寨二大王
  - 后续第二章、第三章、第四章的巡逻 Boss
- 它的代码侵入面很小：
  - `Spawner` 负责把区域配置传进去
  - `Monster` 负责解释配置
  - `Zone spawns` 负责写实际巡逻覆盖区
  - `MonsterTypes` 只负责战斗属性和掉落

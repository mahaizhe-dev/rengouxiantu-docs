## 一、设计结论

第六章新增 **两界阵石** 系统：

- 中央两界村围墙外四角各放置 1 根可交互阵法石柱。
- 四根阵石分别培养攻击、防御、生命上限、生命回复。
- 每根阵石使用 `谪仙令` 激活，激活后从 `Lv.0` 培养至 `Lv.30`。
- 阵石只提供永久平面属性，不附带传送、开门、解封或 BOSS 房间功能。
- 四根阵石使用共享 PNG 石柱素材，运行时用不同阵纹颜色和激活光效区分。
- 四根阵石均配置 NPC 姓名板：未激活显示 `未激活`，培养中显示 `Lv.N`，满级显示 `MAX`。
- 激活光效强度固定，不随等级变化；等级反馈只通过姓名板和交互面板表达。
- 每根阵石首次激活一次性获得 `1000万经验`，奖励与激活状态同事务保存、不可重复领取。
- 单根满级消耗 `50500` 谪仙令，四根全满消耗 `202000` 谪仙令。
- 不新增普通、王级、皇级、圣级 BOSS 的固定谪仙令直掉数量；只将皇级及以上稀有池调整为整池 `6%`，并加入 `谪仙令盒`。
- 规范等级档案是永久属性的唯一真值；客户端提交的聚合属性只能作为缓存，读档与服务端写档都必须重算覆盖。

### 2.3 两套系统的共同骨架

```text
配置定义
  ↓
四个稳定 ID
  ↓
未激活/未解锁
  ↓ 消耗章节令牌
Lv.0 已激活
  ↓ 逐级消耗章节令牌
Lv.N 永久属性
  ↓
四属性聚合到 Player
  ↓
存档 + 角色信息页展示
```

两界阵石应继承这套骨架，但采用以下组合：

- 地图形态采用海神柱：四个独立对象、四个稳定位置。
- UI 信息结构采用海神柱单对象详情。
- 等级按第六章需求扩展到 30 级。
- 保存事务采用青莲长生体的强保存与失败回滚，不继续沿用旧系统只发 `save_request` 的弱确认方式。

### 2.4 三系统实现复核结论

| 关键链路 | 海神柱 | 祀剑池 | 两界阵石 | 结论 |
|----------|--------|--------|----------|------|
| 规范档案 | `seaPillar[id].repaired + level` | `swordPool[id].unlocked + level` | `liangjieStones.stones[id].activated + level + activationRewardClaimed` | 均以对象状态和等级为唯一真值 |
| 缺档处理 | 初始化四柱并把派生属性清零 | 初始化四剑并把派生属性清零 | 初始化四石并把派生属性清零 | 一致 |
| 非法等级 | 客户端读档钳制；服务端再次规范化 | 客户端读档钳制；服务端再次规范化 | 客户端读档钳制；服务端再次规范化 | 一致 |
| 未激活等级 | `repaired=false` 强制 0 级 | `unlocked=false` 强制 0 级 | `activated=false` 强制 0 级 | 一致 |
| 未知 ID | 客户端不读取；服务端丢弃 | 客户端不读取；服务端丢弃 | 客户端不读取；服务端丢弃 | 一致 |
| 客户端属性 | 读档后重算四个 `seaPillar*` 字段 | 读档后重算四个 `swordPool*` 字段 | 读档后重算四个 `liangjieStone*` 字段 | 一致 |
| 服务端属性 | `SaveWriteService` 按等级覆盖派生值 | `SaveWriteService` 按等级覆盖派生值 | `SaveWriteService` 按等级覆盖派生值 | 一致 |
| 最终公式 | 四属性进入 `Player` 总属性 | 同左 | 同左 | 一致 |
| 操作保存 | `save_request`，3 秒防抖落盘 | `save_request`，3 秒防抖落盘 | `SaveSystem.Save(callback)` 强保存 | 阵石保留更强方案 |
| 保存失败 | 旧系统无本次操作回滚 | 旧系统无本次操作回滚 | 回滚令牌、档案、经验、基础属性、称号、技能和派生属性 | 阵石保留更强方案 |

复核后的明确决策：

1. **属性与档案规则完全一致**：三套系统都坚持“等级档案是唯一真值，派生属性只能重算”。
2. **两界阵石不退回弱保存**：阵石激活包含 `1000万经验`，强保存成功确认和失败回滚具有确定收益。
3. **存档结构保留嵌套版本块**：`version + stones` 比旧系统平铺结构更利于后续迁移，不为表面一致而降级。
4. **保存超时由持有 requestId 的网络层先结算**：`CloudStorage` 在 30 秒触发原失败回调；`SaveSystem` 仅在 35 秒做极端兜底，避免提前重试导致业务回调丢失、阵石永久卡在保存中。
5. **经验回滚覆盖升级副作用**：除 `level / exp / maxHp / atk / def / hpRegen / hp` 外，同时恢复称号和技能解锁状态；青莲长生体采用同一回滚边界。

## 三、两界阵石正式设定

### 3.1 定位

两界阵石是第六章的长期令牌消耗与固定属性成长线，用来填补当前 `T11 / Lv.140` 缺少章节专属常规平面养成的问题。

系统不承担以下功能：

- 不解开中央两界村四门封印。
- 不解开虎王试炼封印。
- 不传送玩家。
- 不召唤怪物或开放 BOSS 房间。
- 不参与主线任务完成判定。

玩家进入第六章后即可看到并交互，是否完成 `shadow_village_zone` 不影响阵石激活和升级。

### 3.2 四根阵石与地图档案

中央两界村区域为 `x=33~48, y=33~48`。四根阵石放在主城三格清空缓冲区内，避开四门道路和现有村外围怪物点。

| 稳定对象 ID | 名称 | 内部 ID | 位置 | 坐标 | 属性 |
|-------------|------|---------|------|------|------|
| `ch6_liangjie_stone_xuanbi` | 两界阵石·玄壁 | `xuanbi` | 西北角外侧 | `(32, 32)` | 防御 |
| `ch6_liangjie_stone_tianfeng` | 两界阵石·天锋 | `tianfeng` | 东北角外侧 | `(49, 32)` | 攻击 |
| `ch6_liangjie_stone_houtu` | 两界阵石·厚土 | `houtu` | 西南角外侧 | `(32, 49)` | 生命上限 |
| `ch6_liangjie_stone_huiyuan` | 两界阵石·回元 | `huiyuan` | 东南角外侧 | `(49, 49)` | 生命回复 |

地图配置统一使用：

```lua

isObject = true
decorationType = "liangjie_stone"
```

坐标属于设计锁定值。代码落地时必须通过第六章地图合同测试检查可达性、地形阻挡、怪物重叠和四门通路；如确有冲突，只允许在对应角落内移动不超过 1 格，稳定对象 ID 不变。

### 3.3 PNG 与场景表现

最低资源需求为 1 张共享透明背景 PNG：

```text

```

建议规格：

| 项目 | 要求 |

状态表现：

| 状态 | 表现 |

## 四、激活、等级与属性

### 4.1 状态机

```text
未激活
  ↓ 首次交互
激活确认弹窗（参考天青白莲）
  ├─ 取消：关闭弹窗，不扣除、不发奖
  └─ 确认：消耗 1000 谪仙令 + 一次性获得 1000万经验
已激活 Lv.0
  ↓ 后续交互打开养成面板，消耗目标等级费用
Lv.1 ... Lv.30
  ↓
满级，不再显示升级按钮
```

规则：

1. 每根阵石独立激活、独立升级。
2. 激活成功后等级为 `0`，不提供属性。
3. 属性只按当前等级计算：`bonusPerLevel × level`。
4. 重复激活不得再次扣令牌。
5. 等级不得低于 `0` 或高于 `30`。
6. 四根阵石没有互相前置，玩家可以优先培养任意一项。
7. 每根阵石首次激活一次性发放 `10000000` 角色经验，发放规则与天青白莲一致。
8. 激活状态、经验奖励防重标记、令牌扣除、经验与可能发生的升级结果必须原子保存；保存失败全部回滚。
9. 未激活阵石首次交互只打开激活确认弹窗，不直接打开养成面板，也不在弹窗打开时预扣资源。
10. 激活保存成功后，玩家后续再交互该阵石时直接进入常规养成面板。

### 4.2 属性定稿

| 阵石 | 属性字段 | 每级加成 | 10级 | 20级 | 30级 |
|------|----------|---------:|-----:|-----:|-----:|
| 天锋 | `atk` | 5 | 50 | 100 | 150 |
| 玄壁 | `def` | 4 | 40 | 80 | 120 |
| 厚土 | `maxHp` | 50 | 500 | 1000 | 1500 |
| 回元 | `hpRegen` | 5 | 50 | 100 | 150 |

回元阵石定稿为生命回复 `+5/级`，与祀剑池回复档一致，30 级满额 `+150`。

四根阵石全满后新增固定属性：

```text
攻击 +150
防御 +120
生命上限 +1500
生命回复 +150
```

按当前 `T11` 固定构成快照观察，这组增量属于章节级固定大副轴，但仍低于装备、丹药、图鉴和角色成长等主轴总和。

## 五、谪仙令消耗

### 5.2 30级升级价格

升级价格沿用海神柱 / 祀剑池的 10 档价格，每档覆盖连续 3 级：

| 目标等级 | 单级消耗 |

| Lv.4~6 | 750 |
| Lv.7~9 | 1000 |
| Lv.10~12 | 1250 |
| Lv.13~15 | 1500 |
| Lv.16~18 | 1750 |
| Lv.19~21 | 2000 |
| Lv.22~24 | 2250 |
| Lv.25~27 | 2500 |
| Lv.28~30 | 3000 |

配置函数口径：

```lua
local band = math.ceil(targetLevel / 3)
return UPGRADE_COSTS[band]
```

其中 `UPGRADE_COSTS` 直接沿用：

```lua
{ 500, 750, 1000, 1250, 1500, 1750, 2000, 2250, 2500, 3000 }
```

### 5.3 总消耗

| 范围 | 激活 | 升级 | 合计 |

该结构允许玩家先把最需要的单项培养至满级，再决定是否追求四石圆满。

## 六、谪仙令产出配套

### 6.1 第六章主世界池

当前第六章所有 BOSS 都会触发一次 `ch6` 世界掉落池，并从以下 5 项中等权抽取 1 项：

- 龙髓
- 修炼果
- 灵韵果
- 谪仙令
- 上品修炼果

因此当前谪仙令的理论产出是：

```text
每次第六章 BOSS 击杀，20% 概率获得 1 枚谪仙令
```

## 七、UI 与交互

### 7.1 首次激活弹窗

未激活阵石首次交互时，参考 `QinglianBodySystem.Show()` 的天青白莲确认界面，单独弹出激活确认弹窗，不显示升级控件。

弹窗必须显示：

- 阵石名称与属性方向。
- 激活说明。
- 激活消耗：`1000 谪仙令`。
- 当前持有谪仙令数量，以及是否满足条件。
- 一次性奖励：`1000万经验`。
- `取消` 与 `激活阵石` 两个按钮。

交互规则：

1. 点击遮罩、关闭或取消只关闭弹窗，不扣令牌、不发经验、不写档。
2. 条件不足时激活按钮禁用，并明确显示缺少的谪仙令数量。
3. 点击激活后锁定按钮，进入保存中状态，不能连续提交。
4. 保存成功后显示激活成功弹窗，更新姓名板为 `Lv.0`。
   - 弹窗明确显示“经验奖励已写入角色存档”。
   - 显示本次实际增加的经验；若触发升级，同时显示升级前后等级。
   - 实际值按 `Player:GainExp` 结果计算，可正确反映称号经验加成与境界经验上限。
5. 保存失败显示错误弹窗并完整回滚；玩家可重新交互再次尝试。
6. 已激活阵石不再显示该弹窗，后续交互直接打开养成面板。

### 7.2 单阵石养成面板

已激活后的交互面板参考 `SeaPillarUI`，显示：

- 阵石名称与 PNG 图标。
- 当前可用谪仙令数量。
- 未激活 / 当前等级 / 满级状态。
- 当前属性加成。
- 下一等级加成与消耗。
- `升级1级` 按钮。

当前版本只提供单级升级，保证每次扣令牌、改等级和强保存是一笔明确事务。`本档升满` 属于后续可选扩展；若实施，目标语义为一次升级到当前 3 级价格档的末级，例如：

- `Lv.4 → Lv.6`
- `Lv.17 → Lv.18`
- `Lv.28 → Lv.30`

后续批量升级若实施，必须：

1. 先计算完整总成本。
2. 一次校验可用谪仙令。
3. 一次扣除。
4. 一次修改等级。
5. 一次强保存。
6. 保存失败时整体回滚，不能只回滚最后一级。

### 7.3 总进度（后续扩展）

## 八、存档与多人模式边界

项目当前 `.project/settings.json` 为多人模式：

```text
@runtime.multiplayer.enabled = true
```

两界阵石仍属于角色永久存档，不属于副本局内同步状态。保存继续走现有 `SaveSystem / SavePersistence / SaveWriteService` 链路，不新增场景复制节点或局内远程事件。

### 8.1 规范存档

当前顶层字段：

```lua
liangjieStones = {
    version = 1,
    stones = {
        xuanbi = { activated = true, level = 12, activationRewardClaimed = true },
        tianfeng = { activated = false, level = 0, activationRewardClaimed = false },
        houtu = { activated = true, level = 30, activationRewardClaimed = true },
        huiyuan = { activated = true, level = 8, activationRewardClaimed = true },
    },
}
```

规范状态只认 `liangjieStones.stones[*].activated / level / activationRewardClaimed`。其中 `activationRewardClaimed` 与首次激活在同一事务内置为 `true`，用于明确防止经验奖励重复发放。

聚合属性字段只是派生缓存；为兼容当前 `Player` 序列化结构可以继续写入角色数据，但不能作为真值：

```text
liangjieStoneAtk
liangjieStoneDef
liangjieStoneMaxHp
liangjieStoneHpRegen
```

读档后必须根据四根阵石规范等级重新计算并覆盖聚合字段，不能信任客户端存档里单独提交的聚合值。服务端写档校验也必须执行同一公式，保证客户端、云档和线上玩家迁移结果一致。

存档优先级：

1. `liangjieStones.stones[*]` 是唯一真值。
2. `Player` 上四个 `liangjieStone*` 字段只能由规范等级重算。
3. 客户端提交的派生值、旧缓存值、负数、超上限值都不能影响最终属性。
4. 旧档缺少整个字段时，四根阵石安全默认为未激活、0 级、奖励未领取。
5. `activated=false` 时强制 `level=0` 且 `activationRewardClaimed=false`。
6. `activated=true` 但旧档缺少奖励字段时，迁移为 `activationRewardClaimed=true`，避免补发 1000万经验。

### 8.2 保存事务

激活、单级升级统一采用：

```text
检查存档可用
  ↓
建立令牌、阵石状态、角色经验/等级和派生属性快照
  ↓
扣除谪仙令
  ↓
修改 activated / level / activationRewardClaimed
  ↓
首次激活发放 1000万经验并处理可能的连续升级
  ↓
重算四属性并刷新缓存
  ↓
强保存
  ├─ 成功：确认结果
  └─ 失败：回滚令牌、激活/等级/奖励标记、经验/等级、聚合属性
```

保存过程中锁定该阵石操作按钮，防止连续点击重复提交。

### 8.3 服务端校验

`SaveWriteService` 至少校验：

- 只接受 `xuanbi / tianfeng / houtu / huiyuan` 四个 ID。
- `activated` 必须为布尔值。
- `level` 必须为整数并限制在 `0~30`。
- `activationRewardClaimed` 必须为布尔值。
- `activated = false` 时强制 `level = 0`。
- 根据合法等级重算四个聚合属性。
- 聚合上限为 `ATK 150 / DEF 120 / MaxHP 1500 / HPRegen 150`。
- 不接受客户端通过修改聚合字段增加属性；服务端必须覆盖而不是只做上限钳制。
- 未知字段只记录日志，不参与属性计算。

### 8.4 旧档与异常档迁移

| 输入档案 | 规范化结果 |
|----------|------------|
| 缺少 `liangjieStones` | 四石未激活、0 级、奖励未领取、派生属性全 0 |
| 已激活但缺少 `activationRewardClaimed` | 置为 `true`，不补发经验 |
| 未激活但等级大于 0 | 强制等级 0 |
| 等级为小数、负数或大于 30 | 取整数并钳制到 `0~30` |
| 未知阵石 ID | 丢弃并记录日志 |
| 派生字段与等级不一致 | 忽略派生字段，按规范等级覆盖 |
| 玄冰海神柱派生值与等级不一致 | 按 `玄冰等级 ×2` 重算，10 级固定为 20 |
| 陷仙剑封印旧满级派生防御为 50 | 按 `陷仙等级 ×4` 重算，10 级覆盖为 40 |

### 8.5 需要接入的保存文件

- `scripts/systems/save/SaveDTO.lua`
- `scripts/systems/save/SavePersistence.lua`
- `scripts/systems/save/SaveLoader.lua`
- `scripts/systems/save/SaveSerializer.lua`
- `scripts/network/SaveWriteService.lua`

不要只在 `SavePersistence` 写入字段而遗漏 `SaveDTO` 边界登记。

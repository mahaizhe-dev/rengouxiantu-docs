# 阵营法宝挑战系统执行文档

> 用途：阵营法宝挑战系统下阶段执行稿。  
> 更新时间：2026-05-26。  
> 范围：旧挑战退役评估与解耦、额外掉落优化、`T10 / R8` 法宝挑战接入。  
> 原则：**新系统继续扩展，旧系统只做兼容隔离，不做高风险硬删。**  
> 参考：`阵营法宝挑战系统文档（代码基准版）.md`、`阵营法宝挑战额外掉落分析.md`

## 一、执行结论

本轮建议按下面顺序推进：

1. 先做 **旧挑战解耦与兼容隔离**
2. 再做 **额外掉落重构**
3. 最后接入 **`R8 / T10` 法宝挑战**

本轮已经明确的执行口径如下：

- **旧挑战不建议本轮物理删除**
  - 性能收益很小
  - 存档/迁移耦合仍然很深
  - 如果目标是“绝对可靠”，正确做法是 **运行时退役 + 兼容层保留**
- **额外掉落改成第 3 个候选项**
  - 平时仍是两件法宝二选一
  - 命中额外掉落时，变成 **两件法宝 + 一件额外道具 三选一**
  - 阵营 Boss 本身没有基础材料掉率，所以额外掉落就是挑战内唯一材料出口
- **额外掉落不再走等权池**
  - 改成 **共享 1 个位置，但按单项显式概率配置**
  - 每个物项直接写最终出现率和数量
- **`T10` 挑战直接接入新版声望体系**
  - 即新增 `R8 / T10`
  - 解锁货币使用 `taixu_jianling`
  - 同步接入第 6 种精华 `ganggu_essence` 与 `钢骨丹`
  - `钢骨丹` 在**中洲城炼制**

## 二、事项 1：移除废弃旧挑战

### 2.1 是否有必要

| 维度 | 结论 | 说明 |
|------|------|------|
| 玩法必要性 | 高 | 当前主流程已经完全是新版 `reputation`，旧挑战只剩兼容意义 |
| 维护必要性 | 高 | 旧 `tiers / progress / 奖励 / 怪物 / 成就` 仍混在主模块里，持续增加认知负担 |
| 性能收益 | 低 | 删除后对帧率几乎无感，主要只是少量配置、分支、序列化字段 |
| 数据风险 | 很高 | 旧进度和旧丹药仍处在存档迁移链路里，硬删容易破坏老档 |

结论：

- **要退役，但不建议本轮一步物理删除**
- 本轮目标应是：**旧挑战从运行时主逻辑里退出，只保留存档兼容层**

### 2.2 当前耦合点清单

| 层级 | 具体耦合 | 文件 |
|------|----------|------|
| 配置 | `tiers` 旧挑战配置、`legacyNpcId` | `scripts/config/ChallengeConfig.lua` |
| 怪物 | `challenge_shenmo_t1~t8`、`challenge_luqingyun_t1~t8` | `scripts/config/MonsterTypes_ch3.lua` |
| 旧装备 | `blood_sea_lv1~lv3`、`haoqi_seal_lv1~lv3` | `scripts/config/EquipmentData.lua` |
| 旧丹药 | `xuesha_dan`、`haoqi_dan` | `scripts/systems/ChallengeSystem.lua` |
| 旧进度 | `progress[faction][tier]` | `scripts/systems/ChallengeSystem.lua` |
| 旧玩法 API | `Unlock(faction, tier)`、`Enter(faction, tier)`、`GrantReward()` | `scripts/systems/ChallengeSystem.lua` |
| UI 旧分支 | 旧版 `tier` 交互、旧版胜利弹窗确认逻辑 | `scripts/ui/ChallengeUI.lua` |
| 成就 | `young_hero` 仍引用 `challenge_shenmo_t4 / challenge_luqingyun_t4` | `scripts/config/TitleData.lua` |
| 存档迁移 | `data.progress`、`xueshaDanCount`、`haoqiDanCount` 仍参与迁移/保底恢复 | `scripts/systems/ChallengeSystem.lua`、`scripts/systems/save/*` |
| 测试 | 当前测试主要覆盖新版，但迁移链仍默认旧字段存在 | `scripts/tests/test_challenge_fabao.lua` |

### 2.3 绝对可靠的退役方式

如果要求“没有任何耦合，绝对可靠”，本轮不应追求“零旧代码”，而应追求：

- **运行时零依赖旧挑战**
- **老存档仍能正常加载**
- **旧丹药/旧进度迁移仍能生效**

因此推荐拆成两步：

### 第一步：本轮执行

- 将旧挑战从主玩法链路中彻底隔离
- 保留旧挑战读取与迁移能力
- 旧 `progress` 不再参与新功能扩展
- 旧 `tier` UI 和旧奖励逻辑不再保留可达入口
- 旧成就暂不处理

### 第二步：观察一版后再决定

- 如果确认不再需要支持更老存档，才考虑删除旧怪物定义、旧装备模板和旧序列化字段
- 如果仍要求支持历史存档，则旧兼容层应长期保留，不建议再删

### 2.4 本轮具体执行

### A. 运行时解耦

- `ChallengeUI.lua`
  - 删除或封存旧 `tier` 按钮点击分支
  - 胜利弹窗只保留新版声望奖励路径
- `ChallengeSystem.lua`
  - 新版 API 保持主入口
  - 旧 `Unlock / Enter / GrantReward` 保留到兼容区，不再由 UI 调用

### B. 存档兼容隔离

- 将旧 `progress`、旧丹药迁移逻辑明确视为 `legacy compatibility`
- 推荐把旧迁移逻辑收拢到单独兼容段或兼容模块
- `Serialize()` 可以继续带旧字段 1 个版本，避免回滚时出现兼容断层

### C. 成就处理

当前 `young_hero` 使用旧怪物 ID，但本轮**暂不处理**。  

执行口径：

- 允许该称号暂时保持死成就状态
- 本轮不迁移条件
- 本轮不归档称号
- 等旧挑战兼容层处理稳定后，再统一决定是否迁移或清理旧成就

### 2.5 性能评估

| 项目 | 预期收益 |
|------|----------|
| 帧率 | 基本无变化 |
| 内存 | 小幅下降，仅限旧怪物/旧配置常驻表 |
| 存档体积 | 小幅下降，主要来自旧 `progress` 写入减少 |
| 维护成本 | 明显下降 |

结论：

- 旧挑战退役的价值主要在 **维护性和正确性**
- 不是性能优化项，不应以“提性能”为主要立项理由

### 2.6 验收口径

- 新版玩法全链路不再调用旧 `tier` API
- 老存档仍可正常加载并保留已有迁移结果
- `rg` 检索旧挑战标识后，只允许残留在兼容层、历史资源或明确保留的模板中

## 三、事项 2：额外掉落优化

### 3.1 本轮目标

本轮目标已经明确：

1. **额外掉落改为第 3 个候选项**
2. **额外掉落共享 1 个位置，但不再等概率**
3. **阵营 Boss 没有基础材料掉率，所有材料都从额外掉落口径进入**
4. **非精华类采用固定概率，主要靠数量分档**
5. **精华线单独定档，`R8` 新增第 6 种精华**
6. **角色信息面板同步更新**
   - 新增 `钢骨丹` 展示
   - 福缘果数量上限改为动态读取，不再写死

### 3.2 结构改法

当前重复通关奖励是：

- 选项 A：随机法宝
- 选项 B：随机法宝
- 如果命中额外掉落：当前代码会把选项 B 替换成道具

本轮改成：

- 未命中额外掉落：仍然是两件法宝二选一
- 命中额外掉落：变成 **法宝 A / 法宝 B / 额外掉落 三选一**
- 额外掉落仍然只占 **1 个共享位置**
- 共享位置内部按 **显式单项概率表** 决定最终显示哪 1 个物品

这意味着：

- 玩家不会因为刷材料而丢掉第二件法宝候选
- 额外掉落仍然是机会奖励，不是直接入包
- 文案必须从“额外掉落替换法宝”改成“额外掉落追加候选”

## 四、事项 3：增加 T10 法宝挑战

### 4.1 目标定义

本轮按下面口径落地：

| 项目 | 设定 |
|------|------|
| 新等级 | `R8 / T10` |
| 解锁货币 | `taixu_jianling`（太虚剑令） |
| 解锁费用 | `3000` |
| 入场费用 | 继续沿用 `ATTEMPT_COST = 10`，货币同样为 `taixu_jianling` |
| Boss 等级 | `120` |
| Boss 境界 | `dujie_1`（谪仙初期） |
| Boss 类别 | `saint_boss` |
| 首通奖励 | 两件 `T10 cyan` 法宝二选一 |
| 重复奖励 | 平时两件 `T10` 法宝二选一；命中额外掉落时，变成两法宝 + 一额外道具三选一 |

### 4.2 当前底层支持情况

`T10` 不是从零开始，当前代码已经具备以下基础：

| 能力 | 当前状态 |
|------|----------|
| `tier=10` 装备系统 | 已支持 |
| `T10` 属性倍率 | 已支持 |
| `T10` 品质表 | 已支持 |
| `taixu_jianling` 物品 | 已存在 |
| `dujie_1~dujie_4` 境界 | 已存在 |
| `saint_boss` 类别 | 已存在 |
| 动态法宝生成 | 已支持 `tier=10`，但阵营法宝模板未补 `iconByTier[10] / sellPriceByTier[10]` |

### 4.3 本轮必须改的点

### A. 声望结构扩容

当前新版写死了大量 `1~7`：

- `ChallengeConfig.REPUTATION_*`
- `ChallengeUI.lua` 中 `for repLevel = 1, 7`
- `ChallengeSystem.lua` 的注释、断言、初始化逻辑
- `test_challenge_fabao.lua` 中 `R1~R7` 断言

本轮必须统一改成：

- **配置驱动**
- **动态读取最大声望等级**

### B. 四阵营新增 `R8` Boss

新增：

- `challenge_shenmo_r8`
- `challenge_luqingyun_r8`
- `challenge_yunshang_r8`
- `challenge_liwuji_r8`

建议规则：

- 复用 `R7` 的技能骨架
- 继续三阶段
- 保留狂暴
- 只增强专属机制参数，不在本轮扩第四阶段

### C. 阵营法宝模板补 `T10`

四件阵营法宝模板需要补：

- `iconByTier[10]`
- `sellPriceByTier[10]`

建议本轮直接复用现有最高档图标资源，不阻塞功能上线。

### D. `T10` 品质上限

当前 `LootSystem.CreateFabaoEquipment()` 对 `tier >= 9` 的法宝上限仍是 `cyan`。  
本轮建议改成：

- `T3~T4 -> purple`
- `T5~T8 -> orange`
- `T9 -> cyan`
- `T10 -> red`

这样 `T10` 才是真正的高一档。

### 4.4 `R8` 额外掉落并入方案

`R8` 需要与新挑战同步接入以下内容：

- `gold_bar` 固定 `1.0%`，数量 `×8`

## 六、一句话结论

这三件事里，真正高风险的是 **旧挑战硬删除**，真正高价值的是 **额外掉落从“替换法宝”改成“第 3 候选”并落成显式概率表**，真正能拉开版本体感的是 **`R8 / T10 / 太虚剑令 / 谪仙圣级 Boss / 钢骨丹`**。  
因此本轮正确做法是：**旧挑战只做兼容隔离，不做硬删；额外掉落改成三选一候选并按单项概率配置；`T10` 直接按新版声望扩容落地。**

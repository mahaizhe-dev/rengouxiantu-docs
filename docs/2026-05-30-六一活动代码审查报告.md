## 3. 主要问题

### [P0] 大宝箱两种奖励道具 ID 写错，客户端回放时会丢奖励

位置：

- `scripts/config/EventConfig.lua:127`
- `scripts/config/EventConfig.lua:135`
- `scripts/systems/InventorySystem.lua:596-599`
- `scripts/ui/EventExchangeUI.lua:1273-1278`
- `scripts/ui/EventExchangeUI.lua:1314-1315`

现状：

1. `big_exp_pill` 奖励写成了 `premium_exp_pill`
2. `big_ly_fruit` 奖励写成了 `premium_lingyun_fruit`
3. 实际配置表中的正确 ID 是：
   - `exp_pill_superior`
   - `lingyun_fruit_superior`

风险：

1. 服务端会把这两个错误 ID 当普通 `consumable` 发进存档背包
2. 客户端回包后调用 `InventorySystem.AddConsumable()` 时找不到配置，直接返回失败
3. UI 当前没有检查返回值，仍然继续 `SavePersistence.Save()`
4. 结果可能出现“服务端奖励已发，但客户端本地回放失败并触发即时保存覆盖”的风险

结论：

这是本轮最严重的问题，必须先修。

建议修复：

1. 把 `premium_exp_pill` 改成 `exp_pill_superior`
2. 把 `premium_lingyun_fruit` 改成 `lingyun_fruit_superior`
3. 客户端回放 `AddConsumable()` 后增加失败日志或断言，至少不能静默继续

### [P1] 超级大奖弹窗覆盖不完整，五连时更容易被漏掉

位置：

- `scripts/network/EventHandler.lua:494-502`
- `scripts/network/EventHandler.lua:529-555`
- `scripts/ui/EventExchangeUI.lua:91-216`
- `scripts/ui/EventExchangeUI.lua:225-300`
- `scripts/ui/EventExchangeUI.lua:1282-1308`

现状：

1. 服务端只有在“抽到未拥有的新皮肤”时才回 `UnlockedSkinId`
2. 若抽到重复皮肤，只会把补偿折进 `TotalLingYun`
3. 客户端弹窗判断逻辑只有两条：
   - 有 `UnlockedSkinId` -> 显示皮肤弹窗
   - `Results` 中存在 `small_upgrade` -> 显示传说物品弹窗

直接后果：

1. 大宝箱抽到重复皮肤时，不会触发任何“超级大奖”弹窗
2. 五连开里即使命中多个大奖，也不会按“大奖队列”逐个展示
3. 当前逻辑只对 `small_upgrade` 这一个小箱头奖做了硬编码，不具备通用性

为什么你会感觉“五连时尤其不触发”：

1. 五连比单开更容易出现“大奖不是第一优先分支”的情况
2. 当前 UI 没有“逐个 special reward 入队显示”机制
3. 只要本轮没有新皮肤、又没有 `small_upgrade`，就只会显示普通完成提示

建议修复：

1. 服务端回包新增结构化字段，例如：
   - `SpecialRewards`
   - 或 `PopupEvents`
2. 至少要区分：
   - 新皮肤解锁
   - 重复皮肤补偿
   - 小箱头奖 `childday_pinwheel`
3. 客户端按列表顺序弹窗，而不是只做单一 `if/else`
4. 不要继续只硬编码 `small_upgrade`

### [P1] 皮肤落库链路弱于商店实现，存在主存档成功但账号皮肤写失败的窗口

位置：

- `scripts/network/EventHandler.lua:427-504`
- `scripts/network/EventHandler.lua:521-557`
- `scripts/network/SkinShopHandler.lua:214-227`
- `scripts/network/SaveWriteService.lua:137-140`

对比五一：

1. 五一没有账号级皮肤奖励，不存在这类问题
2. 六一新增 `account_cosmetics` 侧写入，因此必须对比皮肤商店实现

当前六一做法：

1. 先提交 `save_{slot}` 和积分
2. 提交成功后，再异步读取/写入 `account_cosmetics`
3. 即使 `account_cosmetics` 写失败，也不会影响开箱成功回包

风险：

1. 玩家可能已经收到“开箱成功”与本地皮肤解锁表现
2. 但账号级持久化失败后，重登可能丢失皮肤
3. `account_cosmetics` 读取失败时，还会保守地把第一发皮肤当“新解锁”处理

建议修复：

1. 至少把 `account_cosmetics` 写失败打成高优先级错误日志
2. 最稳方案是对齐商店模式：
   - 主事务成功后
   - 必须确认 `account_cosmetics` 写成功
   - 再回最终成功
3. 如果当前平台限制不允许完全原子化，就需要明确补偿策略，而不是静默放行

## 5. 超级大奖弹窗问题专门结论

你提到的现象“超级大奖的弹窗似乎无法触发，尤其是 5 连的时候”，结论如下：

1. 这不是纯表现问题，确实是当前代码设计不完整
2. 当前弹窗只能稳定覆盖：
   - 新皮肤
   - 小箱 `small_upgrade`
3. 当前弹窗不能覆盖：
   - 重复皮肤补偿
   - 五连中多个大奖依次展示
   - 未来新增的其它大奖类型

因此：

1. 如果你把“重复皮肤补 10000 灵韵”也视为超级大奖，当前一定会漏弹窗
2. 如果五连里大奖不满足当前两个硬编码分支，也会表现成“没触发”

## 11. 最终结论

六一活动当前不是“整体不可用”，但还不适合无修正直接上线。

核心原因不是单一点故障，而是：

1. 存在一个明确的 `P0` 奖励发放错误
2. 存在多个 `P1` 级规则偏移与大奖表现缺口
3. 其中“超级大奖弹窗在五连时不稳定”是当前实现方式天然导致的，不是偶发现象

如果要以最小代价恢复上线质量，优先修：

1. 奖励 ID
2. 稀有掉落 BOSS 限制
3. 大奖弹窗语义
4. 四仙剑/境界掉率数值回调

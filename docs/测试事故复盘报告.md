## 一、背景：测试架构的根本缺陷

### 1.1 InventorySystem 的全局单例结构

```lua
-- InventorySystem.lua 顶层
local manager_ = nil  -- 全局单例，全局唯一

function InventorySystem.Init()
    -- 创建新的空 manager，替换全局 manager_
    local UI = require("urhox-libs/UI")
    manager_ = UI.InventoryManager.new({
        inventorySize = GameConfig.BACKPACK_SIZE,
        equipmentSlots = {},
    })
end
```

`manager_` 是模块级变量。整个游戏只有一个 `InventorySystem`，只有一个 `manager_`。
玩家的背包数据、装备数据，全部存放在这个 `manager_` 对象内。

### 1.2 TestRegistry 在真实引擎中执行测试

测试通过 `TestRegistry.lua` 在游戏启动时自动运行。`enabled=true` 的条目会被
`TestRunner` 在真实引擎环境中 `dofile` 执行。这意味着测试代码与玩家存档运行在
**同一个 Lua 虚拟机、同一份全局状态**中。

### 🔴 事故 #1（上次 session）：玩家背包第一次被清空

**触发时间**：2026-05，上次 session

**直接原因**：

`test_jiefeng_forge_pipeline.lua` 中的 `ResetInventory()` 函数调用了 `InventorySystem.Init()`：

```lua
-- 事故版本（已删除）
local function ResetInventory(size)
    InventorySystem.Init()         -- ← 致命调用
    return InventorySystem.GetManager()
end
```

`InventorySystem.Init()` 在真实引擎中执行时：
1. 创建空的 `InventoryManager`
2. 将其赋给全局 `manager_`
3. 原有的玩家背包数据（含所有物品和装备）从内存中消失

随后 auto-save 定时器（120秒）触发，将空背包数据写入云端存档，**玩家数据永久丢失**。

**该测试 TestRegistry 当时状态**：`enabled=true, mode="run_file", gate="blocking"`  
**游戏启动即运行，无任何保护。**

**上次 session 修复（不完整）**：

修复了6处 `RemoveInventoryItem` → `SetInventoryItem(slot, nil)`（实际 API 名称问题），
以及将 `ResetInventory()` 改为：

```lua
-- 不完整修复版（仍有 bug）
local function ResetInventory(size)
    local mgr = MakeManager(size)
    return mgr                     -- ← 只返回，没有调用 SetManager！
end
```

`InventorySystem` 内部的 `manager_` 仍为 `nil`（或上次 Init 后的状态）。

### 🔴 事故 #3（本 session）：玩家背包第二次被清空 + GetAllEquipment 崩溃

**触发时间**：2026-05-23，本 session 中期

**错误报告**（4条）：
```
[2026-05-23 13:17:52.537] [ERROR] systems/InventorySystem:485:
  attempt to call a nil value (method 'GetAllEquipment')
stack traceback:
  systems/InventorySystem:485: in field 'GetActiveSetBonuses'
  ui/InventoryUI:637: in field 'UpdateStats'
  ui/InventoryUI:540: in field 'Show'
  ui/InventoryUI:562: in field 'Toggle'
  main:1659: in function 'HandleKeyDown'
```

**根本原因**：

事故 #2 的修复引入了新的根本性错误。

`SetManager(mockMgr)` 替换全局 `manager_` 后：

| 阶段 | 发生了什么 |
|------|-----------|
| 测试运行中 | `manager_` = mock，mock 无 `GetAllEquipment` 方法 |
| 测试跑完 | `manager_` 仍然指向 mock，未恢复 |
| auto-save 触发（120s内） | SaveSerializer 序列化 mock 的空背包 → 写入云端 → **玩家数据再次永久丢失** |
| 玩家按键打开背包 | `InventoryUI.Show()` → `UpdateStats()` → `GetActiveSetBonuses()` → `manager_:GetAllEquipment()` → **nil 方法崩溃** |

mock manager 缺失的方法清单（与真实 `InventoryManager` 对比）：

| 方法 | 真实 InventoryManager | jiefeng mock | sword mock | bm mock |
|------|-----------------------|-------------|------------|---------|
| `GetAllEquipment` | ✅ 定义于 InventoryManager.lua:271 | ❌ 无 | ❌ 无 | ✅ 返回 `{}` |
| `GetInventorySize` | ✅ | ❌ 无 | ❌ 无 | 未确认 |
| `GetFreeSlotCount` | ✅ | ❌ 无 | ❌ 无 | 未确认 |

**结论**：`SetManager()` 本质上与 `Init()` 同等危险。  
区别在于：`Init()` 注入空 manager（无任何方法缺失），`SetManager(mock)` 注入残缺 mock（会立即崩溃），两者都会覆盖玩家存档。

## 四、最终修复方案

### 4.1 TestRegistry 禁用全部 4 个危险测试

将以下4个条目从 `enabled=true, mode="run_file", gate="blocking"` 改为：

```lua
enabled       = false,
mode          = "skip",
gate          = "non_blocking",
skip_reason   = "lupa_only: 使用 InventorySystem.SetManager() 注入 mock，
                  在真实引擎中会替换玩家背包并触发自动存档，
                  只能在离线 lupa 环境运行",
```

被禁用的测试：

| 测试 ID | 文件 |
|---------|------|
| `jiefeng_forge_pipeline` | `test_jiefeng_forge_pipeline.lua` |
| `sword_forge_comprehensive` | `test_sword_forge_comprehensive.lua` |
| `bm_warehouse_consistency_01a` | `test_black_merchant_warehouse_consistency.lua` |
| `bm_s2_unified_sync_gate` | `test_bm_s2_unified_sync_gate.lua` |

### 4.2 构建验证

## 五、当前仍存在的问题（未解决）

**错误时间**：2026-05-23 13:17:52（本 session 最新构建后）  
**版本**：`72ad264d (main@0688f017)`

仍然报 `GetAllEquipment` nil 崩溃。

**可能原因分析**：

1. **云端存档已损坏**（最大可能）：事故 #3 的 auto-save 将空背包写入云端。即使测试已禁用，下次游戏加载时从云端读取的数据已是空背包，`manager_` 加载的是空数据但对象本身类型正确，不会引起此崩溃。

2. **仍有其他未发现的注入点**：搜索结果显示以下位置调用了 `SetManager()`，均已被 TestRegistry 禁用，但需确认 `bm_s2_unified_sync_gate` 的第 316 行是否在 `enabled=false` 后确实不再执行。

3. **`InventorySystem.Init()` 未被调用**：若游戏启动时某路径跳过了 `Init()`，`manager_` 本身就是 `nil`。但第 481 行有 `if not manager_ then return {} end` 守卫，`nil` 时直接返回空表，不会触发"调用 nil 方法"错误。所以 `manager_` 一定不是 `nil`，而是一个缺少 `GetAllEquipment` 方法的对象。

4. **上一次构建的缓存**：用户测试的可能是事故 #3 发生时的旧构建，而不是最新禁用测试后的版本。

## 六、根因总结

| 事故 | 根本原因 | 后果 |
|------|---------|------|
| #1 | 测试代码调用 `InventorySystem.Init()`，在真实引擎中替换全局 manager_ | 玩家背包第一次被清空，数据永久丢失 |
| #2 | 修复 Init 时只 return manager 未注入，manager_ 为 nil | 测试7个 FAIL，不影响玩家数据 |
| #3 | 引入 `SetManager()` 修复 #2，但未意识到 mock 会留在全局，且 mock 缺少真实方法 | 玩家背包第二次被清空；GetAllEquipment 崩溃 |

**统一根因**：这4个测试文件设计为在离线 Python/lupa 环境运行，但被错误地注册到 TestRegistry 并在真实引擎中执行。测试所做的任何全局状态修改都与玩家数据共享同一个 Lua VM。

## 七、永久规则（已写入 rules.md）

```
🔴 测试文件（scripts/tests/）中绝对禁止：
  - 调用 InventorySystem.Init()（替换全局 manager_，触发存档丢失）
  - 调用 InventorySystem.SetManager(mock)（同等危险：注入残缺 mock，
    auto-save 触发后玩家背包被覆盖为空数据；mock 缺少真实方法导致崩溃）

凡使用 SetManager() 或任何操纵全局单例的测试文件，
必须在 TestRegistry.lua 中将对应条目标记为：
  enabled = false, mode = "skip", gate = "non_blocking"
  skip_reason = "lupa_only: ..."
这类测试只能在离线 Python/lupa 环境中运行，禁止在真实引擎中执行。
```

**当前已禁用的 lupa_only 测试**：
- `jiefeng_forge_pipeline`
- `sword_forge_comprehensive`
- `bm_warehouse_consistency_01a`
- `bm_s2_unified_sync_gate`

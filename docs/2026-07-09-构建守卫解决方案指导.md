## 二、真正的根因（三因叠加，非"模型不可控"）

1. **工具 schema 自我教唆**：`multiplayer` 参数 description 写 "REQUIRED on first build if no local config exists"——新会话无法确证本地配置存在，出于保险必带该参数。这是第一事故源。
2. **few-shot 锚定差异**：老对话上下文里有历史成功构建调用的 JSON，模型照抄自己；新对话只有 schema+文档。这完整解释"老对话正常、新对话失灵"——不是能力丧失，是上下文差异。
3. **deny 消息不可照抄**：exit 2 的 stderr 若是散文式劝告，模型在"schema 说 required"与"报错说别传"间反复横跳；7 次重试全带 multiplayer 说明纠正信号无效。

**"慢 2-3 倍"由重试次数直接解释**：每次阻断=一轮完整工具调用+反思；每次假成功=数分钟完整构建后再发现错。无需另找性能原因。

**触发事件待核实**："从昨天开始"高度可能对应某次版本/schema 变更（sce-urhox MCP 更新、Claude Code 更新、或新对话默认模型变更）——落地时顺手确认（§五-4）。

## 三、解法核心：方案 C 可行 —— guard 改"参数清洗器"

PreToolUse hook 通过 stdout 输出 JSON 可改写工具入参后放行：

```
读 stdin JSON 的 tool_input
→ stdout: {
    "hookSpecificOutput": {
      "hookEventName": "PreToolUse",
      "permissionDecision": "allow",
      "permissionDecisionReason": "build 参数已按 rules.md §6 清洗",
      "updatedInput": { "entry_client": "client_main.lua", "entry_server": "server_main.lua", "scriptsPath": "scripts" }
    }
  }
→ exit 0
```

**清洗器行为规格**（不阻断、只重写）：
- 白名单键：`entry_client`、`entry_server`、`scriptsPath`（缺失/空补 "scripts"）；
- 强制值：两个 entry 恒写死 `client_main.lua` / `server_main.lua`（传错也纠正，不阻断）；
- 黑名单键：`multiplayer`、`entry` **从 JSON 中整键删除**（不是置 null）；
- 永不 exit 2 → 新对话第一次调用即成功；时序正确（改写在工具执行/上传之前）；
- 现有 PostToolUse repair 保留为双保险。

**前提验证（5 分钟）**：在 /workspace 机器 `claude --version`，对照官方 hooks 文档确认该版本 PreToolUse 支持 `updatedInput`；不支持先升级。

## 四、分层方案

| 层 | 方案 | 条件 |
|---|---|---|
| **首选** | §三 参数清洗器 | updatedInput 可用（或升级后可用） |
| 兜底 1 | JSON deny，reason 携带**逐字正确调用 JSON** + 明确"multiplayer/entry/scriptsPath 三键不得出现（非 null，是键不存在）" | updatedInput 不可用且暂无法升级 |
| 兜底 2（结构性） | **无参数代理**：本地 MCP proxy（或 CLI 脚本+skill）暴露零参数 `build_rensheng`，内部写死正确参数调真 build；hook 永久 deny 原始 build 工具。模型没有自由度就不会犯错，不依赖任何模型配合 | 任何时候都值得做，一劳永逸 |
| 源头 1 | CLAUDE.md 加两句："本项目 .project/settings.json 已有多人配置，**任何构建都绝不是首次构建**"+ 一条逐字正确调用示例（文档 few-shot 替代老对话锚定） | 立即 |
| 源头 2 | 向引擎团队反馈：① build 在 settings.json 已有配置时应忽略入参 multiplayer；② description "REQUIRED on first build" 措辞是事故源，建议改 "OMIT if local config exists" | 渠道反馈 |
| 验证闸 | PostToolUse 校验构建输出必须含"多人游戏/服务端入口 server_main.lua"（rules §6.3 manifest 检查自动化），不符即向模型回明确失败信号，杜绝"假成功"被当完成 | 与任一层并行 |

## 五、执行 AI 落地顺序

1. `claude --version` + hooks 文档核对 `updatedInput` → 支持则改清洗器；验收标准：**全新对话第一次 build 调用即产出"运行模式=多人游戏"输出**。
2. 不支持 → 升级 Claude Code；不能升级 → 兜底 1 立即上 + 着手兜底 2。
3. 无论走哪条：CLAUDE.md 两句话 + PostToolUse 验证闸都加上。
4. 核实触发事件：昨天 sce-urhox MCP / Claude Code 是否升版、新旧对话模型是否一致——确认"突然发生"的原因，防同类突变再次无预警出现。
5. 兜底 2（无参代理）建议无论如何排期——它是唯一不依赖模型行为、不依赖 hook 能力的架构级答案。

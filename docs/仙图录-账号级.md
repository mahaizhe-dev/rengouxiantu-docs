# 仙图录（仙途录）系统现状与优化方案

> 状态：按当前代码基线更新  
> 更新时间：2026-06-06  
> 命名说明：代码仍使用“仙图录 / AtlasSystem”，本文中的“仙途录”视为产品名称  
> 本轮目标：先修仙图录自身问题与账号级汇总，再评估是否合并称号

---

## 1. 结论摘要

- 当前仙图录已经是线上代码的一部分，不是历史草案。
- 当前实际范围是 `12` 枚道印和 `4` 套神器，旧文档对数量和范围已过时。
- 账号级能力目前只做到了“账号级存储共享”，还没有真正做到“账号级进度判定共享”。
- `先人一步·化神 / 合体` 不是单点故障，而是一个触发链路问题，至少包含：客户端入口过窄、服务端缺少真实达成校验、排行榜计分方向高概率有误、重复提交缺少幂等保护。
- 当前 atlas 只正式支持第一类账号级需求：单个角色达成后取最高值。它还没有解决第二类需求：多个角色共同累计一条长期进度。
- 按当前优先级，不建议马上合并称号。应该先把 atlas 的竞速、最高值判定、累计值汇总三件事做稳，再决定是否并轨称号。

## 3. 当前代码现状

### 3.1 账号级存储已经落地

当前仙图录通过独立 key `account_atlas` 持久化，加载时会覆盖角色存档中的 `saveData.atlas`。

当前链路：

- `SaveLoader.Load` 读取 `account_atlas`
- `SaveLoader.ProcessLoadedData` 调用 `AtlasSystem.Init()`
- 若存在 atlas 数据，则 `SaveSerializer.DeserializeAtlas(saveData.atlas)`
- 之后执行 `AtlasSystem.PostLoadSync()`
- `AtlasSystem.SaveImmediate()` 可直接回写 `account_atlas`

这说明 atlas 已经是账号级旁路存储，而不是单纯的角色存档字段。

当前 atlas UI 不是旧文档里的简化版，而是：

- 顶层 2 个页签：`道印`、`神器`
- 神器页下有 4 个子页签：
  - 上宝逊金钯
  - 文王八卦盘

## 5. 当前问题清单

### 5.1 P0：账号级存储和角色级判定错位

当前 atlas 虽然存到 `account_atlas`，但 `value_check` 的数据源读取仍然是当前角色运行态：

- `major_realm` 读取当前角色境界
- `class_level_*` 读取当前角色职业和等级
- `trial_floor` 读取当前角色的试炼塔进度

直接后果：

- 角色 A 达成过高试炼层数，切到角色 B 时，道印升级未必能正确体现
- 各职业道印只会在对应职业角色在线时推进
- 账号级 atlas 变成了“共享结果”，不是“共享进度来源”

### 5.2 P0：`先人一步` 当前不是一个稳定的竞速发奖链路

#### 已确认的问题

1. 客户端触发点过窄  
   当前只在突破到 `huashen_1` / `heti_1` 时主动发起竞速请求，`PostLoadSync()` 虽然有回溯补偿，但本质仍是事后补救。

2. 服务端没有校验“这次提交是否真的达成了对应境界”  
   `RankHandler.HandleTryUnlockRace()` 只校验 `medalId` 合法，不校验玩家是否真的达到化神或合体。

3. 缺少幂等保护  
   服务端没有“已有记录直接返回”的分支，重复提交会改写排名分数。

4. `alreadyHad` 语义未落地  
   客户端留了 `alreadyHad` 处理逻辑，但服务端当前固定回 `false`。

#### 高概率问题（基于同文件排行榜用法推断）

当前竞速榜写入分数是 `os.time()`，而同文件其他排行榜整体呈现为“大分数排前面”。

如果竞速榜也遵循同一排序方向，那么当前实现会产生反向结果：

- 后达到的人时间戳更大
- 后达到的人会排在前面
- 竞速榜会变成“后到更优”，而不是“先到更优”

这会直接破坏“先人一步”的定义。

### 5.3 P1：atlas 初始化存在重复订阅风险

`AtlasSystem.Init()` 每次调用都会：

- `EventBus.On("inventory_item_added", ...)`
- `SubscribeToEvent(...)` 订阅多个远程事件

而 `SaveLoader.ProcessLoadedData()` 在每次加载流程中都会再调一次 `AtlasSystem.Init()`。

如果运行态没有做订阅去重，则存在以下风险：

- 同一事件多次响应
- 竞速回调重复处理
- 首获检测重复触发

### 5.4 P1：atlas 保存契约仍存在双写心智负担

当前 atlas 一方面：

- 通过 `SaveSerializer.SerializeAtlas()` 进入普通存档 DTO

另一方面：

- 服务端会写独立 `account_atlas`
- 客户端也支持 `SaveImmediate()` 直接写 `account_atlas`

这说明 atlas 已账号级这一点没有问题，但保存契约还不够单一，后续扩系统时容易引入歧义。

### 5.5 P1：累计类数据当前没有账号级汇总模型

这是本轮新增确认的问题。

当前系统只显式支持“最高值型”逻辑，缺少“累计型”逻辑所需的三个基础能力：

- 稳定角色唯一标识
- 账号级源快照
- 增量累计规则

因此像“所有角色累计击杀多少 BOSS”这类需求，现在没有可靠承载结构。

### 5.6 P1：BOSS 计数存在双源漂移风险

当前相关来源至少有两套：

- `GameState.bossKills`
- `TitleSystem.killStats`

同时还存在：

- 运行时累计
- 角色存档恢复
- 旧档迁移回填

如果不先统一权威口径，后续做账号累计时会出现：

- 同一角色重复计入
- 历史数据回放不一致
- 某些来源累计了，另一些来源没累计

### 5.7 P1：当前代码未体现稳定 `charId`

当前系统是多槽位存档，但未看到可直接用于账号长期聚合的角色唯一 ID。

这意味着：

- `slot` 只能表示存档位置
- 不能表示长期角色身份

一旦出现删角重建、旧槽位复用，就会影响累计类数据正确性。

### 5.8 P1：字段中文语义不统一

当前至少存在以下命名混用：

- `constitution`
- `physique`
- 根骨
- 体质
- 体魄

短期不会直接导致功能错误，但会持续抬高维护和对表成本。

### 5.9 已排除或不是主因的项

以下项目不是“先人一步无法触发”的主因，但属于需要记录的上下文：

- badge 相关字段当前未真正接线
- 天帝剑痕、神器加成本身已接入数值层，不是本次竞速问题的根因
- 称号系统虽然是角色级，但它不是当前 `先人一步` 触发失败的直接原因

## 6. “先人一步”专项整改方案

### 6.1 目标

本专项只解决一件事：让“前 10 先达成化神 / 合体”成为可验证、可回溯、可幂等的服务端链路。

### 6.2 建议改造顺序

#### 第一步：修正服务端竞速判定

在 `HandleTryUnlockRace` 增加真实达成条件校验：

- `pioneer_huashen` 必须验证账号当前达成化神门槛
- `pioneer_heti` 必须验证账号当前达成合体门槛

不应只相信客户端发来的 `medalId`。

#### 第二步：修正竞速榜计分方向

必须明确竞速榜的排序语义，并让“越早达成的人排越前”：

- 如果排行榜是大分在前，则应写入“反向时间值”
- 如果排行榜是小分在前，则可直接使用时间戳

不要继续维持当前这种“从周边代码推断有风险”的状态。

#### 第三步：竞速结果独立冻结

建议服务端显式区分三种情况：

- 本次首次成功占位
- 之前已经占位成功
- 本次失败或未进入前 10

客户端据此再处理提示和 atlas 解锁状态，不要把幂等交给前端猜。

#### 第四步：补充自动化验证

至少补以下回归：

1. 两个玩家顺序提交，确认先提交者排名更优
2. 同一玩家重复提交，不改变原排名
3. 未达成境界玩家提交，被服务端拒绝
4. 读档回溯触发不会重复获奖

## 7. 账号级进度层与汇总方案

### 7.1 目标拆分：两类账号级能力

推荐把 atlas 账号级进度正式拆成两类：

| 类型 | 定义 | 典型例子 | 聚合方式 |
|---|---|---|---|
| `account_max` | 账号内任一角色达到的最高值 | 最高境界、各职业最高等级、试炼塔最高层 | `max` |
| `account_sum` | 账号内所有角色累计贡献的总量 | 所有角色累计击杀 BOSS、累计击杀指定怪、累计消耗指定道具 | 增量 `sum` |

当前 atlas 的 `value_check` 全部属于第一类，还没有一条真正意义上的第二类。

### 7.2 为什么第二类以前一直被规避

之前没有合并累计类，不是简单的“没做”，而是因为当前代码确实缺少第二类的基础设施：

1. 没有稳定 `charId`  
   槽位会复用，`slot` 不能代表角色身份。

2. 没有账号级角色源快照  
   现在没有“这个角色上次已经为账号累计贡献了多少”的记录。

3. 没有累计增量规则  
   当前只有“看当前值是否达到阈值”，没有“只结算新增量”的机制。

4. 累计源存在双口径  
   例如 BOSS 统计同时依赖 `GameState.bossKills` 和 `TitleSystem.killStats`。

### 7.3 推荐数据结构

当前阶段建议仍挂在 `account_atlas` 下做独立子域，降低接入面；但实现上应把它当作独立进度引擎，不再把它视为 `medals` 的附属字段。

```lua
account_atlas = {
    version = 3,

    medals = { ... },

    artifact = { ... },
    artifact_ch4 = { ... },
    artifact_tiandi = { ... },
    artifact_ch5 = { ... },

    progress = {
        aggregate = {
            max = {
                majorRealm = 0,
                classLevel = {
                    taixu = 0,
                    zhenyue = 0,
                    luohan = 0,
                    tianyan = 0,
                },
                trialFloor = 0,
            },
            sum = {
                bossKillTotal = 0,
                killStats = {},
                guardianConsumeExp = 0,
            },
        },

        sources = {
            -- [charId] = {
            --     slot = 1,
            --     name = "角色名",
            --     max = { ... },
            --     checkpoints = {
            --         bossKillTotal = 123,
            --         killStats = { boss_a = 4, boss_b = 2 },
            --         guardianConsumeExp = 6000,
            --     },
            -- }
        },
    },
}
```

核心区别：

- `aggregate` 给 atlas 判定层读取
- `sources[charId]` 给聚合层做增量结算

### 7.4 必须先补 `charId`

如果没有 `charId`，第二类累计值不建议上线。

原因很直接：

- `slot = 1` 今天可能是角色 A
- 删除后重新建角，`slot = 1` 又可能变成角色 B
- 如果累计水位绑定槽位，新角色会继承旧角色的“已计入进度”

因此需要在角色创建时生成稳定 `charId`，并进入：

- `saveData.player.charId` 或同级稳定字段
- `slots_index` 摘要
- `account_atlas.progress.sources[charId]`

迁移策略：

- 旧存档首次加载时补发 `charId`
- 补发后随下一次正常存档固化
- 不依赖槽位号作为长期身份

### 7.5 第一类：最高值型 `account_max`

这类是对现有 atlas 的纠偏，而不是新增复杂功能。

推荐做法：

- `majorRealm = max(所有角色大境界)`
- `classLevel.<classId> = max(所有该职业角色等级)`
- `trialFloor = max(所有角色历史试炼层数)`

然后把现有 `value_check` 全部改成读 `progress.aggregate.max`，不再直接读当前角色运行态。

### 7.6 第二类：累计型 `account_sum`

累计型不能直接每次都“扫描所有角色相加后覆盖保存”，更稳的做法是“高水位增量累计”：

```lua
delta = max(0, currentValue - lastAppliedValue)
aggregateSum = aggregateSum + delta
lastAppliedValue = max(lastAppliedValue, currentValue)
```

这样可以保证：

- 正常推进时只累计新增量
- 重复存档不会重复加
- 角色回档后不会把旧数据再算一遍

### 7.7 BOSS 汇总的推荐口径

这里是第二类最敏感的一块，建议先统一口径，再做账号汇总。

不推荐直接以 `GameState.bossKills` 作为账号累计主源，原因：

- 它是运行态标量
- 它和 `TitleSystem.killStats` 同时存在
- 两套来源长期并行，迟早会漂

更稳的方案：

- 角色级主源统一以 `TitleSystem.killStats` 为准
- 账号级 `progress.aggregate.sum.killStats` 按怪物类型累计
- `bossKillTotal` 优先由聚合后的 `killStats` 过滤 `boss / king_boss / emperor_boss` 后再计算

这样可以同时支持：

- “累计击杀 BOSS 总数”
- “累计击杀某个指定 BOSS”
- “累计击杀某个系列怪物”

### 7.8 边界场景处理

| 场景 | 推荐处理 |
|---|---|
| 角色重复存档 | 只吃增量，不重复累计 |
| 角色回档 | 不回退账号累计，不重放旧数据 |
| 删除角色 | 不扣减账号累计，保留账号历史结果 |
| 新角色占用旧槽位 | 因 `charId` 不同，不继承旧角色累计水位 |
| 手工修档 | 提供一次性“按所有槽位重建 aggregate”的维护工具 |

### 7.9 atlas 规则层改造方式

推荐把 `MedalConfig` 的触发语义改明确，不再让 `value_check` 既像角色级又像账号级：

```lua
trigger = { type = "account_max", source = "major_realm", thresholds = {...} }
trigger = { type = "account_max", source = "trial_floor", thresholds = {...} }
trigger = { type = "account_sum", source = "boss_kill_total", thresholds = {...} }
trigger = { type = "account_sum", source = "kill_target", targetType = "wu_wanhai", thresholds = {...} }
```

这样后续新增累计型道印时，不需要再绕开设计。

### 7.10 聚合与重建时机

推荐分两层：

1. 轻量增量更新  
   在存档成功后、关键事件后更新当前角色对应 `sources[charId]`，并结算增量。

2. 全量重建兜底  
   在登录读取多槽位、旧档迁移、数据修复时，扫描所有槽位重建一次 `aggregate`。

这样可以兼顾：

- 日常运行成本
- 旧档迁移可控
- 数据修复能力

## 8. 完整优化方案

### 8.1 Phase 1：先修 `先人一步`

目标：

- 服务端校验玩家是否真的达到 `huashen_1 / heti_1`
- 修正竞速榜计分方向
- 增加“已有记录直接返回”的幂等分支
- 明确 `alreadyHad` 协议语义
- 补竞速回归用例

这是独立线上 P0，应最先处理。

### 8.2 Phase 2：把 atlas 从“角色判定”改成“账号最高值判定”

目标：

- 新增 `progress.aggregate.max`
- `realm_master / taixu / zhenyue / luohan / tianyan / trial` 全部改为读取账号聚合值
- `PostLoadSync()` 不再依赖“当前角色是谁”才能补齐道印等级

这一步完成后，第一类账号级能力才算真正落地。

### 8.3 Phase 3：补第二类累计型汇总能力

目标：

- 为角色补 `charId`
- 新增 `progress.sources[charId]`
- 为累计类来源补高水位快照
- 第一批先支持 `boss_kill_total`
- 第二批再支持指定目标击杀、累计消耗类道印

推荐顺序：

1. 先统一 BOSS 主源口径
2. 再接入账号累计聚合
3. 再上线首个累计型道印

### 8.4 Phase 4：清理 atlas 结构遗留

目标：

- 清理 `Init()` 重复订阅风险
- 收敛 atlas 保存契约，降低双写心智负担
- 统一 `constitution / physique / 根骨 / 体质 / 体魄` 命名
- 对未生效 badge 字段做“删除或补完”二选一

### 8.5 Phase 5：验收通过后，再评估是否合并称号

当前结论不是“不合并称号”，而是“现在还不到时候”。

前置条件至少包括：

- `先人一步` 竞速链路稳定
- `account_max` 已正式替代当前角色判定
- `account_sum` 至少有一条真实业务链路跑通
- 多槽位切换、删角、回档场景已经验证

只有这四项通过后，再考虑把称号并入 atlas，风险才可控。

## 10. 最终建议

本轮优化目标应收敛为三件事：

1. 修稳 `先人一步`
2. 把 atlas 的第一类账号级能力做真
3. 新增 atlas 的第二类账号级累计能力

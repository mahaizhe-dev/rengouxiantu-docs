#### 1. 藏真圣衣、圣器披风生成错字段

文档意图是：

受影响文件：

- `scripts/systems/LootSystem.lua`
- `scripts/ui/SwordForgeUI.lua`

### P1：第五章与第四章稳定模式发生错误分叉

设计原意是复用第四章思路，但当前第五章单独分出 `SwordForgeUI`，并且行为已经偏离：

1. 第四章稳定链路是成熟参考
2. 第五章单独 UI 链路没有完全沿用第四章的验证、生成、提交流程
3. 第五章测试又没有覆盖真实生产路径，导致错误持续存在

### P2：资源扣除顺序非原子

当前第五章打造流程存在：

1. 先扣金币
2. 先删材料/背包物
3. 再生成产物

这个问题第四章也有，但第四章已线上稳定，不作为本次大范围回溯对象。  
本次至少要保证第五章链路不再继续扩大这个风险。

## 阶段 2：修正第五章打造提交流程

### 目标

让第五章打造流程至少达到“先验证、再提交、失败不吞资源”的可上线水平。

### 问题点

当前 `SwordForgeUI.DoForgeSword()` 存在：

1. 先扣钱
2. 先删背包材料
3. 再生成产物
4. 再写入背包或装备栏

这会导致失败时资源损失。

### 执行要求

#### A. 通用打造路径

对于 `道藏圣衣 / 圣器披风 / 帝尊圣戒 / 龙魂令`：

1. 先检查背包格子、金币、材料、输入物是否充足
2. 在不改动背包的前提下先尝试构造产物
3. 只有当产物构造成功时，才进入正式扣费和扣料
4. 扣费扣料后立即落入背包
5. `AddItem` 若理论上仍失败，必须视为严重错误并避免静默吞资源

#### B. 解封古剑路径

对于 `一重解封古剑`：

1. 先确认装备栏武器正确
2. 先确认背包辅材、金币、材料足够
3. 先基于当前武器构造 `newItem`
4. 构造成功后再正式扣费、扣料、删辅助装备
5. 最后原地替换装备栏武器数据

### 事件与存档要求

成功后必须继续触发：

1. `InventorySystem.RecalcEquipStats()`
2. `EventBus.Emit("save_request")`
3. `EventBus.Emit("equipment_changed")`

失败时不得触发成功提示，不得伪装成成功。

### 完成标准

1. 任意生成失败不会导致金币和材料已扣但产物为空
2. 解封古剑成功后装备栏数据完整
3. 背包模式产物成功进入背包
4. 失败分支提示明确

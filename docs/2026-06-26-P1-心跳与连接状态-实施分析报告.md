## 2. P1 方案评估

### 2.1 P1-1：Heartbeat 协议

**方案设计**：
- C2S_Heartbeat 每 5s 发送（requestId, clientTime, activeSlot, saveDirty, saving）
- S2C_Heartbeat 服务端 echo（requestId, serverTime）
- 不写存档，不做业务重放

**实施评估**：

| 维度 | 评分 | 分析 |
|------|------|------|
| 必要性 | 8/10 | 当前只有 3s 采样 `GetServerConnection()`，这是 transport 层存在性，不代表业务可达。心跳是唯一可靠方式 |
| 复杂度 | 中 | 需改 SaveProtocol（新增 2 个事件）、server_main（echo handler）、NetworkStatus（消费 ACK） |
| 服务端负载 | 低 | 5s × N 玩家，包体 <100 bytes，纯内存 echo 无 IO |
| 客户端负载 | 极低 | 1 个 VariantMap + 1 次 SendRemoteEvent/5s |
| 风险 | 低 | echo handler 无副作用，最坏情况是心跳不回包→退化为当前行为 |

**实施建议**：

```lua
-- SaveProtocol 新增
SaveProtocol.C2S_Heartbeat = "C2S_Heartbeat"
SaveProtocol.S2C_Heartbeat = "S2C_Heartbeat"

-- server_main.lua 新增 handler（极简 echo）
SubscribeToEvent(SaveProtocol.C2S_Heartbeat, function(eventType, eventData)
    local conn = eventData["Connection"]:GetPtr("Connection")
    if not conn then return end
    local resp = VariantMap()
    resp["requestId"] = eventData["requestId"]
    resp["serverTime"] = Variant(os.time())
    conn:SendRemoteEvent(SaveProtocol.S2C_Heartbeat, true, resp)
end)
```

**关键约束**：
1. 心跳只 echo，不做任何存档/业务操作
2. 客户端不能因心跳成功就清 dirty 或放行黑市
3. 心跳超时不代表"必须断线"——可能是服务端繁忙
4. 心跳恢复不代表"保存链路健康"——仍需等 game_saved

### 2.2 P1-2：连接状态改为业务健康判定

**方案设计**：
- connected → unstable → reconnecting → disconnected
- 恢复条件：heartbeat ACK 或 S2C_SaveResult ok=true

**实施评估**：

| 维度 | 评分 | 分析 |
|------|------|------|
| 必要性 | 9/10 | 当前二值（连/断）无法表达"波动中"，导致 UI 提示要么不出现要么直接断线 |
| 复杂度 | 高 | 状态机 4 态 + 转移条件 + 超时 + 恢复 + 与 SaveSystem/UI 联动 |
| 风险 | 中 | 状态机 bug 可能导致误判断线或误恢复 |
| 依赖 | P1-1 | 必须有心跳才能判断"unstable"和"reconnecting" |

**状态转移建议**：

```
connected ──(1次心跳超时)──→ unstable
unstable  ──(收到ACK)──────→ connected
unstable  ──(15s无ACK)─────→ reconnecting
reconnecting ──(收到ACK)───→ connected
reconnecting ──(45s无ACK)──→ disconnected
disconnected ──(收到ACK)───→ connected（需要补保存确认）
```

**关键约束**：
1. 状态机不能影响 dirty 保护——任何状态下 `_dirtyWarehouse` 仍只在 `game_saved` 后清
2. `unstable` 状态下仍允许普通操作（查询、买入），只阻断高风险写操作（出售）
3. `disconnected` 状态下全面阻断 + 全屏提示
4. 不能因为心跳恢复就认为"保存成功了"

### 2.3 P1 保存恢复行为

**方案设计**：
- connection_restored 后延迟 1-3s 随机补保存
- 不重放非幂等 C2S

**当前代码已有基础**（SaveSystem.Init 中）：
```lua
SaveSystem._onConnRestored = function()
    SaveSystem._disconnected = false
    if SaveSystem.loaded and SaveSystem.activeSlot then
        SaveSystem._dirty = true  -- 标记 dirty，由防抖流程统一调度
    end
end
```

**评估**：当前已有"重连后标 dirty"的逻辑。P1 的增量是：
1. 将恢复条件从 `GetServerConnection() ~= nil` 改为"心跳 ACK 收到"
2. 加随机延迟避免保存风暴（当前缺失）
3. 恢复后第一次保存成功时清除 UI 警告

## 6. 对方案提出 AI 的建议

### 6.1 方案质量确认

P1 设计合理、边界清晰、禁止项完备。以下是基于实测的微调建议：

### 6.2 建议调整

1. **心跳间隔建议 10s 而非 5s**：当前 `UpdateCritical` 中 serverConn 采样已是 3s 一次。心跳 5s 与采样 3s 重叠过多。10s 心跳 + 3s transport 采样 = 双层检测，既省带宽又不失灵敏。

2. **Heartbeat payload 精简**：方案建议发送 `activeSlot, saveDirty, saving`。建议只发 `requestId + clientTime`，服务端不需要知道客户端保存状态（这些信息对 echo 无意义，只增加包体）。如果未来需要服务端感知客户端状态，单独做。

3. **unstable 阈值建议 10s 而非"1 次心跳超时"**：单次 10s 心跳超时可能是正常网络抖动。建议连续 2 次心跳无 ACK（20s）才进 unstable，避免过于敏感。

4. **disconnected 恢复后必须走 RequestImmediateSave**：不能只标 dirty 等防抖。用户可能已经等了 60s+，恢复后应尽快推动保存。建议恢复后 1-3s 随机延迟 + 直接调 RequestImmediateSave。

5. **EventBus.On 不可靠问题需在 P1 前解决**：P0 实测发现 EventBus.On 在当前运行环境存在注册失效问题（warehouse_changed 和 game_saved 的 On 回调均不触发）。P1 的心跳 ACK 消费如果也用 EventBus.On 订阅 S2C_Heartbeat，可能遇到同样问题。建议 P1 的 ACK 处理**不走 EventBus**，改为在 SaveProtocol 的 S2C handler 注册中直接 dispatch（与现有 S2C_SaveResult 相同模式）。

### 6.3 P1 前置验证建议

在正式实施 P1 前，建议先做一个 10 分钟的验证实验：
1. 在 server_main 注册一个临时 echo handler
2. 客户端手动发一次 C2S 事件
3. 确认 S2C 回包能被客户端收到
4. 确认 RTT 日志正确

如果这个简单回包都收不到，说明有更底层的协议问题需要先解决。

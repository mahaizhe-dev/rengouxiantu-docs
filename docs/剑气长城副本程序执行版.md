# 剑气长城副本程序执行版

## 1. 执行目标

本文给执行 AI 使用，目标是在现有代码基准上落地 `剑气长城` 门票型多波挑战副本。

只实现以下内容：

1. 第五章 `栖剑别院` 附近新增副本入口 NPC。
2. 入口 NPC 提供 `进入剑气长城` 与 `剑气积分商店`。
3. 进入消耗 `1个仙人精血`。
4. 角色传送到第五章现有 `剑气长城` 区域，不进入异空间。
5. 副本区域刷新 `3波临时挑战怪`。
6. 全部击杀后生成大型结算宝箱。
7. 宝箱一次性结算 `10~30剑气积分 + 1件随机T10青色灵器`。
8. 剑气积分商店支持指定兑换四仙剑、灵兽丹、上品灵韵果、上品修炼果、金砖。

不做以下内容：

1. 不接入多人副本系统。
2. 不重建地图。
3. 不调用声望挑战 `_createArena`。
4. 不设计渡劫、仙阶、谪仙进阶。
5. 不在本文固定 BOSS 技能细节、图标、精确坐标、UI 外观。

## 5. 副本状态机

### 5.1 进入

流程：

1. 玩家与入口 NPC 交互。
2. UI 显示 `进入剑气长城` 与 `剑气积分商店`。
3. 点击进入后校验玩家是否在第五章。
4. 校验是否已有挑战、副本、塔、多人副本等活跃状态。
5. 校验并消耗 `immortal_essence_blood x1`。
6. 生成 `runId`。
7. 记录 `entryReturn` 为入口 NPC 附近安全点。
8. 传送玩家到 `ch5_sword_corridor` 战斗区域。
9. 刷新第 1 波。

门票扣除原则：

1. 如果不能进入战斗区，不扣门票。
2. 如果门票已经扣除但创建副本失败，必须返还门票或阻断进入前置流程。
3. 进入成功后失败不返还门票。

### 5.2 波次推进

流程：

1. `waveIndex = 1` 时刷新 6 只 `ch5_blood_general`。
2. 本波所有 `isSwordWall` 怪死亡后，延迟短时间刷新下一波。
3. `waveIndex = 2` 时刷新 3 只魔帅。
4. 第二波全灭后刷新第 3 波。
5. `waveIndex = 3` 时刷新 `ch5_sword_wall_luohou`。
6. BOSS 死亡后设置 `state = "chest_ready"`，生成大型宝箱。

怪物死亡判断：

1. 只处理 `monster.isSwordWall == true` 的死亡事件。
2. 必须校验 `monster.challengeRunId == current runId`。
3. 不允许野外同名怪推进副本波次。

### 5.3 宝箱结算

流程：

1. 玩家点击大型宝箱。
2. 本地状态必须是 `chest_ready`。
3. `chestOpened == false`。
4. 调用服务端或权威结算接口。
5. 服务端校验 `runId` 未结算。
6. 随机生成 `10~30` 点剑气积分。
7. 随机生成 1 件 T10 青色灵器。
8. 写入积分、奖励物品和结算日志。
9. 成功后本地标记 `chestOpened = true`、`rewardClaimed = true`。
10. 展示奖励结果。
11. 传回入口 NPC。

宝箱实体规则：

1. 宝箱只在第 3 波 BOSS 死亡后出现。
2. 同一 `runId` 只能生成一个宝箱。
3. 同一宝箱只能点击成功一次。
4. 结算中重复点击必须被锁住。
5. 结算失败时不应本地伪造奖励。

### 5.4 失败与退出

失败条件：

1. 玩家死亡。
2. 玩家点击退出传送阵。
3. 玩家离开有效战斗区域。
4. 玩家切章节。
5. 玩家断线、重登或加载存档时发现副本状态未完成。

失败处理：

1. 删除所有 `isSwordWall` 临时怪。
2. 删除宝箱。
3. 删除退出传送阵。
4. 清理 `active/runId/waveMonsters/chestOpened/rewardClaimed`。
5. 传回入口 NPC。
6. 不发积分。
7. 不发装备。
8. 不保留波次进度。

## 6. 积分与商店的防作弊实现

### 6.1 货币模型

推荐服务端 key：

`sword_wall_points`

查询返回：

1. 当前剑气积分余额。
2. 商店商品表。
3. 如有 WAL 待补偿，先尝试补偿再返回。

禁止：

1. 客户端直接修改积分。
2. 把积分作为 consumable 放进背包。
3. 仅靠本地存档保存积分。
4. 商店购买先发物品后扣积分。

### 6.2 宝箱加积分

推荐接口：

`SwordWallHandler.HandleClaimChest`

输入：

1. `runId`
2. `slot`
3. 可选客户端展示用副本状态摘要

服务端校验：

1. userId 合法。
2. `runId` 格式合法。
3. `runId` 未结算。
4. 请求未在 pending 中。
5. 当前角色存档可加载。
6. 背包可以接收奖励，或可写入 WAL。

推荐原子流程：

1. 生成 `points = RandomInt(10, 30)`。
2. 生成奖励装备序列化数据。
3. `BatchCommit("剑气长城宝箱结算")`。
4. `MoneyAdd(userId, "sword_wall_points", points)`。
5. 写入 `CLAIM_LOG_KEY` 或等价结算标记。
6. 写入 `WAL_ITEM_KEY`，内容包含 `runId` 与装备数据。
7. `BatchCommit` 成功后，把装备写入背包存档。
8. 背包写入成功后清理对应 WAL。
9. 背包写入失败时保留 WAL，下次登录补发装备。

如果当前云接口不能保证 `runId` 唯一写入，至少要做到：

1. 本地会话有 `pendingClaims[runId]` 锁。
2. 存档中保留最近 N 个已结算 `runId`。
3. 服务器日志记录 `runId/userId/points/itemId/time`。
4. 重复结算请求必须拒绝。

### 6.3 商店扣积分

推荐接口：

`SwordWallHandler.HandleShopBuy`

输入：

1. `shopItemId`
2. `amount`，第一版建议只允许 `1`

服务端校验：

1. 商品存在。
2. 价格合法。
3. 余额足够。
4. 背包可接收，或可写入 WAL。
5. 请求没有 pending。

推荐原子流程：

1. 查询余额。
2. 构造奖励物品。
3. `BatchCommit("剑气积分商店购买")`。
4. `MoneyCost(userId, "sword_wall_points", price)`。
5. 写入 `WAL_ITEM_KEY`。
6. `BatchCommit` 成功后写背包。
7. 背包写入成功后清理 WAL。
8. 背包写入失败时保留 WAL。

失败处理：

1. 余额不足直接拒绝。
2. `MoneyCost` 失败直接拒绝。
3. 物品构造失败不得扣积分。
4. 扣积分成功但背包保存失败，必须通过 WAL 补发，不能吞物品。

### 6.4 日志

至少记录：

1. `runId`
2. `userId`
3. `slot`
4. `pointsDelta`
5. `balanceAfter`
6. `rewardItem`
7. `shopItemId`
8. `reason`
9. `time`

日志可参考黑市个人交易日志和公共交易日志。

## 7. 奖励装备生成

参考文件：

`scripts/systems/LootSystem.lua`

现状：

1. `GenerateRandomEquipment` 支持 `tierOnly`，可用于普通 T10 装备生成。
2. `GenerateSetEquipment` 当前硬编码 `tier = 9`。
3. `ForgeRandomLingqi` 当前硬编码 `tier = 9`。
4. `ForgeRandomSetLingqi` 当前复用硬编码 T9 的 `GenerateSetEquipment`。

执行要求：

1. 不允许直接使用硬编码 T9 的函数生成宝箱奖励。
2. 需要新增或参数化 T10 生成函数。
3. 普通奖励固定 `tier = 10`、`quality = "cyan"`。
4. 套装奖励固定 `tier = 10`、`quality = "cyan"`。
5. 套装 `setId` 只从 `xuesha/qingyun/fengmo/haoqi` 抽取。
6. 套装概率为 `30%`，普通概率为 `70%`。

推荐新增函数：

```lua
LootSystem.GenerateSwordWallRewardEquipment()
```

内部逻辑：

1. `math.random() < 0.30` 走 T10 套装。
2. 否则走 T10 普通青色灵器。
3. 返回完整可入背包的装备对象。

## 11. 必测用例

基础流程：

1. 门票不足，无法进入。

### 13.3 必须注意的实现细节

**CDR 30% 实现**：

`Monster.lua:180` 中 `self.skills[i] = skillDef` 是直接引用全局 `MonsterData.Skills[skillId]`。改 `skill.cooldown` 会污染全局定义，导致同 ID 野外怪也受影响。

正确做法：创建副本怪时 deepcopy 技能表再修改：

```lua
for i, skillId in ipairs(monster.skillIds) do
    local orig = MonsterData.Skills[skillId]
    if orig then
        local copy = {}
        for k, v in pairs(orig) do copy[k] = v end
        copy.cooldown = (orig.cooldown or 3) * 0.7
        monster.skills[i] = copy
    end
end
```

**怪物清理**：

副本怪与世界怪共存于 `GameState.monsters`。清理时按 `isSwordWall + challengeRunId` 双标记过滤，禁止清空整个列表。

**BOSS 击杀冷却**：

`GameEvents.lua:61` 会为 BOSS 记录 `bossKillTimes`。副本 BOSS 的 `category` 不应设为 `GameConfig.BOSS_CATEGORIES` 中的值，或在死亡事件中优先拦截 `isSwordWall` 怪跳过记录。

**互斥接入点清单**（6处）：

1. `TownReturnSystem.lua:75` — 阻止回城
2. `GameEvents.lua:536` — 死亡触发失败
3. `ChallengeSystem` 进入校验 — 阻止同时开声望挑战
4. `TrialTowerSystem.lua:80` — 互斥
5. `PrisonTowerSystem.lua:80` — 互斥
6. `NPCDialog.lua` — 副本中阻止其他 NPC 交互（可选）

## 一、问题描述

**现象**：新建的 Claude Code 对话（新会话）在执行 `mcp__sce-urhox__build` 构建时，要么无限失败，要么"成功"但产出的构建包实际是单机模式（服务端不启动）。

**影响**：
- 新对话无法完成正常的构建-测试-发布流程
- 服务端不启动导致玩家无法连接
- 开发者被迫在稳定老对话中执行所有构建

**对比**：本稳定对话（在问题出现前启动）可以正常构建，输出正确的多人模式信息。

### 2.1 项目架构

- **类型**：多人 C/S 联网游戏

```

```

注意：没有"多人游戏"、"服务端入口"等信息 — 被当作单机处理。

## 四、根因分析

### 4.1 AI 模型的工具调用行为

`mcp__sce-urhox__build` 工具的 JSON Schema 定义了以下**可选参数**：

```json
{
  "entry": { "type": "string", "description": "Script entry for single-player games..." },
  "entry_client": { "type": "string", "description": "Client entry for multiplayer..." },
  "entry_server": { "type": "string", "description": "Server entry for multiplayer..." },
  "scriptsPath": { "type": "string", "description": "Scripts directory..." },
  "multiplayer": { "type": "object", "description": "Multiplayer game configuration. REQUIRED on first build if no local config exists..." }
}
```

**关键发现**：AI 模型（Claude/其他）在调用工具时，**无法控制不包含某些可选参数**。即使指令明确说"只传 entry_client 和 entry_server"，模型的实际 tool call JSON 仍然携带：
- `multiplayer: {enabled: true}` 或 `multiplayer: null`
- `entry: "main.lua"` 或 `entry: "client_main.lua"`
- `scriptsPath: ""` 或 `scriptsPath: null`

这不是指令理解问题，而是模型层面的工具调用序列化行为。模型反复声称"这次只传两个参数"，但实际发出的 JSON 总是包含额外字段。

### 4.2 build 工具的参数处理行为

build 工具对收到的参数有破坏性副作用：

| 参数 | 工具行为 | 破坏性 |
|------|---------|--------|
| `multiplayer: {enabled: true}` | 写入 settings.json，**替换**整个 multiplayer 块 | 丢失 background_match/max_players 等 |
| `entry: "main.lua"` | 写入 project.json 的 `entry` 字段，平台视为单机 | 服务端不被启动 |
| `scriptsPath: ""` | 可能改变脚本目录 | 找不到脚本文件 |

**关键**：这些写入发生在**上传阶段之前**，上传包已经携带了错误配置。Post-hook 修复本地文件为时已晚。

### 4.3 "警告放行"策略为什么失败

| 策略 | 结果 |
|------|------|
| multiplayer 警告放行 | build 工具用不完整的 multiplayer 对象覆盖 settings.json → 上传包的多人配置损坏 → 单机模式 |
| entry 警告放行 | build 工具将 entry 写入上传包 → 平台按单机处理 → 服务端不启动 |
| scriptsPath="" 回退默认 | **唯一有效的修复** — 空字符串不传递给工具 |

### 4.4 "严格阻断"策略的问题

当前 hook（严格模式）对 multiplayer 和 entry 一律 exit 2 阻断。效果：
- AI 第一次调用（携带 multiplayer + entry）→ 被阻断
- AI 第二次调用 → **应该**去掉 multiplayer 和 entry → 如果成功去掉，构建正确
- **实际观察**：AI 在 7+ 次重试中始终无法去掉 multiplayer 参数 → 无限阻断

### 5.1 文件位置

```

/workspace/.claude/hooks/repair_rensheng_build_config.py — PostToolUse（修复本地）
/workspace/.claude/settings.json — hook 配置
```

### 5.2 当前 guard 逻辑

```python
# multiplayer: 阻断（exit 2）
if "multiplayer" in args and args["multiplayer"] is not None:
    → exit 2

# entry: 阻断（exit 2）  
if entry:
    → exit 2

# entry_client/entry_server 必须正确
if client != "client_main.lua" or server != "server_main.lua":
    → exit 2

# scriptsPath 空值容错
scriptsPath = args.get("scriptsPath") or "scripts"
```

### 5.3 当前 repair 逻辑（PostToolUse）

```python
# 删除 project.json 中任何 entry 字段
# 确保 entry@client = "client_main.lua"
# 确保 entry@server = "server_main.lua"  
# 恢复 settings.json 的标准 multiplayer 配置
```

## 六、核心矛盾

```
AI 模型无法不传 multiplayer 参数
      ×
build 工具收到 multiplayer 就会覆盖正确配置
      ×
Pre-hook 只能阻断或放行，不能修改参数
      =
死局
```

**阻断** → AI 无限重试（因为它无法不传 multiplayer）
**放行** → 构建"成功"但配置被破坏（单机模式）

## 七、可能的解决方向

### 方案 A：模型层面

让 AI 模型学会不传 `multiplayer`。
- **可行性**：极低。多次实验证明模型无法控制工具调用的可选参数序列化。
- **已尝试**：CLAUDE.md、project-level 规则、错误消息提示 — 均无效。

### 方案 B：build 工具层面

修改 build 工具行为：当 settings.json 已有 multiplayer 配置时，忽略参数中的 multiplayer。
- **可行性**：需要引擎团队支持。本项目无权修改 build 工具。
- **效果**：如果实现，彻底解决问题。

### 方案 C：Pre-hook 参数修改

让 Pre-hook 能够**修改**工具参数（去掉 multiplayer/entry），而不只是阻断或放行。
- **可行性**：取决于 Claude Code hook 框架是否支持参数修改。当前 PreToolUse hook 只能返回 exit 0（放行）或 exit 2（阻断），无法修改参数。
- **效果**：如果框架支持，彻底解决问题。

### 方案 D：Pre-hook 文件预写 + 特殊标记

Pre-hook 在放行前：
1. 强制写入正确的 settings.json
2. 在 settings.json 中设置一个"锁定标记"
3. build 工具如果识别锁定标记，跳过 multiplayer 覆盖

- **可行性**：需要 build 工具配合识别标记。不现实。

### 方案 E：接受现状

- 保持 hook 严格阻断
- 接受新对话第一次必然失败
- 如果 AI 第二次能去掉 multiplayer（部分模型可以），构建成功
- 如果 AI 无法去掉（当前观察），则新对话无法构建 — 只能在稳定老对话中操作

### 方案 F：multiplayer=null 的特殊处理

观察到有些模型传 `multiplayer: null` 而非 `multiplayer: {enabled: true}`。
当前 hook 对 `null` 放行（`args.get("multiplayer") is not None` 为 False）。
如果能让模型传 `null` 而非对象，就能通过。

- **可行性**：取决于模型行为。有些模型传 null，有些传空对象 `{}`。

### 方案 G：Hook 阻断 + 自动重建

Pre-hook 阻断后，不只是打印错误信息，而是：
1. 阻断当前调用（exit 2）
2. 同时自动触发一次正确的构建（spawn subprocess）

- **可行性**：hook 内部触发异步构建可能有并发问题。复杂且危险。

---

## 八、推荐方案

**短期（立即可用）**：方案 E — 保持严格阻断，在稳定对话中执行构建和发布。

**中期（需平台支持）**：方案 B 或 方案 C — 向引擎团队反馈：
1. build 工具应支持"当 settings.json 已有配置时忽略传入的 multiplayer"
2. 或 Claude Code hook 框架应支持 PreToolUse 修改参数（而非只能阻断/放行）

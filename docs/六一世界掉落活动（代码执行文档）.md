## 1. 任务目标

在当前“五一活动骨架”基础上，落地“六一世界掉落活动”。

这不是简单换一套活动配置，而是把现有“单福袋 + 单奖池 + 单开/十连开 + 开启次数榜”的实现，扩成：

1. 双世界掉落物。
2. 双宝箱开启。
3. 单开 / 五连开。
4. 抽取记录保留。
5. 排行榜改为积分榜。
6. 兑换功能关闭。
7. 大宝箱支持账号级宠物皮肤解锁。
8. 服务端负责开箱权威校验、扣物、开奖、积分和皮肤解锁。

交付要求：

1. 直接改代码，不要只输出方案。
2. 代码完成后，给出改动文件清单。
3. 说明服务端同步与校验链路。
4. 说明跑了哪些测试，哪些没跑。
5. 说明剩余风险。

## 3. 当前实际代码结论

必须先接受这些现状，不能假设活动系统已经通用化：

1. `EventHandler.lua` 当前仍是“五一单福袋实现”。
2. `HandleOpenFudai` 只允许 `Count = 1 或 10`。
3. 开启物写死为 `mayday_fudai`。
4. 奖池写死为 `EventConfig.ACTIVE_EVENT.fudaiPool`。
5. 奖励汇总只支持 `exp / gold / consumable`。
6. 排行榜当前加的是“开启次数”，不是“积分”。
7. 抽取记录 key 仍写死为 `evt_fudai_records`。
8. 排行附加信息 key 仍写死为 `evt_fudai_info`。
9. `LootSystem.GetEventDropRates()` 只支持：
   - `monsterOverrides[typeId]`
   - `dropRates[chapterId][monster.category]`
10. 当前活动掉落逻辑没有 `realm` 分支。
11. `EventExchangeUI.lua` 当前 UI 仍是：
   - 兑换页
   - 福袋页
   - 排行页
12. `EventExchangeUI.lua` 当前发包仍是 `Count=1 / Count=10`。
13. `EventExchangeUI.lua` 当前本地回放奖励也只支持：
   - `TotalExp`
   - `TotalGold`
   - `Consumables`
14. `pet_premium_fuyuan` 当前仍为 `hidden = true`。
15. `SkinShopConfig.lua` 当前未把 `pet_premium_fuyuan` 放进商店列表。
16. 账号级皮肤的权威存储是 `account_cosmetics`，不是活动自己的 `event_data`。
17. `systems/GameEvents.lua` 在怪物死亡时直接调用 `LootSystem.RollEventDrops(dropResult, monster)`，所以世界掉落当前是客户端判定。
18. 五一服务端并不校验活动世界掉落来源，服务端权威点从“开箱/兑换请求”开始。
19. 五一开箱的背包空位策略只检查“至少 1 个空位”，发奖时通过 `AddToBackpack()` 先堆叠、再占新槽位，放不下的剩余数量只打印 `WARNING: backpack full, lost ...`。
20. `main.lua` 中 `eventBound` NPC 的显示条件只有 `EventConfig.IsActive()`，没有按 `eventId` 精确绑定。
21. 活动物品说明文案当前是 `GameConfig.EVENT_ITEMS` 中的静态 `desc` 字段，不存在按活动状态自动切换的 tooltip 逻辑。
22. `SaveSerializer.DeserializeInventory()` 当前只会用 `PET_FOOD / PET_MATERIALS` 刷新消耗品图标和名称，不会刷新 `EVENT_ITEMS`。
23. `EventHandler.lua:249` 白名单校验硬编码为 `requestCount ~= 1 and requestCount ~= 10`。
24. `SaveProtocol.lua:58` 注释写死 `count=1或10`。
25. `EventExchangeUI.lua:453-455` 十连开按钮条件判断为 `fudaiCount >= 10`。
26. `EventExchangeUI.lua:460` 十连开发包为 `SendToServer(..., { Count = 10 })`。

直接结论：

1. 六一活动不能只改 `EventConfig.lua`。
2. 六一活动必须同时修改：
   - 配置层
   - 掉落层
   - 协议层
   - 服务端 Handler
   - 客户端 UI
   - 皮肤商店链路
3. 六一本次保持五一的“客户端世界掉落 + 服务端开箱权威”架构，不在本次把活动掉落迁到服务端。
4. 六一本次保持五一的背包空位策略，不额外升级为“N 空位预检”。
5. 六一相对五一新增的最高风险点，仍然是 `account_cosmetics` 的双写一致性。

## 4. 最终活动规则

### 4.1 活动 ID

活动 ID 使用：

- `childrensday_2026`

### 4.2 活动物品

| 物品 ID | 名称 | 用途 |
|---|---|---|
| `childday_rattle` | 童趣拨浪鼓 | 小宝箱开启物 |
| `childday_pinwheel` | 流光风车 | 大宝箱开启物 |

要求：

1. 两个玩具直接作为开启物，不再额外造“钥匙 -> 箱子”中间道具。
2. UI 可写成“玩具开启宝箱”，但代码直接扣玩具。
3. `sellPrice` 与五一活动物品保持一致，继续使用 `1000`。

### 4.3 世界掉落规则

所有活动掉落都只能来自 BOSS。

普通开启物掉率：

| 章节 | `boss` | 高阶 BOSS |
|---|---:|---:|
| `ch1` | `0.5%` | `king_boss 1.0%` |
| `ch2` | `1.0%` | `king_boss 2.0%` |
| `ch3` | `2.0%` | `emperor_boss 3.0%` |
| `ch4` | `3.0%` | `emperor_boss 4.0%` |
| `ch5` | `4.0%` | `saint_boss 5.0%` |

稀有开启物掉率：

| 怪物境界 | 掉率 |
|---|---:|
| `yuanying_1` | `0.10%` |
| `huashen_1` | `0.20%` |
| `heti_1` | `0.25%` |
| `dacheng_1` | `0.30%` |
| `dujie_1 ~ dujie_4` | `0.40%` |

四仙剑特殊覆盖：

| `typeId` | 掉率 |
|---|---:|
| `ch5_sword_zhu` | `0.50%` |
| `ch5_sword_xian` | `0.50%` |
| `ch5_sword_lu` | `0.50%` |
| `ch5_sword_jue` | `0.50%` |

实现要求：

1. 普通开启物走“章节 + category”。
2. 稀有开启物走“monster.realm 阶梯”，不是 `player.realm`。
3. 四仙剑继续走 `monsterOverrides[typeId]`。
4. 世界掉落判定沿用五一：客户端在 `systems/GameEvents.lua -> LootSystem.RollEventDrops()` 执行，不在本次迁移到服务端。
5. 服务端权威点从“开箱扣物 + 抽奖 + 皮肤解锁 + 积分写入”开始。

### 4.4 开启规则

| `boxType` | 开启物 | 积分 | 支持次数 |
|---|---|---:|---|
| `small` | `childday_rattle` | `1` | `1 / 5` |
| `big` | `childday_pinwheel` | `10` | `1 / 5` |

要求：

1. 只允许单开和五连开。
2. 禁止十连开。
3. 禁止客户端自行扣开启物，必须服务端确认后再做本地回放。

### 4.5 排行与记录

排行榜改为积分榜。

积分规则：

1. 小宝箱 `+1`
2. 大宝箱 `+10`

建议配置：

- `rankKey = "evt_childday_score"`
- `infoKey = "evt_childday_info"`
- `recordsKey = "evt_childday_records"`
- `rewardHint = "活动结束后，积分榜前3的玩家将在交流群(QQ群: 1054419838)获得小额现金红包奖励！"`

要求：

1. 排行榜文案必须保留“前三现金红包/额外奖励”描述。
2. 排行榜文案里必须带群信息，至少要出现 `交流群(QQ群: 1054419838)`。
3. 由于当前五一代码只透传 `rewardHint` 给 UI，六一最省改法是继续把“群信息 + 前三奖励说明”直接写进 `rewardHint`。
4. 抽取记录保留全服共享列表。
5. 记录项新增 `boxType` 字段。
6. 前端展示记录时需要带箱型标签。

### 4.6 皮肤规则

目标皮肤固定为：

- `pet_premium_fuyuan`

规则：

1. 大宝箱抽中该皮肤概率为 `1%`。
2. 若账号未拥有，直接解锁。
3. 若账号已拥有，改发 `10000 灵韵`。
4. 该皮肤同时上架大黑无天商店。
5. 商店价格保持 `1000 仙石`。

### 4.7 NPC 与物品文案边界

这里必须按当前代码现实处理，不要自行脑补成“动态活动系统”：

1. 当前活动 NPC 入口是青云城的 `npc_event_exchange`，并通过 `eventBound = true` 受 `EventConfig.IsActive()` 控制。
2. 当前 `eventBound` 只是“有活动就显示”，不是“按某个活动 ID 精确显示”。
3. 当前活动物品说明文案是 `GameConfig.EVENT_ITEMS` 的静态 `desc`。
4. 六一新物品上线版本应写活动期间文案，不要直接写“活动已结束，可以出售换金币”。
5. 若未来需要在活动结束后切成“可出售”文案，必须单独改配置，或另做动态 tooltip 逻辑，本次不包含该重构。

## 8. 服务端同步与校验要求

### 8.1 权威性原则

重要边界：

1. 世界掉落延续五一，仍由客户端在怪物死亡时调用 `LootSystem.RollEventDrops()` 判定。
2. 六一本次不重构“怪物击杀 -> 掉落生成”到服务端。
3. 因此下面的“服务端权威”范围，是从开箱请求开始，而不是从世界掉落开始。

以下内容从开箱请求开始必须以服务端结果为准：

1. 玩家是否有资格开箱。
2. 玩家持有多少开启物。
3. 实际开启数量。
4. 奖池抽取结果。
5. 金币、灵韵、消耗品发放。
6. 排行分数增加。
7. 皮肤是否已拥有。
8. 皮肤是否应解锁还是转补偿。

客户端只能做：

1. 发起请求。
2. 回放服务端已确认结果。
3. 本地即时保存。

### 8.2 锁与并发

必须保留并正确使用：

1. 用户级内存锁。
2. 槽位锁。
3. 断线清锁。

禁止：

1. 移除五一已有锁逻辑。
2. 在锁释放前发送可重复点击的成功状态。

### 8.3 `save_{slot}` 与 `account_cosmetics`

这里必须特别注意。

当前实际代码中：

1. 角色主存档写在 `save_{slot}`。
2. 账号级皮肤权威 key 是 `account_cosmetics`。
3. `SaveLoadService.lua` 会在加载时把 `account_cosmetics` 注入到 `saveData.accountCosmetics`。
4. `SaveWriteService.lua` 会在保存时把 `coreData.accountCosmetics` 写回 `account_cosmetics`。

因此，大宝箱出皮肤时，不能只改角色内存，也不能只改活动 `event_data`。

最低要求：

1. 服务端开奖时，若抽中新皮肤，必须把最新 `accountCosmetics` 写入本次 `saveData`。
2. 服务端还必须把 `account_cosmetics` 独立持久化。
3. 客户端收到成功回包后，还要调用 `PetSkinSystem.UnlockPremiumSkin()`，并触发即时保存，让客户端本地与服务端收敛。

### 8.4 推荐持久化顺序

若服务器 API 支持同批写入 `save_{slot}`、排行榜分数、`account_cosmetics`，优先用单批次。

若不支持单批次，按以下优先级处理：

1. 奖励主结果持久化。
2. 排行分数持久化。
3. `account_cosmetics` 持久化。
4. `evt_childday_info` 与 `evt_childday_records` 可以后置。

更具体地说：

1. 没有抽中新皮肤时，关键链路至少保证：
   - `save_{slot}` 成功
   - 积分成功
2. 抽中新皮肤且未拥有时，关键链路至少保证：
   - `save_{slot}` 成功
   - 积分成功
   - `account_cosmetics` 成功

### 8.5 两阶段失败处理

如果使用“两阶段写入”：

1. 第一阶段成功写入 `save_{slot}` 与积分。
2. 第二阶段写入 `account_cosmetics`。

则必须处理第二阶段失败。

可接受方案：

1. 若第二阶段失败，参考 `SkinShopHandler.lua` 进行补偿回滚。
2. 或者在第一阶段里已经把 `saveData.accountCosmetics` 一并写进 `save_{slot}`，并且在第二阶段失败时不发送成功回包，而是继续补偿 / 重试，直到状态一致或明确失败。

不可接受方案：

1. 皮肤没有真正落库，却直接返回成功。
2. 客户端本地解锁了皮肤，但服务端下次加载又丢失。
3. 已消耗大宝箱开启物，结果皮肤和补偿都没到位。

### 8.6 记录与排行附加信息

`evt_childday_info` 和 `evt_childday_records` 是展示型数据。

处理要求：

2. 即使它们后置，也要记录失败日志。
3. 不允许因为记录写失败而重复发奖。

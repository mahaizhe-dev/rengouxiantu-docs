# 服务端连接异常排查执行方案

> 适用问题：客户端长时间停留在"正在连接服务器..."，可能数十分钟到数小时后恢复。  
> 目标：由执行 AI 先完成证据收集和归因，再决定是否进入热修。  
> 核心判断：这不是单纯的客户端 UI 问题，而是"客户端长期拿不到可用服务端连接"的问题。

---

## 一、执行目标

本次任务不是先改代码，而是先把问题归到以下三类之一：

1. 构筑/发布问题
   - 线上包遗漏 `entry_server`
   - `CODE_VERSION` 未递增导致旧实例继续服务
   - 发布产物与仓库代码不一致

2. 服务端启动失败/启动脆弱
   - `server_main.lua` 顶层或 `Start()` 期间崩溃
   - 共享模块被服务端启动链误带起
   - 启动时外部依赖访问导致实例不可用

3. 平台/匹配/外部依赖问题
   - 服务端实例未健康
   - 平台分配不到健康实例
   - `serverCloud` 或相关后端短时异常，之后平台自愈

最终必须输出明确结论：

- 最可能根因
- 排除掉了什么
- 还缺什么证据
- 是否需要热修

---

## 二、已知线索

执行 AI 必须把以下线索当成高优先级证据，不能再轻率下结论为"正常冷启动"：

1. 问题曾持续 1 小时不恢复。
2. 问题曾持续 8 小时不恢复。
3. 两次都在 AI 排查期间恢复。
4. 当前客户端文案为"正在连接服务器..."。
5. 客户端连接等待逻辑位于 `scripts/client_main.lua`。
6. 常规情况下，另一端登录同一在线环境时，当前端会被停掉。
7. 故障期间，两端可以同时停留在"正在连接服务器..."，不会触发互踢。

结论约束：

1. 不能把"5 秒后显示连接中"误判成正常。
2. 不能把"排查期间恢复"直接当作巧合。
3. 不能只做静态代码 review 就宣布"本次改动无关"。
4. 不能把"双端登录冲突"直接当成主因，必须先验证故障是否发生在 `ClientIdentity` 之前。

---

## 三、排查原则

1. 先拿证据，再下结论。
2. 优先查真实线上产物和日志，不先猜。
3. 所有判断都要区分"代码正确"和"服务可用"。
4. 如果日志拿不到，就必须明确说明"结论可信度受限"，不能硬判。
5. 如果发现启动链脆弱，即使尚未拿到崩溃日志，也要提出止血热修方案。

---

## 四、执行顺序

## 阶段 A：确认是否为构筑/发布问题

### 目标

排除最常见的双入口遗漏问题，以及版本守卫失效问题。

### 必做项

1. 读取当前线上发布产物的 `manifest-*.json`。
2. 确认是否同时包含：
   - `entry_client="client_main.lua"`
   - `entry_server="server_main.lua"`
3. 对照仓库 `rules.md` 中的构筑规则。
4. 如能拿到最近一次发布记录，确认是否使用了正确 build 命令。
5. **【补充】确认 `CODE_VERSION` 是否在最近发版时同步递增。**
   - `manifest 存在 ≠ 服务端真正被替换`：平台可能接受了双入口 manifest，但因调度策略继续用旧实例。
   - 如果某次发版遗漏了递增 `CODE_VERSION`，旧服务端实例会继续跑旧代码，导致版本守卫失效。
   - 检查 `scripts/config/GameConfig.lua` 中 `CODE_VERSION` 的值，与上一次发版是否有差异。

### 输出

必须明确写出：

- 线上 manifest 是否包含 `server_main.lua`
- 本次线上构筑是否符合双入口规则
- `CODE_VERSION` 是否在发版时同步递增，线上版本是否与仓库一致
- 如果不符合，直接定性为 P0 根因

### 对应文件

- `docs/架构文档/rules.md`
- 线上实际 `manifest-*.json`
- `scripts/config/GameConfig.lua`

---

## 阶段 B：确认服务端是否成功启动

### 目标

判断问题是"服务端根本没起来"，还是"起来了但后续业务失败"。

### 【补充】执行前先验证 boot 标记是否真实存在于代码

**不能假设 boot 标记已存在。执行前必须先 grep `server_main.lua`，确认以下字符串确实被打出来：**

- `[Server-BOOT]`
- `[Server] 人狗仙途 服务端已启动`
- `[Server] serverCloud available`

如果 grep 不到，说明这些 boot 标记在代码里不存在，日志中查不到属于正常，不能误判为启动失败。此时需在阶段 C 热修中补充这些埋点。

### 必做项

1. 获取服务端启动日志。
2. 检查以下 boot 标记是否出现：
   - `[Server-BOOT] Start() called`
   - `[Server-BOOT] Before SaveProtocol.RegisterAll()`
   - `[Server-BOOT] After SaveProtocol.RegisterAll()`
   - `[Server] 人狗仙途 服务端已启动`
   - `[Server] serverCloud available: true/false`

3. 检查客户端是否有以下日志：
   - `No server connection at Start(), waiting for ServerReady...`
   - `ServerReady received!`
   - `ServerReady fired but no server connection!`
   - `No server connection after 120s`

4. **【补充】检查客户端超时后的实际行为：**
   - 120 秒等待结束后，客户端是停在界面上不重试，还是有重连逻辑？
   - 如果是永久卡住不重试，用户的唯一出路是手动重启。这本身也是需要确认的已知缺陷。

5. 检查服务端是否有以下身份阶段日志：
   - `ClientIdentity: connKey=... userId=...`
   - `Duplicate login userId=... (engine auto-kicks)`

6. 对照故障现象确认：
   - 正常情况下双端是否会互踢
   - 故障期间双端是否能共存
   - 共存时两端是否都停在"正在连接服务器..."

### 判定标准

1. 如果服务端 boot 日志根本没有出现：
   - 优先怀疑构筑/平台分配问题

2. 如果有 `Start()`，但没有"服务端已启动"：
   - 优先怀疑服务端启动中途崩溃

3. 如果服务端已启动，但客户端长时间拿不到 `ServerReady`：
   - 优先怀疑平台匹配/实例可达性/连接层问题

4. 如果故障期间双端可以共存，且日志中没有 `ClientIdentity` / `Duplicate login`：
   - 优先判定故障发生在 `pre-ClientIdentity` 阶段
   - 说明问题不是"登录后互踢"，而是"根本没进入服务端身份阶段"

5. 如果故障期间能看到 `ClientIdentity`，并伴随 `Duplicate login`：
   - 才允许把"双端登录冲突"列为候选原因

6. **【补充】`serverCloud available: false` 的特殊分支：**
   - 如果 boot 日志显示 serverCloud 为 `false`，且 `RegisterRedeemCodesFromConfig()` 在此之后立即执行，必须追查该函数在 serverCloud 不可用时的行为。
   - 如果该函数在 serverCloud 不可用时挂起或抛出未捕获异常，这就是服务端实例进入不健康状态的直接原因。

### 输出

必须给出一张时间线：

1. 客户端开始等待的时间
2. 服务端实例开始启动的时间
3. `ServerReady` 是否发出
4. 恢复时刻与实例重启/替换是否相关
5. 故障窗口内是否曾进入 `ClientIdentity`
6. 客户端 120s 超时后的实际行为

---

## 阶段 C：检查服务端启动链是否脆弱

### 目标

确认是否存在"不是语法错误，但任何共享模块异常都会让服务起不来"的结构性问题。

### 必查链路

按以下顺序检查顶层加载链：

1. `scripts/server_main.lua`
2. `scripts/network/BlackMerchantHandler.lua`
3. `scripts/systems/LootSystem.lua`
4. `scripts/config/EquipmentData.lua`
5. `scripts/config/GameConfig.lua`
6. `scripts/network/RedeemHandler.lua`

### 必做项

1. 标出 `server_main.lua` 顶层所有硬 `require`。
2. 标出哪些模块没有 `pcall` 防护。
3. 标出哪些模块在顶层就会访问外部依赖或共享状态。
4. 标出哪些模块只是业务期才需要，却被放进启动链。

### 当前高优先级怀疑点

#### 1. 黑市链路误带起掉落系统

- `BlackMerchantHandler.lua` 顶层 `require("systems.LootSystem")`
- 但真正调用 `LootSystem.CreateSpecialEquipment()` 只在少数购买/补偿分支
- **风险评估**：`LootSystem` 是重型共享模块，启动时若读取 `EquipmentData` 或 `MonsterData` 出现任何运行时异常，无 pcall 保护的 require 会直接炸掉服务端。改懒加载可切断这条不必要的启动期耦合。

要求执行 AI 判断：

1. 这是否可以改为懒加载
2. 这条依赖链是否把第五章共享代码错误拖进了服务端启动路径

#### 2. 启动尾部主动写 serverCloud

- `server_main.lua` 启动末尾直接执行 `RedeemHandler.RegisterRedeemCodesFromConfig()`
- 该函数会遍历兑换码并写入 `serverCloud`
- **风险评估**：serverCloud 是外部依赖，启动时不保证立即可用（云原生环境常见）。若 serverCloud 在启动前几秒未就绪，此写操作可能失败、挂起、或抛出未捕获异常，导致整个实例进入不健康状态。这是间歇性故障的高度可疑点。

要求执行 AI 判断：

1. 该逻辑是否会导致间歇性启动失败
2. 是否应改成：
   - `pcall` 包裹
   - 延迟执行
   - 后台任务
   - 或启动失败不阻断服务

### 输出

必须产出"启动链风险表"，至少包含：

- 模块名
- 是否顶层加载
- 是否访问外部依赖
- 是否必须在启动时加载
- 是否建议改成懒加载/降级

---

## 阶段 D：验证"本次改动是否可能间接触发"

### 目标

避免只看 `server_main.lua` 是否修改过，而漏掉共享模块回归。

### 必做项

1. 拉出最近两天影响服务端启动链的变更文件。
2. 不只看"服务端文件"，还要看：
   - `scripts/systems/LootSystem.lua`
   - `scripts/config/EquipmentData.lua`
   - `scripts/config/GameConfig.lua`

3. 判断这些改动是否会被服务端启动链间接加载。
4. **重点检查顶层执行语句**：纯数据定义改动（只有 table 赋值）风险极低，但如果有顶层函数调用、全局 table 修改等，必须列入候选原因。
5. 如果会被间接加载且含顶层执行语句，必须把它们列入"可能导致启动不稳定"的候选原因。

### 输出

1. 一张"本次改动 → 服务端启动链影响图"
2. 结论说明：
   - 直接影响
   - 间接影响
   - 无影响（需明确说明该改动为纯数据定义）

---

## 阶段 E：验证"恢复是否与平台自愈/实例替换相关"

### 目标

解释"为什么长时间不恢复，却在排查期间恢复"。

### 必做项

如果拿得到平台侧信息，检查：

1. 恢复时是否发生实例重启
2. 恢复时是否发生新实例替换
3. 恢复时是否发生匹配重新分配
4. 恢复时是否有 `serverCloud` 或外部依赖恢复

如果拿不到平台侧信息，也要基于现象给出最可能解释。

补充约束：

1. 如果故障窗口内双端能够共存，执行 AI 必须优先给出 `pre-ClientIdentity` 解释。
2. 只有在拿到 `ClientIdentity` / `Duplicate login` 证据后，才允许继续追"双端互踢"方向。
3. "两次都在 AI 排查期间恢复"最合理的解释是**平台检测到实例不健康后替换了实例**，不能归为巧合。如果平台侧日志不可达，必须标注为"高概率推断，置信度受限于平台侧证据缺失"。

### 允许的结论

可以给出：

- "高概率是平台自愈/实例替换"
- "高概率是外部依赖恢复"
- "高概率是服务端启动链间歇性失败"

不允许给出：

- "没有证据，但应该只是正常等待"

---

## 五、执行 AI 的决策分支

### 分支 1：确认是构筑问题

如果确认 manifest 缺少 `entry_server` 或 `CODE_VERSION` 未递增：

1. 立即定性为 P0
2. 停止继续猜服务端代码
3. 输出修复动作：
   - 重新构筑双入口
   - 递增 `CODE_VERSION` 强制替换实例
   - 校验 manifest
   - 重新发布

### 分支 2：确认是服务端启动失败

如果 boot 日志显示在启动期崩溃或中断：

1. 立即进入热修评估
2. 优先改启动链，不先碰业务逻辑
3. 候选热修必须至少包含：
   - `BlackMerchantHandler` 对 `LootSystem` 改懒加载
   - `RegisterRedeemCodesFromConfig()` 改降级执行
   - `server_main` 顶层/启动期加 boot-fence 日志和 `pcall` 护栏

### 分支 3：确认是平台/匹配层问题

如果服务端已成功启动但客户端仍长期拿不到连接：

1. 定位为平台可达性问题
2. 代码层仍需考虑增加观测点
3. 补做：
   - 客户端连接阶段埋点
   - 服务端实例 ready 埋点
   - `ClientIdentity` / `VersionHandshake` / `ServerReady` 时间点日志

---

## 六、若进入热修，优先级顺序

执行 AI 不得一上来大重构，必须按以下顺序做最小止血：

1. 给 `server_main` 启动链补完整 boot 日志（先有观测，再做止血）
2. **把 `BlackMerchantHandler` 对 `LootSystem` 的顶层依赖改懒加载**（优先于兑换码，因为 LootSystem require 失败会使服务端根本到不了兑换码注册这一步）
3. 给启动尾部的兑换码注册加 `pcall` 和降级（保护外部依赖访问）
4. 只在必要时再处理共享模块

禁止先做：

1. 大规模服务化重构
2. 无证据地改客户端等待 UI
3. 无证据地回滚整批第五章改动

---

## 七、执行 AI 的交付要求

最终必须交付以下内容：

### 1. 结论摘要

只允许三类之一：

1. 构筑/发布问题
2. 服务端启动失败
3. 平台/匹配层问题

### 2. 证据列表

必须带：

1. manifest 证据
2. 服务端 boot 日志证据（及 boot 标记是否真实存在于代码的 grep 结果）
3. 客户端连接日志证据（含 120s 超时后行为）
4. 启动链依赖证据
5. `CODE_VERSION` 版本一致性证据

### 3. 风险清单

至少说明：

1. 哪些模块对服务端启动链耦合过深
2. 哪些启动期外部依赖未做降级
3. 哪些判断因为拿不到日志而只能给高概率结论
4. 客户端超时无重连是否为已知缺陷，是否需要独立修复

### 4. 后续动作建议

按优先级给出：

1. 立即修复项
2. 本周应完成的稳定性改造
3. 可延后的结构优化

---

## 八、禁止事项

1. 禁止仅凭静态 review 就宣布"本次改动无关"。
2. 禁止把"排查期间恢复"直接归为巧合。
3. 禁止把 1 小时、8 小时的连接等待解释成正常冷启动。
4. 禁止没有 manifest 和 boot 日志就宣布"服务器没问题"。
5. 禁止在未 grep 确认 boot 标记存在的情况下，把"日志中查不到 boot 标记"判定为启动失败。

---

## 九、执行起点

执行 AI 从以下顺序开始：

1. 先取线上 manifest，并核查 `CODE_VERSION`
2. grep `server_main.lua` 确认 boot 标记是否真实存在
3. 再取服务端 boot 日志
4. 再画服务端启动依赖链
5. 最后才决定是否热修

如果第 1 步和第 3 步拿不到，必须明确写：

`当前无法拿到线上构筑或启动日志，结论仅为高概率推断，不足以排除线上启动故障。`

## 三、黑市存档链审计

### 3.1 核心结论

黑市把每笔交易拆在**两个一致性域**上：money 域（仙石/库存，BatchCommit 原子、**不随整档回档**）+ save 域（物品/灵韵，可被整档覆盖/回退），加三条"**读到即删、先删后用**"的 WAL。而唯一写档出口 `WriteSaveBack` 是**全服务端唯一不持 slot 锁的直写方**。回档发生时 money 域永不跟随：**买入回档=玩家钱货两空，卖出回档=玩家无限刷**。

### 3.3 P1/P2 择要

- **P1**：Exchange 无任何内存锁（灵韵→仙石可并发双花，且服务端扣减可被客户端快照抹掉=免费兑换）；Buy/Sell/Exchange 三链互不互斥互相丢写；WAL 无 slot 归属（跨槽位串货+洗白）；Buy 背包溢出静默吞货（返回值被忽略，仙石已扣）；成功买入清 WAL 是**整表清空**（把其他待补偿条目一起删掉）；OnDisconnect 释放 per-item 锁但异步链还活着（TOCTOU 窗口重开）。
- **P2**：buyActive_/sellActive_/recycleRunning_ 无 TTL（回调挂起即卡死到重启）；Sell max_stock 校验非原子；SellGuard 死代码误导审计。
- **正面**：RecycleTick（自动收购）是全文件质量最高部分——quota 原子占坑、失败节流、双 cap 熔断，且不碰 save_{slot}，免疫回档。

## 五、跨域架构结论（三条线共同的根）

1. **三个一致性域互不同步**：save 域（可被整档覆盖/回退）、money 域（仙石/积分，原子但永不回滚）、quota/独立 key 域（每日额度、里程碑、兑换记录，免疫回档）。所有"钱货两空 / 重复刷 / 已领没到账 / 永不能再领"都是同一笔交易的两半落在不同域、回档只回滚一半的产物。
2. **服务端直写 vs 客户端全量快照无 CAS** 是放大器：任何服务端写进 save 的东西，只要客户端内存没同步，下一次客户端保存就抹掉。金币灵韵类"客户端内存双活"的数据幸存，物品/pity/标记类"只有 save 一条命"的数据丢失——这条规律解释了全部"部分能存部分不能存"的玩家反馈。
3. **两套防护规范都存在标杆实现，但覆盖不完整**：
   - slot 锁：8/10 handler 有，黑市 WriteSaveBack 和 XianyuanChest 没有；
   - 写前备份：黑市/Wubao/TriggeredBattle/Redeem 有，Trial/DaoTree/SealDemon/Event 没有；
   - connKey/userId 内存标志 TTL：只有 Trial（T3 后）和 itemSellLock_ 有，Event/SkinShop/SwordWall/BM buyActive_ 等没有；
   - 金标准"先落盘后回包"：Wubao/TriggeredBattle 对，XianyuanChest Pick、SwordWall 错。
4. **TriggeredBattleHandler 是全场唯一 A~F 全绿的 handler**，可作为后续统一收口的模板（幂等缓存 + release() 封装 + 先落盘后回包）。

## 六、下一批修复优先级建议（不开发，供出执行指令）

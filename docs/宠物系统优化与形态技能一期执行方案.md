# 宠物系统优化与形态技能一期执行方案

## 1. 文档定位

这是一份一期执行案，不是最终数值定稿。

本案解决两件事：

1. 先把当前宠物系统里已经确认的高风险问题收口
2. 在不放大旧问题的前提下，上线“三形态手动切换”能力

本案以当前代码为基准，默认后续所有实现都遵守以下原则：

- 先止血，再扩展
- 不做破坏性存档迁移
- 新功能必须可灰度、可回退、可审计
- 形态系统与现有 10 格技能书系统解耦

## 2. 一期目标

一期目标不是把宠物系统重做，而是把它从“高风险可运行”推进到“可继续扩展”。

本期完成后，应达到以下状态：

1. 宠物存档不再因为缺字段或异常数据造成技能、形态、外观被错误清空
2. `coreData.pet` 有服务端边界校验，至少能挡住明显非法值
3. 宠物高价值操作拥有统一持久化策略
4. 宠物新增三形态，支持手动切换、存档、加载、回退
5. 狂暴形态不会演化成“故意送狗换超长玩家常驻增益”的玩法漏洞

## 3. 一期前置问题

以下问题必须纳入本期，不建议只做形态功能而跳过。

### 3.1 P0 存档与线上安全

1. `DeserializePet()` 先清空 `pet.skills`，再按存档回填。缺 `skills` 或异常 `skills` 会造成永久清空风险。
2. `SaveWriteService` 目前没有 `coreData.pet` 专项校验，宠物等级、阶级、技能、外观、后续形态字段都缺少服务端白名单边界。
3. 宠物加载与账号级外观加载存在顺序问题，首轮属性重算结果不稳定。

### 3.2 P1 规则一致性

1. `攻速强化`、`减伤强化` 现在仍属于“已配置、已投放、未完整接战斗结算”的状态。
2. 宠物给主人加成 `petOwnerBonuses` 由 `Pet:RecalcStats()` 写入，但当前宠物死亡时没有明确清空路径。形态系统如果要改死亡收益，这个问题必须先收。
3. 宠物高价值操作缺统一保存策略，当前只有“精研升级”明确发了 `save_request`。

### 3.3 P2 扩展性与维护性

1. `PetPanel.lua` 体量过大，后续再直接往里塞形态逻辑，维护成本会继续恶化。
2. 宠物拾取仍是线性扫描，当前能跑，但会拖累后续扩展。

## 4. 形态系统一期结论

### 4.1 基本形态边界

一期按以下边界落地：

- 三形态不占技能格
- 三形态不消耗技能书
- 三形态不走随机学习
- 三形态作为宠物独立系统能力存在
- 玩家手动切换
- 默认形态始终存在

形态列表：

- `normal`：默认形态，初始可用
- `battle`：战斗形态，`pet.tier >= 2` 解锁
- `guard`：守护形态，`pet.tier >= 3` 解锁
- `rage`：狂暴形态，`pet.tier >= 5` 解锁

### 4.2 切换规则

一期默认按保守方案执行：

- 手动切换
- 不消耗资源
- 切换后立即生效
- 默认限制为非战斗切换

说明：

- 非战斗切换是一期建议，不是永远限制
- 如果后续一定要做战斗中切换，单独作为 `v2` 处理，并补冷却、打断和套利约束

### 4.3 形态定位

- `battle`：稳定输出形态，偏 Boss 战和长线输出
- `guard`：生存形态，偏高压图、持续战、宠物站场
- `rage`：高风险爆发形态，偏短时收益和死亡补偿

## 5. 形态效果草案

这部分先锁方向，不在本案里锁死全部细节数值。

### 5.1 战斗形态

已确认方向：

- 宠物最大攻击力提升
- 灵噬易伤持续时间延长

当前讨论版本：

- 最大攻击力 `+20%`
- 灵噬易伤持续 `4s -> 8s`

实现含义：

- 攻击提升作用于宠物最终攻击值
- 灵噬只延长持续，不改当前基础易伤比例和基础 CD

### 5.2 守护形态

已确认方向：

- 宠物最大生命值提升
- 宠物复活时间缩短

当前讨论版本：

- 最大生命值 `+15%`
- 复活时间缩短，具体秒数待定

实现含义：

- 生命提升作用于宠物最终最大生命值
- 复活缩短只改宠物死亡空窗，不额外赠送别的玩家收益

### 5.3 狂暴形态

已确认方向：

- 宠物最大攻击力提升
- 宠物最大生命值降低
- 宠物死亡后，触发“替换当前犬魂狂暴”的狂暴收益

当前讨论边界：

- 最大攻击力 `+30%`
- 最大生命值 `-15%`
- 死亡后替换当前犬魂狂暴
- 不叠加旧犬魂狂暴
- 不延长到宠物复活
- 只走正常固定持续时间覆盖
- 在现有 `+30%攻速 +30%移速` 基础上，额外增加 **+15%全伤害**

这条边界很重要：

- 狂暴形态不能做成“宠物一死，玩家一直爽到宠物复活”
- 也不能做成“新旧犬魂狂暴同时叠加”

否则会直接把玩法带向“故意送狗换玩家爆发”的错误方向

## 6. 必做规则修正

这些修正应视为形态系统上线前置，不建议拆开。

### 6.1 存档安全修正

1. `DeserializePet()` 改为防御式恢复：
   - 缺 `skills` 不清空现有技能
   - 非法 `skills` 逐项过滤，不整表归零
   - 缺 `formId` 回退到 `normal`
   - 非法 `formId` 回退到 `normal`
2. `SaveWriteService` 增加 `coreData.pet` 专项校验：
   - `tier`
   - `level`
   - `exp`
   - `skills`
   - `appearance`
   - `formId`
3. 对宠物异常存档命中校验时打结构化日志，方便线上追查

### 6.2 死亡收益修正

1. 宠物死亡时，必须明确清理 `petOwnerBonuses`
2. 宠物复活后，再通过正常 `RecalcStats()` 恢复给主人加成
3. 狂暴形态死亡收益改造时，不能依赖“当前状态碰巧被刷新”，必须有显式清理和显式恢复

这是防止狂暴形态上线后出现隐藏收益泄漏的关键点。

### 6.3 高价值操作保存修正

以下操作纳入统一持久化策略：

- 突破
- 学技能
- 技能升级
- 删除技能
- 切换外观
- 切换形态
- 高级外观解锁

执行建议：

- 高价值消耗走即时 `save_request`
- 高频低风险操作可接 `SaveSession` 做会话收口

### 6.4 规则落地修正

`攻速强化`、`减伤强化` 二选一处理：

1. 立即接入真实战斗结算
2. 在接入前暂停投放和展示

不建议继续维持“玩家能拿到，但实际不完整生效”的状态。

## 7. 技术方案

### 7.1 配置层

新增独立配置文件：

- `scripts/config/PetFormConfig.lua`

职责：

- 形态基础配置
- 解锁阶级
- 形态说明文案
- 数值参数
- 后续灰度或开关字段

不建议把形态塞进 `PetSkillData.lua`，因为它不属于技能书体系。

### 7.2 实体层

核心改动放在：

- `scripts/entities/Pet.lua`

新增内容：

1. `self.formId = "normal"`
2. `Pet:GetCurrentFormId()`
3. `Pet:IsFormUnlocked(formId)`
4. `Pet:CanSwitchForm(formId)`
5. `Pet:SwitchForm(formId)`
6. `Pet:GetFormConfig(formId)`

属性接入原则：

- 基础属性
- 同步率
- 技能被动
- 每级成长
- 最后叠加形态修正

这样可以把形态固定在最终乘区，降低对旧技能逻辑的破坏。

### 7.3 玩家与事件层

需要改造：

- `scripts/entities/Player.lua`
- `scripts/systems/GameEvents.lua`

处理内容：

1. 现有 `ApplyPetBerserk(duration, atkSpeedBonus, moveSpeedBonus)` 需要扩展，支持狂暴形态的额外攻击收益
2. 当前 `pet_death` 事件要改成按“当前形态”决定是否触发默认犬魂狂暴，还是触发狂暴形态专用版本
3. 死亡和复活都要显式处理 `petOwnerBonuses` 和相关属性刷新

建议不要继续把这类规则散在多个 UI 入口里做隐式触发。

### 7.4 存档层

涉及：

- `scripts/systems/save/SaveSerializer.lua`
- `scripts/systems/save/SaveLoader.lua`
- `scripts/network/SaveWriteService.lua`

执行要求：

1. `SerializePet()`/`DeserializePet()` 增加 `formId`
2. 旧档无 `formId` 时默认 `normal`
3. `SaveLoader` 调整恢复顺序或增加统一重算阶段，避免首轮属性错误
4. 服务端校验 `formId` 的合法性与解锁条件

### 7.5 UI 层

一期不建议新增顶级 tab。

建议先把形态入口放进宠物信息页卡片中，展示：

- 当前形态
- 已解锁形态
- 每个形态说明
- 切换按钮

但在真正加 UI 前，建议先把 `PetPanel.lua` 至少做职责拆分。

最低拆分建议：

- `PetPanelInfo.lua`
- `PetPanelSkills.lua`
- `PetPanelAppearance.lua`
- `PetPanelProgression.lua`
- `PetPanelForms.lua`

主面板只保留容器、tab 和公共刷新入口。

## 8. 开发阶段

### Phase A：止血与底座

目标：

- 修线上风险
- 建形态系统接入底座

任务：

1. 修 `DeserializePet()` 的技能和形态恢复策略
2. 补 `coreData.pet` 服务端校验
3. 补宠物死亡时 `petOwnerBonuses` 清理
4. 调整宠物与账号外观加载后的统一重算顺序
5. 补宠物关键链路回归测试

交付标准：

- 不引入新功能 UI
- 先把旧风险收住

### Phase B：形态数据与运行时

目标：

- 让三形态在运行时能合法存在、切换、保存、恢复

任务：

1. 新增 `PetFormConfig.lua`
2. `Pet` 接入 `formId`
3. `Pet:RecalcStats()` 接入形态乘区
4. `GameEvents` 接入形态相关死亡收益分支
5. `Player` 接入新的犬魂狂暴收益结构
6. 存档接入 `formId`

交付标准：

- 形态不依赖 UI 也能通过测试驱动跑通

### Phase C：UI 与交互

目标：

- 玩家可见、可切、可理解

任务：

1. 宠物信息页增加形态卡片
2. 接入切换提示、锁定提示、解锁提示
3. 切换后刷新宠物面板、角色属性面板、战斗提示
4. 接入统一保存触发

交付标准：

- 玩家能完成完整闭环操作

### Phase D：灰度与上线

目标：

- 低风险上线

任务：

1. 先内部验证
2. 再灰度配置开启
3. 观察日志、异常存档命中、形态使用率、宠物死亡率
4. 视数据决定是否继续开放战斗中切换或追加数值微调

## 9. 测试清单

至少覆盖以下内容：

### 9.1 存档与加载

1. 旧档无 `formId` 正常加载为 `normal`
2. 缺 `skills` 不清空原有技能
3. 非法技能项被过滤，但合法技能保留
4. 非法 `formId` 回退到 `normal`
5. 非法 `tier/level/exp` 被服务端校验拦截或回退

### 9.2 形态解锁与切换

1. 未到阶级不能切对应形态
2. 到阶级后可切换
3. 非战斗切换限制生效
4. 切换后 `atk/maxHp/def` 按预期变化
5. 切换后立即存档，再读档，形态不丢

### 9.3 狂暴形态

1. 狂暴形态死亡时，只触发替换后的犬魂狂暴
2. 不与旧犬魂狂暴叠加
3. 不覆盖到复活，只按固定时长结束
4. 结束后玩家属性完全回退
5. 宠物死亡期间不残留 `petOwnerBonuses`
6. 宠物复活后给主人加成正确恢复

### 9.4 回归与性能

1. 宠物普攻、灵噬、突破、学技能、删技能、外观切换不回归
2. `攻速强化`、`减伤强化` 的规则状态与实现一致
3. 自动拾取不会因形态系统接入产生额外高频扫描

## 10. 上线风险与约束

### 10.1 线上风险点

1. 存档修正和形态接入不能一起无测试直上
2. 狂暴形态最容易引发死亡收益套利，必须带日志观察
3. UI 改动不应直接绕过服务层去写宠物状态

### 10.2 回退约束

本期必须保证以下回退能力：

1. 关闭形态系统后，旧宠物至少能安全回退到 `normal`
2. 服务端校验异常时，不应再出现“静默写坏数据”
3. 即使临时关闭狂暴形态，默认犬魂狂暴链路仍能独立工作

## 11. 当前待拍板参数

执行案先锁框架，不先锁死这些细节：

1. 守护形态复活时间缩短到多少秒
2. 狂暴形态额外攻击收益的具体数值
3. 狂暴形态固定持续时间是否继续沿用当前 `10s`
4. 战斗形态是否只延长灵噬持续，不改灵噬 CD
5. 一期是否严格限制为非战斗切换

## 12. 一期建议结论

先按以下顺序推进：

1. 先修存档安全、死亡收益清理、高价值操作保存
2. 再接形态底座和运行时逻辑
3. 最后接 UI、灰度和数值微调

如果跳过第 1 步，直接做三形态，后面所有数值讨论都会被存档、回档、异常收益和 UI 耦合问题反复打断。

## 13. 代码审计评估（2026-06-07）

以下为针对当前代码基准的审计结论，非理论推演，全部经代码验证。

### 13.1 致命风险（必须在 Phase A 堵死）

#### 致命 #1：`DeserializePet` 空表清空技能（V7 事故复发风险）

**现状**（`SaveSerializer.lua:630`）：

```lua
pet.skills = {}   -- 无条件清空
if data.skills then
    -- 回填...
end
```

当 `data.skills` 存在但为空表 `{}` 时（网络异常/序列化 bug），技能永久清空——这正是 V7 事故根因。

**修复规则**：
- `data.skills == nil` → 不碰内存中的 `pet.skills`
- `data.skills == {}` (空表) → 视为异常，不清空，打日志
- `data.skills` 有合法内容 → 正常回填（当前逐项过滤逻辑保留）

#### 致命 #2：宠物死亡时 `petOwnerBonuses` 无清理路径

**现状**（`Pet.lua:190-205`）：

`RecalcStats()` 把 `skillPct._ownerFlat` 写入 `player.petOwnerBonuses`。宠物死亡后不再调 `RecalcStats()`，加成残留。

当前不算 bug（固定值不变），但形态系统上线后，若狂暴形态改变了 `_ownerFlat`，死亡后残留等于白送属性。

**修复规则**：
```
宠物死亡 → player.petOwnerBonuses = {} → player._statsCacheFrame = nil
宠物复活 → RecalcStats() 自然恢复
```

#### 致命 #3：宠物高价值操作无即时存档

**现状**：`Pet.lua` 内部零处 `save_request`。PetPanel 中只有**精研升级**一处触发了存档。

突破、普通学技能、删技能、切外观 → 全部无即时存档。

**修复规则**：以下操作必须补 `save_request`：
- 突破成功
- 学习技能
- 高级外观解锁/购买

其他操作（普通升级、切外观、删技能、喂食）经讨论确认不需要即时存档，走会话收口即可。

### 13.2 高风险但非致命

| # | 问题 | 结论 |
|---|------|------|
| 4 | 狂暴形态"送狗换爆发"套利 | 方案边界锁得住（固定10s、不延长、不叠加） |
| 5 | `攻速强化` / `减伤强化` 已配置未接入 | **本期接入**（见 §14 决议） |
| 6 | `formId` 旧档兼容 | 缺值→normal，方案已覆盖 |
| 7 | 形态乘区位置 | 放最后叠加，不干扰旧技能，方案已覆盖 |
| 8 | PetPanel 2175行继续膨胀 | Phase C 开始前必须先拆分 |

### 13.3 方案无问题的部分（确认）

1. Phase 分层顺序正确
2. 形态不占技能格、不走技能书体系
3. 解锁绑定 pet.tier
4. 独立配置文件 PetFormConfig.lua
5. 狂暴形态替换式犬魂狂暴

## 14. 设计决议补充（2026-06-07 讨论确认）

### 14.1 UI 展示位置

形态入口放在**技能 Tab 页内**，位于"固有技能"与"主动技能"之间，新增一栏：

```
固有技能
  犬魂狂暴 / 拾荒伴侣
形态（新增）
  [战斗] [守护] [狂暴]    ← 三个并排按钮，当前激活高亮
主动技能
  灵噬 Lv.3
技能槽
  10/10
```

宠物跟随展示时（场景中的宠物名字/状态条旁），也显示当前形态图标，参考魔兽世界姿态栏的处理方式。

### 14.2 形态对已有技能的增强描述

当某种形态激活时，受影响的已有技能需要在面板中追加一行**形态增强描述**（高亮色），让玩家清楚知道当前形态改变了什么。

**灵噬**（主动技能区域）：

```
灵噬 Lv.3
```

### 14.3 切换规则（含 CD 和边界）

| 规则 | 值 | 说明 |
|------|-----|------|
| 切换 CD | **30 秒** | 切换后 30 秒内不能再切 |
| 生效时机 | **延后生效** | 发起切换后，等当前攻击动作结束再生效 |
| 战斗中可切 | **允许** | 不限制战斗/非战斗，CD 本身就是约束 |
| 死亡中可切 | **禁止** | 宠物死亡期间不能切换形态 |
| CD 持久化 | **不存档** | 重登后 CD 重置，不滥用（30s 够短） |

**边界状态处理**：

1. **切换瞬间宠物正在攻击**：标记 `pendingFormId`，等本次攻击动画/判定结束后再执行切换，刷新属性
2. **切换瞬间宠物死亡**：取消 pending，不生效，不消耗 CD
3. **切换后立即被秒杀**：形态已生效，走新形态的死亡收益逻辑（如狂暴形态则触发增强版犬魂）
4. **CD 内再次点切换**：UI 置灰，提示"冷却中(Xs)"

## 15. 最终实现风险评估（第二轮深度审计）

> 基于代码实际审查，列出每一个可能导致致命问题的实现陷阱。
> 优先级：🔴 致命（会丢数据/崩溃）> 🟠 高危（玩法漏洞/数值异常）> 🟡 中危（体验问题）

### 🔴 致命风险 1：形态切换导致 maxHp 变化时的瞬间死亡

**场景**：
- 宠物当前 hp=120, maxHp=1000（战斗形态）
- 玩家切换到守护形态（maxHp+15%），RecalcStats → maxHp=1150，安全
- 但如果从守护切到狂暴（maxHp-15%），maxHp 可能从 1150 → 850
- 如果当前 hp > 新 maxHp，需要钳位；如果当前 hp 已经很低（如 hp=100, maxHp 无影响），无事
- **真正致命点**：`RecalcStats()` 不修改 `self.hp`，但宠物 hp 不能超过 maxHp
  - 当前代码 `Revive()` 里 `self.hp = self.maxHp` 是安全的
  - 但**常规切换后没有 hp 钳位逻辑**！如果 hp > 新 maxHp，显示会异常（血条溢出），且存档会记录非法 hp 值

**修复方案**：
```lua
-- 切换形态后，在 RecalcStats() 之后加一行：
self.hp = math.min(self.hp, self.maxHp)
```

**严重程度**：如果不加钳位，存档会写入 hp > maxHp 的非法值，下次加载时产生"满血以上"的 bug。

---

### 🔴 致命风险 2：DeserializePet 缺少 formId 反序列化 → 形态丢失

**场景**：
- 玩家选了守护形态 → 存档写入 `formId = "guard"`
- 加载时 `DeserializePet` 没有读取 `data.formId`
- 结果：每次加载都重置为默认形态（战斗），玩家选择"消失"

**当前代码**（SaveSerializer.lua:605+）：
```lua
-- 只恢复了 name/level/exp/tier/skills/appearance
-- 没有 formId 字段
```

**修复方案**：
```lua
-- SerializePet 中加入：
formId = pet.formId or "battle"

-- DeserializePet 中加入：
pet.formId = data.formId or "battle"
-- 然后触发 RecalcStats() 时要基于 formId 计算修正值
```

**严重程度**：不加载 formId = 100%形态丢失，每次启动都重置。

---

### 🔴 致命风险 3：pet_death 时 petOwnerBonuses 未清除 → 死后加成残留

**场景**：
- 守护形态宠物，技能给主人加了 `constitution=50`
- 宠物死亡 → `pet_death` 事件只触发了 `ApplyPetBerserk`
- 但 `petOwnerBonuses = { constitution=50 }` 仍然挂在 Player 上！
- 主人继续享受宠物加成，直到宠物复活并 RecalcStats 重新计算

**当前代码**（GameEvents.lua:440-453）：
```lua
RegisterEvent("pet_death", function()
    local player = GameState.player
    if player and player.alive then
        player:ApplyPetBerserk(10, 0.3, 0.3)  -- 只做了这一件事
        -- ❌ 没有清 petOwnerBonuses
    end
end)
```

**修复方案**：
```lua
RegisterEvent("pet_death", function()
    local player = GameState.player
    if player and player.alive then
        -- 清除宠物给主人的加成
        player.petOwnerBonuses = {}
        player._statsCacheFrame = nil  -- 失效缓存
        player:ApplyPetBerserk(10, 0.3, 0.3)
    end
end)
```

**严重程度**：加成残留 = 数值膨胀。守护形态技能给主人加体质/智慧，死后不清等于永久增益。虽然下次 RecalcStats 会覆盖，但在死亡到复活期间（默认 10s + 复活动画）属性是错的。

### 🟠 高危风险 4：ApplyPetBerserk 签名变更导致 "狂怒形态" 逻辑断裂

**场景**：
- 当前 `ApplyPetBerserk(duration, atkSpeedBonus, moveSpeedBonus)` 只有 3 个参数
- 狂怒形态需要额外传 `bonusDmg`（+15% 伤害加成）
- 如果改为 4 参数：`ApplyPetBerserk(duration, atkSpeedBonus, moveSpeedBonus, bonusDmg)`
- 现有调用点 `GameEvents.lua:443` 只传 3 个参数 → 第 4 个为 nil → 不会崩溃但 bonusDmg 不生效

**修复方案**：使用 options table 或确保所有调用点同步更新：
```lua
-- 方案 A：第 4 参数默认 0
function Player:ApplyPetBerserk(duration, atkSpeedBonus, moveSpeedBonus, bonusDmg)
    self.petBerserkBonusDmg = bonusDmg or 0  -- 兼容旧调用
end

-- 方案 B：只在狂怒形态的 pet_death 分支传入 bonusDmg
```

**严重程度**：不崩溃，但狂怒形态的核心卖点（+15% dmg）可能不生效，变成"无感升级"。

### 🟠 高危风险 5：formId 参与 RecalcStats 的时机问题

**场景**：
- `Pet:RecalcStats()` 被 7 个地方调用（构造、升级、学技能、装备变化、复活、反序列化等）
- 形态修正必须在 `GetSkillBonuses()` 中被纳入计算
- 如果形态修正是在 RecalcStats **之后**单独叠加的（Post-RecalcStats modifier），那每次外部触发 RecalcStats 都会把形态修正冲掉

**修复方案**：
```lua
-- 形态修正必须在 GetSkillBonuses() 内部计算，不能外挂
function Pet:GetSkillBonuses()
    local pct = { ... }
    -- 在最后叠加形态修正
    local form = self:GetActiveForm()
    if form then
        pct.atkSpeedPct = pct.atkSpeedPct + (form.atkSpeedPct or 0)
        pct.dmgReducePct = pct.dmgReducePct + (form.dmgReducePct or 0)
        pct.maxHp = pct.maxHp + (form.maxHpPct or 0)
        -- ...
    end
    return pct
end
```

**严重程度**：如果形态修正不进入 `GetSkillBonuses()` → 属性只在切换瞬间正确，之后任何 RecalcStats 触发都会回退到无形态状态。玩家会看到"切了但过一会属性又没了"。

---

### 🟠 高危风险 6：pendingFormId 与存档竞态

**场景**：
1. 玩家点了切换狂怒形态 → `pendingFormId = "rage"`，等待当前攻击结束
2. 在等待期间，自动存档/手动存档触发
3. 此时 `pet.formId` 还是旧的（"battle"），但 `pendingFormId` 在哪？
4. 如果存档不保存 pending 状态 → 丢失用户操作
5. 如果存档保存 pending 状态 → 加载时需要恢复 pending 并继续切换

**修复方案**：
```lua
-- 存档时，如果有 pending，直接写入 pending 的 formId（视为"已生效"）
-- 因为玩家的意图是明确的，存档 = 确认
SerializePet:
  formId = pet.pendingFormId or pet.formId or "battle"
```

**严重程度**：中高。不处理会导致"我明明切了但重进又回去了"的反馈。

### 🟠 高危风险 7：狂暴形态 pet_death 逻辑需要按形态分支

**场景**：
- 当前 `pet_death` 事件固定给 `+30% atkSpeed, +30% moveSpeed`
- 狂怒形态的犬魂狂暴应该是 `+15% bonusDmg`（不是攻速/移速）
- 如果不分支，狂怒形态死亡后触发的增益跟形态无关

**修复方案**：
```lua
RegisterEvent("pet_death", function()
    local pet = GameState.pet
    local player = GameState.player
    if player and player.alive and pet then
        local formId = pet.formId or "battle"
        if formId == "rage" then
            player:ApplyPetBerserk(10, 0, 0, 0.15)  -- 只加 bonusDmg
        elseif formId == "guard" then
            player:ApplyPetBerserk(8, 0.2, 0.2)  -- 守护形态较温和
        else
            player:ApplyPetBerserk(10, 0.3, 0.3)  -- 战斗形态（当前默认）
        end
    end
end)
```

**严重程度**：不分支 = 狂怒形态的特色完全不体现在犬魂狂暴中。

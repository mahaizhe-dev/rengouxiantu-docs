## 一、背景与问题描述

**现象**：玩家进入游戏后客户端长期停留在"正在连接服务器..."界面（持续 1~8 小时），无法正常进入游戏。  
**自愈特征**：在 AI 介入排查过程中问题自动恢复，玩家可正常进入。  
**重要约束**：自愈是排查的干扰项，需区分"已恢复"与"根因已消除"。

## 四、阶段 B：Boot 标记确认 + 客户端连接逻辑

### B-1 服务端 Boot 标记

**文件**: `scripts/server_main.lua`

经 grep 确认，以下 boot 标记均真实存在于代码中：

| 行号 | 标记字符串 | 含义 |
|------|-----------|------|
| 130 | `[Server-BOOT] Start() called at {timestamp}` | Start() 函数被调用 |
| 136 | `[Server-BOOT] Before SaveProtocol.RegisterAll()` | 注册远程事件前 |
| 139 | `[Server-BOOT] After SaveProtocol.RegisterAll()` | 注册远程事件后 |
| 251 | `[Server] 人狗仙途 服务端已启动` | **服务端最终健康标记** |
| 252 | `[Server] serverCloud available: {true/false}` | serverCloud 可用性快照 |
| 253 | `[Server] CODE_VERSION={5}` | 版本号确认 |

**标记间隙分析**（对排查连接异常至关重要）：

```
BOOT-1 (Start 调用)
  ↓
BOOT-2 (RegisterAll 前)
  ↓  ← 在此区间：若顶层 require 失败，服务端崩溃，不会打印后续任何标记
BOOT-3 (RegisterAll 后)  ← 打到此标记 = require 链安全通过
  ↓  ← 在此区间：订阅大量 C2S 事件
"服务端已启动"          ← 打到此标记 = Start() 主体逻辑全部执行完
"serverCloud available"
"CODE_VERSION=5"
  ↓
RegisterRedeemCodesFromConfig()  ← 此调用在"已启动"标记之后
```

**关键诊断规则**：
- 若日志中只有 BOOT-1，无 BOOT-2 → 崩溃发生在 SampleStart() 调用中
- 若日志中有 BOOT-1、BOOT-2，无 BOOT-3 → SaveProtocol.RegisterAll() 内部崩溃
- 若日志中有 BOOT-3，无"已启动" → 某个 C2S 事件订阅或 Handler 初始化崩溃
- 若日志中有"已启动"，且 `serverCloud available: false` → RedeemHandler 注册会在此之后崩溃

### B-2 客户端连接超时行为

**文件**: `scripts/client_main.lua`（第 32-154 行）

```lua
local CONNECT_TIMEOUT = 120.0  -- 秒：匹配系统首次连接约需60秒，留足余量
```

**完整客户端行为时序**：

| 时间节点 | 客户端行为 |
|---------|---------|
| 0 秒 | Start() 调用，尝试立即初始化网络 |
| 5 秒 | 若未连接，显示 "正在连接服务器..." UI 提示 |
| 120 秒 | 超时触发，显示 "连接超时，等待重试..." |
| 120 秒后 | **不切换到单机模式**，保持 NETWORK 模式，允许后续 ServerReady 触发 |

**关键特性**：客户端超时后不放弃，等到服务端恢复（ServerReady 事件）时仍然接受连接。这直接解释了现象——平台替换服务端实例后，客户端无需重启即可自动恢复。

### C-2 高风险点详细说明

#### 高风险点 #1：BlackMerchantHandler → LootSystem 顶层 require 无 pcall

**证据**：
- `server_main.lua` 第 28 行：`local BlackMerchantHandler = require("network.BlackMerchantHandler")`（无 pcall）
- `BlackMerchantHandler.lua` 第 14 行：`local LootSystem = require("systems.LootSystem")`（无 pcall）
- `XianyuanChestHandler.lua` 第 11 行：`local LootSystem = require("systems.LootSystem")`（无 pcall）

**崩溃路径**：
```
LootSystem.lua 或 EquipmentData.lua 中任何语法错误 / 顶层运行时异常
→ require("systems.LootSystem") 抛出错误
→ require("network.BlackMerchantHandler") 抛出错误（未被 pcall 捕获）
→ server_main.lua 顶层执行中断
→ Start() 从未被调用
→ 日志中只有 BOOT-1（若 Boot-1 在 require 之后）或完全没有日志
→ 服务端静默崩溃，客户端永远等待
```

**当前防护等级**: 无。对比 DungeonHandler 和 EventHandler 已有 pcall 保护，BlackMerchantHandler/XianyuanChestHandler 是遗留问题。

#### 高风险点 #2：RedeemHandler.RegisterRedeemCodesFromConfig() 内 serverCloud 无 nil guard

**证据**（`RedeemHandler.lua` 第 208-250 行）：

```lua
function M.RegisterRedeemCodesFromConfig()
    local ok, RedeemCodes = pcall(require, "config.RedeemCodes")
    if not ok or not RedeemCodes or not RedeemCodes.CODES then
        print("[Server][Redeem] RedeemCodes config not found, skip.")
        return
    end
    -- ... 遍历 codes ...
    for _, entry in ipairs(codes) do
        local code = entry.code
        if code and code ~= "" then
            local codeKey = "redeem_" .. code
            local codeData = { ... }
            serverCloud:Set(REDEEM_SYSTEM_UID, codeKey, codeData, {  -- ⚠️ 无 nil guard
                ok = function() ... end,
                error = function(errCode, reason) ... end,
            })
        end
    end
end
```

**崩溃条件**: `serverCloud == nil`（平台偶发：serverCloud 服务尚未就绪）

**特殊之处**: 这个崩溃发生在 `print("[Server] 服务端已启动")` **之后**。因此：
- 日志中会出现"已启动"标记
- 但 Start() 函数以抛出异常结束
- 引擎是否能继续正常处理客户端连接，取决于引擎对 Start() 异常的处理方式

## 八、连接异常"自愈"原因推断

结合以上分析，"排查期间自动恢复"最可能的原因：

**结论：平台层自动替换服务端实例**

推断依据：
1. 客户端 120 秒超时后保持 NETWORK 模式，接受 ServerReady 事件（已验证）
2. 平台有健康检查机制，检测到服务端长时间无响应后自动重启新实例
3. 新实例在 serverCloud 就绪后启动，`serverCloud available: true`，Start() 正常完成
4. 客户端收到 ServerReady 事件，自动恢复连接，无需用户操作

这一模式与文档阶段 E 描述的"排查期间恢复 = 平台自愈"高度一致。

**是否说明根因已消除？不一定。**  
结构性风险（高风险点 #1 和 #2）仍然存在，下一次触发相同条件时，现象可能重现。

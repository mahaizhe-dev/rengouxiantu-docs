## 1. 系统概述

仙缘宝箱是章节地图上的固定点位宝箱系统。当前代码已实装前五章，每章 4 个，共 20 个，分别对应：

- `constitution` 根骨
- `fortune` 福源
- `wisdom` 悟性
- `physique` 体魄

每个宝箱只检查自己的对应属性。玩家满足门槛后，可进行 `3.0` 秒读条，完成后获得该章节档位的 `3 选 1` 奖励。

当前系统还有两个关键判定：

- 宝箱不是在“读条完成”时立刻记为已开，而是在玩家真正选中 1 件奖励并成功入包后，才记为已开启。
- 若奖励面板关闭而未选择，待选奖励会持久化保存，不会直接丢失。

## 4. 奖励生成逻辑

当前奖励由 `LootSystem.GenerateXianyuanReward(tier, quality, guaranteedSubStat)` 生成。

### 4.1 生成步骤

1. 从 `XianyuanChestConfig.SLOT_POOL` 中随机抽取 3 个不重复槽位。
2. 每个槽位调用 `EquipmentUtils.CalcSlotMainStat(slot, tier, qualityMultiplier)` 生成主属性。
3. 调用 `LootSystem.GenerateSubStats(qualityConfig.subStatCount, tier, mainStatType, quality)` 生成随机副属性。
4. 计算保底仙缘副属性值：`floor(tier * qualityMultiplier)`，最小值为 `1`。
5. 若随机副属性中已存在保底属性，则将该条数值提升到 `max(现值, 保底值)`。
6. 若随机副属性中不存在保底属性，则直接用保底属性替换最后一条随机副属性。
7. 当品质为 `cyan` 及以上时，额外生成 `spiritStat`。

### 4.2 结果特征

- 3 件候选装备槽位必不重复。
- 每件装备都带 `isXianyuanReward = true`。
- `cyan` 灵器会设置：
  - `sellCurrency = "lingYun"`
  - `sellPrice = max(1, tier - 4)`

## 5. 客户端交互流程

### 5.1 入口与查看面板

移动端交互键 / `E` 键的当前优先级中，仙缘宝箱位于 NPC、封印之后，福源果之前。

客户端近身检测逻辑：

- 只检查当前章节宝箱
- 只检查未开启宝箱
- 检测范围为 `1.5` 瓦片

命中后先打开查看面板，面板内容包括：

- 对应属性名
- 门槛值
- 当前值
- 奖励预览，例如 `T9 青装 3 选 1`
- 保底说明，例如 `悟性 +1 条`

满足门槛时，按钮为“开始解封”；不满足时显示“未达标”。

### 5.2 读条与取消

`XianyuanChestSystem` 的本地读条时长为 `3.0` 秒。

当前代码中的取消条件是：

- 主动调用 `CancelChannel()`
- 玩家不存在
- 玩家死亡
- 玩家离开交互半径

当前代码没有显式实现以下中断条件：

- 受击即中断
- 在范围内移动即中断

因此文档以代码现状为准，不再写成“移动 / 受击必定打断”。

### 5.3 待选奖励的重开

如果同一个宝箱已经存在待选奖励：

- 再次交互同一宝箱时，客户端会直接重新打开奖励面板
- 不会重新请求服务端生成新奖励

如果玩家已有另一个宝箱的待选奖励：

- 当前代码不允许同时挂两个不同宝箱的待选状态

### 5.4 奖励面板

奖励面板为独立 `XianyuanChestUI`：

- 横向展示 3 张装备卡片
- 每张卡单独“选择”
- 底部有“暂不选择”

“暂不选择”的当前行为只是关闭面板：

- 不会清空待选奖励
- 不会把宝箱标记为已开

## 6. 服务端流程与校验

### 6.1 协议事件

当前协议常量为：

| 方向 | 常量 | 实际事件名 |
|------|------|------------|
| C2S | `C2S_XianyuanChest_StartOpen` | `C2S_XCStart` |
| C2S | `C2S_XianyuanChest_CancelOpen` | `C2S_XCCancel` |
| C2S | `C2S_XianyuanChest_CompleteOpen` | `C2S_XCComplete` |
| C2S | `C2S_XianyuanChest_Pick` | `C2S_XCPick` |
| S2C | `S2C_XianyuanChest_StartResult` | `S2C_XCStartResult` |
| S2C | `S2C_XianyuanChest_Reward` | `S2C_XCReward` |
| S2C | `S2C_XianyuanChest_PickResult` | `S2C_XCPickResult` |

### 6.2 StartOpen

服务端 `HandleStartOpen` 当前会校验：

- `chestId` 是否存在
- 该宝箱是否已开启
- 客户端上传的 `current` 是否达到 `cfg.req`

然后把 `openingChest_[connKey] = chestId`。

当前代码的两个关键现状：

- 服务端门槛校验使用的是客户端上传的 `current`，不是服务端重新读取玩家真实属性。
- 服务端当前没有做位置 / 距离校验。

### 6.3 CompleteOpen

服务端 `HandleCompleteOpen` 当前会校验：

- `openingChest_[connKey] == chestId`
- 宝箱仍未开启
- 客户端上传的 `current` 仍满足门槛

随后处理奖励：

- 若 `saveData.pendingXianyuanRewards[chestId]` 已存在且有 3 件装备，则直接复用
- 否则新生成 3 件候选装备，并立刻持久化到 `pendingXianyuanRewards[chestId]`

此阶段不会把宝箱记为已开。

### 6.4 Pick

服务端 `HandlePick` 当前会校验：

- `pickIndex` 是否在 `1~3`
- 当前连接是否存在该宝箱的内存待选奖励
- 背包是否有空间

成功后：

1. 选中的装备写入服务端背包
2. `saveData.openedXianyuanChests[chestId] = true`
3. 清除 `saveData.pendingXianyuanRewards[chestId]`
4. 回写存档

也就是说，只有选中成功后，宝箱才真正进入“已开启”状态。

### 6.5 断线与待选奖励

断线时 `CleanupConnection(connKey)` 会清掉运行时状态：

- `openingChest_[connKey]`
- `pendingReward_[connKey]`
- `chestActive_[connKey]`

但已写入存档的 `pendingXianyuanRewards[chestId]` 不会丢。

因此当前实际行为是：

- 断线后，本次读条上下文会丢失
- 但如果该宝箱已经生成过待选奖励，玩家再次发起同箱流程并走到 `CompleteOpen` 时，服务端会复用原来的 3 件装备

## 7. 渲染与集成

### 7.1 主循环集成

当前主流程已接入：

- `main.lua` 中初始化 `XianyuanChestSystem.Init()`
- `main.lua` 中创建 `XianyuanChestUI.Create(overlay)`
- 每帧调用：
  - `XianyuanChestSystem.Update(dt)`
  - `XianyuanChestUI.Update(dt)`

### 7.2 渲染表现

当前世界渲染入口为：

`WorldRenderer -> EntityRenderer.RenderXianyuanChests`

实际表现分为两类：

- 未开启：
  - 显示对应属性贴图
  - 带浮动、外发光、旋转光环、属性标题
  - 玩家靠近时显示“按 E 查看”提示
  - 读条时显示进度条
- 已开启：
  - 贴图灰化、半透明
  - 显示“此箱机缘已尽”

### 7.3 交互入口

移动端交互按钮的优先级中，仙缘宝箱位于：

1. NPC
2. 封印
3. 仙缘宝箱
4. 福源果

这意味着玩家按交互键时，若附近没有更高优先级对象，才会进入仙缘宝箱查看面板。

## 8. 存档与版本

### 8.1 当前存档字段

当前仙缘宝箱相关存档字段有两个：

```lua
openedXianyuanChests = {
    -- [chestId] = true
}

pendingXianyuanRewards = {
    -- [chestId] = { item1, item2, item3 }
}
```

其中：

- `openedXianyuanChests` 由 `XianyuanChestSystem.Serialize()` 序列化
- `pendingXianyuanRewards` 在客户端属于透传缓存，核心来源仍是服务端存档

## 9. 代码现状注意点

以下内容是当前代码的真实行为，不是设计预期：

- 服务端当前信任客户端上传的 `current` 属性值。
- 服务端当前没有校验玩家和宝箱的距离。
- 关闭奖励面板不会丢奖励，奖励会保存在 `pendingXianyuanRewards`。
- 宝箱在 `Pick` 成功前都不算真正开启。
- 当前文档中的点位应直接以 `XianyuanChestConfig.CHESTS` 为准，不再写“建议坐标”。

> **版本**：v0.3

## 一、需求整理

### 1.1 原始需求归并

| 模块 | 需求结论 |
|------|------|
| 入口 | 为宠物增加一个**外观选择界面** |
| 基础外观 | 通过**宠物进阶**解锁，每个阶段对应 1 套基础外观 |
| 基础预览 | 当前未解锁的后续进阶外观也要展示，但以**锁定态预览** |
| 基础加成 | **无属性加成**，纯表现 |
| 高级外观 | **账号级解锁**，未来通过仙玉或其他运营方式获得 |
| 高级共享 | 高级外观一旦解锁，**所有角色都可以给自己的宠物使用** |
| 高级加成 | 有很低的添头属性，本期先做 **1 套**，效果为 **宠物攻击 +1%** |
| 加成层级 | 高级外观的属性加成放在**宠物最终属性最外围百分比层** |

### 1.2 默认解释

为避免后续开发歧义，本方案默认按以下规则落地：

| 问题 | 默认方案 |
|------|------|
| 基础外观是角色级还是账号级 | **角色级**。它跟随该角色宠物的进阶状态自然解锁 |
| 高级外观的"解锁"和"装备"是否同层 | **解锁账号级，装备角色级**。装备只决定表现，不决定是否吃加成 |
| 高级外观属性何时生效 | **解锁后立即生效**，与当前是否装备该高级外观无关 |
| 高级外观加成给谁 | **只加宠物，不加玩家本体** |
| 同一时刻能穿几套外观 | **只能装备 1 套**，基础/高级互斥 |
| 进阶后是否能切回旧外观 | **可以**，已解锁的基础外观都可回切 |
| `selectedId` 数据类型 | **字符串**（如 `"pet_base_t0"`），与配置表 ID 一致，可读性好、扩展灵活 |

## 二、现状与目标

### 2.1 当前系统现状

当前宠物系统已经具备以下基础：

- 宠物面板 `PetPanel` 现有 `属性/喂食`、`突破`、`技能` 三个页签。
- 宠物当前有 5 个阶级：`幼犬(0)`、`灵犬(1)`、`妖犬(2)`、`灵兽(3)`、`圣兽(4)`。
- 本期同步扩展 3 个新阶级：`天犬(5)`、`神兽(6)`、`仙兽(7)`，需在 `GameConfig.PET_TIERS` 中追加对应条目。
- 渲染层已经按阶级贴图展示宠物外形，现有贴图为：
  - `Textures/pet_dog_right.png`
  - `Textures/pet_dog_tier1.png`
  - `Textures/pet_dog_tier2.png`
  - `Textures/pet_dog_tier3.png`
  - `Textures/pet_dog_tier4.png`
- 宠物数据目前保存在**角色存档** `saveData.pet` 中，不是账号共享。
- 账号级存储能力已经存在，当前有 `account_atlas` 等独立 key，可复用同类模式。

### 2.2 设计目标

本次外观系统目标：

1. 在不改动现有宠物养成主线的前提下，增加一个清晰的外观选择入口。
2. 基础外观直接复用现有进阶贴图，最小成本完成第一版。
3. 高级外观从第一天开始按**账号级共享**设计，避免后续仙玉售卖时返工。
4. 高级外观属性加成保持极低，只做"有价值但不破坏平衡"的收藏型收益。

### 2.3 非目标

本期**不包含**以下内容：

- 不做宠物染色、局部换装、部位组合。
- 不做多宠物编队或不同宠物种族切换。
- 不做高级外观升星、培养、碎片收集等二次成长。
- 不做玩家角色外观系统联动。

## 三、系统总览

### 3.1 总体结构

```text
宠物进阶成功
  -> 解锁对应基础外观
  -> 外观页签中可选择该阶段及以下基础外观
  -> 更高阶段外观继续以锁定态展示

账号获得高级外观
  -> 账号级解锁记录写入 account_cosmetics
  -> 所有角色的宠物外观页都能看到并使用
  -> 一旦解锁，即对账号内所有角色的宠物生效极低属性加成
```

### 3.2 分类规则

| 分类 | 解锁范围 | 解锁条件 | 是否共享 | 是否有属性 |
|------|------|------|------|------|
| 基础外观 | 当前角色宠物 | 宠物达到对应进阶 | 否 | 否 |
| 高级外观 | 账号 | 仙玉/活动/礼包等 | 是 | 是，且很低 |

## 六、高级外观方案

### 6.1 本期首套配置

本期先做 1 套高级外观：

| 外观 ID | 外观名 | 解锁范围 | 获取方式 | 属性 |
|------|------|------|------|------|
| `pet_premium_chiyan` | 赤焰天犬 | 账号级 | 仙玉/活动占位 | 宠物攻击 +1% |

贴图：`Textures/pet_skin_chiyan.png`（✅ 已就位，256x256 px，透明背景）

**赤焰天犬美术参考**：全身烈焰纹路，赤红+橙金主色调，眼部火焰特效，威严霸气，与基础外观的青绿灵纹形成强烈色系对比。

### 6.2 规则说明

- 高级外观是**账号级收藏**。
- 角色 A 解锁后，角色 B / C / D 的宠物面板都能看到并使用。
- 高级外观不依赖宠物当前阶级；只要账号已解锁，就可以装备。
- 高级外观的属性在**解锁后立即生效**，当前是否装备该外观只影响表现，不影响属性是否开启。
- 高级外观和基础外观互斥，同一时间只能装备一套。

### 6.3 获取方式预留

虽然本期不实现完整获取链路，但配置层需要预留来源字段：

| sourceType | 说明 |
|------|------|
| `xianyu_shop` | 仙玉直购 |
| `event_reward` | 活动奖励 |
| `gift_pack` | 礼包赠送 |
| `gm_grant` | GM/运营发放 |

### 6.4 GM 命令（测试与运营必备）

外观系统必须提供以下 GM 命令，用于测试和运营发放：

| 命令 | 说明 | 示例 |
|------|------|------|
| `/pet_skin unlock <skinId>` | 解锁指定高级外观（写入 account_cosmetics） | `/pet_skin unlock pet_premium_chiyan` |
| `/pet_skin equip <skinId>` | 强制装备指定外观（跳过解锁检查） | `/pet_skin equip pet_base_t7` |
| `/pet_skin reset` | 清空所有高级外观解锁（测试用） | `/pet_skin reset` |
| `/pet_skin list` | 列出当前账号所有已解锁外观 | `/pet_skin list` |

> GM 命令仅在 `GameConfig.GM_ENABLED = true` 时可用（发布前必须关闭）。

## 七、属性结算规则

### 7.1 当前宠物攻击公式

当前宠物攻击大致为：

```lua
rawAtk = petBaseAtk + petGrowthAtk * (level - 1) + playerTotalAtk * syncRate
atkAfterSkill = floor(rawAtk * (1 + skillPct.atk / 100))
```

### 7.2 新增外观加成后的公式

高级外观加成挂在**最外围百分比层**。

由于已确认"**解锁后生效**"，因此最终取值来自**账号已解锁的全部高级外观加成汇总**：

```lua
-- 汇总所有已解锁高级外观加成（有封顶）
local atkPctTotal = 0
for _, skin in pairs(unlockedPremiumSkins) do
    atkPctTotal = atkPctTotal + (skin.bonus.atkPct or 0)
end
atkPctTotal = math.min(atkPctTotal, MAX_PREMIUM_ATK_PCT)  -- 封顶 0.05 (5%)

rawAtk = petBaseAtk + petGrowthAtk * (level - 1) + playerTotalAtk * syncRate
atkAfterSkill = floor(rawAtk * (1 + skillPct.atk / 100))
finalPetAtk = floor(atkAfterSkill * (1 + atkPctTotal))
```

本期首个高级外观为：

```lua
atkPctTotal = 0.01  -- 赤焰天犬 +1%
```

### 7.3 设计原则

- 只影响**宠物最终攻击**，不影响玩家 `GetTotalAtk()`。
- 不参与同步率计算，不反向喂给玩家属性，避免循环放大。
- 基础外观 `appearanceBonus = 0`。
- 多个高级外观并存时，默认按**已解锁项累计**；因此单套数值必须长期维持极低。
- **累计封顶**：`MAX_PREMIUM_ATK_PCT = 0.05`（5%），定义在 `PetAppearanceConfig` 中。防止后续外观堆叠导致数值膨胀。
- 后续如果高级外观要加生命/防御，也走同一层的最终乘区，各属性独立封顶。

## 八、存储设计

### 8.1 角色级存档

宠物当前"装备哪套外观"属于角色级选择，继续放在 `saveData.pet` 下。

建议结构：

```lua
pet = {
    name = "小黄",
    level = 1,
    exp = 0,
    tier = 0,
    hp = 40,
    alive = true,
    skills = { ... },
    appearance = {           -- 新增（v20）
        selectedId = "pet_base_t0"   -- 字符串类型，与配置表 ID 一致
    }
}
```

说明：

- `selectedId` 类型为**字符串**（如 `"pet_base_t0"`、`"pet_premium_chiyan"`）。
- 基础外观是否解锁不单独存布尔值，**直接由 `pet.tier` 推导**。
- 角色级只记录"当前选中的外观 ID"。
- 旧存档无 `appearance` 字段时，回退到 `pet.tier` 对应的默认基础外观，**不需要迁移函数主动写入**。

### 8.2 账号级存档

高级外观解锁记录建议单独使用新 key：`account_cosmetics`。

选择独立 key，而不是并入 `account_atlas`，原因：

- 后续还会有**人物时装**。
- 后续还会有**账号级称号**。
- 三者都属于"账号级外观/表现资产"，与仙图录不是同一语义层。

建议结构：

```lua
account_cosmetics = {
    version = 1,
    petAppearances = {
        pet_premium_chiyan = {
            unlocked = true,
            unlockTime = 1746758400,
            sourceType = "xianyu_shop"
        }
    },
    characterCostumes = {},   -- 预留：人物时装
    accountTitles = {}        -- 预留：账号级称号
}
```

这样做的好处：

- 不污染当前 `account_atlas` 的图录语义。
- 后续如果要做人物时装、账号级称号、坐骑外观，也能继续扩展同一个账号级 cosmetic key。

### 8.3 读档校验

载入角色时做以下兜底：

| 场景 | 处理规则 |
|------|------|
| `selectedId` 为空或 nil | 回退到当前阶级对应基础外观 |
| 选中了未到阶的基础外观 | 回退到当前阶级对应基础外观 |
| 选中了未解锁的高级外观 | 回退到当前阶级对应基础外观 |
| 外观配置被删或 ID 不在配置表中 | 回退到当前阶级对应基础外观 |

### 8.4 存档版本迁移

```lua
-- SaveState.lua
SaveState.CURRENT_SAVE_VERSION = 20  -- v20: 宠物外观系统

-- SaveMigrations.lua
MIGRATIONS[20] = function(data)
    -- 角色级：pet.appearance 不存在时无需主动写入，
    -- nil 的兜底逻辑是回退到 tier 默认外观（§8.3）。
    -- 这是最安全的迁移——不修改任何数据，只升版本号。
    data.version = 20
    return data
end
```

> **账号级** `account_cosmetics` 是独立 key，不走角色存档迁移链。在 `server_main.lua` 的 `HandleLoadGame` 中做 nil 初始化：为空时初始化为 `{ version=1, petAppearances={}, characterCostumes={}, accountTitles={} }`。

## 十一、执行注意事项

### 11.1 高级外观属性何时生效

已确认：

- **解锁后立即生效**。
- 当前是否装备该高级外观只影响显示效果，不影响属性是否生效。

这意味着：

- 高级外观更接近"账号级收藏加成"。
- 如果后续出现多套高级外观，默认按**已解锁项累计**（有封顶，见 §7.3）。
- 因此单套加成必须长期维持极低，避免堆叠膨胀。

### 11.2 高级外观是否允许低阶宠物直接使用

已确认：

- **允许** 0 阶宠物直接使用已解锁高级外观。

含义：

- 账号级高级外观没有宠物阶级门槛。
- 新角色/低阶宠物也可直接展示该外观。

### 11.3 账号级存储 key 选型

定案为：使用独立 `account_cosmetics`。

原因：

- 后续还有**人物时装**、**账号级称号**。
- 这些内容都应挂在统一的账号级表现资产 key 下，不塞进 `account_atlas`。
- 现在单独立 key，后面扩展成本最低。

### 11.4 server_main 加载顺序（关键）

`HandleLoadGame` 中，`account_cosmetics` 必须**先于** pet 属性计算加载完成：

```text
FetchSlots -> 加载 account_cosmetics -> 加载角色 saveData
  -> pet 初始化（读取 appearance.selectedId）
  -> pet:RecalcStats()（此时已能读到 account_cosmetics 的高级外观加成）
```

如果顺序颠倒，pet 属性计算时尚未拿到账号级高级外观加成，导致首次登录属性偏低。

### 11.5 当前代码里的执行注意点

1. `PetPanel.DoBreakthrough()` 当前只有 `PetPanel.Refresh()`，**没有立即触发 `save_request`**。如果基础外观解锁跟突破绑定，建议在突破成功后补即时存档。
2. 账号级新 key 不是只改一处就够，至少要补齐：
   - `server_main.lua` 的 `FetchSlots` / `LoadGame` / `SaveGame`
   - `SaveSystemNet.lua` 的登录态反序列化
   - 账号级系统自身的即时保存入口
3. `Pet:RecalcStats()` 当前最终攻击只考虑基础成长、玩家同步和技能百分比；高级外观的 `+1%攻击` 应插在最终外层，不要混进同步率或玩家攻击。
4. 由于高级外观属性按"解锁即生效"，因此**账号级解锁到账**时也要触发一次 `pet:RecalcStats()` 或 `_statsDirty = true`，否则当前在线角色不会立刻吃到加成。
5. 宠物外观切换本身主要影响表现；如果切换不改变属性，则切换时不强制重算属性，只需刷新渲染和面板即可。

## 十三、验收标准

### 13.1 功能验收

1. 新角色进入游戏后，宠物外观页能看到全部基础外观（T0-T7），其中只有 `幼犬形态` 可用。
2. 宠物突破到 1/2/3/4/5/6/7 阶后，对应基础外观自动变为可用。
3. 未达到的更高阶外观始终可见，但保持锁定态预览。
4. 账号解锁高级外观后，所有角色的宠物外观页都能看到并装备该外观。
5. 装备基础外观时，宠物属性不发生变化。
6. 解锁首套高级外观后，即使当前装备的是基础外观，宠物最终攻击也能正确提升 1%，且仅作用于宠物。
7. 外观切换后重新登录，角色已选外观保持不丢失。
8. 账号高级外观解锁后切换角色重新登录，解锁状态保持不丢失。

### 13.2 存档兼容验收

9. v19 旧存档加载后，宠物正常显示，外观默认为当前阶级基础外观。
10. v17/v18 旧存档经完整迁移链（v17->v18->v19->v20）后正常加载。
11. 旧账号无 `account_cosmetics` 时，登录不报错，外观页不展示任何高级外观。

### 13.3 安全验收

12. 客户端发送未解锁的外观 ID，服务端校验拒绝，回退到默认外观。
13. 多次触发高级外观解锁，幂等处理，不重复计算加成。
14. GM 命令仅在 `GM_ENABLED = true` 时可用。

## 十六、结论

本方案将宠物外观拆为两层：

- **基础外观**（T0-T7）：跟随宠物进阶解锁，负责成长展示，不加属性。
- **高级外观**（赤焰天犬等）：账号级共享，负责长期收藏与极低数值添头（封顶 5%）。

这样可以同时满足：

- 进阶外观可视化成长（8 个阶段完整覆盖 Lv1-Lv160+）；
- 后续仙玉化/活动化的账号级外观扩展；
- 不破坏当前宠物属性和平衡结构；
- 最大限度复用现有贴图、面板和存档体系。

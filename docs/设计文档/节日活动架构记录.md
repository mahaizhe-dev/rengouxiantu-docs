## 三、掉落系统实现

### 3.1 触发链路

```
monster_death 事件 (GameEvents.lua:67)
    │
    ├── 排除条件:
    │   ├── monster.isDungeonBoss → return (副本BOSS由服务端驱动)
    │   ├── monster.isTrainingDummy → return (训练假人不掉落)
    │   └── monster.isChallenge → return (挑战怪奖励由挑战系统发)
    │
    ├── LootSystem.GenerateDrops(monster.dropTable, monster)  ← 常规掉落
    │       └── 完全独立于活动系统，不引用 EventConfig
    │
    └── LootSystem.RollEventDrops(dropResult, monster)  ← 活动掉落(独立追加)
              │
              ├── 第一行: if not EventConfig.IsActive() then return end
              │   └── enabled=false 时直接返回，不修改 dropResult
              │
              └── 写入 dropResult.consumables[itemId] += 1
```

**⚠️ 安全隔离保证**：`GenerateDrops` 位于 `loot/generation.lua`，完全不引用 `EventConfig`。关闭活动 `enabled=false` 只影响 `RollEventDrops`，常规掉落（装备/金币/经验/消耗品）不受任何影响。

### 3.2 两段式掉落判定 (`loot/rewards.lua:41-88`)

```lua
function M.RollEventDrops(dropResult, monster)
    local EventConfig = require("config.EventConfig")
    if not EventConfig.IsActive() then return end  -- ← 总开关，enabled=false 时此处 return

    local event = EventConfig.ACTIVE_EVENT
    local chapterId = GameState.currentChapter or 1
    if not dropResult.consumables then dropResult.consumables = {} end

    -- ═══ 第一段：普通开启物（拨浪鼓）═══
    -- 条件：所有怪物（包括四仙剑），只要 category 存在于 dropRates[chapter]
    -- 查找：event.dropRates[chapterId][monster.category]
    -- 判定：Utils.Roll(chance) → math.random() < chance
    local rates = M.GetEventDropRates(event, monster, chapterId)
    if rates then
        for itemId, chance in pairs(rates) do
            if Utils.Roll(chance) then
                dropResult.consumables[itemId] = (dropResult.consumables[itemId] or 0) + 1
            end
        end
    end

    -- ═══ 第二段：稀有开启物（风车）═══
    -- 前提：BOSS_CATEGORIES[monster.category] == true（非BOSS直接return）
    -- 优先级链（互斥，命中override不再走realm）:
    --   1. event.monsterOverrides[monster.typeId]  ← 四仙剑专属
    --   2. event.realmDropRates[monster.realm]     ← 按境界阶梯
    if not BOSS_CATEGORIES[monster.category] then return end
    -- ... (override > realm 互斥逻辑)
end
```

### 3.7 下架安全性证明

```
enabled = false 时的执行路径：

GameEvents.lua:109 → LootSystem.RollEventDrops(dropResult, monster)
    → rewards.lua:42 → EventConfig = require("config.EventConfig")
    → rewards.lua:43 → if not EventConfig.IsActive() then return end
                             ↑ IsActive() = (enabled == true) = false
                             ↑ not false = true → 执行 return
    → 函数直接返回，dropResult 未被修改
    → 后续 GameState.AddLootDrop(dropResult) 只包含常规掉落内容

结论：活动道具不会出现在任何掉落中，常规掉落完全不受影响。
```

### 5.2 开箱流程 (`EventHandler.HandleOpenFudai`)

```
1. 前置校验
   ├── EventConfig.IsActive() → 活动开关
   ├── fudaiLocks_[userId] → 内存锁（防并发）
   ├── requestCount ∈ {1, 5} → 白名单
   ├── boxType ∈ event.openBoxes → 合法性
   └── Session.AcquireSlotLock() → 槽位锁

2. 读取存档 (serverCloud:Get)
   ├── 校验背包持有数 (BackpackUtils.CountUnlockedItem)
   ├── 校验背包空位
   └── actualCount = min(requestCount, heldCount)

3. 执行 N 次抽取
   ├── EventSystem.RollFudaiReward(pool) → 加权随机
   ├── 保底逻辑:
   │   ├── 自然命中保底目标 → pityCount = 0
   │   └── pityCount >= threshold → 强制替换为保底物品, pityCount = 0
   └── 累计 totalRewards (exp/gold/lingYun/consumables/petSkin)

4. 皮肤处理 (如有)
   ├── 读 account_cosmetics 判断是否已拥有
   ├── 未拥有 → unlockedSkinId（稍后写入）
   └── 已拥有 → dupLingYunTotal 补偿

5. 原子提交
   ├── RemoveFromBackpack(targetItemId, actualCount) ← 扣宝箱道具
   │   └── 返回值 > 0 → 锁定物品导致扣除不足 → 拒绝
   ├── 发放奖励到 saveData (lingYun/gold/consumables)
   ├── 更新 event_data.pity (保底计数器)
   ├── serverCloud:BatchCommit("event_open_fudai")
   │   ├── ScoreSet(userId, saveKey, saveData) ← 存档
   │   └── ScoreAddInt(userId, rankKey, scoreAdd) ← 排行榜
   └── Commit ok 回调:
       ├── WriteRankInfo(userId, ...) ← 排行附加信息
       ├── WritePullRecords(...) ← 全服稀有记录
       ├── account_cosmetics 写入（如有皮肤解锁）
       └── SafeSend → S2C 回复客户端

6. 释放锁
   └── releaseLocks() (fudaiLocks_ + SlotLock)
```

### 5.3 抽奖算法 (`EventSystem.RollFudaiReward`)

```lua
function M.RollFudaiReward(pool)
    local totalWeight = 0
    for _, entry in ipairs(pool) do
        totalWeight = totalWeight + entry.weight
    end
    local roll = math.random() * totalWeight
    local cumulative = 0
    for _, entry in ipairs(pool) do
        cumulative = cumulative + entry.weight
        if roll < cumulative then return entry end
    end
    return pool[#pool]  -- 兜底
end
```

### 5.4 保底机制

```lua
EventConfig.ACTIVE_EVENT.pity = {
    small = { threshold = 60,  targetId = "small_upgrade" },   -- 小箱60次保底
    big   = { threshold = 120, targetId = "big_skin" },        -- 大箱120次保底
}
-- 计数器存储在 saveData.event_data[eventId].pity[boxType]
-- 服务端 HandleOpenFudai 中每次抽取 pityCount++，命中 target 或 >= threshold 时重置
```

### 5.5 皮肤奖励特殊处理

```
奖池条目 rewards 含 type="petSkin" 时：
  1. 读 account_cosmetics.petAppearances[skinId]
  2. 未拥有 → 解锁 (unlockedSkinId)
  3. 已拥有 → 补偿灵韵 (dupLingYun=5000)
  4. 同轮多次抽中 → 第一次解锁，后续按重复补偿
  5. 写入路径: serverCloud → account_cosmetics → petAppearances[skinId] = { unlocked=true }
  6. 容错: 读取失败 → 重试1次 → 仍失败按新解锁处理（幂等写入无害，不发补偿防多发）
```

## 十一、架构设计原则（经验总结）

### 11.1 单点配置原则

所有活动参数集中在 `EventConfig.ACTIVE_EVENT` 一个配置块中，避免散落在多文件。换活动只需替换这个块 + 注册新道具。

### 11.2 客户端掉落 + 服务端权威

- **掉落判定**在客户端（GameEvents → RollEventDrops），因为怪物击杀是客户端事件
- **开箱/兑换/积分**在服务端（EventHandler），保证经济系统安全
- **存档同步**通过 serverCloud:BatchCommit 原子提交

### 11.3 事件数据隔离

`saveData.event_data[eventId]` 结构保证多活动数据互不干扰。旧活动数据保留（不删除），方便后续统计或补发。

### 11.4 幂等容错

- 皮肤写入 `account_cosmetics` 是幂等的（`unlocked = true` 重复写入无害）
- 失败时宁可不发补偿灵韵也不多发（防止利用重试套利）
- 全服记录用 recordKey 前缀去重

### 11.5 锁的层次

内存锁（毫秒级）→ 槽位锁（请求级）→ 原子事务（数据级），三层保护确保并发安全。

## 一、最终方案

藏宝图是第四章以后终局 Boss 产出的长期抽奖凭证。玩家必须回到第四章，从背包打开藏宝图界面使用；服务端在扣图时预先抽出最终奖励，并将玩家传送到第四章八卦海中对应奖励类别的真实宝藏岛。

每张图只授权一座岛上的一个宝箱。开箱时不再随机，只结算使用藏宝图时已持久化的奖励。

### 1.1 已确定规则

| 项目 | 最终方案 |
| --- | --- |

| 黑市价格 | 玩家购买 `12` 仙石，玩家出售 `6` 仙石 |
| 首次入口 | 第四章内从背包打开藏宝图界面 |
| 其他章节 | 只能查看说明，不能扣图、ROLL 或传送 |

### 1.2 地图方案评审结论

最终采用第四章真实地图方案，不创建第二套藏宝海世界：

1. 九座宝藏岛直接散布在第四章八卦海内外，平时肉眼可见。
2. 普通宝藏岛为 `4~7` 格，仙缘大奖岛为 `9~10` 格，均与其他陆地隔海断开。
3. 小地图不显示宝藏标记；玩家即使通过异常移动抵达其他岛，服务端也只允许开启当前 `active.chestId`。
4. 使用藏宝图只改变玩家在第四章地图中的位置和 `worldMode`，不替换地图、不切换章节。
5. 登录恢复时根据服务端 sidecar 的 `active.islandId` 直接传送到权威岛屿出生点，不能依赖客户端保存的坐标选择岛屿。

### 1.4 开发前 P0 前置项

以下三项不完成，不进入万宝琉璃体增量开发：

1. **普通客户端存档必须参与同一槽位写锁。**
   当前 `SaveWriteService` 只检查槽位是否已锁，没有在普通保存期间持有锁。藏宝图 handler 和普通全量保存可能并发覆盖 `save_{slot}`。需要让普通保存从开始写入到回调结束期间持有同一 `Session` 槽位锁。
2. **`account_cosmetics` 改为服务端拥有。**
   当前普通客户端存档会提交并覆盖 `account_cosmetics`，可能回退服务端新解锁皮肤。需要建立统一 `AccountCosmeticsService`，普通角色保存不再直接写客户端提交的账号外观快照。
3. **`account_immortal_bodies` 改为服务端拥有。**
   当前普通客户端存档仍会提交并覆盖 `account_immortal_bodies`。万宝琉璃体属于账号资产，必须使用统一 `AccountImmortalBodiesService`，普通角色保存只读取服务端注入结果，不得用客户端快照覆盖账号仙体。

藏宝图状态不放入客户端可覆盖的角色 DTO，改用独立服务端 sidecar key。

### 1.5 可靠性优先级

藏宝图的异常处理按以下固定顺序决策：

1. **防刷、防重复发奖、防藏宝图回写为最高优先级。**
2. **保证正常流程能够继续，包括原图恢复、归航和开启下一张图。**
3. **玩家补偿和资格保留最低。** 当服务器无法证明某个资格、奖励或扣图结果时，允许失败关闭或损失当前资格，不得为了自动补偿引入重复 ROLL、重复发奖或返还藏宝图的风险。

具体规则：

- 使用请求结果不明时，只能重发同一个 `requestId` 并查询服务端状态，不能创建新 `requestId`。
- 服务端存在 `active` 或 `session` 时，客户端必须恢复该事务，不能开启下一张图。
- 服务端确认不存在 `active/session` 时，客户端立即归航并解除下一张图的流程锁。
- 开箱授权超时只丢弃本地授权并查询状态，不猜测奖励、不补发奖励。
- 发奖结果不明时只读取 `completedRuns.response` 回放，绝不重新 ROLL。
- 主动使用归航法阵视为放弃；服务端清除资格后不返图、不发奖。
- 服务器权威藏宝图槽位快照未成功应用时，客户端禁止传送、开箱和普通存档。

### 1.6 与节日活动存档/服务器结构对比

| 范围 | 节日活动实现 | 藏宝图最终实现 | 结论 |
| --- | --- | --- | --- |
| 普通活动状态 | `event_data` 嵌入角色主存档，客户端加载后缓存并原样回写 | `session/active/pendingReward/completedRuns/requestHistory` 放独立服务端 sidecar | 藏宝进度不允许客户端整包覆盖 |
| 扣除与普通奖励 | 服务端读取主存档，扣开启物、ROLL、发奖励后写回主存档 | 扣图、预抽结果、会话标记和角色落点同批保存 | 均为保存成功后响应；藏宝图额外持久化未领取结果 |
| 响应丢失 | 福袋没有持久化请求结果，重试视为下一次开启 | 同 `requestId` 恢复同一 `runId`；完成结果从 `completedRuns.response` 回放 | 藏宝图不得重新 ROLL 或重复扣图 |
| 请求生命周期 | 活动开启请求没有长期请求历史 | 已完成或放弃的请求写入 `requestHistory` | 归航后的旧使用请求仍不能再次执行 |
| 皮肤账号资产 | 活动主奖励先提交，皮肤再异步写 `account_cosmetics` | 槽位锁后获取账号锁，主存档、sidecar、账号皮肤同一批保存 | 藏宝图不接受“奖励成功但皮肤保存失败” |
| 仙体账号资产 | 节日活动无对应资产 | 主存档、sidecar、`account_immortal_bodies` 同批保存，并记录 `sourceRunId` | 首次和重复均幂等 |
| 独立权威 key | 活动里程碑使用 `evt_ms_*`，普通存档不能覆盖 | 藏宝 sidecar 使用 `treasure_map_state_*` | 复用活动里程碑的服务端权威思路 |
| 普通存档竞争 | 活动依赖槽位锁，但 `event_data` 仍由客户端缓存回写 | 同一槽位锁 + `treasureSessionId` + `treasureRevision` + 藏宝图槽位快照 | 旧保存跨会话或 ABA 回绕均失败关闭 |
| 背包满 | 活动部分奖励可能记录日志后损失 | 发奖前容量校验失败则保留 `active.pendingReward` | 玩家可重试原宝箱，不重抽 |
| 服务重启 | 未保存的抽取结果不存在 | sidecar 保存预抽结果；仅开箱临时 token 丢失，可重新授权 | 重启不改变奖励 |
| 断线中的云写入 | 活动及其他玩法过去会在断线时立即释放槽位锁 | 不再强制释放；由原回调释放，异常时依赖 `30s` TTL | 防止快速重连产生新旧写入重叠 |

最终采用：

1. 复用节日活动“服务端读取、槽位锁、克隆修改、同批提交、保存成功后响应”的事务骨架。
2. 不复用客户端 `event_data` 回写结构。
3. 不复用账号皮肤后置异步保存。
4. 不复用无 `requestId`、无结果回放的短事务模型。
5. 排行榜、全服抽取记录等非核心展示写入不得进入藏宝奖励主事务。
6. 断线不能提前释放仍由异步云写入持有的槽位锁；宁可短暂等待 TTL，也不允许第二写入者并发覆盖。

### 2.3 黑市

现有黑市字段语义：

- `sell_price`：玩家购买时支付。
- `buy_price`：玩家出售时获得。

```lua
treasure_map = {
    buy_price = 6,
    sell_price = 12,
    category = "consumable_mat",
    max_stock = 10,
}
```

| 行为 | 数值 |
| --- | ---: |
| 玩家购买 | `12` 仙石 |

### 3.5 万宝琉璃体

#### 3.5.1 最终定位

万宝琉璃体是藏宝图长期奖池中的极稀有账号资产，也是仙体系统第一次加入仙缘属性成长。

| 字段 | 最终值 |
| --- | --- |
| 名称 | 万宝琉璃体 |
| `bodyId` | `immortal_body_wanbao_liuli` |
| 获取概率 | `0.1%` |
| 首次折合 | `5000` 灵韵 |
| 重复补偿 | `5000` 灵韵 |
| 四维成长 | 生命 `+15/级`、攻击 `+3/级`、防御 `+2/级`、生命回复 `+0.3/级` |
| 仙缘成长 | 福源 `+1/级` |
| 起算等级 | Lv1 即获得 `1` 点 |
| 成长上限 | Lv300 共获得 `300` 点 |
| 解锁范围 | 账号级解锁，各角色独立选择是否装备 |
| 自动装备 | 否 |

配置建议使用可扩展结构，不把福源硬编码为万宝琉璃体特例：

```lua
immortal_body_wanbao_liuli = {
    name = "万宝琉璃体",
    maxHp = 15,
    atk = 3,
    def = 2,
    hpRegen = 0.3,
    xianyuanGrowth = {
        fortune = {
            perLevel = 1,
            startLevel = 1,
            maxLevel = 300,
        },
    },
}
```

仙缘成长公式：

```text
万宝琉璃体福源 = min(max(当前等级, 1), 300) × 1
```

该公式故意与四维成长的 `(等级 - 1)` 不同：

- 四维属性已经有 Lv1 初始值，因此只计算 Lv2 以后成长。
- 现有职业仙缘成长本来就使用 `当前等级 × 每级成长`，Lv1 即生效。
- 万宝琉璃体作为仙体系统首次仙缘成长，沿用现有仙缘属性语义，避免同类属性出现两套起算口径。

#### 3.5.2 等级收益

| 角色等级 | 仙体福源 | 击杀金币 | Boss 灵韵额外收益 | 宠物同步率 |
| ---: | ---: | ---: | --- | ---: |
| 1 | `1` | 每次击杀 `+1` | 未形成 `5` 点档位 | `+0%` |
| 50 | `50` | 每次击杀 `+50` | `20%` 概率额外 `+1` | `+4%` |
| 100 | `100` | 每次击杀 `+100` | `40%` 概率额外 `+1` | `+8%` |
| 120 | `120` | 每次击杀 `+120` | `48%` 概率额外 `+1` | `+8%` |
| 200 | `200` | 每次击杀 `+200` | `80%` 概率额外 `+1` | `+16%` |
| 300 | `300` | 每次击杀 `+300` | 保底 `+1`，另有 `20%` 概率再 `+1` | `+24%` |

仙体福源属于真实 `GetTotalFortune()` 来源，因此正常参与：

- 击杀金币加成。
- Boss 灵韵溢出。
- 宠物同步率。
- 仙缘属性技能伤害。
- 浩气套装等以福源为伤害来源的效果。
- 仙缘宝箱的福源门槛。
- 后续统一读取 `GetTotalFortune()` 的合法系统。

首版不为万宝琉璃体单独增加宠物同步率上限。现有宠物系统允许同步率超过 `100%`，现有数值设计也已覆盖 `300/600/1000` 仙缘属性档位；万宝琉璃体在 Lv300 提供的 `+24%` 属于当前体系内的线性增量，不应在仙体内部制造隐藏衰减。

#### 3.5.3 战斗机会成本

万宝琉璃体的四维成长与凡体完全相同。它不是在仙人之体上额外附送 `300` 福源，而是用仙人四维成长换取福源流派。

| 等级 | 对比仙体 | 生命差值 | 攻击差值 | 防御差值 | 回复差值 | 获得福源 |
| ---: | --- | ---: | ---: | ---: | ---: | ---: |
| 120 | 仙人之体 | `-1190` | `-238` | `-119` | `-23.8` | `+120` |
| 120 | 青莲长生体 | `-1785` | `0` | `0` | `-83.3` | `+120` |
| 300 | 仙人之体 | `-2990` | `-598` | `-299` | `-59.8` | `+300` |
| 300 | 青莲长生体 | `-4485` | `0` | `0` | `-209.3` | `+300` |

数值结论：

1. 仙人之体仍是直接攻击成长最高的通用战斗仙体。
2. 青莲长生体仍是生命和回复最高的持续生存仙体。
3. 万宝琉璃体定位为福源、宠物同步、浩气套装和长期资源收益仙体。
4. `300` 福源在现有数值体系中属于强力但已被设计基准覆盖的档位，不构成脱离系统量级的新属性。
5. `5000` 灵韵折合仅用于奖池期望和重复补偿口径，不代表它必须在所有战斗场景严格强于其他仙体。

#### 3.5.4 禁止影响的概率

万宝琉璃体提供的福源不得影响：

- 藏宝图 Boss 独立掉率。
- 黑市藏宝图刷新、库存或价格。
- 使用藏宝图时的奖池 ROLL。
- 万宝琉璃体自身 `0.1%` 概率。
- 宠物皮肤、金盒或其他藏宝奖励概率。

藏宝图全部概率只读取服务端静态权重，不读取角色福源、装备、职业、仙体或客户端传入值。这样可以避免“先抽到仙体，再提高后续抽奖概率”的自我强化循环。

#### 3.5.5 装备与切换

- 抽中后只解锁账号仙体，不自动装备当前角色。
- 玩家后续在仙体页手动选择，仍消耗 `1000` 灵韵并保存、退出、重登生效。
- 万宝琉璃体可以作为账号获得的第一个非凡仙体；不要求先完成首次渡劫。
- 未完成首次渡劫但已抽中万宝琉璃体时，允许仙体页显示并装备该仙体。
- 后续首次渡劫仍正常解锁仙人之体，不覆盖万宝琉璃体。
- 同账号新角色 Lv1 装备后获得 `1` 点福源，并按该角色当前等级计算，不继承其他角色等级。

#### 3.5.6 派生而非存档累计

仙体福源不得写入 `pillFortune`、`fruitFortune` 或新的可累加玩家存档字段。

正确实现：

```text
总福源 = 装备 + 图鉴 + 法宝 + 福源果 + 丹药 + 宠物 + 美酒
       + 勋章 + Buff + 道树 + 皮肤 + 命格 + 神器
       + 当前仙体按等级派生的福源
```

只保存：

- 账号是否已解锁万宝琉璃体。
- 当前角色是否装备万宝琉璃体。
- 现有仙体切换 pending。

不保存：

- `bodyFortune = 120` 之类结果值。
- 每次升级累计写入的仙体福源。
- 客户端计算后的总福源。

这样可以保证升级、降级修复、切换、重登、读档重试和公式版本迁移不会重复加点。仙体切换预览单独显示：

```text
福源变化 = 目标仙体派生福源 - 当前仙体派生福源
```

#### 3.5.7 重复策略决策

最终采用：**万宝琉璃体始终保留在奖池中，允许重复命中；已拥有时补偿 `5000` 灵韵。**

不采用“首次抽中后从账号奖池移除”的不放回方案。

| 比较项 | 允许重复并补偿 | 账号不放回 |
| --- | --- | --- |
| ROLL 是否依赖账号读取 | 否 | 是 |
| 使用藏宝图时是否需要账号锁 | 否 | 是，或命中仙体后补读 |
| 多角色同时开图 | 各自正常预抽，开箱时串行结算 | 需要账号级预占，否则可能同时预抽到唯一仙体 |
| 玩家弃图 | 无额外处理 | 必须释放唯一仙体预占 |
| 断线和服务重启 | `pendingReward` 原样恢复 | 还要恢复或清理账号预占 |
| 概率审计 | 所有账号固定 `0.1%` | 已拥有和未拥有账号奖池不同 |
| 幂等恢复 | 与现有皮肤重复补偿一致 | 需要新增预占、领取、释放状态机 |
| 长期灵韵增量 | 每图仅 `0.1% × 5000 = 5` | 无重复补偿，但实现复杂度明显更高 |

不放回方案的主要竞态：

```text
角色A使用藏宝图并预抽到仙体，但尚未开箱
  -> 账号资产中仍未正式拥有
角色B同时使用藏宝图
  -> 若只检查“是否已拥有”，仍可能再次预抽到仙体
```

要严格保证只能抽到一次，必须在使用藏宝图时就向账号数据写入 `reservedRunId`，并处理：

- A 放弃藏宝图后释放预占。
- A 永久不打开宝箱时预占是否长期保留。
- 服务重启后的预占恢复。
- 两个槽位竞争同一预占。
- 账号写入成功但藏宝 sidecar 写入失败时的回滚。

这些状态会显著扩大高价值账号资产的故障面。允许重复并补偿不需要预占：ROLL 始终固定，最终在开箱时通过账号锁判定首次或重复即可。经济影响也很小，长期每张图只增加 `5` 灵韵期望，因此可靠性优先于不放回。

## 四、藏宝海地图

### 4.1 地图类型

藏宝岛属于第四章八卦海真实主地图，不创建独立世界，也不替换 `ZoneData`。

进入时：

- `GameState.currentChapter` 保持为 `4`。
- 设置 `GameState.worldMode = "treasure_map"`，用于暂停普通刷怪和跨章传送。
- 根据服务端返回的 `active.islandId`，在第四章现有地图中传送到对应岛屿出生点。
- 岛上允许正常存档和保存退出；开箱结算、归航事务与普通存档互斥。

退出时：

- 已开箱时清除 `session`，返回第四章固定返回点。
- 未开箱时使用归航法阵视为主动放弃，清除 `active` 和 `session`，不返还藏宝图。
- 直接退出游戏不会放弃；重登后根据 sidecar 自动回到原岛安全出生点。

寻宝模式必须阻止：

- 跨章传送。
- 回城。
- 挑战、试炼塔、神器竞技场和多人副本入口。
- 普通世界 NPC 交互。
- 普通怪物生成和掉落。

| 修炼岛 | `exp_pill` | 修炼果×6 |
| 金砖岛 | `gold_brick` | ×3 / ×6 / ×9 / ×12 |
| 灵韵岛 | `lingyun_fruit` | 灵韵果×4 |
| 上品灵韵岛 | `lingyun_fruit_superior` | ×1 / ×2 / ×3 |
| 上品修炼岛 | `exp_pill_superior` | ×1 / ×2 / ×3 |
| 极品修炼岛 | `exp_pill_supreme` | ×1 |
| 极品灵韵岛 | `lingyun_fruit_supreme` | ×1 |
| 金盒岛 | `gold_box_10m` | ×1 |
| 仙缘秘岛 | `account_treasure` | 三款宠物皮肤或万宝琉璃体，条件权重 `4 : 4 : 4 : 3` |

连续两张图抽中同类时允许留在当前岛：

- 新图仍生成新的 `runId`。
- 服务端重新授权同一宝箱。
- 客户端播放一次短暂重置光效。
- 不排除上一张图命中的岛，不修正原概率。

### 4.4 宝箱视觉

项目当前是二维瓦片和 NanoVG 渲染，不使用 3D 模型。

九岛统一复用：

```text
image/wubao_treasure_chest_20260704023406.png
```

通过以下参数区分大类：

1. 宝箱上方品类图标。
2. “修炼宝藏”“金砖宝藏”等短铭牌。
3. 普通、上品、极品、金币大奖、账号大奖五档光效。
4. 岛面少量复用装饰物。
5. 仙缘秘岛显示仙缘轮廓、金色外圈与账号珍藏标识，不提前显示宠物或仙体剪影。

宝箱不提前展示：

- 具体数量档。
- 具体皮肤款式或是否为万宝琉璃体。
- 是否会重复补偿。

### 4.5 第四章使用入口

藏宝图只从背包打开 `TreasureMapUI`，第四章地图不再配置额外 NPC 或传送阵入口。

客户端在请求前校验当前内存章节为第四章，并完成一次第四章主存档检查点；服务器继续以主存档章节为门禁。检查点尚未成功时不发送使用请求，期间禁止切章。

固定返回点：

```lua
treasure_map_return_ch4 = { x = 39.5, y = 41.5 }
```

第四章地图合同测试必须确认不存在 `ch4_treasure_map_entry` 常驻实体。

### 4.6 位置校验边界

当前普通章节的位置与移动主要由客户端维护，服务端没有实时权威的普通世界坐标。

因此首版服务端可以基于当前服务端存档与会话校验：

- 槽位已经加载。
- 服务端当前保存的角色章节为第四章。
- 没有副本或其他服务端事务。
- 背包中真实持有藏宝图。
- 是否存在未完成藏宝进度。

普通世界的章节值和坐标最初仍来自客户端玩法状态，因此“存档章节为第四章”只能作为产品门禁，不能等同于服务端权威移动证明。服务端也不能把“距离入口 `1.5` 格”声明为强防作弊边界。入口附近判断用于正常交互体验，不能作为奖励安全的唯一依据。

真正的防作弊边界是：

- 服务端扣图。
- 服务端 ROLL。
- 服务端持久化 `runId + islandId + chestId + pendingReward`。
- 服务端只允许该授权宝箱结算一次。

以后普通世界接入服务端权威位置后，再升级入口距离校验。

## 五、玩家流程与 UI

### 5.1 首次使用

1. 玩家在第四章从背包打开藏宝图界面。
2. 显示入口面板：
   - 当前藏宝图数量。
   - “消耗 1 张藏宝图进入藏宝海”。
   - 奖池预览。
   - 固定返回第四章的说明。
3. 玩家确认后发送使用请求。
4. 服务端加锁、读取存档、校验并完成两段 ROLL。
5. 服务端克隆存档，扣除一张图，并写入 sidecar `active`。
6. 保存成功后返回背包 patch 和目标岛授权。
7. 客户端应用 patch，再进入藏宝海目标岛。

保存失败时不扣图、不创建进度、不传送。

### 5.2 开箱

1. 玩家靠近宝箱，按 E 或移动端交互键。
2. 显示查看面板，只显示奖励大类。
3. 玩家确认后发送 `StartOpen`。
4. 服务端校验授权，返回开箱 token 和服务端开始时间。
5. 客户端播放 `3` 秒读条；移动、死亡、关闭面板或断线会取消。
6. 客户端发送 `CompleteOpen`。
7. 服务端校验 token、最短读条时间和持久化 `active`。
8. 服务端读取 `pendingReward`，不重新随机。
9. 容量足够时写入奖励和完成记录。
10. 保存成功后下发结果。

### 5.3 背包容量

奖池中的修炼果、灵韵果、金砖和金盒都是背包消耗品。

规则：

- 使用藏宝图时不因奖励可能占背包而拒绝，避免提前泄露奖励。
- 开箱时按已持久化的具体奖励计算精确可写容量。
- 优先写入同 ID、未满且非 `bmNoResell` 的普通堆叠。
- 空间不足时不发奖、不完成、不重新 ROLL。
- `active` 保持不变，提示“背包空间不足，奖励已保留”。
- 皮肤或仙体首次解锁，以及对应重复灵韵补偿，均不占背包。

### 5.4 连续开图

当前 `active` 未完成：

- 禁止使用下一张图。
- 不扣图。
- 不 ROLL。
- 提示“请先完成当前藏宝图指向的宝藏”。

当前图已完成且 `session` 仍存在：

- 结果面板只允许确认奖励并归航。
- 服务端在 `session` 清除前拒绝使用下一张藏宝图。
- 玩家必须归航，再在第四章打开藏宝图界面开启下一次寻宝。

### 5.5 继续、放弃与退出

有未完成进度且玩家仍在岛上时：

- **保存或保存退出**：保留同一个 `runId`、岛屿、宝箱和预抽奖励。
- **使用归航法阵**：视为主动放弃，清除 `active` 和 `session`，不返还藏宝图，不发奖励。
- **取消归航**：关闭确认面板，不改变状态。

放弃是在已经看见奖励大类后的主动弃奖。地图已经消耗，不能靠放弃重抽同一张图，因此允许该行为。

不存在“保留进度并归航”选项。保留进度只服务于岛上存档、退出游戏、断线和崩溃恢复，不能作为返回龟背岛后再次传送的玩法分支。

### 5.6 重登

| 状态 | 重登处理 |
| --- | --- |
| 扣图保存前断线 | 视为未使用 |
| sidecar 已保存但响应丢失 | 查询后继续原岛 |
| 已登岛未开箱 | 自动传送回原岛安全出生点 |
| 开箱保存失败 | 保留原奖励，重新开箱 |
| 发奖保存成功但响应丢失 | 回放完成结果，不重复发奖 |
| 当前图完成但仍有 session | 回放奖励，确认后归航 |

登录流程：

1. 正常角色存档加载完成。
2. UI 事件监听和第四章地图引用准备完成。
3. 客户端发送 `QueryState`。
4. 服务端读取主存档与 sidecar，并返回服务端权威藏宝图槽位快照。
5. 客户端先清除本地所有藏宝图堆叠，再按权威槽位快照重建；校验或应用失败时停止传送、开箱和保存。
6. 有 `active` 时客户端直接进入 `active.islandId` 对应岛屿，不弹“继续寻宝”作为正常路径。
7. 进入岛屿后排队执行一次普通存档，保存岛上落点。
8. 客户端坐标不参与岛屿选择，目标岛始终由服务端 sidecar 决定。

岛上普通存档携带 `treasureSessionId`。服务端写入主存档前，将它与当前云端主存档中的会话标记比较；不一致说明这是进入前、归航前或结算前滞留的旧保存请求，必须以 `treasure_state_changed` 拒绝，不能覆盖新的藏宝状态。旧版本遗留的 active 状态在 `QueryState` 时由服务端先补写会话标记和权威岛屿坐标，再允许客户端继续保存。

藏宝图槽位快照是会话标记之外的第二道门禁。仅校验 `treasureSessionId` 不足以防止“服务端已扣图、使用回包丢失、客户端旧背包仍有图、岛上普通存档把图写回”的情况。客户端必须在排队岛上检查点保存前完成权威藏宝图槽位同步。

主存档还必须持久化单调递增的 `treasureRevision`。会话 ID 在“入岛前”和“归航后”都会为空，仅比较会话 ID 存在 ABA 风险：归航后到达的入岛前旧存档可能再次变成“空会话对空会话”。每次扣图、发奖、放弃和归航都递增修订号；普通存档必须同时匹配当前会话 ID 和修订号，否则拒绝写入。

### 6.3 结果 patch

- `backpackPatch`：`{ slot, item }` 或 `{ slot, clear = true }`。
- `mapSnapshotJson`：服务端主存档中全部藏宝图堆叠及槽位；`State` 与 `UseResult` 必须携带。
- `playerPatch`：最终 `gold / lingYun / exp` 等绝对值，不发送增量。
- `cosmeticsPatch`：新解锁的皮肤 ID 和最终账号外观版本。
- `immortalBodiesPatch`：新解锁的仙体 ID、最终账号仙体版本与本次是否重复。

- 可以应用 patch 更新本地显示。
- 不得调用 `InventorySystem.AddConsumable` 作为奖励来源。
- 不得在结果响应后立即发起全量保存。
- patch 应用失败时提示重新登录，以服务端存档为准。
- 有 `active/session` 时，`mapSnapshotJson` 应用失败必须阻断传送、开箱和普通存档。

## 七、服务端状态与事务

### 7.1 存储边界

藏宝进度使用独立服务端 key：

```text
treasure_map_state_{slot}
```

不放入普通客户端 `coreData`，不加入 `SaveDTO`，也不需要角色存档版本迁移。

sidecar 自带内部版本：

```lua
{
    version = 1,
    revision = 0,
    session = nil,
    active = nil,
    completedRuns = {},
}
```

角色主存档 `save_{slot}` 只保存：

- 藏宝图扣除后的背包。
- 开箱发放后的背包或灵韵。
- 第四章和角色基础数据。

账号 key：

- `account_cosmetics` 只由统一 `AccountCosmeticsService` 写入。
- `account_immortal_bodies` 只由统一 `AccountImmortalBodiesService` 写入。
- 普通客户端存档不得覆盖以上两个账号资产 key。

### 7.2 sidecar 结构

```lua
{
    version = 1,
    revision = 12,
    session = {
        sessionId = "tm-session-...",
        returnPointId = "treasure_map_return_ch4",
        currentIslandId = "treasure_island_gold",
        startedAt = 0,
        updatedAt = 0,
    },
    active = {
        runId = "tm-run-...",
        requestId = "...",
        islandId = "treasure_island_gold",
        chestId = "treasure_chest_gold",
        rewardType = "gold_brick",
        status = "entered", -- entered / skin_pending / body_pending
        pendingReward = {
            kind = "consumable", -- consumable / pet_skin / immortal_body
            itemId = "gold_brick",
            amount = 12,
            skinId = nil,
            bodyId = nil,
        },
        createdAt = 0,
        updatedAt = 0,
    },
    completedRuns = {
        ["tm-run-..."] = {
            completedAt = 0,
            islandId = "...",
            chestId = "...",
            rewardSummary = {},
            response = {},
        },
    },
}
```

完成记录最多保留 `32` 条，按完成时间淘汰最旧记录。

### 7.3 使用藏宝图事务

```text
校验会话与槽位
  -> 获取 slot lock
  -> 读取 save_{slot} + treasure_map_state_{slot}
  -> 校验请求幂等 requestId
  -> 校验第四章入口模式或合法 chain session
  -> 校验真实背包持有 treasure_map
  -> 两段式 ROLL
  -> 生成 runId、岛屿和 pendingReward
  -> 克隆主存档和 sidecar
  -> 克隆中扣除藏宝图
  -> 写入 session + active
  -> 预覆盖备份
  -> 同一批次写主存档和 sidecar
  -> 保存成功
  -> 释放锁
  -> 发送绝对背包 patch 和传送授权
```

任何保存失败：

- 原主存档不变。
- sidecar 不变。
- 不传送。
- 不向客户端泄露抽中的具体结果。

使用藏宝图阶段不读取账号皮肤或账号仙体：

- 所有账号始终按同一静态权重 ROLL。
- 已拥有万宝琉璃体的账号仍可能预抽到仙体。
- 首次或重复只在开箱时通过账号锁判定。
- 因此使用藏宝图事务不新增账号锁、账号预占或不放回状态。

### 7.4 开箱事务

普通道具奖励：

```text
获取 slot lock
  -> 读取主存档 + sidecar
  -> completedRuns 命中则回放结果
  -> 校验 runId / islandId / chestId / openToken
  -> 读取 pendingReward
  -> 精确检查背包容量
  -> 克隆后写入奖励
  -> active 清空，写 completedRuns
  -> 同批保存主存档 + sidecar
  -> 成功后发送结果 patch
```

不得先修改原存档对象。所有业务修改都在深拷贝上完成。

账号级大奖：

```text
获取 slot lock
  -> 读取主存档 + sidecar
  -> 校验 runId / chestId / openToken
  -> 按固定顺序获取对应 account lock
  -> 读取 account_cosmetics 或 account_immortal_bodies
  -> 用 sourceRunId 判定本次是否已经结算
  -> 判定首次解锁或重复补偿
  -> 克隆角色、sidecar 和账号数据
  -> 重复时在角色克隆中增加 5000 灵韵
  -> 首次时在账号克隆中写入解锁记录
  -> active 清空并写 completedRuns
  -> 同批保存主存档 + sidecar + 对应账号 key
  -> 成功后发送绝对 patch
```

锁顺序固定为 `slot lock -> account lock`。皮肤和仙体 handler 不得使用相反顺序，避免同账号多槽位并发时形成死锁。

### 7.5 普通存档并发

`SaveWriteService` 必须：

1. 获取与玩法 handler 相同的 `slot lock`。
2. 在整个异步写入结束前保持锁。
3. 成功、失败、异常和断线路径都释放锁。
4. 被玩法事务占用时返回 `slot_locked`，客户端按现有重试机制稍后保存。

这样可以保证：

- 普通保存先开始时，藏宝图操作等待或失败重试。
- 藏宝图事务先开始时，普通保存不会用旧背包覆盖奖励。

### 7.6 重放与幂等

| 场景 | 处理 |
| --- | --- |
| 同一 `requestId` 重复使用 | 返回同一 `runId`，不再扣图 |
| 使用回包超时 | 只重发同一 `requestId`，并行查询权威状态 |
| 使用保存成功但回包丢失 | `QueryState` 返回原 `active` 和权威藏宝图槽位快照，应用后才允许岛上保存 |
| 迟到的旧使用回包 | 请求 ID 不匹配则忽略，并重新查询权威状态 |
| 入岛前旧存档在归航后迟到 | 即使会话 ID 都为空，`treasureRevision` 落后仍拒绝写入 |
| 已结束的旧使用请求在归航后重放 | 命中 sidecar 的 `requestHistory`，拒绝执行且不扣下一张图 |
| 有 active 使用第二张 | 拒绝，不扣图 |
| 旧 `runId` 开箱 | 若在完成记录中则回放，否则拒绝 |
| 同时点击多个宝箱 | 只有授权 `chestId` 可进入结算 |
| 作弊位移到其他宝藏岛 | 其他岛的 `islandId/chestId` 与 active 绑定不符，服务端不签发开箱令牌 |
| 连点完成开箱 | 首个保存，其他返回处理中或完成结果 |
| 发奖成功、响应丢失 | 从 `completedRuns.response` 回放 |
| 归航回包丢失 | 保持本地交互锁并查询；无 `active/session` 时自动归航，有状态时解除锁允许原操作重试 |
| 旧归航请求在下一次寻宝中迟到 | `runId/sessionId` 不匹配，拒绝且不改变新会话 |
| 同一仙体 `runId` 重试 | 从 `bodyRewardSources` 和 `completedRuns` 恢复，不重复解锁或补偿 |
| 两个槽位同时抽中仙体 | 开箱时由账号锁串行；第一个首次解锁，后一个按重复补偿 |
| 服务重启 | 从 sidecar 恢复，不依赖内存缓存 |
| 断线 | 清理 open token 和运行时锁，保留 sidecar |

## 八、账号级大奖

### 8.1 皮肤池

| ID | 名称 | 属性 |
| --- | --- | --- |
| `pet_premium_chiyan` | 赤焰天犬 | 宠物攻击 `+1%` |
| `pet_premium_xuanbing` | 玄冰天犬 | 宠物防御 `+1%` |
| `pet_premium_biling` | 樱华天犬 | 宠物生命 `+1%` |

皮肤大奖总概率 `0.4%`，三款等权，单款全局概率约 `0.1333%`。

### 8.2 统一账号外观服务

新增 `AccountCosmeticsService`：

- 统一读取和写入 `account_cosmetics`。
- 统一账号级锁。
- 所有皮肤来源都应逐步迁入该服务。
- 普通客户端保存不直接覆盖该 key。
- 读取失败时保持藏宝奖励待结算，不按“新皮肤”兜底。

### 8.3 首次与重复

命中皮肤档时，具体 `skinId` 已在使用藏宝图时预抽并写入 `pendingReward`。

开箱时：

- 未拥有：解锁该款皮肤，不额外发灵韵。
- 已拥有：不重抽其他皮肤，补偿 `5000` 灵韵。
- 同一款可以重复抽到。

### 8.4 皮肤结算状态机

账号皮肤和角色存档不是同一个业务对象，因此使用 `sourceRunId`：

```lua
petAppearances[skinId] = {
    unlocked = true,
    unlockTime = 0,
    sourceType = "treasure_map",
    sourceRunId = "tm-run-...",
}
```

首次解锁流程：

```text
获取 slot lock + account cosmetics lock
  -> 读取 sidecar、主存档、account_cosmetics
  -> 若已存在且 sourceRunId == 当前 runId，说明本次已解锁，继续完成角色结算
  -> 若已存在且来源不是当前 runId，走重复补偿
  -> 若不存在，将 active.status 保存为 skin_pending
  -> 写 account_cosmetics，带 sourceRunId
  -> 写 completedRuns 并清 active
  -> 返回新皮肤结果
```

账号写入成功但后续完成保存失败时：

- 重试读取到相同 `sourceRunId`。
- 视为本次首次解锁已成功。
- 不发 `5000` 重复补偿。

账号外观读取失败时：

- 不解锁。
- 不补偿。
- 不完成宝箱。
- 保留原 `active` 等待重试。

### 8.5 万宝琉璃体账号服务

新增 `AccountImmortalBodiesService`：

- 统一读取和写入 `account_immortal_bodies`。
- 使用独立账号锁资源 `account_immortal_bodies`。
- 普通客户端保存不再写入该 key。
- 保证 `mortal` 永久存在。
- 按只增不减原则合并已解锁仙体。
- 维护 `bodyRewardSources`，记录首次和重复结算的 `runId`。

推荐账号结构：

```lua
{
    version = 2,
    unlockedBodies = {
        mortal = {
            unlockedAt = 0,
            source = "default",
        },
        immortal_body_wanbao_liuli = {
            unlockedAt = 0,
            source = "treasure_map",
            sourceRunId = "tm-run-...",
            firstUnlockSlot = 1,
        },
    },
    bodyRewardSources = {
        ["tm-run-..."] = {
            bodyId = "immortal_body_wanbao_liuli",
            duplicate = false,
            settledAt = 0,
            sourceType = "treasure_map",
        },
    },
}
```

`bodyRewardSources` 不能省略。仅在 `unlockedBodies[bodyId]` 保存首次 `sourceRunId`，无法可靠识别“账号原本已拥有，本次属于重复补偿”的重试。

### 8.6 万宝琉璃体首次与重复

命中仙体时，`pendingReward` 固定为：

```lua
{
    kind = "immortal_body",
    rewardType = "account_treasure",
    bodyId = "immortal_body_wanbao_liuli",
}
```

开箱时：

- 未拥有：账号解锁万宝琉璃体，不额外发灵韵。
- 已拥有：不重抽、不改成皮肤，补偿 `5000` 灵韵。
- 同一个账号允许未来再次抽中该仙体。
- 是否首次只由服务端账号数据决定，客户端本地仙体列表不参与判定。

服务端准备函数应与皮肤逻辑对齐：

```text
PrepareBodyGrant(accountBodies, bodyId, sourceRunId)
  -> bodyRewardSources 已存在当前 runId：返回上次相同结果
  -> unlockedBodies 已存在 bodyId：记录 duplicate=true
  -> 否则写入 unlockedBodies，记录 duplicate=false
  -> version + 1
```

异常处理：

- 账号仙体读取失败：不解锁、不补偿、不完成宝箱，保留 `active`。
- 同批保存失败：角色灵韵、账号仙体和 sidecar 均以旧数据为准。
- 保存成功但响应丢失：通过 `completedRuns.response` 回放。
- 两个槽位同时开出仙体：账号锁串行，第一个成功写入后，第二个读取到已拥有并获得重复补偿。
- 普通客户端随后保存：不得覆盖服务端刚写入的账号仙体。

## 十四、运营观察

上线后记录：

- 各 Boss 每日藏宝图产量。
- 黑市库存、购买量和掉落图出售量。
- 藏宝图使用率、出售率和放弃率。
- 单角色日均开图数与连续开图长度。
- 各奖励实际命中分布。
- 背包满导致的待领取次数。
- 未完成进度平均保留时长。
- 极品修炼果、极品灵韵果、金盒和皮肤产量。
- 三款皮肤首次解锁与重复补偿次数。
- 万宝琉璃体首次解锁、重复补偿、装备率和平均获得图数。
- 各等级万宝琉璃体使用率及其福源、宠物同步率分布。
- 单图实际平均灵韵折合。
- sidecar 恢复、结果回放和幂等拦截次数。
- `slot_locked` 重试率和存档冲突率。

重点观察：

1. 金盒 `1000万` 金币、`1.5%` 概率是否导致金币产量过高。
2. 黑市购买价 `12` 仙石是否形成稳定成交。
3. 皮肤重复补偿是否成为异常灵韵来源。
4. 万宝琉璃体重复补偿是否形成异常灵韵来源。
5. 万宝琉璃体是否因宠物同步率收益成为无条件最优仙体。
6. 第四章真实宝藏岛的进入、重登自动恢复和放弃归航是否稳定。
7. 普通存档与服务端奖励事务是否仍存在覆盖冲突。

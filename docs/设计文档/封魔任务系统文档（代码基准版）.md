# 封魔任务系统文档（代码基准版）

> 用途：基于当前代码整理封魔任务系统的真实实现，并补充第五章新增计划。  
> 更新时间：2026-05-27。  
> 口径：**以当前实装代码为准，不以旧设计、旧注释或预期方案为准。**

---

## 一、系统定位

封魔任务是替代旧“除魔任务”的现行系统，核心由两部分组成：

1. 一次性封魔任务  
   每章一组，按玩家等级解锁，接取后在当前地图额外生成 1 只“魔化 BOSS”。
2. 日常封魔任务  
   完成指定章节全部一次性封魔后解锁；每日限 1 次，所有章节共享配额；奖励体魄丹和宠物食物。

当前代码已经支持 **第 1~4 章封魔任务**，**第 5 章仅有 NPC 占位，尚未接通系统**。

## 四、核心规则

### 4.1 魔化 BOSS 公共增幅

`SealDemonConfig.DEMON_BUFF` 当前配置如下：

| 项目 | 数值 |
|------|------|
| 狂暴攻速倍率 | `1.3` |
| 狂暴移速倍率 | `1.3` |
| 狂暴技能冷却倍率 | `0.7` |
| 生命倍率 | `1.5` |

魔化 Boss 的创建逻辑：

- 基于原 Boss `typeId` 深拷贝
- 标记 `isDemon = true`
- 标记 `isSealDemon = true`
- 视配置覆盖 `level / category / realm`
- 应用 `berserkBuff`
- 应用 `sealDemonHpMult`
- 名称前缀变为 `魔·`

### 4.2 体魄丹规则

`SealDemonConfig.PHYSIQUE_PILL` 当前配置：

| 项目 | 数值 |
|------|------|
| 最大累计使用量 | `50` |
| 每日获取上限 | `1` |
| 阶梯配置 | `10 / 20 / 30 / 40 / 50` |
| 备注 | 配置里保留了 `bonusPerTen = 2`，但当前实际人物属性加成是**每颗 +1 体魄** |

当前实际生效口径：

- 服务端写入 `player.pillPhysique`
- 客户端角色属性直接把 `pillPhysique` 累加到 `GetTotalPhysique()`
- `SealDemonUI` 与 `CharacterUI` 都按“每颗 +1 体魄”展示

### 4.3 日常配额规则

当前服务端以角色槽位为维度，使用：

- `serverCloud.quota:Add(userId, "seal_demon_daily_s" .. slot, 1, 1, "day", ...)`

因此当前真实规则是：

- **每日 1 次**
- **章节共享**
- **角色隔离**
- 是否已完成由服务端 quota 权威校验
- 客户端 `dailyCompletedDate` 主要用于本地 UI 与预检

### 4.4 日常 Boss 强度定义

日常封魔 Boss 的强度定义方式是：

- 先从该章 `bossPool` 随机一个 `sourceBoss`
- 再直接使用该条目写死的 `level / category / realm`
- 最后叠加封魔公共增幅

当前文档口径统一如下：

| 第1章 | `Lv20 / boss / lianqi_3` | 当前代码 |
| 第2章 | `Lv40 / king_boss / jindan_1` | 当前代码 |
| 第3章 | `Lv70 / emperor_boss / huashen_1` | 当前代码 |

## 六、实际运行链路

### 6.1 入口与 UI

- 第 1~4 章封魔使使用 `interactType = "seal_demon"`
- `NPCDialog` 进入 `SealDemonUI.Show(npc)`
- 面板**只显示当前 NPC 所属章节**
- 打开面板时才调用 `SealDemonSystem.RefreshQuests(player.level)` 刷新锁定状态

### 6.2 一次性任务接取

客户端预检：

- 任务状态必须是 `available`
- 只检查“是否已有进行中的一次性封魔”
- 不检查是否已有进行中的日常封魔

服务端权威校验：

- 玩家等级是否满足
- 任务是否存在
- 任务是否已完成
- 当前连接是否已有任意封魔任务进行中

通过后：

- 服务端仅返回 `questId + bossId + spawnX + spawnY`
- 客户端本地生成 1 只魔化 Boss
- `questStates[questId]` 切为 `active`

### 6.3 日常任务接取

客户端预检：

- `dailyCompletedDate` 不能等于今天
- 只检查“是否已有进行中的日常封魔”
- 必须先完成该章全部一次性封魔

服务端权威校验：

- `DAILY[chapter]` 是否存在
- `requiredQuests` 是否全部 `completed`
- quota 是否已用完

通过后：

- 服务端从该章 `bossPool` 随机 1 只
- 回传 `bossId / bossLevel / bossCategory / bossRealm / spawnX / spawnY`
- 客户端据此生成魔化 Boss
- `activeDailyChapter` 记录当前日常章节

### 6.4 击杀、上报与奖励

客户端监听 `monster_death`：

- 只处理 `isSealDemon = true`
- 必须是当前 `activeQuestMonster` 或 `activeDailyMonster`

击杀后流程：

1. 本地先清空活动怪物引用
2. 向服务端发送 `C2S_SealDemonBossKilled`
3. 服务端依据 `sealDemonActive_[connKey]` 判定是一次性还是日常
4. 服务端写档并回发 `S2C_SealDemonReward`
5. 客户端发放灵韵、道具、体魄丹显示和 toast

容错逻辑：

- 如果无网络连接，客户端直接走本地兜底发奖
- 如果服务端返回 `save_failed / slot_locked / no_save`，客户端也会本地兜底发奖，并主动触发 `save_request`

### 6.5 放弃与章节切换

`SealDemonSystem.AbandonActiveTask()` 会：

- 本地移除活动魔化 Boss
- 一次性任务状态从 `active` 退回 `available`
- 清空 `activeQuestId / activeDailyChapter`
- Fire-and-forget 发送 `C2S_SealDemonAbandon`

章节切换时：

- `main.lua` 会自动放弃当前封魔任务
- 原因是封魔怪物只存在于当前章节地图

### 6.6 存档规则

当前持久化字段只有两部分：

| 字段 | 含义 |
|------|------|
| `saveData.sealDemon.quests` | 一次性任务状态；只存 `completed` 和 `available` |
| `saveData.sealDemon.dailyCompletedDate` | 本地记录的日常完成日期 |
| `saveData.player.pillPhysique` | 体魄丹累计值 |

不持久化的内容：

- 当前正在进行的任务
- 当前活动魔化 Boss 实例

读档后的实际行为：

- `activeQuestId / activeDailyChapter / active monsters` 全部清空
- `active` 状态回退为 `available`
- 玩家需要重新接取

### 6.7 迁移规则

当前迁移点：

- `SaveMigrations[10]`
- 旧 `exorcism` 数据被清空
- 初始化新 `sealDemon = { quests = {} }`
- 初始化 `player.pillPhysique = 0`

## 八、当前实现差异与风险

下面这些点都属于“当前代码已经存在”的实现差异，文档需要明确记录：

### 8.1 UI 和注释仍残留“三章共享”口径

当前真实配置已经是 4 章，但以下位置仍写旧口径：

- `SealDemonUI.lua` 顶部注释写“显示三章封魔任务列表”
- 面板说明文案写“日常封魔每日限1次（三章共享）”
- `SealDemonHandler.lua` quota 注释也写“三章共享”
- `SealDemonSystem.lua` 对 `activeDailyChapter`、`GetHighestUnlockedDailyChapter()`、`AcceptDaily()` 的注释仍写 `1/2/3`

结论：

- **真实功能是 4 章**
- **文案和注释未同步**

### 8.2 一次性任务的客户端回包使用了错误字段名

`SealDemonConfig.QUESTS` 使用的是：

- `overrideLevel`
- `overrideCategory`
- `overrideRealm`

但 `SealDemonSystem.HandleAcceptResult()` 在一次性任务分支里读取的是：

- `questDef.level`
- `questDef.category`
- `questDef.realm`

这意味着当前一次性任务客户端生成魔化 Boss 时：

- 可能**没有正确应用覆盖等级**
- 可能**没有正确应用覆盖类别**
- 可能**没有正确应用覆盖境界**

而日常任务分支使用的是服务端回包里的 `bossLevel / bossCategory / bossRealm`，这部分是正确的。

这个问题不影响“任务能接、能打、能领奖”的主流程，但会影响一次性封魔 Boss 的**实际强度口径**。

### 8.3 `SealDemonSystem.Update()` 已存在，但当前未接主循环

系统里已经写了 `Update(dt)`，用于：

- 发现活动怪物已失效时回收状态
- 保持 `activeDemonMonster` 同步

但当前全项目里**没有实际调用** `SealDemonSystem.Update()`。

结论：

- 正常击杀路径依赖 `monster_death` 事件，主流程仍可工作
- 但“怪物被系统异常清掉”的兜底回收逻辑当前没有实际运行

### 8.4 客户端预检只拦同类型，不拦任意活动任务

当前预检逻辑：

- `AcceptQuest()` 只拦“已有一次性任务”
- `AcceptDaily()` 只拦“已有日常任务”

真正拦“任意封魔任务并存”的，是服务端 `already_active` 校验。

结果：

- 客户端可能发出本不该发的请求
- 但最终会被服务端拒绝

这属于**体验层冗余请求**，不是数据一致性问题。

### 8.5 日常章节反查兜底仍偏旧

日常回包落地时，客户端会先按 `bossId + bossLevel` 反查 `dailyChapter`；如果反查失败，再按等级做粗略兜底。

这个兜底分支当前只有：

- `<=20 -> 1`
- `<=40 -> 2`
- `else -> 3`

正常情况下因为反查会成功，所以第 4 章不会误判；但这个兜底本身仍然是旧时代口径。

### 8.6 第五章 NPC 只有占位，没有入口

当前 `seal_master_ch5`：

- 只有名字、立绘、对话
- `subtitle = "暂未开放"`
- **没有** `interactType = "seal_demon"`
- **没有** `chapter = 5`

因此第五章封魔使当前无法打开面板。

## 九、执行部分

### 9.1 日常和一次性奖励优化

执行逻辑：

- 保持当前奖励协议不变，继续使用 `lingYun + petFood + petFoodCount + physiquePill`
- 一次性奖励按第 `1~5` 章统一重配
- 日常奖励按第 `1~5` 章统一重配
- 日常 quota 统一定义为：**所有章节共享每日 1 次**，不再保留“三章共享”或“第 1~4 章共享”口径

执行清单：

- 更新 `SealDemonConfig.QUESTS` 一次性奖励

| ID | 奖励 |
|------|------|

## 1. 设计目标

### 1.1 玩法描述

八卦海·四兽岛是一个**传送副本区域**，玩家通过主世界传送点进入。副本内有世界BOSS「文王仙魂」（圣级×4血量），3 小时刷新一次，全服共享状态。

**核心规则**：

| 规则 | 说明 |
|------|------|
| 进入方式 | 主世界传送点交互 → Loading → 进入副本 |
| 退出方式 | 走到出口 / 按钮传送 → 回到主世界；死亡 → 回到副本入口（主世界侧） |
| 人数限制 | 0~8 人随时进出，满员拒绝进入（服务端返回失败，客户端提示"副本已满"） |
| 单人可否 | 可以，但 BOSS 极其强大，单人需要很长时间 |
| BOSS 刷新 | 3 小时（10800 秒），全服共享击杀/刷新状态 |
| 社交模式 | 偶遇式——进入后自动看到其他在场玩家，默契协作 |

### 1.2 奖励机制

**伤害贡献制 + 名次灵韵奖励**：

| 名次 | 奖励 | 前提 |
|------|------|------|
| 第 1 名 | 500 灵韵 | 伤害 ≥ 总 HP 的 5% |
| 第 2 名 | 400 灵韵 | 伤害 ≥ 总 HP 的 5% |
| 第 3 名 | 300 灵韵 | 伤害 ≥ 总 HP 的 5% |
| 第 4~8 名 | 100 灵韵 | 伤害 ≥ 总 HP 的 5% |
| 伤害不足 | 无 | 低于 5% 不获取奖励 |

**伤害累计规则**：
- 退出副本或死亡后，累计伤害**保留**，再次进入继续累计，直到 BOSS 被击杀后清零
- 场内玩家：BOSS 死亡时直接弹窗显示奖励
- 场外玩家（已退出/死亡未回来）：奖励记录在副本入口 NPC，可补领

### 1.3 与匹配制的区别

| 维度 | 匹配制 | 持久区域（本方案） |
|------|--------|------------------|
| 进入方式 | 点"匹配"，等凑齐人 | 直接传送进入 |
| 人数 | 固定（2-4 人开局） | 动态（0-8 人随时进出） |
| BOSS 状态 | 每局独立 | 全服共享，3 小时刷新 |
| 单人可否 | 不行 | 可以，但极难 |
| 社交感 | 匹配陌生人 | 偶遇→协作 |
| 退出 | 结算后自动 | 随时走，随时回（伤害累计保留） |

## 2. 架构总览

### 2.1 双区域模型

```
┌──────────────────────┐          ┌──────────────────────────────┐
│  主世界（单机体验）    │          │  副本区域（多人实时）           │
│                      │          │                              │
│  现有全部游戏逻辑     │  传送切换  │  位置同步 + BOSS 战斗          │
│  服务端仅做存档中继   │ ◄──────► │  客户端本地Monster实体(AI/战斗) │
│  玩家之间互不可见     │          │  服务端HP权威+伤害校验+结算     │
│                      │          │  玩家之间互相可见              │
│                      │          │  伤害排名 + 灵韵结算            │
└──────────────────────┘          └──────────────────────────────┘
              │                              │
              └───── serverCloud 共享存档 ─────┘
```

### 2.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 多人模式 | background_match + 应用层虚拟房间 | V1 验证 persistent_world 数据隔离，不可用；保持现有 background_match，DungeonHandler 管理副本房间 |
| 副本形式 | 传送副本（独立场景） | 场景隔离最简单，Loading 画面是天然切换边界 |
| BOSS 控制 | 混合架构：客户端本地Monster实体运行AI/战斗/技能/渲染，服务端仅HP权威+伤害校验+结算 | 复用现有Monster/CombatSystem/SkillSystem/GameEvents全套系统，避免重复实现 |
| 位置同步 | 服务端中继广播 | 不用 Scene Replication（2D NanoVG 无法用引擎节点同步） |
| 战斗判定 | 客户端乐观执行 + 服务端校验 | 保持手感，服务端结果覆盖 |

### 2.3 为什么不用 Scene Replication

当前游戏是 **2D NanoVG 渲染**，所有实体（玩家、怪物、BOSS）都是 NanoVG 绘制的精灵，不是引擎 Scene 中的 Node。引擎的 REPLICATED 节点自动同步机制无法直接应用。

因此采用**手动 RemoteEvent 广播**方案：服务端收集各客户端上报的位置 → 打包成快照 → 广播给副本内所有客户端 → 客户端根据快照数据用 NanoVG 绘制远程玩家。

#### V1 验证方案（风险 · 测试 · 回退）

##### 1. 风险分析

| # | 风险 | 概率 | 影响 | 说明 |
|---|------|------|------|------|
| R1 | **启动流程改变** | 高 | 高 | `background_match` 下引擎行为：Start() 立即调用 → serverConn 随后通过 ServerReady 到达。`persistent_world` 下行为未知，可能 Start() 时 serverConn 就已就绪（类似传统 dedicated server），也可能 ServerReady 根本不触发 |
| R2 | **ServerReady 不触发** | 中 | 高 | 当前 client_main.lua 严重依赖 `ServerReady` 事件作为 fallback 路径。若 persistent_world 不触发此事件，SaveSystemNet 排队的 FetchSlots 将永远不执行，玩家无法加载角色 |
| R3 | **连接超时回退失效** | 低 | 中 | CONNECT_TIMEOUT=120s 后调用 FallbackToStandalone()，但此函数不切换模式（clientCloud=nil）。persistent_world 模式下，连接失败恢复路径可能不同 |
| R4 | **多客户端连接冲突** | 低 | 高 | 当前 HandleClientIdentity 已有"同 userId 踢旧连接"逻辑。persistent_world 下多个**不同 userId** 同时连入是否正常需要验证 |
| R5 | **服务器 idle 关闭** | 中 | 中 | persistent_world 服务器可能有 idle timeout（所有玩家断开后 N 秒自动关闭）。如果 timeout 太短，单人挂机回来可能断线 |
| R6 | **存档队列竞态** | 低 | 中 | SaveSystemNet 中的 `_queuedFetchSlots` 是单值，persistent_world 模式下如果连接时序变化，可能出现覆盖 |

```lua
            local saved = cjson.decode(data)
            bossStatePersistent_.alive = saved.alive
            bossStatePersistent_.lastSpawnAt = saved.lastSpawnAt or 0
            bossStatePersistent_.killedAt = saved.killedAt or 0
            bossStatePersistent_.totalKills = saved.totalKills or 0

            if bossStatePersistent_.lastSpawnAt > 0 then
                local elapsed = now - bossStatePersistent_.lastSpawnAt
                if elapsed >= BOSS_RESPAWN_INTERVAL then
                    DungeonManager.RespawnBoss()
                elseif not bossStatePersistent_.alive then
                    -- 已被击杀但还没到刷新时间
                    bossStateRuntime_.hp = 0
                else
                    bossStateRuntime_.hp = bossStateRuntime_.maxHp
                end
            else
                -- 首次启动，初始化刷新时间
                bossStatePersistent_.lastSpawnAt = now
                bossStateRuntime_.hp = bossStateRuntime_.maxHp
            end
        end
        if callback then callback() end
    end)
end
```

### B2：3 小时刷新计时器

```lua
local BOSS_RESPAWN_INTERVAL = 10800  -- 3 小时 = 10800 秒
local respawnCheckTimer_ = 0

-- 固定刷新：每秒检查一次，距上次刷新超过 3 小时就强制刷新（无论 BOSS 是否存活）
function DungeonManager.UpdateRespawn(dt)
    respawnCheckTimer_ = respawnCheckTimer_ + dt
    if respawnCheckTimer_ < 1.0 then return end  -- 每秒检查一次
    respawnCheckTimer_ = 0
    
    local elapsed = os.time() - (bossStatePersistent_.lastSpawnAt or 0)
    
    if elapsed >= BOSS_RESPAWN_INTERVAL then
        -- 固定刷新到期：无论 BOSS 存活与否，强制刷新新 BOSS
        if bossStatePersistent_.alive then
            print("[Dungeon] BOSS still alive but respawn timer expired, force respawn")
        end
        DungeonManager.RespawnBoss()
    end
end

function DungeonManager.RespawnBoss()
    bossStatePersistent_.alive = true
    bossStatePersistent_.lastSpawnAt = os.time()  -- 记录本次刷新时间（固定刷新起点）
    bossStatePersistent_.killedAt = 0
    bossStateRuntime_.hp = bossStateRuntime_.maxHp
    bossStateRuntime_.aiState = "idle"
    bossStateRuntime_.targetConnKey = nil
    
    -- 清理所有伤害记录（在线玩家的 totalDamage 归零 + 离线表清空）
    for _, p in pairs(dungeonPlayers_) do
        p.totalDamage = 0
    end
    DungeonManager.ClearLeftPlayers()
    
    DungeonManager.SaveBossState()
    DungeonManager.Broadcast("S2C_BossRespawned", {
        hp = bossStateRuntime_.maxHp,
        maxHp = bossStateRuntime_.maxHp,
    })
    
    print("[Dungeon] BOSS respawned")
end

-- 获取距下次刷新的剩余时间（秒），0 = 刚刷新
```

### B4：伤害处理（客户端上报 + 服务端校验）

```lua
-- 客户端 DungeonClient.Update(dt) 中：
-- 每 0.3s 批量上报 Monster:TakeDamage 中 _onDamageTaken 回调累积的伤害

-- 服务端 HandlePlayerAttack 接收上报：
local ATTACK_COOLDOWN = 0.2          -- 上报频率限制
local MAX_DAMAGE_PER_REPORT = 50000  -- 单次上报伤害上限（防作弊）

function M.HandlePlayerAttack(connKey, eventData)
    if not bossStatePersistent_.alive then return end
    local p = dungeonPlayers_[connKey]
    if not p then return end

    -- 频率限制
    local now = os.clock()
    if p.lastAttackTime and (now - p.lastAttackTime) < ATTACK_COOLDOWN then return end
    p.lastAttackTime = now

    -- 读取并校验伤害
    local damage = eventData:GetInt("damage")
    if damage <= 0 then return end
    damage = math.min(damage, MAX_DAMAGE_PER_REPORT)

    -- 扣血（服务端权威HP）
    bossStateRuntime_.hp = math.max(0, bossStateRuntime_.hp - damage)

    -- 广播
    M.Broadcast(SaveProtocol.S2C_AttackResult, {
        attacker = connKey, damage = damage, isCrit = false,
        bossHp = bossStateRuntime_.hp, bossMaxHp = bossStateRuntime_.maxHp,
    })

    if bossStateRuntime_.hp <= 0 then M.OnBossDefeated() end
end
```

```lua
                ratio = p.totalDamage / bossStateRuntime_.maxHp,
                name = p.name,
                online = true,  -- 场内
            })
        end
    end
    
    -- 场外玩家（已退出/死亡/断线，但有伤害贡献）
    for userId, lp in pairs(leftPlayers_) do
        table.insert(ranking, {
            userId = lp.userId,
            slot = lp.slot,
            connKey = nil,
            damage = lp.totalDamage,
            ratio = lp.totalDamage / bossStateRuntime_.maxHp,
            name = lp.name,
            online = false,  -- 场外
        })
    end
    
    table.sort(ranking, function(a, b) return a.damage > b.damage end)
    
    -- 灵韵奖励配置（纯灵韵，无其他奖励）
    local LINGYUN_REWARDS = { 500, 400, 300, 100, 100, 100, 100, 100 }  -- 第1~8名
    local MIN_DAMAGE_RATIO = 0.05  -- 至少 5% 伤害才有奖励
    
    for rank, entry in ipairs(ranking) do
        local lingYun = LINGYUN_REWARDS[rank] or 100
        local qualified = entry.ratio >= MIN_DAMAGE_RATIO
        
        if qualified then
            -- 写入灵韵奖励
            ApplyDungeonRewards(entry.userId, entry.slot, lingYun)
        end
        
        if entry.online then
            -- 场内玩家：直接弹窗通知
            local conn = connections_[entry.connKey]
            if conn then
                SafeSend(conn, "S2C_BossDefeated", {
                    rank = rank,
                    totalPlayers = #ranking,
                    damage = entry.damage,
                    ratio = entry.ratio,
                    lingYun = qualified and lingYun or 0,
                    qualified = qualified,
                    fullRanking = BuildRankingSummary(ranking),
                })
            end
        else
            -- 场外玩家：写入待领取队列（副本入口 NPC 补领）
            if qualified then
                local pendingKey = "dungeon_pending_reward_" .. entry.slot
                serverCloud.score:Set(entry.userId, pendingKey, cjson.encode({
                    rank = rank, lingYun = lingYun,
                    bossName = "文王仙魂", timestamp = os.time(),
                }))
            end
        end
    end
    
    -- 清理离线伤害表
    DungeonManager.ClearLeftPlayers()
    
    print(string.format("[Dungeon] BOSS defeated! %d players participated, next respawn in %d seconds",
        #ranking, BOSS_RESPAWN_INTERVAL))
end

function GetLingYunReward(rank)
    local rewards = { 500, 400, 300, 100, 100, 100, 100, 100 }
    return rewards[rank] or 100
end
```

## 11. 同步方案与带宽评估

### 11.1 同步频率设计

| 数据类型 | 方向 | 频率 | 说明 |
|---------|------|------|------|
| 玩家位置 | C→S | 10fps | x, y, dir, anim |
| 快照广播 | S→C | 10fps | 所有玩家 + BOSS 位置 |
| 攻击指令 | C→S | ~2次/s | 由攻速决定 |
| 攻击结果 | S→C | ~2×N次/s | 每次攻击广播 |
| BOSS 技能 | S→C | ~0.3次/s | 低频 |
| 刷新倒计时 | S→C | 1fps | BOSS 死亡期间 |

### 11.2 带宽评估（8 人满员场景）

```
上行（每个客户端 → 服务器）：
  位置上报：16 bytes × 10fps           = 160 B/s
  攻击指令：~20 bytes × 2次/s          = 40 B/s
  合计：~200 B/s / 人

下行（服务器 → 每个客户端）：
  快照广播：(7人 × 20B + BOSS 30B) × 10fps = 1700 B/s
  攻击结果：~30 bytes × 16次/s（8人×2）     = 480 B/s
  BOSS 技能：~50 bytes × 0.3次/s            = 15 B/s
  合计：~2200 B/s ≈ 2.2 KB/s / 人

服务端总出口：
  2.2 KB/s × 8 人 = 17.6 KB/s

结论：极低带宽需求，手机 4G 绰绰有余。
```

### 11.3 延迟容忍度

| 延迟范围 | 体验影响 |
|---------|---------|
| < 100ms | 几乎无感 |
| 100-200ms | 远程玩家位置有轻微漂移，可接受 |
| 200-500ms | 攻击命中反馈有感知延迟，但 RPG 战斗节奏可容忍 |
| > 500ms | 需要客户端预测补偿 |

**设计选择**：客户端乐观执行（本地先播攻击动画），服务端结果回来后修正。对于 RPG 自动攻击模式，这种方案足够好。

## 15. 线上存档安全评估（2026-04-12）

> **核心关切**：副本系统开发中，对线上玩家数据造成不可逆影响的风险有多大？如何消除？
>
> 用户原话：*"我甚至不担心多人副本的测试问题。如果多人副本有问题，可以直接掐断多人副本。我更担心多人副本开发中对线上数据、线上玩家状态造成不可逆的影响。"*

### 15.1 serverCloud 写入路径审计

副本系统中所有对 serverCloud 的写操作（即可能影响线上玩家数据的路径）：

| # | 函数 | 写入 key | 操作类型 | 影响范围 | 风险等级 |
|---|------|----------|---------|---------|---------|
| W1 | `_WritePendingReward` | `dungeon_pending_{dungeonId}_{slot}` | Set（独立 key） | 仅副本奖励 | 🟢 安全 |
| W2 | `SaveBossState` | `dungeon_boss_sishoudao`（SYSTEM_UID=0） | Set（全局 key） | BOSS 状态 | 🟢 安全 |
| W3 | `ClaimPendingReward` | `dungeon_pending_{dungeonId}_{slot}` | Delete | 仅副本奖励 | 🟢 安全 |
| W4 | `HandleLoadGame` 旧 key 迁移 | `dungeon_pending_{dungeonId}_{slot}` + 旧 key | Set+Delete | 仅副本奖励 | 🟢 安全 |

**关键保证**：**没有任何路径直接写入 `save_{slot}`（玩家主存档）**。

### 15.2 已消除的竞态风险：`_ApplyRewardToOnline`

#### 问题描述

旧版 `_ApplyRewardToOnline` 函数执行的操作序列：

```
① serverCloud:Get(userId, "save_{slot}")     → 读取完整存档 blob（~50KB）
② saveData.player.lingYun += reward           → 内存中修改灵韵字段
③ serverCloud:Set(userId, "save_{slot}", blob) → 写回完整存档
```

这是经典的 **GET-MODIFY-SET 竞态（Read-Modify-Write race condition）**：

```
时间线：
  T1: BOSS 击杀 → _ApplyRewardToOnline → Get(save_1)  → 读到 lingYun=1000
  T2: 玩家在主世界存档 → SaveGame → Set(save_1, {lingYun=1200, ...})  ← 玩家赚了200灵韵
  T3: _ApplyRewardToOnline 回调 → Set(save_1, {lingYun=1500, ...})  ← 覆盖！玩家的200灵韵丢失
```

**后果**：玩家在 T1~T3 之间的所有游戏进度（不仅是灵韵，包括装备、位置、任务等整个存档 blob）可能被覆盖为 T1 时刻的快照。**这是不可逆的数据丢失**。

#### 触发条件

- BOSS 被击杀时，贡献者恰好在主世界（场外玩家），且主世界正在执行自动存档
- 自动存档间隔 30 秒，BOSS 击杀是随机时刻 → 概率约 `伤害处理延迟 / 30s`
- 多人场景下，多个场外贡献者同时触发，竞态窗口叠加

#### 修复方案

**彻底废弃 `_ApplyRewardToOnline` 和 `_ApplyRewardToOffline`**，统一使用 `_WritePendingReward`：

```lua
-- 旧方案（已删除）：
-- _ApplyRewardToOnline  → Get(save) → modify → Set(save)  ❌ 竞态
-- _ApplyRewardToOffline → Get(save) → modify → Set(save)  ❌ 竞态

-- 新方案：
function M._WritePendingReward(userId, slot, rewardData)
    local pendingKey = "dungeon_pending_" .. rewardData.dungeonId .. "_" .. slot
    serverCloud:Set(userId, pendingKey, rewardData, { ... })
end
-- → 写入独立 key，完全不触碰 save_{slot}  ✅ 无竞态
```

**为什么新方案安全**：
- `dungeon_pending_{dungeonId}_{slot}` 是**只有副本系统读写**的独立 key
- 写入操作是原子的 `Set`，不是 `Get-Modify-Set`
- 与主存档 `save_{slot}` 完全隔离，无论何时写入都不会影响玩家存档

### 15.3 奖励领取流程（NPC 方案）

奖励不再由服务端自动写入存档，而是由玩家主动通过 NPC 领取：

```
BOSS 击杀
  ↓
服务端 OnBossDefeated()
  ↓ 对每个合格贡献者调用 _WritePendingReward()
  ↓ 写入 dungeon_pending_{dungeonId}_{slot} = { lingYun, rank, ratio, bossName, ... }
  ↓ （不触碰 save_{slot}）
  ↓
玩家登录 / game_loaded 事件
  ↓ DungeonClient 自动发送 C2S_QueryDungeonReward
  ↓ 服务端读取 pending key → 返回 S2C_DungeonRewardInfo
  ↓
副本入口 NPC 显示感叹号指示器（EntityRenderer）
  ↓
玩家与 NPC 交互
  ↓ NPCDialog 显示领取面板（BOSS名、排名、伤害比例、灵韵数量、过期警告）
  ↓
玩家点击"领取奖励"
  ↓ C2S_ClaimDungeonReward
  ↓
服务端 ClaimPendingReward()
  ↓ 读取 pending key → Delete pending key → 返回 S2C_ClaimDungeonRewardResult { ok, lingYun }
  ↓ （仍然不触碰 save_{slot}）
  ↓
客户端收到成功结果
  ↓ 将 lingYun 加到玩家内存状态 playerMemory.lingYun
  ↓ 下次正常自动存档时，由 SaveSystem 统一写入 save_{slot}
  ↓ （走已有的、经过充分验证的存档流程）
```

**关键设计**：灵韵加到内存后，由**正常的 SaveSystem 自动存档流程**写入 `save_{slot}`。这条存档路径是整个游戏最核心、最稳定的代码路径，经过长期线上验证。副本系统完全不引入新的 `save_{slot}` 写入路径。

### 15.4 HandleLoadGame 改造

旧版 HandleLoadGame 中有一段"副本离线灵韵奖励恢复"逻辑，执行的是：

```
读取 dungeon_pending_reward_{slot}
  → 取出 lingYun → 加到 saveData.player.lingYun
  → BatchSet(save_{slot}, 新saveData) + Delete(pending key)
```

这又是一个 GET-MODIFY-SET 模式，且嵌入了**所有玩家的登录流程**（即使从未进过副本的玩家也会执行这段 Get 查询）。

**改造后**：
- 更新 key 格式：`dungeon_pending_reward_{slot}` → `dungeon_pending_sishoudao_{slot}`
- 移除自动发放：不再修改 `saveData`，不再写回 `save_{slot}`
- 仅做旧 key 迁移：如果检测到旧格式 key，迁移数据到新格式 key 后删除旧 key
- 奖励由客户端通过 NPC 独立领取（§15.3 流程）

### 15.5 安全保证总结

| 保证 | 实现方式 |
|------|---------|

#### P0-2. ClaimPendingReward 内存锁（rules.md §8 合规）

**问题**：没有 `claimActive_[connKey]` 内存锁，快速重复请求可并发执行 Get+Delete。

**修复方案**（参考悟道树金标准）：

```lua
-- DungeonHandler.lua
local claimActive_ = {}  -- connKey → true

function M.ClaimPendingReward(connKey, dungeonId)
    if claimActive_[connKey] then return end  -- ① 内存锁
    claimActive_[connKey] = true
    
    local userId = Session.userIds_[connKey]
    local slot = Session.connSlots_[connKey]
    if not userId or not slot then
        claimActive_[connKey] = nil
        return
    end
    
    local pendingKey = DungeonConfig.MakePendingKey(dungeonId, slot)
    
    serverCloud:Get(userId, pendingKey, {
        ok = function(scores)
            local pendingData = scores[pendingKey]
            if not pendingData or type(pendingData) ~= "table" or not pendingData.lingYun then
                claimActive_[connKey] = nil  -- 释放锁
                SendTo(connKey, S2C_ClaimResult, { ok = false, reason = "no_pending" })
                return
            end
            
            local reward = pendingData.lingYun
            serverCloud:Delete(userId, pendingKey, {
                ok = function()
                    claimActive_[connKey] = nil  -- 释放锁
                    SendTo(connKey, S2C_ClaimResult, { ok = true, lingYun = reward })
                end,
                error = function(code, reason)
                    claimActive_[connKey] = nil  -- 释放锁
                    -- Delete 失败也返回 ok（幂等：下次再删）
                    SendTo(connKey, S2C_ClaimResult, { ok = true, lingYun = reward })
                end,
            })
        end,
        error = function()
            claimActive_[connKey] = nil  -- 释放锁
            SendTo(connKey, S2C_ClaimResult, { ok = false, reason = "server_error" })
        end,
    })
end

-- HandlePlayerDisconnected 中也要清理：
function M.HandlePlayerDisconnected(connKey)
    claimActive_[connKey] = nil
    -- ... 现有逻辑
end
```

### 16.8 v2.1 架构变更（2026-04-14）

**问题背景**：v2.0 重写时丢失了 3 项设计文档中的关键机制，导致"伤害每次进入副本重新累计"等 BUG。

**偏离排查结果**：

| # | 偏离项 | 影响 | 修复状态 |
|---|--------|------|---------|
| 1 | `leftPlayers_` 离线伤害表被移除，PlayerEnter 设 `totalDamage=0` | 🔴 玩家离开后伤害丢失（用户报告的 BUG） | ✅ 已恢复 |
| 2 | 结算未合并离线玩家（fighters 注册表 60s 超时导致离线玩家消失） | 🔴 离线超 60s 的玩家不计入结算排名 | ✅ 已修复 |
| 3 | `bossStatePersistent_` 重命名为 `bossMeta_`（代码清理） | ⚪ 无影响，仅命名变更 | ✅ 文档已知 |
| 4 | `ClearLeftPlayers()` 函数被移除 | 🟡 RespawnBoss 未清空离线表 | ✅ 已内联修复 |
| 5 | P0-3 攻击距离校验在重写中丢失 | 🔴 玩家可远程打 BOSS | ✅ 已恢复 |

**方案 A 实施：内存累计 + 定期 BatchSet**

v2.0 每次攻击执行 2 次 `Add()` 调用（HP + 伤害），8 人满攻时写入频率约 478 次/分钟，超出 300 次/分钟配额限制。

v2.1 架构：
```
攻击处理（每次）:
  - 内存扣血: bossStateRuntime_.hp -= damage     (0 次云端写入)
  - 内存累计: p.totalDamage += damage             (0 次云端写入)
  - batchDirty_ = true

定期刷写（每 3 秒）:
  - BatchSet(0):
      SetInt("boss_hp", hp)                       -- 1 个 key
      SetInt("boss_dmg_{charKey}", totalDamage)    -- N 个 key（在线+离线玩家）
  - = 1 次 BatchSet = 1 次写配额

写入频率: ~20 次/分钟（远低于 300/分钟限制）
```

**新增/修改的内存变量**：
```lua
local leftPlayers_ = {}           -- charKey → { userId, slot, name, totalDamage }
local BATCH_FLUSH_INTERVAL = 3.0  -- BatchSet 刷写间隔
local batchFlushTimer_ = 0
local batchDirty_ = false
```

# 万界黑商 — NPC 集市交易系统设计

> **版本**：v0.12.1（P0 事故修复）
> **日期**：2026-05-09
> **定位**：有限的 NPC 做市商交易，非玩家间直接交易
> **前置依赖**：C4 Money.Cost 验证 ✅、C5 BatchCommit 验证 ✅（无灵韵迁移依赖）

---

## 1. 系统概述

「万界黑商」是一个游走于各界的神秘商人 NPC。他以固定价格、限量收购的方式向玩家收购神器碎片，再以加价出售给其他有需要的玩家。

### 1.1 核心规则

| 规则 | 说明 |
|------|------|
| **NPC 做市** | 所有交易通过黑商 NPC，玩家之间不直接交易 |
| **固定价格** | 每种商品有独立的收购价和出售价，**不随库存变动** |
| **限量收购** | 每种商品有库存上限（碎片类 `max_stock = 5`，材料类 `max_stock = 5~10`），满仓不收 |
| **零初始库存** | 黑商初始没有任何商品，所有货物来自玩家出售 |
| **不刷新库存** | 无每日/每周重置，库存纯粹跟随交易实时变动 |
| **仙石交易** | 所有交易使用专用货币「仙石」，玩家需先用灵韵兑换仙石 |
| **开始即可见** | 图标从游戏开始即显示，境界不足时可浏览商品但不可操作（筑基初期解锁操作） |
| **背包直接操作** | 物品持有量从背包（`saveData.backpack`）直接计数和扣减，**无 Money 域物品迁移** |

### 1.2 交易流向

```
                         兑换
玩家灵韵 ──100灵韵:1仙石──→ 仙石余额
                              │
                              ↓
玩家A（有多余碎片）──卖给黑商──→ 黑商库存 ──买走──→ 玩家B（急需碎片）
                   收购价(仙石)              出售价(仙石)

仙石余额 ──1仙石:50灵韵──→ 玩家灵韵（反向兑换，50%折损）
```

### 1.3 仙石货币

| 属性 | 说明 |
|------|------|
| **名称** | 仙石 |
| **用途** | 万界黑商专用货币，不可在其他系统使用、出售或丢弃 |
| **正向兑换** | 100 灵韵 → 1 仙石（整数兑换，不足 100 不可兑换） |
| **反向兑换** | 1 仙石 → 50 灵韵（50% 折损，灵韵净回收） |
| **数据域** | Money 域原生货币，key = `xianshi`，天然支持 `Cost/Add/Get` |

**经济安全性**：

- **正向兑换（灵韵→仙石）**：灵韵在存档中先扣除并 `WriteSaveBack`，成功后 `BatchCommit:MoneyAdd(xianshi)`。崩溃场景：灵韵扣了但仙石未加 → 写 `WAL_SELL_KEY` 补偿，下次登录 `CheckWALOnLogin` 自动补发仙石
- **反向兑换（仙石→灵韵）**：`BatchCommit` 原子执行 `MoneyCost(xianshi)` + `ListAdd(WAL_KEY, lingYunGain)`。成功后通知客户端加灵韵，再清除 WAL。崩溃场景：仙石扣了但灵韵未加 → WAL 条目保留，下次登录 `CheckWALOnLogin` 自动补偿灵韵到存档
- **套利不可能**：100 灵韵 → 1 仙石 → 50 灵韵，往返损失 50%

### 1.4 UI 入口

- **入口按钮**：小地图容器（`minimapContainer`）下方，72×90 像素，显示 NPC 图标（`Textures/icon_black_merchant.png`）+ "黑市" 文字
- **NPC 入口**：天机阁地图区域 NPC "大黑无天"（`interactType = "black_merchant"`），与 NPCDialog 系统集成，对话后打开交易面板
- **可见性**：**游戏开始即可见**，无需任何前置条件
- **操作权限**：
  - 境界 < 筑基初期（`order < 4`）：点击可打开交易面板，展示所有商品（名称、价格、库存），但所有操作按钮（买/卖/兑换）**置灰不可点击**，面板顶部显示提示：`"你修为太低了，筑基初期修士才可以入内"`
  - 境界 ≥ 筑基初期（`order >= 4`）：正常交易
- **服务端校验**：无论客户端状态如何，所有 C2S 请求（兑换/买/卖）均校验境界 ≥ 筑基初期，不满足则拒绝

## 2. 定价模型

### 2.1 固定价格

所有商品采用**固定价格**，不随库存变动，不波动：

| 参数 | 含义 |
|------|------|
| `buy_price` | 收购价（黑商从玩家买入，玩家获得仙石） |
| `sell_price` | 出售价（黑商卖给玩家，玩家花费仙石） |

**黑商永远赚差价**：`sell_price = buy_price × 2`。

### 2.4 经济效果

**所有套利路径均不可盈利**：

| 套利路径 | 计算 | 结果 |
|---------|------|------|
| 买入-转卖同种 | 花 20 买 → 卖回得 10 | 亏损 10 仙石 |
| 灵韵↔仙石往返 | 100灵韵→1仙石→50灵韵 | 亏损 50% |
| 跨品套利（低→高） | 花 20 买钉耙 → 无法卖成八卦 | 不同品类不互通 |
| 买碎片→仙石→灵韵 | 花 20 买碎片 → 卖回 10 仙石 → 500 灵韵 | 净亏 1000 灵韵 |

## 3. serverCloud 数据模型

### 3.1 方案：仙石 → Money 域，物品 → 背包直操

仙石天然是 Money 域货币。**物品持有量不使用 Money 域**，而是直接从存档背包（`saveData.backpack`）中计数和扣减，通过 `BackpackUtils` 共享模块操作。

买入操作：`BatchCommit(扣仙石 + 扣库存 + 写背包WAL)` → 背包写入 → 清除 WAL。
卖出操作：背包扣除 → `WriteSaveBack` → `BatchCommit(加仙石 + 加库存)`。

**v0.10 装备类扩展**：装备类商品（`itemType = "equipment"`）使用相同的 Money 域库存 key 格式，但背包操作使用专用的 `BackpackUtils.CountEquipmentItem / AddEquipmentToBackpack / RemoveEquipmentFromBackpack` 函数，按 `equipId` 匹配。Handler 内部通过 `CountItemHeld()` 统一分派。

### 3.2 键设计

| 域 | userId | key 模式 | 值 | 用途 |
|----|--------|---------|-----|------|
| Money | `playerId` | `xianshi` | 整数 | 玩家仙石余额 |
| Money | `SYSTEM_UID(0)` | `bm_stock_{consumableId}` | 整数 | 黑商当前库存 |
| List | `SYSTEM_UID(0)` | `bm_pub_trade` | JSON 字符串 | 公共买卖记录（含玩家名） |
| List | `playerId` | `bm_trade_log` | JSON 字符串 | 个人兑换记录 |
| List | `playerId` | `bm_wal` | JSON | WAL：灵韵补偿（仙石→灵韵崩溃） |
| List | `playerId` | `bm_wal_bp` | JSON | WAL：背包物品补偿（购买后背包未写入） |
| List | `playerId` | `bm_wal_sell` | JSON | WAL：仙石补偿（出售背包已扣但仙石未到） |

**与 v0.8 文档的关键差异**：
- ~~`mat_{consumableId}` Money 域键~~：**已废弃**。物品持有量直接从背包读取（`BackpackUtils.CountBackpackItem`），不存在 Money 域物品键
- WAL 从 1 种扩展为 3 种（`bm_wal` / `bm_wal_bp` / `bm_wal_sell`），覆盖所有崩溃场景

**无交易限额**：个人购买、出售不限次数。库存上限（`max_stock`）是唯一的数量约束。

**不需要的数据**：
- 不需要价格存储（价格固定，写在 `BlackMerchantConfig.lua` 中）
- 不需要物品 Money 域键（从背包直接计数）
- 不需要交易计数器（无交易次数限制）
- 不需要 NPC 钱包（黑商仙石无限，不做收支平衡）

### 3.3 数据规模

| 指标 | 数量 | 说明 |
|------|------|------|
| 商品种类 | 48 | 每种占 1 个 SYSTEM Money key（含 4 种装备、3 种令牌盒） |
| SYSTEM Money key | 48 | `bm_stock_{id}`（消耗品用 consumableId，装备用 equipId） |
| 玩家 Money key | 1 | `xianshi`（物品不走 Money） |
| BatchCommit 操作数/笔交易 | 2~3 | 买入: `Cost xianshi + Cost stock + ListAdd WAL`；卖出: `Add xianshi + Add stock` |

### 3.4 交易记录（List 域，公共+个人双轨）

每笔交易完成后，服务端向 List 域追加流水记录。**买/卖操作写公共记录**（所有玩家可见），**兑换操作写个人记录**（仅自己可见）。

#### 键设计

| 域 | userId | key | 用途 | 上限 |

| List | `SYSTEM_UID(0)` | `bm_pub_trade` | 公共买卖记录（含玩家角色名 `pn`） | 50 条 |

## 5. 原子操作

### 5.0 服务端内存锁（§8 三层保护）

> **v0.10.1 新增**：遵循 `rules.md §8` 三层 C2S 保护机制，防止同一玩家并发请求导致重复扣款/重复发货。

**三层保护**：

| 层 | 机制 | 实现 |
|----|------|------|
| ① 服务端内存锁 | `buyActive_[connKey]` / `sellActive_[connKey]` | Handler 入口设 `true`，**所有**出口（同步+异步）设 `nil` |
| ② 原子 BatchCommit | `MoneyCost + MoneyCost + ListAdd` 原子执行 | 框架保证 |
| ③ 客户端按钮禁用 | `pendingRequest_` + 倒计时确认弹窗 | UI 层防重复点击 |

**内存锁覆盖范围**：

```lua
-- BlackMerchantHandler.lua 顶部
local buyActive_  = {}  -- connKey -> true
local sellActive_ = {}  -- connKey -> true
```

- `HandleBuy`：入口检查 + 设锁，13 个出口点全部释放（含同步校验失败 4 处、异步回调失败/成功 9 处）
- `HandleSell`：入口检查 + 设锁，12 个出口点全部释放（含同步校验失败 4 处、异步回调失败/成功 8 处）
- `HandleExchange` / `HandleQuery` / `HandleHistory`：无资源授予操作，不需要内存锁

**断线清理**：`OnDisconnect(connKey)` 中同时清除 `buyActive_` 和 `sellActive_`。

### 5.1 出售操作（玩家 → 黑商）

```
C2S_BMSell { consumableId = "rake_fragment_1", amount = 1 }

HandleSell:
  1. 境界校验：IsRealmOk(connKey)
  2. 白名单校验：BMConfig.ITEMS[consumableId] 存在
  3. amount 校验：amount >= 1
  4. 读存档 backpack：CountItemHeld(backpack, consumableId) → held >= amount
     - 消耗品：CountBackpackItem(backpack, consumableId)
     - 装备类：CountEquipmentItem(backpack, cfg.equipId)
  5. 读系统库存：stock + amount <= max_stock（满仓不收）
  6. 从 backpack 扣除：
     - 消耗品：RemoveFromBackpack(backpack, consumableId, amount)
     - 装备类：RemoveEquipmentFromBackpack(backpack, cfg.equipId, amount)
       → 按 equipId 匹配，不论附魔/洗练状态
  7. WriteSaveBack（存档写回）
  8. 成功后 BatchCommit:
     MoneyAdd(userId, "xianshi", total),                    -- 加仙石
     MoneyAdd(SYSTEM_UID, "bm_stock_{id}", amount),         -- 入库
  9. 失败处理：背包已扣但 BatchCommit 失败 → 写 WAL_SELL_KEY { xianshi = total }
  10. TradeLogger.LogPublic(charName, "sell", ...)
  11. → S2C_BMResult
```

### 5.2 购买操作（黑商 → 玩家）

```
C2S_BMBuy { consumableId = "rake_fragment_1", amount = 1 }

HandleBuy:
  1. 境界校验：IsRealmOk(connKey)
  2. 白名单校验 + amount 校验
  3. 前置校验：读 playerMoneys.xianshi >= total, 读 systemMoneys.stock >= amount
  4. BatchCommit（原子操作）:
     MoneyCost(userId, "xianshi", total),                    -- 扣仙石
     MoneyCost(SYSTEM_UID, "bm_stock_{id}", amount),         -- 出库
     ListAdd(userId, WAL_BP_KEY, { consumableId, amount,     -- 背包 WAL
             itemType, equipId }),                            -- v0.10: 装备字段
  5. 成功后：读存档 → 写背包 → WriteSaveBack → 清除 WAL_BP_KEY
     - 消耗品：AddToBackpack(backpack, consumableId, amount, cfg.name)
     - 装备类：逐件 LootSystem.CreateSpecialEquipment(cfg.equipId) →
              AddEquipmentToBackpack(backpack, newEquip)（初始状态装备）
  6. TradeLogger.LogPublic(charName, "buy", ...)
  7. → S2C_BMResult
```

### 5.3 批量交易

固定价格下批量定价极其简单：

```lua
total = price × amount  -- 不再需要逐件累加
```

### 5.4 交易后立即存档

交易涉及背包变动时，通过 `WriteSaveBack` 立即将修改后的存档写回 serverCloud。

**实现要点**：
- 买入：BatchCommit 成功 → 读存档 → 写背包 → `WriteSaveBack`（异步）→ 成功后清 WAL，失败则 WAL 保留
- 卖出：先 `WriteSaveBack` 扣背包 → 成功后 `BatchCommit` 加仙石+库存 → 失败写 `WAL_SELL_KEY`
- 兑换灵韵→仙石：先 `WriteSaveBack` 扣灵韵 → 成功后 `BatchCommit` 加仙石 → 失败写 `WAL_SELL_KEY`

### 5.5 库存轮询与性能参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 轮询间隔 | **30 秒** | UI 打开期间每 30 秒发一次 C2S_BMQuery |
| C2S 全局节流 | **0.5 秒** | `THROTTLE_INTERVAL`，所有请求共享，防止短时间内重复发送 |
| 系统库存缓存 TTL | **10 秒** | `GetSystemStockCached()`，SYSTEM_UID Money 读取带缓存 |
| 缓存失效 | 买/卖成功后 | `InvalidateSystemStockCache()` 主动清缓存 |
| 轮询内容 | 库存 + 仙石余额 + 背包持有量 | 价格固定不需轮询 |
| 关闭 UI | 停止轮询 | 释放网络资源 |
| 境界缓存 | `playerRealms_` 字典 | 首次 Query 时 `EnsureRealmCached` 从存档读取，断线时清理 |

```lua
pcall(BlackMerchantHandler.OnDisconnect, connKey)

-- 登录时 WAL 补偿检查（在 HandleLoadGame 加载存档后调用）
```

### 7.3 系统独立性原则

| 原则 | 说明 |

### 9.3 WAL 补偿机制（3 种 WAL）

`CheckWALOnLogin(userId, slot, saveData, onDone)` 在玩家登录时自动执行，**并行检查三种 WAL**：

| WAL key | 触发场景 | 补偿方式 |
|---------|---------|---------|
| `bm_wal` | 仙石→灵韵兑换：Cost 成功但灵韵未到账 | 灵韵写入存档 `saveData.player.lingYun` → `WriteSaveBack` |
| `bm_wal_bp` | 购买：仙石已扣+库存已扣但背包未写入 | 消耗品：`AddToBackpack` → `WriteSaveBack`；装备类（v0.10）：逐件 `CreateSpecialEquipment` → `AddEquipmentToBackpack` → `WriteSaveBack` |
| `bm_wal_sell` | 出售：背包已扣但仙石未到账；或灵韵→仙石：灵韵已扣但仙石 Add 失败 | `BatchCommit:MoneyAdd(xianshi)` 直接补发 |

**v0.10 WAL 数据扩展**：`bm_wal_bp` 条目新增 `itemType` 和 `equipId` 字段，`CheckWALOnLogin` 据此判断补偿装备还是消耗品。

**补偿完成后清除对应 WAL 条目**（`serverCloud.list:Delete`）。

## 12. 自动收购系统（大黑无天 NPC）

> **版本**：v0.12 新增
> **目标**：让系统 NPC "大黑无天" 以自己的名义定期收购满库存的黑商商品，缓解滞销商品积压、维持市场流通感

### 12.1 系统概述

"大黑无天" 是万界黑商体系中的 **系统级自动收购 NPC**，独立于玩家交易。他以固定的频率检测黑市库存，对所有 **库存已满** 的商品进行随机收购，其行为会记录在公共交易日志中（玩家可见）。

| 属性 | 说明 |
|------|------|
| **NPC 名称** | 大黑无天（显示在公共交易记录中） |
| **角色定位** | 系统做市商的库存回收助手，非玩家角色 |
| **仙石消耗** | **无**——大黑无天拥有无限资金，不消耗任何仙石 |
| **运行环境** | 服务端（`server_main.lua` → `HandleServerUpdate`） |
| **执行频率** | 每日一次（新日期首次 tick 触发） |

### 12.2 触发规则

#### 12.2.1 自动触发（每日一次）

- **时机**：服务器每帧 `HandleServerUpdate` 中调用 `RecycleTick(dt)`
- **日期判断**（两层快速路径）：
  1. **内存快速路径**：比较 `os.date("%Y%m%d")` 与内存变量 `lastRecycleDate_`
     - 若相同 → 今天已执行过，跳过（绝大多数 tick 命中此路径，零开销）
  2. **云端持久化**：`lastRecycleDate_` 为 `nil`（首次启动/重启后），从 `serverCloud.money:Get(SYSTEM_UID)` 读取 `bm_recycle_date`
     - 云端值与今日相同 → 已执行过，填充内存变量，跳过
     - 云端值为空或不同 → 新的一天，执行收购
- **重启安全**：日期持久化到 `serverCloud.money` 的 `bm_recycle_date` key（SYSTEM_UID=0），存储数值型 `YYYYMMDD`。重启后从云端恢复，**不会重复执行**
- **日期更新时机**：所有逐件收购完成后，通过 `MoneyAdd(SYSTEM_UID, "bm_recycle_date", delta)` 原子更新

#### 12.2.2 手动触发（GM 命令）

- **命令**：`bm_recycle`
- **触发方式**：GM 面板 → "系统" 分类 → "黑市收购" 按钮
- **注意**：GM 手动触发 **不受每日一次限制**，每次点击都会执行
- **用途**：测试验证、紧急库存调控

**执行路径（⚠️ 与常规 GM 命令不同）**：

```
常规 GM 命令：客户端 → 服务端鉴权 → 返回 ok → 客户端 callback 执行业务逻辑
bm_recycle：  客户端 → 服务端鉴权 → 服务端执行 ExecuteRecycle → 异步完成后返回结果
```

现有 `GMHandler` 鉴权通过后直接返回 `ok=true`，客户端在 callback 中执行业务逻辑。但 `bm_recycle` 的业务逻辑（读库存 → 随机 → BatchCommit）必须在**服务端**执行且是**异步**的。因此需要：

1. `GMHandler` 中新增 `cmd == "bm_recycle"` 分支，鉴权通过后 **不立即返回响应**
2. 调用 `BlackMerchantHandler.ExecuteRecycle(callback)`
3. 在 `callback` 中构建 `S2C_GMCommandResult` 并发送（携带收购结果摘要）
4. 客户端 callback 仅做 `ShowLog` 显示，不执行业务逻辑

参考先例：`C2S_GMResetDaily` 也是服务端执行后异步返回（§GMConsole 第 305-322 行）

### 12.3 收购逻辑

#### 12.3.1 商品筛选

遍历 `BMConfig.ITEM_IDS` 中所有商品，仅对 **库存已满** 的商品执行收购：

```
筛选条件：stock >= max_stock（该商品的 cfg.max_stock 或 BMConfig.MAX_STOCK）
```

- 库存未满的商品 → 跳过
- 库存已满的商品 → 进入加权随机阶段

#### 12.3.2 加权随机收购数量

对每个符合条件的商品，独立执行一次加权随机：

| 收购数量 | 概率 | 说明 |
|---------|------|------|
| **0 件** | 25% | 不收购（该商品本轮"逃过一劫"） |
| **1 件** | 50% | 收购 1 件（最常见情况） |
| **2 件** | 25% | 收购 2 件（运气好的清仓） |

- **期望值**：每个满库存商品平均每天被收购 1.0 件
- **实际收购量**：`min(randomAmount, currentStock)`，不会超过当前库存
- **各商品独立随机**：A 商品收 2 件不影响 B 商品的随机结果

#### 12.3.3 逐件独立提交

每个符合条件的商品独立执行一次 `BatchCommit`，**顺序串行**（前一个完成后再提交下一个）：

```
for each eligible item (sequential):
    BatchCommit("大黑无天收购_" .. itemId)
      :MoneyCost(SYSTEM_UID, "bm_stock_" .. itemId, amount)
      :Commit(function(ok)
          if ok then
              TradeLogger.LogPublic(...)
          end
          -- 无论成功失败，继续下一件
          processNext()
      end)
```

- **逐件独立**：A 商品 Commit 失败不影响 B 商品（消除级联回滚风险）
- **顺序串行**：避免并发 Commit 竞态，保证执行可预测
- **性能预估**：单次 Commit ≈ 200ms，100 种商品 ≈ 20s，远在 1 小时日窗口内
- 全部完成后：调用 `InvalidateSystemStockCache()` 刷新库存缓存
- 全部完成后：通过 `MoneyAdd` 更新云端日期标记（见 §12.2.1）
- 单件失败时：跳过该商品，继续处理剩余商品

#### GMHandler 异步分支详细说明（文件 #4）

现有 GMHandler 的标准流程是：鉴权 → 返回 `ok=true` → 客户端 callback 执行业务逻辑。`bm_recycle` 需要改为**服务端异步执行**模式：

```lua
    BlackMerchantHandler.ExecuteRecycle(function(ok, summary)

        local resultData = {
            ok = ok,
            msg = summary  -- "收购完成: 3种商品共5件" 或 "无满库存商品"

        SendEventToClient(userId, "S2C_GMCommandResult", resultData)

```

### 12.10 P0 事故报告：RecycleTick 无限重入导致经济崩溃 🔴

> **事故等级**：P0（线上经济系统崩溃）
> **发生时间**：2026-05-08（自动收购系统上线当日）
> **影响范围**：全服经济——仙石通胀，玩家获取无限仙石
> **当前状态**：`RECYCLE_ENABLED = false`（紧急关闭），经济损失不可逆

#### 12.10.1 事故根因

**原始缺陷代码**（`RecycleTick` 的 `money:Get` error 回调）：

```lua
serverCloud.money:Get(BMConfig.SYSTEM_UID, {
    ok = function(moneys) ... end,
    error = function(code, reason)
        lastRecycleDate_ = nil   -- ← 致命 BUG：清除了内存日期守卫
        print("[BlackMerchant][Recycle] 云端读取失败: " .. tostring(reason))
    end,
})
```

**触发链**：

1. `RecycleTick` 每帧被 `server_main.lua` 的 `HandleServerUpdate` 调用
2. 内存守卫 `lastRecycleDate_` 在 `error` 回调中被重置为 `nil`
3. **下一帧**：`lastRecycleDate_ == today` 判断失败（因为已被清空）→ 再次进入
4. 如果网络持续异常 → **每帧都重新进入** → 无限次 `ExecuteRecycle`
5. 即使网络恢复，已积压的异步回调会批量执行 → NPC 无限扣减库存

**灾难放大链**：

```
RecycleTick 无限重入
  → NPC "大黑无天" 无限执行 MoneyCost(SYSTEM_UID, stock)
  → 所有商品库存被清零
  → 库存清零后，玩家可以卖出商品补充库存（获得仙石）
  → 库存刚补充，下一轮 RecycleTick 又清零
  → 玩家反复卖出 → 无限获取仙石
  → 仙石通胀 → 经济系统崩溃
```

#### 12.10.2 修复措施

**修复 1：移除 error 回调中的 `lastRecycleDate_ = nil`**

```lua
-- ✅ 修复后
error = function(code, reason)
    -- 不重置 lastRecycleDate_，本进程当天不再重试
    print("[BlackMerchant][Recycle] 云端读取失败: " .. tostring(reason))
end,
```

**修复 2：ExecuteRecycle 读库存前强制刷新缓存**

原代码使用 10 秒 TTL 缓存的 `GetSystemStockCached()` 判断库存是否满仓。在高频重入场景下，缓存数据可能与实际库存不一致，导致误判"库存已满"而执行不必要的收购。

```lua
-- ✅ 修复后：ExecuteRecycle 入口处
InvalidateSystemStockCache()   -- 强制清缓存
local systemMoneys = GetSystemStockCached(function(...) ... end)
```

**修复 3：紧急关闭开关**

```lua
-- BlackMerchantConfig.lua
BMConfig.RECYCLE_ENABLED = false   -- 紧急关闭，待经济评估后再考虑重开
```

#### 12.10.3 修复后的 RecycleTick 逻辑

```
每帧调用 RecycleTick(dt):
  ├─ RECYCLE_ENABLED == false → return（当前状态）
  ├─ recycleRunning_ == true → return
  ├─ lastRecycleDate_ == today → return（绝大多数帧命中此路径）
  ├─ lastRecycleDate_ = today（占位，本进程当天只跑一次）
  ├─ money:Get(SYSTEM_UID) 读云端日期
  │   ├─ ok: cloudDate >= today → 已执行，return
  │   ├─ ok: cloudDate < today → ExecuteRecycle → 成功后 MoneyAdd 更新日期
  │   └─ error: 打印日志，return（不清除 lastRecycleDate_）
  └─ 无论成功失败，当天不再重入
```

**核心保证**：`lastRecycleDate_ = today` 在进入异步调用前赋值，且在所有出口路径中**永不被清除**。每个服务器进程每天最多执行一次。

#### 12.10.4 经济损失评估

| 影响 | 详情 | 可恢复性 |
|------|------|---------|
| **仙石通胀** | 玩家在 BUG 期间通过反复卖出获得大量仙石 | ❌ 不可逆——无法确定每个玩家的非法获利量 |
| **商品库存** | 所有商品库存被反复清零 | ✅ 已自动恢复——玩家卖出会补充库存 |
| **交易记录污染** | 公共交易记录中充满"大黑无天"的虚假收购记录 | ⚠️ 部分可恢复——FIFO 淘汰会自动清除旧记录（上限 50 条） |

**仙石回收不可行**：`serverCloud.money:MoneyCost(userId, "xianshi", amount)` 在已观察的代码中仅用于已连接玩家的请求上下文。是否支持对离线玩家或任意 UID 操作**未经验证**，不应贸然执行批量扣减。

#### 12.10.5 事故教训

| # | 教训 | 对应规则 |
|---|------|---------|
| 1 | **异步 error 回调中绝不能清除安全守卫变量** | 守卫变量一旦设置，只有业务成功完成后才可按逻辑推进，不可因失败而回退 |
| 2 | **每帧调用的函数必须考虑异步回调堆积** | RecycleTick 被 HandleServerUpdate 每帧调用，error 回调清除守卫 = 每帧重入 |
| 3 | **上线前必须模拟网络异常场景** | 71 项单元测试全部通过，但未覆盖"money:Get 连续失败多帧"的集成场景 |
| 4 | **不要声称未验证的能力** | "技术上能扣除玩家仙石"——实际上 serverCloud API 对离线玩家的行为未知 |
| 5 | **经济系统的 BUG 损失通常不可逆** | 货币发放容易、回收困难；经济系统代码必须以最高安全标准对待 |

#### 12.10.6 预防措施

1. **RECYCLE_ENABLED 默认 false**：自动收购功能在未经充分线上验证前保持关闭
2. **重开前必须**：在测试环境模拟 `money:Get` 连续失败场景，确认 RecycleTick 不重入
3. **若需重开**：建议设置每日执行后的"冷却期"（如 1 小时内不再尝试），作为 `lastRecycleDate_` 之外的第二道防线

### 12.11 审查结论（事故后修订）

> ~~2026-05-08 完成全面审查，核心逻辑无安全隐患~~ → **已被 P0 事故推翻**

原 §12.10 审查（71 项测试全通过）未能发现 RecycleTick error 回调中的守卫变量清除问题。测试覆盖了加权随机、日期检测、并发锁等维度，但**缺失了关键的网络异常多帧重入场景**。

**修正后的审查结论**：
- 收购核心逻辑（筛选、随机、BatchCommit、日志）经测试验证，逻辑正确
- **RecycleTick 的重入保护存在致命缺陷**，已修复（§12.10.2）
- 自动收购功能当前处于关闭状态（`RECYCLE_ENABLED = false`），待经济影响评估完成后再决定是否重开

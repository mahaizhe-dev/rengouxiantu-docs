# 万界黑商 — NPC 集市交易系统设计

> **版本**：v0.7.1（实施前一致性审查）
> **日期**：2026-04-10
> **定位**：有限的 NPC 做市商交易，非玩家间直接交易
> **前置依赖**：C4 Money.Cost 验证 ✅、C5 BatchCommit 验证 ✅（无灵韵迁移依赖）

---

## 1. 系统概述

「万界黑商」是一个游走于各界的神秘商人 NPC。他以固定价格、限量收购的方式向玩家收购神器碎片，再以加价出售给其他有需要的玩家。

### 1.1 核心规则

| 规则 | 说明 |
|------|------|
| **NPC 做市** | 所有交易通过黑商 NPC，玩家之间不直接交易 |
| **固定价格** | 每种商品有独立的收购价和出售价，**不随库存变动** |
| **限量收购** | 每种商品有库存上限（`max_stock = 5`），满仓不收 |
| **零初始库存** | 黑商初始没有任何商品，所有货物来自玩家出售 |
| **不刷新库存** | 无每日/每周重置，库存纯粹跟随交易实时变动 |
| **仙石交易** | 所有交易使用专用货币「仙石」，玩家需先用灵韵兑换仙石 |
| **开始即可见** | 图标从游戏开始即显示，境界不足时可浏览商品但不可操作 |

### 1.2 交易流向

```
                         兑换
玩家灵韵 ──100灵韵:1仙石──→ 仙石余额
                              │
                              ↓
玩家A（有多余碎片）──卖给黑商──→ 黑商库存 ──买走──→ 玩家B（急需碎片）
                   收购价(仙石)              出售价(仙石)

仙石余额 ──1仙石:50灵韵──→ 玩家灵韵（反向兑换，50%折损）
```

### 1.3 仙石货币

| 属性 | 说明 |
|------|------|
| **名称** | 仙石 |
| **用途** | 万界黑商专用货币，不可在其他系统使用、出售或丢弃 |
| **正向兑换** | 100 灵韵 → 1 仙石（整数兑换，不足 100 不可兑换） |
| **反向兑换** | 1 仙石 → 50 灵韵（50% 折损，灵韵净回收） |
| **数据域** | Money 域原生货币，key = `xianshi`，天然支持 `Cost/Add/Get` |

**经济安全性**：

- **正向兑换**：灵韵是本地状态，仙石是 Money 域。先扣灵韵（本地）再 `Money.Add(xianshi)`。崩溃场景：灵韵扣了但仙石未加 → 玩家亏损但系统安全
- **反向兑换**：仙石是 Money 域，灵韵是本地。先 `Money.Cost(xianshi)` 再加灵韵（本地）。崩溃场景：仙石扣了但灵韵未加 → 使用 Write-Ahead Log 在下次登录时补偿（WalManager 通过 ClientReady 事件自动触发，见 §9.2）
- **套利不可能**：100 灵韵 → 1 仙石 → 50 灵韵，往返损失 50%

### 1.4 UI 入口

- **位置**：日常任务下方，圆形功能图标
- **可见性**：**游戏开始即可见**，无需任何前置条件
- **操作权限**：
  - 境界 < 金丹期：点击可打开交易面板，展示所有商品（名称、价格、库存），但所有操作按钮（买/卖/兑换）**置灰不可点击**，面板顶部显示提示：`"你修为太低了，金丹期修士才可以入内"`
  - 境界 ≥ 金丹期：正常交易
- **服务端校验**：无论客户端状态如何，所有 C2S 请求（兑换/买/卖）均校验境界 ≥ 金丹期，不满足则拒绝

## 2. 定价模型

### 2.1 固定价格

所有商品采用**固定价格**，不随库存变动，不波动：

| 参数 | 含义 |
|------|------|
| `buy_price` | 收购价（黑商从玩家买入，玩家获得仙石） |
| `sell_price` | 出售价（黑商卖给玩家，玩家花费仙石） |

**黑商永远赚差价**：`sell_price = buy_price × 2`。

### 2.4 经济效果

**所有套利路径均不可盈利**：

| 套利路径 | 计算 | 结果 |
|---------|------|------|
| 买入-转卖同种 | 花 20 买 → 卖回得 10 | 亏损 10 仙石 |
| 灵韵↔仙石往返 | 100灵韵→1仙石→50灵韵 | 亏损 50% |
| 跨品套利（低→高） | 花 20 买钉耙 → 无法卖成八卦 | 不同品类不互通 |
| 买碎片→仙石→灵韵 | 花 20 买碎片 → 卖回 10 仙石 → 500 灵韵 | 净亏 1000 灵韵 |

> **暂不纳入**：炼丹材料（虎骨、灵蛇果等）、龙神材料（龙鳞等）。后续视运营情况再扩展。

## 3. serverCloud 数据模型

### 3.1 方案：仙石 + 物品 → Money 域

仙石天然是 Money 域货币，无需灵韵迁移。可交易物品数量也作为 Money 域的命名货币管理，「扣物品 + 扣/加仙石 + 增/减库存」在一个 `BatchCommit` 中原子完成。

### 3.2 键设计

| 域 | userId | key 模式 | 值 | 用途 |
|----|--------|---------|-----|------|
| Money | `playerId` | `xianshi` | 整数 | 玩家仙石余额 |
| Money | `playerId` | `mat_{consumableId}` | 整数 | 玩家持有的可交易物品数量 |
| Money | `SYSTEM(0)` | `bm_stock_{consumableId}` | 整数 | 黑商当前库存（`bm_` = black merchant） |

> `bm_` 前缀区分黑商数据与其他系统数据，避免 key 冲突。

**不需要灵韵的 Money 键**：灵韵保持本地存储不变，兑换时通过客户端本地扣减/增加灵韵 + `Money.Add/Cost(xianshi)` 完成。

**无交易限额**：个人购买、出售不限次数。库存上限（`max_stock = 5`）是唯一的数量约束。

**玩家持有量**：
- 通过 `Money.Get(playerId, "mat_{consumableId}")` 读取
- 在 Query 响应和 Result 响应中均返回，保证客户端显示一致

**不需要的数据**：
- 不需要价格存储（价格固定，写在配置表中）
- 不需要交易计数器（无交易次数限制）
- 不需要 NPC 钱包（黑商仙石无限，不做收支平衡）

### 3.3 数据规模

| 指标 | 数量 | 说明 |
|------|------|------|
| 商品种类 | 19 | 每种占 1 个 Money key（钯8 + 八卦8 + 天帝3，后续增至25） |
| SYSTEM Money key | 19 | `bm_stock_{id}`（SYSTEM 名下） |
| 玩家 Money key | 19 + 1 = 20 | `mat_{id}` + `xianshi` |
| BatchCommit 操作数/笔交易 | 3 | `Cost/Add 物品` + `Cost/Add 仙石` + `Add/Cost 库存` |

### 3.4 交易记录（List 域）

每笔交易完成后，服务端向 List 域追加一条流水记录，用于玩家回顾和 GM 审计。

#### 键设计

| 域 | userId | key | 值 | 用途 |
|----|--------|-----|-----|------|
| List | `playerId` | `bm_trade_log` | JSON 字符串 | 玩家个人交易记录 |

#### 单条记录结构

```lua
{
    t  = 1712745600,           -- 时间戳（服务端 Unix 秒）
    a  = "sell",               -- 动作："buy" / "sell" / "exchange_buy" / "exchange_sell"
    id = "rake_fragment_1",    -- 商品 ID（兑换时为 "xianshi"）
    n  = 1,                    -- 数量
    p  = 10,                   -- 单价（仙石）；兑换时为汇率
    xs = 60,                   -- 交易后仙石余额
}
```

> 字段名刻意缩短（`t/a/id/n/p/xs`），减小 List 条目体积。

#### 记录策略

| 策略 | 值 | 说明 |
|------|-----|------|
| 写入时机 | BatchCommit 成功后 | 只记录成功交易，失败不记 |
| 写入方式 | `serverCloud.list:Add(playerId, "bm_trade_log", record)` | 追加，不覆盖 |
| 最大条数 | 100 条 | Add 后检查总数，超出则 Delete 最旧条目（手动 FIFO） |
| 读取方式 | `serverCloud.list:Get(playerId, "bm_trade_log", events)` | 返回全量，客户端侧分页 |

> **API 对齐说明**：serverCloud List 域的 `Get` 返回某 key 下全部条目（无原生 offset/limit），分页在客户端内存中切片。100 条上限（~8KB）全量拉取开销可接受。FIFO 淘汰需在 Add 后主动检查并 Delete。

#### 数据开销

| 指标 | 估算 |
|------|------|
| 单条记录体积 | ~80 字节（JSON） |
| 100 条上限 | ~8 KB / 玩家 |
| 写入频率 | 每笔交易 1 次 List.Add，远低于限制 |

## 4. 协议设计

> **简化说明**：由于价格固定不随库存变化，v0.5 的二阶段锁（Lock → Confirm）不再是强需求。固定价格下不存在"锁定时看到的价格与成交时不同"的问题。因此 v0.6 采用**单步交易**——玩家点击买/卖，服务端直接执行 BatchCommit。
>
> **二阶段锁保留为可选**：如果产品层面希望保留"确认弹窗"的体验，可以在客户端本地实现确认对话框（纯 UI 层面），无需服务端锁机制。

### 4.1 C2S 协议

| 协议 | 字段 | 说明 |
|------|------|------|
| `C2S_BlackMerchantQuery` | — | 查询全部商品库存 + 玩家仙石余额 + 物品持有量 |
| `C2S_BlackMerchantBuy` | `consumableId: string, amount: int` | 从黑商购买物品（花仙石） |
| `C2S_BlackMerchantSell` | `consumableId: string, amount: int` | 向黑商出售物品（得仙石） |
| `C2S_BlackMerchantExchange` | `direction: "buy"/"sell", amount: int` | 仙石兑换：`"buy"` = 灵韵→仙石，`"sell"` = 仙石→灵韵 |
| `C2S_BlackMerchantHistory` | — | 查询全部交易记录（≤100 条，客户端侧分页） |

### 4.2 S2C 协议

| 协议 | 字段 | 说明 |
|------|------|------|
| `S2C_BlackMerchantData` | `items, xianshi, realm_ok` | 全部商品列表 + 仙石余额 + 是否达到境界要求 |
| `S2C_BlackMerchantResult` | `ok, action, id, amount, total, newStock, newXianshi, newHeld, reason?` | 交易结果 |
| `S2C_BlackMerchantExchangeResult` | `ok, direction, amount, newXianshi, lingYunDelta, reason?` | 兑换结果 |
| `S2C_BlackMerchantHistory` | `records: array` | 交易记录全量列表（按时间倒序，≤100 条） |

**S2C_BlackMerchantData 字段详情**：

```lua
{
    realm_ok = true,          -- 是否达到金丹期（客户端据此决定按钮是否可用）
    xianshi = 50,             -- 玩家仙石余额
    items = {
        rake_fragment_1 = {
            stock = 3,        -- 黑商库存
            buy_price = 10,   -- 收购价（固定）
            sell_price = 20,  -- 出售价（固定）
            held = 2,         -- 玩家持有量
        },
        -- ...
    }
}
```

### 4.4 兑换协议详细

#### 正向兑换（灵韵 → 仙石）

```
C2S_Exchange { direction = "buy", amount = 5 }

HandleExchange:
  1. 境界校验：玩家境界 >= 金丹期
  2. amount 校验：amount >= 1
  3. 计算灵韵消耗：lingYunCost = amount × 100
  4. 检查客户端灵韵：player.lingYun >= lingYunCost（由客户端报告）
  5. Money.Add(playerId, "xianshi", amount)
  6. 通知客户端扣减灵韵：lingYunDelta = -lingYunCost
  7. → S2C_ExchangeResult { ok=true, direction="buy", amount, newXianshi, lingYunDelta }
  客户端收到后：player.lingYun = player.lingYun + lingYunDelta
```

**崩溃安全性**：灵韵是客户端本地状态（本地扣减），仙石是 Money 域（服务端增加）。若 `Money.Add` 成功后崩溃、客户端未扣灵韵：玩家下次登录从存档加载灵韵（未扣状态），但仙石已增加 → 玩家"白赚"仙石。**防护**：先让客户端扣灵韵 → 客户端确认扣减后再 `Money.Add` → 若 Add 失败则客户端回滚灵韵。

#### 反向兑换（仙石 → 灵韵）

```
C2S_Exchange { direction = "sell", amount = 3 }

HandleExchange:
  1. 境界校验：玩家境界 >= 金丹期
  2. amount 校验：amount >= 1
  3. Money.Cost(playerId, "xianshi", amount)  -- 原子扣减，不足回滚
  4. 计算灵韵增加：lingYunGain = amount × 50
  5. Write-Ahead Log：记录 { playerId, lingYunGain, pending=true }
  6. 通知客户端增加灵韵：lingYunDelta = +lingYunGain
  7. 清除 WAL：pending = false
  8. → S2C_ExchangeResult { ok=true, direction="sell", amount, newXianshi, lingYunDelta }
```

**崩溃安全性**：仙石是 Money 域（先原子扣减），灵韵是本地状态（后增加）。若 `Money.Cost` 成功后崩溃、灵韵未加 → WAL 记录存在，下次登录时补偿灵韵。

## 5. 原子操作

### 5.1 出售操作（玩家 → 黑商）

```
C2S_BlackMerchantSell { consumableId = "rake_fragment_1", amount = 1 }

HandleSell:
  1. 境界校验：玩家境界 >= 金丹期，否则拒绝
  2. 白名单校验：MARKET_ITEMS[consumableId] 存在
  3. amount 校验：amount >= 1
  4. 前置校验（快速排除）：
     - BatchGet 读取：stock = bm_stock_{id}, held = mat_{id}
     - stock + amount <= max_stock（满仓不收）
     - held >= amount（物品不足）
  5. 查配置表获取收购价：buy_price = ITEMS[id].buy_price
  6. 计算总价：total = buy_price × amount
  7. 执行 BatchCommit:
     Money.Cost(playerId, "mat_" .. id, amount),         -- 扣物品
     Money.Add(playerId, "xianshi", total),               -- 加仙石
     Money.Add(SYSTEM_UID, "bm_stock_" .. id, amount),   -- 入库
  8. 写入交易记录：TradeLogger.Log（内部调用 `serverCloud.list:Add`，见 §3.4）
  9. 存档（见 §5.4）
  10. 计算交易后数据 → S2C_Result
```

### 5.2 购买操作（黑商 → 玩家）

```
C2S_BlackMerchantBuy { consumableId = "rake_fragment_1", amount = 1 }

HandleBuy:
  1. 境界校验：玩家境界 >= 金丹期，否则拒绝
  2. 白名单校验：MARKET_ITEMS[consumableId] 存在
  3. amount 校验：amount >= 1
  4. 前置校验（快速排除）：
     - BatchGet 读取：stock = bm_stock_{id}, xianshi
     - stock >= amount（无货不卖）
     - xianshi >= sell_price × amount（仙石不足）
  5. 查配置表获取出售价：sell_price = ITEMS[id].sell_price
  6. 计算总价：total = sell_price × amount
  7. 执行 BatchCommit:
     Money.Cost(playerId, "xianshi", total),              -- 扣仙石
     Money.Cost(SYSTEM_UID, "bm_stock_" .. id, amount),  -- 出库
     Money.Add(playerId, "mat_" .. id, amount),           -- 给物品
  8. 写入交易记录：TradeLogger.Log（内部调用 `serverCloud.list:Add`，见 §3.4）
  9. 存档（见 §5.4）
  10. 计算交易后数据 → S2C_Result
```

### 5.3 批量交易

固定价格下批量定价极其简单：

```lua
total = price × amount  -- 不再需要逐件累加
```

### 5.4 交易后立即存档

交易成功（BatchCommit 完成）后，服务端**立即触发一次玩家存档**（调用现有存档接口，不修改 HandleSaveGame 函数本身）。

**为什么需要**：
- Money 域的变更（仙石、物品数量）已经在 BatchCommit 中原子生效
- 但玩家的其他客户端状态（如 UI 缓存、灵韵余额等）仍在内存中
- 如果交易成功后服务端崩溃、玩家断线，未存档的客户端状态可能与 Money 域不一致
- 立即存档确保交易结果和客户端状态同步落盘

**实现要点**：
- 仅在 BatchCommit **成功**后触发，失败不存档
- 存档是异步的，不阻塞 S2C_Result 的发送

### 5.5 库存轮询（UI 打开期间）

| 参数 | 值 | 说明 |
|------|-----|------|
| 轮询间隔 | 10 秒 | UI 打开期间每 10 秒发一次 C2S_Query |
| 轮询内容 | 库存 + 仙石余额 + 持有量 | 价格固定不需轮询，但库存和余额实时变化 |
| 关闭 UI | 停止轮询 | 释放网络资源 |

**服务端 Query 开销**：单次 BatchGet 39 个 key（19 库存 + 19 持有量 + 1 仙石），6 次/分钟/人，远低于 300 次/分钟的 serverCloud 读限制。

## 7. 与现有系统的关系

### 7.1 前置依赖链

```
C4 Money.Cost 验证 ✅
C5 BatchCommit 验证 ✅
        │
        ↓
   万界黑商上线（无灵韵迁移依赖！）
        ├─ BlackMerchant/（独立系统目录，见 §9.1）
        │   ├─ BlackMerchantHandler.lua（路由 + 交易 + 兑换）
        │   ├─ MarketConfig.lua（商品配置 + 汇率）
        │   ├─ TradeLogger.lua（交易记录）
        │   └─ WalManager.lua（WAL 补偿）
        ├─ ui/BlackMerchantUI.lua（客户端）
        └─ 可交易物品 → Money 迁移（16 种，同 6.1 模式）
```

**关键简化**：仙石方案**完全绕过灵韵迁移（6.1）**。灵韵保持本地存储不变，仙石作为 Money 域原生货币直接使用。这消除了 v0.5 中最大的风险项——涉及 15+ 文件、30+ 修改点的灵韵迁移工程。

### 7.2 物品迁移方案

可交易物品仍需迁移到 Money 域（与 6.1 同模式，但与灵韵迁移解耦）：

| 阶段 | 操作 | 说明 |
|------|------|------|
| M1 双写期 | 物品获取时：背包 + Money.Add 双写 | 新增物品同时写入两处 |
| M2 存量迁移 | 首次登录时：遍历背包/仓库中的可交易物品 → Money.Add | 一次性同步历史数据 |
| M3 切换读取 | 背包 UI 中可交易物品数量从 Money.Get 读取 | 真值从 Money 域获取 |
| M4 废弃旧数据 | 停止写入背包中的可交易物品 | Money 域成为唯一数据源 |

> 注意：可交易物品仍然**显示在背包中**（UI 层面），只是数量的真值改为从 Money 域读取。

### 7.3 系统独立性原则 🔴

**万界黑商是完全独立的系统模块，不并入 `server_main.lua` 或任何待优化的现有代码。**

| 原则 | 说明 |
|------|------|
| **独立文件** | 全部代码在 `BlackMerchant/` 目录下，不向 `server_main.lua` 添加任何逻辑 |
| **自注册** | Handler 通过 `Register(session)` 自行订阅协议路由，`server_main.lua` 只需一行 `require` + 一行 `Register` 调用 |
| **不依赖 B4** | 不修改 `HandleCreateChar`、`HandleLoadGame`、`HandleSaveGame/V4` 三个核心函数 |
| **WAL 自治** | 兑换 WAL 的写入和补偿检查均在 `BlackMerchantHandler` 内部完成，通过 `ClientReady` 事件钩子触发，不侵入 `HandleLoadGame` |
| **配置自包含** | `MarketConfig.lua` 独立维护商品表和汇率，不依赖外部配置文件 |
| **数据隔离** | Money 域 key 以 `bm_` / `mat_` 前缀隔离，List 域 key 以 `bm_` 前缀隔离，不与其他系统冲突 |

**与 server_main.lua 的唯一接口**：

```lua
-- server_main.lua 中仅需添加以下两行（不修改任何现有函数）：
local BlackMerchant = require("BlackMerchant.BlackMerchantHandler")
BlackMerchant.Register(session)  -- 在 session 创建后调用
```

**为什么不并入 server_main.lua**：
1. `server_main.lua` 当前承载了过多职责（B4 待拆分），继续往里塞功能会加剧维护困难
2. 黑商系统的生命周期独立——上线、下线、配置变更都不应影响其他系统
3. 独立模块便于测试——可以单独对 BlackMerchantHandler 编写单元测试

### 7.4 与路线图的关系

| 路线图项 | 关系 |
|---------|------|
| 6.2 系统回收（NPC 回收） | **被替代**：黑商收购覆盖了 NPC 回收功能，且更有趣 |
| 6.5 拍卖行 | **互补**：拍卖行是玩家对玩家，黑商是玩家对 NPC，定位不同 |
| ~~6.1 灵韵迁移~~ | **不再依赖**：仙石方案绕过灵韵迁移 |
| B4 SaveHandler 拆分 | **无直接依赖**：黑商独立模块，不增加 B4 复杂度 |

### 9.1 独立目录结构

> `server_main.lua` 仅需 `require` + `Register` 两行胶水代码（见 §7.3）。

## 10. 风险与缓解

| 风险 | 严重度 | 缓解方案 |
|------|--------|---------|
| **兑换崩溃导致资产丢失** | 🟡 中 | 反向兑换使用 WAL（§4.4），正向兑换先扣灵韵后加仙石（天然安全） |
| **物品迁移数据不一致** | 🟡 中 | 与 6.1 同风险同方案：Money 域为真值，背包为展示缓存 |
| **BatchCommit 失败** | 🟢 低 | 交易失败，客户端提示"余额不足，请重试"。无资产损失 |
| **恶意刷库存** | 🟢 低 | 库存上限 5 约束总量。小号也需要先获得物品（BOSS 掉落），成本不低 |
| **碎片稀缺度破坏** | 🟡 中 | 库存上限 5，售价 2 倍收购价。碎片仍然稀缺，只是多了一个应急渠道 |
| **仙石通胀** | 🟢 低 | 仙石只能兑换获得（100 灵韵 = 1 仙石），本质是灵韵的代币化，不增加总量 |
| ~~**灵韵迁移风险**~~ | ~~🟠~~ | ✅ **已消除**：仙石方案完全绕过灵韵迁移 |
| ~~**价格竞态**~~ | ~~🟡~~ | ✅ **已消除**：固定价格无竞态可能 |

**风险对比（v0.5 → v0.6）**：

| 维度 | v0.5（灵韵直接交易） | v0.6（仙石方案） |
|------|---------------------|-----------------|
| 灵韵迁移 | 必须，涉及 15+ 文件 | 不需要 |
| 定价复杂度 | 动态公式，逐件累加 | 固定价格，`price × amount` |
| 二阶段锁 | 必须（价格随库存变） | 不需要（固定价格） |
| 并发复杂度 | 锁管理 + 过期 + 断线清理 | 仅 BatchCommit 原子性 |
| 服务端代码量 | ~450 行 | ~470 行（4 模块，但逻辑更简单） |
| 总风险等级 | 🟠 中高 | 🟢 低 |

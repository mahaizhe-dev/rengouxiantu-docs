## 一、活动概览

### 1.1 核心玩法

击败 BOSS 有独立概率掉落五种**活动限定道具**，集齐后可在中洲城 NPC 处兑换灵韵（总计可兑换 **20,000 灵韵**），或通过黑市交易。**天庭福袋在 NPC 处开启**（支持十连开），福袋开启数量有全服排行榜，排名前三的玩家活动结束后在交流群获得现金红包奖励。活动手动下架后，所有相关系统同步关闭，不影响常规掉落。

### 1.2 活动道具一览

| 道具 ID | 名称 | 图标 | 品质 | 可黑市交易 | 说明 |
|---------|------|------|------|-----------|------|
| `mayday_wu` | 五 | 🎏 | 紫色 | 是 | 五一快乐·五字 |
| `mayday_yi` | 一 | 🎏 | 紫色 | 是 | 五一快乐·一字 |
| `mayday_kuai` | 快 | 🎏 | 紫色 | 是 | 五一快乐·快字 |
| `mayday_le` | 乐 | 🎏 | 紫色 | 是 | 五一快乐·乐字 |
| `mayday_fudai` | 天庭福袋 | 🧧 | 橙色 | 否 | 在 NPC 处开启，随机获得一种奖励 |

> 正式实现时为五个道具各自生成 64×64 像素风图标（透明背景）。emoji 仅为文档占位符。

### 1.3 活动控制

- **开关方式**：`EventConfig.ACTIVE_EVENT.enabled`（手动开关，发布时控制）
- **不使用时间判断**：上架 = `enabled = true`，下架 = `enabled = false`，由版本发布控制
- **活动 ID**：`mayday_2026`（唯一标识，用于存档隔离）

## 二、通用活动掉落框架设计

### 2.1 框架目标

活动掉落是一个**通用框架**，五一活动是第一个实例。框架需满足：

1. **配置驱动**：换一套 `items` + `dropRates` + `exchanges` + `fudaiPool` 即可上线新活动
2. **独立性**：活动掉落与常规掉落完全独立，互不影响
3. **热插拔**：`enabled = false` 时所有活动逻辑静默跳过，零运行时开销
4. **存档隔离**：每个活动用 `eventId` 隔离存档数据
5. **排行榜**：福袋开启数量全服排行，支持奖励引导

### 2.2 框架数据结构

### 2.3 LootSystem 对接方式（独立性保证）

```lua
-- LootSystem.lua 末尾追加（唯一的改动点）
-- 活动掉落判定 —— 与常规掉落完全独立，互不干扰
function LootSystem.RollEventDrops(monster, chapterId)
    local EventConfig = require("config.EventConfig")
    if not EventConfig.IsActive() then return end  -- 活动未开启，直接跳过

    local event = EventConfig.ACTIVE_EVENT
    local rates = LootSystem.GetEventDropRates(event, monster, chapterId)
    if not rates then return end

    local dropped = {}
    for itemId, chance in pairs(rates) do
        if math.random() < chance then
            local ok = InventorySystem.AddConsumable(itemId, 1)
            if ok then
                dropped[#dropped + 1] = { itemId = itemId, count = 1 }
            end
        end
    end

    if #dropped > 0 then
        EventBus.Emit("event_item_drop", { items = dropped, eventId = event.eventId })
    end
end
```

**独立性保证**：
- `RollEventDrops` 在常规 `GenerateDrops` 之后调用，不修改常规掉落的任何逻辑
- 活动关闭后 `IsActive()` 返回 false，函数第一行 return，零开销
- 活动道具走 `InventorySystem.AddConsumable`，与常规装备/消耗品掉落互不干扰

### 3.4 掉率查找优先级

```lua
function LootSystem.GetEventDropRates(event, monster, chapterId)
    -- 优先级1：怪物 ID 覆盖（四龙等特殊 BOSS）
    if event.monsterOverrides and event.monsterOverrides[monster.typeId] then
        return event.monsterOverrides[monster.typeId]
    end
    -- 优先级2：章节 + 阶级
    local chapterRates = event.dropRates[chapterId]
    if not chapterRates then return nil end
    return chapterRates[monster.category]
end
```

### 4.3 抽奖算法

```lua
-- 通用加权随机抽取（服务端执行）
function EventSystem.RollFudaiReward(pool)
    local totalWeight = 0
    for _, entry in ipairs(pool) do
        totalWeight = totalWeight + entry.weight
    end

    local roll = math.random() * totalWeight
    local cumulative = 0
    for _, entry in ipairs(pool) do
        cumulative = cumulative + entry.weight
        if roll < cumulative then
            return entry
        end
    end
    -- 兜底：返回最后一个
    return pool[#pool]
end
```

### 4.4 福袋开启逻辑（服务端权威）

```lua
-- ═══ 模块级变量 ═══
local SYSTEM_UID = 0                         -- 全服公共记录所有者（与黑市一致）
local PULL_RECORD_KEY = "evt_fudai_records"  -- 全服抽取记录列表 key
local MAX_PULL_RECORDS = 50                  -- 最多保留最近 50 条
local fudaiLocks_ = {}                       -- 内存锁（key = userId）

-- ═══ FIFO 截断（参考 BlackMerchantTradeLogger.TrimList）═══
local function TrimPullRecords()
    serverCloud.list:Get(SYSTEM_UID, PULL_RECORD_KEY, {
        ok = function(list)
            if #list <= MAX_PULL_RECORDS then return end
            table.sort(list, function(a, b)
                local tA = (a.value and a.value.ts) or 0
                local tB = (b.value and b.value.ts) or 0
                return tA < tB  -- 最旧的在前面
            end)
            local toDelete = #list - MAX_PULL_RECORDS
            for i = 1, toDelete do
                if list[i] and list[i].list_id then
                    serverCloud.list:Delete(list[i].list_id)
                end
            end
        end,
    })
end

-- 服务端处理 C2S_EventOpenFudai
function HandleEventOpenFudai(eventType, eventData)
    local connection = eventData["Connection"]:GetPtr("Connection")
    local connKey = ConnKey(connection)
    local userId = GetUserId(connKey)
    local currentSlot = GetCurrentSlot(connKey)  -- 当前存档槽位

    -- ═══ 防护 #1：内存锁（防并发重放，key = userId）═══
    if fudaiLocks_[userId] then return end
    fudaiLocks_[userId] = true

    local requestCount = eventData["Count"]:GetInt()  -- 1 或 10

    -- 请求数量白名单
    if requestCount ~= 1 and requestCount ~= 10 then
        fudaiLocks_[userId] = nil
        return  -- 非法数量，静默拒绝
    end

    -- ═══ 防护 #2：存档槽位锁（防止与自动存档 read-modify-write 竞态）═══
    if not ServerSession.AcquireSlotLock(userId, currentSlot) then
        fudaiLocks_[userId] = nil
        SendError(connection, "操作中，请稍后")
        return
    end

    -- 释放所有锁的辅助函数（确保所有退出路径调用）
    local function releaseLocks()
        fudaiLocks_[userId] = nil
        ServerSession.ReleaseSlotLock(userId, currentSlot)
    end

    -- 读取玩家存档（键名 = "save_" .. slot，与 TrialHandler 等一致）
    local saveKey = "save_" .. currentSlot
    serverCloud:Get(userId, saveKey, {
        ok = function(scores)
            local saveData = scores[saveKey] or {}
            local backpack = saveData.backpack or {}

            -- ═══ 防护 #3：服务端校验背包福袋数量 ═══
            local heldCount = 0
            for _, slot in pairs(backpack) do
                if slot.consumableId == "mayday_fudai" then
                    heldCount = heldCount + (slot.count or 0)
                end
            end

            if heldCount <= 0 then
                releaseLocks()
                SendError(connection, "没有天庭福袋")
                return
            end

            -- 实际开启数量 = min(请求数, 持有数)
            local actualCount = math.min(requestCount, heldCount)

            -- ═══ 抽奖：服务端执行 N 次独立抽取 ═══
            local pool = EventConfig.ACTIVE_EVENT.fudaiPool
            local rollResults = {}
            local totalRewards = { exp = 0, gold = 0, consumables = {} }

            for i = 1, actualCount do
                local entry = EventSystem.RollFudaiReward(pool)
                rollResults[#rollResults + 1] = {
                    id = entry.id, name = entry.name, rarity = entry.rarity
                }

                for _, r in ipairs(entry.rewards) do
                    if r.type == "exp" then
                        totalRewards.exp = totalRewards.exp + r.count
                    elseif r.type == "gold" then
                        totalRewards.gold = totalRewards.gold + r.count
                    elseif r.type == "consumable" then
                        totalRewards.consumables[r.id] =
                            (totalRewards.consumables[r.id] or 0) + r.count
                    end
                end
            end

            -- ═══ 原子事务提交（扣福袋 + 发奖励 + 排行递增）═══
            -- ⚠️ 开发注意：AddToBackpack / RemoveFromBackpack 目前定义在
            -- BlackMerchantHandler.lua 的 local 作用域，需提取到共享模块
            -- （如 BackpackUtils.lua）供 EventHandler 复用。
            RemoveFromBackpack(backpack, "mayday_fudai", actualCount)

            saveData.player.exp = (saveData.player.exp or 0) + totalRewards.exp
            saveData.player.gold = (saveData.player.gold or 0) + totalRewards.gold

            for consumableId, count in pairs(totalRewards.consumables) do
                AddToBackpack(backpack, consumableId, count)
            end

            local commit = serverCloud:BatchCommit("event_open_fudai")
            commit:ScoreSet(userId, saveKey, saveData)

            -- 排行榜计数（服务端权威递增）
            commit:ScoreAddInt(userId, "evt_fudai_opened", actualCount)

            commit:Commit({
                ok = function()
                    -- ═══ 全服抽取记录：稀有+最稀有写入 serverCloud.list ═══
                    local playerName = saveData.player and saveData.player.name or "???"
                    -- 内联脱敏：取首尾字符，中间替换为 *（与黑市 TradeLogger 对齐，后续可提取为工具函数）
                    local chars = {}
                    for _, c in utf8.codes(playerName) do chars[#chars + 1] = utf8.char(c) end
                    local masked = #chars <= 2 and (chars[1] .. "*")
                        or (chars[1] .. string.rep("*", #chars - 2) .. chars[#chars])

                    for _, r in ipairs(rollResults) do
                        if r.rarity == "rare" or r.rarity == "legendary" then
                            serverCloud.list:Add(SYSTEM_UID, PULL_RECORD_KEY, {
                                displayName = masked,
                                name = r.name,
                                rarity = r.rarity,
                                ts = os.time(),
                            })
                        end
                    end
                    -- 异步截断超出上限的旧记录
                    TrimPullRecords()

                    -- 发送结果给客户端
                    local reply = VariantMap()
                    reply["Success"] = Variant(true)
                    reply["Count"] = Variant(actualCount)
                    reply["Results"] = Variant(cjson.encode(rollResults))
                    reply["TotalExp"] = Variant(totalRewards.exp)
                    reply["TotalGold"] = Variant(totalRewards.gold)
                    reply["Consumables"] = Variant(cjson.encode(totalRewards.consumables))
                    SafeSend(connection, SaveProtocol.S2C_EventOpenFudaiResult, reply)

                    releaseLocks()
                end,
                error = function(code, reason)
                    releaseLocks()
                    SendError(connection, "开启失败，请重试")
                end,
            })
        end,
        error = function()
            releaseLocks()
        end,
    })
end
```

```lua
-- ═══ 客户端：收到服务端返回 ═══
function HandlePullRecordsData(eventType, eventData)
    local raw = eventData["Records"]:GetString()
    local records = cjson.decode(raw)  -- { {displayName, name, rarity, ts}, ... }

    local displayList = {}
    local now = os.time()
    for _, r in ipairs(records) do

                playerName = r.displayName,     -- 已脱敏，直接使用
                rewardName = r.name,
                rarity = r.rarity,
                timeAgo = FormatTimeAgo(r.ts),  -- "3分钟前"
            }
        end
    end
    UpdatePullRecordUI(displayList)
end

-- ═══ 服务端：处理 C2S_EventGetPullRecords ═══
function HandleEventGetPullRecords(eventType, eventData)
    local connection = eventData["Connection"]:GetPtr("Connection")

    serverCloud.list:Get(SYSTEM_UID, PULL_RECORD_KEY, {
        ok = function(list)
            -- list: 数组，每项 { list_id, value = { displayName, name, rarity, ts } }
            local records = {}
            for _, item in ipairs(list) do
                records[#records + 1] = item.value
            end
            -- 按时间降序（最新在前）
            table.sort(records, function(a, b)
                return (a.ts or 0) > (b.ts or 0)
            end)

            local reply = VariantMap()
            reply["Records"] = Variant(cjson.encode(records))
            SafeSend(connection, SaveProtocol.S2C_EventPullRecordsData, reply)
        end,
        error = function()
            -- 查询失败返回空数组
            local reply = VariantMap()
            reply["Records"] = Variant("[]")
            SafeSend(connection, SaveProtocol.S2C_EventPullRecordsData, reply)
        end,
    })
end
```

### 5.4 排行榜防作弊

| 防线 | 机制 | 说明 |
|------|------|------|
| **服务端权威计数** | `ScoreAddInt` 原子递增 | 客户端无法伪造排行分数，只有服务端在扣除福袋成功后才递增 |
| **福袋来源可追溯** | 福袋仅从 BOSS 掉落和字→福袋兑换获得 | 无法凭空产生福袋，掉率受控 |
| **内存锁 + SlotLock 防并发** | `fudaiLocks_[userId]` + `ServerSession.AcquireSlotLock` | 同一用户并发请求被拦截，防与自动存档竞态 |
| **背包服务端校验** | 先读 saveData 校验实际持有量 | 客户端伪造数量无效 |
| **请求频率限制** | 全局 `RateLimitedSubscribe` 30次/10秒（滑动窗口） | 自动化脚本受限 |
| **请求数量白名单** | 仅允许 count=1 或 count=10 | 非法数量直接丢弃 |

**核心原则**：排行榜分数（`evt_fudai_opened`）**只能**在服务端通过 `BatchCommit` 原子递增，递增前提是 `RemoveFromBackpack` 成功扣除了对应数量的福袋。客户端无法直接写入此 iscores 键。

- **字的价值阶梯**：组合对（500/2字 = 250灵韵/字）> 单字（0灵韵，仅换福袋）→ 有对的字远比落单的字值钱

| 系统 | 行为 | 独立性保证 |
|------|------|-----------|
| **掉落** | `IsActive()` 返回 false，第一行 return | 常规掉落零影响 |
| **NPC** | `eventBound` 过滤隐藏 | 其他 NPC 不受影响 |
| **兑换** | 入口关闭（NPC 隐藏） | — |

| **常规掉落** | **完全不受影响** | 活动掉落是独立追加逻辑 |

### 10.2 兑换 Handler 伪代码（EventHandler.lua）

> 与 §4.4 开福袋 Handler 对应，覆盖 `C2S_EventExchange` 的完整服务端处理流程。

```lua
-- ═══ 兑换处理（服务端） ═══
-- 注册：RateLimitedSubscribe(SaveProtocol.C2S_EventExchange, HandleEventExchange)

function HandleEventExchange(connection, eventData)
    -- ═══ 前置校验 ═══
    if not EventConfig.ACTIVE_EVENT.enabled then return end

    local connKey = ServerSession.ConnKey(connection)
    local userId = ServerSession.GetUserId(connKey)
    local currentSlot = ServerSession.GetCurrentSlot(connKey)
    if not userId or not currentSlot then return end

    local exchangeId = eventData["ExchangeId"]:GetString()

    -- 查找兑换配置
    local exchangeCfg = nil
    for _, ex in ipairs(EventConfig.ACTIVE_EVENT.exchanges) do
        if ex.id == exchangeId then exchangeCfg = ex; break end
    end
    if not exchangeCfg then
        SendError(connection, "无效的兑换项")
        return
    end

    -- ═══ 槽位锁 ═══
    if not ServerSession.AcquireSlotLock(userId, currentSlot) then
        SendError(connection, "操作中，请稍后")
        return
    end

    local function releaseLock()
        ServerSession.ReleaseSlotLock(userId, currentSlot)
    end

    -- ═══ 读取存档 ═══
    local saveKey = "save_" .. currentSlot
    serverCloud:Get(userId, saveKey, {
        ok = function(scores)
            local saveData = scores[saveKey] or {}
            local backpack = saveData.backpack or {}
            local eventData_ = saveData.event_data or {}
            local evtState = eventData_[EventConfig.ACTIVE_EVENT.eventId] or {}
            local exchanged = evtState.exchanged or {}

            -- ═══ 校验限购 ═══
            local usedCount = exchanged[exchangeId] or 0
            if exchangeCfg.limit > 0 and usedCount >= exchangeCfg.limit then
                releaseLock()
                SendError(connection, "已达兑换上限")
                return
            end

            -- ═══ 校验背包字卡数量 ═══
            for itemId, needCount in pairs(exchangeCfg.cost) do
                local held = 0
                for _, slot in pairs(backpack) do
                    if slot.consumableId == itemId then
                        held = held + (slot.count or 0)
                    end
                end
                if held < needCount then
                    releaseLock()
                    SendError(connection, "材料不足")
                    return
                end
            end

            -- ═══ 扣字卡 ═══
            -- ⚠️ 复用 BackpackUtils（见 §4.4 注释）
            for itemId, needCount in pairs(exchangeCfg.cost) do
                RemoveFromBackpack(backpack, itemId, needCount)
            end

            -- ═══ 发奖励 ═══
            local reward = exchangeCfg.reward
            if reward.type == "lingYun" then
                saveData.player.lingYun = (saveData.player.lingYun or 0) + reward.count
            elseif reward.type == "consumable" then
                AddToBackpack(backpack, reward.id, reward.count)
            end

            -- ═══ 记录兑换次数 ═══
            exchanged[exchangeId] = usedCount + 1
            evtState.exchanged = exchanged
            eventData_[EventConfig.ACTIVE_EVENT.eventId] = evtState
            saveData.event_data = eventData_

            -- ═══ 原子提交 ═══
            local commit = serverCloud:BatchCommit("event_exchange")
            commit:ScoreSet(userId, saveKey, saveData)
            commit:Commit({
                ok = function()
                    local reply = VariantMap()
                    reply["Success"] = Variant(true)
                    reply["ExchangeId"] = Variant(exchangeId)
                    reply["UsedCount"] = Variant(usedCount + 1)
                    reply["Reward"] = Variant(cjson.encode(reward))
                    SafeSend(connection, SaveProtocol.S2C_EventExchangeResult, reply)
                    releaseLock()
                end,
                error = function()
                    releaseLock()
                    SendError(connection, "兑换失败，请重试")
                end,
            })
        end,
        error = function()
            releaseLock()
        end,
    })
end
```

### 10.3 防作弊体系总览

#### 三层防护架构（基础防护，服务端权威优先）

```
┌─────────────────────────────────────────────────┐
│ L1：请求验证 + 频率控制（网络层 + 服务端）         │
│  · RateLimitedSubscribe：30 次/10 秒（滑动窗口） │
│  · 内存锁 fudaiLocks_[userId] 防同一用户并发      │
│  · SlotLock 防与自动存档 read-modify-write 竞态   │
│  · 请求 count 白名单校验（仅允许 1 或 10）        │
│  · 背包福袋数量服务端读取校验                      │
├─────────────────────────────────────────────────┤
│ L2：原子事务（服务端）                             │
│  · serverCloud:BatchCommit 一次性提交              │
│  · 扣背包 + 发奖励 + 排行递增，要么全成功要么全失败│
│  · 排行键 evt_fudai_opened 仅通过 ScoreAddInt 递增│
├─────────────────────────────────────────────────┤
│ L3：客户端 UI 防护（辅助层，非安全边界）            │
│  · 点击后立即禁用按钮，等待服务端响应              │
│  · 结果动画播放期间禁止再次操作                    │
└─────────────────────────────────────────────────┘
```

> **设计原则**：仅做基础防护，服务端权威控制为核心。不引入复杂的审计层或行为分析。

#### 关键防线详解

| 攻击向量 | 防护层 | 防护措施 |
|---------|--------|---------|
| **客户端伪造 count=999** | L1 | 白名单校验，仅允许 1/10 |
| **并发重放请求** | L1 | 频率限制 30次/10秒 + 内存锁 + SlotLock |
| **伪造背包数据** | L1 | 服务端读取 saveData 校验实际持有 |
| **直接写入排行分数** | L2 | `evt_fudai_opened` 只能通过 `ScoreAddInt` 递增，无 `SetInt` 入口 |
| **不扣福袋直接加排行分** | L2 | `BatchCommit` 原子性：扣背包和加分在同一事务 |
| **自动化脚本刷福袋** | L1 | 频率限制 30次/10秒 + 福袋来源受掉率控制（0.5%~5%） |
| **活动结束后篡改排行** | — | 活动关闭后 NPC 隐藏，无开启入口；排行数据只增不减 |

## 十三、经济影响评估（v3.0 修订）

### 13.1 掉落产出估算

（与 v2.0 一致，掉率未变）

**一周产出约 8~10 个活动道具**（字+福袋混合）

### 13.2 福袋期望值分析（v3.0 · 16 种奖池 · 3 层稀有度）

中度玩家一周约获得 4~5 个福袋（掉落+字兑换），按 16 种奖励池：

| 资源 | 单次期望 | 5次期望（周产出） | 说明 |
|------|---------|-----------------|------|
| 经验 | 841.4 | 4,207 | 含直接经验+食物经验+修炼果+稀有食物 |
| 金币（去金砖） | 144.4 | 722 | 含直接金币+金条+食物卖价 |
| 灵韵 | 1.5 | 7.5 | 仅灵韵果（3% × 50灵韵），池内无直接灵韵 |
| 食物 | ~0.6 份 | ~3 份 | 4 种一般食物+2 种稀有食物加权 |
| 金砖 | 0.005 个 | 0.025 个 | 约 200 次出 1 个，全服级稀有事件 |

> **与 v2.0 对比**：v2.0 单次约等于精英掉落的 25%，v3.0 提升至约 **1.6 个精英掉落**（经验 841 / 金币 144），但灵韵几乎不产出（1.5 vs 5.9），保护兑换系统核心价值。
> **灵韵保护**：池内无直接灵韵产出，仅 3% 概率获得灵韵果（+50），灵韵获取只能走集字兑换。
> **惊喜感**：0.5% 金砖（价值 10 万金）作为唯一最稀有奖励，全服抽取记录可见，社交传播价值极高。

### 13.3 灵韵产出估算

| 场景 | 灵韵收入 | 说明 |
|------|---------|------|
| **轻度玩家**（一周 5~6 个字） | ~1,000~2,000 | B 组 1+1 兑换 2~4 次，多余字换福袋 |
| **中度玩家**（一周 8~12 个字） | ~3,000~5,000 | B 组满兑（10,000）需 10+10 个配对字 |
| **重度玩家**（一周 20+ 个字） | ~10,000~15,000 | A 组全兑 + B 组大部分 |
| **肝帝**（持续多周） | 20,000 上限 | A 组 1W + B 组 2×5000 全兑完 |

> **v3.3 变更说明**：单字不再直接兑换灵韵，仅可 1:1 换福袋。灵韵只来自 A 组（四字齐全 1W）和 B 组（1+1 组合 500×10 次×2 条 = 1W）。这使得灵韵获取需要**配对字**，而非任意字通吃，提升了集字策略性。

### 13.4 排行榜经济影响

- 排行奖励为**现金红包**（群外发放），不影响游戏内经济
- 福袋来源受掉率控制（0.5%~5%），刷排行的唯一途径是大量击杀 BOSS
- 预估活动期间排行第一名开启量 ~100~200 个福袋

### 13.5 金砖风险评估

| 项目 | 数据 |
|------|------|
| 金砖概率 | 0.5%（200 次期望出 1 个） |
| 金砖价值 | 100,000 金币（sellPrice） |
| 排名第一（200 个福袋） | 期望出 1 个金砖（37% 概率一个都不出） |
| 全服 100 人各开 10 个 | 1000 次总开启，期望出 5 个金砖 |
| 经济冲击 | 50 万金币 / 全服 ≈ 每人 5000 金（Lv.50 约 50 次精英掉落） |
| **结论** | 冲击可控，金砖通过全服抽取记录产生社交传播价值远超其经济影响 |

> 金砖 0.5% 概率保证了它是**社交事件**而非**经济通道**。
> 全服抽取记录中金砖出现频率约每 200 次开启一次，既足够稀有，又不至于活动全程无人出。
> 极端情况（某人连出 2 个金砖）概率仅 0.0025%，即便发生，20 万金对整体经济影响可接受。

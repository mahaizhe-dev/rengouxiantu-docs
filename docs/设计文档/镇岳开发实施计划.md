## 零、影响评估（存档/线上数据/账号级属性）

> **本节回答核心问题**: 镇岳上线后，对现有玩家的存档数据、线上数据、账号级属性有何影响？是否需要存档版本迁移？

### 0.1 存档结构影响

**当前存档版本**: `CURRENT_SAVE_VERSION = 17`（SaveState.lua）

| 变更项 | 操作 | 影响范围 | 风险级别 | 说明 |
|--------|------|---------|---------|------|
| `bloodAccumulator` | SerializePlayer/DeserializePlayer 新增字段 | 仅镇岳角色 | 🟢 无风险 | `data.bloodAccumulator or 0` 兜底，旧存档无此字段时默认 0 |
| `classId` | 已在 v16 迁移中加入 | 所有角色 | ✅ 已完成 | 旧存档 `classId or "monk"` 兜底 |
| `skillBar` | 现有 SerializeSkills 已支持 | 仅镇岳角色 | 🟢 无风险 | 镇岳只有 2 个主动槽，skillBar 格式不变 |
| `skillState` | **不持久化**（瞬态） | 无 | ✅ 无影响 | `SkillSystem.skillState` 每次 Init 重置为 `{}`，焚血 toggle 状态重登后回到 `defaultActive=true` |

**结论: 不需要版本迁移（不需要 v18）**

新增的 `bloodAccumulator` 字段使用 `or 0` 默认值模式（与 `pillKillHeal or 0`、`fruitFortune or 0` 等几十个已有字段完全一致），旧存档自然兼容。不新增 MIGRATIONS 条目，不递增 CURRENT_SAVE_VERSION。

### 0.2 线上数据影响

| 数据类别 | 影响 | 说明 |
|---------|------|------|
| **现有罗汉/太虚角色存档** | ✅ 零影响 | 不修改任何已有字段，不改变已有公式，不改变已有属性衍生链 |
| **已有装备数值** | ✅ 零影响 | 不修改 EquipmentData、TIER_MULTIPLIER、品质公式 |
| **已有技能 TypeHandler** | ✅ 零影响 | 新增 4 个 TypeHandler，不修改现有 10+ 个 TypeHandler 的任何代码 |
| **已有伤害计算公式** | ✅ 零影响 | CalcDamage、ApplyCrit、TakeDamage 函数体不做任何修改；镇岳技能通过调用这些函数接入管线 |
| **cloudScore（clientCloud/排行榜）** | ⚠️ UI 补全 | 服务端零影响（classId 自动写入）；客户端 LeaderboardUI 需补充镇岳子 Tab（见 §12.1） |
| **MedalConfig / AtlasSystem** | ✅ 已存在 | `zhenyue` 条目在 MedalConfig（L44,110-121,253）和 AtlasSystem（L43,121,124,793,834）中已预埋 |

### 0.3 账号级属性影响

| 属性类别 | 影响 | 说明 |
|---------|------|------|
| **角色槽位** | ✅ 零影响 | MAX_SLOTS=4，不变；镇岳作为新 classId 选项，不影响槽位数 |
| **跨角色共享数据** | ✅ 零影响 | 本游戏无跨角色共享的账号级属性；每个角色槽独立存档 |

## 六、技能 TypeHandler 实现（通用机制设计）

> **核心原则**: 所有 TypeHandler 必须是**通用的数据驱动实现**，不能硬编码 skillId 或 classId。  
> 任何新职业如果有类似机制（HP 系数伤害、toggle 被动、累计触发），应能**零代码修改**复用同一 TypeHandler，只需在 SkillData 中配置参数。
>
> **架构依据**: 参见 `战斗系统架构.md` §7 TypeHandlers 注册表 — "架构层通用性良好"、§13.6 通用性评估 — TypeHandlers 注册表评级为 ✅ 通用。

### 6.1 伤害管线集成方式

镇岳技能接入现有伤害管线的方式（参见 `战斗系统架构.md` §3.2-3.4）：

| 技能 | 伤害来源类型 | effectiveAtk 来源 | CalcDamage | ApplyCrit | TakeDamage | skillDmg% |
|------|------------|------------------|:----------:|:---------:|:----------:|:---------:|
| 裂山拳 | HP 系数 | `maxHP × coefficient` | ✅ | ✅ | ✅ | ✅ 吃 |
| 地脉回涌 | HP 系数 | `maxHP × coefficient` | ✅ | ✅ | ✅ | ✅ 吃 |
| 焚血之躯 | 真实伤害 | `maxHP × burnPercent × dt` | ❌ 无视DEF | ✅ | ✅ | ❌ 不吃 |
| 血爆 | 真实伤害 | `maxHP × damagePercent × (1+bloodRageValue)` | ❌ 无视DEF | ✅ | ✅ | ❌ 不吃 |

> **"吃 skillDmg%"的含义**: `raw = maxHP × coefficient × (1 + player:GetSkillDmgPercent())`，应用在 CalcDamage 之前的 effectiveAtk 阶段，与普通技能的 `effATK × multiplier × (1 + skillDmg%)` 同层级。

### 6.2 `hp_damage` TypeHandler（通用 HP 系数伤害）

**适用范围**: 任何"以自身 HP 为基础计算伤害"的技能，不限职业。

```lua
-- 通用设计：读取 skill 表的数据字段，不硬编码任何 skillId
SkillSystem.RegisterType("hp_damage", {
    cast = function(skill, player)
        local maxHp = player:GetMaxHp()

        -- 1. 释放限制: HP 不足以支付消耗则拒绝
        local cost = maxHp * (skill.hpCost or 0)
        if player.hp <= cost then
            return false  -- 拒绝释放，不进 CD
        end

        -- 2. HP 消耗
        player.hp = player.hp - cost

        -- 3. 累计消耗记录（通用字段，供 accumulate_trigger 类型技能读取）
        if skill.feedsAccumulator ~= false then
            player._bloodAccumulator = (player._bloodAccumulator or 0) + cost
        end

        -- 4. 伤害计算（HP 系数管线）
        local raw = maxHp * (skill.hpCoefficient or 0)
        if skill.skillDmgApplies then
            raw = raw * (1 + player:GetSkillDmgPercent())
        end
        -- → 进入标准管线: CalcDamage(raw, DEF) → ApplyCrit → TakeDamage

        -- 5. 范围筛选 + 施加伤害（复用 CastDamageSkill 的 AoE 逻辑）
        -- 实际调用: SkillSystem.ApplySkillDamage(player, raw, skill)

        -- 6. EventBus 广播（供血怒等机制监听）
        -- EventBus.Emit("skill_cast", { player=player, skillId=skill.id })

        return true
    end,
})
```

**数据驱动字段**:

| 字段 | 类型 | 含义 | 默认值 |
|------|------|------|--------|
| `hpCost` | float | HP 消耗比例（占 maxHP） | 0 |
| `hpCoefficient` | float | 伤害系数（占 maxHP） | 0 |
| `skillDmgApplies` | bool | 是否吃 skillDmg% | false |
| `feedsAccumulator` | bool | 消耗是否计入 `_bloodAccumulator` | true |

> **通用性**: 如果未来有其他职业/法宝技能需要"消耗 HP 造成伤害"，只需在 SkillData 中定义 `type = "hp_damage"` 并配置系数即可，无需改 TypeHandler 代码。

### 6.3 `hp_heal_damage` TypeHandler（通用 HP 系数回复+伤害）

**适用范围**: 任何"同时回复自身 HP 并造成 HP 系数伤害"的技能。

```lua
SkillSystem.RegisterType("hp_heal_damage", {

        end
        -- → CalcDamage(raw, DEF) → ApplyCrit → TakeDamage

        -- 3. 范围筛选 + 施加伤害

        return true
    end,
})
```

**数据驱动字段**:

| 字段 | 类型 | 含义 | 默认值 |
|------|------|------|--------|
| `healPercent` | float | 回复比例（占 maxHP） | 0 |
| `hpCoefficient` | float | 伤害系数（占 maxHP） | 0 |

### 6.4 `toggle_passive` TypeHandler（通用可切换被动）

**适用范围**: 任何"有激活/关闭两种状态、点击切换"的被动技能。

> **设计原则**: toggle 是一个通用概念，不特化于焚血。TypeHandler 读取 skill 表中的 `onActive` / `onInactive` 配置或回调来决定激活和关闭时的行为。实际激活/关闭的状态切换通过 `SkillSystem.skillState[skillId].active` 管理。

```lua
SkillSystem.RegisterType("toggle_passive", {
    -- 不走 AutoCast（不占技能槽）
    passive = function(skill, player, skillId, dt)
        local state = SkillSystem.GetSkillState(skillId, { active = skill.defaultActive or false })
        if not state.active then return end  -- 关闭状态：不执行任何逻辑

        -- === 通用 tick 框架 ===
        -- 由 skill 表的 onTick 回调定义具体行为
        if skill.onTick then
            skill.onTick(skill, player, skillId, dt, state)
        end
    end,

    -- toggle 接口：切换 active 状态
    toggle = function(skill, player, skillId)
        local state = SkillSystem.GetSkillState(skillId, { active = skill.defaultActive or false })
        state.active = not state.active

        -- 通知 UI 更新
        -- EventBus.Emit("passive_toggle", { skillId=skillId, active=state.active })

        -- 播放开/关音效与特效
        if state.active and skill.onActivate then
            skill.onActivate(skill, player)
        elseif not state.active and skill.onDeactivate then
            skill.onDeactivate(skill, player)
        end
    end,
})
```

### 6.5 `accumulate_trigger` TypeHandler（通用累计触发）

**适用范围**: 任何"某个数值累计到阈值后自动触发效果"的被动技能。

```lua
SkillSystem.RegisterType("accumulate_trigger", {
    passive = function(skill, player, skillId, dt)
        -- 读取累计源（通用字段名，由 skill 表指定）
        local accumField = skill.accumulatorField or "_bloodAccumulator"
        local accumulated = player[accumField] or 0
        local threshold = player:GetMaxHp() * (skill.threshold or 1.0)

        if accumulated < threshold then return end

        -- === 触发 ===
        -- 1. 消耗累计值
        player[accumField] = accumulated - threshold  -- 溢出保留 or 归零

        -- 2. 计算伤害（由 skill 数据字段决定公式）
        local baseDmg = player:GetMaxHp() * (skill.damagePercent or 0)
        if skill.useBloodRageValue then
            baseDmg = baseDmg * (1 + player:GetPhysiqueBloodRageValue())
        end
        local finalDmg = player:ApplyCrit(baseDmg)
        -- 真实伤害 → TakeDamage(trueDamage=true)

        -- 3. AoE 施加
        -- SkillSystem.ApplyTrueDamageAoE(player, finalDmg, skill.aoeRange, skill.maxTargets)

        -- 4. 特效+音效
        -- 血爆爆炸特效 + 屏幕闪红
    end,
})
```

**数据驱动字段**:

| 字段 | 类型 | 含义 | 默认值 |
|------|------|------|--------|
| `accumulatorField` | string | 读取玩家身上哪个字段的累计值 | `"_bloodAccumulator"` |
| `threshold` | float | 触发阈值（× maxHP） | 1.0 |
| `damagePercent` | float | 伤害比例（× maxHP） | 0 |
| `useBloodRageValue` | bool | 是否乘以血怒值 | false |
| `trueDamage` | bool | 是否真实伤害 | false |
| `aoeRange` | float | AoE 范围（格） | 2 |
| `maxTargets` | int | 最大目标数 | 5 |

### 6.6 属性-技能交互机制总览

以下表格描述镇岳技能如何与属性系统交互（参见 `战斗系统架构.md` §5 属性衍生链）：

| 属性 | 来源 | 影响的技能 | 交互方式 |
|------|------|----------|---------|
| **体魄 (physique)** | 基础 + 职业(+1/级) + 装备 + 洗髓 | 全部（通过 maxHP） | `GetPhysiqueHpBonus()` → maxHP 增加 → 所有 HP 系数伤害/消耗/回复等比增加 |
| **血怒概率** | 体魄衍生 + 职业(+10%) | 血爆（间接） | 血怒触发频率影响战斗体验，不影响血爆本身伤害公式 |
| **血怒值** | 体魄衍生 | 血爆 | `GetPhysiqueBloodRageValue()` 直接乘入血爆伤害 |
| **skillDmg%** | 法宝/装备/buff | 裂山拳、地脉回涌 | 技能数据 `skillDmgApplies=true` 时乘入 raw 伤害 |
| **暴击** | 现有系统 | 全部 | 所有技能最终都经过 `ApplyCrit` |

> **通用原则**: 属性→技能的交互通过**数据字段**（如 `skillDmgApplies`、`useBloodRageValue`）控制，不在 TypeHandler 中硬编码"如果是镇岳职业则..."的判断。TypeHandler 只读取 skill 数据字段。

### 6.7 与 EventBus 的集成

镇岳技能需要触发的 EventBus 事件（参见 `战斗系统架构.md` §8）：

| 事件 | 触发时机 | 触发者 | 监听者 |
|------|---------|--------|--------|
| `skill_cast` | 裂山拳/地脉回涌释放成功 | `hp_damage` / `hp_heal_damage` TypeHandler | 血怒系统（震山掌类的保证叠层通过此事件） |
| `player_deal_damage` | 技能命中敌人 | 伤害管线末端 | 血怒概率触发 |
| `passive_toggle` | 焚血开关切换 | `toggle_passive.toggle()` | BottomBar UI 更新图标状态 |

> **不新增事件**: 镇岳使用现有事件即可。`passive_toggle` 是唯一新增事件，且是 UI 层面的通用事件（任何 toggle 被动都会触发）。

### 7.1 方案：复用现有被动图标位 + 点击切换

**不新增任何 UI 元素**。

## 九、职业隔离架构

### 9.1 隔离原则

镇岳的实现必须与罗汉、太虚**完全隔离**，不产生交叉依赖。具体要求：

| 规则 | 说明 | 检查方法 |
|------|------|---------|
| **数据隔离** | 镇岳的技能定义（SkillData）、职业数据（CLASS_DATA）、解锁序列（UnlockOrder）都通过 `classId = "zhenyue"` 标识 | `grep "zhenyue"` 只出现在数据定义和测试中 |

| **测试隔离** | 镇岳自测脚本独立存在，不修改其他职业的测试 | 独立 `test_zhenyue.lua` |

## 十一、实施总结与建议开发顺序

### 11.1 改动规模对比

| 维度 | 太虚（第三步） | 镇岳 | 差异原因 |
|------|--------------|------|---------|
| CLASS_DATA | 1 条目 | 1 条目 | 相同 |
| CLASS_SKILL_PREVIEW | 1 条目 | 1 条目 | 相同 |
| 技能定义 | 4 个 | 4 个 | 相同 |
| 新增 TypeHandler | 2 个（已有类型可复用） | **4 个**（全部新增🆕） | 镇岳每个技能都是全新机制 |
| Player 新增字段 | 0 | **1 个**（`_bloodAccumulator`） | 镇岳需要持久累计状态 |
| SkillState 新增 | 0 | **1 个**（`blood_burn.active`） | toggle 状态走通用 skillState |
| 存档新增字段 | 0 | **1 个**（`bloodAccumulator`） | 配合 Player 字段 |
| UI 改动 | 0 | **1 处**（被动图标 toggle 分支） | 复用现有图标位，不新增元素 |
| 预计代码行数 | ~250 行 | **~400 行** | 镇岳机制更复杂 |
| 自测用例 | — | **~25 项** | 全链路覆盖 |

### 11.2 风险评估

| 风险项 | 级别 | 缓解措施 |
|-------|------|---------|
| 4 个新 TypeHandler 同时开发 | 🔴 高 | 按复杂度递增开发（见下方顺序） |
| `_bloodAccumulator` 多来源写入 | 🟡 中 | 单线程无并发问题；自测覆盖多来源写入场景 |
| 焚血每帧 tick 精度 | 🟡 中 | 使用 dt 累加器，每满 1s 触发整数伤害 |
| HP 消耗扣血与受伤扣血视觉混淆 | 🟢 低 | 消耗扣血用不同颜色（橙色）区分 |
| 存档新字段兼容 | 🟢 低 | 默认值兜底，无需版本迁移 |

### 11.3 建议开发顺序

```
阶段 1：数据层（零风险，可立即开始）
    ├─ 第四节: CLASS_DATA + Player 属性
    ├─ 第五节: SkillData 定义
    └─ 第二节: CharacterSelectScreen 数据

阶段 2：核心机制（按复杂度递增）
    ├─ C4 accumulate_trigger（最简单，只检查阈值 + AoE 真伤）
    │   └→ 验证 _bloodAccumulator 机制
    ├─ C1 hp_damage（中等，HP 消耗 + CalcDamage）
    │   └→ 验证 HP 系数伤害管线
    ├─ C2 hp_heal_damage（中等，回复 + 伤害双效）
    │   └→ 验证回复逻辑
    └─ C3 toggle_passive（最复杂，每帧 tick + 开关 + AoE 真伤）
        └→ 整合所有机制

阶段 3：UI 与特效
    ├─ 第七节: BottomBar toggle 分支
    └─ 第三节: 特效与音效

阶段 4：验收
    ├─ 第十节: 自测脚本
    ├─ 第九节: 隔离性检查
    └─ 全链路测试
```

> **核心工作量在阶段 2 的 4 个 TypeHandler**。每完成一个 TypeHandler 都应立即写对应测试用例，不要等全部完成再测。

#### 12.3.1c 公式说明

| 维度 | 公式 | 伤害类型 |
|------|------|---------|
| 普攻 | CalcDmg(ATK, 怪DEF) × 暴击 × 攻速 | 常规 |
| 重击 | (ATK+重击值) × (1+根骨加伤) × 暴击 × 重击率 × 攻速 | 真伤 |
| 金刚掌 | CalcDmg(ATK×1.5, 怪DEF) × 暴击 / CD5s | 常规 |
| 伏魔刀 | CalcDmg(ATK, 怪DEF) × 暴击 / CD8s | 常规 |
| 龙象功 | 同重击公式 / 充能周期(18层) | 真伤AoE |
| 连击 | 连击率 × (金刚掌DPS + 伏魔刀DPS) | 同技能 |
| 裂山 | CalcDmg(HP×10%, 怪DEF) × 暴击 / CD3s，自伤5%HP | 常规 |
| 地涌 | CalcDmg(HP×30%, 怪DEF) × 暴击 / CD15s，回血30%HP | 常规 |
| 焚血 | 总回血/s × burnRatio(1.0)，不暴击不过CalcDmg | 真伤DOT |
| 血怒 | HP×10% × 暴击 / 叠层间隔。叠层: 普攻+重击+裂山(概率) + 裂山(强制+1) + 地涌(概率)。焚血isDot不触发 | 真伤 |
| 血爆 | HP × 系数 × (1+血怒值) × 暴击 / 触发间隔。累积: 焚血自伤 + 裂山自伤(5%HP/3s) | 真伤 |

**结论**: 镇岳(0.10)在T1-T3弱于罗汉(81%→96%)，T4反超(105%)。镇岳(1.00)从T2开始即超越罗汉(111%)，T4达到140%。镇岳属于**"前弱后强"**的成长型职业，T4档的DPS成长倍率(2.46x/2.71x)显著高于罗汉(1.90x)。

#### 12.3.4 成长曲线分析

```
和尚/镇岳 DPS 比率
  120% ┤         ×(T1高阶)
       │        /
  115% ┤    ×(Lv30)
       │   /     \

       │ /         \

       │              \

       │                  \

       ├──┬──┬──┬──┬──┬──┬──┤
       Lv10 Lv20 Lv30 T1  T2  T3  T4
                  └─断层区─┘
```

**三阶段成长模型**:

| 阶段 | 范围 | 比率 | 原因 |
|------|------|------|------|
| **低阶持平** | Lv10~Lv20 | 100%~106% | 焚血真伤弥补暴击/重击缺失 |
| **中阶断层** | Lv30~仙缘300 | 110%~120% | 和尚暴击×重击乘法加速；镇岳 maxHP 不够大，HP 比例伤害绝对值低 |
| **高阶反超** | 仙缘600+ | 100%→92% | maxHP 突破 10 万后，焚血+血爆 HP 比例伤害进入指数增长区 |

#### 12.3.5 平衡性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 高阶平衡 | ✅ 良好 | T3 交叉，T4 反超 8%，双向浮动合理 |
| 低阶体验 | ✅ 可接受 | Lv10 持平，Lv20~30 弱 6~15%，但焚血提供独特的"持续灼烧"体验感 |
| 中阶断层 | ⚠️ 需关注 | 仙缘 100~300 区间和尚领先 10~20%，是镇岳最弱的阶段 |
| DPS 来源多样性 | ✅ 优秀 | 镇岳有 6 个 DPS 来源（普攻+裂山+地涌+焚血+血怒+血爆），操作感丰富 |
| 真实伤害占比 | ✅ 合理 | 高阶 T4 镇岳真伤（焚血+血爆+血怒）占比约 41%，面对高 DEF Boss 有结构性优势 |
| 生存能力 | ✅ 优势 | 焚血净回复=0（自损=回复），但地涌回复 30% maxHP(CD15s) + 体魄 HP 加成提供高续航 |
| 成长曲线 | ✅ 合理 | "持平→弱势→追平→反超"的四段式，有明确的成长期待感 |

**中阶断层缓解方案**（可选，不急于实施）:
1. 调整焚血 `regenPercent` 从 2% 提升到 3%（直接增加焚血 DPS）
2. 或调整裂山拳 `hpCoefficient` 从 10% 提升到 12%（回归设计文档值）
3. 或降低血爆 `threshold` 从 1.0 到 0.8（加快触发频率）

> **建议**: 先按当前数值上线观察实战数据，中阶断层（10~20%）在实际游戏中因镇岳的**生存优势**（高 HP + 地涌回血）和 **AoE 持续输出**（焚血范围灼烧）会被部分抵消。纯 DPS 差距不等于综合战斗效率差距。

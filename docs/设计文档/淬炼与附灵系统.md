## 二、Tab1：萃取

### 2.1 规则

- **输入**：同一套装的 **3 件**装备（T9 原装套装件 **或** 已附灵装备均可） + **100 灵韵**
- **输出**：该套装对应的 **1阶附灵玉**（确定性，非随机）

| 输入（3件同套装） | 输出 |
|----------------|------|
| 3× 血煞装备 | 1阶附灵玉·血煞（`lingyu_xuesha_lv1`）|
| 3× 青云装备 | 1阶附灵玉·青云（`lingyu_qingyun_lv1`）|
| 3× 封魔装备 | 1阶附灵玉·封魔（`lingyu_fengmo_lv1`）|
| 3× 浩气装备 | 1阶附灵玉·浩气（`lingyu_haoqi_lv1`）|

**可萃取装备判定**（`GetItemExtractSetId(item)`）：
- 有 `setId` 且在四大套装中 → 取 `setId`
- 有 `enchantSetId` 且在四大套装中 → 取 `enchantSetId`
- 两者都不符合 → 不可萃取

**前置检查（任一不满足则禁止）**：
- 三件装备 `setId`（或 `enchantSetId`）**完全相同**
- 玩家灵韵 ≥ 100

提交的 3 件装备**永久销毁**（**仅限背包中的装备**，装备栏中已穿戴的装备不出现在列表中）。

### 2.2 执行逻辑（TemperUI.lua `DoExtract()`）

```lua
-- 1. 验证：3件同套装 + 灵韵 >= 100
-- 2. 先移除选中的3件背包装备（释放槽位，同时备份用于回滚）
-- 3. InventorySystem.AddConsumable(outputId, 1)  -- 此时至少有3个空槽，必定成功
--    如果极端异常失败 → 回滚已删除的装备，返回错误
-- 4. player.lingYun -= 100（放在 AddConsumable 成功之后，确保不会白扣）
-- 5. EventBus.Emit("player_lingyun_change", player.lingYun)
-- 6. EventBus.Emit("temper_extract", sid, outputId)
-- 7. EventBus.Emit("save_request")  -- 即时存档
-- 8. 显示萃取成功弹窗 ShowObtainPopup(outputId)
```

**操作顺序设计**：先删除 3 件装备（释放背包空间），再添加附灵玉，最后扣灵韵。
这样确保：(a) 添加附灵玉时一定有空间；(b) 如果 AddConsumable 异常失败，可回滚装备且灵韵未扣。

**附灵玉占用背包格子**：附灵玉作为 consumable 存储在背包槽位中（与装备共享 40 格），
同类附灵玉可堆叠（最大 9999）。理论上萃取不会因背包满而失败（删 3 占 1），
但代码中仍有防御性回滚逻辑。

## 三、Tab2：附灵

### 3.1 规则

**目标装备**：
- **装备栏中所有已穿戴装备**均可选择（包括有 setId 的、有 enchantSetId 的、无套装的）
- **已有 `enchantSetId` 的装备可再次附灵**（覆盖旧值）
- 背包中的装备不可附灵

**费用**：1× 对应附灵玉 + **100 灵韵**

**结果**（确定性）：
- 使用 1阶附灵玉·血煞 → `item.enchantSetId = "xuesha"`
- 使用 1阶附灵玉·青云 → `item.enchantSetId = "qingyun"`
- 使用 1阶附灵玉·封魔 → `item.enchantSetId = "fengmo"`
- 使用 1阶附灵玉·浩气 → `item.enchantSetId = "haoqi"`

**附灵后装备改名**：名称更新为 `"套装名·基础名"` 格式（如 "青云·龙骨坠"），使用 `StripSetPrefix()` 去除旧前缀再拼接新前缀。

**附灵后立即存档**（`EventBus.Emit("save_request")`）。

### 3.2 套装计件效果

`enchantSetId` 让装备在计算套装效果时**额外贡献一个套装件数**：

```
item.setId        = "xuesha"   → 血煞套装 +1件（原装）
item.enchantSetId = "qingyun"  → 青云套装 +1件（附灵贡献）
```

**计件规则**（`InventorySystem.RecalcEquipStats()` / `GetActiveSetBonuses()`）：
- 原装 `setId`：按 `slot` 去重（同 slot 只计1件），通过 `setSlotSeen` 防重
- 附灵 `enchantSetId`：每件装备直接 +1，不去重

例：玩家装备 5 件青云原装套装 + 1 件附灵了"青云"的非套装防具 → 视为 6 件青云。

### 3.3 执行逻辑（TemperUI.lua `DoEnchant()`）

```lua
-- 1. 验证：selectedEquipSlot_ + selectedLingyuId_ + 灵韵 >= 100
-- 2. 验证：附灵玉数量 >= 1
-- 3. InventorySystem.ConsumeConsumable(selectedLingyuId_, 1)  -- 先消耗附灵玉
-- 4. player.lingYun -= 100
-- 5. EventBus.Emit("player_lingyun_change", player.lingYun)
-- 6. item.enchantSetId = setId
-- 7. item.name = "套装名·基础名"  -- 更新装备名称
-- 8. InventorySystem.RecalcEquipStats()  -- 重算套装加成
-- 9. EventBus.Emit("temper_enchant", selectedEquipSlot_, setId, item)
-- 10. EventBus.Emit("save_request")  -- 即时存档
-- 11. 显示附灵成功弹窗 ShowEnchantPopup(enchantedItem)
```

**执行顺序设计**：先消耗附灵玉，成功后再扣灵韵。避免附灵玉消耗失败但灵韵已扣的问题。

### 7.1 InventorySystem 套装计件逻辑

```lua
-- RecalcEquipStats() 和 GetActiveSetBonuses() 中：
for slotId, item in pairs(equipped) do
    if item then
        -- 原装 setId：按 slot 去重
        if item.setId then
            if not setSlotSeen[item.setId] then setSlotSeen[item.setId] = {} end
            if not setSlotSeen[item.setId][origSlot] then
                setSlotSeen[item.setId][origSlot] = true
                setCounts[item.setId] = (setCounts[item.setId] or 0) + 1
            end
        end
        -- 附灵 enchantSetId：直接 +1，不去重
        if item.enchantSetId then
            setCounts[item.enchantSetId] = (setCounts[item.enchantSetId] or 0) + 1
        end
    end
end
```

## 九、已知差异与设计演进

| 项目 | 原设计文档 v2.0 | 实际代码 v3.0 | 原因 |

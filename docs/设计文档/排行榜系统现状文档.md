# 排行榜系统现状文档

> 状态日期：2026-07-17  
> 适用项目：人狗仙途 TapTap Maker 多人模式  
> 当前主榜顺序：修仙榜 → 仙人榜 → 试炼榜 → 镇狱榜

## 1. 文档范围

本文档基于排行榜完整代码链路整理，覆盖：

- 客户端排行榜面板与四个主榜页签。
- 客户端到服务端的排行榜查询协议。
- 服务端权威排行计算、存档更新与登录重建。
- 修仙榜、仙人榜、试炼榜、镇狱榜、仙石榜和活动榜的数据结构。
- 谪仙角色从修仙榜迁移到仙人榜的规则。
- 仙人榜角色 UID、当前仙体和两行 UI 排版。

## 2. 总体架构

项目启用了多人模式，排行榜由服务端权威写入。

### 2.1 查询链路

```text
排行榜 UI
  → CloudStorage.GetRankList
  → C2S_GetRankList
  → RankHandler.HandleGetRankList
  → serverCloud:GetRankList
  → 按 userId 补查附加字段
  → S2C_RankListData
  → UI 合并四个角色槽位并渲染
```

`serverCloud:GetRankList` 只返回排行键本身。角色名、境界、仙体等附加信息由服务端按榜单返回的 `userId` 再执行单用户 `BatchGet` 补查。

### 2.2 写入链路

```text
客户端保存角色
  → C2S_SaveGame
  → SaveWriteService.Execute
  → 主存档 BatchSet.Save 成功
  → RankHandler.ComputeRankFromSaveData
  → RankHandler.UpsertCultivationRank
  → 按境界分流到修仙榜或仙人榜
  → 单个 BatchCommit 原子删除另一榜并写入目标榜
  → RankHandler.UpdateTowerRanksFromSave
  → 试炼榜、镇狱榜仅在当前成绩更高时更新
```

排行榜旁路只在主存档成功后执行，避免存档失败但榜单提前变化。修仙榜与仙人榜的一次更新包含“删除另一榜四个键”和“写入目标榜四个键”，由同一个 `BatchCommit` 提交，不会留下角色同时存在于两个修炼榜的中间状态。

### 2.3 登录重建链路

```text
读取角色槽位
  → SlotReadService.Execute
  → GetUserNickname
  → RankHandler.RebuildRankOnLogin
  → 遍历 1~4 号角色槽位
  → 重建修仙榜或仙人榜
  → 同时修复试炼榜、镇狱榜和仙石榜
```

登录重建承担历史数据修复职责。仙阶角色第一次登录新版后，即使暂时没有主动保存，也会进入仙人榜并从修仙榜移除。

登录重建还会：

- 清理没有角色数据的空槽位在修仙榜和仙人榜中的残留记录。
- 遍历全部 1~4 号槽位，重新计算账号真实最高试炼层和镇狱层。
- 允许账号级塔榜在全量重建时降低或删除，以修复角色删除、异常旧数据或历史高分残留。

## 3. 通用排序规则

修仙榜和仙人榜使用相同的原始分数：

```text
rawScore = realmOrder × 1000 + level
```

最终云排行分数：

```text
composite = rawScore × 10^10 + (10^10 - 1 - achieveTime)
```

服务端按整数降序排列，因此：

1. 境界顺序高的角色优先。
2. 境界相同时等级高的角色优先。
3. 境界和等级都相同时，先达成者优先。

`achieveTime` 完全由服务端维护：

- 新的原始分数使用服务端 `os.time()`。
- 原始分数相同，保留云端原有首次达成时间。
- 新原始分数低于云端已有分数时拒绝覆盖。
- 客户端传入的 `rankData.achieveTime` 不参与排行榜计算。

因此客户端不能通过伪造早期时间抢占同分排名。

## 4. 修仙榜

### 4.1 入榜规则

- 角色等级必须大于 5。
- 境界必须存在于 `GameConfig.REALMS`。
- 境界顺序必须小于 23，即最高到大乘巅峰。
- 每个角色槽位独立入榜，一个账号最多四条记录。
- 仙阶角色不会继续写入修仙榜。

### 4.2 云键

| 类型 | 键 |
|---|---|
| 排行分数 | `rank2_score_{slot}` |
| BOSS 击杀 | `rank2_kills_{slot}` |
| 达成时间 | `rank2_time_{slot}` |
| 附加信息 | `rank2_info_{slot}` |

`rank2_info_{slot}` 字段：

```lua
{
    name = "角色名",
    realm = "境界ID",
    level = 120,
    bossKills = 100,
    taptapNick = "TapTap昵称",
    classId = "monk",
}
```

### 4.3 客户端展示

- 合并四个槽位各自的前 50 条结果。
- 合并后按复合分数再次降序排序。
- 最终展示前 50 名。
- 支持全部、罗汉、太虚、镇岳职业子页签。
- 展示排名、职业图标、角色名、TapTap 昵称、境界、等级和 BOSS 击杀。

## 5. 仙人榜

### 5.1 入榜规则

- 从谪仙开始进入仙人榜。
- 判定口径为 `realmOrder >= 23`。
- 同时覆盖旧兼容境界 `dujie_1` 和新仙阶境界 `asc_1`、`zhexian_1` 及其后续仙阶。
- 不设置罗汉、太虚、镇岳职业子榜。
- 每个角色槽位独立入榜。
- 排序规则与修仙榜一致。

### 5.2 从修仙榜迁移

当角色达到谪仙或更高仙阶时，存档更新或登录重建会：

1. 在一个原子 `BatchCommit` 中删除该槽位的 `rank2_score`、`rank2_kills`、`rank2_time`、`rank2_info`。
2. 在同一提交中写入同槽位的仙人榜分数、击杀、时间和附加信息。
3. 后续只更新仙人榜，不再写修仙榜。

反向分流也使用相同规则：修仙角色写入修仙榜时会同步删除该槽位的仙人榜键，用于修复旧数据或异常残留。服务端会跨修仙榜、仙人榜比较该槽位的最高历史原始分数；角色存档发生境界或等级回退时，保留更高成绩并清理较低榜位，不允许仙人被异常降回修仙榜。

### 5.3 云键

| 类型 | 键 |
|---|---|
| 排行分数 | `xianren_score_{slot}` |
| BOSS 击杀 | `xianren_kills_{slot}` |
| 达成时间 | `xianren_time_{slot}` |
| 附加信息 | `xianren_info_{slot}` |

`xianren_info_{slot}` 字段：

```lua
{
    name = "角色名",
    realm = "asc_1",
    level = 121,
    bossKills = 100,
    taptapNick = "TapTap昵称",
    classId = "taixu",
    roleUid = "123456789:2",
    currentBodyId = "immortal_body_qinglian",
    currentBodyName = "青莲长生体",
}
```

### 5.4 角色 UID 定义

当前存档没有独立、永久的角色 GUID。平台云排行返回的是账号 `userId`，而同一账号最多有四个角色槽位。

仙人榜使用以下稳定组合标识作为“角色 UID”：

```text
角色UID = TapTap userId : 角色槽位
```

示例：`123456789:2` 表示账号 `123456789` 的 2 号角色。

如果未来引入真正的角色 GUID，应新增持久化字段，并在兼容期优先显示新 GUID、回退到当前组合 UID。

### 5.5 当前仙体来源

当前仙体候选值读取自角色最新存档：

```text
coreData.immortalBody.activeBodyId
```

服务端写榜时还会读取账号级权威数据：

```text
account_immortal_bodies.unlockedBodies
```

只有同时满足以下条件，候选仙体才会进入榜单快照：

- 仙体 ID 存在于 `AscensionConfig.GROWTH_PROFILES`。
- 仙体已存在于服务端维护的账号解锁列表。

未知、未解锁、空值仙体统一回退为 `mortal`，名称回退为“凡人之躯”。客户端提交的 `accountImmortalBodies` 快照不会被保存，仙体解锁由服务端奖励流程维护。

榜单在以下时机更新仙体快照：

- 角色主存档成功并执行排行榜旁路时。
- 账号登录并执行排行榜重建时。

因此仙人榜展示的是该角色最近一次榜单更新时、并通过服务端解锁校验的当前仙体。

### 5.6 两行条目排版

仙人榜每条记录分为两行：

```text
第一行：排名 | 职业图标 + 角色名 / TapTap昵称 | 境界 / 等级 / BOSS击杀
第二行：角色UID | 当前仙体
```

职业只作为角色信息展示，不用于拆分子榜或筛选排行。

## 6. 试炼榜

### 6.1 数据模型

试炼榜按账号记录所有角色中的最高试炼层数，不是每槽位独立榜。

| 类型 | 键 |
|---|---|
| 排行分数 | `trial_floor` |
| 附加信息 | `trial_info` |

### 6.2 规则

- 遍历账号 1~4 号角色。
- 选取 `trialTower.highestFloor` 最高的角色。
- 普通角色存档时只允许提高账号记录，不允许较低槽位覆盖更高成绩。
- 登录重建或删除角色后会根据所有剩余槽位重算，允许降低或清除失效记录。
- 层数必须在 1~300，并满足该层要求的境界和等级。
- 同层数时使用时间倒序复合分数。
- 展示角色名、TapTap 昵称、职业、境界、等级和试炼层数。

## 7. 镇狱榜

镇狱榜同样是账号级最高记录。

| 类型 | 键 |
|---|---|
| 排行分数 | `prison_floor` |
| 附加信息 | `prison_info` |

服务端存档更新和登录重建均包含反作弊校验：

- 最高层不得超过 150。
- 当前角色必须满足目标层对应的境界顺序和等级要求。
- 普通存档只允许提高账号记录；登录重建或删除角色后允许按剩余槽位降低或清除。

## 8. 仙石榜

仙石榜由 `XianshiRankUI` 独立展示，不在主排行榜四页签中。

| 类型 | 键 |
|---|---|
| 排行分数 | `xianshi_rank` |
| 附加信息 | `xianshi_info` |

当前规则为账号仙石大于 0 时入榜。一旦入榜不会因余额归零自动删除，主要用于长期记录和异常监控。

## 9. 其他排行能力

### 9.1 活动榜

节日活动通过 `EventHandler` 和活动专用排行键实现，与主排行榜分离。活动榜通常包含积分键、附加信息键和全服记录列表。

### 9.2 竞速道印

`RankHandler` 还维护：

- `pioneer_huashen`
- `pioneer_heti`

竞速道印采用先写入占位、再查询真实名次、超额则回滚删除的乐观占位方式。

## 10. 黑名单与处罚

- 服务端查询统一经过 `AdminPenaltyControl.FilterRankList`。
- `suppressRank` 用户不执行存档排行写入和登录重建。
- 客户端继续通过 `config.Blacklist` 过滤指定账号和角色名。
- 仙石榜保留部分处罚账号用于监控，其余主榜正常屏蔽。
- 一次性处罚在一个 `BatchCommit` 中扣除仙石、清零修仙榜/仙人榜/塔榜整数键，并删除对应附加信息键。

## 11. 角色生命周期

### 11.1 创建角色

创建角色的主存档和槽位索引成功后，`CharacterService` 会清理该槽位在修仙榜、仙人榜中的历史残留，避免复用槽位继承已删除角色的排名。

### 11.2 删除角色

删除角色成功后会并行执行：

- 删除该槽位修仙榜和仙人榜的分数、击杀、时间、附加信息。
- 遍历剩余 1~4 号槽位，重建账号级试炼榜和镇狱榜。

即使排行清理或重建失败，角色删除结果仍会返回成功，同时服务端记录失败日志；下次登录重建会再次修复。

## 12. 当前限制

1. 修仙榜和仙人榜各发起四次槽位查询，最多先合并 200 条记录再取全局前 50；附加字段补查也会增加服务端云读取次数。
2. 仙人榜当前使用组合角色 UID，不是真正的服务端角色 GUID。
3. 历史仙阶角色需要至少登录一次或保存一次，才会完成修仙榜到仙人榜迁移；当前没有全服离线批量迁移任务。
4. TapTap 昵称属于账号信息，榜单附加字段保存的是最近一次写入时的昵称快照。
5. 排行榜旁路与主存档仍是两个先后独立的云提交；当前已保证“先存档、后写榜”，但排行榜提交失败时不会回滚已成功的主存档，需要依赖日志和下次登录重建修复。
6. 排行分数虽然由服务端计算并执行范围校验，但境界、等级和当前仙体候选值仍来自客户端提交的角色存档。要实现完全服务端权威的修炼进度，还需要把突破、升级和仙体切换改为服务端事件状态机。

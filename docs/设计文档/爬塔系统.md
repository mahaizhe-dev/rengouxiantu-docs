## 2. 青云试炼

### 2.1 当前代码实装

代码基线：

- `scripts/config/TrialTowerConfig.lua`
- `scripts/systems/TrialTowerSystem.lua`
- `scripts/ui/TrialTowerUI.lua`
- `scripts/network/TrialHandler.lua`
- `scripts/network/RankHandler.lua`

当前结论：

- `TrialTowerConfig.FLOOR_MONSTERS` 当前生成到 `300` 层。
- `TrialTowerConfig.MAX_DAILY_TIER = 30`，服务端 `TrialHandler.lua` 内联上限也为 `30`。
- 当前青云试炼增幅作用于所有试炼内生成怪物（普通怪 / 精英怪 / BOSS）：攻击 `atkMult=1.5`（`+50%`）、生命 `hpMult=2.0`（`+100%`）、狂暴节奏为攻速 `atkSpeedMult=1.3`、移速 `speedMult=1.3`、冷却 `cdrMult=0.7`；当前没有额外防御倍率。
- 进入试炼只校验境界，不额外校验等级。
- 试炼怪物生成时会覆盖等级与境界：优先使用 `FLOOR_MONSTERS[floor].realm`；未配置时才由 `GameConfig.GetMaxRealmByLevel(floorLevel)` 反推。
- 注意：`261-300` 层当前 `TrialTowerConfig.REALMS` 使用试炼专属粗分段（`zhexian_1 / zhexian_4 / zhexian_7 / zhexian_9`），其中 `reqLevel / maxLevel` 是塔内等级插值用，不等同于 `AscensionConfig.ASCENSION_REWARDS.requiredLevel` 的逐小阶限制等级；下方 BOSS 预设的仙阶反推以 `AscensionConfig` 为准。
- 小怪层沿用原逻辑：每个 `10` 层段前 `9` 层仍按 `8N / 6N+2E / 4N+4E` 自动铺怪，小怪和精英只借用原怪 ID / race / category / 技能，强度统一由 `floorCfg.level + floorCfg.realm` 覆盖，不套用 BOSS 的“原等级 +10~15”规则。
- 每层首通即时发奖励；每日供奉按“里程碑追领优先，追满后再领当日最高档”执行。

当前代码公式：

```lua
realm, realmIndex = TrialTowerConfig.GetRealmByFloor(floor)
span = realm.floors[2] - realm.floors[1]
progress = floor - realm.floors[1]

monsterLevel = realm.reqLevel
    + math.floor((realm.maxLevel - realm.reqLevel) * progress / math.max(1, span))

firstClearLingYun = math.ceil(floor / 2)
firstClearGoldBar = math.floor(floor / 2)

dailyTier = math.min(math.floor(highestFloor / 10), MAX_DAILY_TIER)
dailyReward = CalcFirstClearReward(dailyTier * 10)
```

### 2.2 BOSS 规则优化（已落地）

本节记录青云试炼 BOSS 层的专属数值规则。当前已在 `TrialTowerConfig.GetBossOverride()` 与 `TrialTowerSystem._spawnFloorMonsters()` 中落地；小怪层仍沿用 2.1 所述楼层段逻辑。

规则口径：

1. 只作用于每个 `10` 层段的 BOSS 层，即 `10 / 20 / ... / 300`。
2. BOSS 个体沿用原怪物 `category` 与技能/掉落配置。
3. BOSS 等级按“原 BOSS 等级 + `10-15` 级”形成候选区间，然后取该区间内最高的 `5` 的倍数。
   - 等价公式：`targetLevel = floor((原等级 + 15) / 5) * 5`，且应满足 `targetLevel >= 原等级 + 10`。
   - 例：`Lv.8 -> Lv.20`，`Lv.10 -> Lv.25`，`Lv.12 -> Lv.25`。
4. BOSS 境界不再直接上抬固定小境界，而是按“目标等级可达到的最大境界”反推；例如 `Lv.20` 可达到的最大境界为 `练气后期`，`Lv.25` 为 `筑基初期`，`Lv.30` 为 `筑基中期`。
5. 无显式 `realm` 的早期 BOSS，原 BOSS 设定列按原始等级可达最高境界补充说明；实际预设仍只由目标等级反推。
6. 仙阶按 `AscensionConfig.ASCENSION_REWARDS.requiredLevel` 的最新限制等级处理：目标等级达到某阶 `requiredLevel` 即反推为该阶；每个大仙阶覆盖 `20` 级，每个小阶间隔 `2` 级；`谪仙1阶` 限制等级为 `Lv.120`，`谪仙2-9阶` 依次为 `Lv.122 / 124 / 126 / 128 / 130 / 132 / 134 / 136`；`人仙1阶` 限制等级为 `Lv.140`，`人仙2-9阶` 依次为 `Lv.142 / 144 / 146 / 148 / 150 / 152 / 154 / 156`。

## 3. 镇狱塔

### 3.1 当前代码实装

代码基线：

- `scripts/config/PrisonTowerConfig.lua`
- `scripts/systems/PrisonTowerSystem.lua`
- `scripts/ui/PrisonTowerUI.lua`
- `scripts/network/RankHandler.lua`
- `scripts/network/SaveWriteService.lua`

当前结论：

- 当前实际开放 `150` 层。
- 当前 BOSS 池固定 `30` 个，对应 `30 * 5 = 150` 层。
- 当前不是“所有已解锁 BOSS 混池轮换”，而是“新 BOSS + 上一段 BOSS”交接模型。
- 当前每层 BOSS 数量不是 `1/2/3/4/5`，而是 `1/2/3/4/4`。
- 当前塔柱不是每层都有，只在 `5 / 10 / 15 ...` 这类里程碑层生成。
- 里程碑层的塔柱不会替换 BOSS 位，而是在 `4` 只 BOSS 之外额外添加。
- 非里程碑层如果当前层只有 `1` 只 BOSS，这只 BOSS 会站在中心点；里程碑层中心点留给塔柱。
- 镇狱塔内 BOSS 品质不看原怪物类别，而是按 `PrisonTowerConfig.GetEffectiveCategory()` 结算，不低于 `皇级`。

当前代码公式：

```lua
segmentIndex = math.ceil(floor / 10)
localFloor = ((floor - 1) % 10) + 1

monsterLevel = realm.reqLevel
    + math.floor((realm.maxLevel - realm.reqLevel) * (localFloor - 1) / 9)

bossCount = math.min(((floor - 1) % 5) + 1, 4)   -- 1/2/3/4/4
unlockCount = math.min(30, math.ceil(floor / 5))

if firstFiveFloors then
    activeBossIds = { currentBoss repeated bossCount times }
else
    activeBossIds = { currentBoss repeated bossCount times, previousBoss filled to 4 }
end
```

- 入场要求：`REQUIRED_REALM_ORDER = 7`、`REQUIRED_LEVEL = 40`

- 塔柱 AOE：`damagePct = 0.6`，按参考 BOSS 攻击计算
- 里程碑：每 `5` 层一次

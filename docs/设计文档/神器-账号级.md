> 状态：第三、四、五、六章神器已实装；中洲神器开放 6/9 格
> 代码审查日期：2026-07-11
> 原则：本文以当前代码为准。设计目标与代码不一致时，必须显式标记为“待实现”或“待确认”。

## 一、审查结论

当前神器系统不是单一模块，而是“章节运行时模块 + 账号级仙图录 + 角色存档镜像”的组合：

1. 每件神器有独立 `ArtifactSystem*.lua`，负责激活、属性、Boss 与被动。
2. `AtlasSystem` 将神器进度并入独立云变量 `account_atlas`，实现账号内角色共享。
3. 第三、四、五章神器同时保留在角色槽位存档中，作为兼容镜像与恢复来源。
4. 读档时先恢复角色槽位镜像，再以 `account_atlas` 做只增不减的并集合并。
5. 神器属性不直接存最终面板值，读档后由激活格重新计算到 `Player` 派生字段。

因此，“神器是账号级”描述的是最终共享语义，不等于“代码中只写账号档”。当前真实口径是：

```text
章节神器运行时状态
  ├─ 角色槽位镜像：artifact / artifact_ch4 / artifact_ch5
  └─ 账号权威副本：account_atlas.artifact*
          ↓
读档并集合并
          ↓
按激活格重算 Player 神器派生属性
```

## 五、账号级存档真相

### 5.1 账号档结构

`AtlasSystem.CreateEmpty()` 当前包含：

```lua
artifact = {
    activatedGrids = { false, false, false, false, false, false, false, false, false },
    bossDefeated = false,
    passiveUnlocked = false,
},
artifact_ch4 = {
    activatedGrids = { false, false, false, false, false, false, false, false },
    bossDefeated = false,
    passiveUnlocked = false,
},
artifact_tiandi = {
    activatedGrids = { false, false, false, false, false, false, false, false, false },
    bossDefeated = false,
    passiveUnlocked = false,
},
artifact_ch5 = {
    activatedGrids = { false, false, false, false, false, false, false, false, false },
    bossDefeated = false,
    passiveUnlocked = false,
},
```

### 5.2 写档路径

正常角色保存时：

1. `SavePersistence` 生成角色槽位中的 `artifact`、`artifact_ch4`、`artifact_ch5`。
2. `SaveSerializer.SerializeAtlas()` 调用 `AtlasSystem.Serialize()`。
3. `AtlasSystem.Serialize()` 将当前神器状态并入账号数据。
4. `SaveWriteService` 在同一服务端批处理中：
   - 写入 `save_{slot}`；
   - 将 `coreData.atlas` 写入独立 `account_atlas`。

第五章神器因此同时存在：

```text
save_{slot}.artifact_ch5
account_atlas.artifact_ch5
```

### 5.3 读档与合并

读档时：

1. 角色槽位中的 `artifact_ch5` 先由 `ArtifactSystem_ch5.Deserialize()` 恢复。
2. 服务端读取独立 `account_atlas` 并注入 `saveData.atlas`。
3. `AtlasSystem.Deserialize()` 恢复账号数据。
4. `AtlasSystem.PostLoadSync()` 将账号数据同步到当前角色。
5. 同步采用“只把 `true` 合并进去”的并集语义，不会因为某个旧角色档缺格而倒退账号进度。
6. `RecalcAttributes()` 根据激活格重算派生属性并失效属性缓存。

### 5.4 属性字段

| 神器 | Player 派生字段 |
|------|-----------------|
| 上宝逊金钯 | `artifactHeavyHit`、`artifactConstitution`、`artifactPhysique` |
| 文王八卦盘 | `artifactCh4Wisdom`、`artifactCh4Fortune`、`artifactCh4Defense` |
| 天帝剑痕 | `artifactTiandiAtk`、`artifactTiandiKillHeal`、`artifactTiandiHpRegen` |
| 诛仙阵图 | `artifactCh5Wisdom`、`artifactCh5Constitution`、`artifactCh5Physique` |

这些字段均已进入 `Player:GetTotal*()` 或击杀回血公式。

## 六、审查发现与技术欠账

### 6.1 高优先级

1. **Boss胜利没有立即强保存**
   - 第三、四、五章 Boss 胜利会设置 `bossDefeated/passiveUnlocked` 并发 UI 事件。
   - 当前没有在胜利分支调用 `SaveSystem.Save(callback)` 或发出 `save_request`。
   - 若胜利后在下一次自动保存前退出，解锁存在丢失窗口。

2. **激活仍是弱事务**
   - 激活先扣碎片、金币、灵韵，再修改格子，最后只发 `save_request`。
   - 保存失败没有回滚本次消耗和激活状态。
   - 第六章不应照搬，应采用天青白莲/两界阵石的“快照、强保存、失败回滚”模式。

3. **`AtlasSystem.SaveImmediate()` 需要专项验证**
   - 当前客户端分支调用 `clientCloud:SetScore(...)`。
   - 引擎示例的任意类型写入接口为 `clientCloud:Set(...)`。
   - 网络模式下 `CloudStorage.BatchSet(...):Save("atlas")` 又属于非主存档批次。
   - 在完成真实接口测试前，不应把该函数视为账号神器唯一可靠落盘路径；当前最确定的路径仍是正常 `SaveSystem` 服务端保存。

4. **服务端未按神器规则规范化账号档**
   - `SaveWriteService` 会直接写入客户端提交的 `coreData.atlas`。
   - 当前没有针对格数、布尔字段、Boss解锁关系、属性上限的神器专用服务端校验。
   - 第六章接入时应至少做结构规范化，并禁止 `passiveUnlocked=true` 但 `bossDefeated=false` 等非法组合。

### 6.2 中优先级

1. **第五章完整测试实际缺失**
   - `TestRegistry.lua` 有“第五章神器完整测试”注释，但后面没有对应测试项。
   - 现有测试只覆盖 SaveDTO 包含 `artifact_ch5`，没有覆盖激活消耗、账号并集、Boss胜利、四段伤害和冷却。

2. **账号并集只增不减**
   - 该策略能防止旧角色覆盖新进度。
   - 但普通 `Reset()` 无法清理账号档，GM重置或数据修复必须提供明确的账号级重置路径。

3. **中洲实现仍是半完成态**
   - 当前只有 6 格，Boss与被动均未接线。
   - 文件尾存在重复 `return ArtifactTiandi`，虽不影响首个 `return` 前的加载结果，但应清理。

4. **旧文档数值漂移**
   - 钉耙被动已从 50% 变为 100%。
   - 八卦护盾已从 10秒改为 30秒。
   - 第五章被动已从旧文档部分位置的 120秒改为 60秒。

## 八、第六章正式设计

### 8.1 基本信息

| 项目 | 定案 |
|------|------|
| 名称 | 界匙 |
| 定位 | 账号级生存神器、确定性防猝死 |
| 格数 | 9 |
| UI形式 | 3×3界匙钥阵 |
| 属性顺序 | 生命 → 生命回复 → 福源 |
| 被动 | 两界替命 |
| 当前状态 | 已实装并完成定向回归 |

### 8.2 激活消耗

| 资源 | 单格 | 9格总计 |
|------|-----:|---------:|
| 金币 | 3000万 | 2.7亿 |
| 灵韵 | 30000 | 270000 |
| 对应碎片 | 1 | 9 |

每格必须同时扣除金币、灵韵和对应编号碎片。材料不足、存档状态不安全、断线或正在保存时均不得先扣资源。

### 8.3 属性

沿用现有神器“9格基础累计 + 全激活额外一次”的成熟结构：

| 属性 | 每格 | 9格合计 | 全激活额外 | 最终满额 |
|------|-----:|---------:|-----------:|---------:|
| 生命 | +150 | +1350 | +150 | **+1500** |
| 生命回复 | +10/s | +90/s | +10/s | **+100/s** |
| 福源 | +10 | +90 | +10 | **+100** |

属性面板、神器面板、仙图录和角色详情中的展示顺序必须保持一致，不允许改为按字段名或数值动态排序。

### 8.4 被动：两界替命

| 项目 | 定案 |
|------|------|
| 解锁 | 9格全激活并击败神器Boss |
| 触发 | 本次最终伤害将使存活玩家死亡 |
| 结果 | 免疫该次致命结果，生命直接设置为最大生命的25%，最低为1 |
| 影界状态 | 3秒无敌 |
| 内置冷却 | 180秒 |
| 冷却口径 | 账号级绝对时间戳持久化，重登与切换角色不能刷新 |
| 技能定位 | 其他护盾与装备免死都失败后的最后兜底 |

推荐且固定的伤害结算顺序：

```text
伤害减免
→ 受伤前钩子
→ 普通护盾
→ 文王护体
→ 扣除生命
→ 装备 death_immunity
→ 两界替命
→ 正常死亡
```

边界规则：

1. 装备 `death_immunity` 成功时，不触发两界替命，也不消耗180秒冷却。
2. 伤害被闪避、护盾完全吸收或扣血后仍存活时，不触发。
3. 已死亡、复活流程中或明确标记为“不可免死”的剧情强制死亡，不触发。
4. 触发后最终生命是 `max(1, floor(maxHp × 0.25))`，不是“1点再加25%”。
5. 3秒无敌复用玩家统一无敌计时器，不另建与其他无敌互相覆盖的布尔状态。
6. 被动只处理玩家受伤结算，不得由属性重算、读档补血或UI预览触发。

### 8.5 神器与战斗表现

#### 激活面板

1. 整体为3×3残缺钥阵，九块组成一把纵向界匙和中央钥孔。
2. 未激活格使用暗灰断裂金属、闭合钥齿和低饱和影界纹理。
3. 持有材料且可激活的格子显示青白描边脉冲；只提示边框，不改变格子尺寸。
4. 激活成功后，对应钥齿由外向中央亮起，并生成一段连接中央钥孔的细线。

### 9.3 数据模型

账号权威结构：

```lua
account_atlas.artifact_jieshi = {
    version = 1,
    activatedGrids = {
        false, false, false,
        false, false, false,
        false, false, false,
    },
    bossDefeated = false,
    passiveUnlocked = false,
    rescueReadyAt = 0,
}
```

角色槽位兼容镜像：

```lua
artifact_ch6 = {
    activatedGrids = { boolean × 9 },
    bossDefeated = boolean,
    passiveUnlocked = boolean,
}
```

规则：

1. 账号结构保存进度与180秒冷却；角色镜像只保存永久进度，不保存冷却。
2. 属性最终值不写入任何神器档案，只保存格子，再由格子重算。
3. 新账号直接创建完整默认结构。
4. 旧账号第一次发现 `artifact_jieshi` 字段缺失时，才允许从当前角色镜像迁移。
5. 一旦账号字段存在，账号档覆盖角色镜像；不再每次用旧角色镜像做无条件并集，避免账号重置后被旧槽位重新“复活”。
6. 显式GM重置必须同时重置账号结构、当前角色镜像、派生属性和冷却，并立即强保存。

这是相对旧神器“永久并集”的确定性改进：兼容首次迁移，同时消除旧槽位覆盖账号重置的歧义。

### 9.4 激活强保存事务

复用 `LiangjieStoneSystem` 与 `QinglianBodySystem` 的成熟方式，流程固定为：

```text
预检查存档状态与材料
→ 获取全局操作锁
→ 建立回滚快照
→ 本地扣除金币、灵韵、碎片
→ 设置格子并重算属性
→ 同步账号内存结构
→ SaveSystem.Save(callback)
   ├─ 成功：释放锁、刷新UI、播放激活表现
   └─ 失败：恢复快照、重算属性、释放锁、显示失败原因
```

保存前检查：

1. 角色存档已加载完成。
2. 当前角色槽位有效。
3. `SaveState.saving` 为假。
4. 不存在 `_retryTimer`。
5. 客户端连接有效。
6. 当前没有另一个神器、阵石或青莲强事务占用全局锁。
7. 目标格未激活，材料与货币足够。

快照至少包含：

| 范围 | 内容 |
|------|------|
| 神器系统 | 9格、Boss状态、被动状态 |
| 账号内存 | `AtlasSystem.data.artifact_jieshi` 完整深拷贝 |
| 背包 | 完整背包序列化，确保对应碎片和锁定状态可恢复 |
| 玩家 | 金币、灵韵、当前生命、三项神器派生字段、属性缓存相关状态 |
| UI事务 | 当前操作格、保存中状态，不持久化但必须正确复位 |

回调必须有一次性完成保护，防止超时与迟到回包导致成功/失败逻辑执行两次。保存成功前不得发“激活成功”事件，不得播放完整激活动画，也不得开放Boss按钮。

### 9.5 Boss首胜强保存

Boss首胜是永久账号解锁，流程必须与格子激活同级：

1. 胜利结算先建立神器与账号快照。
2. 设置 `bossDefeated=true`、`passiveUnlocked=true`。
3. 重算并刷新被动卡，但维持“保存中”遮罩。
4. 调用 `SaveSystem.Save(callback)`。
5. 成功后才显示“神器觉醒”弹窗、播放完整觉醒动画并退出竞技场。
6. 失败时先恢复持久状态快照，同时保留一次仅驻留内存的“待确认首胜”凭据，让玩家可在当前会话内重新保存；若离开当前会话仍未成功，则凭据作废，下次需要重新挑战。
7. 重试保存成功后再重新应用Boss与被动状态并进入成功流程；不得只保存已经回滚的旧状态。
8. 不再依赖普通 `save_request` 等待下一次自动保存。

第三、四、五章Boss胜利分支应同步改造，消除现有退出即可能丢解锁的窗口。

### 9.6 账号级同步与迁移

加载顺序：

```text
读取角色槽位镜像
→ 读取独立 account_atlas
→ 规范化 account_atlas
→ 若 artifact_jieshi 缺失：从角色镜像迁移一次
→ 若已存在：账号档覆盖角色镜像
→ ArtifactCh6.Deserialize()
→ RecalcAttributes()
→ 刷新仙图录、角色面板和场景入口
```

切换角色时：

1. 新角色从同一 `account_atlas.artifact_jieshi` 获取进度。
2. 当前角色镜像被刷新为账号进度，供兼容和灾难恢复。
3. 三项派生属性先清零再按9格重算，不做增量累加。
4. `rescueReadyAt` 始终读取账号值，不能因换角色变为可用。
5. 账号数据未成功加载时，不允许以空默认结构覆盖云端。

`account_atlas` 建议从版本2升级为版本3，并提供显式迁移函数；迁移必须幂等，多次执行结果一致。

### 9.7 服务端规范化

`SaveWriteService` 在写入 `account_atlas` 前必须执行神器专用规范化：

1. 只接受固定键：`version/activatedGrids/bossDefeated/passiveUnlocked/rescueReadyAt`。
2. `activatedGrids` 固定截断或补齐为9项，每项严格转换为布尔。
3. 丢弃未知键、客户端提交的属性总值和运行时视觉状态。
4. 未全激活时强制 `bossDefeated=false`、`passiveUnlocked=false`。
5. `passiveUnlocked` 只能在 `bossDefeated=true` 时为真。
6. `rescueReadyAt` 必须是有限整数时间戳。
7. 服务端按当前时间把新冷却上限限制在 `now + 180` 附近，拒绝异常远期值。
8. 旧账号冷却尚未结束时，客户端不得提交更小时间戳提前清冷却；服务端应取合法旧值与新值的安全合并。
9. 缺失神器结构时写入完整默认结构，不保留半结构。
10. 服务端只保存格子和解锁关系，属性由客户端加载后按同一常量表重算。

第三、四、五章神器也应补固定格数和解锁关系规范化，形成统一安全边界。

### 9.8 属性接线

新增玩家派生字段：

```lua
player.artifactCh6MaxHp
player.artifactCh6HpRegen
player.artifactCh6Fortune
```

接入位置：

| 字段 | 权威总属性函数 |
|------|----------------|
| `artifactCh6MaxHp` | `Player:GetTotalMaxHp()` |
| `artifactCh6HpRegen` | `Player:GetTotalHpRegen()` |
| `artifactCh6Fortune` | `Player:_RecalcStatsCache()` 的福源汇总 |

重算公式：

```text
count = 已激活格数量
fullBonus = count == 9

生命 = count × 150 + (fullBonus and 150 or 0)
生命回复 = count × 10 + (fullBonus and 10 or 0)
福源 = count × 10 + (fullBonus and 10 or 0)
```

重算要求：

1. 每次从零计算后覆盖字段，不允许在旧值上累加。
2. 修改后统一调用 `InvalidateStatsCache()`。
3. 最大生命变化时按项目既有规则处理当前生命，不利用激活/读档重复补满。
4. 缺失或非法档案必须把三项字段清零，不能保留上一角色的神器属性。
5. UI从重算后的派生值读取，不自行复制另一套公式。

### 9.9 两界替命接线

在 `Player:TakeDamage()` 中，放在装备 `death_immunity` 失败之后、正常死亡之前：

```text
if hp <= 0
and passiveUnlocked
and now >= rescueReadyAt
and not bypassArtifactRescue
then
    hp = max(1, floor(GetTotalMaxHp() * 0.25))
    invincibleTimer = max(invincibleTimer, 2.0)
    rescueReadyAt = now + 180
    同步 account_atlas 内存值
    发出 artifact_ch6_rescue_triggered
    请求耐久保存
    return 本次实际造成的伤害
end
```

冷却保存与激活事务不同：

1. 替命已改变战斗结果，保存失败时不能把玩家重新判死。
2. 本地冷却必须继续生效，并进入有界重试队列。
3. 在冷却成功落盘前禁止通过切角色刷新状态。
4. 每次最多180秒触发一次，因此触发时立即保存的频率可接受。
5. 服务器需要用旧 `readyAt` 防止客户端缩短未结束冷却。

### 9.10 UI与表现接线

1. `ArtifactUI_ch6` 复用第五章3×3格子布局与确认弹窗结构。
2. 保存等待态和成功/失败弹窗复用青莲，不重新发明异步反馈。
3. 激活格动画由系统成功事件驱动，不由按钮点击直接驱动。
4. 两界替命由 `ArtifactSystem_ch6.TryRescuePlayer()` 完成战斗结算后，调用 `CombatSystem.AddJieshiRescueEffect(player, 3.0)` 写入只读表现队列；`CombatSystem.Update()` 维护持续时间，`EffectRenderer.RenderJieshiRescueEffects()` 每帧渲染裂隙、残像、冲击波、钥孔、光屑和无敌环。
5. 仙图录第五个子页展示9格、三项当前属性、Boss状态、被动说明与冷却状态。
6. 角色面板保持神器被动卡样式一致，但使用钥匙图标和青白/紫双色。
7. 表现层不得改写生命、冷却或存档，只消费只读事件数据。
8. 低性能模式减少残像层数与粒子数，但保留裂隙、浮字和无敌环三项核心反馈。

### 9.11 复用成熟方式与历史优化

必须继承的成熟做法：

| 来源 | 必须复用 |
|------|----------|
| 两界阵石 | 保存状态预检查、全局忙锁、深拷贝快照、一次性回调、失败回滚、重算属性 |
| 天青白莲 | 账号解锁进入快照、保存中弹窗、成功回包后再展示奖励、失败恢复账号与角色状态 |
| 旧神器 | 独立章节模块、格子驱动属性、角色槽位镜像、仙图录集中展示、Boss竞技场备份恢复 |
| SaveDTO优化 | 明确字段白名单，避免账号旁路与角色档互相污染 |
| PostLoadSync优化 | 所有属性系统加载完成后统一失效并重建属性缓存 |

明确不再沿用的旧方式：

推荐人工验收顺序：

1. 切到第六章，确认 `角色 → 无敌模式` 处于关闭状态。
2. 点 `资源 → 界匙神器材料包`，等待“材料包保存成功”反馈。
3. 点 `系统 → 传送到界匙`，逐格激活九格，核对每次保存弹窗、属性增量和材料扣除。
4. 第九格保存成功后，从界匙面板进入双王挑战；确认必须同时击败魔化与仙化呱大王。
5. 首胜觉醒保存成功后，点 `系统 → 界匙替命表现自测`，检查25%生命、3秒无敌、双浮字和整套裂隙表现。
6. 冷却期间再次承受致命伤应正常死亡；复活后点 `界匙冷却清零`，再测一次替命。
7. 切换另一个角色槽位并重登，确认九格、Boss、被动和剩余冷却仍按账号同步。
8. 全部验收结束后点 `神器6账号重置`；重登确认旧角色镜像不能把已清进度恢复。

注意：`神器6免费激活` 用于快速检查UI和Boss入口，不适合验证真实材料扣除。材料消费测试必须使用 `界匙神器材料包` 后保持免费激活关闭。

现有全量测试中的 `server_main.lua` 行数阈值失败属于既存欠账，但新增代码不得引入新的失败；最终报告必须把既存失败与本次失败分开列出。

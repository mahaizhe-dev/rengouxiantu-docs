## 1. 系统定位

仙图录是**账号级收集与养成系统**，记录玩家的里程碑和收藏品，提供账号级属性加成。

- 道印：修行里程碑铭刻，提供属性加成，部分可升级
- 神器：收集养成，展示贴图、激活情况和属性加成

**核心原则**:
- 所有数据为**账号级**，任意角色的行为都可以推进进度
- 属性加成作用于**账号内所有角色**

---

## 2. 系统架构

### 2.1 存储方案：独立 CloudStorage key（账号级）

```
account_atlas = { ... }          -- 仙图录独立 key（勋章 + 神器）
save_1 = { player = {...} }     -- 角色1存档
save_2 = { player = {...} }     -- 角色2存档
```

- 数据不绑定角色槽，切换角色不影响
- 任意角色的行为（境界突破、游戏时长等）都推进账号级进度

### 2.2 即时存储规则

道印升级、神器碎片激活等**不可逆进度变更**，必须在操作完成后**立即写入 CloudStorage**，不依赖定时存档。

| 触发时机 | 存储内容 |
|---------|---------|
| 道印升级（含首次获得） | `account_atlas.medals` 整段 |
| 神器碎片激活 | `account_atlas.artifact` 整段 |
| 神器 BOSS 击败 | `account_atlas.artifact` 整段 |

> 避免玩家达成条件后因意外退出导致进度丢失。

## 3. 入口与展示规则

### 3.1 入口位置

仙图录入口在**登录界面**，替代原来的"退出游戏"按钮。

| 字段 | 说明 |
|------|------|
| 位置 | `LoginScreen` 按钮列表，原"退出游戏"按钮位置 |
| 按钮文本 | "仙图录" |
| 点击行为 | 打开仙图录全屏面板（覆盖登录界面） |

> 退出游戏功能移至仙图录面板内或角色选择界面，登录界面不再直接提供退出按钮。

### 3.2 道印隐藏规则

**无法获得且未获得**的道印在列表中**隐藏**，不展示给玩家。

| 条件 | 是否显示 | 示例 |
|------|---------|------|
| 已获得（任意等级） | 显示 | 境界大师 Lv3 |
| 未获得但可获得 | 显示（灰色/锁定态） | 青云试炼（未通关 30 层） |
| 未获得且不可获得 | **隐藏** | 先人一步·化神（名额已满 10 人） |

**判断"不可获得"的逻辑**：

| trigger.type | 不可获得条件 |
|-------------|-------------|
| `server_race` | 排行榜已满（已达到 `max_slots` 上限） |
| `value_check` | 永远可获得（不会隐藏） |
| `item_consume` | 永远可获得（不会隐藏） |

> 已获得的道印**永远显示**，即使是后来名额已满的 `server_race` 类道印。

## 5. 道印系统

### 5.1 定义

道印 = **修行里程碑的铭刻**，记录修仙路上的关键成就，提供账号级属性加成。

- 每枚道印都有**属性加成**（非纯展示）
- 部分道印可**升级**，属性随等级递增
- 全部为**账号级**，任意角色的行为都可推进进度

### 5.2 图标资源

每枚道印需要一张**大尺寸专属图标**，用于弹窗展示和仙图录列表。

| 字段 | 规格 |
|------|------|
| 尺寸 | 256 × 256 px |
| 格式 | PNG，透明背景 |
| 风格 | 仙侠篆刻风，金色/青铜主色调，带光晕底纹 |
| 路径 | `Textures/atlas/seal_{id}.png`（如 `seal_realm_master.png`） |

### 5.3 获得表现

道印首次获得或升级时，触发**全屏弹窗**：

| 阶段 | 表现 |
|------|------|
| 铭刻动画 | 道印大图标（256px）从虚到实浮现，金光涟漪向四周扩散 |
| 信息展示 | 道印名称 + 当前等级 + 本次新增属性加成 |
| 退出 | 点击任意位置关闭，或 2 秒后自动消失 |

> 弹窗期间**不暂停游戏**，仅叠加 UI 层。

### 5.4 统一结构

所有道印遵循同一定义模板：

| 字段 | 说明 |
|------|------|
| `id` | 唯一标识 |
| `name` | 显示名称 |
| `type` | `levelable`（可升级）或 `one_time`（一次性） |
| `max_level` | 最高等级（一次性 = 1） |
| `trigger.type` | 触发类型（见下） |
| `trigger.source` | 检测数据来源 |
| `trigger.thresholds` | 每级解锁阈值 |
| `bonus.attrs` | 加成属性 |
| `bonus.per_level` | 每级加成值 |

**三种触发类型**：

| trigger.type | 逻辑 | 适用道印 |
|-------------|------|---------|
| `value_check` | 取账号内某数值的最大值，达到阈值即升级 | 境界大师、职业四印、青云试炼 |
| `item_consume` | 消耗指定道具累积经验，每满额升 1 级 | 仙途守护者 |
| `server_race` | 全服前 N 名达成条件，一次性获得 | 先人一步 ×2 |

### 5.5 全部道印

#### ① 数值检测类（value_check）— 6 枚

|----|------|------|---------|------|---------|----------|

| `medal_taixu` | 太虚之道 | 10 | 最高太虚（剑修）角色等级 | 10, 20, …, 100 | +1 悟性 | +10 悟性 |
| `medal_zhenyue` | 镇岳之力 | 10 | 最高镇岳（体修）角色等级 | 10, 20, …, 100 | +1 体魄 | +10 体魄 |
| `medal_luohan` | 罗汉之躯 | 10 | 最高罗汉（武僧）角色等级 | 10, 20, …, 100 | +1 根骨 | +10 根骨 |
| `medal_tianyan` | 天衍之机 | 10 | 最高天衍（阵法师）角色等级 | 10, 20, …, 100 | +1 福源 | +10 福源 |

#### ② 道具消耗类（item_consume）— 1 枚

| ID | 名称 | 级数 | 道具 | 每级经验 | 每级加成 |

> 仙途印符为运营发放的消耗品，每使用 1 枚 +100 经验。用于奖励积极反馈 BUG、提供有效建议的玩家。

#### ③ 全服竞速类（server_race）— 2 枚

| ID | 名称 | 名额 | 达成条件 | 加成 |
|----|------|------|---------|------|

### 5.7 属性生效

道印属性为**账号级全局加成**，在角色属性计算时叠加：

```
HP  += 境界大师等级 × 30
根骨 += 仙途守护者等级 × 1 + 罗汉之躯等级 × 1
悟性 += 仙途守护者等级 × 1 + 太虚之道等级 × 1
福缘 += 仙途守护者等级 × 1 + 天衍之机等级 × 1
体魄 += 仙途守护者等级 × 1 + 镇岳之力等级 × 1
DEF += 青云试炼等级 × 4
ATK += (化神已解锁 ? 5 : 0) + (合体已解锁 ? 5 : 0)
```

## 7. 大事件道印实现（代码审查基准）

> **状态**: 🔴 存在 P0 级 BUG — 多人模式下道印永远无法激活
> **涉及文件**: 5 个（配置 1 + 系统 1 + 服务端 1 + UI 1 + 存档 1）
> **添加日期**: 2026-05-09
> **目的**: 供外部 AI 审查代码质量和 BUG 根因

### 7.1 功能概述

「大事件：无天醉酒」是第 ⑤ 类触发类型 `event_grant` 的首个道印：

| 字段 | 值 |
|------|------|
| ID | `wutian_drunk` |
| 名称 | 大事件：无天醉酒 |
| 类型 | `one_time`（一次性） |
| 触发类型 | `event_grant` |
| 触发条件 | 玩家在 `xianshi_rank` 排行榜上有记录（score > 0） |
| 加成 | 福源 +3 |
| 背景 | 26年5月8日夜，大黑无天当值醉酒，仙石市场紊乱 |

**设计意图**: 服务端在玩家登录时自动将持有仙石的玩家写入 `xianshi_rank` 排行榜，客户端检测到自己在榜上后永久发放道印。这是一次性历史事件奖励，所有持有仙石的玩家都能获得。

### 7.2 数据流（设计预期）

```
┌─────────────┐     ┌─────────────────┐     ┌──────────────────┐
│  服务端登录  │ ──→ │ 仙石榜写入       │ ──→ │ 客户端查询榜单   │
│  RankHandler │     │ serverCloud      │     │ clientCloud      │
│  .lua        │     │ :SetInt          │     │ :GetMyScore      │
└─────────────┘     └─────────────────┘     └──────────────────┘
                                                     │
                                                     ▼
                                              ┌──────────────┐
                                              │ 道印解锁      │
                                              │ medals[       │
                                              │ wutian_drunk] │
                                              │ = true        │
                                              └──────────────┘
```

**预期时序**:

```
1. [服务端] RebuildRankOnLogin → serverCloud.money:Get → 仙石>0 → serverCloud:SetInt("xianshi_rank", xianshi)
2. [客户端] SaveLoader 完成存档加载 → AtlasSystem.PostLoadSync()
3. [PostLoadSync step 3c] → AtlasSystem.CheckEventGrants()
4. [CheckEventGrants] → clientCloud:GetMyScore("xianshi_rank", {...})
5. [回调] scoreInfo.score > 0 → medals.wutian_drunk = true → SaveImmediate()
```

**评价**: 配置层无问题。

#### (B) 服务端: `scripts/network/RankHandler.lua`

```lua
-- RebuildRankOnLogin() 第 ~160 行
-- 仙石榜入榜逻辑：
serverCloud.money:Get(userId, {
    ok = function(playerMoneys)
        local xianshi = playerMoneys and playerMoneys.xianshi or 0
        if xianshi > 0 then
            serverCloud:SetInt(userId, "xianshi_rank", xianshi, { ... })
            -- BatchSet 写入附加信息
            local xsBatch = serverCloud:BatchSet(userId)
            xsBatch:Set("xianshi_info", { xianshi = xianshi, taptapNick = taptapNick })
            xsBatch:Save("仙石榜入榜", { ... })
        end
    end,
})
```

**评价**: 服务端逻辑正确。日志已确认入榜成功：
```
[Server] xianshi_rank: enrolled userId=1092360835 xianshi=26
```

### 7.4 🔴 BUG 根因分析

#### 问题表现

客户端日志显示 `PostLoadSync complete`，但 `CheckEventGrants` 内部 **无任何输出**——既没有 "Event grant medal unlocked"，也没有 "not on leaderboard"，也没有 "Event grant check ERROR"。

#### 根因：多人模式下 `clientCloud` 全局变量为 nil

**项目配置** (`.project/settings.json`):
```json
{
  "@runtime": {
    "multiplayer": {
      "enabled": true         // ← 多人模式
    }
  }
}
```

**`client_main.lua` 第 43 行已明确记录**:
```lua
print("[ClientMain] clientCloud=" .. tostring(clientCloud) .. " clientScore=" .. tostring(clientScore))
```

**`client_main.lua` 第 136-143 行注释**:
```lua
--- 在 multiplayer 构建下 standalone 路径不可用（clientCloud/clientScore 为 nil），
...
-- 因为 multiplayer 构建下 standalone 路径无法工作（clientCloud=nil）
```

**结论**: 在 `multiplayer.enabled = true` 的构建下：
- `clientCloud` 全局变量 = **nil**
- `clientScore` 全局变量 = **nil**
- 所有云操作应通过 `CloudStorage` 模块封装层访问（CloudStorage 内部在网络模式下使用 C2S 协议转发到服务端）

#### BUG 影响范围

`AtlasSystem.lua` 中 **3 处** 直接访问 `clientCloud` 全局变量：

| 行号 | 函数 | 调用 | 影响 |
|------|------|------|------|
| 477 | `CheckEventGrants` | `clientCloud:GetMyScore(...)` | 🔴 **event_grant 道印永远无法激活** |
| 538 | `CheckRaceSlots` | `clientCloud:GetLeaderboard(...)` | ⚠️ 竞速道印名额状态无法查询（UI 隐藏规则失效） |
| 816 | `SaveImmediate` | `clientCloud:SetScore(...)` | ⚠️ 有 CloudStorage 回退路径，影响较小 |

#### 为什么 `SaveImmediate` 没有完全崩溃

`SaveImmediate` 有 `else` 分支回退到 `CloudStorage.BatchSet()`，所以其他道印（value_check 等）的存储在多人模式下仍然能工作。但 `CheckEventGrants` 和 `CheckRaceSlots` 没有回退路径。

#### 为什么静默失败无日志

```lua
if clientCloud then
    clientCloud:GetMyScore(...)
end
-- ← clientCloud 为 nil 时，整个 if 块被跳过，无 else 分支，无日志
```

### 7.5 修复方案（待实施）

`event_grant` 的排行榜查询（`GetMyScore`）不在 `CloudStorage` 封装中。有两个方案：

**方案 A**: 在 `CloudStorage` 中添加 `GetMyScore` 和 `GetLeaderboard` 的网络模式封装（需要新增 C2S/S2C 协议）

**方案 B**: 直接在 `CheckEventGrants` 中改用服务端查询——通过 `SaveProtocol` C2S 事件让服务端代查排行榜并返回结果

**推荐方案 B**（改动最小）：
1. 定义新的 C2S/S2C 事件对：`C2S_CheckEventGrant` / `S2C_CheckEventGrantResult`
2. 客户端发送 `{ leaderboard = "xianshi_rank" }` 到服务端
3. 服务端用 `serverCloud:GetUserRank(userId, leaderboard)` 查询
4. 服务端将结果通过 `S2C_CheckEventGrantResult` 返回
5. 客户端收到后执行道印解锁逻辑

# 法宝挑战生成完整链路

> **目的**: 新增一个阵营的法宝挑战时，按本文档逐步操作，避免遗漏。
> **已验证阵营**: 血煞盟（xuesha）、浩气宗（haoqi）
> **创建时间**: 2026-04-25

---

        [4] = { name = "声望4级", tier = 6, tokenId = "sha_hai_ling", unlockCost = 500,  monsterId = "challenge_xxx_r4" },
        [5] = { name = "声望5级", tier = 7, tokenId = "sha_hai_ling", unlockCost = 1000, monsterId = "challenge_xxx_r5" },
        [6] = { name = "声望6级", tier = 8, tokenId = "taixu_token",  unlockCost = 1000, monsterId = "challenge_xxx_r6" },

    -- ④ 旧版兼容（仅已上线阵营需要，全新阵营留空即可）

### 关键约定

| 字段 | 约定 | 参考 |
|------|------|------|

## 第二步：EquipmentData.FabaoTemplates 添加法宝模板

**文件**: `scripts/config/EquipmentData.lua` → 文件末尾 `FabaoTemplates` 表

### 模板结构

```lua
EquipmentData.FabaoTemplates.fabao_xxx = {
    name          = "法宝中文名",       -- 显示名（实际显示为 "法宝名Lv.N"）
    slot          = "exclusive",        -- 固定为 exclusive 槽位
    mainStatType  = "wisdom",           -- 主属性类型: wisdom / fortune / constitution / physique
    mainStatFormula = "xianyuan",        -- 公式: "xianyuan"(仙缘) 或 "standard"(标准)
    mainStatBase  = 5,                  -- 仙缘公式的倍数因子

    skillId       = "xxx_skill",        -- 法宝主动技能 ID（SkillData 中定义）

    -- 图标（3 档，T3-T4 / T5-T8 / T9+）
    iconByTier = {
        [3] = "icon_fabao_xxx_lv1.png",
        [4] = "icon_fabao_xxx_lv1.png",
        [5] = "icon_fabao_xxx_lv2.png",
        [6] = "icon_fabao_xxx_lv2.png",
        [7] = "icon_fabao_xxx_lv3.png",
        [8] = "icon_fabao_xxx_lv3.png",
        [9] = "icon_fabao_xxx_lv3.png",
    },

    -- 出售价格（金币）
    sellPriceByTier = {
        [3] = 300, [4] = 400, [5] = 500,
        [6] = 600, [7] = 800, [8] = 1000, [9] = 1200,
    },
}
```

### 属性公式说明

| 公式 | 计算方式 | 典型用途 |
|------|---------|---------|
| **xianyuan（仙缘）** | `floor(tier × qualityMult) × mainStatBase` | 仙元属性（福缘、悟性等） |
| **standard（标准）** | `base × TIER_MULTIPLIER[tier] × qualityMult` | 战斗属性（攻击、防御等） |

**已有法宝的主属性选择**:

| 法宝 | mainStatType | 说明 | 状态 |
|------|-------------|------|------|
| 血海图 | wisdom（悟性） | 击杀经验+N、技能伤害+1%/5点 | ✅ 已上线 |
| 浩气印 | fortune（福缘） | 击杀金币+N、灵韵概率+2%/5点 | ✅ 已上线 |
| 第三阵营 | constitution（根骨） | 每5点+1%防御、每25点+1%重击概率/伤害 | ⬜ 待设计 |
| 第四阵营 | physique（体魄） | 每5点+1%生命、每10点+1%回血、每25点+1%血怒 | ⬜ 待设计 |

> **注意**: 四种仙元属性各对应一个法宝阵营。wisdom 和 fortune 已被占用，剩余 constitution 和 physique 可用于后续两个阵营。四个阵营全部上线后，仙元属性即全部分配完毕。

## 存档防刷方案（2-Save 模式）

> **创建时间**: 2026-04-25
> **背景**: 挑战 BOSS 为 emperor_boss（HP×20, ATK×1.8），战斗持续 30+ 秒。每次挑战消耗 10 令牌（ATTEMPT_COST=10）。

### 存档时间线

```
Save① 入场存档（令牌已扣除 → 立即持久化）
  ↓
  🔒 挑战进行中 — 不存档
    ├─ BOSS 击败: completed=true 仅在内存中标记
    ├─ 自动存档（60s 周期）: 被 ChallengeSystem.active 抑制
    └─ 强退/重启: 令牌已消耗 + completed 未持久化 = 需重新挑战
  ↓
Save③ 领奖存档（奖励入库 → 一次性持久化所有状态）
```

### 防刷效果

| 漏洞场景 | 玩家行为 | 实际后果 |
|---------|---------|---------|
| **看奖不满意强退** | 击败 BOSS → 看到掉落 → 强退 | 令牌已消耗，completed 未存档 → 需重打 30+ 秒 emperor_boss |
| **领奖前断线** | 击败 BOSS → 掉线 | 同上 → 重新挑战（令牌不退） |
| **自动存档偷跑** | 挑战中等待 60s | `ChallengeSystem.active = true` 时 SaveSystem 跳过自动存档 |

### 代码实现位置

| 组件 | 文件:行 | 作用 |
|------|---------|------|
| Save① | `ChallengeSystem.lua:EnterReputation()` / `Enter()` | 入场时 `SaveSystem.Save()` |
| 中间不存档 | `ChallengeSystem.lua:_onRepMonsterDeath()` / `OnMonsterDeath()` | 已移除 `SaveSystem.Save()` |
| auto-save 抑制 | `SaveSystem.lua:Update()` 行 129 | `if ChallengeSystem.active then skip` |
| Save③ | `ChallengeSystem.lua:GrantReputationChoice()` | 领奖后 `SaveSystem.Save()` |

### 为什么不存 3 次

旧方案在 BOSS 击败时存档（Save②），导致：
1. 玩家可在看到掉落后强退重进，用 10 令牌无限重掷随机掉落
2. Save① 和 Save② 间距仅 30s，在 `SavePersistence.saving` 互斥锁下可能产生竞态

2-Save 方案彻底消除这两个问题。

## 数据流全景图

```
玩家与 NPC 交互
    ↓
ChallengeUI 读取 ChallengeConfig.FACTIONS[factionKey]
    ↓ 显示声望等级列表、令牌余额、解锁消耗
    ↓
玩家选择声望等级 → 扣除令牌 (ATTEMPT_COST=10)
    ↓
💾 Save① — SaveSystem.Save()（令牌扣除持久化）
    ↓
ChallengeSystem 生成 ARENA_SIZE×ARENA_SIZE 副本
    ↓ 放置 BOSS: MonsterTypes_ch3.Types[repCfg.monsterId]
    ↓
🔒 挑战中（ChallengeSystem.active=true，auto-save 被抑制）
    ↓
玩家击败 BOSS → completed=true（仅内存，不存档）
    ↓
ChallengeSystem._onRepMonsterDeath(factionKey, repLevel)
    ├─ 查 factionCfg.fabaoTplId → 传入 LootSystem.CreateFabaoEquipment
    ├─ 首通？→ 保底品质 2 件法宝（二选一）
    │       品质 = REPUTATION_FIRST_CLEAR_QUALITY[repLevel]
    └─ 非首通？→ fabaoA(随机) + [物品掉落(repLevel×1%) | fabaoB(随机)]
        ↓
    LootSystem.CreateFabaoEquipment(templateId, tier, quality)
        ├─ EquipmentData.FabaoTemplates[templateId] → 模板
        ├─ quality==nil → RollQuality(white, maxQ)
        │   maxQ: T3-T4→purple, T5-T8→orange, T9+→cyan
        ├─ 仙缘公式: floor(tier × qualityMult) × mainStatBase
        ├─ 副属性: GenerateSubStats(品质条数, tier, 排除主属性)
        ├─ 灵器+: spiritStat + sellCurrency="lingYun"
        └─ 输出完整 item 对象 (isFabao=true, isSpecial=true)
        ↓
    EventBus.Emit("challenge_victory", { choices={item1, item2} })
        ↓
    ChallengeVictoryUI 显示二选一
        ↓
    ChallengeSystem.GrantReputationChoice(selectedItem)
        ├─ 法宝 → InventorySystem:AddEquipment(item)
        └─ 物品 → _grantExtraDrop → AddConsumable
        ↓
    💾 Save③ — SaveSystem.Save()（completed + 奖励一次性持久化）
```

## 已知踩坑记录

### 1. FabaoTemplates 不是独立文件

法宝模板定义在 `EquipmentData.lua` 文件末尾的 `EquipmentData.FabaoTemplates` 表中，**不是** `FabaoTemplates.lua` 独立文件。

### 2. CreateFabaoEquipment 不在 FabaoSystem.lua

法宝创建函数在 `LootSystem.lua:522` 的 `LootSystem.CreateFabaoEquipment()` 中，**不存在** `FabaoSystem.lua` 文件。

### 3. icon 字段是图片路径，不是 emoji

法宝的 `icon` 字段存储的是图片文件名（如 `"icon_fabao_xuehaitu_lv1.png"`），由 `iconByTier[tier]` 映射。这与普通道具的 emoji icon 不同（参见 rules.md §11）。

### 4. 灵器品质的特殊售价

灵器（cyan）及以上品质的法宝，售价会被覆盖为灵韵货币：
```lua
sellCurrency = "lingYun"
sellPrice = max(1, tier - 4)  -- T5=1, T6=2, ..., T9=5
```

### 5. 消耗品掉落的晚绑定陷阱

物品掉落中的 `consumableId` 必须在 `GameConfig.CONSUMABLES` 中已注册，否则入库静默失败。详见 rules.md §13。

### 6. 旧版 tiers 字段不可删除

已上线阵营（血煞盟、浩气宗）的 `tiers` 字段必须永久保留，供旧存档 v2→v3 迁移使用。

### 7. 储备 BOSS 已用完

沈墨（血煞盟）和陆青云（浩气宗）是仅有的两个预设计 BOSS 角色。后续阵营**没有现成 BOSS 可复用**，必须从零设计：
- BOSS 名称 / 技能机制 / 阶段配置 / 专属机制（如血煞印记、正气结界）
- BOSS 肖像 256×256（不能再复用 NPC 肖像）
- 法宝图标 3 档 64×64

### 8. 已有 BOSS 技能模式参考

| 阵营 | BOSS | 基础技能 | 专属机制 | 阶段特殊技能 |
|------|------|---------|---------|-------------|
| 血煞盟 | 沈墨 | line_charge, whirlwind, cross_smash | bloodMark（血煞印记，每层每秒扣%HP） | shenmo_blood_flood, shenmo_shadow_clone |
| 浩气宗 | 陆青云 | shield_bash, rect_sweep, ground_pound | barrierZone（正气结界，周期性伤害区域） | luqingyun_sword_array, luqingyun_prison |

新阵营 BOSS 应设计**差异化的专属机制**，避免与已有机制雷同。

### 9. 挑战中禁止存档

2-Save 方案要求挑战进行中（`ChallengeSystem.active=true`）不触发任何存档。如果未来新增其他系统在挑战中可能调用 `SaveSystem.Save()`，必须检查 `ChallengeSystem.active` 守卫。详见本文"存档防刷方案"章节。

### 10. InventorySystem 主属性累加遗漏（已修复 v1.8.4）

**问题**: `InventorySystem.lua` 中 `RecalcEquipStats()` 累加装备主属性时，只处理了 `atk/def/maxHp/speed/hpRegen/critRate/dmgReduce/skillDmg/wisdom` 9 种属性，**完全遗漏了 `fortune`（福缘）、`constitution`（根骨）、`physique`（体魄）**。

**影响**: 浩气印的 fortune 主属性和青云塔的 constitution 主属性在穿戴后**不生效**，面板数值为 0，实际损失 15~90+ 仙元属性点（取决于法宝 Tier 和品质）。

**根因**: 添加仙元属性系统时，EquipmentData 和 LootSystem 正确生成了 `mainStat.fortune/constitution/physique`，但 InventorySystem 的累加循环未同步更新。

**修复**: 在 `InventorySystem.lua` 主属性累加块中补充 3 行：
```lua
totalFortune = totalFortune + (item.mainStat.fortune or 0)
totalConstitution = totalConstitution + (item.mainStat.constitution or 0)
totalPhysique = totalPhysique + (item.mainStat.physique or 0)
```

**教训**: 新增属性类型时，必须全链路验证——生成（LootSystem）→ 存储（SaveSystem）→ 累加（InventorySystem）→ 显示（Player/UI），任何一环遗漏都会导致属性静默丢失。

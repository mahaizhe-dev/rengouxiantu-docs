# 伤害跳字规范与视觉优化执行方案

> 文档状态：执行前方案（已完成方案自审 v1）
>
> 创建日期：2026-07-14
>
> 适用范围：主世界战斗表现
>
> 明确排除：多人副本、`DungeonHUD`、`DungeonHandler`、副本网络协议与伤害上报
>
> 核心约束：不修改伤害公式，不改变 `HitResolver` 结算顺序，不改变服务器权威逻辑

---

## 1. 结论摘要

当前伤害跳字已经不是单纯的渲染样式问题，而是“表现语义缺失、调用方式分散、视觉优先级混杂”的综合问题。

当前代码搜索结果中，`AddFloatingText` 在 `scripts/` 下原始匹配为 296 处、分布于 58 个 Lua 文件。这个数字包含定义、兼容调用和测试引用，但足以说明不能通过一次性替换字符串完成治理。

当前 `CombatSystem.lua` 物理行数约为 1079 行，`test_file_budget.lua` 中预算为 1090 行。新逻辑不能继续直接展开写入该文件。

本方案采用以下路线：

1. 新增独立的 `CombatFeedback` 表现系统，接收结构化战斗反馈。
2. 保留 `CombatSystem.AddFloatingText` 作为旧接口，不一次迁移所有调用点。
3. 普通攻击只显示主数字。
4. 特殊技能、被动和装备触发使用“来源标签 + 主数字”的复合跳字。
5. 暴击、天诛、重击、DOT 等采用可组合字段，不用单一类型枚举互相覆盖。
6. 主数字表达伤害结果，来源标签表达技能身份，角标和动画表达暴击、天诛等命中特征。
7. 加入分轨、短窗聚合、标签去重、每目标上限和全局上限。
8. 将动画、格式化、聚合、分轨设计为纯 Lua 逻辑，先自动测试，再接 NanoVG 渲染。
9. 扩充现有 GM 特效压测入口，建立可重复的运行态视觉测试矩阵。
10. 通过功能开关分阶段启用，出现问题可立即回退旧表现。

### 2.3 当前实现

`CombatSystem.AddFloatingText` 当前接口为：

```lua
CombatSystem.AddFloatingText(x, y, text, color, lifetime, baseScale)
```

它只保存：

- 世界坐标
- 裸字符串
- 颜色
- 生命周期
- 基础缩放
- 已经过时间
- 垂直偏移

当前统一更新逻辑为：

```lua
ft.offsetY = ft.offsetY - 40 * dt
```

当前渲染逻辑的主要特征：

- 基准字号来自 `UITheme.worldFont.damage = 18`
- 出现阶段从约 `1.5x` 缩回
- 最后 30% 生命周期线性淡出
- 只有一次右下偏移的黑色阴影
- 没有真正描边
- 没有目标归属
- 没有分轨
- 没有聚合
- 没有标签去重
- 没有每目标或全局数量上限

### 2.4 当前明确问题

#### 问题 A：普通攻击存在重复表现

`CombatSystem.AutoAttack` 在暴击或天诛时直接创建特殊跳字，随后又发出 `player_attack`。

`player_attack` 监听器无条件创建一次普通数字。

因此暴击和天诛可能同时出现：

```text
天诛 12345
12345
```

或者：

```text
暴击 12345
12345
```

#### 问题 B：特殊技能依赖字符串表达语义

当前存在多种拼接方式：

```text
诛仙剑气 12345
诛仙暴击 12345
天诛连击 12345
连击暴击 12345
神塔镇压 12345
血神暴击 12345
技能图标 + 暴击 + 数字
```

渲染器无法知道：

- 哪一部分是技能名
- 哪一部分是数字
- 是否为暴击
- 是否为天诛
- 是否为多段技能
- 是否允许聚合
- 是否应该重复显示技能名

#### 问题 C：非伤害提示与伤害共用同一队列

同一队列还包含：

- 技能开启和关闭
- Buff 层数
- 背包已满
- 地图加载异常
- 挑战结果
- 掉落提示
- 解锁失败
- NPC 或系统错误

这导致战斗数字无法建立稳定的优先级、聚合和数量上限。

#### 问题 D：参数接口容易误用

`passives.lua` 中已有调用把 `textScale` 作为第 5 参数传入，实际修改的是 `lifetime`，没有设置 `baseScale`。

这类问题来自位置参数缺乏语义，后续新接口必须改为结构化 table 参数。

#### 问题 E：契约存在但表现层没有消费

`BattleContracts.lua` 已定义：

- `DamageTag`
- `HitContext`
- `HitResult`

但当前生产表现仍由调用点拼接字符串。现有契约可以作为来源分类参考，但不应为了跳字改造而强行重写现有结算主链。

## 3. 目标与非目标

### 3.1 目标

1. 玩家在 0.2 秒内可以区分普通伤害、技能伤害、暴击、天诛、重击、DOT、治疗和自身受伤。
2. 特殊技能身份能够保留，不被降级为普通纯数字。
3. 多段、AOE 和高攻速场景下，信息仍然可读。
4. 表现语义不再依赖调用点拼字符串。
5. 相同结算结果只产生一次跳字。
6. 动画和排布在不同帧率下结果稳定。
7. 新系统可以独立测试、分批启用和快速回退。

### 3.2 非目标

本方案不做以下事项：

- 不调整攻击、防御、暴击、天诛、重击和真实伤害公式
- 不修改 `Player:ApplyCrit`
- 不修改 `Monster:TakeDamage` 的结算规则
- 不把表现逻辑塞入 `HitResolver`
- 不修改多人副本
- 不修改副本客户端上报与服务端校验
- 不一次性清理全部旧世界提示
- 不重构全部战斗事件
- 不引入新的字体资源

## 4. 设计原则

### 原则 1：结果、来源和命中特征分开表达

跳字不能再用一个字符串同时承载全部含义。

- 主数字：表达实际伤害或恢复值
- 来源标签：表达技能、被动、装备或宠物技能名称
- 角标：表达暴击、天诛、吸收等短语义
- 颜色与动画：增强层级，不作为唯一识别手段

### 原则 2：字段可组合，不使用互斥大枚举

“技能暴击重击”同时具有：

- 来源是技能
- 伤害标签是重击
- 暴击等级是暴击

因此不能只设计：

```lua
type = "skill" -- 或 "crit" 或 "heavy"
```

应拆分为：

```lua
sourceType = "skill"
damageTag = "heavy"
criticality = "crit"
```

### 原则 3：表现层只消费结算结果

`CombatFeedback` 不重新计算伤害，不决定暴击，不改变目标生命值。

正确顺序：

```text
命中结算
  -> 得到 actualDamage / isCrit / isTianzhu
  -> 生成 CombatFeedbackEvent
  -> 表现系统排队和渲染
```

### 原则 4：特殊技能保留身份，但不能刷屏

特殊技能伤害不归入普通纯数字。

技能名默认附着在该技能对该目标造成的第一条可见伤害上。多段伤害和 DOT 后续数字不重复刷技能名。

### 原则 5：旧入口兼容，新入口严格

旧 `AddFloatingText` 继续服务未迁移的世界提示。

新战斗代码只使用结构化接口，禁止继续拼接：

```lua
skill.name .. "暴击 " .. damage
```

## 5. 表现语义模型

### 5.1 通道划分

| 通道 | 用途 | 位置 |
|---|---|---|
| `target_damage` | 对敌伤害 | 目标头顶 |
| `self_combat` | 玩家受伤、治疗、吸收、闪避 | 玩家专属轨道 |
| `entity_status` | Buff、Debuff、技能开启、层数变化 | 实体状态轨道 |
| `world_hint` | 背包、地图、交互、系统提示 | 保留旧入口 |

第一版只重构前三个战斗通道，不迁移 `world_hint`。

### 5.2 建议事件结构

```lua
---@class CombatFeedbackEvent
---@field kind string
---@field value number|nil
---@field source table|nil
---@field target table|nil
---@field x number|nil
---@field y number|nil
---@field sourceType string|nil
---@field sourceId string|nil
---@field sourceName string|nil
---@field sourceShortName string|nil
---@field damageTag string|nil
---@field criticality string|nil
---@field impact string|nil
---@field castId string|number|nil
---@field effectId string|nil
---@field hitIndex number|nil
---@field labelPolicy string|nil
---@field aggregatePolicy string|nil
---@field priority number|nil
---@field color table|nil
```

### 5.3 字段定义

#### `kind`

| 值 | 含义 |
|---|---|
| `damage` | 对目标造成伤害 |
| `hurt` | 玩家或宠物受到伤害 |
| `heal` | 恢复生命 |
| `absorb` | 护盾吸收 |
| `evade` | 闪避 |
| `immune` | 免疫 |
| `status` | Buff、Debuff、层数或状态切换 |

#### `sourceType`

| 值 | 含义 |
|---|---|
| `basic` | 玩家普攻 |
| `skill` | 主动技能 |
| `passive` | 职业或技能被动 |
| `equipment` | 装备触发 |
| `artifact` | 神器触发 |
| `pet` | 宠物普攻 |
| `pet_skill` | 宠物技能 |
| `boss` | Boss 机制 |
| `environment` | 场地和环境 |

#### `criticality`

| 值 | 含义 |
|---|---|
| `normal` | 普通命中 |
| `crit` | 暴击 |
| `tianzhu` | 天诛 |

优先级固定为：

```text
tianzhu > crit > normal
```

#### `impact`

| 值 | 含义 |
|---|---|
| `normal` | 普通冲击 |
| `heavy` | 重击或高冲击 |
| `dot` | 持续伤害 |
| `minor` | 低优先级追加伤害 |

#### `damageTag`

优先复用 `BattleContracts.DamageTag`：

- `normal`
- `skill`
- `heavy`
- `pet`
- `set_aoe`
- `dot`
- `blood_rage`
- `artifact`

表现层可以接受额外标签，但不能据此改变结算。

### 5.4 标识生成规则

#### `castId`

- 主动技能在一次施法开始时生成。
- 使用客户端本地单调递增序号，不使用随机数。
- 同一次施法产生的全部目标、全部段数共享同一个 `castId`。
- 连击如果属于同一次技能施法，沿用原 `castId`，并通过 `hitIndex` 区分。
- 普攻可以使用本地攻击序号作为 `castId`。

#### `effectId`

- DOT、Buff、装备、神器和被动使用稳定配置 ID。
- 不使用显示名称作为唯一 ID。
- 同名但规则不同的效果必须有不同 `effectId`。

#### `targetKey`

- 优先使用实体稳定 ID。
- 没有稳定 ID 时，由 `CombatFeedback` 为对象引用分配运行期 ID。
- `targetKey` 只用于表现队列，不写入存档和网络协议。

这些标识必须由调用方或表现桥接层明确提供，不能在渲染阶段通过字符串推断。

## 6. 特殊技能伤害规范

### 6.1 核心结论

特殊技能伤害不是普通输出样式。

它在结算层仍然是 `damage`，在表现层使用：

```text
来源标签
主伤害数字
可选角标
```

示意：

```text
诛仙剑气    暴击
   12.8万
```

主数字始终是第一视觉层级，技能名不能比数字更大。

### 6.2 主动单次技能

规则：

- 技能名作为小号来源标签
- 数字使用技能伤害预设
- 暴击或天诛改变主数字和角标
- 不再额外在玩家头顶重复显示一次技能名

示例：

```text
神塔镇压
  8.64万
```

暴击：

```text
神塔镇压    暴击
  12.9万
```

### 6.3 主动多段技能

规则：

1. 同一 `castId` 对同一目标的第一条可见伤害显示技能名。
2. 后续伤害只显示数字。
3. 100ms 内满足聚合条件的普通段可以累计为一条。
4. 聚合后可显示小号次数标记，例如 `x3`。
5. 暴击和天诛不与普通段聚合。

示例：

第一段：

```text
剑影连斩
  1.28万
```

后续聚合：

```text
3.64万  x3
```

### 6.4 AOE 技能

规则：

- 每个目标的第一条伤害可以显示技能名。
- 同一目标在标签冷却时间内不重复显示相同技能名。
- 默认标签冷却为 `0.8s`。
- 当同屏目标过多时，低优先级来源标签可以被裁掉，但主数字不能被裁掉。
- 暴击和天诛标签不因来源标签裁剪而消失。

### 6.5 DOT 技能

规则：

- 第一个有效 tick 显示技能名。
- 后续 tick 默认只显示较小数字。
- 同一 `effectId + target` 在 `160ms` 内允许聚合。
- DOT 不与直接伤害聚合。
- DOT 不显示夸张弹跳。
- DOT 标签默认 `1.0s` 内只出现一次。

示例：

首跳：

```text
焚血
  2380
```

后续：

```text
2460
```

### 6.6 被动、装备和神器触发

规则：

- 使用短来源标签，例如“裂风”“灭影”“连击”“天蓬遗威”。
- 标签不使用图标或 emoji 作为核心语义。
- 低频触发默认每次显示来源。
- 高频触发采用 `0.5-0.8s` 标签冷却。
- 固定伤害、真实伤害不通过特殊颜色暗示公式，只通过来源标签区分。

### 6.7 重击、暴击和天诛组合

重击与暴击、天诛可以同时成立。

表现解析顺序：

1. `criticality` 决定主数字颜色和核心尺寸。
2. `impact=heavy` 决定更强的位移和停顿。
3. `sourceName` 决定来源标签。
4. `damageTag` 只做辅助样式与聚合隔离。

示例：

```text
裂地    天诛
  26.8万
```

不能退化成单行：

```text
天诛裂地重击暴击 268000
```

### 6.8 非伤害技能

以下内容不进入伤害轨道：

- 技能开启和关闭
- 层数变化
- 护盾生成
- 气血不足
- 技能释放失败
- Buff 获得或消散

它们进入 `entity_status` 或 `self_combat` 通道。

## 7. 文案规范

### 7.1 主数字

- 只显示数值
- 治疗使用 `+`
- 玩家受伤使用 `-`
- 对敌伤害默认不显示 `-`
- 不显示“伤害”二字
- 不显示 HP 后缀，只有主动消耗生命时可以显示 `HP`

### 7.2 来源标签

- 优先使用 `sourceShortName`
- 没有短名时使用 `sourceName`
- 中文建议不超过 6 个汉字
- 超出时截断并加省略号
- 禁止把技能等级、图标和数字全部塞入标签
- 宠物技能等级可放入调试信息，不放在主战斗标签

### 7.3 角标

允许的核心角标：

- 暴击
- 天诛
- 吸收
- 免疫

“真实伤害”“破防”等不默认做大角标，优先使用来源标签或较弱的辅助标记，避免信息过载。

### 7.4 禁止形式

新战斗反馈禁止：

```text
emoji + 技能名 + 暴击 + 天诛 + 数字
```

禁止依赖 emoji 作为技能类别和伤害类型的唯一表达。

## 8. 数字格式规范

### 8.1 格式目标

- 保持约 4-6 个主要可见字符
- 不使用千分位逗号
- 保留真实数值供结算、日志和统计
- 缩写只影响显示

### 8.2 建议规则

| 数值范围 | 示例 |
|---|---|
| `< 10,000` | `9999` |
| `10,000-99,999` | `1.25万` |
| `100,000-999,999` | `12.5万` |
| `1,000,000-99,999,999` | `125万`、`1250万` |
| `100,000,000-999,999,999` | `1.25亿` |
| `>= 1,000,000,000` | `12.5亿`、`125亿` |
| `>= 1,000,000,000,000` | `1.25万亿` |

格式化采用最多 3 个有效数字，不保留无意义尾零：

```text
1.20万 -> 1.2万
12.0万 -> 12万
```

### 8.3 边界规则

- `value <= 0` 的普通伤害不显示数字。
- 闪避显示“闪避”。
- 完全吸收显示“吸收”或吸收值。
- 无敌显示“免疫”。
- 小数伤害统一取与实际扣血一致的整数结果，不在表现层二次计算。

## 9. 视觉规范

### 9.1 字体层级

当前基准字号为 `18`，新方案继续以此为基准。

| 类型 | 主数字字号 | 来源标签 | 角标 |
|---|---:|---:|---:|
| 普通伤害 | 18 | 无 | 无 |
| 技能伤害 | 19 | 11-12 | 10-11 |
| 暴击 | 22 | 12 | 11 |
| 天诛 | 25 | 12 | 12 |
| 重击 | 21 | 11-12 | 10-11 |
| DOT | 15 | 10-11 | 无 |
| 治疗 | 18 | 11 | 无 |
| 玩家受伤 | 20 | 无 | 无 |

- 暴击必须始终可识别
- 天诛必须始终保持最高层级
- 部分技能色对浅色地图缺乏对比度

- 可附加一次低透明度右下阴影，但不能代替描边

来源标签：

- 使用一次阴影 + 一次主文字
- 不使用完整 4 向描边，控制绘制成本

不在每帧创建字体，不在每条跳字创建 NanoVG 资源。

## 10. 动画规范

### 10.1 通用时间段

动画分为：

1. 出现：快速建立可读性
2. 冲击：表达命中强度
3. 停留：保证数字可读
4. 上浮：与目标拉开
5. 淡出：清理画面

### 10.2 动画预设

| 预设 | 时长 | 垂直位移 | 水平漂移 | 特征 |
|---|---:|---:|---:|---|
| `normal` | 0.82s | 42px | 4-8px | 轻弹出 |
| `skill` | 0.92s | 48px | 6-10px | 标签跟随 |
| `crit` | 1.02s | 54px | 8-12px | 更强弹出和短停顿 |
| `tianzhu` | 1.15s | 60px | 8-12px | 高层级、亮度闪现 |
| `heavy` | 1.00s | 50px | 5-8px | 短暂停顿后上浮 |
| `dot` | 0.62s | 24px | 2-5px | 不弹跳、快速淡出 |
| `heal` | 0.90s | 38px | 3-6px | 平稳上浮 |
| `hurt` | 0.85s | 34px | 4-8px | 玩家专属方向 |
| `status` | 0.95s | 28px | 0-4px | 低运动量 |

### 10.3 缩放曲线

普通伤害：

```text
0ms    0.88x
70ms   1.06x
150ms  1.00x
结束   1.00x
```

暴击：

```text
0ms    0.82x
80ms   1.24x
180ms  1.06x
结束   1.00x
```

天诛：

```text
0ms    0.78x
90ms   1.36x
210ms  1.08x
结束   1.00x
```

不再让所有文字统一从 `1.5x` 缩回。

### 10.4 透明度曲线

- 前 40-60ms 从 0 到 1
- 中段保持完全不透明
- 最后约 30% 时长使用缓出淡出
- `dt` 突然增大时必须夹紧到 `[0, 1]`
- 到期后必须当帧移出队列

### 10.5 帧率一致性

位置、缩放和透明度全部由：

```lua
progress = elapsed / duration
```

计算，不能依赖固定帧数。

10 FPS、30 FPS 和 60 FPS 下相同时间点的结果应基本一致。

## 11. 排布与拥挤治理

### 11.1 目标归属

每条战斗反馈必须能够得到稳定 `targetKey`：

优先顺序：

1. 目标稳定 ID
2. 目标运行时 ID
3. 目标对象引用的稳定映射
4. 无目标时使用位置桶

### 11.2 分轨

每个目标使用固定水平轨道序列：

```text
0, -16, +16, -30, +30
```

新跳字优先选择当前占用最少的轨道。

高优先级跳字优先靠近中心：

- 天诛
- 暴击
- 玩家受伤
- 闪避和免疫

### 11.3 垂直起点

同目标连续出现时，在基础锚点上增加小幅垂直层级偏移，避免新旧数字完全覆盖。

起点偏移不得持续累积。活跃项清空后恢复基础锚点。

### 11.4 聚合键

聚合键至少包含：

```text
targetKey
sourceType
sourceId/effectId
damageTag
criticality
impact
castId（存在时）
```

### 11.5 聚合窗口

| 玩家受伤 | 默认不聚合 |

### 11.6 聚合显示

聚合项保留：

- 总数值

## 12. 技术架构

### 12.1 建议模块

```text
scripts/
├── config/
│   └── CombatFeedbackConfig.lua
├── systems/
│   └── combat/
│       ├── CombatFeedback.lua
│       └── CombatFeedbackBridge.lua
├── rendering/
│   └── effects/
│       └── combat_feedback.lua
└── tests/
    ├── test_combat_feedback.lua
    ├── test_combat_feedback_aggregation.lua
    └── test_combat_feedback_integration.lua
```

### 12.2 `CombatFeedbackConfig.lua`

职责：

- 颜色 Token
- 字号
- 动画预设
- 数量上限
- 聚合窗口
- 优先级
- 标签冷却

不包含：

- 玩家或怪物状态
- 伤害计算
- NanoVG 句柄

### 12.3 `CombatFeedback.lua`

职责：

- 事件校验与默认值
- 数字格式化
- 标签去重
- 聚合
- 分轨
- 数量上限
- 优先级淘汰
- 生命周期更新
- 输出与渲染无关的绘制模型

建议 API：

```lua
CombatFeedback.Emit(event)
CombatFeedback.EmitDamage(event)
CombatFeedback.EmitHeal(event)
CombatFeedback.EmitHurt(event)
CombatFeedback.EmitStatus(event)
CombatFeedback.Update(dt)
CombatFeedback.GetItems()
CombatFeedback.Clear()
```

`Emit(event)` 是统一入口，分类方法只是补默认值的薄封装。桥接层统一调用 `Emit(event)`。

测试辅助 API：

```lua
CombatFeedback.ResetForTest()
CombatFeedback.SetClockForTest(fn)
CombatFeedback.BuildRenderModel(item, now)
```

`CombatFeedback` 本身不负责旧接口回退，也不依赖 `CombatSystem`，避免循环依赖。

### 12.4 `combat_feedback.lua`

职责：

- 世界坐标转屏幕坐标
- 可见范围裁剪
- 按绘制模型调用 NanoVG
- 描边、阴影、主数字、来源标签和角标绘制
- 接管或由 `basic_attacks.lua` 委托当前 `RenderFloatingTexts` 渲染入口

不包含：

- 聚合判断
- 标签冷却
- 数字格式化
- 伤害分类
- 结算逻辑

### 12.5 `CombatFeedbackBridge.lua`

职责：

- 读取 `COMBAT_FEEDBACK_V2`
- 保证新旧表现互斥
- 启用时调用 `CombatFeedback.Emit(event)`
- 关闭时调用传入的旧表现回退数据
- 将 `EmitCombatFeedback` 注入 `CombatSystem`

建议沿用现有子模块注入模式：

```lua
require("systems.combat.CombatFeedbackBridge")(CombatSystem)
```

桥接模块通过传入的 `CombatSystem` 引用调用 `AddFloatingText`。`CombatFeedback` 本身不反向 require `CombatSystem`，避免循环依赖。

### 12.6 与 `CombatSystem` 的关系

`CombatSystem` 只做轻量接入：

- `Init` 时清空新系统
- `Update` 时调用新系统更新
- 旧 `floatingTexts` 继续保留给未迁移提示
- 通过一行子模块注入获得新旧互斥的表现桥接入口

建议桥接 API：

```lua
CombatSystem.EmitCombatFeedback(event, legacyFallback)
```

职责：

```text
COMBAT_FEEDBACK_V2 = true
  -> 只调用 CombatFeedback.Emit(event)

COMBAT_FEEDBACK_V2 = false
  -> 只调用 AddFloatingText 使用 legacyFallback
```

临时回退数据结构：

```lua
{
    text = oldText,
    color = oldColor,
    lifetime = oldLifetime,
    baseScale = oldBaseScale,
}
```

迁移期间允许保留这份旧表现参数，以确保关闭开关后能恢复原表现。Phase 5 完成后删除已迁移路径的 `legacyFallback`。

不能把全部新逻辑继续写进 `CombatSystem.lua`，原因：

- 当前约 1079 行，预算为 1090 行
- 聚合和布局属于独立表现职责
- 独立模块更容易纯 Lua 测试

### 12.7 与 `HitResolver` 的关系

第一版不修改 `HitResolver.Hit` 的返回形式。

调用方式：

```lua
local damage, isCrit, isTianzhu =
    HitResolver.Hit(player, monster, rawDamage, opts)

local criticality = isTianzhu and "tianzhu"
    or (isCrit and "crit" or "normal")

CombatSystem.EmitCombatFeedback({
    kind = "damage",
    value = damage,
    source = player,
    target = monster,
    sourceType = "skill",
    sourceId = skill.id,
    sourceName = skill.name,
    damageTag = "skill",
    criticality = criticality,
    castId = castId,
}, {
    -- 迁移期旧表现回退；Phase 5 删除
    text = oldDamageText,
    color = oldDamageColor,
    lifetime = oldLifetime,
    baseScale = oldBaseScale,
})
```

不建议让 `HitResolver` 自动发跳字，原因：

- `HitResolver` 不知道合适的技能短名
- `HitResolver` 不知道 `castId`
- 部分伤害明确使用 `noEvent`
- 自动表现容易与现有调用方重复
- 表现逻辑会污染结算层

### 12.8 兼容层

旧接口继续存在：

```lua
CombatSystem.AddFloatingText(x, y, text, color, lifetime, baseScale)
```

兼容策略：

- 未迁移世界提示继续进入旧队列
- 已迁移战斗路径不再由调用点直接调用旧入口
- 功能开关关闭时，只允许统一桥接层调用旧入口
- 渲染阶段先绘制战斗反馈，再绘制必要的旧世界提示
- 旧接口不自动猜测字符串是不是伤害

禁止通过正则解析 `"暴击 12345"` 来生成新事件。字符串反向解析不可靠。

## 18. 性能约束

### 18.1 队列约束

- 全局活跃战斗项不超过 36
- 每目标不超过 5
- 所有缓存必须有清理条件
- 切换章节或清理战斗状态时必须清空

### 18.2 渲染约束

主数字最多：

```text
4 次描边 + 1 次主文字
```

来源标签最多：

```text
1 次阴影 + 1 次主文字
```

角标最多：

```text
1 次阴影 + 1 次主文字
```

低优先级项目在压力场景中可以省略来源标签和角标，但不能省略主数字。

### 18.3 分配约束

- 不在每帧创建字体
- 不在每帧创建资源
- 更新过程使用原地紧凑数组
- 避免每帧为每个项目创建临时 table
- 格式化文本只在创建或聚合时更新
- 渲染阶段不重复格式化数值

### 18.4 压测目标

建议运行：

```text
持续 30 秒，每秒输入 200 条反馈事件
```

验收：

- 活跃项稳定在上限内
- 内存不持续线性增长
- 标签缓存不持续增长
- 压测停止后队列能够清空
- 同一场景平均 FPS 相比旧表现下降不超过 5%

FPS 指标需要在实际 Maker 运行环境中记录，纯 Lua 测试不能代替。

## 22. 自审记录

### 22.1 范围自审

结论：通过。

- 已明确排除多人副本。
- 不修改 `DungeonHUD`、`DungeonHandler` 和副本协议。
- 不把“以后可能统一副本”写入本次实施阶段。

### 22.2 特殊技能自审

结论：通过。

特殊技能没有被归入普通纯数字，已明确分为：

- 主动单次技能
- 主动多段技能
- AOE 技能
- DOT 技能
- 被动技能
- 装备触发
- 神器触发
- 宠物技能

每类都定义了技能名显示和去重规则。

### 22.3 组合语义自审

初版设想如果使用单一 `hitType`，无法同时表达“技能 + 重击 + 暴击”。

本方案已修正为：

- `sourceType`
- `damageTag`
- `criticality`
- `impact`

四个可组合维度，避免互相覆盖。

### 22.4 结算边界自审

结论：通过。

- 不在表现层计算伤害。
- 不让 `HitResolver` 自动生成表现。
- 使用实际扣血结果作为反馈数值。
- 保留旧事件签名和后处理时机。

这样可以把伤害公式风险降到最低。

### 22.5 重复表现自审

结论：已识别并给出明确修复。

普通攻击当前的直接表现与 `player_attack` 监听存在双写。方案明确规定：

- `AutoAttack` 拥有伤害反馈
- `player_attack` 监听只保留刀光

并将“每次命中正好一条反馈”列入自动测试。

### 22.6 技能名重复自审

初版若同时保留玩家头顶技能名和目标伤害标签，会产生重复信息。

本方案已修正：

- 直接伤害技能不再额外显示独立技能名
- 技能名附着在目标的第一条伤害上
- 非伤害技能继续使用状态通道

### 22.7 聚合语义自审

结论：采用保守策略。

- 暴击默认不聚合
- 天诛不聚合
- 普通与暴击不聚合
- 不同技能不聚合
- DOT 与直接伤害不聚合
- 聚合项显示 `xN`

这样优先保证信息真实性，再逐步调优画面密度。

### 22.8 性能自审

结论：有约束，但仍必须运行态测量。

已加入：

- 全局上限
- 每目标上限
- 原地更新
- 不在每帧格式化
- 不在每帧创建字体
- 标签在压力下可降级
- 200 事件/秒压测

纯 Lua 测试不能证明 NanoVG 最终耗时，因此性能不能只凭单元测试验收。

### 22.9 可测试性自审

结论：通过。

可以自动验证：

- 格式化
- 样式解析
- 标签策略
- 聚合
- 分轨
- 数量上限
- 优先级
- 动画曲线
- 结算值与反馈值一致
- 单次命中只产生一次反馈

不能完全自动判断：

- 字体实际观感
- 地图背景上的对比度
- 高频战斗是否“舒服”
- 动画是否有足够打击感

这些项目已经安排 GM 规范矩阵和运行态人工检查，不会以自动测试替代。

### 22.10 迁移风险自审

结论：不能一次性迁移，分阶段方案合理。

原因：

- 原始匹配达到 296 处、58 个 Lua 文件
- 队列承载大量非战斗提示
- 旧测试依赖字符串
- `CombatSystem.lua` 接近文件预算

因此保留旧接口、按战斗路径迁移是必要条件。

### 22.11 回退能力自审

初版只提出功能开关，但没有规定统一桥接位置，仍可能在调用点出现：

```text
新系统写一次 + 旧接口写一次
```

本方案已补充 `CombatSystem.EmitCombatFeedback(event, legacyFallback)`：

- 使用现有功能开关体系
- 新旧路径互斥
- 开关只在桥接层判断
- 关闭开关不改变结算
- 未迁移提示继续使用旧入口

需要在 Phase 5 明确移除临时双路径，避免长期维护成本。

### 22.12 标识稳定性自审

初版提出 `castId` 和 `effectId`，但没有规定生成方式，可能导致：

- 同一次多段技能无法正确去重
- 不同技能被错误聚合
- 测试结果依赖随机数

本方案已补充：

- `castId` 使用单调递增序号
- 同一次施法共享 `castId`
- `effectId` 使用稳定配置 ID
- `targetKey` 只用于本地表现

结论：通过。

### 22.13 文件预算自审

检查发现：

- `CombatSystem.lua` 实施后物理行数为 1085
- `test_file_budget.lua` 预算为 1090
- 直接在主文件实现完整桥接函数会有较高概率触发文件预算门禁

本方案已修正为：

- 新增 `CombatFeedbackBridge.lua`
- 沿用 `require(...)(CombatSystem)` 子模块注入模式
- `CombatSystem.lua` 只增加最小加载和生命周期接入

结论：通过。专项静态守卫已验证文件不超过 1090 行。

### 22.14 最终自审结论

本方案已经完成首版实施，仍必须坚持以下四条红线：

1. 不修改伤害公式和副本逻辑。
2. 不让新旧系统为同一次命中双写。
3. 不把聚合用于暴击、天诛或不同来源伤害。
4. 未完成 Maker 运行态视觉检查前，不宣称视觉优化已经完全验证。

## 0. 本工单的优先级与三波结构（先读）

用户明确的黑市防范优先级：

1. **第一优先：防止利用存档/回档重复刷资源**（经济安全红线）
2. **第二优先：不影响存档、服务器稳定**
3. **第三优先：交易失败补偿正确性**

据此把黑市修复分三波，**第一波立即可执行**，第二/三波是规格与排期：

```
第一波（防刷 + 稳定，本工单主体，现在做）
  W1 WriteSaveBack 接 slot 锁      ← ①防刷(并发型) + ②存档稳定，双料地基
  W2 WAL 补发补打 bmNoResell        ← ①防刷（堵洗白），廉价
  W3 Exchange 补内存锁 + TTL        ← ①防刷（并发双花）
  W4 buy/sell 内存锁加 TTL          ← ②服务器稳定（防卡死）
  W5 买入背包溢出预检               ← ②存档完整（防吞货）
        │
第二波（防刷根治，需前置，规格待排期）
  R1 黑市交易幂等台账               ← ①防刷根治（回档无限卖 + 买入 WAL 双发）
     依赖：B1 保存 CAS + D1 FeatureFlags 云端下发
        │
第三波（补偿正确性，用户第三优先级，延后）
  C1 WAL_KEY/WAL_BP 先用后删
  C2 HandleQuery 停止消费 WAL
  C3 WAL 补偿告知客户端
```

### 0.1 两个必须讲清的判断（决定为什么这样排）

**判断一：防刷根治绕不开存档稳定。** "卖了→回档→再卖"的无限刷，前提是发生回档；回档正是存档被并发覆盖/回退旧备份造成的。所以①和②在黑市这里耦合——**W1（接锁，消除并发覆盖）既是防刷也是稳定**，是第一波第一个做的。纯应用层防刷（如加交易次数限制）挡不住回档刷，因为回档把一切计数打回。真正根治必须把"物品扣减"和"仙石到账"绑到不会被回档的锚点（交易票据 / save 版本号）= R1。

**判断二：WAL_BP 双发防刷不能做廉价的第一波项（收回上一版的 W4b）。** 双发的机理是"买入写档失败→仍回 ok→客户端本地加物并存档→登录时 WAL_BP 又补一份"。一个"补发前查存档已有就跳过"的朴素幂等**不安全**——会把"玩家本来就持有该物品"误判成"已补偿"而跳过合法补偿。正确幂等必须给每笔交易一个**唯一票据**，同时打进 WAL 条目和存档"已应用集"，靠票据判重——这与 R1 交易台账是同一套机制。**故双发防刷归入 R1，不在第一波单列。**

# 第一波 · 防刷 + 稳定（立即执行）

## W1 · WriteSaveBack 接入 slot 锁（① 防刷 + ② 稳定，地基）

**目标**：`WriteSaveBack`（`BlackMerchantHandler.lua:132-154`）是全服务端唯一不持 slot 锁的写档出口，与客户端全量保存（`SaveWriteService.lua:87` 检查 `IsSlotLocked`）双向互踩——客户端快照可把黑市刚扣的物品复活（**并发型刷**），黑市写档也可吞掉客户端保存的其他进度（**存档不稳**）。给它加锁互斥。**调用方（Buy/Sell/Exchange/WAL）零改动**——这是集中在 WriteSaveBack 内部改的最大好处。

**定位锚点**：`local function WriteSaveBack(userId, slot, saveData, desc, onOk, onError)`（:132）；slot 锁 API 见 `ServerSession.lua`：`Session.AcquireSlotLock`(:95)、`Session.ReleaseSlotLock`(:104)、`SLOT_LOCK_TTL=30`(:81)。

**当前代码**（:132-154）：
```lua
local function WriteSaveBack(userId, slot, saveData, desc, onOk, onError)
    local saveKey = "save_" .. slot
    saveData.timestamp = os.time()
    SaveBackupService.BeforeOverwrite(userId, slot, desc, function()
        serverCloud:BatchSet(userId)
            :Set(saveKey, saveData)
            :Save(desc, {
                ok = function()
                    print("[BlackMerchant] SaveBack OK: " .. desc)
                    if onOk then onOk() end
                end,
                error = function(code, reason)
                    print("[BlackMerchant] SaveBack FAILED: " .. desc
                        .. " code=" .. tostring(code) .. " " .. tostring(reason))
                    if onError then onError() end
                end,
            })
    end, function(code, reason)
        print("[BlackMerchant] SaveBack BACKUP FAILED: " .. desc
            .. " code=" .. tostring(code) .. " " .. tostring(reason))
        if onError then onError() end
    end)
end
```

**改造规格**（按此结构重写函数体，不是简单 find/replace）：

1. **入口取锁**：函数体第一步 `Session.AcquireSlotLock(userId, slot)`。
   - 成功 → 进入写档流程；
   - 失败 → 短退避重试（复用现有延迟机制，200ms×3）；耗尽 → `print("[BM-LOCK-BUSY] userId=" .. tostring(userId) .. " slot=" .. tostring(slot))` 后调 `onError()`（**绝不无锁硬写**）。
2. **只释放一次的本地闭包**，四个出口都经它：
   ```lua
   local released = false
   local function releaseOnce()
       if not released then released = true; Session.ReleaseSlotLock(userId, slot) end
   end
   ```
   出口清单（每个先 `releaseOnce()` 再调回调）：
   - `:Save` 的 `ok`（原 :139-142）→ releaseOnce() → `if onOk then onOk() end`
   - `:Save` 的 `error`（原 :143-147）→ releaseOnce() → `if onError then onError() end`
   - `BeforeOverwrite` 的 backup 失败回调（原 :149-153）→ releaseOnce() → `if onError then onError() end`
   - 取锁失败分支 → 无需 releaseOnce（未持锁），直接 onError

**风险与前置**：加锁本身纯互斥、不改数据、不动协议。风险全在"某条异步链某个出口忘释放锁"→ 硬约束 4 要求先画 Buy/Sell/Exchange/WAL 四条链状态图。**参考范式**：`HandleSell` 的 12 出口成对释放 `sellActive_+itemSellLock_`（:582~842）是本项目"多出口成对释放"的严谨标杆。

**注意**：slot 锁与各 handler 的业务锁（buyActive_/sellActive_）是两层，互不替代，W1 只加 slot 锁这层。

**验收**：四个出口都经 releaseOnce()；取锁失败不写档；review 无出口遗漏；LSP 0 error。

# 第二波 · 防刷根治（需前置，规格待排期，不在本次施工）

## R1 · 黑市交易幂等台账（① 防刷根治）

**根治的两类刷**：
- **回档无限卖**：卖出→整档回档→物品在 save 域复活、仙石在 money 域已入账→再卖。
- **买入 WAL 双发**：买入写档失败→客户端本地加物并存档→登录 WAL_BP 再补一份。

**为什么第一波做不了**：两者都需要"这笔交易是否已结算"的可靠判据。仙石/库存在 money 域（永不回档）、物品在 save 域（可回档），跨域且无票据 → 只能靠交易票据 + save 版本号绑定。朴素的"背包已有就跳过"不安全（见 §0.1 判断二）。

**设计（依赖 B1 保存 CAS 的 rev 字段 + D1 FeatureFlags 云端下发）**：
```
前提：B1 落地后 save_{slot} 带单调 rev。
卖出：读 save(rev=R) → 扣物 → WriteSaveBack(rev R→R+1, CAS 校验) 成功 → 才 BatchCommit 加仙石，
      仙石入账绑定 saveRev=R+1。回档使 save 打回 <R+1 → 再卖时 rev 与已结算流水不符 → 拒绝。
买入：BatchCommit 扣仙石+库存(带 saveRev) → WriteSaveBack 写物品(CAS)；
      WAL_BP 补发带 saveRev + 唯一票据，登录补发前校验票据是否已在 save 的"已应用集"，已应用则不重发。
```
**冲突策略（决策点）**：黑市交易小额高频、天然可重试，推荐"rev 不匹配则本笔失败回包'请重试'"，不做复杂 merge——比主链整档 merge 简单，可先行验证。
**上线**：log-only 打 `[BM-CAS-REJECT]` ≥2 周量化影响面 → 正常玩家几乎不触发 → 再启用拒绝。
**备选（若想早于 B1 部分缓解买入双发）**：改买入写档失败路径——失败时**不让客户端乐观本地加物**（服务端回"物品将于重登补发"，客户端不 local-add），消除客户端副本。属行为+客户端改动，中风险，需单独评审。

## W1 详细：WriteSaveBack 接入 slot 锁

### 持锁状态图

```
tryAcquire(0)
  ├─ AcquireSlotLock 成功 → doWrite()
  │    ├─ BeforeOverwrite ok → BatchSet:Save
  │    │    ├─ :Save ok    → releaseOnce() → onOk()
  │    │    └─ :Save error → releaseOnce() → onError()
  │    └─ BeforeOverwrite error → releaseOnce() → onError()
  ├─ 失败 attempt<3 → 延迟200ms → tryAcquire(attempt+1)
  └─ 失败 attempt>=3 → [BM-LOCK-BUSY] → onError()（未持锁，无需释放）
```

**四出口核对**：
1. `:Save` ok → releaseOnce() ✅
2. `:Save` error → releaseOnce() ✅
3. BeforeOverwrite backup 失败 → releaseOnce() ✅
4. 取锁失败(3次) → 未持锁，直接 onError() ✅

### 额外收益
- 集成了 T5 对账日志（`[ROLLBACK-WRITE] BM`）在 BeforeOverwrite ok 回调中

### 释放覆盖

| 出口位置 | 释放方式 |
|---------|---------|
| userId=nil（:903 之前，未置位） | 跳过 |
| invalid request（:919） | 显式 nil ✅ |

**设计决策**：深层异步回调（BatchCommit ok/error → WriteSaveBack ok/error → 嵌套）的出口太多且太深，逐个加释放的漏错概率高于 TTL 不释放的 15s 短暂占用。选择 TTL 兜底。

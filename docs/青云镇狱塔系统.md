## 1. 概述

中洲城中的青云塔保留原有装饰表现，同时新增一个可交互入口，扩展为全新玩法内容"青云镇狱塔"。

镇狱塔以现有青云试炼为代码基底，但职责独立：

- 副本尺寸一致
- 出入逻辑一致
- 覆盖层副本逻辑一致
- 无日常奖励
- 无小怪，仅有 BOSS
- 每 5 层解锁 1 个新 BOSS 进入候选池
- 每层实际上阵 BOSS 数量在 1 到 4 之间逐层变化，第 5 层塔柱替换 1 个 BOSS 位
- 每 5 层通关提供角色级永久数值加成
- 拥有独立排行榜
- 整体开放节奏后移到金丹期
- 怪物境界永远比玩家解锁境界高 2 个小境界
- 所有 BOSS 数值档位不低于皇级 BOSS
- 镇狱塔内怪物统一挂载"镇狱BUFF"

## 3. 交互与副本流程

### 3.1 入口

- 中洲城原 `trial_tower` 装饰继续保留。
- 在塔前新增一个可交互对象，`interactType = "prison_tower"`。
- 文案统一为"青云镇狱塔"。
- 位置建议：青云区域 `(42, 51)` 附近（青云塔正前方，与 `qingyun_master` 相邻）。

### 3.2 进入流程

- 复用青云试炼现有覆盖层副本模式。
- 进入时备份地图、怪物、NPC、掉落、镜头、玩家位置。
- 房间底部生成退出传送阵。
- 玩家出生点固定为"传送阵正上方 1 格"。

### 3.3 通关与退出

- 击杀本层所有战场 BOSS 后通关。
- 镇狱塔柱不计入通关条件。
- 玩家可继续下一层。
- 中途离开视为失败，本层进度不保留。
- `highestFloor` 仅在成功通关后更新。

## 4. 层数与境界推进

镇狱塔难度高于试炼塔，但开放节奏整体后移。

新的硬约束是：

- 玩家达到 `金丹初期`
- 玩家等级不低于 `40`
- 镇狱塔怪物永远比当前挑战所需玩家境界高 `2` 个小境界
- 所有 BOSS 实际数值档位不低于 `emperor_boss`
- 每个小境界只解锁 `10` 层
- 有多少个 BOSS，就做多少关卡段，不额外补后续空层

当前 BOSS 池共有 `30` 个，因此总层数直接定为 `30 x 5 = 150` 层。

### 4.1 基础公式

```lua
segmentIndex = math.ceil(floor / 10)            -- 1..15
localFloor = ((floor - 1) % 10) + 1             -- 1..10
requiredRealmOrder = 6 + segmentIndex           -- 7..21
monsterRealmOrder = requiredRealmOrder + 2      -- 9..23
bossUnlockCount = math.min(30, math.ceil(floor / 5))
bossCountThisFloor = math.min(((floor - 1) % 5) + 1, 4)  -- 1/2/3/4/4
```

### 4.2 怪物等级公式

镇狱塔怪物等级不取"玩家解锁境界"，而取"怪物目标境界"的 `requiredLevel ~ maxLevel` 区间，按 10 层线性插值：

```lua
monsterLevel = monsterRealm.requiredLevel
    + math.floor((monsterRealm.maxLevel - monsterRealm.requiredLevel) * (localFloor - 1) / 9)
```

说明：

- 境界判定以 `GameConfig.REALMS[realmId].order` 为准。
- 玩家的挑战解锁境界与怪物境界不是同一张表。
- 入口最低玩家境界是 `jindan_1`，但最低怪物境界直接从 `jindan_3` 开始。

### 4.3 BOSS 数值档位规则

镇狱塔中的 BOSS 一律使用"不低于皇级 BOSS"的数值档位：

```lua
effectiveCategory = originalCategory
if categoryOrder[originalCategory] < categoryOrder.emperor_boss then
    effectiveCategory = "emperor_boss"
end
```

当前代码基线中，`MonsterData.CategoryMultipliers`（`scripts/config/MonsterData.lua:51-79`）的关键倍率如下：

| 档位 | HP | ATK | DEF | SPEED |
|------|----|-----|-----|-------|
| `boss` | 5.0 | 1.3 | 1.3 | 1.3 |
| `king_boss` | 10.0 | 1.5 | 1.5 | 1.5 |
| `emperor_boss` | 20.0 | 1.8 | 1.8 | 1.6 |

镇狱塔执行规则：

- 原始为 `boss` 的，拉升到 `emperor_boss`
- 原始为 `king_boss` 的，拉升到 `emperor_boss`
- 原始为 `emperor_boss` 的，保持不变
- 若未来出现高于 `emperor_boss` 的更高档位，则保持原档位，不下压

## 5. 房间规则

### 5.1 基础规则

- 每层仅刷新 BOSS，不刷普通怪和精英怪。
- 每层实际上阵 `1-4` 只 BOSS。
- 房间中央固定存在 1 根镇狱塔柱。
- 退出传送阵固定在房间底部中央。
- 玩家出生在传送阵上方 1 格。
- BOSS 出生点使用固定位置表，不做随机。

### 5.2 固定出生点原则

- 优先外圈，后补中圈，最后补内圈。
- 避开底部中央传送阵通道，给玩家保留最小起步空间。
- 镇狱塔柱占据中心点，不参与出生点竞争。
- 同一层已解锁 BOSS 按顺序填入固定位置表。

### 5.3 固定出生点位置表

基于 10x10 竞技场（中心点 `(6,6)`），BOSS 固定出生点如下：

```lua
PrisonTowerConfig.BOSS_SPAWN_POSITIONS = {
    { x = 3.0, y = 3.0 },   -- 位置1：左上外圈
    { x = 9.0, y = 3.0 },   -- 位置2：右上外圈
    { x = 3.0, y = 9.0 },   -- 位置3：左下外圈（避开底部中央传送阵）
    { x = 9.0, y = 9.0 },   -- 位置4：右下外圈
}
```

说明：

- 第 5 层上阵 4 只 BOSS 时，4 个位置全部填满。
- 中心点 `(6,6)` 始终留给镇狱塔柱。
- 底部中央 `(6,10)` 区域是传送阵和玩家出生区，不放 BOSS。

## 6. 镇狱塔柱

### 6.1 定位

镇狱塔柱是环境压制单位，不是通关目标。

| 项目 | 规则 |
|------|------|
| 是否可击杀 | 否 |
| 是否计入通关 | 否 |
| 是否掉落 | 否 |
| 是否参与排行榜 | 否 |
| 位置 | 房间正中央 |

### 6.2 技能职责

- 普通攻击改为地刺
- 定时释放全屏 AOE

### 6.3 技能参数

```lua
PrisonTowerConfig.PILLAR_SKILLS = {
    groundSpike = {
        interval = 2.0,            -- 地刺触发间隔（秒）
        warningTime = 0.5,         -- 预警持续时间（秒），参考试炼塔 0.3s 适度放宽
        warningRange = 1.0,        -- 预警圆半径（格），参考试炼塔 0.8 适度放大
        warningColor = { 200, 50, 50, 160 },  -- 红色预警，区分试炼塔紫色
        count = 3,                 -- 同时释放地刺数量（随机落点）
    },
    fullScreenAOE = {
        interval = 8.0,            -- AOE 触发间隔（秒）
        warningTime = 1.5,         -- 预警持续时间（秒），给玩家反应窗口
        warningColor = { 255, 200, 0, 120 },  -- 金色预警
        damagePct = 0.6,           -- 伤害为基准 BOSS 攻击力的 60%
        canDodge = true,           -- 可通过无敌帧/盾值/闪避类技能规避
    },
}
```

### 6.4 数值规则

镇狱塔柱不单独配置伤害与难度，不额外写独立攻击值。

统一规则如下：

- 镇狱塔柱所有伤害都以"本层最新解锁的 BOSS"作为数值标杆。
- 也就是取 `activeBossIds[#activeBossIds]` 对应的 BOSS 作为当前层基准 BOSS。
- 塔柱地刺与全屏 AOE 的攻击强度、等级压制、难度感受，全部与该基准 BOSS 保持一致。
- 塔柱仅单独保存自己的触发频率、预警表现和技能类型，不单独保存攻击倍率。
- 塔柱继承的是"基准 BOSS 在镇狱塔内结算完成后的最终数值"，包含皇级保底与镇狱BUFF。

实现要求：

```lua
referenceBossId = activeBossIds[#activeBossIds]
referenceBoss = MonsterData.Types[referenceBossId]

pillar.damageSource = referenceBossId
pillar.groundSpikeDamageSource = referenceBossId
pillar.aoeDamageSource = referenceBossId
```

结论：

- 地刺难度、攻击与房间内 BOSS 保持一致
- 全屏 AOE 使用基准 BOSS 攻击力 x `damagePct`（默认 60%）
- 不为塔柱地刺单独赋值

## 7. BOSS 解锁与上阵规则

### 7.1 解锁规则

镇狱塔每 `5` 层解锁 `1` 个新 BOSS 进入候选池。

```lua
unlockedBossIds = bossPool[1 .. math.min(30, math.ceil(floor / 5))]
```

### 7.2 每层上阵数量规则

每个 `5` 层段内部，BOSS 上阵数量逐层变化：

| 层内位置 | BOSS上阵数 | 塔柱 | 场上总单位 | 设计意图 |
|----------|-----------|------|-----------|----------|
| 第1层 | 1 | 1 | 2 | 试探层，熟悉新 BOSS |
| 第2层 | 2 | 1 | 3 | 压力递增 |
| 第3层 | 3 | 1 | 4 | 中等强度 |
| 第4层 | 4 | 1 | 5 | 高压力，BOSS 满编 |
| 第5层 | 4 | 1 | 5 | 里程碑层，BOSS 满编 + 解锁新 BOSS |

公式：

```lua
bossCountThisFloor = math.min(((floor - 1) % 5) + 1, 4)  -- 1/2/3/4/4
```

设计说明：

- 原版 v0.3 第 5 层为 5 BOSS + 1 塔柱 = 6 同屏单位，移动端性能压力过大。
- v0.4 将第 5 层 BOSS 数量从 5 降为 4，由塔柱替补该单位位置。
- 场上最大同屏单位数恒定为 5（4 BOSS + 1 塔柱），确保移动端帧率稳定。
- 第 5 层虽然 BOSS 数量与第 4 层相同，但新 BOSS 解锁带来了种类变化，体验不重复。

### 7.3 上阵选择规则

每层不把全部已解锁 BOSS 全部摆上场，而是从已解锁池中按顺序取出 `bossCountThisFloor` 只。

推荐固定规则：

```lua
unlockedBossIds = bossPool[1 .. bossUnlockCount]
bossCountThisFloor = math.min(((floor - 1) % 5) + 1, 4)

-- 倒序优先：越新解锁的 BOSS 优先上阵
rotationPool = reverse(unlockedBossIds)
activeBossIds = takeCycle(rotationPool, bossCountThisFloor)
```

效果：

- 新解锁的 BOSS 一定优先登场
- 早期已解锁种类不足时，允许同一 BOSS 重复上阵
- 中后期虽然候选池越来越大，但单层实际上阵数量始终不超过 `4`

### 7.4 版本终点

- `1-150` 层：每 5 层解锁 1 个新 BOSS
- `146-150` 层：最后一个新增 BOSS 解锁完成，版本内容收束

## 9. 难度模型

镇狱塔整体难度必须高于青云试炼。

组成方式：

- 高于试炼塔的总层数
- 更长境界推进
- BOSS 纯战场
- 每层 BOSS 数量在 1 到 4 之间变化
- 镇狱塔柱持续压场
- 塔内怪物专属增幅强于青云试炼

### 9.1 镇狱BUFF

镇狱塔新增独立专属 BUFF：`镇狱BUFF`。

定位：

- 强度高于青云试炼的 `trial_buff`（试炼为 atkMult=1.5, hpMult=2.0, atkSpeed=1.3, speed=1.3, cdr=0.7）
- 强度高于常规 `狂暴`
- 定位高于现有"入魔"类强化
- 作为镇狱塔内所有怪物的统一最终强化层

建议配置：

```lua
PrisonTowerConfig.ZHENYU_BUFF = {
    hpMult = 2.0,
    atkMult = 2.0,
    defMult = 2.0,
    speedMult = 2.0,
    atkSpeedMult = 2.0,
    cdrMult = 0.5,
}
```

解释：

- 生命翻倍
- 攻击翻倍
- 防御翻倍
- 移速翻倍
- 攻速翻倍
- 冷却效率翻倍，等价于实际技能 CD 减半

生效顺序：

```lua
基础等级属性（MonsterData.CalcBaseStats）
-> 种族修正
-> BOSS类别修正（至少皇级，MonsterData.CategoryMultipliers）
-> 境界修正
-> 镇狱BUFF（PrisonTowerConfig.ZHENYU_BUFF）
```

### 9.2 镇狱BUFF覆盖范围

镇狱塔内以下单位全部吃 `镇狱BUFF`：

- 场上 BOSS
- 镇狱塔柱
- 其他未来新增的塔内怪物单位

数值原则：

- 镇狱塔怪物增幅单独配置
- 强度整体高于 `TrialTowerConfig.TRIAL_BUFF`
- 具体倍率在 `PrisonTowerConfig` 中配置，不写死在系统逻辑中
- 镇狱BUFF结算在 BOSS 档位与境界修正之后

### 9.3 最终数值示例（校验参考）

以第 150 层为例，怪物等级 140（`dujie_1` 境界区间上限），BOSS 为 `dragon_sand`（原 `emperor_boss`）：

```
基础属性（Lv.140）:
  HP  = 50 + 140*50 + (140-55)^2*5 = 7050 + 36125 = 43175
  ATK = 5 + 140*2.5 = 355
  DEF = 2 + 140*2 = 282

emperor_boss 倍率:
  HP  = 43175 * 20.0 = 863500
  ATK = 355 * 1.8 = 639
  DEF = 282 * 1.8 = 507.6

镇狱BUFF (x2):
  HP  = 863500 * 2.0 = 1727000
  ATK = 639 * 2.0 = 1278
  DEF = 507.6 * 2.0 = 1015

最终: HP ≈ 172.7万, ATK ≈ 1278, DEF ≈ 1015
同屏 4 只 = 总 HP ≈ 690.8万
```

## 10. 永久加成

### 10.1 规则

- 每 5 层通关提供 1 次永久数值成长
- 永久成长为角色级，不是账号级
- 不做手动领取，按 `highestFloor` 自动反推
- 只加战斗平值，不加攻速、CD、暴击、重击这类高敏感属性
- 设计目标是"稳定抬底"，不是替代装备与境界成长

### 10.2 设计原则

镇狱塔的永久加成建议直接走"平值四维"：

- 攻击 `atk`
- 防御 `def`
- 生命上限 `maxHp`
- 生命回复 `hpRegen`

不建议首版直接加入以下永久成长：

- 攻速
- 技能冷却
- 暴击率
- 暴击伤害
- 重击
- 击杀回血

原因：

- 这些属性对构筑波动太大，容易把职业和装备权重冲散
- 当前 `Player` 的平值聚合链路已经很成熟（`seaPillarAtk` 先例），接入成本最低
- 镇狱塔本身已经通过 `镇狱BUFF` 把怪物端速度和 CD 拉满，玩家端永久成长更适合走稳态平值

### 10.3 推荐数值方案

镇狱塔总计 `150` 层，对应 `30` 个 5 层里程碑。

建议按三个阶段递增：

| 里程碑区间 | 对应楼层 | 每次加成 |
|------------|----------|----------|
| 前期 | 5-50 | `atk +2`、`def +2`、`maxHp +20`、`hpRegen +0.2` |
| 中期 | 55-100 | `atk +4`、`def +3`、`maxHp +40`、`hpRegen +0.3` |
| 后期 | 105-150 | `atk +6`、`def +4`、`maxHp +60`、`hpRegen +0.5` |

总计收益：

| 属性 | 总加成 |
|------|--------|
| 攻击 | `+120` |
| 防御 | `+90` |
| 生命上限 | `+1200` |
| 生命回复 | `+10.0` |

这个总量的特点：

- 明显高于海神柱（`seaPillarAtk=50, seaPillarDef=30, seaPillarMaxHp=200, seaPillarHpRegen=5`），符合镇狱塔终局内容定位
- 仍然是"平值成长"，不会把角色伤害模型打穿
- 后期加成更重，能体现越往后收益越高

### 10.5 配置方式

建议独立配置为：

```lua
PrisonTowerConfig.MILESTONE_INTERVAL = 5
PrisonTowerConfig.MILESTONE_BONUS = {
    [5]   = { atk = 2, def = 2, maxHp = 20, hpRegen = 0.2 },
    [10]  = { atk = 2, def = 2, maxHp = 20, hpRegen = 0.2 },
    ...
    [150] = { atk = 6, def = 4, maxHp = 60, hpRegen = 0.5 },
}
```

也可以进一步压成函数：

```lua
function PrisonTowerConfig.GetMilestoneBonus(floor)
    if floor <= 50 then
        return { atk = 2, def = 2, maxHp = 20, hpRegen = 0.2 }
    elseif floor <= 100 then
        return { atk = 4, def = 3, maxHp = 40, hpRegen = 0.3 }
    else
        return { atk = 6, def = 4, maxHp = 60, hpRegen = 0.5 }
    end
end
```

如果后续需要热调，我更建议直接写完整表，而不是公式。

### 10.6 存储方式

- 进度存档：`saveData.prisonTower`
- 聚合属性字段：参考海神柱系统，最终汇总到玩家总属性
- 推荐支持字段：
  - `prisonTowerAtk`
  - `prisonTowerDef`
  - `prisonTowerMaxHp`
  - `prisonTowerHpRegen`

建议结构：

```lua
saveData.prisonTower = {
    highestFloor = 0,
}
```

说明：

- 不需要单独存"已领奖励列表"
- 所有永久属性都可以由 `highestFloor` 反推
- 这样最稳，不会出现重复领取或补发遗漏

### 10.7 计算方式

建议完全照海神柱系统做总和聚合：

```lua
function PrisonTowerSystem.GetAllBonuses()
    local bonuses = { atk = 0, def = 0, maxHp = 0, hpRegen = 0 }
    local highestFloor = PrisonTowerSystem.highestFloor or 0

    for floor, bonus in pairs(PrisonTowerConfig.MILESTONE_BONUS) do
        if highestFloor >= floor then
            bonuses.atk = bonuses.atk + (bonus.atk or 0)
            bonuses.def = bonuses.def + (bonus.def or 0)
            bonuses.maxHp = bonuses.maxHp + (bonus.maxHp or 0)
            bonuses.hpRegen = bonuses.hpRegen + (bonus.hpRegen or 0)
        end
    end

    return bonuses
end
```

## 11. 排行榜

### 11.1 排行榜键

- 主键：`prison_floor`（Int 类型，存最高层数）
- 附加信息：通过 `serverCloud:BatchSet` 写入分字段

### 11.2 排行榜附加字段编码

参照试炼塔 `trial_floor` 的 `BatchSet` 模式（`RankHandler.lua:RebuildRankOnLogin`）：

```lua
serverCloud:BatchSet(userId, {
    prison_floor_level = coreData.level,
    prison_floor_realm = coreData.realm,
    prison_floor_class = coreData.classId,
    prison_floor_name  = coreData.charName,
})
```

不使用自定义编码格式，直接拆分为独立字段，与试炼塔保持一致。

### 11.3 排行规则

- 每个账号取所有角色中的最高镇狱塔成绩
- 先按最高层数排
- 同层时按达成时间先后排
- UI 作为独立"镇狱榜"页签展示

### 11.4 展示字段

- 角色名
- TapTap 昵称
- 境界
- 等级
- 最高层
- 职业图标

### 11.5 反作弊验证

在 `RankHandler.ComputeRankFromSaveData` 中新增镇狱塔验证：

```lua
-- 镇狱塔排行验证
local prisonFloor = coreData.prisonTower and coreData.prisonTower.highestFloor or 0
if prisonFloor > 0 then
    -- 验证楼层范围
    if prisonFloor > 150 then return nil, "prison floor exceeds max" end
    -- 验证境界门槛（金丹初期 order=7）
    if realmOrder < 7 then return nil, "prison requires jindan_1" end
    -- 验证等级门槛
    if level < 40 then return nil, "prison requires level 40" end
end
```

## 15. 实现约束总结

以下内容视为本方案的硬约束：

1. 镇狱塔入口开放条件为金丹初期（order=7），且等级不低于 40。
2. 镇狱塔副本尺寸、出入逻辑、继续挑战逻辑与青云试炼一致。
3. 怪物起始境界从金丹后期开始，玩家永远越 2 个小境界打怪。
4. 镇狱塔没有日常奖励。
5. 镇狱塔没有小怪，只有 BOSS。
6. 每 5 层解锁 1 个新 BOSS 进入候选池，每层实际上阵数量按 `1/2/3/4/4` 逐层变化。
7. 所有 BOSS 实际数值档位不低于 `emperor_boss`，低于皇级的统一上抬到皇级。
8. 镇狱塔内全部怪物统一挂载 `镇狱BUFF`，生命、攻击、防御、移速、攻速翻倍，技能实际 CD 减半。
9. 镇狱塔柱固定在房间中央，不可击杀，不计通关。
10. 塔柱地刺与 AOE 伤害不单独赋值，统一与本层基准 BOSS 保持一致。
11. 玩家出生点固定为退出传送阵上方 1 格。
12. 排行榜独立于试炼榜。
13. 场上最大同屏单位数 = 4 BOSS + 1 塔柱 = 5 单位。
14. 存档版本号递增到 22，迁移函数补充 `prisonTower` 字段。

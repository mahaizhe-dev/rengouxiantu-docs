## 2. 自动回收真实流程链路

### 2.1 调用入口

```
server_main.lua HandleServerUpdate (每帧)
  └─ pcall(BlackMerchantHandler.RecycleTick, dt)
```

### 2.2 RecycleTick 完整决策流程 (L1323-1410)

```
RecycleTick(dt)
  │
  ├─ Guard 1: RECYCLE_ENABLED == false? → return (开关关闭)
  ├─ Guard 2: recycleRunning_ == true?  → return (防重入)
  ├─ Guard 3: lastRecycleDate_ == today? → return (内存快速路径)
  │
  ├─ 内存占坑: lastRecycleDate_ = today
  │
  ├─ serverCloud.money:Get(SYSTEM_UID)  ← 异步
  │   ├─ error → lastRecycleDate_ = nil (允许重试) → return
  │   └─ ok → cloudDate = moneys["bm_recycle_date"]
  │
  ├─ 计算间隔:
  │   ├─ numericGap = todayNum - cloudDate  (YYYYMMDD 数值差，用于 money:Add)
  │   └─ realGap = RealDayGap(todayNum, cloudDate)  (真实天数差，用于决策)
  │
  ├─ realGap <= 0? → return (云端已是今天，不重复)
  │
  ├─ realGap > MAX_RECYCLE_GAP(2)?
  │   ├─ 异常路径: money:Add(numericGap) 仅推日期
  │   │   ├─ ok → 日期已推进，不执行回收
  │   │   └─ error → lastRecycleDate_ = nil (允许重试)
  │   └─ return
  │
  └─ 正常路径 (1 <= realGap <= 2):
      ├─ money:Add(numericGap) 占坑
      │   ├─ ok → ExecuteRecycle(callback)
      │   └─ error → lastRecycleDate_ = nil (允许重试)
      └─ return
```

### 2.3 ExecuteRecycle 执行流程 (L1212-1318)

```
ExecuteRecycle(callback)
  ├─ recycleRunning_ = true (防重入)
  ├─ InvalidateSystemStockCache()
  ├─ GetSystemStockCached → 读取全部系统库存
  │   ├─ error → recycleRunning_ = false → callback(false)
  │   └─ ok → 遍历 NPC "胤" 的商品列表
  │       ├─ 过滤: stock >= maxStock 的商品
  │       ├─ WeightedRecycleAmount() → 0/1/2 件 (25%/50%/25%)
  │       └─ 逐件 BatchCommit(MoneyCost) + TradeLogger.LogPublic
  └─ recycleRunning_ = false → callback(ok, summary)
```

### 2.4 状态存储位置

| 状态 | 存储位置 | 生命周期 | 说明 |
|------|---------|---------|------|
| `lastRecycleDate_` | 内存 (module local) | 进程级 | 快速路径防重入，进程重启后丢失 |
| `bm_recycle_date` | 云端 (serverCloud.money, SYSTEM_UID=0) | 持久化 | 跨进程持久化，防重启后重复回收 |
| `recycleRunning_` | 内存 (module local) | 进程级 | ExecuteRecycle 执行锁 |

### 2.5 双 gap 设计

| gap | 计算方式 | 用途 | 跨月示例 (0531→0601) |
|-----|---------|------|---------------------|
| `numericGap` | `todayNum - cloudDate` | `money:Add` 增量值 | 70 |
| `realGap` | `RealDayGap()` (时间戳) | 决策逻辑 | 1 |

## 3. 问题根因说明

### "首日触发，后续不触发" — 不是代码 bug

经完整代码审计和 4 场景验证，确认 RecycleTick 逻辑正确。**该现象的真实原因**：

1. **P0 事件后 `RECYCLE_ENABLED` 曾被设为 `false`**（见 `万界黑商.md` §12.10.2），回收功能被关闭。
2. **回收设计本身就是"每天最多触发一次"**，三层防护（内存占位 + 云端日期 + 执行锁）确保同一天不重复。
3. **次日过天后，只要开关打开、云端/内存状态正常，回收会正确触发** — 这已被 T13(连续4天)、T14(跨月5天)、T15(跨年5天) 测试证明。

### 防护机制正确性

| 场景 | 内存防护 | 云端防护 | 结果 |
|------|---------|---------|------|
| 同天重复 tick | `lastRecycleDate_ == today` 拦截 | 不到达云端 | 正确跳过 |
| 进程重启同天 | 内存丢失，不拦截 | `realGap <= 0` 拦截 | 正确跳过 |
| 次日首次 tick | 内存不匹配，放行 | `realGap = 1`，正常路径 | 正确回收 |
| 跨月 (0531→0601) | 放行 | `realGap = 1`（非 70） | 正确回收 |
| 跨年 (1231→0101) | 放行 | `realGap = 1`（非 8870） | 正确回收 |
| 大跨度 (>2天) | 放行 | `realGap > 2`，异常路径 | 仅推日期 |
| 网络失败 | `lastRecycleDate_ = nil` | 未到达 | 允许重试 |

## 8. 残余风险

| 风险 | 说明 | 等级 |
|------|------|------|
| 云端并发竞争 | 多个服务端实例同时读取 `bm_recycle_date` 后各自 Add，可能导致日期跳跃。但当前架构为单服务端实例，风险极低 | 低 |
| `RECYCLE_ENABLED` 需人工确认 | 如果生产环境仍为 `false`（P0 事件遗留），回收永远不会触发。当前 `BlackMerchantConfig.lua` 中值为 `true`，但需确认部署版本一致 | 低 |
| `MAX_RECYCLE_GAP` 阈值 | 当前为 2 天，超过 2 天不回收。如果服务器停机超过 2 天后恢复，会跳过中间的回收。这是设计行为（防时间篡改），非 bug | 无 |

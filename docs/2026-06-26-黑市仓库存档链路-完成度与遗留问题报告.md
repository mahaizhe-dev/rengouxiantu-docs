## 1. 原始问题

玩家反馈：使用仓库后黑市出售永久被锁，显示"数据同步中"无法恢复。

## 2. 根因（已确认）

```
仓库操作 → _dirtyWarehouse = true → 黑市出售全局阻断
                                            ↓
                            只有 game_saved 才清除 dirty
                                            ↓
                问题1: EventBus.On("game_saved") 在运行时不触发（引擎模块缓存问题）
                问题2: os.clock() 在 WASM 下返回 0，防抖逻辑永远不满足条件
                问题3: 关闭仓库时 warehouse_changed 事件不到达订阅者（同源问题）
```

## 5. 遗留问题

### 5.1 已确认但未修复

| 问题 | 影响 | 修复方向 |
|------|------|----------|
| **EventBus.On 文件末尾注册在运行时不生效** | game_saved/warehouse_changed 回调从不触发 | 已用防御性直调绕过，但根因未解 |

### 5.2 架构层面的已知局限

| 局限 | 说明 |
|------|------|
| 黑市出售是全局阻断（非物品粒度） | 服务端 WriteSaveBack 写完整快照，无法安全做物品粒度放行 |
| 保存链路单点依赖 serverConn | 无 serverConn = 完全无法保存 = dirty 永远不清 |
| 断线判断仍是 transport 层 | GetServerConnection() 为 nil ≠ 业务不可达 |

## 8. 给下一位执行者的上下文

1. **核心安全不变量**：`_dirtyWarehouse` 只在真实保存成功后才清。任何改动不得违反此规则。
2. **EventBus.On 不可靠**：文件末尾的模块级 EventBus.On 注册在运行时可能不触发。关键回调必须用防御性直调或 SubscribeToEvent 全局注册。
3. **os.clock() 在 WASM 返回 0**：客户端所有计时必须用 `time.elapsedTime`。服务端用 `os.time()`。
4. **心跳已就绪**：5s 间隔，RTT/missed 数据可用。可直接在 NetworkStatus 上构建状态机。
5. **测试依赖 GM 后台**：发布前关 GM_ENABLED，测试通过 TestRegistry 在 GM 后台手动触发。

## T2. 让 pending 超时清扫在选角/加载阶段真正运行（P0，最小 diff）

**目标**：`SaveSystemNet.Tick()`（20s pending 超时清扫）当前只被 `SaveSystem.UpdateCritical` 驱动，而 `main.lua` 的 HandleUpdate 在 `gameStarted_=false` 或 `GameState.paused=true` 时提前 return，UpdateCritical 不执行——FetchSlots/Load 的 pending 恰恰只存在于这两个阶段。把清扫驱动挪到门之前。

**文件**：`scripts/main.lua`（仅此一个文件）

**定位锚点**：`CloudStorage.Tick()`（约 :1410-1417，HandleUpdate 内、`if not gameStarted_` 之前）

```lua
-- 改前
    -- P1-SAVE-2: CloudStorage 回调超时清理（每 5 秒一次）
    cloudStorageTickTimer_ = (cloudStorageTickTimer_ or 0) + dt
    if cloudStorageTickTimer_ >= 5.0 then
        cloudStorageTickTimer_ = 0
        CloudStorage.Tick()
    end

-- 改后
    -- P1-SAVE-2: CloudStorage 回调超时清理（每 5 秒一次）
    cloudStorageTickTimer_ = (cloudStorageTickTimer_ or 0) + dt
    if cloudStorageTickTimer_ >= 5.0 then
        cloudStorageTickTimer_ = 0
        CloudStorage.Tick()
        -- P0-fix2: pending 超时清扫必须在 gameStarted_/paused 门之前驱动。
        -- FetchSlots/Load 的 pending 只存在于选角与加载阶段，
        -- 此时 UpdateCritical（门之后）不会执行，不能依赖它驱动清扫。
        if CloudStorage.IsNetworkMode() then
            pcall(function() require("network.SaveSystemNet").Tick() end)
        end
    end
```

**注意**：
- 5 秒粒度对 20 秒的 `PENDING_TIMEOUT` 足够，不要为此新增计时器。
- `SaveSystem.UpdateCritical` 顶部已有的那次 `SaveSystemNet.Tick()` 调用**保留不动**（Tick 幂等，重复调用无害）。
- 不要改 `SaveSystemNet.lua` 本身。

**验收**：静态确认新增调用位于 `if not gameStarted_ then` 之前；LSP 0 error。

## T3. 修复青云试炼领取的标志泄漏与静默无回包（P0）

**目标**：`TrialHandler.HandleTrialClaimDaily` 有两条不释放 `trialClaimingActive_` 且不回包的路径，导致部分玩家"点领取完全没反应"且永久失效。三处改动全部在 `scripts/network/TrialHandler.lua`。

**改动 1 — skip 路径加标识日志 + 回包 + 标志 TTL**

定位锚点：`if trialClaimingActive_[connKey] then`（约 :34-38）

```lua
-- 改前
    -- ① 内存锁：防止同一用户并发请求
    if trialClaimingActive_[connKey] then
        print("[Server] TrialClaimDaily: already processing, skip")
        return
    end
    trialClaimingActive_[connKey] = true

-- 改后
    -- ① 内存锁：防止同一用户并发请求（带 15s TTL，防回调挂起导致永久卡死）
    local activeAt = trialClaimingActive_[connKey]
    if activeAt then
        local age = os.time() - (type(activeAt) == "number" and activeAt or 0)
        if age <= 15 then
            print("[Server] TrialClaimDaily: already processing, skip connKey=" .. connKey
                .. " userId=" .. tostring(userId) .. " age=" .. age .. "s")
            local data = VariantMap()
            data["ok"] = Variant(false)
            data["msg"] = Variant("正在处理中，请稍后再试")
            SafeSend(connection, SaveProtocol.S2C_TrialClaimDailyResult, data)
            return
        end
        print("[Server] TrialClaimDaily: stale claiming flag expired (age=" .. age .. "s), releasing")
    end
    trialClaimingActive_[connKey] = os.time()
```

> 说明：标志值从 `true` 改为 `os.time()`。全工程该表的其他触点只有"置 nil 清理"（server_main.lua 断连清理与重复登录清理），truthy 语义不变，无需改其他文件。执行前 grep `trialClaimingActive_` 确认触点仅此两类。

**改动 2 — `slot == nil` 早退路径：清标志 + 回包**

定位锚点：`connSlots_ not loaded for connKey`（约 :42-46）

```lua
-- 改前
    local slot = connSlots_[connKey]
    if not slot then
        print("[Server] WARNING: connSlots_ not loaded for connKey=" .. connKey .. ", rejecting")
        return
    end

-- 改后
    local slot = connSlots_[connKey]
    if not slot then
        trialClaimingActive_[connKey] = nil  -- P0-fix: 必须清标志，否则该连接永久静默失败
        print("[Server] WARNING: TrialClaimDaily connSlots_ not loaded, connKey=" .. connKey
            .. " userId=" .. tostring(userId) .. ", rejecting")
        local data = VariantMap()
        data["ok"] = Variant(false)
        data["msg"] = Variant("存档未就绪，请稍后再试")
        SafeSend(connection, SaveProtocol.S2C_TrialClaimDailyResult, data)
        return
    end
```

**改动 3（防御性，可选但推荐）** — `ServerGetSlotData` 回调挂起时靠改动 1 的 TTL 兜底，无需额外代码；但确认函数内**所有** `return` 前都成对执行了 `ReleaseSlotLock` + `trialClaimingActive_[connKey] = nil`（基准提交中 :62-63、:86-87、:120-121、:134-135、:156-157、:170-171 已成对，勿动）。

**执行前自检**：
- grep 客户端对 `S2C_TrialClaimDailyResult` 的订阅处理（应在试炼塔相关 UI/系统文件中），确认其 `ok=false` 分支会展示 `msg` 或至少不会崩溃。如果客户端对 `msg` 字段读取不是 pcall/nil 安全，回包字段名保持 `msg`（与本 handler 既有失败回包一致，见 :55、:67），**不要**改成 `message`。

**验收**：函数内每条 `return` 路径要么发生在标志置位之前，要么之前已把 `trialClaimingActive_[connKey]` 置 nil（TTL 路径除外）；每条业务拒绝路径都有 S2C 回包；LSP 0 error。

## T5. 回档写入对账日志（P0，log-only，严禁阻断）

**目标**：量化"旧档覆盖新档"的发生率与来源。**只记日志，不拦截、不修改任何数据**（遵守 rules.md §12 灰度原则：先 log-only 一个版本）。

**文件**：`scripts/network/SaveBackupService.lua` + `scripts/network/SaveWriteService.lua`

**改动 1 — SaveBackupService.BeforeOverwrite 把旧档传给 onReady**

定位锚点：`onReady()`（该文件共两处调用，约 :30 与 :43）

```lua
-- 改前（两处）
                    onReady()
-- 改后（两处）
                    onReady(current)   -- :43 处（备份写成功后）
                    onReady(nil)       -- :30 处（旧档不存在/非 table 时）
```

> 精确对应：`if type(current) ~= "table" then onReady() return end` 这一处改为 `onReady(nil)`；备份写成功回调里的 `onReady()` 改为 `onReady(current)`。参数是**追加**，其他调用方（BlackMerchantHandler、SaveLoadService、RedeemHandler 等）以无参函数接收会自动忽略，**不需要改它们**。

**改动 2 — SaveWriteService.Execute 的 CommitSave 前比较时间戳**

定位锚点：`SaveBackupService.BeforeOverwrite(userId, slot, "SaveGame", CommitSave, function(code, reason)`（约 :353）

```lua
-- 改前
    SaveBackupService.BeforeOverwrite(userId, slot, "SaveGame", CommitSave, function(code, reason)

-- 改后
    SaveBackupService.BeforeOverwrite(userId, slot, "SaveGame", function(oldSave)
        -- [ROLLBACK-WRITE] 对账日志（log-only，绝不阻断写入）
        if oldSave and type(oldSave) == "table"
            and type(oldSave.timestamp) == "number"
            and type(coreData.timestamp) == "number"
            and coreData.timestamp < oldSave.timestamp - 60 then
            print("[ROLLBACK-WRITE] userId=" .. tostring(userId)
                .. " slot=" .. slot
                .. " newTs=" .. coreData.timestamp
                .. " oldTs=" .. oldSave.timestamp
                .. " gap=" .. (oldSave.timestamp - coreData.timestamp) .. "s source=client_snapshot")
        end
        CommitSave()
    end, function(code, reason)
```

**硬性要求**：
- 这段代码**任何分支都必须最终调用 `CommitSave()`**，不允许出现任何提前 return 或条件跳过写入的路径。
- 比较容错：任一 timestamp 缺失/非 number 时不打日志、直接 CommitSave。
- 不要在其他直写路径（BM/Event/Trial）加同样逻辑——那是下一批次的事，本批次只覆盖主保存链。

**验收**：构造性 code review 确认 CommitSave 在所有分支必达；LSP 0 error；lupa save 组无新增红项（特别关注 `r3_service_shell_contract`）。

## 本批次不做（明确排除，防止执行扩散）

| 项 | 原因 |
|---|---|
| TrialHandler 的 `ScoreSet` 接入 BeforeOverwrite | 会延长 slot 锁持有窗口，需先评估锁 TTL 配合，属下一批 |
| timestamp 旧档覆盖**拒绝**（非日志） | 必须先跑 T5 的 log-only 数据确认误报率（rules §12.3 灰度） |
| 客户端快照 CAS / 版本号 | 协议级改动，需单独设计 |
| `sealDemonActive_` / `daoTreeMeditating_` 等同类标志 TTL 统一化 | 与 T3 同模式，但需逐 handler 核对退出路径，单独批次 |
| SLOT_LOCK_TTL 30s→15s、slot_locked 阈值 10→5 | 数值调整需先看 `[LOCK-EXPIRED]` 日志分布 |

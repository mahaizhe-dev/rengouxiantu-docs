# 服务端连接异常止血与修复实施方案

> 目标：不是继续写归因报告，而是以最小风险、最快节奏把“长时间卡在正在连接服务器...”这类事故压下去，并形成可上线方案。  
> 适用范围：`pre-ClientIdentity` 阶段的服务端不可用、启动链脆弱、可选业务模块拖垮整个服务端实例。  
> 执行对象：执行 AI / 开发同学按本文顺序直接实施。

---

## 一、方案目标

本方案只解决一件事：

1. 即使部分可选模块失败，服务端也必须能启动。
2. 即使 `serverCloud` 短时未就绪，服务端也不能在启动期被拖死。
3. 下次再出现事故，必须能在日志里区分：
   - 根本没启动
   - 启动中途失败
   - 已启动但没进入 `ClientIdentity`
   - 已进入 `ClientIdentity` 但后续业务失败

本方案不要求一次性找出唯一根因；要求先把“整服不可用”降级为“个别功能不可用”。

---

## 二、总体策略

采用三层处理：

1. `P0 止血`
   - 先切断会拖垮整服的启动期硬依赖。
   - 允许黑市、仙缘宝箱、兑换码注册降级，但不允许整服起不来。

2. `P1 观测`
   - 补全启动阶段和身份阶段日志。
   - 让下次事故能直接判到 `pre-ClientIdentity`、`post-ClientIdentity`、还是 `serverCloud` 问题。

3. `P2 稳定化`
   - 清理启动链，把业务期模块从冷启动路径移走。
   - 增加发版前检查，避免双入口、`CODE_VERSION`、资源遗漏这类低级错误。

## 三、风险评估

### 变更风险分级

| 任务 | 风险级别 | 主要风险 | 风险说明 | 控制方式 |
|------|---------|---------|---------|---------|
| `RedeemHandler` 加 `serverCloud` 保护 | 低 | 兑换码启动期不注册 | 只影响兑换码初始化，不影响登录与存档主链路 | 判空 + `pcall` + 跳过注册日志 |
| 黑市懒加载 `LootSystem` | 中 | 黑市请求路径遗漏改造 | 若部分调用点仍直接使用旧变量，可能在运行期报空 | 搜索全量调用点 + 黑市功能冒烟 |
| 仙缘宝箱懒加载 `LootSystem` | 中 | 宝箱奖励路径遗漏改造 | 行为与黑市类似，但影响范围更窄 | 搜索全量调用点 + 宝箱功能冒烟 |
| `server_main` 可选模块改为可失败加载 | 中 | 订阅条件分支漏判空 | 可能出现模块加载失败后仍引用其方法 | 只处理可选模块 + 订阅/断开两侧同时判空 |
| 启动日志增强 | 低 | 日志噪音增加 | 不影响业务，只增加日志量 | 统一前缀 + 仅关键节点打印 |
| 身份阶段日志增强 | 低 | 无 | 只增加观测信息，不改流程 | 固定日志格式，便于 grep |

### 上线风险判断

本方案整体属于 **低到中风险热修**，原因如下：

1. 不改存档格式。
2. 不改网络协议字段。
3. 不改客户端状态机。
4. 只改服务端启动期保护和可选模块加载方式。

真正需要防的不是“改完起不来”，而是：

1. 黑市或仙缘宝箱某个运行期调用点漏改。
2. 可选模块判空不完整，导致加载失败后在别处继续被调用。
3. 启动日志补了，但没有形成固定 grep 口径，导致下次仍难排查。

### 风险接受原则

本次允许的副作用：

1. 兑换码注册在某次冷启动时被跳过。
2. 黑市或仙缘宝箱在异常条件下临时不可用。

本次不允许的副作用：

1. 登录、读档、进游戏、保存任一核心链路被影响。
2. 新增未捕获异常。
3. 为了保护可选模块而把核心模块错误“吞掉”。

## 四、自测方案总览

### 自测目标

自测不是证明“所有问题都解决了”，而是先证明两件事：

1. 可选模块异常时，服务端不会再整服不可用。
2. 核心登录链路在本次改动后没有回归。

### 自测分层

#### L0：静态自测

必须完成：

1. 搜索确认黑市和仙缘宝箱顶层不再直接 `require("systems.LootSystem")`
2. 搜索确认 `server_main.lua` 中可选模块已用 `pcall(require, ...)`
3. 搜索确认 `RedeemHandler.RegisterRedeemCodesFromConfig()` 有 `serverCloud` 判空
4. 搜索确认新增 boot 日志和身份阶段日志存在

#### L1：降级路径自测

必须验证：

1. 模拟 `serverCloud = nil`
   - 预期：兑换码注册跳过，服务端继续启动
2. 模拟 `LootSystem` 加载失败
   - 预期：黑市/宝箱功能失败，但登录链路正常

#### L2：核心链路自测

必须验证：

1. 单端登录
2. 角色列表读取
3. 进档
4. 保存
5. 断线清理

#### L3：冲突场景自测

必须验证：

1. 正常情况下，同账号双端仍会互踢
2. 若再次出现“双端共存且都卡在连接中”，日志里能看出没有进入 `ClientIdentity`

#### L4：功能回归自测

至少覆盖：

1. 黑市查询
2. 黑市购买
3. 宝箱开启
4. 兑换码注册成功路径

### 自测记录要求

执行 AI 必须记录：

1. 每一层自测的输入条件
2. 关键日志
3. 实际结果
4. 是否通过
5. 未通过时的阻塞点

## 六、P0 止血方案

### 任务 1：给 `RedeemHandler.RegisterRedeemCodesFromConfig()` 加降级保护

**目的**

避免 `serverCloud` 短时未就绪或配置异常时，服务端在启动尾部崩掉。

**修改点**

文件：[RedeemHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs_latest/scripts/network/RedeemHandler.lua)

必须实现：

1. 函数开头先判断 `serverCloud` 是否可用。
2. 整个注册过程包一层 `pcall` 或等价保护。
3. 失败时只打印 `WARN`，不得抛出未捕获异常。
4. 明确记录：
   - 进入注册
   - 跳过注册
   - 注册成功条数
   - 注册失败原因

**预期行为**

1. `serverCloud` 可用：兑换码正常注册。
2. `serverCloud` 不可用：跳过注册，服务端继续启动。
3. `RedeemCodes` 配置缺失：跳过注册，服务端继续启动。

**验收标准**

1. 即使把 `serverCloud` 视为 `nil`，代码路径也不会抛出未捕获异常。
2. `server_main.Start()` 末尾不会因为兑换码注册失败而中断。

---

### 任务 2：把 `LootSystem` 从黑市和仙缘宝箱的顶层 `require` 改为懒加载

**目的**

避免 `LootSystem -> EquipmentData -> MonsterData` 这条重依赖链在启动期拖死整个服务端。

**修改点**

文件：

1. [BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs_latest/scripts/network/BlackMerchantHandler.lua)
2. [XianyuanChestHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs_latest/scripts/network/XianyuanChestHandler.lua)

必须实现：

1. 删除顶层 `local LootSystem = require("systems.LootSystem")`
2. 增加缓存式 `GetLootSystem()`：
   - 首次调用时 `pcall(require, "systems.LootSystem")`
   - 成功则缓存
   - 失败则打印错误并返回 `nil`
3. 所有实际使用 `LootSystem` 的位置改为先取 `GetLootSystem()`
4. 若 `LootSystem` 取不到：
   - 当前请求返回失败
   - 不能抛出异常
   - 不能影响其他系统

**预期行为**

1. `LootSystem` 正常：黑市/宝箱逻辑不变。
2. `LootSystem` 异常：黑市/宝箱功能降级，但服务端仍可启动、登录、读档、进游戏。

**验收标准**

1. `server_main.lua` 顶层不再通过黑市/宝箱间接硬依赖 `LootSystem`。
2. 搜索结果中，黑市和仙缘宝箱文件顶层不再出现 `require("systems.LootSystem")`。

---

### 任务 3：在 `server_main.lua` 中把高风险可选模块改为“可失败加载”

**目的**

把“某个业务模块加载失败 = 整服起不来”改成“某个业务模块不可用，但整服还能跑”。

**修改点**

文件：[server_main.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs_latest/scripts/server_main.lua)

建议本次至少处理以下模块：

1. `network.BlackMerchantHandler`
2. `network.XianyuanChestHandler`
3. `network.SkinShopHandler`

处理方式：

1. 顶层 `require` 改为 `pcall(require, "...")`
2. 失败时打印 `[Server][WARN] ... load failed`
3. 对应事件订阅改为“模块存在才订阅”
4. 对应断开清理也要判空后再调用

**不要处理为可失败加载的模块**

以下属于核心链路，本次仍保留硬依赖：

1. `network.SaveProtocol`
2. `network.ServerSession`
3. `network.SlotReadService`
4. `network.SaveLoadService`
5. `network.SaveWriteService`

原因很简单：这些模块失败时，本来就已经无法提供线上服务，不能假装启动成功。

**验收标准**

1. 可选模块加载失败不会阻断 `Start()`
2. 服务端最少仍能跑通：
   - `ClientIdentity`
   - `VersionHandshake`
   - `FetchSlots`
   - `LoadGame`
   - `SaveGame`

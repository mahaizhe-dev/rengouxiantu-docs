## 一、核心判断

线上所有"回档、丢东西、重复刷、点了没反应、属性异常"的表象，根在同一处架构决定：

> 存档 = 客户端全量快照直写 save_{slot}，无版本号/无 CAS；奖励与资源散落在三个回档行为不一致的一致性域（save/money/quota）。

止血批已堵住"永久卡死"，但数据一致性的根未动。本路线图从"仍在流血"到"根治"分五步推进。

## 三、风险评估

### 高风险（需设计评审）

| 项 | 风险等级 | 风险描述 | 缓解措施 |
|---|---|---|---|
| A2 WriteSaveBack 接锁 | 中高 | 黑市三路并行（WAL+回包+Query），排队/重试可能引入超时/死锁 | 短退避3次放弃+保留WAL；画状态图后执行 |
| A3 WAL 先用后删 | 中高 | 支付对价核心链路，"用"失败容错设计复杂 | FeatureFlags 灰度；失败保留WAL下次重试 |
| A5 加载分模块 pcall | 中 | 降级部分若被后续保存写回等效丢数据 | 降级系统标 readonly 禁止序列化 |
| B1 保存 CAS | 高 | merge 逻辑无银弹（物品id碰撞、装备冲突） | log-only 灰度≥2周→rev高者胜（简单版）→后期merge |

### 低风险（可快速执行）

| 项 | 风险等级 | 原因 |
|---|---|---|
| 删 .pre-s1 | 极低 | 纯删除无运行时影响 |
| EventBus unsub | 极低 | 照抄现有正确范式 |
| pcall require | 极低 | 纯加法保护 |
| A1 EventHandler 锁 TTL | 低 | 已有 Trial T3 成功范例 |
| A4 库存空表护栏 | 低 | 加法检查不改正常路径 |
| A6/A7 性能削峰 | 低 | encode复用是纯优化 |
| F 性能缓存 | 低 | 加法缓存，最坏退回每帧计算 |

## 一、核心判断：一个根问题，五个域的投影

线上所有"回档、丢东西、重复刷、点了没反应、属性异常"的表象，**根都在同一处架构决定**：

> **存档 = 客户端全量快照直写 `save_{slot}`，无版本号/无 CAS；而奖励与资源散落在三个回档行为不一致的一致性域。**

三个域：
- **save 域**（物品、灵韵、属性、坐标、领取标记）——可被整档覆盖 / 回退旧备份；
- **money 域**（仙石、积分）——serverCloud 原子，**永不随存档回档**；
- **quota / 独立 key 域**（每日额度、里程碑、兑换记录）——免疫回档。

任何一笔"跨域"交易（买入 = 扣 money + 加 save 物；卖出 = 减 save 物 + 加 money；抽奖 = 扣 save 材料 + 服务端直写 save 奖励），回档时只回滚一半，于是必然产生"玩家亏 / 玩家刷 / 已扣没到账"。**震荡只是把这个长期存在的窗口从毫秒放大到秒级，让它变成可稳定复现的线上事故。** 震荡前同样的代码没暴露，不是因为没缺陷，而是窗口太小。

止血批已经堵住了"永久卡死"类问题（锁 TTL、pending 超时、identity 回包），但**数据一致性的根（无 CAS + 三域分裂 + 服务端直写无锁）没有动**，这是本方案的主攻方向。

## 三、分域问题清单（P0/P1 摘要，行号见各专项审计）

### 3.1 存档（主链）

- **P0** 库存反序列化 mgr=nil 静默跳过 + 序列化空表无退化护栏 → 空背包/空装备可覆盖云端（collection 有护栏、inventory 没有）。`SaveSerializer.lua:320/338/411`
- **P0** 加载单层大 pcall"全有或全无"：单件脏物品 → 整户禁载禁存。`SaveLoader.lua:37-609`
- **P0** 无断线重连状态机；`InitializeNetwork` 一次性。`client_main.lua:161`
- **P1** `data.version` 丢失按 v1 重跑全部迁移；paused/gameStarted_ 门仍冻结 UpdateCritical 本体；网络模式 checkpoint/登录备份假成功、SaveRecovery 三重失效；>1MB 无预检；dataLost 单向固化。
- **回归残留** N2（`_queuedFetchSlots` 二次悬挂）、N5（20s 超时对弱网大档偏激进）。

### 3.2 黑市

- **P0** `WriteSaveBack` 是全服务端唯一不持 slot 锁的直写方，与客户端全量保存双向互踩。`BlackMerchantHandler.lua:132`
- **P0** WAL"先删后用"→ 补偿蒸发 / 双倍补偿；写档失败仍回 ok → 客户端本地加物 + 登录 WAL 再补 = **双发**，且补发不打 bmNoResell = **洗白**。`:529/:1189/:1237`
- **P0** 卖出对整档回档零防御 + SellGuard 已成死代码 + 客户端 2 分钟自动保存必然复活已扣物 = **无限重复卖**（线上实锤）。
- **P1** Exchange 无内存锁（并发双花/免费兑换）；三链互不互斥；WAL 无 slot 归属（跨槽串货）；Buy 背包溢出静默吞货；WAL 清理整表清空。

### 3.3 C/S 架构

- **P0** 副本伤害纯客户端上报整数，服务端仅 `min(dmg,50000)`+0.2s CD → 8 次上报秒杀 3.5M BOSS，独占灵韵。`DungeonHandler` `HandlePlayerAttack`、`DungeonClient.lua:820`
- **P0** 多实例内存态失效：slot 锁/限流/handler 标志都是本实例内存，serverCloud 全局 → 跨房并发直写同一 save 无 CAS（回档/吞档根源之一）；BOSS HP/伤害 flush 用 SetInt 覆盖式（多实例丢伤害）。
- **P1** 协议无版本号（靠 VersionGuard 全量踢人，灰度期无法新旧共存，已发生字段错位事故）；错误字段 msg/message/reason/error 四种混用（实测 39/27/35/5 次）。
- **P1** 副本伤害/位置事件绕过限流（仅靠 CD）。

### 3.4 属性

- **P0** 挑战丹"计数自愈但属性不自愈"：SaveLoader 以 challenge progress 重算 xuesha/haoqi 计数，但已烙进 base 的 atk/maxHp 只 log-only 不修 → 属性永久虚高/虚低且档面对不上。`SaveLoader.lua:141-209` vs `challenge/rewards.lua:322-344`
- **P1** V5c 服务端校验 + 客户端 Step4 公式**均与实际脱节**（未计仙阶奖励、仙体成长、凝X丹）→ 120+ 玩家 expected 系统性偏低，×3 容忍被吃掉，WARN 噪声掩盖真异常（rules §12 同型事故模式，靠 log-only 免疫）。
- **P1** 影神丹单向棘轮（派生值反推计数只升不降）；仙体切换差量补丁使 base 不可重放（未来重构最大障碍）；B 类小计字段 section 缺失静默双真源；服务端 clamp 与客户端重算的休眠互搏（改任一侧数值即点燃）。

### 3.5 性能

- **P1** 存档管道：单次 Save 对 63KB 做 **2 次 encode**（measureSize 一次 + 发送一次），每 10 次审计再逐字段 encode（≈第 3 次全量），每 5 次 checkpoint 又 encode + 同步 File 写（且网络模式下 checkpoint 落入无效分支白 encode）；一帧内连发 5 个 RemoteEvent ~63KB 突发。尖峰 10~25ms（丢 1~2 帧）。
- **P1** 日志洪水：全代码 1461 处 print，Logger error/warn/info 无法线上关闭；AutoAttack 每次命中 print（挂机每秒 2~10 条字符串拼接+console IO）。
- **P1** 服务端云存储放大：每次保存 = 63KB 读 + ~130KB 写 + 3~6 独立 SetInt RPC + 备份串行 +2 RTT。
- **P2** NanoVG 瓦片层每帧全量程序化重绘（静态地形）；怪物名每帧 nvgTextBounds 测宽。

## 四、优化方案（三阶段，全程契合线上运营约束）

**全局约束**（贯穿所有阶段）：不改已有存档 key/字段/版本号、不删迁移函数、新字段默认值兼容旧档、服务端校验先 log-only 一版再启用、所有行为变更走 FeatureFlags 灰度。

### 阶段一 · 止血收尾 + 削峰（小 diff，可即刻分批出执行指令，零/低回归）

面向"仍在流血"和"低成本高收益"的点，全部是机械改动或加保护，不动数据结构。

| 编号 | 项 | 域 | 收益 | 风险 |
|---|---|---|---|---|
| A1 | EventHandler 三把 userId 锁加 15s TTL + 占用回包 + OnDisconnect 无条件清理（disconnect 前取 userId 快照）| 存档/日常 | 修复端午抽奖"点了没反应" | 低 |
| A2 | 黑市 WriteSaveBack 接入 slot 锁（Acquire 失败短退避重试）| 黑市 | 消除唯一无锁写档方，堵双写互踩 | 低-中 |
| A3 | 黑市 WAL 改"先用后删"（补偿落盘成功才删）+ 补发路径补打 bmNoResell + WAL 清理逐条删 | 黑市 | 堵补偿蒸发/双发/洗白 | 中 |
| A4 | 库存空表退化护栏（对齐 collection 的 `_lastKnownItemCount` + 骤减 >50% 用缓存替代）| 存档 | 堵空背包覆盖云端 | 低 |
| A5 | 加载改分模块 pcall（单件/单系统失败降级跳过而非整户禁载）+ 库存区分"无数据"与"反序列化失败" | 存档 | 单件脏数据不再拖垮整户 | 中 |
| A6 | 存档管道 encode 一次复用（DoSave encode→字符串透传，measureSize 用 #json）+ S3 审计改采样/GM 触发 + 网络模式 checkpoint 死代码短路 | 性能 | 保存尖峰 -30~50% | 低 |
| A7 | 战斗/拾取热路径 print 降级 + Logger 加全局级别开关（release 关 info）| 性能 | 稳态 CPU+GC 双降 | 低 |
| A8 | 属性阶段0：修 V5c/Step4 公式（计入仙阶+仙体+凝X丹）改从配置推导上限；影神丹去棘轮；凝X丹补 V5 clamp（log-only 一版）| 属性 | 消除校验噪声，为重构铺路 | 低 |
| A9 | 服务端 prewrite backup 条件化（10min 内跳过）+ rank/trial/prison SetInt 变更才写并回 batch | 性能 | 云存储读写 -40%，保存延迟 -2RTT | 中 |
| A10 | 主链回归 N2/N5：`_queuedFetchSlots` 纳入超时管理；LOAD 超时按 Begin 到达重置计时 | 存档 | 弱网选角不再二次悬挂 | 低 |

A1~A5 是数据安全，优先级最高；A6~A7 是体验（卡顿/发热），见效快、玩家可感知；A8~A10 是防患。

### 阶段二 · 根治数据一致性（需设计评审 + FeatureFlags 灰度，中改动）

这一阶段解决"回档"和"重复刷"的根，是本方案的核心。

**B1 · 保存 CAS 版本号协议（最高优先级，一根治多）**
- `save_{slot}` payload 增 `rev` 字段；服务端写前校验 `client.rev == cloud.rev`，一致才 `Set(rev+1)`，否则回 `S2C_SaveResult{ok=false, reason="stale", serverRev}`，客户端 reload-merge-retry。
- 用 `serverCloud.quota` 原子 Add 做写序号发号器可进一步防并发。
- **一次性根治**：多实例并发覆盖（回档/吞档）、客户端旧快照覆盖服务端直写（端午抽奖丢物、黑市回档）。
- FeatureFlags 已预留 `SERVER_SAVE_VALIDATOR_V2` 开关位。**前置条件见 D1**（当前 flag 重启即回 false，无法灰度）。

**B2 · 奖励落档原则（配合 B1）**
- 服务端发放的物品/货币必须写入 save 或独立可对账 key，**禁止"只进回包靠客户端自存"**。改造对象：SwordWall 商店/宝箱装备、SealDemon 一次性任务奖励、Trial 每日供奉奖励。
- 改造前至少先加"服务端发放流水日志"供客服对账（低成本先行）。
- 解决"钱花了没东西""今日已领取但没到账"。

**B3 · 副本伤害服务端 clamp（架构权威渐进迁移第一步）**
- 不追求服务端完全模拟（房间制 Lua VM 算力有限），而是服务端持有玩家攻击力上界（登录从存档快照缓存 atk），把上报 damage clamp 到 `min(reported, serverAtk × critMax × skillMult)` + 每秒 DPS 上限，替代固定 5 万。
- 把作弊成本从"改一个整数"提到"需伪造合理属性"。迁移清单：副本伤害 → 位置（加速度/边界）→ 触发式战斗结算 → 属性快照登录校验。

**B4 · 跨实例状态收敛**
- 凡涉及资源发放/唯一性的操作（领奖、接任务、竞速占位）统一走 `serverCloud.quota:Add(maxLimit)` 原子占坑（黑商自动收购已是范本），内存标志仅作本实例防抖。
- 副本 BOSS 伤害/HP flush 从 SetInt 覆盖改为 Add 增量（消除跨实例丢伤害）。

### 阶段三 · 结构收敛与治理（大版本，降低长期维护成本）

**C1 · 属性来源注册表**：建 `StatSourceRegistry`（id/countField/权威模块/perCount/cap/serializeKey），Serialize/Deserialize/SaveLoader/服务端 V5 clamp/UI 全部遍历注册表驱动 → 新丹药从"7 文件 12 点"收敛到"1 条配置"，漏环由结构消灭。配合"成长账本 baseline"消化仙体切换历史债（分阶段迁移，见属性专项 §5）。

**C2 · WAL 重构**：先用后删、带 slot 归属、幂等票据（凭据先于发货、消费后置删除）。

**C3 · 服务端写档统一收口**：抽象统一写档服务（slot 锁 + BeforeOverwrite + CAS），黑市/日常所有直写点接入，消灭"每个 handler 各写各的"。

**C4 · 协议 envelope 统一**：`{pv, rid, data}` 外层 + 错误统一 `{ok, code, msg}`，废弃四字段混用；`rid` 同时解决重连幂等和 requestId 匹配。借 VersionGuard 踢人窗口分批迁移。

**C5 · 断线重连状态机**：客户端 session epoch + resume token（userId+slot+lastAckSeq），断线后 background 重匹配携带 token，服务端以 serverCloud 为会话真相源恢复；分级 unstable→reconnecting→disconnected；所有写操作带 client request-id 幂等去重。

**C6 · 渲染**：瓦片层烘焙到离屏 render target（仅相机跨半瓦片重绘）或精细瓦片降 LOD；怪物名 nvgTextBounds 宽度按 name 缓存。

### 贯穿性基础设施（越早做越好）

**D1 · FeatureFlags 支持 serverCloud 下发 override（阻塞项）**：当前 6 个 flag 全 `default=false` 且 override 内存态、重启即回滚 → **阶段二所有 v2 路径都无法安全灰度/回滚**。必须先让 FeatureFlags 能从 serverCloud 系统 key 读 override。这是阶段二的前置条件。

**D2 · 可观测性标准化**：NetDiag 增服务端指标（存档写失败率/限流命中/结算异常/各云域失败）；异常事件按 userId 低频采样入库（现在只有全局桶，无法定位单个玩家）；GM 只读 dashboard（查最近 N 分钟 + 查指定 userId + 强制释放锁）。已有日志标签 `[ROLLBACK-WRITE]`/`[LOAD-FROM-BACKUP]`/`[LOCK-EXPIRED]` 的日频与用户分布应作为阶段一验证重点。

## 五、落地路线图

```
阶段一（止血收尾+削峰）  ── 可即刻，分 2~3 个小版本
  数据安全: A1 A2 A3 A4 A5 A10
  体验:     A6 A7
  防患:     A8 A9
        │
        ▼
D1 FeatureFlags 云端下发  ── 阶段二前置，必须先做
        │
        ▼
阶段二（根治一致性）  ── 大版本，log-only 灰度 ≥1 版
  B1 保存CAS  →  B2 奖励落档  →  B3 副本clamp  →  B4 跨实例收敛
  （B1 是总闸，优先；B2/B3 可并行）
        │
        ▼
阶段三（结构收敛）  ── 随后续大版本滚动
  C1 属性注册表 / C2 WAL重构 / C3 写档收口 / C4 协议envelope / C5 重连 / C6 渲染
  D2 可观测性  ── 全程并行推进
```

**优先级一句话**：先做 A1~A5（还在流血的数据安全）+ A6~A7（玩家可感知的卡顿发热）；同时把 D1 做掉解锁灰度能力；然后 B1 保存 CAS 是整个方案的总闸——它一旦落地，回档/吞档/抽奖丢物/黑市回档刷这一大类问题被系统性根治，剩下的 B2~B4、C 系都是收尾和加固。

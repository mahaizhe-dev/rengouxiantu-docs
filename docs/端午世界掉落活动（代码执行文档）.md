## 1. 任务目标

在 `origin/main` 已完成的六一双掉落活动骨架上，切换为端午活动，并补齐端午相对六一新增的两条能力：

1. 新增 `仙界精品粽`：
   - 高阶 BOSS 稀有掉落
   - 彩绳宝箱大奖
   - 香囊宝箱稀有奖励
   - 可手动食用
   - 永久 `福缘 +1`
   - 单角色最多食用 `10`
   - 可上黑市
2. 新增服务端校验的一次性 BOSS 击杀里程碑任务页：
   - `300 / 600 / 900 / 1200 / 1500 / 1800 / 2100 / 2400 / 2700 / 3000`
   - 每档奖励 `辟邪香囊 ×1`

补充要求：

1. 端午大奖皮肤改为 `pet_premium_wuxing`。
2. 该皮肤重复补偿固定为 `5000 灵韵`。
3. 该皮肤同步进入宠物皮肤商店放出。
4. `仙界精品粽` 必须显示在角色信息面板，且便于核对数据。
5. `彩绳`、`香囊` 黑市库存不上调。
6. 方案必须覆盖自动测试与人工验收。

## 2. 执行前强制前提

### 2.1 先接受这个现实

当前本地工作树并不是完整的六一成品状态，至少已确认：

1. `scripts/config/EventConfig.lua` 当前仍是 `mayday_2026`。
2. `scripts/config/GameConfig.lua` 当前 `EVENT_ITEMS` 仍是 `mayday_*`。
3. `scripts/config/BlackMerchantConfig.lua` 当前活动分类 `event` 已下架。
4. `scripts/systems/InventorySystem.lua` 当前仍内嵌旧版批量使用实现，而 `origin/main` 已拆出 `scripts/systems/inventory/batch_use.lua`。
5. 当前工作树与 `origin/main` 的活动主链、掉落链路、测试脚本存在分叉。

直接结论：

1. 不能把“当前工作树看起来像五一”当成端午开发基线。
2. 必须先参考 `origin/main` 的六一落地代码，再做端午增量。
3. 如果当前工作树缺失 `origin/main` 的六一重构文件或目录，先把六一基线结构对齐，再做端午改动。

### 2.2 执行顺序约束

执行 AI 必须按以下顺序做：

1. 先阅读 `origin/main` 的六一实现。
2. 判断当前工作树与 `origin/main` 是否结构一致。
3. 若不一致，优先把当前工作树对齐到六一骨架。
4. 再叠加端午改动。

禁止：

1. 直接在旧五一逻辑上生长一套“端午特判”。
2. 忽略 `origin/main` 已有的六一改造结果。
3. 只改文档或只改配置，不改协议/UI/存档/测试。

## 4. 当前代码基线结论

基于现代码与 `origin/main`，本次必须接受以下边界：

1. 端午活动应建立在六一“双开启物 + 双奖池 + 排行 + 保底”骨架上，不再沿用五一“单福袋”结构。
2. 角色总福缘当前由 `Player:GetTotalFortune()` 统一计算，但还没有 `pillFortune` 字段。
3. `CharacterUI` 已有属性页与丹药/成长统计区，适合接入 `仙界精品粽` 的属性贡献与食用次数。
4. 当前项目已有黑市交易保护锁、未锁定库存计数、批量使用测试与黑市锁纯 Lua 回归测试，可以复用。
5. 当前 `BlackMerchantConfig.lua` 的 `event` 分类已经下架，不应再把端午商品挂在不可见分类里。
6. `pet_premium_wuxing` 已在 `PetAppearanceConfig.lua` 中配置完成，但当前 `hidden = true` 且未进入 `SkinShopConfig.SKIN_IDS`。
7. `SkinShopConfig.SKIN_PRICE = 1000` 是当前统一价；如无新增策划口径，本次沿用 `1000 仙石`。
8. `origin/main` 中六一小/大奖池、保底、自测脚本已经是可参考成品。

补充提醒：

1. 六一文档里写过的部分数值与 `origin/main` 实装值存在差异时，以代码为准。
2. 但端午奖池的最终策划口径已经单独拍板，本次执行以本端午文档为准：
   - `仙界精品粽` 奖池估值按 `1000 灵韵`

## 7. 协议与服务端设计

### 7.1 活动主链

开箱主链继续沿用六一活动协议和 `EventHandler.lua`，只替换端午配置与开奖内容，不另起一套活动 Handler。

### 7.2 新增里程碑协议

建议新增 4 个协议常量：

1. `SaveProtocol.C2S_EventQueryBossMilestones = "C2S_EvtBossMilestonesQuery"`
2. `SaveProtocol.S2C_EventBossMilestonesData = "S2C_EvtBossMilestonesData"`
3. `SaveProtocol.C2S_EventClaimBossMilestone = "C2S_EvtBossMilestoneClaim"`
4. `SaveProtocol.S2C_EventClaimBossMilestoneResult = "S2C_EvtBossMilestoneClaimResult"`

建议字段：

```lua
-- query response
{
    EventId = "dragonboat_2026",
    BossKills = 1234,
    Claimed = { 300 = true, 600 = true }
}

-- claim request
{
    Milestone = 1200
}

-- claim result
{
    EventId = "dragonboat_2026",
    Milestone = 1200,
    BossKills = 1234,
    Claimed = true,
    Rewards = {
        { Type = "consumable", Id = "dragonboat_sachet", Count = 1 }
    }
}
```

### 7.3 领取校验顺序

服务端处理里程碑领取时，顺序必须固定：

1. 校验活动开启且 `eventId == "dragonboat_2026"`
2. 校验连接、用户、槽位有效
3. 加用户锁和槽位锁
4. 读取 `saveData`
5. 读取 `saveData.bossKills`
6. 读取 `saveData.event_data["dragonboat_2026"].bossMilestones.claimed`
7. 判断目标档位是否存在
8. 判断是否达标
9. 判断是否已领取
10. 发放 `dragonboat_sachet ×1`
11. 标记 `claimed[milestone] = true`
12. 持久化

### 7.4 实时性边界

这里必须明确写给执行 AI：

1. 当前服务端能看到的是最近一次已保存的 `bossKills`
2. 它不是服务端实时战斗计数
3. 因此“服务端校验”不等于“服务端实时累计”

客户端推荐流程：

1. 打开任务页时先触发一次 `save_request`
2. 存档完成后查询里程碑状态
3. 点击领取前再触发一次 `save_request`
4. 然后再发 `C2S_EventClaimBossMilestone`

不接受：

1. 客户端把本地 `GameState.bossKills` 直接作为领取依据
2. 服务端相信客户端传来的击杀数

## 8. 仙界精品粽的实现口径

### 8.1 存档字段

推荐新增 3 个字段：

1. `player.pillFortune`
2. `player.premiumZongEaten`
3. `player.pillCounts.zong`

权威关系：

1. `premiumZongEaten` 是次数权威字段
2. `pillFortune` 是属性镜像字段
3. `pillCounts.zong` 是统计镜像字段

加载/保存校验口径：

1. `premiumZongEaten` 夹紧到 `0..10`
2. `pillFortune` 强制与 `premiumZongEaten` 对齐
3. `pillCounts.zong` 强制与 `premiumZongEaten` 对齐

### 8.2 为什么不能复用旧字段

不允许把精品粽叠到 `fruitFortune`：

1. `fruitFortune` 是既有福缘果系统字段
2. 上限和来源口径完全不同
3. 混用后无法核对活动丹药进度

### 8.3 使用逻辑

`xianjie_premium_zong` 必须走独立使用分支：

1. 先检查未锁定库存
2. 计算本次最多可食用 `10 - premiumZongEaten`
3. 自动夹紧使用数量
4. 调用 `ConsumeConsumable()` 扣除
5. 成功后：
   - `premiumZongEaten += actualCount`
   - `pillFortune += actualCount`
   - `pillCounts.zong = premiumZongEaten`
6. 若已满 `10`，返回“已达食用上限”

### 8.4 批量与单次入口

执行要求：

1. `EquipTooltip.lua` 里把 `xianjie_premium_zong` 加入批量按钮白名单。
2. `UseBatchConsumable` 或 `batch_use.lua` 中新增显式分支：
   - 位置必须在 `cfgData.category == "event"` 的出售分支之前
3. 单次点击和批量点击都走同一套上限校验。

### 8.5 黑市锁要求

`仙界精品粽` 是当前项目第一种“可吃且可黑市交易”的属性丹药，必须按黑市 `B 类` 商品防护执行：

1. 消耗必须走 `ConsumeConsumable()`
2. 锁定中的精品粽不能食用
3. 只能消耗未锁定堆叠
4. 提示语复用现有 `TradeLock.LOCK_MESSAGE`

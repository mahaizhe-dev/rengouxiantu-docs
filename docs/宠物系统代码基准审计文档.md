## 1. 审计范围与结论摘要

本次审计覆盖宠物实体、成长与突破、技能与外观、自动拾取、存档与联网写档、UI入口、以及当前自动化测试覆盖情况。

结论先行：

1. 当前宠物系统的核心玩法链路是可运行的，且相对旧审计基线，`学技能 / 普通升级 / 精研升级 / 突破` 的消耗校验已经补上，`Pet:Destroy()` 也已补齐。
2. 当前线上最高风险仍然是**服务端对 `coreData.pet` 没有专门校验**。客户端会直接上传并在加载时直接恢复宠物等级、阶级、技能和外观，这在联网项目里仍是 `P0`。
3. 宠物给主人加成、账号级外观加成、宠物同步属性三者之间仍存在**重算顺序问题**，并且加载顺序会放大这个问题，属于 `P1`。
4. `攻速强化`、`减伤强化` 两条技能线已完整配置到掉落、商店和UI，但当前仍未真正进入战斗结算，属于明显的规则与实现偏差，属于 `P1`。
5. 宠物高价值操作还没有形成统一的持久化策略。当前只有“精研升级”在宠物模块内显式发出 `save_request`，其余大量资源消耗操作仍依赖通用自动存档窗口，属于 `P2`。
6. 从扩展性和代码质量看，`scripts/ui/PetPanel.lua` 仍然过大且耦合严重；自动拾取仍是线性扫描；高级外观加成是账号已解锁总和结算而不是按装备结算。这些都不是立刻炸线上的问题，但会持续增加后续扩展成本。

## 4. 关键链路事实

### 4.1 客户端运行链路

1. `scripts/main.lua` 创建 `Player` 后直接创建 `Pet.New(GameState.player)`。
2. 游戏主循环逐帧执行 `pet:Update(dt, gameMap)`。
3. 宠物的绝大多数玩家操作都从 `scripts/ui/PetPanel.lua` 进入。
4. 宠物拾取的真实搜索与结算逻辑在 `scripts/systems/loot/pickup.lua`。

### 4.2 存档与联网链路

1. `scripts/systems/save/SavePersistence.lua:157-210` 将宠物与账号级外观合并进 `coreData.pet` 和 `coreData.accountCosmetics`。
2. `scripts/systems/save/SaveSerializer.lua:569-598` 会序列化宠物的 `name/level/exp/tier/hp/alive/deathTimer/skills/appearance`。
3. `scripts/systems/save/SaveLoader.lua:83-84` 先恢复 `saveData.pet`。
4. `scripts/systems/save/SaveLoader.lua:393-396` 后恢复 `accountCosmetics`。
5. `scripts/network/SaveWriteService.lua:334-501` 是联网写档时的服务端校验入口，但当前没有 `coreData.pet` 专项校验。

## 5. 审计发现

### P0. 服务端没有校验 `coreData.pet`，宠物存档仍然是强客户端权威

**代码依据**

- `scripts/systems/save/SavePersistence.lua:157-210` 将宠物直接写入 `coreData.pet`。
- `scripts/network/SaveWriteService.lua:334-501` 校验了 `player/shop/challenges/warehouse/seaPillar` 等字段，但没有发现任何 `coreData.pet` 校验。
- `scripts/systems/save/SaveSerializer.lua:603-646` 会直接把 `level/exp/tier/skills/appearance` 恢复到运行态宠物对象。

**风险**

- 玩家只要篡改本地状态或拦截 `C2S_SaveGame`，理论上就可以伪造宠物等级、阶级、技能等级、外观选择。
- 该问题直接关联线上项目的数值安全、回档治理、异常账号甄别和客服追责，属于必须优先收口的问题。
- 根据 `rules.md`，这是线上项目，且存档兼容优先级最高，因此不能用“直接改字段名”或“删旧存档字段”这类破坏性方式收口。

**建议**

1. 在 `SaveWriteService._ValidateCoreData()` 中补齐 `pet` 专项校验。
2. 至少校验并必要时钳制以下字段：
   - `tier` 必须在 `GameConfig.PET_TIERS` 范围内。
   - `level` 不能超过当前 `tier.maxLevel`，不能小于 1。
   - `exp` 必须落在当前等级的合理区间内。
   - `skills` 只能来自 `PetSkillData.SKILLS` 白名单，且 `tier` 只能在 `1..4`。
   - `appearance.selectedId` 必须同时满足“当前宠物阶级可用”与“账号已解锁”。
3. 对命中钳制的账号做结构化日志，至少记录 `userId / slot / petTier / petLevel / suspiciousSkillIds`。
4. 如后续要提高安全级别，应逐步把“突破、学技能、升技能、外观切换”迁移为服务端确认型事务；但迁移时必须遵循 `rules.md` 的兼容原则，只能做增量协议和版本迁移。

### P1. 宠物给主人加成、账号级外观加成、宠物同步输入之间存在重算顺序问题

**代码依据**

- `scripts/entities/Pet.lua:183-189` 在 `Pet:RecalcStats()` 里先读取 `owner:GetTotalMaxHp()/GetTotalAtk()/GetTotalDef()`。
- `scripts/entities/Pet.lua:224-226` 在同一次 `RecalcStats()` 的末尾，才把 `owner.petOwnerBonuses` 更新为当前宠物技能提供的主人属性加成。
- `scripts/entities/Player.lua:279-360` 的角色总属性缓存依赖 `petOwnerBonuses` 与 `PetAppearanceConfig.GetAllPremiumBonuses()`。
- `scripts/systems/save/SaveLoader.lua:83-84` 先反序列化宠物，`scripts/systems/save/SaveLoader.lua:393-396` 再恢复账号级外观。

**影响**

- `fortune_owner` 对同步率的影响，不会在本次宠物重算里立即生效。
- `constitution_owner / physique_owner / wisdom_owner` 给角色带来的属性提升，也不会在同一次宠物同步计算里被完整纳入。
- 加载完成后的首轮重算，宠物和角色都可能先吃到一版“旧的主人属性”或“未恢复的账号外观加成”。

**建议**

1. 把重算拆成明确的两阶段：
   - 第一阶段：更新 `owner.petOwnerBonuses`。
   - 第二阶段：显式失效角色属性缓存，再读取 `owner:GetTotal*()` 重算宠物。
2. 在 `Player` 侧增加显式的“属性缓存失效”接口，不要把一致性依赖在“下一帧自然刷新”上。
3. 调整加载顺序：
   - 先恢复 `accountCosmetics`
   - 再恢复 `pet`
   - 最后统一执行一次“角色 -> 宠物”的完整重算
4. 这类修正不能直接改存档字段结构，必须按 `rules.md` 的兼容要求，以加载顺序和运行时重算逻辑修正为主。

### P1. `攻速强化` 与 `减伤强化` 已完整投放，但没有进入真实战斗结算

**代码依据**

- `scripts/config/PetSkillData.lua:85-155` 已定义 `atkSpd_basic` 与 `dmgReduce_basic`。
- `scripts/entities/Pet.lua:214-218` 会把 `atkSpeedPct` 和 `dmgReducePct` 写入宠物运行态。
- `scripts/entities/Pet.lua:63` 和 `scripts/entities/Pet.lua:522` 仍使用固定 `self.attackInterval = 1.2` 控制普攻节奏。
- `scripts/entities/Pet.lua:817-839` 的 `TakeDamage()` 直接按原始伤害扣血，没有使用 `dmgReducePct`。

**风险**

- 玩家可以拿到“掉落了、买到了、装上了、UI也显示了，但实际不生效”的技能书。
- 这会直接破坏线上项目的数值可信度，后续如果继续投放更多宠物技能书，问题会越来越难解释。

**建议**

1. 明确只做二选一：
   - 立即接入真实战斗结算。
   - 或在接入前，先从掉落、售卖、UI展示中撤掉这两条技能线。
2. 如果接入：
   - `atkSpeedPct` 应参与计算有效攻击间隔。
   - `dmgReducePct` 应参与最终受伤乘区。
3. 把这两条能力补上专门的自动化测试，避免以后再次出现“配置存在但逻辑未接”的回归。

### P2. 宠物高价值操作缺少统一的持久化策略

**代码依据**

- 当前宠物模块内显式发出 `save_request` 的入口只有 `scripts/ui/PetPanel.lua:1959`，即“精研升级”。
- 以下宠物高价值操作当前没有在本模块内显式持久化：
  - 喂食：`scripts/ui/PetPanel.lua:1230-1249`
  - 突破：`scripts/ui/PetPanel.lua:1252-1267`
  - 学技能：`scripts/ui/PetPanel.lua:1477-1493`
  - 普通升级：`scripts/ui/PetPanel.lua:1863-1900`
  - 删除技能：`scripts/ui/PetPanel.lua:1966-1986`
  - 外观切换/重置：`scripts/systems/PetSkinSystem.lua:96-129`
- `scripts/systems/PetSkinSystem.lua:134-158` 的高级外观解锁 API 自身也不拥有持久化职责，依赖调用方记得保存。

**风险**

- 在“客户端先改状态、SaveSystem 延迟保存”的架构下，会形成宠物资源回档窗口。
- 宠物技能书、突破材料、灵韵和外观解锁都不是低价值操作，不适合继续依赖隐式自动保存收口。

**建议**

1. 给宠物系统建立统一的操作服务层，例如 `PetProgressionService / PetSkillService / PetAppearanceService`。
2. 明确每类操作的持久化等级：
   - 必须即时：突破、学技能、精研升级、高级外观解锁
   - 可会话收口：连续喂食、普通升级尝试、普通外观切换
3. 如果担心频繁写档，优先接入 `SaveSession.MarkDirty()` 这类会话收口，而不是完全不记账。

### P2. `PetPanel.lua` 仍然过大，UI、业务规则、资源消耗和持久化耦合严重

**代码依据**

- `scripts/ui/PetPanel.lua` 当前约 `2007` 行。
- 同一文件同时承担 UI 构建、技能学习、技能升级、突破、喂食、外观切换、角色 UI 联动、即时保存等职责。

**风险**

- 任一宠物功能改动都会触发大文件修改，线上热修和冲突合并成本都偏高。
- 宠物规则难以脱离 UI 做自动化测试。
- 未来如果继续扩宠物技能线、外观页、运营活动入口，复杂度会继续堆在单文件里。

**建议**

1. 拆分为：
   - `PetPanelView.lua`
   - `PetProgressionService.lua`
   - `PetSkillService.lua`
   - `PetAppearanceService.lua`
2. 让 UI 只负责交互和展示，业务层只负责判定、扣资源、触发保存与事件。
3. 后续所有新宠物功能都不要继续直接塞回 `PetPanel.lua`。

### P2. 高级外观当前是“账号已解锁总和加成”，装备只影响显示，不影响数值来源

**代码依据**

- `scripts/config/PetAppearanceConfig.lua:240-266` 的 `GetAllPremiumBonuses()` 会遍历 `accountCosmetics.petAppearances` 中所有已解锁外观并求和。
- `scripts/entities/Player.lua:279-360` 直接把这组汇总结果并入角色总属性缓存。
- `scripts/systems/PetSkinSystem.lua:96-119` 的 `EquipSkin()` 只切换 `pet.appearance.selectedId`，不决定高级外观加成来源。

**风险**

- 对玩家而言，“装备了哪件高级外观”和“当前真正吃到哪些高级外观加成”并不是同一个概念。
- 如果后续扩展外观售卖、活动返场、外观套装或外观洗练，这条隐式规则会变成定价和客服解释风险。

**建议**

1. 把这条规则明确固化为可配置设计：
   - `bonusScope = account_unlocked_total`
   - 或 `bonusScope = equipped_only`
2. 在商店文案、外观说明、运营配置里同步这条规则，避免策划和玩家各自按不同口径理解。
3. 如果未来需要改成“按装备结算”，必须通过新增字段和迁移逻辑处理，不能直接破坏当前账号级解锁数据结构。

### P3. 宠物自动拾取仍是线性全量扫描，扩展后会持续吃性能预算

**代码依据**

- `scripts/entities/Pet.lua:563-566` 每 `0.5` 秒调用一次 `LootSystem.FindPetPickable()`。
- `scripts/systems/loot/pickup.lua:255-274` 对 `GameState.lootDrops` 做线性遍历，按距离找最近目标。

**评估**

- 以当前单宠物、单角色、普通掉落密度来看，不是马上会炸的性能问题。
- 但如果后续提升掉落密度、增加更多宠物、或引入大范围自动拾取效果，线性全量扫描会持续抬高帧内开销。

**建议**

1. 给掉落物建立简单的空间分桶或九宫格索引。
2. 至少把“宠物可拾取候选集合”与“玩家磁吸候选集合”拆开缓存。
3. 如果短期不做索引，至少把复杂度边界写进设计约束，避免未来玩法无上限扩张。

### P3. `UnlockAllSkins()` 的计数逻辑仍有明显错误

**代码依据**

- `scripts/systems/PetSkinSystem.lua:185-189`：

```lua
if pet.tier < maxTier then
    pet.tier = maxTier
    count = count + (maxTier - pet.tier)
end
```

**影响**

- 赋值后再做减法，增量永远是 `0`。
- 当前只影响 GM/工具统计，不影响正式玩法，但属于低成本可修的明显错误。

## 6. 测试覆盖现状

当前直接能看到的宠物相关测试并不强：

- `scripts/tests/test_battle_pipeline_regression.lua` 包含 `Pet:TakeDamage` 语义验证，但使用的是 mock pet，不是实际 `Pet.lua` 运行对象。
- `scripts/tests/test_save_dto_contract.lua` 覆盖了 `pet` 字段在保存 DTO 中存在，但不覆盖宠物业务合法性和服务端校验。
- 未发现对以下关键链路的有效自动化覆盖：
  - 宠物服务端写档校验
  - 宠物给主人加成后的重算顺序
  - 宠物加载顺序与账号级外观加成联动
  - 宠物学技能 / 升技能 / 突破 / 删除技能 / 外观切换 的完整资源与保存闭环
  - `攻速强化` / `减伤强化` 的真实战斗接入

补测建议：

1. 先补 `SaveWriteService` 的 `pet` 校验单测。
2. 再补“宠物给主人加成 + 账号外观 + 同步率”的重算顺序回归测试。
3. 再补高价值宠物操作的保存策略测试。

## 7. 面向线上项目的优化建议

### 7.1 扩展性

1. 把宠物规则从 `PetPanel.lua` 中继续剥离，形成稳定的服务层和数据层边界。
2. 给宠物技能、外观、成长、掉落建立更明确的配置驱动边界，不要让 UI 文件继续承担规则源。
3. 将“高级外观加成范围”“宠物给主人加成是否参与同轮同步”等高影响规则显式配置化或流程化。

### 7.2 代码质量

1. 增加宠物系统的专门测试入口，避免只靠全局战斗回归覆盖。
2. 所有宠物高价值操作都统一走服务层，禁止 UI 直接写状态。
3. 给 `Player` 和 `Pet` 的互相依赖补显式缓存失效接口，减少隐式时序依赖。

### 7.3 安全性

1. 服务端补 `coreData.pet` 白名单校验与钳制日志。
2. 对宠物异常成长、异常技能组合、异常外观组合建立审计日志。
3. 后续若引入服务端确认型宠物事务，必须遵守 `rules.md` 的存档兼容要求，不改旧 key，不做破坏式迁移。

### 7.4 性能

1. 优先收口拾取候选搜索复杂度。
2. 角色外观加成汇总可考虑做更明确的失效缓存，而不是每次属性缓存重算都遍历全部已解锁外观。
3. 宠物面板可以继续减少全量重建，尤其是外观页和技能页切换。

## 8. 优先级建议

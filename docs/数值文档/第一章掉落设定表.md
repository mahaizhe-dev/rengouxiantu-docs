## 五、巡逻 Boss 机制现状

### 1. 当前接入方式

| 维度 | 结论 |
| --- | --- |
| 配置入口 | 已做成 `PatrolPresets.lua + spawn.patrolPreset + spawn.patrol` 的通用接入方式，不是单 Boss 特判 |
| 注入方式 | `Spawner` 在生成怪物时解析巡逻配置，再通过 `Monster:SetPatrolConfig` 注入，后续 Boss 可以直接复用 |
| 当前模板 | 远端当前有 `waypoint_loop`、`large_waypoint_loop`、`fortress_loop` 三个模板 |
| 当前模式 | 现在全部是“固定路点循环”，还不是“任意区域随机游荡” |

### 2. 性能判断

| 维度 | 评估 | 结论 |
| --- | --- | --- |
| 巡逻开销 | 当前实现是“固定路点切换 + 直线移动”，没有引入全局寻路系统 | 开销低 |
| 远处怪物约束 | 现有怪物主循环本来就对远处 `idle/patrol` 怪物有降频处理；这套巡逻逻辑直接复用了该节流 | 风险低 |
| 返位逻辑 | 脱战后回最近路点继续巡逻，不强制回出生点 | 合理 |
| 同区复用 | 1 到 2 只巡逻 Boss 级别没有明显压力；数量继续堆高时会线性增加更新成本 | 当前可接受 |

### 3. 当前主要风险

| 风险点 | 影响 |
| --- | --- |
| 没有寻路 | 路点如果压到障碍、狭缝或装饰边缘，Boss 可能卡走位，所以路点仍需人工校验 |
| 只有固定路点模式 | 现在是“可配置巡逻”，但还不是“指定大区域内随机游荡”；后续若要做区域随机游荡，建议扩成“预采样锚点 + 低频换点” |
| 共享池概率理解偏差 | `10%` 共享池不代表池内两件都 `10%`；当前是池命中 `10%`，单件等权拆分后各 `5%` |

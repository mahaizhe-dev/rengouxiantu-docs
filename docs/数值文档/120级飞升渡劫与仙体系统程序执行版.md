# 120级飞升渡劫与仙体系统程序执行版

> 面向执行 AI 的落地文档。本文以当前 `rengouxiantu-docs_latest` 代码为基准，目标是在不重写既有境界、存档、C/S、属性系统的前提下，新增 120 级后飞升渡劫、仙阶突破与仙体系统首版。

## 1. 目标与边界

### 1.1 本次必须落地

1. `Lv120 / 大乘巅峰` 后，角色突破入口默认切到第二个 tab：`仙阶突破`。
2. `大乘巅峰 -> 谪仙1阶` 废弃旧的 `仙丹200 + 金币5100万` 路径，改为：
   - 投入 `一转仙劫丹` 增加仙阶进度。
   - 进度满后进入渡劫台。
   - 首次渡劫为 `雷火劫`，暂定 60 秒。
   - 成功后进入 `谪仙1阶`，解锁后续小境界，并账号解锁 `仙人之体`，开启仙体系统。
3. `谪仙1阶 -> 谪仙9阶` 先按小境界直接突破处理：
   - 每小阶需要 10 点仙阶进度。
   - 进度满后点击直接成功，不进渡劫台。
4. `一转仙劫丹` 两个来源：
   - 剑气长城积分商店：每角色永久限购 10 颗，每颗 2000 剑气积分。
   - 第六章炼丹炉：100 万金币 + 1 万灵韵炼制 1 颗。
5. 仙体系统首版：
   - 默认仙体：`凡人`。
   - 首次渡劫成功账号解锁：`仙人之体`。
   - 仙体是账号级稀有内容：任一角色解锁后，同账号所有角色可用。
   - 每个角色只保存自己当前装备/生效的仙体模板，可以各自切换。
   - 仙体影响从 1 级开始计算的等级成长属性，不是只影响 120 级以后。
   - 切换仙体消耗 1000 灵韵，参考宠物皮肤切换的操作逻辑，但必须有两次确认并明确显示属性增减。
   - 仙体定位为游戏中最难获取的内容之一，后续投放必须按账号级高价值资产处理。

### 1.2 本次不做

1. 不重写整个 `ProgressionSystem`。
2. 不把全部旧境界 ID 替换成全新仙阶 ID。
3. 不完整实现九次渡劫，只实现第一次渡劫 `雷火劫`，其余八次只保留配置框架。

6. 不做仙体在线热切的复杂全量属性回放。首版使用更稳的存档后重载方案。

### 1.3 旧设定取舍

1. `后120级仙阶设定.md` 只作为阶位名称、九大阶段顺序的参考。
2. 旧的 `Lv121~300` 常规升级、旧 `仙力`、旧 `仙丹 + 金币` 飞升消耗，按废弃处理。
3. 当前代码里的 `dujie_1..dujie_4` 仍可作为兼容 ID 使用，避免装备、怪物、排行榜、章节门槛等大量系统短期失效。

## 3. 推荐新增模块

### 3.1 配置

新增 `scripts/config/AscensionConfig.lua`。

建议内容：

```lua
local C = {}

C.FEATURE_ENABLED = true

C.ITEM_ONE_TURN_PILL = "one_turn_tribulation_pill"
C.ITEM_TWO_TURN_PILL = "two_turn_tribulation_pill"

C.MAJOR_PROGRESS_REQUIRED = 30
C.MINOR_PROGRESS_REQUIRED = 10
C.PROGRESS_OVERFLOW_MAX = 4

C.PILL_GAIN_TABLE = {
    { type = "normal", gain = 1, weight = 775 },
    { type = "crit", gain = 2, weight = 200 },
    { type = "bigCrit", gain = 3, weight = 20 },
    { type = "heavenCrit", gain = 5, weight = 5 },
}

C.TRIBULATION_COOLDOWN_SEC = 600
C.TRIBULATION_DURATION_SEC = 60
C.TRIBULATION_WARMUP_SEC = 3

C.IMMORTAL_BODY_SWITCH_COST_LINGYUN = 1000
C.GROWTH_PROFILES = {
    mortal = { name = "凡人", maxHp = 15, atk = 3, def = 2, hpRegen = 0.3 },
    immortal_body_1 = { name = "仙人之体", maxHp = 25, atk = 5, def = 3, hpRegen = 0.5 },
```

1. `+2` 暴击概率固定为 20%。

3. 单颗期望进度约 `1.26`。
4. 大乘巅峰首段进阶需要 30 点进度，平均约 24 颗，无暴击保底 30 颗。

## 5. 仙阶进度规则

### 5.1 吃丹规则

`一转仙劫丹` 每颗随机增加 `1/2/3/5` 点仙阶进度。

硬规则：

1. 只有进度未满时可以吃丹。
2. 只有最后一颗丹允许溢出。
3. 因为单颗最大 `+5`，理论最大溢出为 `4` 点。
4. 进度达到或超过需求后，不能继续吃丹。
5. UI 进度条显示封顶到满值，存档记录真实值。

伪代码：

```lua
function AscensionSystem.ConsumePill()
    local req = AscensionSystem.GetRequiredProgress()
    if state.progress >= req then
        return false, "进度已满"
    end

    if InventorySystem.CountConsumable(C.ITEM_ONE_TURN_PILL) <= 0 then
        return false, "一转仙劫丹不足"
    end

    local gain, critType = RollGain()
    InventorySystem.ConsumeConsumable(C.ITEM_ONE_TURN_PILL, 1)

    local maxProgress = req + C.PROGRESS_OVERFLOW_MAX
    state.progress = math.min(state.progress + gain, maxProgress)
    state.lastAscensionGain = gain
    state.lastAscensionCritType = critType

    EventBus.Emit("save_request")
    return true
end
```

### 5.2 大境界与小境界

大境界突破：

1. `dacheng_4 -> 谪仙1阶`：需要 30 点进度。
2. 进度满后按钮变为 `前往渡劫台`。
3. 渡劫成功后才推进仙阶。

小境界突破：

1. `谪仙1阶 -> 谪仙2阶` 到 `谪仙8阶 -> 谪仙9阶`：每阶 10 点进度。
2. 进度满后按钮为 `突破`。
3. 点击直接成功，不进入渡劫台。
4. 成功后清空当前目标进度，进入下一目标。

第二大境界：

1. `谪仙9阶 -> 人仙1阶` 使用 `二转仙劫丹`。
2. 当前只保留配置和 UI 禁用提示，不开放产出。

### 6.2 第一次渡劫：雷火劫

时长：60 秒。

参战：

1. 玩家当前宠物、伙伴、可参战单位允许保留。
2. 渡劫目标是生存到结束，不要求击杀怪物。

机制只做两种：

| 机制 | 规则 |
| --- | --- |
| 雷劫 | 每波追着玩家连续劈 9 道天雷；每道落点有短预警；玩家需要持续移动，且需要一定移速 |
| 火劫 | 火焰覆盖渡劫台绝大多数位置；结算前出现少量安全区，玩家需要进入安全区 |

建议 60 秒节奏：

| 时间 | 内容 |
| --- | --- |
| 0-3 秒 | 入场准备与倒计时 |
| 3-12 秒 | 第一波追身九雷 |
| 12-22 秒 | 第一轮火场安全区 |
| 22-35 秒 | 第二波追身九雷，频率提高 |
| 35-50 秒 | 第二轮火场安全区，安全区更少或更偏 |
| 50-60 秒 | 雷火组合收尾 |

伤害：

1. 机关类伤害不在文档中定死具体数值。
2. 首版建议用最大生命百分比为主，全部放配置。
3. 不要写死在机关逻辑里，方便线上调参。

### 6.3 成功失败

成功条件：

1. 玩家存活到 60 秒结束。
2. 只结算一次，使用 `tribulationRunId` 防止重复成功。

成功结算：

1. 推进仙阶到目标阶位。
2. 设置兼容 `player.realm = "dujie_1"`。
3. 发放目标仙阶属性。
4. 解锁后续小境界。
5. 首次成功获得 `immortal_body_1`。
6. 清空当前目标进度。
7. 触发强保存或即时 `save_request`，线上服优先服务端权威结算。

失败条件：

1. 玩家死亡。
2. 主动退出渡劫台。
3. 断线。
4. 切图、回登录、异常离场。

失败结算：

1. 不扣仙阶进度。
2. 不扣道具。
3. 不倒退仙阶。
4. 写入 10 分钟冷却：`tribulationCooldownUntil = now + 600`。
5. 冷却结束后，用原本已满进度重新挑战。

登录修复：

1. 读档发现 `tribulationState = "inProgress"` 时，按失败处理。
2. 写入冷却并清理运行态。
3. 不允许玩家通过断线绕过失败冷却。

## 8. 仙体系统

### 8.1 仙体定义

仙体不是 120 级后的成长，而是从 1 级开始计算的等级成长模板。
仙体解锁是账号级资产，每个角色装备哪个仙体是角色级状态。

定位：

1. 仙体是游戏中最难获取的内容之一，后续不应作为普通角色道具或短周期活动消耗品处理。
2. 任一角色首次渡劫成功并解锁 `仙人之体` 后，同账号所有角色都可以在仙体界面看到并切换该仙体。
3. 同账号不同角色可以装备不同仙体，互不影响当前属性模板。
4. 账号级只记录“是否解锁/何时解锁/由哪个角色首次解锁”等资产信息；角色级记录“当前装备哪个仙体”和切换 pending。

| 仙体 | ID | HP成长 | ATK成长 | DEF成长 | 回复成长 |
| --- | --- | ---: | ---: | ---: | ---: |
| 凡人 | `mortal` | +15/级 | +3/级 | +2/级 | +0.3/级 |
| 仙人之体 | `immortal_body_1` | +25/级 | +5/级 | +3/级 | +0.5/级 |

注意：

1. 当前代码中 `GameConfig.LEVEL_GROWTH.hpRegen` 是 `+0.2/级`。
2. 新的 `凡人` 模板把回复成长改为 `+0.3/级`，这是一次数值迁移，不只是新增仙体。
3. 必须用版本字段保证同一个存档只迁移一次。

### 8.2 为什么不做全量重算

当前项目里很多属性是直接烙进 `player.maxHp/atk/def/hpRegen` 的：

1. `Player:LevelUp()` 直接加等级成长。
2. `ProgressionSystem.Breakthrough()` 直接加境界奖励。
3. 一些丹药、系统奖励也可能直接加进 player 基础字段。
4. 装备、图鉴、神器、宠物等又有各自后加载加成。

因此首版不推荐从 `PLAYER_INITIAL` 全量重算 `player.maxHp/atk/def/hpRegen`。全量重算会误删已经烙进基础字段的永久来源，除非所有来源都有可回放计数。

首版推荐：等级成长差量调整。

公式：

```lua
deltaStat = (newProfile[stat] - oldProfile[stat]) * (player.level - 1)
player[stat] = player[stat] + deltaStat
```

示例：Lv120 从 `mortal` 切到 `immortal_body_1`：

| 属性 | 差值 |
| --- | ---: |
| maxHp | `(25 - 15) * 119 = +1190` |
| atk | `(5 - 3) * 119 = +238` |
| def | `(3 - 2) * 119 = +119` |
| hpRegen | `(0.5 - 0.3) * 119 = +23.8` |

旧存档从代码旧成长迁移到新 `mortal`：

| 属性 | 差值 |
| --- | ---: |
| maxHp | 0 |
| atk | 0 |
| def | 0 |
| hpRegen | `(0.3 - 0.2) * (level - 1)` |

这次 hpRegen 迁移是否立刻执行，需要数值确认。若担心影响存量线上强度，可先把存量档 `mortal` 的 `legacyGrowthProfileId` 记为 `mortal_legacy_0_2_regen`，仅新角色用 `mortal` +0.3。若执行全服迁移，必须有一次性版本号和测试覆盖。

### 8.3 切换流程

首版推荐“保存后重载应用”，不是在线热切。

流程：

1. 玩家打开仙体界面。
2. 从账号已解锁仙体列表中选择本角色未使用的仙体。
3. 第一层确认：预览属性变化。
4. 第二层确认：明确本次消耗 `1000 灵韵`，明确保存成功后返回登录或角色选择界面。
5. 校验灵韵、账号解锁状态、目标仙体与本角色当前仙体不同。
6. 扣除 1000 灵韵。
7. 写入：
   - `pendingImmortalBodySwitch.targetBodyId`
   - `pendingImmortalBodySwitch.fromBodyId`
   - `pendingImmortalBodySwitch.paidLingYun = 1000`
   - `pendingImmortalBodySwitch.createdAt`
8. 调用 `SaveSystem.Save(callback)` 强保存。
9. 保存成功后调用现有 `ReturnToLogin()` 或等价的回角色选择流程。
10. 下次加载角色时，在 post-load sync 阶段应用 pending：
    - 按差量调整基础属性。
    - 设置 `activeImmortalBodyId = targetBodyId`。
    - 设置 `baseGrowthProfileId = targetBodyId`。
    - 清除 pending。
    - 刷新属性缓存并校准 HP。
11. 保存失败则留在游戏内，提示稍后再试；不要回登录，不要应用目标仙体。

### 8.4 切换 UI 要求

参考 `PetPanel` 的宠物外观列表与卡片逻辑：

1. 显示账号已解锁仙体。
2. 本角色当前生效仙体显示 `使用中`。
3. 未解锁仙体可显示来源或隐藏。若显示，必须明确这是账号级稀有解锁内容。
4. 预览不扣费、不生效。
5. 属性变化必须显示增减，而不是只显示切换后总值。

预览至少展示：

| 行 | 内容 |
| --- | --- |
| 每级成长变化 | HP/ATK/DEF/回复每级成长差 |
| 当前等级成长总变化 | 按当前等级计算差值 |
| 自带属性变化 | 首版没有自带属性则显示无，不要空着 |
| 消耗 | 1000 灵韵 |
| 生效方式 | 保存成功后返回登录，重新进入后生效 |

二次确认必须包含：

1. “本次切换将消耗 1000 灵韵”。
2. “保存成功后会返回登录/角色选择界面，重新进入后生效”。
3. 属性下降项如果存在，必须用醒目颜色或明确文字展示。

## 9. 存档设计

### 9.1 推荐保存结构

优先把仙阶、渡劫、仙体做成独立系统序列化，挂到 `coreData`：

```lua
coreData.ascension = AscensionSystem.Serialize()
coreData.tribulation = TribulationSystem.Serialize()
coreData.immortalBody = ImmortalBodySystem.SerializeCharacter()
coreData.accountImmortalBodies = ImmortalBodySystem.SerializeAccount()
```

其中：

1. `coreData.immortalBody` 是角色级仙体状态。
2. `coreData.accountImmortalBodies` 是账号级仙体解锁资产。
3. 线上网络模式建议仿照 `account_atlas / account_cosmetics` 写入独立账号 key，例如 `account_immortal_bodies`，并在读档时注入当前角色的 `saveData`。
4. 如果首版为了少改保存管线先透传到 `coreData`，也必须在服务端 `SaveWriteService` 里按账号级旁路单独写，不能只落到当前角色槽位。

同时，和角色属性强绑定的最小字段可存在 `player` 内，保证加载早期可读：

```lua
player.activeImmortalBodyId
player.baseGrowthProfileId
player.baseStatsRecalcVersion
player.pendingImmortalBodySwitch
```

不要把大段运行时机关状态全部塞进 `player`。
不要在角色存档中保存 `ownedImmortalBodies` 作为权威解锁列表。

### 9.2 仙阶字段

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `ascensionStageIndex` | number | 当前大阶段，谪仙为 1，人仙为 2 |
| `ascensionMinorIndex` | number | 当前大阶段内小阶，1-9 |
| `ascensionTotalIndex` | number | 全局仙阶序号，1-81 |
| `ascensionProgress` | number | 当前目标真实进度，可记录最多 4 点溢出 |
| `ascensionProgressRequired` | number | 当前目标所需进度 |
| `ascensionMaterialId` | string | 当前目标消耗材料 ID |
| `lastAscensionGain` | number | 最近一次吃丹进度 |
| `lastAscensionCritType` | string | `normal/crit/bigCrit/heavenCrit` |
| `firstTribulationCompleted` | boolean | 首次渡劫是否完成 |

### 9.3 渡劫字段

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `tribulationState` | string | `none/ready/inProgress/cooldown` |
| `tribulationStageIndex` | number | 第几次大境界渡劫 |
| `tribulationStartedAt` | number | 开始时间戳 |
| `tribulationCooldownUntil` | number | 失败冷却结束时间戳 |
| `tribulationLastResult` | string | `success/fail/quit/disconnect` |
| `tribulationRunId` | string | 单次实例 ID，防重复结算 |

### 9.4 仙劫丹与商店字段

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `swordWallOneTurnPillPurchased` | number | 剑气长城永久兑换数量 |
| `swordWallOneTurnPillPurchaseLimit` | number | 当前为 10 |

如果剑气长城商店已经有统一 purchase quota 表，优先接入统一结构，不另存重复字段。文档字段仅表达业务需求。

### 9.5 仙体字段

账号级字段，建议独立 key：`account_immortal_bodies`。

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `unlockedBodies` | table/map | 账号已解锁仙体，key 为仙体 ID |
| `unlockedBodies[bodyId].unlockedAt` | number | 解锁时间戳 |
| `unlockedBodies[bodyId].source` | string | 来源，例如 `first_tribulation` |
| `unlockedBodies[bodyId].firstUnlockSlot` | number | 首次解锁角色槽位 |
| `unlockedBodies[bodyId].firstUnlockCharacterId` | string/nil | 若项目有角色唯一 ID，则记录 |
| `version` | number | 账号仙体数据版本 |

角色级字段，存当前角色的生效模板与切换事务：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `activeImmortalBodyId` | string | 当前生效仙体 |
| `baseGrowthProfileId` | string | 当前等级成长模板 |
| `baseStatsRecalcVersion` | number | 基础成长差量迁移版本 |
| `immortalBodyGrowthVersion` | number | 仙体成长公式版本 |
| `lastImmortalBodySwitchAt` | number | 最近切换时间 |
| `immortalBodySwitchCount` | number | 累计切换次数 |
| `pendingImmortalBodySwitch` | table/nil | 保存后重载应用的切换事务 |

### 9.6 迁移要求

新增存档版本，例如 v25：

1. 老档没有 `ascension` 时补默认：
   - `ascensionStageIndex = 0`
   - `ascensionMinorIndex = 0`
   - `ascensionProgress = 0`
2. 老档没有 `tribulation` 时补 `tribulationState = "none"`。
3. 老档没有角色级仙体字段时：
   - `activeImmortalBodyId = "mortal"`
   - `baseGrowthProfileId = "mortal"` 或兼容旧模板 ID
4. 老账号没有账号级仙体数据时：
   - 初始化 `account_immortal_bodies.unlockedBodies.mortal = { source = "default" }`
   - 不自动解锁 `immortal_body_1`
5. 是否把旧 hpRegen +0.2 迁移到新 `凡人` +0.3，必须通过版本字段一次性处理。
6. 所有迁移必须幂等，重复加载不能重复加属性。
7. 任一角色解锁账号级仙体后，切换到同账号其他角色必须能读取该解锁状态。

## 10. C/S 与线上安全

### 10.1 高价值操作清单

以下操作必须按线上高价值操作处理：

1. 吃 `一转仙劫丹` 增加仙阶进度。
2. 剑气积分商店兑换 `一转仙劫丹`。
3. 第六章炼丹炉炼制 `一转仙劫丹`。
4. 渡劫成功结算仙阶与仙体。
5. 仙体切换扣 1000 灵韵。
6. 账号级仙体解锁写入。

### 10.2 客户端与服务端分工

如果当前线上模式开启 `CloudStorage.IsNetworkMode()`：

1. 商店兑换必须服务端权威。
2. 渡劫成功结算至少要有 `runId` 和状态校验，防止客户端重复上报。
3. 首次渡劫成功解锁 `immortal_body_1` 时，必须写账号级资产，不能只写当前角色槽位。
4. 仙体切换如果首版仍由客户端改角色存档，必须用 `SaveSystem.Save(callback)` 确认落盘后再返回登录。
5. 后续最好新增服务端协议处理仙体切换，避免客户端直接扣灵韵。

账号级仙体解锁要求：

1. 服务端读取 `account_immortal_bodies`。
2. 若目标仙体未解锁，则写入 `unlockedBodies[bodyId]`。
3. 若已解锁，本次渡劫成功只结算角色仙阶，不重复改写账号来源，最多补充日志。
4. 账号级写入与角色渡劫成功存档尽量同事务提交；做不到同事务时，必须有可重试补偿，不能出现角色已渡劫成功但账号仙体永久丢失。
5. 账号级解锁是高价值资产，不能依赖客户端本地字段作为最终权威。

建议新增协议：

| 协议 | 用途 |
| --- | --- |
| `C2S_AscensionConsumePill` | 请求吃丹，服务端校验道具并写进度 |
| `S2C_AscensionConsumeResult` | 返回进度、暴击类型、剩余道具 |
| `C2S_TribulationStart` | 请求进入渡劫台 |
| `C2S_TribulationFinish` | 上报渡劫结果，首版可只接受 success/fail 并服务端校验状态 |
| `C2S_ImmortalBodySwitch` | 请求仙体切换 |
| `S2C_ImmortalBodySwitchResult` | 返回保存/事务结果 |
| `C2S_AccountImmortalBodyQuery` | 查询账号已解锁仙体 |
| `S2C_AccountImmortalBodyData` | 返回账号仙体解锁数据 |

如果首版不新增协议，也必须在文档和代码注释里标明是过渡方案，不能把客户端本地结算当成长期线上安全方案。

### 10.3 并发与重复提交

必须防：

1. 连点吃丹导致多扣或越过满进度继续吃。
2. 两个客户端同时吃丹。
3. 渡劫成功上报重复触发两次奖励。
4. 仙体切换保存成功前重复点击导致多扣灵韵。
5. 回登录过程中自动保存覆盖 pending 状态。
6. 两个角色/客户端同时触发账号级仙体解锁导致账号数据互相覆盖。

实现要求：

1. UI 点击后按钮置灰等待返回。
2. 系统层二次校验资源、状态、runId。
3. 保存字段记录 `lastProcessedRunId` 或等价结构。
4. 高价值失败必须提示并保留当前可恢复状态。
5. 账号级仙体数据写入必须做读改写合并，不允许用当前客户端旧缓存覆盖服务端较新的 `account_immortal_bodies`。

## 11. 属性结算

### 11.1 仙阶属性

仙阶每阶属性以最终确认的仙阶设定表为准。执行 AI 不要把旧 `dujie_1` 的 `GameConfig.REALMS.dujie_1.rewards` 当成新系统奖励直接用。

首版建议：

1. 把仙阶奖励写进 `AscensionConfig.ASCENSION_REWARDS`。
2. 大境界和小境界都通过 `AscensionSystem.ApplyAscensionReward(targetIndex)` 加属性。
3. 成功后同样刷新 `player.realmAtkSpeedBonus`，兼容旧显示。
4. 奖励必须只发一次，存档要记录已达成阶位。

### 11.2 HP 处理

属性变化后：

1. 突破成功可回满血，沿用旧突破体验。
2. 仙体切换导致 maxHp 上升或下降时，post-load 后执行 HP 校准：
   - `player.hp = min(savedHp, player:GetTotalMaxHp())`
   - 如果 `player.hp <= 0`，按现有 `SavePostLoadSync` 逻辑回到满血。
3. 切换仙体后必须 `player._statsCacheFrame = -1` 或调用项目已有刷新方法。

### 11.3 不允许的实现

1. 不允许只改 `GameConfig.LEVEL_GROWTH` 然后期待旧角色自动正确。
2. 不允许切换仙体时从初始属性全量覆盖 `player.maxHp/atk/def/hpRegen`。
3. 不允许把装备、丹药、图鉴、神器等后加属性混入仙体预览差值来源。
4. 不允许切换失败后已扣灵韵但没有 pending 或补偿路径。

## 12. UI 与体验

### 12.1 仙阶突破页

至少展示：

1. 当前阶段：例如 `大乘巅峰`、`谪仙1阶`。
2. 下一目标：例如 `谪仙1阶`、`谪仙2阶`。
3. 当前进度：`真实进度/需求`，显示层封顶。
4. 当前材料：`一转仙劫丹` 持有数量。
5. 最近一次吃丹结果：普通、暴击、大暴击、天命暴击。
6. 进度满后：
   - 大境界显示 `前往渡劫台`。
   - 小境界显示 `突破`。
7. 渡劫失败冷却剩余时间。

### 12.2 渡劫台 HUD

至少展示：

1. 倒计时。
2. 当前劫难阶段文字：雷劫/火劫。
3. 机关预警。
4. 失败后冷却提示。

不要做长篇教学弹窗。雷火劫通过预警、节奏和短文案让玩家理解。

### 12.3 仙体界面

仙体界面可以放在角色面板新增入口，也可以作为仙阶突破页的子入口。首版优先减少入口复杂度：

1. 若账号只有默认 `凡人`，入口可以隐藏或置灰。
2. 任一角色账号解锁 `仙人之体` 后，同账号所有角色入口开放。
3. 当前角色未到 120 级也可以切换已解锁仙体，因为仙体从 1 级开始影响等级成长；但 UI 必须清楚显示这是当前角色的属性模板切换。
4. 操作逻辑参考宠物外观，但仙体不是外观，必须强调属性变化。

## 13. 实施顺序

推荐按以下顺序做，避免先做玩法后卡在存档：

   - 加载流程增加 Deserialize。
   - `SaveMigrations` 增加默认字段。
5. 先接账号级仙体存储：
   - 新增或扩展 `account_immortal_bodies` 账号 key。
   - 读档时注入账号仙体数据。
   - 保存或服务端结算时按账号级旁路写入。
6. 改 `RealmPanel` 双 tab，阻断旧 `dujie_1` 突破路径。

## 14. 测试方案

### 14.1 单元与配置测试

1. 吃丹随机表总权重正确。
2. `+2` 概率配置为 20%。
3. `+3/+5` 概率可配置。
4. 进度未满可以吃丹。
5. 进度满后不能继续吃丹。
6. 最后一颗最大只允许溢出 4 点。
7. 大乘巅峰首段需求为 30 点。
8. 谪仙小阶需求为 10 点。
9. 道具不足时不能吃丹。

### 14.2 入口与旧路径回归

### 14.3 存档迁移测试

## 16. 验收标准

首版完成必须满足：

1. `Lv120 / 大乘巅峰` 后，玩家不会再通过旧境界按钮消耗 `仙丹200 + 金币5100万`。
2. 大乘巅峰首段进阶实际需要 30 点仙阶进度。
3. 吃丹进度、暴击、溢出、满后禁用全部正确。
4. 首次渡劫失败不扣进度，只给 10 分钟冷却。
5. 首次渡劫成功推进到 `谪仙1阶`，账号解锁 `仙人之体`。
6. 仙体切换有两次确认，显示明确属性增减，扣 1000 灵韵。
7. 仙体切换保存成功后返回登录/角色选择，重新进入后属性按差量生效。
8. 任一角色解锁仙体后，同账号其他角色可切换该仙体；各角色当前生效仙体互不影响。
9. 老存档可加载，新存档可回读，重复加载不重复加属性。
10. 网络模式下高价值操作没有明显双扣、重复结算、半状态风险。

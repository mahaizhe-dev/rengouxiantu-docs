# 2026-06-25 黑市仓库出售拦截与存档安全复盘方案-执行AI版

## 1. 基线与任务定位

当前代码基线：

- 分支：`main`
- 远端：`origin/main`
- HEAD：`a52bf03 fix: BlackMerchantHandler逻辑修复 + Phase3测试完善`
- 已确认回归：`test_save_phase3_smoke`、`test_save_phase1_smoke`、`test_bm_noresell`、`test_bm_noresell_consume`、`test_bm_noresell_warehouse` 合计 `34/34` 全绿

本方案只处理一个线上高风险问题：

> 玩家反馈：使用仓库后，黑市出售经常出现“数据同步中，请稍后再出售”，有时长时间或一直无法出售。

本方案不是：

- 不是黑市系统重构。
- 不是仓库服务端权威化。
- 不是存档 schema / migration 改造。
- 不是断线重连大改。
- 不是取消黑市出售保护门。

本方案目标：

1. 复盘当前仓库、黑市出售、存档之间的真实代码链路。
2. 评估是否可以放松拦截，以及放松拦截会不会重新引入经济漏洞。
3. 给出执行 AI 可落地的小步方案。
4. 明确风险、禁止事项、测试与验收标准。

## 2. 当前代码事实

### 2.1 仓库操作会标记黑市未同步

文件：`scripts/systems/WarehouseSystem.lua`

成功路径会执行：

- `WarehouseSystem._dirty = true`
- `BlackMarketSyncState.MarkWarehouseOp(...)`
- `EventBus.Emit("save_request")`

覆盖路径包括：

- `StoreItem`
- `StoreItemToSlot`
- `RetrieveItem`
- `UnlockNextRow`
- `SortWarehouse`
- `StoreItemToPage2`
- `RetrieveFromPage2`
- `UnlockPage2NextRow`
- `SortPage2`

注意：

- 单纯打开 `WarehouseUI.Show()` 不会调用 `MarkWarehouseOp`。
- 真正置脏的是存入、取出、整理、解锁等成功操作。
- 玩家口中的“打开仓库就不行”，线上语义大概率是“打开仓库并做了仓库操作后不行”。

### 2.2 黑市出售入口统一走同步门

文件：`scripts/ui/BlackMerchantUI.lua`

出售按钮和确认出售都调用：

```lua
BlackMarketSyncState.GetSellBlockReason(itemId)
```

如果返回 `global_unsync`，当前 UI 会：

1. 触发 `save_request`
2. 尝试 `SaveSession.Flush()`
3. 显示泛化文案：`数据同步中，请稍后再出售`
4. 不发送 `SendSell(...)`

买入入口 `SendBuy(...)` 不走这道 `GetSellBlockReason`，当前主要被拦的是出售。

### 2.3 当前未同步门是全局仓库脏标记

文件：`scripts/systems/BlackMarketSyncState.lua`

当前状态：

```lua
BlackMarketSyncState._dirtyWarehouse = false

function BlackMarketSyncState.MarkWarehouseOp(reason)
    BlackMarketSyncState._dirtyWarehouse = true
end

function BlackMarketSyncState.IsBlocked()
    if BlackMarketSyncState._dirtyWarehouse then
        return true, "warehouse_unsync"
    end
end
```

清除时机：

```lua
EventBus.On("game_saved", function()
    BlackMarketSyncState.ClearAll()
end)

EventBus.On("game_loaded", function()
    BlackMarketSyncState.ClearAll()
end)
```

结论：

- 仓库变更后，所有黑市出售都会被拦。
- 只有 `game_saved` 或 `game_loaded` 才会清除。
- 仅 `save_request`、关闭仓库、打开黑市都不会清除。

### 2.4 2 分钟自动保存不是恢复上限

文件：

- `scripts/systems/save/SaveState.lua`
- `scripts/systems/SaveSystem.lua`
- `scripts/systems/save/SavePersistence.lua`

当前常量：

```lua
AUTO_SAVE_INTERVAL = 120
SAVE_DEBOUNCE_INTERVAL = 3
```

仓库操作后已经发出 `save_request`，正常情况下 `SaveSystem.Update(dt)` 会在防抖后触发保存，理论上不应等 2 分钟。

但存在这些情况会导致 `game_saved` 长时间不来：

1. 当前处于挑战、试炼塔、镇妖塔、副本等模式，`SaveSystem.Update(dt)` 被主循环跳过。
2. 网络断开，`SavePersistence.Save()` 因 `_disconnected` 或无 `serverConn` 跳过。
3. 保存请求遇到 `slot_locked`，进入 3 秒 retry。
4. 保存 timeout，进入线性退避。
5. 保存失败但 `_dirty` 保留，等待后续重试。
6. 黑市查询、排行榜读请求正常，不等于存档写链路健康。

结论：

- “理论最多 2 分钟恢复”不成立。
- 当前黑市出售门禁只认 `game_saved`，如果保存链路不健康，就会长时间卡住。

历史结论不是“误加了仓库拦截”，而是：

1. 仓库与黑市之间曾存在经济漏洞窗口。
2. 黑市服务端出售判断依赖云端 `save_{slot}` 快照。
3. 玩家本地仓库变更如果未存档，服务端可能仍按旧背包快照判断。
4. 因此仓库变更后拦截黑市出售是安全止血策略。

当前问题是止血策略的副作用：

- 文案不清晰，玩家误以为黑市服务器卡。
- 关闭仓库时没有专门的强制落盘收口。
- 卡住后 UI 不区分仓库未存档、网络断开、保存 retry、保存 timeout。
- 测试没有覆盖“仓库变更后出售门禁体验与恢复”。

## 4. 高风险判断

### 4.1 不能直接取消仓库后出售拦截

禁止方案：

> 仓库关闭后直接允许黑市出售。

风险：

1. 仓库变更可能尚未进入云端 `save_{slot}`。
2. 黑市出售服务端可能基于旧快照执行。
3. 服务端黑市出售成功后会写回存档快照。
4. 旧快照写回可能覆盖客户端尚未落盘的仓库变更。
5. 极端情况下可能重新引入“黑市收益到账 + 本地仓库状态后续覆盖”的经济不一致。

### 4.2 P0 不建议做“无关物品可卖”的物品粒度放行

之前有一个看似合理的方向：

> 仓库移动 A 物品后，只拦 A，允许卖 B。

重新复盘后，此方向不适合作为 P0。

原因：

- 当前风险不只在“卖同一个物品”。
- 黑市出售本身是服务端写回存档快照的操作。
- 只要仓库存在未落盘变更，任何黑市出售都可能让服务端用旧快照写回，从而暂时覆盖仓库变更。

因此：

- P0 必须保留“仓库变更未存档时禁止黑市出售”的全局保护。
- 物品粒度放行只能进入 P2，并且前提是黑市服务端写回不再使用会覆盖本地仓库变更的旧全量快照，或引入明确的 merge/CAS 机制。

### 4.3 不能通过清 dirty 来恢复体验

禁止方案：

- 关闭仓库时清 `BlackMarketSyncState._dirtyWarehouse`
- 发出 `save_request` 后清 `_dirtyWarehouse`
- 黑市打开时清 `_dirtyWarehouse`
- 超时后自动清 `_dirtyWarehouse`

原因：

- `save_request` 只是保存意图，不是保存成功。
- 只有 `game_saved` 代表保存成功。
- 自动清 dirty 会恢复旧漏洞窗口。

### 4.4 不能自动重放黑市交易

禁止方案：

- 保存完成后自动帮玩家补发刚才被拦截的出售请求。

原因：

- 黑市出售是非幂等经济操作。
- 玩家持有、库存、价格、境界、连接状态都可能在等待期间变化。
- 自动重放可能造成重复出售、错价出售或误操作。

## 5. 推荐总方案

推荐分为 P0 和 P1。

P0 只做止血与可解释性：

1. 保留仓库未存档时禁止黑市出售。
2. 将“数据同步中”改成真实原因提示。
3. 仓库关闭时如有变更，立即尝试保存一次。
4. 黑市打开时关闭仓库 UI，避免并行操作。
5. 增加测试覆盖，保证不误清 dirty、不误放行出售。

P1 做体验增强与观测：

1. 增加仓库 dirty 持续时间和最近原因。
2. 在黑市 UI 显示更明确的超时/网络提示。
3. 增加日志，方便判断保存没有完成的真实原因。

P2 暂不执行：

1. 仓库 dirty 物品粒度放行。
2. 黑市服务端存档 merge/CAS。
3. 仓库服务端权威化。

## 6. P0 具体执行方案

### P0-1：为黑市出售门禁返回稳定原因码

文件：

- `scripts/systems/BlackMarketSyncState.lua`
- `scripts/ui/BlackMerchantUI.lua`

当前 `GetSellBlockReason(itemId)` 返回：

```lua
reason, detail
```

建议兼容扩展为：

```lua
reason, detail, code
```

示例：

```lua
return "global_unsync", "L2: warehouse_unsync", "warehouse_unsync"
return "global_unsync", "L2: consume_unsync", "consume_unsync"
return "global_unsync", "L2: save_session_active", "save_session_active"
```

兼容性：

- Lua 调用方只接两个返回值时不会受影响。
- UI 新代码可以使用第三返回值。

文案映射建议：

| code | UI 文案 |
| --- | --- |
| `warehouse_unsync` | `仓库道具变更尚未完成存档，暂不能出售` |
| `consume_unsync` | `道具消耗尚未完成存档，暂不能出售` |
| `save_session_active` | `近期操作正在保存，暂不能出售` |
| `network_disconnected` | `网络异常，存档未完成，暂不能出售` |
| unknown | `数据保存尚未完成，暂不能出售` |

说明：

- P0 可以只落地前三个 code。
- `network_disconnected` 可在 UI 层通过 `NetworkStatus.IsDisconnected()` 辅助判断，不要求改保存系统。
- UI 不允许解析 `detail` 字符串来判断业务原因。

必须替换的 UI 位置：

- 确认出售拦截处。
- 出售按钮实时拦截处。

保留行为：

- 仍然触发 `save_request`。
- 仍然调用 `SaveSession.Flush()`。
- 仍然不发送 `SendSell(...)`。

### P0-2：仓库 UI 内增加本次打开期间变更标记

文件：

- `scripts/ui/WarehouseUI.lua`

新增 UI 层局部状态：

```lua
local changedThisOpen_ = false
local lastCloseFlushTime_ = 0
```

在 `WarehouseUI.Show()`：

```lua
changedThisOpen_ = false
```

在以下成功操作后置 `true`：

- 从仓库取出成功。
- 从背包存入仓库成功。
- 整理仓库成功。
- 解锁仓库成功。
- 第二页存取、整理、解锁成功。

注意：

- 只有 `ok == true` 时置 `changedThisOpen_ = true`。
- 失败操作不得置变更标记。
- 单纯打开/关闭仓库不得置变更标记。

### P0-3：关闭仓库时如有变更，立即尝试保存

文件：

- `scripts/ui/WarehouseUI.lua`

在 `WarehouseUI.Hide()` 中，销毁 UI 前或后均可，但必须在函数返回前执行：

```lua
if changedThisOpen_ then
    EventBus.Emit("save_request")
    pcall(function()
        require("systems.save.SaveSession").Flush()
    end)
    pcall(function()
        local SaveSystem = require("systems.SaveSystem")
        if SaveSystem.loaded
                and SaveSystem.activeSlot
                and not SaveSystem.saving
                and not SaveSystem._retryTimer
                and not SaveSystem._disconnected then
            SaveSystem.Save()
        end
    end)
end
```

执行要求：

1. 不要在这里清 `BlackMarketSyncState`。
2. 不要在这里发 `game_saved`。
3. 不要在这里修改 `SaveSystem` dirty 生命周期。
4. 如果 `SaveSystem.Save()` 因断线、无 serverConn、retry、saving 被跳过，仓库 dirty 必须继续保留。
5. `changedThisOpen_` 可以在 Hide 末尾重置为 `false`，因为真正的业务保护状态在 `BlackMarketSyncState`。

风险控制：

- 直接调用 `SaveSystem.Save()` 可能增加保存频率。
- 因此必须检查 `not SaveSystem.saving` 和 `not SaveSystem._retryTimer`。
- 不允许循环重试。
- 不允许在关闭仓库时连续多次调用保存。

可选防抖：

```lua
if os.clock() - lastCloseFlushTime_ >= 3 then
    lastCloseFlushTime_ = os.clock()
    -- flush
end
```

禁止：

- 不要因为强制关闭仓库而允许出售。

### P0-5：保留现有全局仓库出售保护

## 7. P1 具体执行方案

P1 只在 P0 稳定后执行。

### P1-1：记录仓库 dirty 的时间和原因

文件：

- `scripts/systems/BlackMarketSyncState.lua`

可新增：

```lua
BlackMarketSyncState._dirtyWarehouseSince = nil
BlackMarketSyncState._dirtyWarehouseReason = nil
```

`MarkWarehouseOp(reason)` 时记录：

```lua
_dirtyWarehouseSince = os.clock()
_dirtyWarehouseReason = reason
```

`ClearAll()` 时清除。

新增只读接口：

```lua
function BlackMarketSyncState.GetWarehouseDirtyInfo()
    return {
        dirty = BlackMarketSyncState._dirtyWarehouse == true,
        since = BlackMarketSyncState._dirtyWarehouseSince,
        reason = BlackMarketSyncState._dirtyWarehouseReason,
    }
end
```

用途：

- UI 可在超过 10 或 15 秒后显示“存档未完成”而不是一直“请稍后”。
- 日志可定位是 `store`、`retrieve`、`sort` 还是 `unlock` 后卡住。

### P1-2：黑市出售拦截超时提示

文件：

- `scripts/ui/BlackMerchantUI.lua`

当 code 为 `warehouse_unsync` 时：

- 若 dirty 持续小于 10 秒：显示 `仓库道具变更尚未完成存档，暂不能出售`
- 若 dirty 持续大于等于 10 秒：显示 `仓库道具变更存档未完成，网络或服务器繁忙，暂不能出售`
- 若 `NetworkStatus.IsDisconnected()`：显示 `网络异常，仓库变更未完成存档，暂不能出售`

要求：

- 不自动放行。
- 不自动重放出售。
- 不把保存失败伪装成普通同步中。

### P1-3：日志收敛

建议日志：


## 8. 暂缓方案

### 8.1 暂缓：仓库 dirty 物品粒度放行

暂缓原因：

- 当前黑市出售服务端仍可能写回基于云端旧 `save_{slot}` 的快照。
- 仓库有任何未落盘变更时，允许任意黑市出售都可能导致旧快照覆盖未落盘变更。
- 如果只从“卖的是否同一个物品”判断，会低估服务端写回快照的风险。

进入条件：

必须先满足至少一个条件：

1. 黑市服务端出售不再写回完整角色快照。
2. 黑市服务端写回能基于版本号/CAS 检测旧快照。
3. 黑市服务端写回能 merge 客户端最新仓库变更。
4. 仓库系统本身服务端权威化。

在这些条件未满足前，不做物品粒度放行。

### 8.2 暂缓：修改 `BlackMerchantHandler.lua`

本轮不修改服务端黑市 handler。

原因：

- 上一轮 Phase 3 已刚完成服务端短锁和释放路径修复。
- 当前玩家反馈主要是客户端仓库 dirty 与黑市出售门禁体验问题。
- 服务端 handler 改动风险更高，容易影响真实经济交易。

### 8.3 暂缓：修改存档 schema

本轮不修改：

- `CURRENT_SAVE_VERSION`
- SaveDTO
- migration
- save key
- 云端存储结构

## 16. 最终结论

当前问题不是“仓库拦截不该存在”，而是：

1. 拦截提示太泛。
2. 仓库关闭没有形成明确的保存收口动作。
3. 保存失败或延迟时没有把真实原因暴露给玩家。
4. 测试没有覆盖这个体验和安全边界。

P0 正确方向是：

- 保留仓库未存档时禁止黑市出售。
- 关闭仓库有变更时立即尝试保存。
- 黑市出售被拦时显示“仓库道具变更尚未完成存档，暂不能出售”。
- 只在 `game_saved` 后解除门禁。

这能在不重新打开经济漏洞的前提下，显著改善玩家理解和恢复速度。

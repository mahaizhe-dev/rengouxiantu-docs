# 2026-06-17 黑市买入永久禁回售与全商品审计执行文档

## 0. 执行边界

本任务一切以当前代码为准，不以历史文档结论为准。

审计基准：`origin/main`，commit `eefc72b`。

本文件是交给执行 AI 的实现文档，目标是修复黑市买入物品被使用、消耗、出售后，又通过旧存档/旧快照卖回黑市的恶性闭环。

禁止只修 `gold_brick`。本次必须覆盖 `BMConfig.ITEMS` 中全部黑市商品。

## 1. 核心目标

1. 所有从黑市买入的商品，永久不能再卖回黑市。
2. 黑市买入商品仍允许走其正常用途。例如金砖可以在背包中出售换金币，令牌盒可以打开，粽子可以食用，技能书可以给宠物使用，装备可以穿戴。
3. 普通来源的同名商品不受影响。例如掉落/活动/GM/任务获得的金砖、粽子、令牌盒仍可按原规则卖回黑市。
4. 黑市买入来源和普通来源永远不能合并成同一堆叠。
5. 本地金币、灵韵、道具消耗发生后，不能被黑市服务端用旧 `save_{slot}` 快照写回覆盖。
6. 服务端必须硬校验，不能只靠 UI、临时锁、dirty 标记或历史 `SellGuard`。

## 1.5 评估修正与分层策略（基于 commit eefc72b 代码核实，权威口径）

本节为 2026-06-17 二次评估后补入，**优先级高于本文档其余章节**。其余章节如与本节冲突，以本节为准。

### 1.5.1 两个攻击链与本质

用户确认的两个最严重攻击链：

- **链路一（消耗品回售闭环）**：黑市买入消耗品 → 本地使用/消耗 → 再上架卖回黑市。
- **链路二（金砖洗金币 + 金币归零）**：黑市买入金砖 → 本地出售换金币 → 再上架卖回黑市；不仅套利，还导致已花掉的金币被恢复（消耗动作等效变 0 成本）。

两条链路的主语**都是黑市买入物**。经代码核实，本质如下：

- **回售套利的根因**：黑市买入物没有永久来源标记，本地消耗后服务端读到的旧 `save_{slot}` 快照里物品仍在，`held` 校验通过即放行回售。
- **金币归零的根因**：`BlackMerchantHandler.WriteSaveBack` 用**从云端读来的整份旧快照**覆盖写 `save_{slot}`，而 `player.gold` / `player.lingYun` 就嵌在这份快照里（`SaveSerializer.SerializePlayer`），于是本地已扣的金币被旧值盖回。`serverCloud` 无 CAS/版本号/条件写/子字段写，`Set(key,value)` 是整 key 覆盖。

> 关键结论：金币归零的**触发器**就是"黑市卖出成功 → 进入 WriteSaveBack"。只要黑市买入物**永远无法成功卖出**，就**根本不会进入 WriteSaveBack**，金币也不会被覆盖。因此 `bmNoResell` 服务端硬拒绝这一条线，可**同时根治链路一、链路二（含金币归零）**中所有以"黑市买入物"为主语的路径。

### 1.5.2 修复分两条线，安全强度不同，不可混为一谈

| 线 | 目标 | 安全边界 | 能否防作弊 | 本次定位 |
|---|---|---|---|---|
| **线 A：`bmNoResell` 永久来源标记** | 黑市买入物永久不可回售黑市 | 服务端 `HandleSell` 只统计/扣除 `bmNoResell~=true`，**真正的服务端硬边界** | **能** | **P0 主力，必须完成** |
| **线 B-服务端：写回收窄** | 黑市卖出写回不再覆盖最新货币/快照 | 写回前重读云端最新快照、在最新基础上施加扣除（无 CAS 下的尽力收窄） | 部分（窗口收窄到毫秒级，非真 CAS） | P0 加固，针对线 A 之外的普通来源残留竞态 |
| **线 B-客户端：未同步门禁** | 本地有未落盘货币/可卖物变更时阻断卖出 | 客户端 `BlackMarketSyncState`（**已存在**），flush 后放行 | **不能**（作弊者绕 UI 直发协议） | 体验兜底，仅防普通玩家时序意外 |

**务必牢记**：客户端阻断对作弊者无效，绝不能作为唯一防线。链路一、链路二的根治依赖**线 A 的服务端硬边界**；线 B-服务端解决"卖普通来源物时顺带把本地未落盘金币盖回"的残留路径。

### 1.5.3 对原文档的认知修正（执行前必读）

1. **`BlackMarketSyncState` 已存在**，不是"待新增"。它已实现 `MarkConsumeUsed` / `GetSellBlockReason`(L1 锁 + L2 全局未同步) / `IsBlocked`(消耗脏 + 仓库脏 + SaveSession 三合一) / `game_saved` 清脏。§4.8 中"新增 `EconomyDirtyState` 或扩展 `BlackMarketSyncState`"应改为"**复用并强化已有的 `BlackMarketSyncState`**"，**不要新建 `EconomyDirtyState`**。

2. **L2 阻断已被 HOTFIX-SELL-01 主动关闭**：`BlackMerchantUI.lua` 中确认弹窗与卡片按钮均已注释掉 `global_unsync`(L2) 阻断，只保留 `locked_item`(L1)。这说明 §4.8 想要的"未同步阻断卖出"曾上线、因体验问题被关闭。**重启 L2 前必须解决误拦**：本地有未落盘变更时不要直接弹"请稍后再卖"，而应**主动触发一次 `save_request` 并在 `game_saved` 后自动放行**，避免普通玩家被反复拦截。

3. **"禁止不同来源合并"是 P0 最高风险点，不是普通条目**。当前合并入口（`consumables.lua` 的 `AddConsumable` 主循环、`InventorySortUtil.GetMergeKey`）**只看 `TradeLock.IsLocked + bmLockBatchId`，完全不看任何永久字段**。一旦引入 `bmNoResell` 而漏改任一合并入口，"黑市买入堆叠 + 普通堆叠合并 → 保留 `bmNoResell=nil`"会**把黑市买入物洗白成可卖**，直接击穿整个方案。详见修订后的 §4.4。

4. **装备改造不会丢 `bmNoResell`**（§3.8 的担忧已证伪）：`ForgeSystem.GenerateSpecialEquipment` / `ForgeUI` 洗练 / `TemperUI` 附灵 全部**原地改字段**，未被覆写的 `bmNoResell` 会自然保留。装备线只需保证"买入时写入"+"序列化保留"+"卖出按 `bmNoResell` 排除"，无需为改造做特殊处理，但**保留一条改造后回归测试**。

5. **`gold_bar` 不是黑市商品**，无法卖回黑市，§4.8 对它的覆盖条目冗余无效，删除。真正要保护的是 `gold_brick`（`BMConfig` 第 246 行，本地售价 10 万金币）。

6. **代码中没有"银币"货币**。货币只有 金币(gold)、灵韵(lingYun)、仙石(xianshi)。链路二里的"银币"按"金币"处理。

7. **`ConsumeConsumable` 刻意不发 `save_request` 是设计决定且有测试固化**（`test_black_merchant_warehouse_consistency.lua` 明确断言"消耗路径不发 save_request"）。**不要强行改成每次消耗都存档**（会引发高频写盘并破坏现有测试）。消耗后维持"置脏 `_dirtyConsume`"现状即可，由线 B 的门禁/写回保证一致性。

## 2. 当前代码问题

### 2.1 服务端卖出没有永久来源校验

文件：`scripts/network/BlackMerchantHandler.lua`

当前 `HandleSell` 主流程读取玩家存档背包后，只做：

- `BackpackUtils.HasAnyLockedItem(backpack, consumableId)`
- `held < amount`
- `stock + amount > maxStock`
- `RemoveFromBackpack(backpack, consumableId, amount)`
- `MoneyAdd(userId, "xianshi", total)`
- `MoneyAdd(SYSTEM_UID, stockKey, amount)`

问题：

- `SellGuard` 虽然被 `require`，但当前卖出主链没有调用 `SellGuard.CheckSellAllowed()`。
- `held` 是总持有量，不区分普通来源和黑市买入来源。
- `RemoveFromBackpack` 只跳过临时锁，不跳过永久黑市来源。
- 装备类卖出按 `equipId` 统计和扣除，也没有永久来源校验。

### 2.2 当前锁不是安全边界

文件：

- `scripts/systems/BlackMarketTradeLock.lua`
- `scripts/systems/save/SaveSerializer.lua`
- `scripts/network/BackpackUtils.lua`
- `scripts/systems/inventory/consumables.lua`

当前锁字段：

```lua
bmLockUntil
bmLockSource
bmLockBatchId
```

问题：

- `SaveSerializer.SerializeItemFull()` 明确省略这些字段。
- `game_saved` 后会清除锁。
- 5 分钟锁过期后也会失效。
- 重登/保存成功/旧快照写回后，锁不能证明物品来源。

结论：现有临时锁可以保留为 UI 体验提示，但不能作为安全边界。

### 2.3 本地使用/出售路径太多，不能逐个靠补丁拦

黑市商品覆盖大量本地路径：

- `InventorySystem.ConsumeConsumable`
- `inventory/batch_use.lua`
- `_SellBatchConsumable`
- `_UseTokenBox`
- `_UseBatchPremiumZong`
- 宠物技能书消耗
- 炼丹材料消耗
- 神器碎片消耗
- 打造材料消耗
- 活动兑换消耗
- 装备穿戴/仓库存取/改造

继续逐条补丁会遗漏。正确做法是建立统一来源标记和统一服务端可回售统计。

## 4. 统一实现方案

### 4.1 新增永久来源字段

所有黑市买入物品都必须写入：

```lua
bmNoResell = true
bmSource = "black_market_buy"
bmBuyAt = os.time()
bmBuyItemId = itemId
```

字段语义：

- `bmNoResell=true`：永久不可卖回黑市。
- `bmSource="black_market_buy"`：来源为黑市买入，便于日志和排查。
- `bmBuyAt`：买入时间戳，便于追踪。
- `bmBuyItemId`：买入时黑市商品 ID，装备类可用来对应 `equipId`。

这些字段不是临时锁，不得在保存成功、重登、锁过期、排序、仓库存取时清除。

### 4.2 序列化必须落盘

修改文件：`scripts/systems/save/SaveSerializer.lua`

在 `SerializeItemFull(item)` 中保留：

```lua
bmNoResell = item.bmNoResell,
bmSource = item.bmSource,
bmBuyAt = item.bmBuyAt,
bmBuyItemId = item.bmBuyItemId,
```

注意：

- 不要放进 `bmLockUntil / bmLockSource / bmLockBatchId` 的省略逻辑。
- 旧存档中没有这些字段时，默认视为普通来源，可回售。

### 4.3 黑市买入写入永久字段

修改文件：

- `scripts/network/BackpackUtils.lua`
- `scripts/systems/inventory/consumables.lua`
- `scripts/network/BlackMerchantHandler.lua`
- `scripts/ui/BlackMerchantUI.lua`

服务端买入写存档：

- 消耗品：`AddLockedNewStack()` 或新函数必须支持写入 `bmNoResell` 系列字段。
- 装备：`LootSystem.CreateSpecialEquipment(cfg.equipId)` 后，必须给每件 `newEquip` 写入 `bmNoResell=true` 等字段。

客户端收到买入成功后的本地背包同步：

- 装备类本地创建的新装备也必须写入永久字段。

要求：

- 服务端和客户端写法一致。
- 如果客户端本地同步失败，服务端存档仍必须正确。
- 不能只改客户端。

### 4.4 禁止不同来源合并（P0 最高风险点，优先于其他 P0 实现并测试）

这是整个方案最容易被击穿的地方。当前合并逻辑**完全不看任何永久字段**，只看锁状态与批次：

- `consumables.lua` 的 `AddConsumable` 主循环合并条件：相同 `consumableId` + `not TradeLock.IsLocked(item)` + 未满。
- `InventorySortUtil.GetMergeKey`：未锁定堆叠合并键 = `consumableId|unlocked`；锁定堆叠 = `consumableId|locked|batchId`。

若引入 `bmNoResell` 而漏改**任一**入口，"黑市买入堆叠（`bmNoResell=true`）"会与"普通堆叠（`bmNoResell=nil`）"合并，结果堆叠的 `bmNoResell` 取决于合并实现——一旦取到 `nil`，黑市买入物即被**洗白成可卖**，方案彻底失效。

必须同时改造以下**全部三类合并/整理入口**，缺一不可：

1. `scripts/systems/inventory/consumables.lua` → `AddConsumable` 主循环：合并候选判断追加 `item.bmNoResell ~= (新增物的 bmNoResell)` 即跳过。黑市买入仍走 `AddConsumableLockedNewStack` 强制新建堆叠，不参与合并。
2. `scripts/systems/InventorySortUtil.lua` → `GetMergeKey`：合并键追加来源维度，例如 `bmNoResell` 为 true 时键加 `|noresell`，确保整理时不与普通来源合并。
3. `scripts/network/BackpackUtils.lua` → `AddToBackpack` / 任何会把数量并入现有堆叠的函数：同样按 `bmNoResell` 隔离。

统一合并门规则（任意入口判断两堆叠 a、b 能否合并时调用）：

```lua
local aNoResell = a and a.bmNoResell == true
local bNoResell = b and b.bmNoResell == true
if aNoResell ~= bNoResell then
    return false   -- 来源不同，永不合并
end
```

合并保留规则：

- 普通来源之间可以合并。
- 黑市买入来源之间可以合并，但合并后**必须保持 `bmNoResell=true`**（取任一合法黑市来源字段，建议保留最早 `bmBuyAt`）。
- 装备为独立对象不堆叠，不涉及合并，但仓库存取/整理时不得清除 `bmNoResell`。

测试要求：§6.3 合并隔离测试必须对每类（金砖/令牌盒/粽子/技能书/草药）断言"普通 + 黑市来源不会合并、且黑市堆叠合并后仍 `bmNoResell=true`"。

### 4.8 金币归零防御（线 B：服务端写回收窄 + 客户端门禁）

> 重要：本节解决的是**线 A 之外的残留竞态**——即卖**普通来源**物时，服务端 `WriteSaveBack` 用旧快照覆盖 `save_{slot}` 把本地未落盘金币盖回。对于链路二里以"黑市买入金砖"为主语的金币归零，**线 A（`bmNoResell` 服务端拒绝 → 不进入 WriteSaveBack）已根治**，本节是纵深防御。

#### 4.8.1 根因（已核实）

- `player.gold` / `player.lingYun` 嵌在 `coreData.player`，**整份存进 `save_{slot}` 一个 key**（`SaveSerializer.SerializePlayer`）。
- `BlackMerchantHandler.WriteSaveBack` 用 `Session.ServerGetSlotData` 读来的**整份旧快照**做 `serverCloud:BatchSet():Set("save_"..slot, saveData)`，连 `player.gold` 一起覆盖。
- `serverCloud` 无 CAS/版本号/条件写/子字段写，`Set` 是整 key 覆盖。仙石 `xianshi` 与系统库存走 `MoneyAdd` 原子增量，不在快照里，安全。
- `GainGold` / `GainLingYun` 只改内存不落盘，存在"卖出读快照 → 写回"窗口；窗口内若客户端已落盘新金币，会被旧值盖回；玩家若在窗口内断线，金币增量永久丢失。

#### 4.8.2 服务端写回收窄（核心加固，必须做）

修改文件：`scripts/network/BlackMerchantHandler.lua`

把"读旧快照 → 改 → 写旧快照"改为"**算出本次扣除项 → 写回前重读最新快照 → 在最新快照上施加扣除 → 写回**"：

1. `HandleSell` 第一次 `ServerGetSlotData` 读到的快照**仅用于校验**（持有量、可回售数量、库存）。
2. 校验通过后，确定本次要扣除的具体内容（`consumableId`+扣除明细 / 装备 `equipId`+扣除槽位）。
3. `WriteSaveBack` 内部**重新 `BatchGet` 读取一次云端最新 `save_{slot}`**，得到 `saveDataLatest`。
4. 在 `saveDataLatest.backpack`（最新背包）上施加扣除，**而不是在第一次读到的旧快照上**。这样 `player.gold` / `player.lingYun` 以及 backpack 外其他字段全部保留云端最新值。
5. `Set("save_"..slot, saveDataLatest)` 写回。

这是 `serverCloud` 无 CAS 约束下的"重读-应用-写回"收窄：把覆盖窗口从"整个卖出流程"压缩到"重读到写回"的毫秒级。它不是真 CAS，理论窗口仍在，但配合线 A（黑市买入物压根不进 WriteSaveBack）后，残留面已极小。

边界处理：

- 重读后若 `saveDataLatest` 里的可回售数量已不足（玩家在窗口内已消耗），**放弃本次卖出并返回错误，不发仙石、不加库存**。
- 重读失败时**不写回、不发仙石**，返回错误，宁可让玩家重试，绝不用旧快照写回。

#### 4.8.3 客户端未同步门禁（复用已有 `BlackMarketSyncState`，体验友好）

> 不要新建 `EconomyDirtyState`。`BlackMarketSyncState` 已提供 `MarkConsumeUsed` / `MarkWarehouseOp` / `GetSellBlockReason` / `IsBlocked` / `game_saved` 清脏。HOTFIX-SELL-01 关闭了它的 L2 阻断，重启时必须按下面方式避免误拦。

修改文件：`scripts/systems/inventory/batch_use.lua`、`scripts/ui/BlackMerchantUI.lua`、`scripts/systems/BlackMarketSyncState.lua`

必须覆盖的本地变更：

- `gold_brick` 本地卖金币（`_SellBatchConsumable`）。
- `xianjie_premium_zong` 食用（`_UseBatchPremiumZong`）。
- 令牌盒打开（`_UseTokenBox`）。
- 活动道具兑换、技能书/草药/材料/碎片 `ConsumeConsumable()`（已置 `_dirtyConsume`）。
- 金币 `GainGold()` / `GainLingYun()` / `SpendGold()` 变更。

（已删除原"`gold_bar` 本地卖金币"：`gold_bar` 不在 `BMConfig.ITEMS`，无法卖回黑市，覆盖它无效。）

重启 L2 的正确姿势（避免重蹈 HOTFIX-SELL-01 的误拦）：

1. 本地发生上述变更后置脏（消耗走 `MarkConsumeUsed`；金币变更新增一个轻量脏标记，如 `MarkEconomyUsed`，并入 `IsBlocked`）。
2. 玩家点黑市卖出时，若 `GetSellBlockReason` 返回 `global_unsync`：**不直接弹"请稍后再卖"**，而是**主动 `EventBus.Emit("save_request")` 触发一次落盘**，并提示"正在同步…"，待 `game_saved` 清脏后**自动重试该次卖出**或恢复按钮可点。
3. 仅当 flush 在合理超时内仍未完成才提示失败。

提示文案：

```text
物品或金币变动正在同步，请稍后重试
```

注意：客户端门禁只防普通玩家时序意外，**防不住作弊**（作弊者直发协议）。真正的边界是 §4.5/§4.6 的服务端可回售校验 + §4.8.2 的写回收窄。

#### 4.8.4 关于 `ConsumeConsumable` 是否触发保存

维持现状：`ConsumeConsumable` **不发 `save_request`**（设计决定，且 `test_black_merchant_warehouse_consistency.lua` 已固化"消耗路径不发 save_request"）。它只置 `_dirtyConsume`，由 §4.8.3 门禁在卖出前 flush。**不要**为黑市商品消耗强行加 `save_request`，否则破坏既有测试且引发高频写盘。`_SellBatchConsumable` / `_UseTokenBox` / `_UseBatchPremiumZong` 这类"产生货币/重大状态变化"的操作可在成功后显式 `save_request`，与消耗语义分开对待。

### 4.9 不要破坏普通来源交易

普通来源包括：

- 掉落
- 活动奖励
- 任务奖励
- GM 发放
- 炼制产出
- 兑换产出
- 旧存档已有物品

这些物品默认 `bmNoResell ~= true`，仍可按原黑市规则卖出。

不要把某个 `itemId` 全局禁售，除非产品明确决定该商品永久下架。

4. 合并、整理、仓库存取、序列化必须保留 `bmNoResell` 字段。
5. 不允许新增绕过 `BlackMerchantHandler.HandleSell` 的黑市卖出入口。

如果新增商品类型无法走现有统一入口，例如新增非消耗品、非装备的特殊对象类型，必须先补齐：

- 黑市买入来源标记写入。
- 序列化字段保留。
- 可回售数量统计。
- 只扣可回售来源。
- UI 可卖数量显示。
- 全量遍历测试。

否则该商品不得加入 `BMConfig.ITEMS`。

### 9.3 `itemType` 扩展必须 fail-closed

当前黑市商品主要分为：

- 默认消耗品：`itemType == nil`
- 装备：`itemType == "equipment"`

如果未来新增 `itemType`，服务端卖出必须默认拒绝未知类型：

```lua
if cfg.itemType ~= nil and cfg.itemType ~= "equipment" then
    SendError(connection, SaveProtocol.S2C_BlackMerchantResult, "该商品类型暂不支持出售")
    return
end
```

只有当新类型完成来源标记、可回售统计、扣除、序列化、测试后，才允许放行。

### 9.4 测试必须自动覆盖新增商品

测试禁止手写固定商品清单作为唯一覆盖方式。

必须至少有一个测试遍历：

```lua
for _, itemId in ipairs(BMConfig.ITEM_IDS) do
    -- 黑市买入 -> bmNoResell=true
    -- 序列化仍保留
    -- 查询 held 不计入
    -- 服务端卖出拒绝黑市来源
    -- 普通来源仍可卖出
end
```

验收标准：

- 新增任意 `BMConfig.ITEMS` 商品后，如果没有被该遍历测试覆盖，测试必须失败。
- 新增商品如果能从黑市买入但没有 `bmNoResell=true`，测试必须失败。
- 新增商品如果 `held` 把黑市来源算成可卖数量，测试必须失败。
- 新增商品如果服务端会扣黑市来源并发仙石，测试必须失败。

### 9.5 新增商品 PR 检查清单

任何新增黑市商品的提交必须在说明中回答：

1. 商品 ID 是否已加入 `BMConfig.ITEMS`。
2. 商品类型是消耗品、装备，还是新类型。
3. 黑市买入后是否自动写入 `bmNoResell=true`。
4. 是否会和普通来源同类物品合并，答案必须是不会。
5. 是否能本地使用、出售、打开、消耗、穿戴、仓库存取。
6. 本地使用/消耗后是否触发保存或进入经济未同步阻断。
7. 全量 `BMConfig.ITEM_IDS` 测试是否通过。

如果这些问题没有答案，不允许合并。

### 9.6 不允许用配置豁免绕过

不得新增类似以下配置绕过永久禁回售：

```lua
allowResellAfterBuy = true
blackMarketBuyCanResell = true
skipNoResell = true
```

如果产品未来确实需要“黑市买入后可再次卖回黑市”的商品，必须单独设计服务端权威流水或批次系统，不能复用当前背包快照模型。

## 10. 推荐提交说明

建议拆成至少三个提交：

1. `fix(black-market): persist no-resell source for black-market purchases`
2. `fix(black-market): sell only resellable inventory and preserve normal sources`
3. `test(black-market): cover all 123 items no-resell and economy sync`

如果只能一次提交，提交信息应明确：

```text
fix(black-market): permanently block reselling black-market purchases
```

# 2026-05-13 R3-GATE 联合发布门禁收口任务卡

## 1. 任务定位

这是 `R3 + BM-S2R` 联合发布前的门禁收口任务。

本任务只做一件事：

- 查清并收口 `r3_service_shell_contract` 当前 `1/40` 的失败

本任务不是新一轮结构重构，也不是黑市新功能任务。目标只是把联合发布的阻塞点从“预存失败”收成“已解释且可放行”或“已修复归零”。

---

## 2. 当前背景

当前已知状态：

1. `BM-S2R` 已通过，黑市未同步门已完成精确收口
2. `R3` 代码已通过上一轮复核
3. 但 `BM-S2R` 完成报告附带的原始测试输出显示：
   - `python3 scripts/tests/_run_via_lupa.py`
   - `python3 scripts/tests/_run_via_lupa.py --all`
   均存在：
   - `r3_service_shell_contract: 1/40 failed`

因此当前不能直接把 `R3 + BM-S2R` 作为“全绿联合发布包”处理。

---

## 3. 任务目标

本任务只允许达成下面 2 种结果之一：

1. **修复该失败**，让 `r3_service_shell_contract` 回到 `40/40`
2. **严格证明该失败是测试问题或非发布阻塞项**，并在报告中给出足够证据与边界说明

如果两者都做不到，本任务判定为不通过。

## 7. 固定执行步骤

### 步骤 1：先定位失败点

必须在完成报告里明确写出：

1. 失败的是哪一个子测试
2. 失败断言是什么
3. 根因属于哪一类：
   - 真实代码回归
   - 测试假设过期
   - stub/夹具问题
   - 旧行为保留但测试期望错误

如果这 4 点写不清楚，不允许进入改代码阶段。

### 步骤 2：判断是否影响联合发布

必须明确回答：

1. 该失败是否影响 `FetchSlots / LoadGame / SaveGame` 线上行为
2. 该失败是否影响 `BM-S2R` 依赖的 `save_request -> SaveGame 落盘` 语义
3. 该失败是否可能影响黑市当前已上线止血效果

如果任一答案是“可能影响”，优先按代码问题修复；不能只改测试。

### 步骤 3：最小修复

只做最小收口：

- 若是测试错误：改测试
- 若是 `R3` service 壳等价性问题：只修对应 service / 委托边界

不允许借机继续重构。

### 步骤 4：重新跑门禁

至少提交：

```bash
python3 scripts/tests/_run_via_lupa.py
python3 scripts/tests/_run_via_lupa.py --all
```

并附原始输出。

## 8. 成功标准

满足以下全部条件才算通过：

1. `r3_service_shell_contract` 失败原因被明确定位
2. 若是代码问题，已修复并验证
3. 若是测试问题，已给出不影响联合发布的充分证据
4. blocking 输出中不再出现“未知性质的 R3 失败”
5. 完成报告明确给出：
   - `R3 + BM-S2R` 是否允许联合发布
   - 原因是什么

---

## 9. 失败判据

出现以下任一情况视为失败：

1. 只给结果，不解释根因
2. 顺手改了黑市逻辑
3. 顺手改了存档结构/协议
4. 没有原始测试输出
5. 仍然把失败描述成“预存问题”但没有证据

---

## 10. 完成报告必须提交的内容

完成报告必须包含：

1. `commit SHA / parent SHA`
2. 失败子测试定位
3. 根因分类
4. 实际修改文件清单
5. 原始测试命令
6. 原始测试输出
7. `rules.md` 符合性说明
8. 对黑市一致性的影响评估
9. 最终结论：
   - `允许联合发布`
   - 或 `不允许联合发布`

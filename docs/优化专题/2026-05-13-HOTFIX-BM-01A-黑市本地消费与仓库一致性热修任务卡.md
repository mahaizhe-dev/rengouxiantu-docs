# 2026-05-13 HOTFIX-BM-01A 黑市本地消费与仓库一致性热修任务卡

> 这是 `HOTFIX-BM-01` 的升级版任务卡。

> 这张卡把“仓库存取 + 本地使用/消耗”两类未同步漏洞一起纳入。

## 1. 任务定义

任务 ID：`HOTFIX-BM-01A`

任务名称：`黑市-本地消费与仓库一致性热修`

变更类型：`线上漏洞热修`

优先级：`P0`

目标：
- 阻断“黑市卖出成功，但本地真实道具状态已变化”的漏洞链路
- 同时覆盖两类已确认路径：
  - 仓库存取导致的背包/服务端快照分裂
  - 本地使用/消耗导致的背包/服务端快照分裂
- 让黑市卖出在同步完成前进入明确禁止状态

---

## 2. 本次相对 HOTFIX-BM-01 的新增判断

新增确认：

1. `InventorySystem.ConsumeConsumable()` 只改本地背包并发 `consumable_used`，不会自动形成可靠同步闭环
2. 一批“使用类道具”直接依赖它，例如：
   - `UseLingyunFruit()`
   - `UseExpPill()`
   - `UseGuardianToken()`
   - `_UseTokenBox()`（乌堡令盒/沙海令盒/太虚令盒）
3. 因为 `save_request` 只是把存档标记为 dirty，不是立即完成服务端落盘，所以即使某些系统后续会发 `save_request`，中间仍存在可利用窗口
4. 因此本次热修不能只盯仓库，必须把“本地已消费但未同步”一并挡住

---

## 3. 成功标准

成功标准只有 10 条：

1. 仓库存取/扩容成功后，进入“黑市相关背包未同步”状态
2. 黑市敏感道具在本地成功使用/消耗后，进入“黑市相关背包未同步”状态
3. 至少明确覆盖以下典型风险项：
   - `wubao_token_box`
   - `sha_hai_ling_box`
   - `taixu_token_box`
4. 未同步状态下，黑市卖出请求被客户端明确阻止
5. 阻止时有明确提示文案，例如“背包数据未同步，请先保存后再进行黑市交易”
6. 仓库相关操作成功后，补发一次 `save_request`
7. `game_saved` 和 `game_loaded` 后，未同步状态会被清除
8. 黑市卖出成功后的本地同步，若本地扣除失败，不允许继续发 `save_request`
9. 不修改协议、不修改服务端 handler、不修改存档结构、不 bump 版本号
10. 至少新增一组纯 Lua 自动化测试覆盖本次热修的核心语义

## 7. 固定执行步骤

### 步骤 1：把“黑市相关背包未同步”做成显式语义

至少要覆盖：

1. 仓库存入成功
2. 仓库取出成功
3. 仓库扩容成功
4. 黑市敏感道具本地成功使用/消耗
5. `game_saved`
6. `game_loaded`

要求：

- 前 4 项置脏
- 后 2 项清脏
- 生命周期清楚
- 不依赖 UI 是否已打开

### 步骤 2：仓库操作成功后补 `save_request`

在以下成功路径后补发：

1. `StoreItem`
2. `StoreItemToSlot`
3. `RetrieveItem`
4. `UnlockNextRow`

注意：
- 这只是缩短窗口，不是根治
- 不要因为补了 `save_request` 就取消“未同步状态”

### 步骤 3：把“本地消费但未同步”纳入同一套同步门

本任务不要求给所有消耗品都加特殊逻辑。  
本任务要求至少覆盖黑市敏感道具，尤其是：

1. `wubao_token_box`
2. `sha_hai_ling_box`
3. `taixu_token_box`

建议做法二选一：

1. 在 `InventorySystem.ConsumeConsumable()` 成功后识别是否属于黑市敏感道具并置脏
2. 或在更上层的使用入口里单独置脏

但最终必须满足：
- 这类“已使用但未同步”的黑市可卖物，不会再被黑市卖出

### 步骤 4：黑市卖出前增加硬性前置校验

卖出前如果本地背包未同步：

- 不发送卖出请求
- 不进入确认后的真实提交
- 给出明确提示

最低要求：
- 在真正发送 `SendSell(...)` 前拦住

### 步骤 5：黑市卖出成功后的本地同步失败，必须硬拦截

这是本任务最关键的止血点。

必须改成：

1. 本地扣除结果必须可判断成功/失败
2. 如果失败：
   - 不发 `save_request`
   - 记录高优先级错误日志
   - 立即给出错误提示
   - 立即触发一次黑市数据刷新
3. 如果成功：
   - 才允许继续 `save_request`

### 步骤 6：不要扩大到服务端修复

如果执行中判断：
- 客户端热修只能缓解，不能防住强对抗客户端

这是合理结论。  
但本任务不允许直接扩成服务端大改。  
正确做法是：完成客户端热修，并在完成报告里单列“服务端仍存在的理论攻击面”。

### 步骤 7：补纯 Lua 自动化测试

新增：

- `scripts/tests/test_black_merchant_warehouse_consistency.lua`

最低覆盖：

1. 仓库存入成功后置脏并发 `save_request`
2. 仓库取出成功后置脏并发 `save_request`
3. 本地使用黑市敏感道具成功后置脏
4. `game_saved` 后清脏
5. `game_loaded` 后清脏
6. 未同步状态下，黑市卖出前置校验返回禁止
7. 本地卖出扣除失败时，不发 `save_request`
8. 本地卖出扣除成功时，允许发 `save_request`

### 步骤 8：接入现有门禁

至少跑：

```bash
python3 scripts/tests/_run_via_lupa.py
python3 scripts/tests/_run_via_lupa.py --all
```

blocking 不能变红。

## 10. 本任务完成后的判断门

本任务通过后，只说明：

- 这条“正常操作可复现”的刷收益路径已经被热修收口

不代表：

- 黑市与本地背包已经完全服务端权威一致

如果完成报告明确指出“改包客户端仍可绕过客户端前置校验”，这是合理结论。  
不要因为这个结论就否定本任务本身的价值。

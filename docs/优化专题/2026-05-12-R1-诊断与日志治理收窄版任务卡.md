## 1. 任务定义

任务ID：`R1`

任务名称：`诊断与日志治理（收窄版）`

变更类型：`结构准备 + 低风险护栏`

目标：

- 给后续 `R2 / R3 / R4` 提供最小可用的日志分级与诊断开关
- 先收口存档/网络最危险链路的日志输出
- 在不改协议、不改业务逻辑、不改数据结构的前提下，提高线上问题定位能力

---

## 2. 为什么先做这个

当前项目已经上线，且用户无法稳定回传遥测数据。

这意味着后续如果继续做：

- `SaveSystemNet` 状态机收口
- 服务端存档主链拆分
- 排行/奖励事务收口

那么定位问题时主要只能依赖：

1. 自有账号复测
2. 玩家反馈
3. 运行日志

现在关键链路里的日志现状是：

1. `print()` 数量多
2. 级别混乱
3. 没有统一开关
4. 很难做到“默认输出必要信息，深入排查时再开诊断”

所以这一步的价值不是“代码更漂亮”，而是：

- 让后续高风险任务有可用诊断手段
- 降低线上问题只能靠猜的概率

---

## 3. 本任务的成功标准

成功标准只有 8 条：

1. 新增统一日志入口模块，且该模块是纯叶模块，不依赖业务代码。
2. 至少支持 4 个等级：
   - `error`
   - `warn`
   - `info`
   - `diag`
3. 默认行为下：
   - `info` 及以上可输出
   - `diag` 默认关闭
4. `server_main.lua` 中至少以下两条链路改为统一日志入口：
   - `HandleFetchSlots`
   - `HandleSaveGame`
5. `SaveSystemNet.lua` 中存档/加载/迁移状态相关日志改为统一日志入口。
6. `CloudStorage.lua` 中网络模式关键分支日志改为统一日志入口，尤其是：
   - 非存档 `BatchSet`
   - 超时清理
   - 无连接/回退路径
7. 本任务不得改变协议、回包字段、数据结构、存档行为。
8. blocking gate 继续保持绿色。

只要 8 条里有任意一条不满足，本任务不算完成。

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：定义最小 Logger 边界

新增 `scripts/utils/Logger.lua`，但只提供最小能力：

1. 4 个等级函数：
   - `error(tag, msg)`
   - `warn(tag, msg)`
   - `info(tag, msg)`
   - `diag(tag, msg)`
2. 运行时等级/诊断开关读取
3. 输出格式统一

固定要求：

1. 不依赖业务模块
2. 不依赖引擎对象
3. 纯 Lua 环境可加载
4. 不做复杂格式系统

建议输出格式保持简单，例如：

```text
[ERROR][SaveGame] message
[WARN][CloudStorage] message
[INFO][FetchSlots] message
[DIAG][SaveSystemNet] message
```

### 步骤 2：明确默认策略

本任务必须固定默认行为：

1. `error / warn / info`
   - 默认开启
2. `diag`
   - 默认关闭

允许通过简单的运行时开关启用 `diag`，但不要引入复杂配置系统。

可接受方案：

1. `Logger.SetDiagEnabled(true/false)`
2. 或简单常量/环境式开关

不可接受方案：

1. 为了这一步引入新的全局配置体系
2. 为了这一步重做 `FeatureFlags`

### 步骤 3：只改三处关键链路

本任务不是全量替换。

只要求先覆盖：

1. `server_main.lua`
   - `HandleFetchSlots`
   - `HandleSaveGame`
2. `SaveSystemNet.lua`
   - `FetchSlots / Load / CreateCharacter / DeleteCharacter`
   - 网络未就绪 / 重试 / needMigration 已废弃提示
3. `CloudStorage.lua`
   - `BatchSet` 中的非存档路径
   - 超时清理
   - `Get / SetInt` 在网络模式下的降级输出

原则：

1. 保持原信息语义
2. 只做等级归类与入口统一
3. 不顺手删所有旧日志

### 步骤 4：划分日志等级

最低要求如下：

#### `error`

用于：

- 保存失败
- 回调异常
- 请求超时后的错误回调
- 无连接且无法继续重试

#### `warn`

用于：

- 非存档 `BatchSet` 被跳过或不支持
- 已废弃路径仍被命中
- fallback / 降级路径触发
- 可恢复但不正常的状态

#### `info`

用于：

- 关键主链开始/结束
- 成功保存
- 成功读取槽位
- 关键模式切换

#### `diag`

用于：

- 大体量 JSON / slotsIndex 打印
- 高频状态细节
- 仅排障时才需要的信息

### 步骤 5：建立纯 Lua 冒烟测试

新增：

- `scripts/tests/test_logger_smoke.lua`

最低覆盖：

1. Logger 模块可在纯 Lua 下加载
2. 4 个等级函数存在
3. 默认情况下 `diag` 不输出
4. 打开 `diag` 后能输出
5. 输出前缀格式稳定

这一步不要求对 `server_main.lua` 做高复杂度日志捕获测试。
第一阶段只验证 Logger 模块本身行为稳定。

### 步骤 6：接入门禁

如果 `test_logger_smoke.lua` 为纯 Lua 测试，应接入现有 runner。

至少跑：

```bash
python3 scripts/tests/_run_via_lupa.py --gate=blocking
python3 scripts/tests/_run_via_lupa.py --all
```

如果日志测试暂时不进入 blocking，完成报告里必须说明原因。

## 8. 本任务的关键限制

这一步不是“完整 observability 体系建设”。

这一步只做：

1. 最小 Logger 入口
2. 关键链路等级化
3. 诊断开关最小化
4. 纯 Lua 冒烟测试

不做：

1. 全项目替换 `print`

3. 改协议埋点
4. 改性能监控系统

---

## 9. 本任务的停手条件

出现以下情况，停止编码，改交付问题说明：

1. 为了接入 Logger，必须改业务流程或协议字段
2. `server_main.lua` 的某些日志已经和控制流强耦合，替换会改变时序
3. 无法在纯 Lua 下测试 Logger
4. 需要修改超过 4 个核心运行时代码文件之外的额外业务模块
5. 发现 `SaveSystemNet` 的日志问题本质上已经属于状态机逻辑问题，无法在本任务内只靠等级治理收口

如果停手，仍然要交付：

1. 哪些日志点不能安全替换
2. 哪些链路需要推迟到 `R2` 一并处理
3. 为什么 `R1` 不能独立完成

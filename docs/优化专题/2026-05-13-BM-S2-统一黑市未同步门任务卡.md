# 2026-05-13 BM-S2 统一黑市未同步门任务卡

## 1. 任务定位

这是黑市专项的第一张运行时代码任务卡。

本任务只解决一件事：

- 把当前黑市卖出前的客户端门禁，从“仓库 dirty + 3 个敏感道具白名单”升级为“统一黑市未同步门”

这一步是黑市专项的最小运行时收口，不是最终权威修复。

本任务**不**解决：

- 服务端 `HandleSell` 的最终权威校验缺口
- 自动回收重构
- 仙石流水审计重构
- 黑市协议重写

## 2. 前置结论

本任务必须严格继承以下冻结结果：

- `BM-S1 黑市可卖物与本地变更路径风险矩阵`
- `BM-S1 黑市一致性与仙石自动回收合同 v1`

已知当前最高优先级风险物为：

- 附灵玉 4 种（P0）
- `gold_brick` 1 种（P0）

已知当前高风险面总量为：

- `53 / 79` 个黑市可卖物存在未被门禁覆盖的本地变更路径

因此，`BM-S2` 不能再做成“再补几个物品名单”。

## 3. 本次目标

### 3.1 核心目标

建立一个**统一的黑市未同步门**，使黑市卖出在以下情形下被客户端阻断：

- 本地背包中黑市可卖物刚被消耗/转化，但尚未完成可靠同步
- `SaveSession` 仍处于活跃延迟窗口
- 仓库相关操作刚发生，尚未完成可靠同步

### 3.2 必须达到的结果

至少要覆盖以下 5 条路径：

1. 仓库存入后卖出同物
2. 仓库取出后卖出同物
3. 令牌盒使用后卖出
4. `gold_brick` 本地卖钱后卖出
5. 附灵玉在 `TemperUI` 消耗后卖出

## 7. 设计约束

### 7.1 不允许的方案

以下方案直接视为不通过：

- 继续扩 `_BM_SENSITIVE_CONSUMABLES`
- 新增第二个、第三个物品白名单
- 只修 `gold_brick`
- 只修附灵玉
- 只修 `ConsumeConsumable()`，不处理 `SaveSession` 活跃窗口
- 继续复用 `WarehouseSystem._dirty` 作为全部黑市语义的最终承载

### 7.2 允许的方案方向

可接受的实现方向只有两类：

1. **独立黑市未同步状态模块**
   - 例如 `BlackMarketSyncState`
   - 用于表达“当前存在会影响黑市卖出可卖性的未同步本地变化”

2. **以 SaveSession 活跃状态为主门禁**
   - 但必须证明：
     - 能覆盖仓库链路
     - 能覆盖 `ConsumeConsumable` 链路
     - 不会误伤大量无关路径到不可用

如选第二种，完成报告必须单列“为什么不需要独立状态模块”。

## 8. 固定实施步骤

1. 先盘点当前客户端门禁实际入口  
   - `WarehouseSystem.IsDirty()`
   - `_BM_SENSITIVE_CONSUMABLES`
   - `SaveSession`

2. 选定统一门禁实现方式  
   - 独立状态模块
   - 或 SaveSession 主门禁

3. 收口所有“本地先变化、后同步”的黑市相关入口  
   至少包括：
   - `ConsumeConsumable`
   - `WarehouseSystem` 存/取
   - `SaveSession.MarkDirty / Flush / 完成保存后清状态`

4. 把黑市卖出前置检查统一改到新门禁上  
   不再依赖道具名单。

5. 保留 `BM-01 / BM-01A` 已上线止血效果  
   本轮可以替换其内部机制，但不能让这两条路径退化。

## 9. 测试要求

### 9.1 必须新增的自动化覆盖

至少新增或补强以下测试：

1. 仓库存入后卖出被阻断
2. 仓库取出后卖出被阻断
3. 令牌盒使用后卖出被阻断
4. `gold_brick` 使用后卖出被阻断
5. 附灵玉消耗后卖出被阻断
6. 正常未脏状态下，黑市普通卖出仍可成功

### 9.2 测试层次要求

测试必须区分两类：

- **状态层测试**
  - 哪些操作会置脏
  - 哪些事件会清脏

- **行为层测试**
  - 黑市卖出是否在脏状态下被拦截

不接受只有 UI 文案断言、没有状态机断言的测试。

## 10. 黑市相关特别要求

### 10.1 扩展性要求

本任务必须为后续新增黑市商品留出扩展能力。

至少要满足：

- 新增商品时，不需要再给它单独补一条黑市敏感名单
- 只要它走的是“黑市相关本地变更类型”，就能自动进入门禁

### 10.2 与仙石/自动回收的边界

本轮不改自动回收和仙石结算逻辑，但完成报告必须明确说明：

- 本次客户端门禁是否影响自动回收相关 UI / 查询
- 本次是否改变任何仙石余额读写路径

答案预期应为“否”；如果不是，必须停手说明。

## 11. 成功标准

满足以下全部条件，才算 `BM-S2` 通过：

1. 黑市客户端门禁不再依赖敏感物品白名单
2. `gold_brick` 与附灵玉路径被自动化测试覆盖
3. 仓库、令牌盒旧止血路径不退化
4. 正常黑市买卖流程不被误伤
5. 不改服务端代码
6. 不新增黑市商品

## 12. 失败判据

出现以下任一情况，视为失败：

- 继续补名单
- `gold_brick` 仍需专门单独判断
- 附灵玉仍未进入统一门禁
- 黑市正常卖出被大面积误拦
- 运行时代码顺手改到服务端

## 1. 本次任务目标

本次只收 2 件事：

1. **修正解锁闭环**
   - 手动保存成功后，锁定状态必须真正解除
   - 重新登录后不能再从云存档读回旧锁

2. **整类禁售 UI 收口**
   - 只要某个 `itemId` 在背包中存在任意锁定堆
   - 该 `itemId` 在黑市 UI 中就整体视为“当前不可出售”
   - 不再进入“先弹确认框，再告诉玩家不可卖”的怪异交互

本次**不做**：

- 个体精细出售（只卖未锁堆）
- 更复杂的 `heldUnlocked / heldLocked` 排行列展示
- 黑市新功能扩展

## 2. 这次任务的产品口径

本任务采用以下明确规则：

> **只要某个商品存在任意锁定堆，该商品整类暂不可黑市出售；待成功存档或保护期超时后恢复。**

这意味着：

- 玩家背包里原有未锁 `gold_brick x9`
- 黑市新买带锁 `gold_brick x1`

在锁未解除前：

- 整个 `gold_brick` 黑市卖出按钮不可用
- 不做“仍允许卖 9 个旧库存”的精细区分

理由：

- 用户已明确接受“1 个锁定，全不出售”
- 这样能明显降低实现和维护复杂度
- 比“个体判断 + 部分可卖”更适合第五章前阶段

## 6. 需要修的两个问题

### 6.1 问题 A：成功存档后解锁没有真正闭环

现象：

- 本地保存后当前会话可能清锁
- 但重新登录后，锁会从存档里再次读回来

这说明当前大概率是：

- 先把带锁字段写进云存档
- 保存成功后才只清本地运行时锁
- 没把“解锁后的背包状态”同步回权威存档

本次必须修到：

- **成功存档后，本地和后续读档口径一致地视为已解锁**

允许实现方式：

1. 在保存写入前就生成“去锁版背包”参与序列化
2. 或在成功保存后立刻安排一次受控的去锁持久化
3. 或同等级别的闭环方案

但必须满足：

- 重登后锁不会重新出现
- 不能只解决“当前会话里看起来解锁”

### 6.2 问题 B：卖出 UI 口径错误

现象：

- 只要 `held > 0`
- 商品卡仍可点“出售”
- 甚至会弹出确认框
- 最终才在更深层被锁判断拦住

这会造成“界面很奇怪”的体验。

本次必须改成：

- 只要该 `itemId` 存在任意锁定堆
- 黑市商品卡的出售按钮直接置灰或不可点击
- 确认弹窗不应再出现
- 状态提示应明确说明：
  - `该商品存在保护期库存，当前不可出售`
  - `成功存档后自动解除`

## 7. 设计要求

### 7.1 增加整类锁态判断 API

建议新增统一接口，例如：

- `InventorySystem.HasAnyLockedConsumable(consumableId)`
- 或 `BlackMarketTradeLock.HasLockedStack(...)`

用途：

- 黑市 UI 直接拿它决定该 `itemId` 是否整体禁售

### 7.2 UI 决策口径改为“整类禁售”

至少要覆盖：

- 商品卡 `canSell`
- 商品卡按钮状态
- `ShowConfirmDialog("sell", itemId)` 触发前置判断
- 必要时补文案提示

要求：

- 不能再基于单纯 `held > 0` 判断可卖
- 要同时考虑“是否存在任意锁定堆”

### 7.3 成功存档后解锁的存档口径

本次必须明确并落实：

- 解锁不是纯运行时行为
- 解锁后的背包状态必须和未来重登结果一致

不要求你必须用哪种技术路径实现，
但完成报告必须明确说明：

- 锁是在保存前不入档
- 还是保存后通过额外持久化去掉
- 为什么重登后不会复现旧锁

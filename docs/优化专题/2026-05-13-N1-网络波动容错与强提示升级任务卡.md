# 2026-05-13 N1 网络波动容错与强提示升级任务卡

## 1. 任务定位

这是一个高性价比的线上体验缓解任务。

本任务不追求解决所有真实断线问题，也不做底层网络重构。  
它只做两件事：

1. 降低轻微网络波动时的误判断线 / 误失败
2. 在确认是持续断连时，给玩家足够强的感知与操作阻断

---

## 2. 为什么现在做

当前项目已经证明：

1. 黑市、存档、角色链路都依赖网络稳定性
2. 玩家对“弱提示”的感知不足，会继续误操作
3. 现有优化更偏向“保护存档不乱写”，而不是“保护玩家体验和操作预期”

因此，这一步的价值在于：

- 不改大架构
- 不碰高风险服务端协议
- 但能显著缓解“稍微波动就像掉线”的体感问题

---

## 3. 本任务目标

### 3.1 断线判定加宽限

不再允许“一次采样失败就直接 `connection_lost`”。

至少要做到：

1. 引入连续失败阈值或时间宽限
2. 短暂抖动不会立刻触发断线状态
3. 恢复判定也有最小滞回，避免抖动时 `lost/restored` 来回抖

### 3.2 分层提示升级

提示分层必须至少做到：

1. 轻微波动：横幅或状态提示
2. 持续断连：全屏阻断弹窗
3. 弹窗中明确提示：
   - 网络已中断
   - 当前进度可能无法同步
   - 请重新连接或重新登录

### 3.3 持续断连时阻断高风险操作

至少要阻断：

1. 黑市
2. 商城 / 兑换
3. 开箱 / 领奖
4. 其他依赖服务端立即确认的高风险写操作

### 3.4 只读请求补短时容错

对只读 / 幂等请求，允许增加统一短时重试缓冲，例如：

1. 黑市查询
2. 活动查询
3. 排行查询
4. 部分角色列表 / 拉取类请求

---

## 4. 本任务明确不做

1. 不做底层无感重连
2. 不做服务端会话恢复
3. 不对黑市 / 商城 / 兑换等高风险写请求做自动补发
4. 不重写网络协议
5. 不顺手改黑市服务端权威模型（那是 `BM-S3`）

## 8. 固定执行步骤

### 步骤 1：先盘点当前误判点

必须先在完成报告里列出：

1. 当前谁在发 `connection_lost / connection_restored`
2. 当前哪些路径会在 `GetServerConnection()==nil` 时立即失败
3. 哪些路径属于只读 / 幂等
4. 哪些路径属于高风险写操作

没有这张盘点表，不允许直接写代码。

### 步骤 2：收口断线判定

最小可接受做法：

1. 给断线判定加连续失败阈值或时间宽限
2. 给恢复判定加最小滞回

如果执行 AI 想改采样周期，也可以，但必须说明为什么比“当前每 5 秒轮询一次”更稳。

### 步骤 3：升级提示模型

必须把当前“弱横幅”升级成分层提示。

最低要求：

1. 保留轻提示
2. 增加持续断连的全屏阻断弹窗
3. 弹窗期间禁止高风险继续点击

### 步骤 4：只读请求容错

只允许给只读 / 幂等请求补短时容错，不允许对高风险写请求自动补发。

### 步骤 5：补测试

至少要补：

1. 断线判定宽限测试
2. 恢复滞回测试
3. 持续断连时高风险操作被阻断测试
4. 只读请求短时波动不立刻失败的测试（如可脚本化）

---

## 9. 成功标准

满足以下全部条件，才算通过：

1. 轻微网络波动不再立刻触发断线状态
2. 持续断连会升级成强提示 / 阻断弹窗
3. 高风险写操作在持续断连时被明确阻断
4. 没有引入黑市自动重发或资源重复提交风险
5. blocking gate 继续全绿

---

## 10. 失败判据

出现以下任一情况视为失败：

1. 仍然一次采样失败就直接 lost
2. 直接把所有操作都硬弹窗阻断，导致正常抖动也无法玩
3. 对黑市 / 商城 / 兑换高风险写请求做自动补发
4. 顺手改到服务端或协议

---

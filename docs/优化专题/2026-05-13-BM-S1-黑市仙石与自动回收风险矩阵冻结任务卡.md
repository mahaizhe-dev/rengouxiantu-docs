# 2026-05-13 BM-S1 黑市、仙石与自动回收风险矩阵冻结任务卡

## 1. 任务定位

这是黑市专项第一步。

这一步不是热修，不是补名单，不是改协议，也不是改 `BlackMerchantHandler`。

本任务只做一件事：

- 基于当前 Git 代码，把黑市、仙石、自动回收的真实风险面冻结成文档合同

如果这一步没做，后续 `BM-S2 / BM-S3` 仍然会退化成“发现一个物品，补一个判断”。

## 2. 任务目标

交付两份文档：

1. `黑市可卖物-本地变更路径风险矩阵`
2. `黑市一致性与仙石结算合同 v1`

并在上述两份文档中补齐一套最小的“黑市新增商品准入规则”。

两份文档都必须落在 `docs/优化专题/`。

## 6. 固定产出要求

### 6.1 风险矩阵文档必须包含

至少按以下维度列出矩阵：

- 物品 ID
- 分类
- 是否黑市可卖
- 本地可变更类型
  - 使用
  - 卖钱
  - 开盒
  - 入仓/出仓
  - 炼丹消耗
  - 龙锻消耗
  - 灵玉/附灵消耗
  - 宠物书/培养消耗
  - 其他
- 当前是否会立即 `save_request`
- 当前是否只 `SaveSession.MarkDirty`
- 当前是否已被 `BM-01 / BM-01A` 覆盖
- 扩展性分类
  - 安全类
  - 高风险类
  - 未知
- 风险等级
  - `P0`
  - `P1`
  - `P2`
  - `已止血`
  - `未知`
- 准入建议
  - 可准入
  - 需专项审查
  - 当前禁止准入

### 6.2 合同文档必须明确

至少明确以下 6 个结论：

1. 黑市“可卖性”到底由什么定义  
   当前现实口径，不是理想口径。

2. 什么操作会改变黑市可卖性  
   必须按“操作类型”定义，而不是按“物品名单”定义。

3. 哪些状态变化必须进入“黑市未同步门”  
   为 `BM-S2` 提供直接输入。

4. 服务端卖出权威缺口在哪里  
   为 `BM-S3` 提供直接输入。

5. 自动回收和仙石结算哪些边界必须一起治理  
   为 `BM-S4` 提供直接输入。

6. 第五章新增道具接入黑市前，需要满足什么前置检查  
   为 `C5-PREP` 提供直接输入。

7. 黑市未来新增商品的最小准入规则  
   至少说明什么样的商品可以低风险接入，什么样的商品必须升级到专项审查。

## 7. 明确禁止的做法

禁止以下低质量推进方式：

- 只列当前已知漏洞物品，不盘点同类物品
- 只写现象，不写代码入口
- 只写客户端，不写服务端卖出链路
- 把自动回收排除在黑市专项之外
- 把仙石/仙玉问题只当文案问题处理
- 提出“继续补敏感物品名单”作为长期方案

## 8. 成功标准

满足以下全部条件，才算 `BM-S1` 通过：

1. 有一张完整的黑市可卖物风险矩阵
2. 有一份明确的黑市一致性合同 v1
3. 文档中明确指出：
   - 当前热修覆盖了什么
   - 当前热修没覆盖什么
4. 文档中明确冻结：
   - `BM-S2` 要解决什么
   - `BM-S3` 要解决什么
   - `BM-S4` 要解决什么
5. 文档中给出一套可执行的黑市商品准入规则
6. 不改任何运行时代码

## 2. 黑市问题的整体统筹判断

### 2.1 当前黑市的核心问题，不是单个 bug，而是模型冲突

当前黑市卖出服务端判断，核心仍建立在**已保存的旧背包快照**之上。

证据：

- [BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua)
  - `HandleQuery()` 里 `held = CountItemHeld(backpack, id)`
  - `HandleBuy()` 里对 `saveData.backpack` 直接加物品
  - `HandleSell()` 同样围绕 `saveData.backpack` / `CountItemHeld()` 工作

而项目内大量物品变化，是**客户端先变，再异步保存**：

- [SaveSession.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/save/SaveSession.lua)
- [InventorySystem.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/InventorySystem.lua)
- [WarehouseSystem.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/WarehouseSystem.lua)
- 神器、炼丹、附灵、打造等系统

所以黑市真正的问题不是：

- “还有几个物品没补名单”

而是：

**硬通货结算入口，信了一个可能过期的库存视图。**

### 2.2 我对黑市的重新统筹方案

黑市不适合走“服务器完全代管库存”的路线。对这个项目，更合理的是：

#### 第一层：商品静态风险分层

按“是否会本地先变化”分，而不是按旧的 `L1/L2` UI 模型分。

- `A 直接使用型`
  - 例：`gold_brick`、令牌盒
- `B 系统消耗型`
  - 例：附灵玉、神器碎片、炼丹/打造材料
- `C 纯库存型`
  - 例：没有已知本地变化路径的库存物

规则：

- `A/B` 买入后进入黑市交易保护锁
- `C` 不加锁，正常买卖

#### 第二层：运行时状态拆域

必须把两类状态彻底分开：

- **商品级状态**
  - 只描述某个 `itemId` 自己是否被锁
- **黑市级状态**
  - 只描述当前黑市是否在请求中、网络异常、不可提交

不能再把“全局未同步”直接塞进单个商品卡的可售状态里。

正确表现应是：

- 商品卡只显示 `出售` / `🔒锁定`
- 黑市顶部显示 `处理中` / `网络异常` / `同步中`

#### 第三层：交易保护锁

保护锁不是“5 分钟内不能使用”这么简单，而应该是：

**5 分钟内，或直到下一次成功存档前，不能发生任何改变该商品状态的本地操作，也不能卖回黑市。**

包括：

- 使用
- 消耗
- 存仓/取仓
- 合成/激活/打造
- 黑市回卖

并且：

- 买入消耗品必须**强制分成新堆**
- 不允许并进老堆
- 锁定堆必须有图标和 UI 提示

#### 第四层：服务端轻量护栏，而不是全权威库存

服务端不需要代管一切，但必须补轻量护栏：

- 单商品短时卖出冷却
- 卖出频率限制
- 单次数量上限
- 可疑异常计数
- 商品级或账号级熔断

目标不是绝对防作弊，而是：

- 阻断正常玩家轻易利用
- 提高改包利用成本
- 控制单次和批量损失

### 2.3 黑市自动回收的判断

当前自动回收代码仍然存在明显架构脆弱性。

证据：

- [BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua)
  - `RecycleTick()`
  - `ExecuteRecycle()`

当前问题不是单个 bug，而是：

- 自己管理“自然日推进”
- 同时负责库存、日志、回收、异常分支
- 仍然与黑市主 handler 高耦合

其中“跨月/跨年日期差”的旧实现是明确错误的；如果已做修正，也仍需**真实流程级验证**，不能只测日期工具函数。

### 2.4 黑市的最终判断

黑市是第五章前最重要的系统之一，但正确方向不是：

- 服务器代管一切
- 或者 79 项全部拒卖

而是：

**商品风险分层 + 交易保护锁 + 商品级锁显示 + 黑市级请求状态 + 轻量服务端护栏 + 自动回收独立验证。**

## 3. 我对“性能开关”的重新判断

### 3.1 之前“泛性能开关”这个判断不够严谨

如果只是泛泛地说“加一个性能开关/低功耗模式”，这确实容易变成无效建议。  
因为当前项目已经做过部分更新节流，尤其怪物更新侧已经不是完全裸跑。

证据：

- [main.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/main.lua)
  - `FAR_MONSTER_RANGE`
  - `FAR_MONSTER_UPDATE_INTERVAL = 0.5`
  - 远处 idle/patrol 怪物半秒更新一次

这说明“手机发热”不能简单归因于主逻辑没节流。

### 3.2 基于渲染方式，真正昂贵的地方在哪里

当前主要热源更像是**每帧 CPU 侧的大量渲染组织 + 高特效 pass**，而不是某个单独的数值系统。

证据：

- [WorldRenderer.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/rendering/WorldRenderer.lua)
  - 每帧串行渲染 tiles、decorations、loot、monsters、players、health bars、effects
- [WorldRenderer.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/rendering/WorldRenderer.lua)
  - `RenderTiles()` 里双层 tile 循环
- [EffectRenderer.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/rendering/EffectRenderer.lua)
  - 每帧十多个 effect pass
- [EntityRenderer.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/rendering/EntityRenderer.lua)
  - `RenderLootDrops()` 里高品质掉落的外光、内光、旋转粒子、文本

这说明：

- 一个“泛性能开关”没意义
- 但一个**针对这些具体渲染路径的移动端降载开关**，是有实际价值的

### 3.3 性能开关的必要性和可行性

#### 必要性

- **中等必要**
- 不是第五章前第一优先级
- 但如果第五章继续增加怪物、地图、掉落、特效，不做任何降载，移动端发热和掉帧风险会被放大

#### 可行性

- **高**
- 但前提是目标明确，不做空泛的“低画质”

### 3.4 真正有价值的 T1 方案

我现在只认可这种“有代码抓手”的降载：

- 关闭高品质掉落外光/粒子
- 关闭或降级部分 `EffectRenderer` pass
  - buff aura
  - bagua aura
  - 部分 slash / zone 特效
- 限制血条显示范围
- 降低顶部额外浮字/装饰层

如果做不到直接关掉某些 pass，那这个开关价值就会很低。

### 3.5 关于性能的最终判断

**T1 有必要，但不是第五章前最高优先级；它应该是一个“小型移动端降载补丁”，不是一轮大性能重构。**

## 4. 当前项目的独立审查结果

下面是不受之前结论干扰、只基于当前代码状态做的项目级判断。

### 4.1 可扩展性

评级：**中低**

原因：

- 主入口和服务端主入口仍然过大
  - [main.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/main.lua)
  - [server_main.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/server_main.lua)
- 新系统非常容易同时碰到：
  - UI
  - Inventory/Consumable
  - Save
  - Server handler
  - Config
- 黑市、附灵、神器、炼丹、存档天然交叉，说明“新增一类道具”已经不是单一模块改动

结论：

- 第五章继续加内容是可行的
- 但如果不先补黑市和存档边界护栏，扩展风险会明显上升

### 4.2 可维护性

评级：**中低**

原因：

- `main.lua` 太大
- `server_main.lua` 太大
- `SavePersistence.lua` 太大
- 大型 UI 文件过多，如：
  - [BlackMerchantUI.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/ui/BlackMerchantUI.lua)
  - [PetPanel.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/ui/PetPanel.lua)
  - 其他系统 UI

但需要注意：

- 当前阶段可维护性的最大问题，不是“文件大”本身
- 而是**状态边界不清**，尤其黑市 / 存档 / 本地消耗这类跨系统状态

### 4.3 线上稳定性

评级：**中**

原因：

- 有不少保护措施
  - `SaveSystem` 保存失败重试
  - 连接丢失提示
  - 一些服务端 rate limit
- 但核心断线判定仍然粗：
  - [SaveSystem.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/SaveSystem.lua)
  - 每 5 秒看一次 `network:GetServerConnection()`
  - 单次状态跳变就触发 lost/restored
- 很多查询/写请求对网络抖动仍然敏感

结论：

- `N1` 这条优化线仍然是有价值的
- 但它不是因为某次发布才需要做，而是本来就是长期稳定性问题

### 4.4 性能

评级：**中低（移动端更低）**

原因不是“某个模块特别慢”，而是：

- 大量 per-frame 渲染调度
- tile 双层循环
- 多特效 pass
- 高品质掉落渲染
- 再叠加主循环大量系统更新

当前已经有：

- 怪物远距节流
- `PerfMonitor` 监测

所以第五章前不该做大重构，而应做：

- **小型移动端降载补丁**

### 4.5 存档安全

评级：**中低**

证据：

- [SavePersistence.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/save/SavePersistence.lua)
  - 序列化范围过大，跨系统数据耦合严重
- [SaveSession.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/save/SaveSession.lua)
  - 仍是旧式会话合并逻辑
- [SaveSystem.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/SaveSystem.lua)
  - 连接状态与保存调度高度耦合

当前存档最大风险不是“马上坏”，而是：

- 黑市 / 仓库 / 消耗品 / 神器 / 附灵等系统都依赖“本地先变，异步落盘”
- 一旦再扩内容，没有更清晰的边界，就会继续出“状态看起来对，但真实提交口径不一致”的问题

### 4.6 C/S 架构

评级：**中低**

原因：

- `server_main.lua` 仍然承担太多
- 虽然已有 handler split，但主链依然集中
- 黑市、存档、排行、奖励等关键路径仍是“服务端部分权威 + 客户端先变”的混合模型

这不是说必须“服务器代管一切”，而是：

**必须先把哪些是服务端结算权威、哪些是客户端展示/缓存状态划清。**

## 5. 第五章前，必做且高性价比的优化

下面这些我认为是**第五章前必须做**的，而且性价比高。

### P0. 黑市卖出模型重置

这是第五章前最高优先级之一。

目标：

- 不再继续 patch 式修黑市
- 重置为：
  - 商品风险分层 `A/B/C`
  - 商品级交易锁
  - 黑市级请求状态
  - 轻量服务端护栏
  - 自动回收真实流程验证

原因：

- 黑市是硬通货系统
- 第五章继续加商品，如果黑市模型不先稳住，只会继续放大问题

### P1. 黑市自动回收流程级校验

不是只修 `RealDayGap()`，而是要验证：

- `RecycleTick()`
- `lastRecycleDate_`
- 云端 `bm_recycle_date`
- `ExecuteRecycle()`

这条一定要单独走流程级测试，不要再只测工具函数。

### P2. N1 网络波动容错与断线强提示

目标：

- 给当前粗糙的断线判定加宽限/滞回
- 把长期弱提示升级成：
  - 短时轻提示
  - 持续断连强提示/阻断弹窗

这条对第五章前的意义是：

- 它提升的是整个项目的线上体验稳定性
- 不是黑市附属问题，也不是本次发布附属问题

### P3. 存档边界收口：不做大重构，只做风险边界明确化

不是继续推大范围 Save 重构，而是做最值钱的一步：

- 冻结哪些系统是“本地先变化 + 异步保存”
- 哪些系统会影响黑市/硬通货/关键资源结算
- 为第五章新增道具和系统设一条准入规则

这一步更多是“边界治理”，不是“继续拆 Save 模块”。

### P4. T1 移动端降载最小补丁

必要，但排在黑市和稳定性之后。

只做：

- 掉落高光/粒子降级
- 关闭部分高开销 effect pass
- 限制部分血条/浮字

不要做大性能重构。

### P5. 第五章开发护栏冻结

在正式开第五章前，必须冻结两条规则：

1. 新增可黑市卖出商品，必须先走风险分类
2. 新增会本地先消耗/先变化的系统道具，必须声明其是否进入黑市保护锁体系

这一条非常值，因为它能避免第五章一开工，黑市问题立刻复发。

## 8. 最终结论

### 8.1 关于黑市

黑市是第五章前最重要的优化主题之一，但正确方向是：

- **不是**服务器完全代管
- **不是**继续补名单
- **不是**全拒卖

而是：

**商品风险分层 + 交易保护锁 + 商品级锁显示 + 黑市级请求状态 + 轻量服务端护栏 + 自动回收流程验证**

### 8.2 关于性能开关

我之前“泛性能开关”的判断不够严谨。  
重新基于代码看，结论是：

- **泛开关没意义**
- **定向降载补丁有意义**

其价值来自：

- 关闭高开销渲染 pass
- 降级掉落特效
- 降低屏上辅助渲染密度

而不是一个空泛的“低画质模式”。

### 8.3 关于项目整体

当前项目在第五章前，不需要继续追求“大而完整”的结构重构。  
更高 ROI 的是：

- 把黑市这个核心经济系统的模型先稳住
- 把网络波动这个长期问题纳入正式优化
- 用最小代价给移动端降载
- 给第五章新增内容建立准入护栏

**一句话总结：第五章前最值钱的优化，不是继续拆大模块，而是先把“商品状态、存档同步、硬通货结算、移动端渲染负荷”这四条真正的高风险主线压稳。**

## 1. 任务定义

任务ID：`P0-2`

任务名称：`增加线上开关与发布护栏`

变更类型：`结构准备`

目标：

- 建立统一 Feature Flag / Kill Switch 入口
- 为后续 `P0-5 / P0-3 / P0-4 / P0-6` 提供灰度和回滚护栏
- 确保当前默认行为不变

---

## 2. 本任务的成功标准

成功标准只有 5 条：

1. 全项目只有一个统一的功能开关入口模块。
2. 第一批高风险开关已经定义，但默认值不能改变当前行为。
3. 代码读取开关时通过统一 API，而不是到处直接访问裸表。
4. 至少有一组纯 Lua 自动化测试验证开关默认值和读取行为。
5. 必须写清楚每个开关未来对应哪一个后续任务。

如果 5 条里任一条不满足，本任务不算完成。

---

## 3. 本任务允许修改的文件

允许新增或修改：

- `scripts/config/FeatureFlags.lua`
- `scripts/config/GameConfig.lua`
- `scripts/main.lua`
- `scripts/client_main.lua`
- `scripts/server_main.lua`
- 与开关读取直接相关的少量接入点文件
- `scripts/tests/test_feature_flags.lua`
- `scripts/tests/TestRegistry.lua`
- 必要时补充 `scripts/tests/README.md`

要求：

- 运行时接入点只能做最小接线
- 不要顺手做结构重构

---

## 4. 本任务禁止修改的文件

不要修改：

- `scripts/systems/save/*` 的实际存档逻辑
- `scripts/network/*` 的协议行为
- 迁移状态机
- 排行榜/奖励业务规则
- 任何 UI 表现层逻辑

特别禁止：

- 借着加开关顺手实现 `SAVE_PIPELINE_V2`
- 借着加开关顺手重写 `server_main.lua`

---

## 5. 执行前先读这些文件

1. `scripts/config/GameConfig.lua`
2. `scripts/main.lua`
3. `scripts/client_main.lua`
4. `scripts/server_main.lua`
5. 搜索仓库中已有的配置读取方式
6. `scripts/tests/TestRegistry.lua`
7. `docs/优化专题/2026-05-12-P0-1A-测试门禁收口与运行环境固化任务卡.md`

执行前必须先回答：

1. 当前配置读取入口是否唯一
2. 开关应该放在哪个模块最不容易引入循环依赖
3. 第一批开关分别服务哪个后续任务

答不出来就不要动代码。

---

## 6. 第一批必须定义的开关

至少定义以下开关：

1. `SAVE_PIPELINE_V2`
2. `SERVER_SAVE_VALIDATOR_V2`
3. `SAVE_MIGRATION_SINGLE_ENTRY`
4. `NEW_MAIN_LOOP_ORCHESTRATOR`
5. `NEW_SERVER_ROUTER`
6. `CLOUDSTORAGE_STRICT_BATCHSET`

每个开关至少要有：

- 名称
- 默认值
- 说明
- 对应任务

建议格式统一，不允许每个文件各写一套。

---

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：建立统一模块

新增：

- `scripts/config/FeatureFlags.lua`

要求：

1. 模块本身尽量纯
2. 不依赖复杂运行时初始化
3. 提供统一读取函数

### 步骤 2：定义 flag 元数据

不要只写：

- `FLAG = true/false`

至少还要保留：

- 描述
- 默认值
- 所属任务

### 步骤 3：最小接入

在必要入口上接入开关读取，但第一轮不改变行为。

可接受做法：

- 保留旧逻辑为默认
- 新逻辑分支只占位或保留为未来接入点

### 步骤 4：补纯 Lua 测试

新增：

- `scripts/tests/test_feature_flags.lua`

至少覆盖：

1. 默认值正确
2. 未定义开关处理明确
3. 统一读取 API 正常
4. 关闭开关时不影响当前旧路径

### 步骤 5：把测试接入当前门禁体系

如果 `P0-1A` 已完成：

- 将 `test_feature_flags.lua` 接入当前阻断门禁或 system/smoke 体系

但不要破坏 `P0-1A` 收口后的门禁定义。

## 8. 本任务的停手条件

出现以下情况，停止编码，改交付问题说明：

1. 加开关必须同时改业务行为才能继续
2. `FeatureFlags.lua` 会引入明显的 require 循环
3. 不同运行入口使用不同配置初始化，无法统一
4. 需要同时修改存档逻辑和网络协议才能把开关接进去

如果停手，仍然要交付：

- 循环依赖风险说明
- 当前配置入口清单
- 拆分建议

# BM-S4E：黑市可售状态一致性与自动回收集成收口任务卡

## 0. 任务定位

本卡合并处理两类仍然阻塞黑市主流程的问题：

1. `BM-S4D` 之后，黑市商品在“锁已消失/已成功保存”后，仍可能出现：
   - 卡片上 `出售` 按钮亮起
   - 但点击后又被提示 `背包数据未同步，请稍后再试`
   - 前端“锁状态”和“可出售状态”不一致

2. 黑市自动回收虽然已修复 `YYYYMMDD` 直接相减的跨月/跨年问题，但当前测试只证明了 `RealDayGap()` 日期工具，不足以证明 `RecycleTick()` 在真实过天流程里稳定工作。

本卡目标不是扩展新规则，而是把**黑市可售状态的单一真相**和**自动回收真实过天链路**一次收清。

---

## 1. 本次目标

### 1.1 黑市可售状态一致性

把当前黑市卖出判定从“分散在 3 处、用户看到的状态和真实拦截原因不一致”，收敛成一条统一规则：

- **同类锁（L1）**
  - 某个 `itemId` 存在任意锁定堆
  - 则该 `itemId` 整类不可出售

- **全局未同步（L2）**
  - 仅在真实存在未同步状态时阻断
  - 且前端卡片、顶部提示、点击行为必须和这个状态一致

要求：

- `出售` 按钮是否可点
- 顶部红字提示
- 是否允许进入确认出售弹窗
- 确认出售时最终是否放行

必须基于**同一套可售判定结果**，不能再“按钮亮着，但点下去才发现不让卖”。

### 1.2 自动回收真实过天链路

当前 `BM-S4D` 只证明：

- `BlackMerchantConfig.RealDayGap()` 在日期计算上跨月/跨年正确

但还没有证明：

- `BlackMerchantHandler.RecycleTick()` 在真实 once-per-day 流程中稳定触发
- `lastRecycleDate_` 内存占位 + 云端 `bm_recycle_date` 推进 + `ExecuteRecycle()` 组合在跨天、跨月、跨年、异常大跨度场景下都符合预期

本卡要求补到**真实流程级**验证，不再只停留在工具函数层。

## 4. 固定实施步骤

### 步骤 A：收敛黑市可售状态为单一判定入口

必须先抽出一个前端统一判定入口，示例方向：

- `GetSellBlockReason(itemId)`
- 返回值只允许是：
  - `none`
  - `locked_item`
  - `global_unsync`

要求：

- 卡片 `出售` 按钮是否可点，不能再只看 `HasAnyLockedConsumable(itemId)`
- 点击 `出售` 是否允许弹确认框，必须复用同一判定
- 顶部提示文案也必须来自同一判定

不允许继续保留“卡片阶段看一套、确认阶段再看另一套”的分裂逻辑。

### 步骤 B：查清并修复“锁已解但仍 global_unsync”的真实来源

这一步不能拍脑袋改，要先明确分支：

- 是 `SaveSession.IsDirty()` 残留
- 还是 `_dirtyConsume`
- 还是 `_dirtyWarehouse`
- 还是顶部状态文案残留但实际已可卖

要求：

- 对“纯买入黑市商品 -> 成功保存 -> 锁消失”路径，最终必须回到 `GetSellBlockReason(itemId) == none`
- 如果是文案残留，则清理文案生命周期
- 如果是 L2 真残留，则修正清理边界

### 步骤 C：补 `RecycleTick()` 流程级测试

不能只测 `RealDayGap()`。

必须新增真实流程测试，至少覆盖：

- 同一天重复 tick，只执行一次
- 次日同月，执行一次
- 月末跨月，只执行一次
- 年末跨年，只执行一次
- 超过 `MAX_RECYCLE_GAP` 时只推进日期，不执行回收
- `lastRecycleDate_` 与云端 `bm_recycle_date` 协同不导致“第一天触发，后面不触发”

测试至少要 stub：

- `os.date`
- `serverCloud.money:Get`
- `serverCloud.money:Add`
- `M.ExecuteRecycle`

并验证：

- 日期是否被正确推进
- 回收是否真正触发/不触发
- 同一自然日不会重复

## 5. 必须补的测试

### 黑市可售状态一致性

至少补：

1. `itemId` 有锁时：
   - 按钮禁用或点击即前置拦截
   - 不进入确认出售弹窗
   - 顶部提示为锁定提示

2. `itemId` 无锁，但 `L2` 全局未同步时：
   - 前端状态与真实拦截一致
   - 不允许出现“按钮亮着，但确认阶段才挡”

3. 纯黑市买入 -> 成功保存 -> 锁清除后：
   - `GetSellBlockReason(itemId) == none`
   - 不再显示 `背包数据未同步`

4. 无锁且无 L2 时：
   - 可正常进入确认出售并继续卖出

### 自动回收流程级

至少补：

1. `RecycleTick()` 同日幂等
2. 次日触发
3. 跨月触发
4. 跨年触发
5. 大跨度只推进日期、不执行回收

## 6. 验收标准

同时满足以下条件才算通过：

1. 黑市前端的“可出售状态”变成单一判定来源  
   不能再出现“按钮亮起，但其实下一步会被另一套门拦住”的交互分裂。

2. 纯“买入黑市商品 -> 保存成功 -> 锁消失”后，不再出现 `背包数据未同步，请稍后再试`。

3. 自动回收有真实 `RecycleTick()` 级测试，而不是只有日期工具测试。

4. blocking / `--all` 原始 runner 输出附在完成报告里。

---

## 7. 失败判据

出现以下任一项即判未通过：

- 无锁商品仍能点出确认框，但确认阶段才被 L2 拦
- 纯买入并成功保存后，仍然是 `global_unsync`
- 自动回收仍只有工具函数测试，没有真实 tick 流程验证
- 完成报告缺少真实 runner 输出

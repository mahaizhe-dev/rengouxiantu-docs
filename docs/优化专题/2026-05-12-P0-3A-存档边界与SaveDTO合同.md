## §3 边界分类表

### 3.1 分类总表

| 边界类别 | 包含的 key family | 合计 key 数 | 写入权威 |
|---------|-----------------|-----------|---------|
| **main_character_save** | `save_{slot}`, `save_data_{slot}`(legacy), `save_bag1/2_{slot}`(legacy) | 4 per slot × 4 slots = 16 | 客户端 DoSave → 服务端 B4 校验 → 写入 |
| **slot_index_meta** | `slots_index`, `player_level` | 2 | 客户端附带 → 服务端规范化 |
| **backup_recovery_meta** | `login_{slot}`, `prev_{slot}`, `deleted_{slot}`, `premigrate_*`, `save_migrate_backup_*`, `migration_backup_done` | ~20 | 客户端各时机写入；服务端仅写 deleted + migrate_backup |
| **account_level_meta** | `account_bulletin`, `account_atlas`, `account_cosmetics` | 3 | 客户端 embedded → 服务端提取写独立 key |
| **server_authority_side_data** | `rank2_*`, `trial_*`, `prison_*`, `_code_ver`, `pending_redeem_*`, `dungeon_pending_*`, v1 `rank_*` | ~20 | 服务端权威计算或旁路写入 |

### 3.2 详细归类

#### 3.2.1 main_character_save

角色主存档正文。`save_{slot}` 是 v11 单 key，包含以下所有序列化模块：

```
save_{slot} = {
  version: number,           -- CURRENT_SAVE_VERSION (23)
  -- 以下来自 SaveSerializer 各 Serialize 函数：
  player: { ... },           -- SerializePlayer: 50+ 字段
  equipment: { ... },        -- SerializeEquipped: 装备栏 13 槽
  backpack: { ... },         -- SerializeBackpack: 背包 42 格
  skills: { ... },           -- SerializeSkills: 已学技能列表
  collection: { ... },       -- SerializeCollection: 收集品
  pet: { ... },              -- SerializePet: 灵宠
  quests: { ... },           -- 任务进度
  shop: { ... },             -- 商店状态 (v13+ 空壳)
  titles: { ... },           -- 称号
  challenges: { ... },       -- 挑战数据
  sealDemon: { ... },        -- 封魔系统
  trialTower: { ... },       -- 青云试炼塔
  artifact: { ... },         -- 法器
  fortuneFruits: { ... },    -- 福缘果
  seaPillar: { ... },        -- 海神柱
  prisonTower: { ... },      -- 镇狱塔
  warehouse: { ... },        -- 仓库
  atlas: { ... },            -- 图鉴 (账号级，透传)
  accountCosmetics: { ... }, -- 外观 (账号级，透传)
  yaochi_wash: { ... },      -- 瑶池洗练
  event_data: { ... },       -- 活动数据
  openedXianyuanChests: {},  -- 已开仙缘宝箱
  pendingXianyuanRewards: {},-- 待领仙缘奖励 (服务端写入)
  bossKillTimes: { ... },    -- Boss击杀次数
}
```

归属判定依据：
- 这些字段全部由 `SaveSerializer.Serialize*()` 序列化
- 全部由 `SavePersistence.DoSave()` 通过 `CloudStorage.BatchSet` 写入 `save_{slot}` key
- 全部由 `SaveLoader.Load()` → `ProcessLoadedData()` 反序列化

⚠️ **特殊字段**：
- `atlas` 和 `accountCosmetics`：虽然 embedded 在 `save_{slot}` 中，但实际是**账号级数据的透传载体**。服务端 HandleSaveGame 会从 coreData 中提取它们，分别写入 `account_atlas` 和 `account_cosmetics` 独立 key。归类说明见 §3.2.4。
- `pendingXianyuanRewards`：服务端通过 HandleLoadGame 附加到存档数据中的旁路字段。客户端保存时透传回去。归类说明见 §3.2.5。

#### 3.2.2 slot_index_meta

| key | 内容 | 权威来源 |
|-----|------|---------|
| `slots_index` | `{ version, slots: { [slot]: { name, level, realm, classId, ... } } }` | 客户端构建 → 服务端 NormalizeSlotsIndex 规范化 → 写入 |
| `player_level` | 等级整数副本 | 客户端附带 → 服务端 SetInt |

归属判定依据：
- `slots_index` 是全局唯一（非 per-slot），包含所有槽位的元数据摘要
- 它不是存档正文——它是存档的**索引**，用于角色选择界面展示
- `player_level` 是等级的整数副本，仅用于排行系统辅助

#### 3.2.3 backup_recovery_meta

| key family | 写入时机 | 写入端 | 用途 |
|-----------|---------|--------|------|

- P0-5 strict mode 已阻止 `login_` / `prev_` 在网络模式下写入

#### 3.2.4 account_level_meta

| key | 内容 | 客户端写入路径 | 服务端写入路径 | 跨角色共享 |
|-----|------|-------------|-------------|-----------|
| `account_bulletin` | 公告/通知数据 | DoSave 附带 batch:Set | HandleSaveGame batch:Set | 是 |
| `account_atlas` | 图鉴/道印/神器 | AtlasSystem.SaveImmediate (standalone), DoSave embedded in coreData.atlas | HandleSaveGame 从 coreData 提取 → batch:Set("account_atlas") | 是 |
| `account_cosmetics` | 角色外观数据 | DoSave embedded in coreData.accountCosmetics | HandleSaveGame 从 coreData 提取 → batch:Set("account_cosmetics") | 是 |

归属判定依据：
- 三个 key 都是**账号级**，不绑定单个角色，所有角色共享
- `account_atlas` 有**双写**问题：AtlasSystem.SaveImmediate() 会在 standalone 模式下直接写 `account_atlas`，同时 DoSave 也会把 atlas embedded 在 coreData 里交给服务端提取
- `account_cosmetics` 是**透传模式**：客户端写入 coreData → 服务端提取写入独立 key → 下次 Load 时服务端读取独立 key 附加到存档数据
- 这三个 key **不属于 SaveDTO**，但 atlas 和 accountCosmetics 的值**来源于 save_{slot} 的子字段**

⚠️ **account_atlas 双写是已确认歧义点**（见 §7）

## §4 SaveDTO v1 合同表

### 4.1 SaveDTO v1 合同

SaveDTO v1 的定义：**角色主存档正文的结构化数据合同**。仅覆盖 `save_{slot}` key 中由 `SaveSerializer` 管理的模块。

| # | 字段域 | 是否进入 SaveDTO v1 | 来源模块 | 当前存储位置 | 决策理由 |
|---|-------|--------------------|---------|-----------|---------| 
| 1 | `player` | **✅ 进入** | SaveSerializer.SerializePlayer | `save_{slot}.player` | 角色核心属性，50+字段，SaveDTO 主体 |

## 附录 A：7 个前置问题的回答

### Q1: 哪些 key 属于角色主存档？

**答**: 仅 `save_{slot}`（v11 统一 key）。legacy `save_data_{slot}` / `save_bag1/2_{slot}` 是 v10 格式的读兼容 key，新写入已全部合并到 `save_{slot}`。

### Q2: 哪些 key 属于槽位元数据？

**答**: `slots_index`（全局唯一索引）和 `player_level`（等级整数副本）。

### Q3: 哪些 key 属于恢复/备份链路？

**答**: `login_{slot}`、`prev_{slot}`、`deleted_{slot}`、`premigrate_{slot}`、`premigrate_bag1/2_{slot}`、`save_migrate_backup_{source}`、`migration_backup_done`。共 7 个家族。

### Q4: 哪些 key 属于账号级持久化？

**答**: `account_bulletin`、`account_atlas`、`account_cosmetics`。三个 key 都是账号级，跨角色共享。

### Q5: 哪些数据虽然出现在 save_{slot} 中，但权威写入实际在服务端？

**答**:
- `atlas`：客户端 embedded → 服务端提取写入 `account_atlas`（服务端是最终写入方）
- `accountCosmetics`：客户端 embedded → 服务端提取写入 `account_cosmetics`
- `pendingXianyuanRewards`：服务端 HandleLoadGame 附加，客户端仅透传
- 排行榜数据（rank2_*、trial_*、prison_*）：完全由服务端从 coreData 计算后写入独立 key

### Q6: 哪些字段属于 rules.md 下必须保持旧 key/旧字段兼容的红线？

**答**: 见 §5.1（绝不能删除/重命名的字段）和 §5.2（绝不能改名的 key 家族）。核心规则：
- 所有已有字段：不删除、不重命名
- 所有已有 key：不改名
- 所有已有迁移函数 MIGRATIONS[2-23]：不删除、不修改
- 配对字段（§5.5）：不拆开

### Q7: 哪些字段若进入 DTO，后续必须伴随 CURRENT_SAVE_VERSION + SaveMigrations 一起变更？

**答**: SaveDTO v1 冻结的 22 个字段域中，**任何结构性变更**（新增子字段、修改字段含义、变更序列化格式）都必须：
1. `CURRENT_SAVE_VERSION` +1（当前 23 → 24）
2. `SaveMigrations.MIGRATIONS[24]` 新增迁移函数
3. `SaveSerializer.Deserialize*()` 添加默认值兼容
4. 服务端 HandleSaveGame 校验逻辑适配

**特别注意**: 如果 P0-3 的 SaveDTO 只是添加类型定义（不改变运行时数据格式），则不需要版本号变更。只有实际改变序列化格式时才需要。

# 2026-05-13 HOTFIX-BM-01 黑市仓库一致性漏洞热修任务卡

> 这是一个线上紧急热修任务。  
> 目标不是顺手重构黑市或仓库系统，而是先把“卖出收益到账但仓库副本被保存”的漏洞路径堵住。  
> 本任务按 `rules.md` 的线上项目红线执行，优先级高于常规重构路线。

配套文档：
- [项目规则 rules.md](../架构文档/rules.md)
- [R3至第五章开工前优化执行方案](./2026-05-13-R3至第五章开工前优化执行方案.md)

## 1. 任务定义

任务 ID：`HOTFIX-BM-01`

任务名称：`黑市-仓库一致性漏洞热修`

变更类型：`线上漏洞热修`

优先级：`P0`

目标：
- 先阻断“黑市卖出成功，但仓库副本后续仍被保存”的漏洞链路
- 收口仓库与黑市之间的未同步窗口
- 给后续更彻底的服务端权威化修复留出空间

---

## 2. 问题定义

当前代码存在一条结构性分裂链路：

1. 黑市买入时，服务端先把道具写入 `saveData.backpack`
2. 玩家把道具放入仓库时，只改本地运行时状态，不立即形成可靠同步闭环
3. 黑市卖出时，服务端仍按旧的 `saveData.backpack` 判断“道具还在”
4. 卖出成功后，客户端本地同步只尝试从背包扣，不会扣仓库
5. 随后如果本地仓库状态被保存，玩家会同时拿到卖出收益和仓库副本

这不是纯显示问题，而是线上经济漏洞。

已知镜像现象也支持这一判断：
- “仓库的道具拿到背包后，要存一次档才能被黑市识别”

这说明黑市识别口径和仓库真实状态目前并不统一。

---

## 3. 本任务为什么现在必须做

这不是一般体验 bug，而是潜在刷收益漏洞。

如果不先热修：
- 玩家可以尝试利用时序窗口刷仙石/黑市收益
- 仓库与黑市之间的数据分裂会继续污染线上存档
- 后续 `R3` 之前的任何存档相关优化，都可能在带病状态下继续叠加复杂度

本任务的价值是：
- 先止血
- 先让客户端正常操作路径不再触发漏洞
- 再决定是否需要单开更重的服务端修复

---

## 4. 本任务的成功标准

成功标准只有 8 条：

1. 玩家在仓库存取或仓库扩容后，会进入“仓库/背包未同步”状态
2. 在该未同步状态下，黑市卖出请求被客户端明确阻止，不能继续发出
3. 该阻止必须有明确提示文案，例如“仓库/背包数据未同步，请先保存后再进行黑市交易”
4. 仓库相关操作成功后，必须请求一次保存，缩短旧快照窗口
5. `game_saved` 和 `game_loaded` 后，未同步状态会被清除
6. 黑市卖出成功后的本地同步，若本地扣除失败，不允许继续发 `save_request`
7. 本任务不修改协议、不修改服务端 handler、不修改存档结构、不 bump 版本号
8. 至少新增一组纯 Lua 自动化测试覆盖本次热修的核心语义

只要其中任意一条不满足，本任务不算完成。

## 8. 固定执行步骤

严格按这个顺序做。

### 步骤 1：先把“未同步状态”做成显式语义

不要继续依赖“玩家自己记得先存档”。

要求：

1. 仓库成功存入后，标记未同步
2. 仓库成功取出后，标记未同步
3. 仓库成功扩容后，标记未同步
4. `game_saved` 后清除未同步
5. `game_loaded` 后清除未同步

这个状态可以放在：
- `GameState` 上
- 或一个很小的叶模块里

但必须满足：
- 默认值明确
- 生命周期清楚
- 不依赖 UI 面板是否已打开

### 步骤 2：仓库操作成功后必须请求保存

在以下成功路径后补 `save_request`：

1. `StoreItem`
2. `StoreItemToSlot`
3. `RetrieveItem`
4. `UnlockNextRow`

注意：
- 这里只是缩短窗口，不是根治全部问题
- 不要因为补了 `save_request` 就删掉未同步状态

### 步骤 3：黑市卖出前增加硬性前置校验

卖出前如果仓库/背包未同步：

- 不发送卖出请求
- 不进入确认后的真实提交
- 给出明确提示

最低要求：
- 在真正发送 `SendSell(...)` 前拦住

更好的做法：
- 在卖出按钮可点条件和确认逻辑里都统一使用同一套前置判断

但不要为了“更优雅”去重写整个黑市 UI。

### 步骤 4：卖出成功后的本地同步失败，必须硬拦截

这是本任务最关键的止血点。

当前问题是：
- 服务端卖出成功后
- 客户端本地只尝试从背包扣
- 如果实际道具已在仓库，本地扣除会失败
- 代码却仍继续 `save_request`

必须改成：

1. 本地扣除结果必须可判断成功/失败
2. 如果失败：
   - 不发 `save_request`
   - 记录高优先级错误日志
   - 立即给出错误提示
   - 立即触发一次黑市数据刷新
3. 如果成功：
   - 才允许继续 `save_request`

这里不要求你顺手做全局强制回档或重新登录流程。  
本任务只要求：**本地扣除失败时，绝不能继续保存当前本地状态。**

### 步骤 5：不要扩大到服务端修复

如果你在执行中判断：
- 仅靠客户端热修无法防住“改包客户端手工发协议”的攻击面

这是合理判断。  
但本任务不允许你直接扩成服务端大改。

正确做法是：
- 先完成当前客户端热修
- 在完成报告里单列“服务端仍存在的理论攻击面”
- 后续如有必要，再单开服务端权威化任务

### 步骤 6：补纯 Lua 自动化测试

新增：

- `scripts/tests/test_black_merchant_warehouse_consistency.lua`

最低覆盖：

1. 仓库存入成功后：
   - 标记未同步
   - 发出一次 `save_request`
2. 仓库取出成功后：
   - 标记未同步
   - 发出一次 `save_request`
3. `game_saved` 后清除未同步
4. `game_loaded` 后清除未同步
5. 未同步状态下，黑市卖出前置校验返回禁止
6. 本地卖出扣除失败时，不发 `save_request`
7. 本地卖出扣除成功时，允许发 `save_request`

如直接测试 `BlackMerchantUI.lua` 太重，可以把纯逻辑抽到一个极小 helper 再测。  
但不要把 helper 做成新的大模块。

### 步骤 7：接入现有门禁

如果测试是纯 Lua，就接进现有 runner。

至少跑：

```bash
python3 scripts/tests/_run_via_lupa.py
python3 scripts/tests/_run_via_lupa.py --all
```

blocking 不能变红。

## 9. 本任务的关键限制

这一轮只做客户端热修止血，不做完整架构修复。

本任务不追求：

1. 黑市与仓库服务端完全权威统一
2. 所有恶意客户端攻击面一次性封死
3. 仓库系统服务端化
4. 黑市协议重做

本任务追求的是：

1. 正常玩家操作路径不再能稳定复现刷收益
2. 本地扣除失败时不再把脏状态继续保存
3. 仓库和黑市之间有显式同步门

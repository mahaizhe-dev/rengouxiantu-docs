## 2. 总体结论

### 2.1 当前最需要解决的问题

1. `main.lua` 和 `server_main.lua` 承担了过多职责。
2. 存档系统虽然已经拆分文件，但“数据层”仍直接依赖 UI 和业务模块。
3. 客户端、服务端、迁移逻辑、旧格式兼容逻辑同时存在，存档链路复杂度过高。
4. 网络模式下 `CloudStorage` 不是严格等价抽象，部分写入路径语义不一致。
5. UI 层直接触发大量业务和存档事件，事务边界不清晰。
6. 主循环、渲染和怪物更新仍以全量遍历为主，内容增长后性能风险会继续上升。
7. 测试脚本数量不少，但缺少统一执行入口和 CI 门禁，回归收益没有完全兑现。

### 2.2 优化顺序

必须按照下面顺序推进，不能倒置：

1. 先补线上护栏和自动化回归能力。
2. 再收敛存档和 C/S 边界。
3. 再拆主入口和巨型模块。
4. 最后做性能和 UI 结构治理。

原因：

- 这是线上项目，重构优先级必须服从“可回滚、可观测、可验证”。
- 没有护栏直接重构，风险会首先落到存档损坏、读档失败、排行榜异常、奖励重复发放。

## 3. 线上项目改造原则

以下原则建议设为硬约束：

### 3.1 存档安全原则

1. 所有存档结构变更必须保持向后兼容至少一个发布周期。
2. 所有迁移逻辑必须只存在一个权威入口。
3. 所有服务端写入必须满足幂等。
4. 所有关键存档变更必须可降级回旧结构读取。
5. 禁止客户端直接决定服务端权威数据。
6. 新增存档字段时只能“追加”，不得先删除旧字段读取能力。

### 3.2 线上发布原则

1. 所有高风险改动必须加开关。
2. 所有高风险路径必须保留旧实现旁路。
3. 所有重构都要允许灰度。
4. 所有变更必须定义失败判据和回滚动作。
5. 先做“等价重构”，再做“行为优化”。

### 3.3 测试原则

1. 每个 P0/P1 任务必须至少有一个脚本化回归测试。
2. 存档链路测试必须覆盖单机、联网、迁移、恢复、断线、锁竞争。
3. 服务端逻辑必须尽量抽离成纯 Lua 可跑测试，不依赖引擎场景。
4. 回归测试必须提供统一 runner，而不是靠手工 `require`。

   - 固化若干真实存档样本：

  - 迁移

## P1-3 重构 `SaveSystemNet` 请求状态机

### 目标

把单槽 pending 结构改成可扩展的 request map。

## 9. 结论

这不是一个适合“大重写”的阶段，而是适合“护栏先行、边界收敛、逐段替换”的阶段。

最优策略不是立刻优化战斗数值或 UI 表现，而是：

1. 先把存档和服务端主链路做成可测试、可回滚、可灰度。
2. 再拆 `main.lua` 和 `server_main.lua`。
3. 最后再做性能和目录清理。

如果只能选一个最高优先级方向，建议优先做：

`P0-1 + P0-2 + P0-3 + P0-5`

这四项完成后，后续重构的线上风险会明显下降。

## 1. 任务定义

任务ID：`P0-3R`

任务名称：`SaveDTO 边界收口与报告补齐`

变更类型：`缺陷修复 + 交付收口`

目标：

- 确认 `SaveDTO` 的代码边界与 `P0-3A` 合同是否一致
- 如果不一致，收口为“合同边界”和“envelope/透传边界”分离的实现
- 补齐 `P0-3` 完成报告，提供正式验收所需的 Git 信息和原始测试输出
- 在不改变默认线上行为的前提下，为 `P0-4` 提供可靠前置

---

## 2. 为什么必须先做这个收口

当前 `P0-3` 主提交已经完成了第一阶段等价重构，但复核时发现两个阻塞项：

1. `SaveDTO.lua` 当前实现把 `22 个 DTO 字段` 与 `6 个 coreData envelope 字段` 合并成了同一个 `SCHEMA_FIELDS`
2. `P0-3` 完成报告尚未进入 Git，缺少正式验收需要的 `commit / parent / 原始测试输出`

如果不先收口：

- `P0-3A` 冻结的边界会被冲淡
- 后续 `P0-4` 容易把 `SaveDTO` 错当成“完整 coreData 合同”
- 我无法按既定工作流正式放行 `P0-4`

---

## 3. 本任务的成功标准

成功标准只有 7 条：

1. `P0-3` 完成报告进入 Git。
2. 完成报告中包含真实 `commit SHA / parent SHA / diff 摘要 / 原始测试输出`。
3. `SaveDTO` 代码层明确区分：
   - `DTO_FIELDS`
   - `ENVELOPE_FIELDS`
   - `PASSTHROUGH_FIELDS`
4. `P0-3A` 合同中的 22 个字段域仍然是 `SaveDTO v1` 的唯一主体。
5. `code_version / timestamp / bossKills / _bossKillsMigrated / bulletin / artifact_ch4` 不再被表述成 DTO 主体字段。
6. `SAVE_PIPELINE_V2=false` 下默认线上行为保持不变。
7. blocking 门禁继续保持绿色。

只要 7 条中有任一条不满足，本任务不算完成。

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：先判断是“代码边界错误”还是“表述错误”

必须先明确：

- 如果 `SaveDTO` 只是做了 coreData 分层包装，但测试和命名把它说成 DTO 主体，那么要收口命名和测试
- 如果 `SaveDTO` 真的把 envelope 字段当成 DTO 主体参与后续推导，那么要收口实现

不要先改，再解释。

### 步骤 2：把 22 + 6 + 3 三层边界说清楚

推荐最小结构：

- `DTO_FIELDS`：22 个字段域
- `ENVELOPE_FIELDS`：6 个附加字段
- `PASSTHROUGH_FIELDS`：3 个旁路/透传字段

如果你使用不同命名，也必须满足同样的职责分离。

固定要求：

1. 22 个字段域保持为 `SaveDTO v1` 主体
2. 6 个字段只能被归类为 envelope / save wrapper / coreData extra
3. 3 个透传字段继续独立于 DTO 主体

### 步骤 3：收口测试表述

`test_save_dto_contract.lua` 必须同步收口：

1. 不再把 28 个 schema 字段当成“DTO 主体字段数”
2. 要明确断言：
   - DTO 主体字段 = 22
   - envelope 字段 = 6
   - passthrough 字段 = 3
3. round-trip 测试可以继续验证整体 coreData 重组能力，但文案和断言不能混淆“DTO 主体”和“完整 coreData”

### 步骤 4：补齐 P0-3 完成报告

必须新增并提交：

- `docs/优化专题/2026-05-12-P0-3-完成报告.md`

报告必须包含：

1. 真实 `commit SHA`
2. 真实 `parent SHA`
3. `diff 摘要`
4. `实际修改文件`
5. `测试命令`
6. `原始测试输出`
7. `rules.md 对照检查`
8. `为什么 P0-3R 是补收口，而不是重做 P0-3`

不要只写摘要，不要只贴结论。

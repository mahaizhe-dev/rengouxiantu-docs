# BM-S4D：黑市状态边界与自动回收过天收口任务卡

## 1. 任务定位

- **任务 ID**：`BM-S4D`
- **任务类型**：`合并收口任务`
- **目标**：一次性收口两类同源问题：
  1. **黑市锁已解除，但仍被“背包数据未同步”挡住**
  2. **黑市自动回收只在首日触发，后续过天不再稳定触发**

这两个问题表面不同，本质相同：

- **黑市相关状态没有在正确的时间边界上被清理、推进或对齐**

所以本轮不再拆 2 张小卡，而是合并成一张“状态边界收口”卡。

---

## 2. 本轮明确问题

### 问题 A：锁已解除，但黑市仍提示“背包数据未同步”

当前代码链路里：

- `BlackMarketSyncState.IsBlocked()` 仍会把 `SaveSession.IsDirty()` 视为 L2 全局未同步门
- `SaveSession._active` 只会在 `SaveSession._flush()` 时清零
- **成功存档（`game_saved`）并不会清除 `SaveSession` 活跃状态**

这意味着：

- 玩家已经手动保存成功
- 交易保护锁已经被清除
- 但如果此前某个炼丹/锻造/洗练会话触发过 `SaveSession.MarkDirty()`
- 且该会话没有通过 `SaveSession.Flush()` 自己收口
- 那 `SaveSession.IsDirty()` 仍可能继续返回 `true`
- 黑市 L2 继续挡卖出

这会造成一种非常糟糕的体验：

- 玩家看到锁没了
- 却仍然卖不出去
- 表现成“黑市系统坏了”

### 问题 B：自动回收只在第一天触发，后续过天不稳定触发

当前 `BlackMerchantHandler.RecycleTick()` 使用：

```lua
local today = os.date("%Y%m%d")
local cloudDate = moneys[BMConfig.RECYCLE_DATE_KEY] or 0
local todayNum = tonumber(today) or 0
local gap = todayNum - cloudDate
```

然后直接把这个 `gap` 同时用于：

1. **日期推进**
2. **时间异常判断（`gap > MAX_RECYCLE_GAP`）**

这是危险的，因为：

- `YYYYMMDD` 不是“天数序号”
- 月切换时：
  - `20260531 -> 20260601`
  - 数值差是 `70`
  - 实际只过了 `1` 天

所以现在的逻辑会把正常过天误判成异常跳天，从而：

- 只推进日期
- 不执行自动回收

这和其他日常系统明显不一致。  
其他日常一般使用：

- `quota(..., "day")`
- 或明确的服务器日边界

而不是自己拿 `YYYYMMDD` 做数值减法。

## 3. 本轮目标

本轮一次性完成 4 件事：

1. **修正 SaveSession 的状态边界**
   - 成功存档后，`SaveSession` 不能继续把黑市挡在 L2 之外

2. **校准 BlackMarketSyncState 对 SaveSession 的使用**
   - 防止“锁已清、存档已成功，但 L2 仍误拦”

3. **修复自动回收的过天判断**
   - 正常过一天必须继续触发
   - 月切换/年切换不能误判

4. **补对应回归测试**
   - 黑市卖出路径
   - 自动回收过天路径

## 6. 固定实施步骤

### Step 1：先收口 SaveSession 边界

执行 AI 必须先回答：

- `SaveSession.IsDirty()` 现在依赖什么状态
- 为什么成功存档后它仍可能保持 `true`
- 应该在哪个事件边界清理

本轮可接受的修法方向：

- 给 `SaveSession` 增加显式 `Reset()` / `Clear()` 之类的公共 API
- 在 `game_saved` 和 `game_loaded` 时清除会话活跃状态

但必须确保：

- 只是清掉“已被成功存档覆盖”的旧会话
- 不是把未保存会话静默吞掉

### Step 2：收口 BlackMarketSyncState 的 L2 门

至少要保证：

- 锁已经清掉
- 且存档已经成功
- 黑市不会再继续显示“背包数据未同步，请稍后再试”

本轮不要求重构整套 L2 模型，只要求：

- 不再让 `SaveSession` 的陈旧活跃态误挡黑市

### Step 3：修复自动回收过天判断

要求：

- 不再用 `YYYYMMDD` 的整数差值直接充当“过了几天”
- 正常跨天、跨月、跨年都能正确判断
- 仍保留“疑似异常跳天不回放多天收益”的护栏

可接受方向：

1. 继续保留 `RECYCLE_DATE_KEY = YYYYMMDD`
   - 但单独引入“真实日差”计算辅助函数
   - **真实日差** 用于异常判断
   - **数值差** 仅用于把旧值推进到今天

2. 或改成更稳的日期键表示
   - 但必须说明迁移方式与线上兼容方式

### Step 4：补测试

至少补 2 组：

#### A. 黑市未同步收口测试

- `SaveSession.MarkDirty()` 后，`IsBlocked()` 命中
- 触发模拟 `game_saved`
- `SaveSession` 清理
- `IsBlocked()` 不再因旧会话而命中

#### B. 自动回收过天测试

至少覆盖：

- 同月隔天：`20260513 -> 20260514`
- 跨月隔天：`20260531 -> 20260601`
- 跨年隔天：`20261231 -> 20270101`
- 异常大跨度：仍触发“只推进日期，不执行回收”的保护逻辑

## 8. 失败判据

出现以下任一项即判失败：

- 仍然能稳定复现“锁没了但仍显示未同步”
- 只是把 L2 全局门暂时关掉，而不是修边界
- 自动回收仍然依赖 `YYYYMMDD` 数值差充当真实天数
- 测试没有覆盖跨月/跨年

## 1. 任务定义

任务ID：`P0-4R`

任务名称：`迁移状态模型收口与报告归档`

变更类型：`缺陷修复 + 交付收口`

目标：

- 让 `MigrationPolicy.Evaluate()` 先区分“现实状态”，再由策略决定动作
- 避免 `flag=true` 时所有情况都被压成同一个 `MIGRATION_DISABLED`
- 把 `P0-4` 完成报告修正到可正式验收状态

---

## 2. 为什么必须先做这个收口

当前 `P0-4` 的方向是对的，但还留了两个阻塞项：

1. `MigrationPolicy.Evaluate()` 先判断 `MIGRATION_GLOBALLY_ENABLED=false`，导致 `needMigration / slotCount` 这些输入在当前线上状态下失去区分能力
2. `P0-4` 完成报告中的 `commit SHA` 与实际 Git 提交不一致，且报告没有落在 `docs/优化专题/`

如果不先收口：

- 后续执行 AI 会误以为“迁移状态模型已经稳定”
- `P0-6` 会建立在不精确的迁移状态语义上
- 当前工作流下，`P0-4` 还不能正式放行

---

## 3. 本任务的成功标准

成功标准只有 6 条：

1. `MigrationPolicy.Evaluate()` 在 `flag=true` 时，能够区分至少以下状态：
   - `NOT_NEEDED`
   - `ALREADY_MIGRATED` 或等价状态
   - `ELIGIBLE_*` 或等价状态
   - `MIGRATION_DISABLED`
2. “迁移全局禁用”只能影响动作决策，不能抹平现实状态本身。
3. `ShouldAutoMigrate()` / `ShouldBlockEntry()` 可以继续全部返回 false，但前提是状态已经被正确区分。
4. `test_migration_state_policy.lua` 同步修正，不再把“全压成 DISABLED”当成正确行为。
5. `P0-4` 完成报告中的 Git 信息与真实提交一致。
6. blocking 门禁继续保持绿色。

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：先拆“状态”和“动作”

必须先明确两层：

1. **状态层**
   - 现实上属于哪一种迁移状态
2. **策略层**
   - 当前产品策略是否允许自动迁移 / 阻断入口 / 提示用户

如果代码里还是把这两层混在一起，本任务就没有完成。

### 步骤 2：修正 Evaluate 优先级

推荐顺序：

1. 先根据 `needMigration / slotCount / 其他现有输入` 判定现实状态
2. 再让 `ShouldAutoMigrate()` / `ShouldBlockEntry()` 根据全局策略决定动作

最低要求：

- 有存档的情况，不应直接被归成 `MIGRATION_DISABLED`
- `needMigration=false` 的情况，应能落到 `NOT_NEEDED` 或等价状态
- “可迁移但当前不执行”应与“根本不需要迁移”区分开

### 步骤 3：同步测试

`test_migration_state_policy.lua` 必须改：

1. 不再断言 “flag=true + global=false → 所有组合都等于 DISABLED”
2. 增加“状态已区分，但动作仍不执行”的断言
3. 保留 `flag=false` 的兼容断言

### 步骤 4：修正完成报告

必须新增或修正：

- `docs/优化专题/2026-05-12-P0-4-完成报告.md`

固定要求：

1. 使用真实 `commit SHA`
2. 使用真实 `parent SHA`
3. 若旧报告在错误目录，需说明最终以 `docs/优化专题/` 版本为准
4. 原始测试输出必须保留

- 改 `SaveSystemNet.lua`

才能完成，那就说明它已经偏离 `P0-4R` 范围，必须停手上报。

## 9. 本任务的停手条件

出现以下情况，停止编码，改交付问题说明：

1. 发现仅靠当前输入根本无法区分“已迁移”和“不需要迁移”
2. 发现 `needMigration` 信号本身已经失真到不能作为状态来源
3. 发现报告修正需要依赖丢失的历史测试输出
4. 需要编辑超过 3 个运行时代码文件

如果停手，仍然要交付：

- 当前状态模型还能保真的最小集合
- 哪些状态必须推迟到后续任务再区分
- 为什么 `P0-4` 不能直接放行

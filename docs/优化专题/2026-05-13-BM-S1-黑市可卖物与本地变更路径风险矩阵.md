## 1. 方法论

### 1.1 数据来源

| 来源 | 文件 | 用途 |
|------|------|------|
| 商品配置表 | `BlackMerchantConfig.lua` | 79 个 BM 可卖物完整清单 |
| 服务端卖出链路 | `BlackMerchantHandler.lua:HandleSell` | 服务端验证逻辑 |
| 客户端卖出门禁 | `BlackMerchantUI.lua` | BM-01 / BM-01A 客户端检查 |
| 背包操作 | `InventorySystem.lua` | 本地消耗/添加/卖出入口 |
| 仓库操作 | `WarehouseSystem.lua` | 入仓/出仓/脏标记机制 |
| 存盘会话 | `SaveSession.lua` | 延迟存盘窗口（≤5 ops / ≤20s） |
| 消耗 UI | `AlchemyUI / DragonForgeUI / PetPanel / TemperUI` | 各系统消耗路径 |
| 服务端背包 | `BackpackUtils.lua` | 服务端背包读写 |

### 1.2 风险等级定义

| 等级 | 定义 |
|------|------|
| **P0** | 当前存在可利用的幻影卖出窗口，玩家可通过正常操作复制仙石 |
| **P1** | 存在理论风险窗口，但需要精确时序或特殊条件才能利用 |
| **P2** | 低风险，需要极端操作或多步组合才可能触发 |
| **已止血** | 已被 HOTFIX-BM-01 或 BM-01A 覆盖，客户端门禁阻断 |
| **安全** | 该物品不存在本地消耗路径，或消耗路径直接触发 `save_request` |

### 1.3 核心漏洞模型

```
客户端消耗物品（本地扣减）
    ↓
SaveSession.MarkDirty()        ← 延迟窗口：≤5 ops 或 ≤20s
    ↓                              在此窗口内，服务端 saveData.backpack 仍有旧数量
打开黑市 → 卖出同一物品
    ↓
HandleSell 读取 saveData.backpack → 校验通过（stale data）
    ↓
物品被"幻影卖出"，仙石凭空增发
```

**BM-01 门禁**: `WarehouseSystem.IsDirty()` → 仅拦截仓库操作（入仓/出仓/解锁行）
**BM-01A 门禁**: `_BM_SENSITIVE_CONSUMABLES` → 仅覆盖 3 个令牌盒

## 4. 自动回收 (RecycleTick) 风险面

| 维度 | 当前状态 |
|------|---------|
| 开关 | `RECYCLE_ENABLED = true` |
| NPC 名称 | "胤"（代码）vs "大黑无天"（设计文档）—— 不一致 |
| 执行频率 | 每日一次，`RecycleTick` 每帧检查日期变化 |
| 防篡改 | `MAX_RECYCLE_GAP = 2`，超过不执行只更新日期 |
| 执行模式 | "先占坑再干活"：先写 cloud date，再处理物品 |
| 故障影响 | `pcall` 保护，单物品失败跳过，不回滚日期 |
| 回收范围 | 库存满（`stock >= max_stock`）的物品 |
| 回收量 | `WeightedRecycleAmount()`：0件(25%) / 1件(50%) / 2件(25%)，期望 1.0 件/天 |
| 仙石影响 | 回收 = 减少系统库存，不涉及仙石增减 |
| WAL 保护 | 无。回收是系统库存操作，不涉及玩家背包/仙石 |

**回收专项风险**:

1. **日期源未验证**: 使用 `os.date` 本地时间判断日期变化，服务端环境下由宿主机时钟决定。如果服务端时区配置异常，可能导致重复回收或漏回收。
2. **fire-and-forget commit**: 每个物品回收使用独立的 `serverCloud.money.add` 提交，不等待确认。如果中间网络抖动，部分库存减少可能丢失。
3. **无审计日志**: 回收操作通过 `TradeLogger.LogPublic` 记录，但日志也是 serverCloud 异步写入，与回收操作之间无事务保证。

## 6. BM-01 / BM-01A 覆盖边界精确描述

### 6.1 BM-01: `WarehouseSystem.IsDirty()` 门禁

**覆盖**: 仅当 `WarehouseSystem._dirty == true` 时阻断 BM 卖出按钮。

**触发 `_dirty = true` 的操作** (来源: `WarehouseSystem.lua`):
- `StoreItem()` — 物品入仓
- `StoreItemToSlot()` — 物品入仓到指定位
- `RetrieveItem()` — 物品出仓
- `UnlockNextRow()` — 解锁仓库新行
- `MarkDirty()` — 被 InventorySystem BM-01A 调用

**清除 `_dirty = false` 的时机**:
- `game_saved` EventBus 事件
- `game_loaded` EventBus 事件

**未覆盖的本地变更**:
- `InventorySystem.ConsumeConsumable()` 中非 `_BM_SENSITIVE_CONSUMABLES` 的物品
- `SaveSession.MarkDirty()` 管理的所有延迟存盘
- PetPanel 的技能书学习
- 任何不经过仓库的背包物品变更

### 6.2 BM-01A: `_BM_SENSITIVE_CONSUMABLES` 门禁

**覆盖**: 仅 3 个令牌盒在 `ConsumeConsumable` 时触发 `WarehouseSystem.MarkDirty()` + `save_request`。

```lua
InventorySystem._BM_SENSITIVE_CONSUMABLES = {
    wubao_token_box  = true,
    sha_hai_ling_box = true,
    taixu_token_box  = true,
}
```

**未覆盖**: 所有草药、附灵玉、技能书、金砖等通过 `ConsumeConsumable` 消耗的物品。

### 6.3 门禁缺口总结

BM-01 + BM-01A 共覆盖了 **31/79** 个物品的安全（含本身安全的 28 个 + 已止血的 3 个）。

**48 个物品存在未被门禁覆盖的本地变更路径**:
- 4 个 P0（附灵玉）
- 40 个 P1（草药 10 + 技能书 30）
- 4 个 P2（特殊装备）

> 注：技能书 P1 实际 34 个，草药 10 个，但部分草药的 AlchemyUI Flush 机制收窄了窗口。总计 P0+P1+P2 = 48 个未覆盖。

## 7. 为后续任务的直接输入

### 7.1 → BM-S2（未同步门扩展）

BM-S2 需要将以下操作纳入"未同步门"：

| 操作类型 | 涉及物品 | 当前入口 |
|---------|---------|---------|
| `ConsumeConsumable` 全量 | 所有消耗品 | `InventorySystem.ConsumeConsumable()` |
| `SaveSession.MarkDirty` | 所有走延迟存盘的操作 | `SaveSession.MarkDirty()` |
| 宠物技能学习 | 技能书 34 种 | `PetPanel` 技能学习逻辑 |

**建议方案**: 不再按物品名单补敏感列表，而是在 `ConsumeConsumable` 统一触发 `WarehouseSystem.MarkDirty()`，或引入独立的 `BackpackDirty` 标记。

### 7.2 → BM-S3（服务端权威补强）

服务端 `HandleSell` 当前验证逻辑：
```
Session.ServerGetSlotData → 读取 saveData.backpack → CountItemHeld → 校验
```

缺口：`saveData.backpack` 是上一次 `save_request` 的快照，不反映客户端本地变更。

BM-S3 需要补强：
1. 服务端在卖出前强制 flush 客户端存档
2. 或引入版本号/序列号验证存档新鲜度
3. 或卖出操作本身成为原子事务（read-modify-write）

### 7.3 → BM-S4（自动回收治理）

1. 确认 `RECYCLE_ENABLED` 当前值 `true` 是否为预期状态（设计文档写 false）
2. NPC 名称 "胤" vs "大黑无天" 统一
3. fire-and-forget commit 模式的容错升级
4. 回收操作与仙石/灵韵结算的边界划清

## 1. 结论

**新路线整体可行，建议采纳，但需要做两处调整。**

新路线相比原路线有三个明显改进：

- 将"大而全"的 P0-6 拆成渐进式的 R3（等价拆分）+ R4（排行收口），降低了单次改动的爆炸半径
- 将日志/诊断治理前移（R1），为后续高风险任务提供定位手段
- 将 SaveSystemNet 状态机收口（R2）前移到服务端拆分（R3）之前，先稳定客户端协议层

需要调整的两处：

1. **R0 可以降级** — rules.md 常量已验证无漂移，R0 不再需要作为独立任务卡
2. **R2 建议拆出一个 R2-prep** — SaveSystemNet 与 CloudStorage 之间的回调分发机制存在隐藏耦合，直接做状态机收口前需要先摸清边界

## 3. 顺序判断

### 3.1 原 P0-6 是否确实范围过大？

**是。** 代码审查证实：

- `server_main.lua` 共 ~1800 行、50+ handler，其中核心 CRUD 5 个函数占 ~1100 行
- `HandleSaveGame` 单函数 500+ 行，承载存档验证 + 排行计算 + 账户级数据 + 版本守卫四重职责
- 排行写入（rank2/trial/prison）散布在 SaveGame、DeleteChar、FetchSlots 三处
- 账户级数据（account_*）与 per-slot 数据混在同一 BatchSet

如果直接做"服务端事务服务化"，改动面覆盖以上全部，单次提交风险过高。文档的拆分建议（R3 只拆 FetchSlots/LoadGame/SaveGame 的壳、R4 单独收排行）是合理的。

### 3.2 R2（SaveSystemNet 状态机收口）是否应前移到 R3 前？

**是，且应进一步强化。** 审查发现 SaveSystemNet 有三个结构性问题：

| 问题 | 描述 | 影响 |
|------|------|------|
| 晚到响应不一致 | FetchSlots 回调给"当前 pending"而非原始发起者，Load 则直接丢弃。行为不统一 | 快速重试场景下回调错配 |
| 无取消语义 | 新请求覆盖旧 pending，旧回调永不调用 | 调用方可能永久挂起等待 |
| 死代码残留 | MigrateToServer/MigrateFromLocalFile 仍是公开函数，S2C_MigrateResult 处理器仍存在 | 增加维护负担和误用风险 |

这些问题如果不先修，R3 拆分服务端链路时可能踩到客户端协议层的暗坑。

### 3.3 R1（日志/诊断治理）是否应前移？

**是，但范围可以收窄。** 理由：

- 用户无稳定遥测，后续定位只能靠日志
- 当前 server_main.lua 中已有大量 `print` 日志，但等级混乱
- 但完整的日志治理工作量大，可以只做"关键链路日志标准化"而不做全局治理

## 4. 拆分建议

### R1 仍然过大，建议收窄

原文 R1 要求覆盖 `server_main.lua`、`SaveSystemNet.lua`、`CloudStorage.lua`、`systems/save/*`，外加"统一开关入口"和"字段标准化"。

**建议只做：**

1. 定义日志等级枚举（error/warn/info/diag）
2. 仅改造 SaveGame 和 FetchSlots 两条最危险链路的日志
3. 不改业务逻辑、不改协议

这可以在 1 张任务卡内完成，不需要拆分。

### R2 需要插入准备任务（R2-prep）

**理由：** SaveSystemNet 与 CloudStorage 之间存在隐藏耦合：

| 耦合点 | 说明 |
|--------|------|
| CloudStorage 的 requestId-回调匹配 | FIFO 兜底可能在多并发时错配 |
| SaveSystemNet 的 pending 覆盖 | 调用方不知道旧回调是否被丢弃 |
| 迁移路径死代码 | 3 个函数 + 1 个 S2C 处理器仍存在 |

**建议 R2-prep 只做：**

1. 画出 SaveSystemNet ↔ CloudStorage ↔ server_main 的完整请求-响应流图
2. 标记所有 pending 状态的生命周期和所有者
3. 输出一份"状态机收口影响范围清单"

R2-prep 是 docs-only，不改代码。

### R3 范围合理，不需要进一步拆分

文档已明确只覆盖 FetchSlots/LoadGame/SaveGame，排除迁移和排行。这是正确的粒度。

## 6. 推荐下一张正式任务卡

### 推荐：R1 诊断与日志治理（收窄版）

**为什么是 R1 而不是其他：**

| 候选 | 不选理由 |
|------|---------|
| R0 | 已完成 — 常量无漂移、基线已确认 |
| R2 | 需要先做 R1 提供定位手段；且建议先做 R2-prep |
| R3 | 依赖 R2 先稳定客户端协议层 |

**为什么是 R1：**

1. **前置依赖最少** — 不依赖任何其他 R 步骤，只依赖当前已通过的基线
2. **风险最低** — 只改日志输出，不改业务逻辑，不改协议，不改数据结构
3. **收益最直接** — 后续 R2/R3 的调试能力立即提升，在用户无遥测条件下尤为关键
4. **范围可控** — 收窄到只覆盖 SaveGame + FetchSlots 两条链路的日志分级，1 张任务卡可完成

**建议 R1 任务卡范围：**

- 在 `scripts/` 中新建 `utils/Logger.lua`，定义 error/warn/info/diag 四级
- 改造 `server_main.lua` 的 HandleSaveGame 和 HandleFetchSlots 中的 print → Logger 调用
- 改造 `SaveSystemNet.lua` 中的 print → Logger 调用
- 添加诊断开关（默认开启 info 及以上，diag 需显式开启）
- 不改业务逻辑，不改协议
- blocking gate 通过即可合入

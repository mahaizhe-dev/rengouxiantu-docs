## 5. 关键实现说明

### strict 模式如何判断 unsupported

在 `builder:Save()` 的 else 分支（`coreData and slot` 都为 nil 时）检查 `FeatureFlags.isEnabled("CLOUDSTORAGE_STRICT_BATCHSET")`。判断逻辑复用原有的 key 遍历结果：如果遍历完所有 `self._sets` 后 `coreData` 和 `slot` 都为 nil，说明没有 `save_{slot}` 或 `save_data_{slot}` 匹配项，即为非存档 batch。

新增 `IsSaveBatch()` helper 提供独立判断能力（供未来需要时使用），`CollectBatchKeys()` 收集所有操作 key 用于诊断。

### 错误码

`CloudStorage.ERR_UNSUPPORTED_BATCH = "UNSUPPORTED_BATCH"` — 字符串常量，稳定可断言。

### reason 中包含了什么

格式：`"Unsupported non-save BatchSet in network mode: desc=<desc> keys=[<key1>, <key2>, ...]"`

- `desc`: 调用方传入的操作描述
- `keys`: 排序后的完整 key 列表（包括 Set/SetInt/Delete），Delete 键带 `(del)` 前缀

### 哪些调用方补了最小处理

**无需修改任何调用方**。所有 5 个受影响调用点的 error callback 已有充分的日志输出：

- SavePersistence checkpoint: `print("[SaveSystem] Checkpoint write failed: "..reason)` — checkpoint 是备份，失败可接受
- SaveLoader login backup: `print("[SaveSystem] Login backup failed: "..reason)` — 登录备份在主数据加载后执行，失败不影响流程
- AtlasSystem SaveImmediate: `print("[AtlasSystem] Immediate save FAILED (batch): "..reason)` — atlas 是非关键数据

所有调用方的 error 回调都只做日志，不会误将错误当成功处理。

## 7. 风险与回滚

### 默认值下是否保持旧行为

**是**。`CLOUDSTORAGE_STRICT_BATCHSET` 默认值为 `false`，所有旧路径（包括假成功）完全保持不变。线上不受任何影响。

### strict 模式下已知 unsupported 清单

| 调用方 | desc | 影响 | 风险等级 |
|--------|------|------|---------|
| SavePersistence checkpoint ×2 | `prev_{slot}` | checkpoint 丢失，不影响主存档 | 低 |
| SaveLoader login backup ×2 | `login_{slot}` | 登录备份丢失，主数据已正常加载 | 低 |
| AtlasSystem SaveImmediate ×1 | `account_atlas` | atlas 数据未持久化 | 中（如有 atlas 跨设备需求则需后续补支持） |

### 回滚方式

1. **秒级回滚**: `FeatureFlags.setOverride("CLOUDSTORAGE_STRICT_BATCHSET", false)` — 运行时生效，无需重启
2. **代码回滚**: revert 本次 commit — 只改 3 个文件，无协议/服务端依赖

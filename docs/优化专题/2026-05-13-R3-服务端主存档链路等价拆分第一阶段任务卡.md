# 2026-05-13 R3 服务端主存档链路等价拆分第一阶段任务卡

这是 `R2` 线上相对稳定后的下一张正式任务卡。

本任务不是“服务端全面服务化”，也不是“顺手把黑市一起重构掉”。
本任务只做一件事：

**把 `server_main.lua` 中最核心的主存档读写链路先抽成 service 壳，保持线上行为等价，并明确不破坏当前黑市热修的一致性约束。**

## 1. 任务定义

任务 ID：`R3`

任务名称：`服务端主存档链路等价拆分第一阶段`

变更类型：`等价重构 + 服务端壳层收口`

目标：
1. 从 `server_main.lua` 中抽出主存档链路的 service 壳
2. 第一阶段只覆盖：
   - `FetchSlots`
   - `LoadGame`
   - `SaveGame`
3. 保持协议、存档结构、迁移链、现有返回语义不变
4. 不破坏当前 `HOTFIX-BM-01A` 已上线的黑市一致性止血效果

一句话说明：
**先把最核心的读写主链从巨型文件里拆出“外壳”，但不趁机重写业务。**

---

## 2. 为什么现在做

当前代码现实是：
1. `R2` 已经把客户端 `SaveSystemNet` 状态机收稳
2. 第五章前还值得再收一层的，不是 UI 大拆，也不是排行系统，而是服务端主存档链路
3. 如果继续把 `FetchSlots / LoadGame / SaveGame` 堆在 `server_main.lua` 里，后续第五章开发会进一步放大误改风险
4. 这一步仍然属于“等价拆分”，比直接碰 `R4/R5/R6` 更适合在上线项目里推进

但同时必须明确：
- 本任务不负责彻底修完黑市服务端权威一致性
- 本任务只要求“不要把黑市相关旁路写入和已上线热修语义改坏”

## 3. 成功标准

本任务只有 8 条成功标准，必须全部满足：

1. `FetchSlots / LoadGame / SaveGame` 的主要逻辑不再直接堆在 `server_main.lua` 大段实现中，而是委托给新 service 模块
2. `server_main.lua` 仍保留原有入口函数与订阅关系，不改协议名、不改入参出参
3. `FetchSlots / LoadGame / SaveGame` 的成功回包、失败回包、旧 fallback 行为保持等价
4. 不修改 `CURRENT_SAVE_VERSION`、不修改 `SaveMigrations`、不修改任意已有存档 key 名
5. `account_atlas / account_cosmetics / rank2_* / trial_* / prison_*` 仍保持当前旁路写法，不被误吞进新 service 的“主存档正文”
6. 黑市相关逻辑不被改坏：
   - 不改 `BlackMerchantHandler.lua`
   - 不改 BM 协议
   - 不把 BM 的 WAL、交易历史、兑换、自动收购顺手卷进本次 service
7. 至少新增一组纯 Lua 或最小契约测试，覆盖 service 壳的路由/边界语义
8. blocking gate 继续全绿

只要任意一条不满足，本任务不算完成。

## 6. 黑市相关硬约束

这是本任务新增的强约束，执行 AI 必须认真看。

### 6.1 本任务不负责修黑市服务端权威漏洞

不要把本任务做成“顺手重写黑市”。
当前黑市权威修复会在后续单开专题任务。

### 6.2 本任务必须避免破坏当前 BM-01A 的止血结果

至少保证以下事实不被改坏：
1. 正常客户端下，仓库存取和 3 个令牌盒本地消费后的未同步状态，仍会阻止黑市卖出
2. `SaveGame` 的服务端主链拆壳后，不能让客户端这类热修路径变成“发了保存但服务端新壳没按旧语义写入”
3. 不允许把黑市相关旁路数据误合并进主存档 service 的“正文 DTO”

### 6.3 本任务必须显式盘点黑市接触面

完成报告里必须单列一张表：
- `SaveGame` 当前会不会间接影响黑市相关数据
- 哪些黑市相关 key / WAL / 旁路写入，本任务明确不碰

至少说明以下点：
1. `BlackMerchantHandler` 不在本次写集
2. `WAL` 不在本次写集
3. `history / exchange / recycle` 不在本次写集
4. 当前 `SaveGame` 拆壳后，对 BM-01A 的前端 `save_request -> 服务端落盘` 语义是否保持不变

## 8. 固定执行步骤

严格按这个顺序做。

### 步骤 1：先交付主链盘点表

完成报告里先给一张表，至少列出：
1. `HandleFetchSlots`
2. `HandleLoadGame`
3. `HandleSaveGame`

每项都要说明：
1. 输入来源
2. 读哪些 key
3. 写哪些 key
4. 当前有哪些旁路写入
5. 当前有哪些明显不该进入本轮 service 壳的逻辑

如果盘点表写不清楚，不允许开始拆壳。

### 步骤 2：先抽 service 壳，不重写业务体

第一阶段目标是：
- `server_main.lua` 负责订阅和转调
- 具体业务放到 service

但 service 内部可以先包住旧逻辑，不要求立刻“洁净重写”。

不允许一上来就：
1. 改返回结构
2. 改错误语义
3. 改 key 访问策略
4. 把大段逻辑一边抽一边重新设计

### 步骤 3：主存档正文和旁路逻辑显式分区

service 壳内部必须把以下几层写清楚：
1. 主存档正文
2. 账号级旁路
3. 排行/试炼/镇狱塔旁路
4. 其他本轮不接管的旁路

允许只是用函数分区，不要求一次性架构完美。
但必须让后续人一眼看出：
- 哪些是主链
- 哪些是旁路
- 哪些是故意先不碰

### 步骤 4：黑市相关边界做 tombstone 式注释/隔离

在新 service 或 `server_main.lua` 委托点附近，必须明确注释：
- 黑市 handler 不在本次 service 范围
- 黑市 WAL 不在本次 service 范围
- 如果有共享存档写入口，只保持原行为，不新增 BM 语义分支

要求的是“边界冻结”，不是“顺手接管”。

### 步骤 5：补最小自动化测试

至少新增一层测试，覆盖：
1. `server_main.lua` 入口仍委托到新 service
2. service 接口边界不吞掉旁路字段
3. 黑市相关未在本轮写集内的事实可被断言或最小验证

可以是纯 Lua 契约测试，也可以是最小桩测试。
重点是“边界能防回归”，不是把整套服务端行为完整模拟。

### 步骤 6：接入现有门禁

至少跑：

```bash
python3 scripts/tests/_run_via_lupa.py
python3 scripts/tests/_run_via_lupa.py --all
```

如果新测试无法进入 blocking，必须在报告里解释原因。

## 9. 推荐实现方式

推荐，但不强制，拆成 3 个 service 壳：

1. `SlotReadService`
   - 负责 `FetchSlots`
2. `SaveLoadService`
   - 负责 `LoadGame`
3. `SaveWriteService`
   - 负责 `SaveGame`

要求：
1. 先把壳立起来
2. 原有逻辑尽量原位平移
3. 不追求本轮就做完进一步抽象

如果执行 AI 评估后认为 3 个壳过大，也允许拆成 2 个壳：
- `SlotAndLoadService`
- `SaveWriteService`

但必须在报告里说明为什么这么拆更安全。

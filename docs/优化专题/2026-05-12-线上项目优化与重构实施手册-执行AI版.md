# 2026-05-12 线上项目优化与重构实施手册（执行 AI 版）

## 1. 文档用途

这份文档不是“为什么优化”，而是“怎么安全落地”。

使用方式：

1. 先看 [2026-05-12 线上项目优化与重构执行报告](./2026-05-12-线上项目优化与重构执行报告.md) 确定优先级和目标。
2. 再按本手册逐任务执行。
3. 如果报告与本手册冲突：
   - 任务顺序、停手条件、测试要求，以本手册为准。
   - 优化目标和风险判断，以执行报告为准。

快速入口：

- [执行方最小操作清单](./2026-05-12-执行方最小操作清单.md)
- [P0-1 统一测试入口首任务卡](./2026-05-12-P0-1-统一测试入口首任务卡.md)

这份手册默认执行者能力有限，因此要求：

- 不允许靠“猜测项目意图”推进。
- 不允许把多个高风险任务混在一个改动里。
- 不允许在没有自动化回归的情况下改存档主链路。

## 2. 强制执行规则

以下规则是硬约束，不满足时必须停止，不得继续编码。

### 2.1 顺序规则

必须按下面顺序执行，不允许跳步：

1. `P0-1`
2. `P0-2`
3. `P0-5`
4. `P0-3`
5. `P0-4`
6. `P0-6`
7. `P1-1`
8. `P1-2`
9. `P1-3`
10. `P1-4`
11. `P2-1`
12. `P2-2`
13. `P2-3`
14. `P2-4`

原因：

- `P0-1` 提供自动化验证入口。
- `P0-2` 提供开关和回滚能力。
- `P0-5` 先修复存档调用语义歧义，再动更深的存档结构。
- `P0-3/P0-4/P0-6` 才是高风险存档与服务端治理。

### 2.2 变更范围规则

每次只做一个任务，不要混做。

允许：

- `P0-1` 只动测试目录和必要的轻量公共工具。
- `P0-2` 只动配置和开关接入点。
- `P0-3` 只做存档 DTO 与加载后同步拆分，不顺手改迁移。

禁止：

- 在 `P0-3` 中顺便重写 `server_main.lua`
- 在 `P0-5` 中顺便改协议格式
- 在 `P1-1` 中顺便改战斗逻辑

### 2.3 线上安全规则

涉及 `存档 / 网络 / 服务端写入` 的任务，必须同时满足：

1. 保留旧路径。
2. 加开关。
3. 有失败判据。
4. 有回滚动作。
5. 有自动化回归。

### 2.4 存档安全规则

涉及存档结构时，必须同时满足：

1. 先做等价重构，再做行为调整。
2. 新字段只能追加，不能先删旧读取能力。
3. 不能在同一批次里同时改：
   - 序列化结构
   - 迁移入口
   - 服务端事务写入
4. 任何存档主路径改动都必须能切回旧路径。
5. 旧 key、旧槽位、旧 fallback 在至少一个发布周期内保留读取能力。

### 2.5 目录规则

除 `P2-4` 外，禁止修改以下历史快照文件：

- `scripts/server_main.lua.pre-s1`
- `scripts/network/CloudStorage.lua.pre-s1`
- `scripts/network/SaveSystemNet.lua.pre-s1`
- `scripts/systems/SaveSystem.lua.pre-s1`
- `scripts/systems/SaveSystem.lua.bak.s2`

这些文件只允许作为“旧实现参考”，不允许作为当前实现编辑入口。

## 4. 通用执行流程

每个任务都按同一流程执行。

### 4.1 第一步：读代码，不读旧结论

先读源码，再读已有文档。

最低要求：

1. 读本任务指定文件。
2. 搜索所有调用点。
3. 记录“谁调用谁、谁负责最终写入、谁负责回包/事件”。

### 4.2 第二步：确认这是哪一种任务

只允许归类为以下三种之一：

1. `等价重构`
   - 目标是拆文件、挪职责、加包装，不改行为。
2. `缺陷修复`
   - 目标是修明确 bug，必须写清楚旧行为错在哪里。
3. `结构准备`
   - 目标是为后续重构打地基，比如加开关、加 runner、加服务层接口。

如果一个任务同时包含两种以上类型，拆开做。

### 4.3 第三步：先建立基线，再动代码

必须先形成“修改前行为基线”。建议使用：

- 调用点列表
- 关键返回值/错误码列表
- 现有测试输出
- 手工构造的小型模拟用例

### 4.4 第四步：只做最小闭环

一个任务的正确形态是：

1. 改最少量的文件。
2. 新增最少量的新模块。
3. 同批补测试。
4. 可验证。
5. 可回滚。

不是：

1. 顺手清理所有历史问题。
2. 重命名大量文件。
3. 修改无关格式或风格。

### 4.5 第五步：提交前自检

每次交付前必须回答：

1. 这次是否改变了线上行为？
2. 如果改变了，是缺陷修复还是预期新行为？
3. 关闭开关后是否能回到旧行为？
4. 自动化测试是否覆盖最危险路径？
5. 有没有修改到不该动的旧快照文件？

## 5. 必须停手上报的情况

出现以下任一情况，停止编码，改为输出问题说明。

### 5.1 存档类停手条件

1. 发现同一个字段有两个“权威来源”。
2. 发现同一个槽位号在客户端和服务端含义不一致。
3. 发现旧逻辑存在“假成功”但调用方依赖该假成功。
4. 发现新旧 serializer 无法对齐关键字段。
5. 发现某些迁移分支会覆盖已有有效存档。

### 5.2 服务端类停手条件

1. 无法证明一次奖励发放是否幂等。
2. 无法证明一次请求失败后是否会重试。
3. 无法证明 `requestId` 是否贯穿读写链路。
4. 排行榜或待领奖写入依赖异步回调时序，且没有事务边界。

### 5.3 测试类停手条件

1. 环境中没有 `lua` 解释器，且当前任务需要纯 Lua 测试。
2. 现有测试脚本实际依赖引擎，但被误当成纯 Lua。
3. 新增模块无法在测试中被隔离加载。

### 5.4 结构类停手条件

1. 需要同时重写协议和服务端写入才能推进。
2. 需要一次性删掉旧逻辑才能推进。
3. 需要编辑超过 8 个核心运行文件才能完成一个单任务。

如果出现这些情况，说明任务切分不对，必须重新拆任务。

## 7. 任务依赖图

### 7.1 P0 依赖关系

- `P0-1` 是所有后续任务前置条件。
- `P0-2` 是所有高风险任务前置条件。
- `P0-5` 必须早于 `P0-3`，因为它会暴露存档调用语义。
- `P0-3` 必须早于 `P0-4`，因为先要厘清存档职责，再收敛迁移入口。
- `P0-4` 必须早于 `P0-6`，因为服务端服务化时需要知道最终迁移边界。

### 7.2 P1 依赖关系

- `P1-1` 完成后再做 `P1-2`
- `P1-2` 完成后再做 `P1-3`
- `P1-3` 完成后再做 `P1-4`

理由：

- 先拆主入口，再削 UI 事务，再改网络状态机，最后收敛高风险写事务。

### 7.3 P2 依赖关系

- `P2-1` 和 `P2-2` 可分开做，但都必须晚于 P1。
- `P2-3` 可独立执行。
- `P2-4` 必须最后做，避免在前期重构时丢掉参考快照。

## P0-1 建立统一测试入口与最小 CI 门禁

### 目标

把当前分散测试整理成“能一条命令批量运行”的入口。

### 执行前先读

1. `scripts/tests/` 全部现有测试文件
2. 重点先读：
   - `scripts/tests/test_save_optimization.lua`
   - `scripts/tests/test_battle_pipeline_regression.lua`
   - `scripts/tests/test_p0_regression.lua`
   - `scripts/tests/test_p1_regression.lua`
   - `scripts/tests/test_prison_tower.lua`

### 本任务允许新增/修改

- `scripts/tests/run_all.lua`
- `scripts/tests/TestRegistry.lua`
- `scripts/tests/TestRunner.lua`
- 必要时少量修改现有测试文件的导出形式

### 本任务禁止修改

- 游戏运行时逻辑
- 存档逻辑
- 网络协议

### 固定步骤

1. 盘点现有测试，分成三类：
   - `pure_lua`
   - `engine_required`
   - `unknown`
2. 第一版 runner 只纳管 `pure_lua`。
3. `engine_required` 不强行接入自动执行，但要在注册表中标记为 `skip`，并写明原因。
4. runner 输出必须机器可读，至少包含：
   - `total`
   - `passed`
   - `failed`
   - `skipped`
   - `failed_cases`
5. runner 失败时返回非 0。

### 先做什么，不要猜

先确认每个测试文件是否“直接执行即可”。

如果一个文件顶层就跑测试，可以先包装成：

- `return function() ... end`

或者：

- 导出 `Run()` 接口

但不要一口气重写全部测试。

### 新增测试建议

- `scripts/tests/test_runner_smoke.lua`

至少覆盖：

1. 注册表能加载
2. 通过 case 能计数
3. 失败 case 能返回非 0
4. skip case 会记录原因

### 完成判定

至少能支持如下形式：

```bash
lua scripts/tests/run_all.lua
lua scripts/tests/run_all.lua save
lua scripts/tests/run_all.lua regression
```

### 停手条件

- 发现大多数测试都依赖引擎，无法直接纯 Lua 跑。

此时不要硬做“假 runner”，而是先把测试分类结果交付出来，再拆“纯 Lua 子集 runner”。

## P0-5 修正 CloudStorage 网络模式语义不一致

### 目标

消灭“调用方以为成功，实际未写入”的假成功路径。

### 执行前先读

1. `scripts/network/CloudStorage.lua`
2. `scripts/systems/save/SavePersistence.lua`
3. `scripts/systems/SaveSystem.lua`
4. `scripts/network/SaveSystemNet.lua`
5. 搜索 `BatchSet(` 全部调用点

### 本任务允许新增/修改

- `scripts/network/CloudStorage.lua`
- `scripts/systems/save/SavePersistence.lua`
- 直接调用 `BatchSet` 的少量调用点

### 本任务禁止修改

- 存档 DTO
- 迁移入口
- 服务端 handler 大结构

### 决策规则

优先做“更小、更安全”的方案。

固定判断：

1. 如果 `BatchSet` 非存档调用点很少，且都能明确分类，优先做方案 A。
2. 如果调用点较多，先做方案 B，先消灭假成功，再考虑 API 拆分。

### 更推荐的安全顺序

第一阶段先做方案 B：

- 保留 `BatchSet()`
- 在网络模式下，对不支持的写入显式返回错误
- 记录一次明确日志

第二阶段再评估是否拆成：

- `SaveGameBatch()`
- `MetaBatch()`

### 固定步骤

1. 列出所有 `BatchSet` 调用点及其用途。
2. 逐个标记：
   - 存档写入
   - 元数据写入
   - 其他用途
3. 改 `CloudStorage` 返回值语义，不允许对 unsupported 直接 `ok()`
4. 补调用方处理，不允许吞错

### 新增测试建议

- `scripts/tests/cloudstorage_network_semantics.lua`
- `scripts/tests/checkpoint_write_behavior.lua`
- `scripts/tests/non_save_batch_rejected.lua`

### 关键验收

必须明确区分三种结果：

1. 真成功
2. 明确失败
3. 明确不支持

### 停手条件

- 发现大量调用方不检查返回值，只依赖回调副作用。

此时不要继续扩大改动，而是先交付“调用方依赖清单”，再拆更小任务。

## P0-3 收敛存档链路职责，建立单一权威 DTO

### 目标

先把存档流程拆成可理解的阶段，不做一次性重写。

### 执行前先读

1. `scripts/systems/save/SaveSerializer.lua`
2. `scripts/systems/save/SaveLoader.lua`
3. `scripts/systems/save/SavePersistence.lua`
4. `scripts/systems/save/SaveState.lua`
5. `scripts/systems/save/SaveMigrations.lua`
6. `scripts/systems/save/SaveRecovery.lua`
7. `scripts/systems/SaveSystem.lua`

### 本任务允许新增/修改

- `scripts/systems/save/SaveDTO.lua`
- `scripts/systems/save/SavePostLoadSync.lua`
- 上述 save 目录文件

### 本任务禁止修改

- 网络协议
- 迁移入口位置
- `server_main.lua` 主体结构

### 这是高风险任务，必须拆成 3 个小阶段

#### 阶段 1：定义 DTO，不改现有 key

新增统一 DTO 结构，但不改变实际落盘 key。

DTO 最少包含：

- `player`
- `inventory`
- `progress`
- `account`
- `runtime_passthrough`

#### 阶段 2：把“加载后同步”从主 loader 中拆出

新增：

- `SavePostLoadSync.lua`

它只负责：

1. 把 DTO 同步到运行时模块
2. 触发必要的后置恢复

它不负责：

1. 迁移决策
2. 云端存储
3. UI 直连采集

#### 阶段 3：加入双路径比对，不立即切流

在诊断开关下支持：

- 旧路径加载/序列化结果
- 新路径加载/序列化结果
- 关键字段比对

### 绝对禁止的事情

1. 不要在这一任务里删除旧 `ProcessLoadedData` 路径。
2. 不要在这一任务里改槽位 key。
3. 不要在这一任务里改迁移状态机。
4. 不要顺手改 UI 流程。

### 固定步骤

1. 先列出 `SaveSerializer` 依赖的 UI/业务模块。
2. 对每个依赖判断：
   - 采集期需要
   - 加载后同步需要
   - 只是诊断/修补残留
3. 只把“加载后同步”移动到新文件。
4. 把序列化期直接依赖 UI 的逻辑改成读 DTO 或读运行时纯数据。
5. 保留旧入口，通过 flag 控制新路径是否启用。

### 新增测试建议

- `scripts/tests/save_roundtrip_equivalence.lua`
- `scripts/tests/save_load_postsync_regression.lua`
- `scripts/tests/save_forward_backward_compat.lua`
- `scripts/tests/save_fixture_suite.lua`

建议新增目录：

- `scripts/tests/fixtures/save/`

样本至少包含：

1. 新角色
2. 中期角色
3. 高等级角色
4. 含待领奖状态
5. 含迁移痕迹
6. 含恢复痕迹

### 停手条件

1. 无法明确某字段应归属 DTO 还是运行时派生值。
2. 同一字段在加载期间被多个系统重复修正。
3. 新旧 round-trip 无法对齐关键字段。

出现这些情况时，先产出“字段归属清单”，不要继续写代码。

## 9. 自动化测试分层规则

为避免弱执行体把所有测试都写成一类，测试需要明确分层。

### 9.1 pure Lua

特点：

- 可直接 `lua xxx.lua`
- 不依赖引擎场景
- 适合作为 CI 门禁

### 9.2 engine required

特点：

- 依赖运行时对象、场景、GM 入口
- 不纳入第一版批量 runner
- 需要在注册表中标为 `skip`

### 9.3 fixture based

特点：

- 输入是固定样本数据
- 非常适合存档、迁移、奖励 patch 测试

### 9.4 benchmark

特点：

- 只比较耗时和数量级
- 不作为阻断上线的硬门禁

## 10. 最后提醒

这份手册的核心思想只有三句：

1. 先建护栏，再动高风险路径。
2. 先做等价重构，再做行为优化。
3. 一次只做一个任务，并且每个任务都能测试、回滚、交接。

如果执行者做不到这三点，就不要进入 `P0-3` 之后的任务。

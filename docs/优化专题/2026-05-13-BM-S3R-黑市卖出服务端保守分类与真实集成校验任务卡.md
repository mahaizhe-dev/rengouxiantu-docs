# 2026-05-13 BM-S3R 黑市卖出服务端保守分类与真实集成校验任务卡

## 1. 任务定位

这是 `BM-S3` 的代码收口任务，不是新的大任务。

当前 `BM-S3` 的主要问题不是“报告没写全”，而是：

1. `SellGuard` 把一批**存在本地先消耗路径**的商品错误归成了服务端安全类
2. 测试主要验证了 `SellGuard.CheckSellAllowed()` 的分类结果，没有真正验证 `HandleSell()` 被拒绝后的服务端副作用

因此，`BM-S3R` 的目标非常明确：

- 把服务端“安全类”定义收紧到真正可由服务端独立信任的范围
- 补一层真实的 `HandleSell()` 拒绝路径集成校验

---

## 2. 为什么现在做

`BM-S3` 当前最危险的误判是：

- 令牌盒（`wubao_token_box / sha_hai_ling_box / taixu_token_box`）
- 神器碎片（`rake_fragment_* / bagua_fragment_* / tiandi_fragment_*`）

被当成了“服务端安全类”。

但这些物品的真实代码路径是：

- 本地先 `ConsumeConsumable()`
- 再触发 `save_request`

这说明它们**并不具备服务端独立证明安全的条件**。  
如果改包客户端绕过 `BM-S2R` 的 UI 门禁，旧快照卖出风险仍然存在。

所以 `BM-S3` 现在不能放行，必须先做 `BM-S3R`。

---

## 3. 本任务目标

### 3.1 收紧服务端安全类定义

本阶段安全类必须遵守一个更硬的规则：

**只要该商品存在明确的本地先消耗 / 先转化 / 先变化路径，就不能被归入服务端安全类。**

这意味着至少要重新审视：

1. 令牌盒
2. 神器碎片
3. 龙鳞

如果执行 AI 证明这些商品同样存在“本地先变化 + 延迟落盘”窗口，则本阶段必须改成：

- 服务端拒卖

### 3.2 补真实服务端拒绝副作用校验

不能再只测：

- `SellGuard.CheckSellAllowed() == false`

必须至少验证 `HandleSell()` 拒绝时：

1. 不会 `MoneyAdd(userId, "xianshi", ...)`
2. 不会写成功交易流水 `TradeLogger.LogPublic(..., "sell", ...)`
3. 返回的是明确错误，而不是假成功

### 3.3 对自动回收的“无影响”结论必须来自真实隔离校验

不能再只写架构推断。

本阶段至少要补：

1. `SellGuard` 不被自动回收调用链引用
2. 自动回收相关测试/契约不会因为本轮收口红掉

## 8. 固定执行步骤

### 步骤 1：重做“服务端安全类证明表”

必须先写一张表，逐类说明：

1. 哪些商品当前仍被当成安全类
2. 它们是否存在本地先变化路径
3. 这些路径是否依赖客户端门禁才能安全
4. 最终结论是：
   - 真安全，保留允许
   - 伪安全，改为拒卖

没有这张表，不允许直接改 `SellGuard`。

### 步骤 2：收紧 `SellGuard`

原则：

- **服务端安全类的标准必须高于客户端安全类**
- “客户端已有热修”不能作为“服务端安全”的证据

如果某类商品无法证明服务端独立安全，本阶段统一按：

- `high_risk_blocked`

处理。

### 步骤 3：补真实 `HandleSell()` 集成测试

至少补以下场景：

1. 高风险商品请求卖出 → 返回失败
2. 失败时没有 `MoneyAdd(xianshi)`
3. 失败时没有 `TradeLogger.LogPublic(..., "sell", ...)`
4. 安全类商品请求卖出仍能继续走旧链路

如果执行 AI 认为纯 Lua 很难完整打到 `HandleSell()`，也必须至少构造：

- 伪 `serverCloud`
- 伪 `connection`
- 伪 `eventData`

并在测试里显式断言这些副作用是否发生。

### 步骤 4：补自动回收不受影响的最小验证

允许是契约级测试，但不能再只写 `assertTrue(true)`。

最少要做到：

1. 证明自动回收测试集仍通过
或
2. 证明 `SellGuard` 相关接口没有被自动回收路径触达，并且用代码级 stub/调用断言表达出来

## 9. 成功标准

满足以下全部条件，才算通过：

1. 令牌盒/神器碎片/龙鳞等“伪安全类”已被重新审视
2. 任何存在本地先变化路径的商品，不再被轻易算作服务端安全类
3. `HandleSell()` 拒绝路径有真实副作用测试，不再只是分类测试
4. 被拒绝时不发仙石、不记成功流水
5. 自动回收不受影响有真实测试或契约支撑
6. blocking gate 继续全绿

---

## 10. 失败判据

出现以下任一情况视为失败：

1. 继续把依赖客户端门禁才能安全的商品归为服务端安全类
2. 测试仍然主要停留在 `SellGuard.CheckSellAllowed()` 分类层
3. 仍然没有验证拒绝时 `MoneyAdd / LogPublic` 不发生
4. 用文档描述替代真实自动回收隔离验证

## 12. 备注

如果执行 AI 最终得出结论：

- “目前能被证明真正服务端安全的商品范围非常小”

这是可以接受的结果。  
本阶段优先级是：

**保守正确 > 功能覆盖面。**

宁可让更多商品暂时拒卖，也不要继续保留“看起来能卖、实际靠客户端门禁兜底”的伪安全模型。

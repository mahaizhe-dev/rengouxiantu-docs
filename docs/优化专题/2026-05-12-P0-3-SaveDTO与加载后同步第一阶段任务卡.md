## 1. 任务定义

任务ID：`P0-3`

任务名称：`SaveDTO 与加载后同步第一阶段`

变更类型：`等价重构`

目标：

- 依据 `P0-3A` 已冻结的边界，落地 `SaveDTO v1` 的第一版实现
- 从 `SaveLoader.ProcessLoadedData()` 中抽出**低歧义**的加载后同步逻辑，形成 `SavePostLoadSync.lua`
- 保持线上存档 key、迁移链、服务端权威写入、备份链路全部不变
- 使用现有开关 `SAVE_PIPELINE_V2` 为新路径提供灰度入口

---

## 2. 为什么这一阶段要收紧

`P0-3A` 已经确认：

1. `SaveDTO v1` 只覆盖主存档正文 22 个字段域
2. `account_* / slots_index / login/prev/deleted / rank2_* / pending_*` 不属于 DTO
3. `atlas / accountCosmetics / pendingXianyuanRewards` 都存在旁路或透传约束
4. `rules.md` 的线上红线要求：不能删旧字段、不能改 key、不能断迁移链

所以当前 `P0-3` 不能做“大一统重构”，只能先做**第一阶段等价拆分**。

---

## 3. 本任务的成功标准

成功标准只有 9 条：

1. 新增 `SaveDTO.lua`，其边界必须与 `P0-3A` 合同完全一致。
2. 新增 `SavePostLoadSync.lua`，且只收纳低歧义的加载后同步逻辑。
3. 不得删除或重命名任何已有存档字段。
4. 不得修改任何已有云存储 key 名。
5. 不得删除或改写任何已有 `SaveMigrations.MIGRATIONS` 函数。
6. `SAVE_PIPELINE_V2=false` 时，默认行为保持旧路径。
7. 若本任务未改变落盘结构，则**不得**无故递增 `CURRENT_SAVE_VERSION`。
8. 至少有 1 个纯 Lua 自动化测试验证 `SaveDTO` 合同边界。
9. 现有 blocking 门禁继续保持绿色。

只要 9 条里有任一条不满足，本任务不算完成。

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：校对文档常量基线

先确认 `rules.md` 中的以下常量是否与当前代码一致：

- `CURRENT_SAVE_VERSION`
- `AUTO_SAVE_INTERVAL`
- `CHECKPOINT_INTERVAL`
- `MAX_SLOTS`

如果不一致：

1. 记录真实代码值
2. 仅同步修正文档常量
3. 不做任何其他 rules 重写

### 步骤 2：创建 SaveDTO v1 模块

新增 `SaveDTO.lua`，目标是把 `P0-3A` 合同转成代码层的边界约束。

最低要求：

1. 明确列出 22 个允许进入 DTO 的字段域
2. 对非法字段给出稳定处理策略：
   - 忽略
   - 透传
   - 报错（仅测试/诊断模式）
3. 提供纯 Lua 可测的接口

推荐最小接口：

- `SaveDTO.FromCoreData(coreData)`
- `SaveDTO.ToCoreData(dto, passthrough)`
- `SaveDTO.GetSchemaFields()`

如果你使用不同命名，也必须在完成报告里写清楚职责映射。

### 步骤 3：只抽低歧义 post-load sync

新增 `SavePostLoadSync.lua`，但**只允许移动低歧义块**。

第一阶段允许优先考虑移动的逻辑：

1. `AtlasSystem.PostLoadSync()`
2. 依赖所有系统已加载完成的 HP 重新校准
3. 明确属于“反序列化之后、校验之前/之后”的纯同步逻辑

第一阶段禁止移动的逻辑：

1. `SaveMigrations.MigrateSaveData`
2. `ValidateLoadedState`
3. `login backup` 写入
4. `SaveRecovery` 候选恢复逻辑
5. `bossKillTimes` 恢复
6. `bossKills` 历史迁移分支
7. `pill validation / correction`
8. 任何服务端旁路字段处理

如果你无法证明某段逻辑属于“低歧义 post-load sync”，就不要移动。

### 步骤 4：接入 SAVE_PIPELINE_V2

必须使用已有开关 `SAVE_PIPELINE_V2`。

固定要求：

1. `flag=false`
   - 旧路径保持默认行为
2. `flag=true`
   - 允许走新 `SaveDTO` / `SavePostLoadSync` 路径
3. 如果当前阶段只做到“影子构建 / 等价包装”，也要在报告里写清楚

不要在默认路径下直接切流。

### 步骤 5：补最小测试

必须新增：

- `scripts/tests/test_save_dto_contract.lua`

最低覆盖：

1. 22 个允许字段域被保留
2. `account_* / slots_index / login/prev/deleted / rank2_* / pending_*` 不进入 DTO
3. `atlas / accountCosmetics / pendingXianyuanRewards` 的处理符合合同
4. `SaveDTO.ToCoreData()` 不改 key 名

如果 `SavePostLoadSync.lua` 能纯 Lua 隔离，再新增：

- `scripts/tests/test_save_postload_sync_smoke.lua`

若做不到纯 Lua 隔离，必须在报告里说明阻塞原因，不允许伪造覆盖。

### 步骤 6：维持版本与迁移纪律

如果本任务**没有改变落盘结构**：

- 不要改 `CURRENT_SAVE_VERSION`
- 不要新增迁移函数

如果本任务**确实改变了落盘结构**：

1. `CURRENT_SAVE_VERSION + 1`
2. 新增对应 `MIGRATIONS[N]`
3. 在报告里明确说明结构变更点

不要为了“看起来规范”而无意义 bump 版本。

这一步不是“完成全部 P0-3”。

这一步只做：

1. 代码层冻结 SaveDTO v1
2. 第一批低歧义 post-load sync 抽离
3. 保留旧路径和旧结构
4. 给后续更深的存档拆分打地基

如果执行 AI 发现要完成目标必须：

- 动服务端
- 动迁移入口
- 动 key 命名
- 动大量验证逻辑

那就说明任务切分还得再细，必须停手上报。

## 9. 本任务的停手条件

出现以下情况，停止编码，改交付问题说明：

1. 要落地 SaveDTO 就必须修改现有 key 名
2. 要抽 post-load sync 就必须同时移动 pill validation / recovery / ranking 逻辑
3. 不改 `server_main.lua` 就无法让新路径成立
4. 需要编辑超过 8 个核心运行文件
5. 无法为 `SaveDTO` 做出纯 Lua 合同测试
6. 必须 bump 版本，但无法证明迁移函数正确

如果停手，仍然要交付：

- 哪些块已确认可抽
- 哪些块无法安全抽离
- 为什么当前 `P0-3` 仍然过大
- 建议拆出的 `P0-3B` 小任务

## 1. 任务定义

任务ID：`P0-3A`

任务名称：`存档边界盘点与 SaveDTO 合同冻结`

变更类型：`结构准备`

目标：

- 在进入真正的 `P0-3` 存档重构前，先冻结“哪些数据属于主存档，哪些不属于”
- 把 `save_{slot}`、`slots_index`、`login/prev/deleted`、`account_*`、`排行/服务端权威数据` 的边界讲清楚
- 产出一份可直接指导后续 `P0-3` 的 SaveDTO 合同文档，避免执行 AI 在实现阶段靠猜测切边界
- 把 `rules.md` 中和线上存档兼容有关的硬约束，直接写进后续 `P0-3` 的实施前置条件

---

## 2. 为什么要先插入这个任务

`P0-5` 已经证明一件事：

1. `main save` 和 `meta/backup/account` 写入在网络模式下不是同一类能力
2. 之前的大任务 `P0-3` 把这些东西放在一起描述，边界过大
3. 如果现在直接开始做 `SaveDTO / SavePostLoadSync`，执行者很容易把：
   - 主存档字段
   - 账号级字段
   - 备份字段
   - 服务端权威字段
   混进同一个 DTO

这一步的目的不是改代码，而是先把边界冻结。

同时还要补一层：

`rules.md` 已明确这是线上项目，并且规定了：

1. 不能删除或重命名已有存档字段
2. 不能修改已有云存储 key 名
3. 改存档结构必须递增版本并写迁移
4. 不能删除已有迁移函数
5. 配对字段必须整体同步

所以 `P0-3A` 不能只讲“模块边界”，还必须把这些线上兼容约束变成后续实现的硬前提。

---

## 3. 本任务的成功标准

成功标准只有 6 条：

1. 必须盘点所有与存档读写直接相关的 key 家族和调用模块。
2. 必须把 key 家族划分到明确边界类别，不能留“看起来差不多”的模糊描述。
3. 必须定义 `SaveDTO v1` 的“包含字段”和“明确不包含字段”。
4. 必须标出每类数据的权威来源：客户端 / 服务端 / 账号级 / 备份级。
5. 必须给出 `P0-3` 实施阶段的禁止混做项。
6. 必须单独列出 `rules.md` 约束下“绝不能动”的 key / 字段 / 迁移链。
7. 本任务不得引入任何运行时行为变更。

只要 7 条里有任一条不满足，本任务不算完成。

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：盘点 key 家族

必须先输出一张 key 家族清单，最少包含这些列：

| key family | 典型 key | 写入模块 | 读取模块 | 网络模式现状 | 备注 |

至少覆盖这些家族：

- `save_{slot}`
- `save_data_{slot}` / `save_bag1_{slot}` / `save_bag2_{slot}`
- `slots_index`
- `login_{slot}`
- `prev_{slot}`
- `deleted_{slot}`
- `premigrate_*`
- `save_migrate_backup_*`
- `account_bulletin`
- `account_atlas`
- `account_cosmetics`
- `rank_*`
- 其他你实际搜索到的存档相关 key

### 步骤 2：按边界类别分组

必须把所有 key 家族归到下面 5 类之一：

1. `main_character_save`
   - 角色主存档正文
2. `slot_index_meta`
   - 槽位摘要/索引
3. `backup_recovery_meta`
   - login / prev / deleted / premigrate / migrate backup
4. `account_level_meta`
   - account_atlas / account_bulletin / account_cosmetics
5. `server_authority_side_data`
   - 排行榜、待领奖、服务端衍生权威信息

不允许出现“暂时归不清”“先放 DTO 再说”。

### 步骤 3：冻结 SaveDTO v1 合同

必须产出一张 `SaveDTO v1` 合同表，最少包含这些列：

| 字段域 | 是否进入 SaveDTO v1 | 来源模块 | 当前存储位置 | 备注 |

至少要回答这些问题：

1. `player / equipment / backpack / skills / progress` 是否进入 DTO
2. `account_bulletin / account_atlas / account_cosmetics` 是否进入 DTO
3. `login/prev/deleted` 是否属于 DTO
4. `slots_index` 是否属于 DTO
5. `rank_*` 是否属于 DTO
6. `pendingXianyuanRewards` 这类透传字段属于 DTO 还是服务端旁路
7. 哪些配对字段必须整体进/整体出 DTO，不能拆开判断

要求：

- 如果进入 DTO，要写明为什么
- 如果不进入 DTO，要写明后续归属到哪里

### 步骤 4：列出 P0-3 禁止混做项

必须明确写出后续 `P0-3` 不能同时做的内容。

最低要求写清楚：

1. 不在 `P0-3` 中顺手支持 `meta_batch` 网络写入
2. 不在 `P0-3` 中顺手改迁移入口
3. 不在 `P0-3` 中顺手改服务端 handler
4. 不在 `P0-3` 中顺手删旧 key fallback
5. 不在 `P0-3` 中顺手改已有 clientCloud/clientScore key 名
6. 不在 `P0-3` 中顺手删除已有迁移函数

### 步骤 5：补 rules 红线清单

必须新增一节，明确列出：

1. 后续 `P0-3` 绝对不能删除/重命名的已有字段
2. 后续 `P0-3` 绝对不能改名的已有 key 家族
3. 后续 `P0-3` 若要新增字段，哪些地方需要默认值兼容
4. 后续 `P0-3` 若要改结构，哪些地方必须同步：
   - `CURRENT_SAVE_VERSION`
   - `SaveMigrations.MIGRATIONS`
   - 旧字段 fallback
5. 哪些字段属于“配对字段”，必须整体同步

### 步骤 6：输出阻塞点和歧义点

必须有一节单独列：

- 当前已确认边界
- 当前未确认边界
- 需要我下一轮重新定方案的点

如果存在无法证明的字段归属，不要硬判，直接列出来。

## 9. 本任务的停手条件

出现以下情况，停止编码，改交付问题说明：

1. 同一个 key 家族被两个模块以不同语义写入
2. 同一份数据既像主存档，又像账号级元数据，无法判定主归属
3. 发现服务端和客户端对同一字段的权威来源判断不一致
4. 要确认边界就必须先改代码才能继续
5. 搜索结果显示真实调用面明显超出当前任务卡假设
6. `rules.md` 的线上兼容要求与当前代码现状发生直接冲突，无法只靠文档冻结解决

如果停手，仍然要交付：

- 已盘点出的清单
- 无法冻结的字段/模块
- 建议拆出的更小过渡任务

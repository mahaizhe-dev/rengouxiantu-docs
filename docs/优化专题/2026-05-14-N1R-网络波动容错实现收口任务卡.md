## 3. 当前已实现内容（不得重复施工）

### 3.1 已实现：断线判定宽限

当前 `SaveSystem.Update()` 已不是“一次失败就 lost”。

真实行为是：

1. 每 3 秒采样一次连接状态
2. 第 1 次失败：进入 `unstable`
3. 连续第 3 次失败：进入 `disconnected`
4. 连续 2 次成功：恢复 `connected`

因此：

- **不得再重做一套新的断线状态机**
- **不得把任务重新理解成“做 N1 初版”**

### 3.2 已实现：分层提示

当前已有：

1. `connection_unstable` → 底部轻横幅
2. `connection_lost` → 全屏阻断弹窗
3. `connection_restored` → 解除提示

因此：

- **不得再新建第二套断线 UI**
- 只能修正现有提示触发条件、文案或边界行为

### 3.3 已实现：部分高风险入口阻断

当前至少已接入：

1. 黑市
2. 炼丹

- 本任务重点不是“再发明阻断模型”

## 4. 本任务真正要解决的问题

### 4.1 问题 A：状态机文档口径和实现不一致

当前 [NetworkStatus.lua](/C:/Users/mahaizhe/Documents/New project/taptapmaker/rengouxiantu-docs/scripts/network/NetworkStatus.lua) 中仍保留：

- `SUSTAINED_DISCONNECT_DELAY`
- `_sustainedTimer`

但 `RecordSample()` 实际并没有使用它们。

这意味着现在存在两套冲突说法：

1. 设计口径：达到失败阈值后，再等一段时间才算持续断连
2. 实现口径：连续失败计数到阈值即断连

本任务必须二选一：

1. **要么**删掉这套未使用设计残留，让代码和文档回到“纯次数阈值”
2. **要么**真的把“持续断连延迟”实现进去

不允许继续保留半拉子状态。

### 4.2 问题 B：只读请求重试的“2 秒”与运行时不一致

当前 `CloudStorage.GetRankList()` 在 `unstable` 状态下会：

- `ScheduleDelayedLocal(2.0, ...)`

但延迟任务是由 `CloudStorage.Tick()` 驱动，而当前主循环对它的调用频率是 **5 秒级**。

因此：

- 文档中的“2 秒重试”
- 和实际运行时的“约 5 秒粒度重试”

并不一致。

本任务必须二选一：

1. **要么**真正把延迟任务驱动做到近似 2 秒粒度
2. **要么**承认现在就是 5 秒级近似重试，并同步修正文档、注释和测试

不能继续口径错位。

### 4.3 问题 C：高风险入口覆盖面尚未冻结

当前虽然已经拦了一批入口，但还没有冻结成一张**真实入口清单**。

本任务必须产出：

1. 已接入 `NetworkStatus.IsDisconnected()` 阻断的 UI 列表
2. 仍未接入，但属于高风险写操作的 UI 列表
3. 当前只读请求容错的真实覆盖列表

没有这张表，后面第五章继续加系统时还会漏。

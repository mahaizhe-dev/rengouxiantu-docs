# 2026-05-13 第五章前优化重审与重排方案

> 适用基线：`origin/main` 截至 `8570053 / 5c2180a / badd9ef / a0aa194 / c5b13c3 / 248b44e`
> 本文为**从当前代码与 Git 事实重新审计**后的新方案。
> 本文**不沿用**旧版“`BM-S3/BM-S3R -> 分层拒卖恢复`”路线作为判断前提。

## 1. 重新审计后的总判断

当前项目在第五章前，不应该再按“大规模结构重构”继续推进。

更合理的目标是：

1. 保住线上安全，尤其是黑市/仙石经济域与存档一致性。
2. 保住功能可用性，不能为了止血直接牺牲黑市卖出能力。
3. 让后续新增第五章商品、材料、消耗品时，风险不会继续按“下一个物品再补一个名单”的方式扩散。
4. 在不把服务器改造成全权威库存的前提下，增加正常玩家和改包脚本的利用成本。

这意味着第五章前的优化主线，应该从原来的：

- `BM-S3 / BM-S3R` 服务端保守拒卖
- `R4 / R5 / R6` 更大结构治理

切换为：

- `BM-L1` 黑市交易保护锁与统一操作闸
- `BM-L2` 黑市商品准入元数据与轻量风控
- `T1` 移动端发热缓解最小补丁
- `C5-PREP` 第五章开发护栏冻结

## 2. 基线重估

### 2.1 已经有价值、应保留的成果

以下工作不应推翻：

- `P0-1 ~ R2`
  - 测试门禁、FeatureFlag、CloudStorage 语义、SaveDTO、迁移入口、日志、SaveSystemNet 状态机等，仍然是当前线上基线的一部分。
- `R3`
  - 服务端主存档主链拆壳第一阶段，虽不是黑市修复，但有助于后续在 `SaveGame / LoadGame` 上局部治理。
- `N1`
  - 网络波动容错与强提示升级，属于长期体验问题的正式护栏。
- `BM-S1 / BM-S1R`
  - 它们把黑市问题从“几个热修 bug”提升成“79 商品 + 本地变化路径 + 仙石结算 + 自动回收”的专项风险矩阵。
- `BM-S2 / BM-S2R`
  - 它们依然有价值，因为它们解决的是**正常客户端下的未同步窗口**，这和未来的交易保护锁并不冲突。

### 2.2 不再作为最终路线的成果

以下路线只保留“探索/止血”价值，不再作为最终方向：

- `BM-S3`
  - 通过“服务端安全类/高风险类分类”尝试恢复卖出，但把若干本地先消耗商品误判为安全。
- `BM-S3R`
  - 通过“79 项全部拒卖”证明了旧服务端卖出模型不能直接继续信，但这个状态在产品上不可接受。

结论：

> `BM-S3/BM-S3R` 不是白做，但它们的最终意义是“证明旧路线不适合当前项目目标”，而不是继续沿着“服务端证明不了就不让卖”推进。

## 3. 黑市问题的重新定义

黑市的核心问题不是“有几个可利用物品”，而是：

> 黑市卖出会发放硬通货仙石，但它依赖的是**服务端存档里的旧背包快照**；与此同时，很多物品状态先在客户端本地变化，再延迟写回存档。

只要同时满足：

1. 商品可在黑市卖出；
2. 该商品能在本地先被使用、消耗、入仓、合成、激活、打造、卖钱或转成其他资源；
3. 存档不是立即同步；

就存在“旧快照卖出”的结构性风险。

这不是单个物品问题，而是经济结算模型与本地库存模型的冲突。

---

## 4. 新目标下的黑市正确方向

用户当前明确目标是：

- 不追求服务器代管一切；
- 不追求超高等级防护；
- 黑市卖出必须保留；
- 后续黑市还要继续加商品；
- 只需要足够基本的防护和利用成本提升。

在这个前提下，黑市的正确路线不是“全量服务端权威库存”，而是：

### 4.1 客户端层

- 保留 `BM-S2R` 的统一未同步门；
- 再增加一层**黑市交易保护锁**：
  - 从黑市买到的商品，在保护期内不能发生任何会改变该商品状态的本地操作；
  - 包括：使用、消耗、存仓、取仓、合成、激活、打造、黑市回卖。

### 4.2 服务端层

- 不做全背包权威托管；
- 只做**很轻的卖出锁校验**：
  - 卖出时检查该商品/该堆叠是否仍处于黑市交易保护期；
  - 若在保护期，则拒绝卖出。

### 4.3 运营与扩展层

- 新增黑市商品时，不再把它视为纯配置接入；
- 必须带黑市风险元数据；
- 高价值商品再叠加轻量限频/冷却/审计。

## 5. 关于“5 分钟保护锁”的代码现实与可行性

### 5.1 这个方向是可行的

它比 `BM-S3R` 更适合当前项目，因为它：

- 不需要服务器重建真实库存；
- 能保留卖出功能；
- 能覆盖“刚买就用 / 刚买就存仓 / 刚买就激活 / 刚买就卖回黑市”这些正常客户端漏洞路径；
- 对未来新增黑市商品具备统一接入能力。

### 5.2 但必须解决“堆叠混合”问题

当前客户端和服务端都是**按 `consumableId` 全局堆叠**：

- 客户端：`InventorySystem.AddConsumable()`
- 服务端存档：`BackpackUtils.AddToBackpack()`

两边都会把同 `consumableId` 的道具堆到现有堆叠里。

这意味着，如果玩家原本就有未加锁的 `gold_brick`，又从黑市买了一个应加锁的 `gold_brick`：

- 如果继续沿用现有堆叠逻辑；
- 被买入的那 1 个会直接混进旧堆叠；
- 之后就无法区分：
  - 哪些数量处于保护期；
  - 哪些数量是旧库存；

所以“5 分钟保护锁”要成立，前提不是只加一个倒计时，而是：

> **拒绝与未加锁同类堆叠。**

### 5.3 现有代码支持低成本落地

现有存档结构不是彻底阻碍：

- 客户端背包物品是 table；
- 服务端 `saveData.backpack` 物品也是 table；
- `SaveSerializer.SerializeItemFull()` 当前已显式列出字段，但扩展字段成本可控；
- `BackpackUtils.AddToBackpack()` / `InventorySystem.AddConsumable()` 都是集中堆叠入口。

因此，最现实的低成本方案是：

1. 为黑市购买物品加**堆叠级锁元数据**，例如：
   - `bmLockUntil`
   - `bmLockSource`
2. 修改客户端与服务端堆叠规则：
   - 未锁定堆叠只能和未锁定堆叠合并；
   - 已锁定堆叠不能和未锁定堆叠合并；
   - 不同锁批次的已锁定堆叠默认也不合并。
3. 本地消费/仓库存取/打造/激活等入口统一检查锁；
4. 黑市卖出服务端只检查锁字段，不做全背包权威校验。

这仍然是“轻量护栏”，不是“服务器代管一切”。

## 6. 从五个维度重新审计剩余优化

### 6.1 可维护性

当前最该避免的是继续堆“特例名单热修”。

黑市后续如果仍沿用：

- `_BM_SENSITIVE_CONSUMABLES`
- “再补一个物品”
- “再补一个分类”

会继续恶化维护成本。

**新的维护性原则应改为：**

- 统一交易保护锁；
- 统一操作闸；
- 商品准入元数据；
- 风险按机制分层，而不是按道具名补丁。

### 6.2 线上安全性

第五章前，真正的线上高优先级不再是“大规模结构拆分”，而是：

1. 黑市卖出漏洞不能再以正常客户端可复现的方式存在；
2. 服务端要有最小护栏，至少不能让新买商品在保护期内直接卖回；
3. 高价值结算入口要可限频、可冷却、可审计。

### 6.3 存档

交易保护锁如果落地，必须当成**存档数据模型的一部分**处理，而不是临时内存态：

- 黑市买入写入服务端 `saveData.backpack` 时就要带锁；
- 客户端本地加物品时也要带锁；
- 存档成功/超时解锁要同步清锁；
- 重登后仍能识别锁状态。

否则会出现：

- 本地有锁、服务端没锁；
- 重登后锁丢失；
- 存档成功后本地/服务端解锁不一致。

### 6.4 C/S 架构

第五章前不应该再追求“黑市全权威库存”。

更合理的 C/S 划分是：

- 客户端负责：
  - 统一操作闸；
  - 保护期内的本地交互阻断；
  - 正常体验层提示；
- 服务端负责：
  - 卖出轻量锁校验；
  - 高价值结算发放；
  - 轻量冷却/限频/审计；
  - 自动回收仍独立运行。

### 6.5 可拓展性

第五章前，黑市必须具备接纳新商品的能力。

这要求未来每个黑市商品至少补这些元数据：

- `bmRiskTier`
- `bmLocalChangeKind`
- `bmProtectionMode`
- `bmSellCooldownSec`
- `bmAuditLevel`

否则第五章每加一种新消耗品，都要重新人工过一遍漏洞路径。

### 6.6 性能

第五章前不建议再做大性能重构。

当前性能主问题仍以移动端发热为主，优先级高于深层渲染优化的，是：

- `T1` 低功耗/低特效/掉落光效降载

黑市与交易保护锁不会明显增加性能压力，风险主要在逻辑复杂度和背包遍历次数，但相对于当前背包规模与操作频率，成本可接受。

## 7. 新的第五章前优化主线

### A. 必做：BM-L1 黑市交易保护锁与统一操作闸

目标：

- 恢复黑市卖出能力；
- 不依赖名单热修；
- 不需要服务器重建真实库存；
- 解决“刚买即用 / 刚买即存 / 刚买即变更 / 刚买即卖回”的主风险。

关键要求：

- 保护锁持续 `5 分钟`，或直到下一次成功存档；
- 锁定品不能与未锁定同类堆叠；
- 保护锁进入客户端背包与服务端存档；
- 本地所有状态变化入口统一阻断；
- 服务端卖出仅做锁校验，不做重库存代管。

### B. 必做：BM-L2 黑市商品准入元数据与轻量风控

目标：

- 为后续新增黑市商品建立准入门槛；
- 给高价值商品增加轻量冷却、限频、审计；
- 不做重服务端托管。

### C. 建议做：T1 移动端发热缓解最小补丁

目标：

- 加低功耗模式；
- 降特效、降高品质掉落光效、减少持续热源。

### D. 必做：C5-PREP 第五章开发护栏冻结

目标：

- 第五章新增商品/材料/消耗品/黑市接入前先过准入清单；
- 新增存档字段遵守 `rules.md`；
- 不把第五章开发重新带回“加内容同时重新引入黑市/存档漏洞”的老路。

---

## 8. 不建议在第五章前继续推进的路线

以下路线不应在第五章前继续作为主线：

- `BM-S3/BM-S3R` 路线的继续修补
- `R4` 排行/奖励事务大收口
- `R5` 客户端结构治理
- `R6` 更大性能重构
- 黑市服务端全权威库存化

理由：

- 性价比低；
- 产品约束不匹配；
- 容易把黑市卖出功能做没；
- 也不利于第五章内容节奏。

---

## 9. 新主线的第一步

下一步不是继续 `BM-S3R`，而是直接进入：

> `BM-L1 / BM-S4A 黑市交易保护锁与统一操作闸`

这是第五章前最值得先做的代码任务。

它完成后：

- 黑市卖出能力可以在较低成本防护下恢复；
- 黑市未来扩展会有统一接入规则；
- `BM-S2R` 与新锁模型可以叠加，而不是互相替代。

---

## 10. 一句话结论

> 第五章前的优化主线，应该从“服务端保守拒卖恢复路线”切换成“黑市交易保护锁 + 统一操作闸 + 轻量风控 + 商品准入元数据”，并把移动端发热缓解与第五章开发护栏作为另外两条并行主线。

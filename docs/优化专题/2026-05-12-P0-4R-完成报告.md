## 2. MigrationPolicy 收口

### Evaluate 现在按什么顺序判定

```
flag=false → return nil（不干预原有逻辑）
flag=true →
  1. not needMigration OR slotCount > 0 → NOT_NEEDED
  2. needMigration=true AND slotCount=0  → ELIGIBLE_CLIENT_CLOUD
```

**关键变化**：`MIGRATION_GLOBALLY_ENABLED` 不再出现在 `Evaluate()` 中。无论全局开关如何，`Evaluate()` 只返回现实状态。

### 状态层与动作层如何分离

| 层 | 职责 | 函数 | 是否读 MIGRATION_GLOBALLY_ENABLED |
|----|------|------|-----------------------------------|
| **状态层** | 判定现实（有没有旧数据） | `Evaluate()` | **否** |
| **动作层** | 判定策略（要不要做迁移） | `ShouldAutoMigrate()`, `ShouldBlockEntry()` | 不直接读，但两者当前均硬返回 false |
| **动作层** | PC 拦截是否需要 | `IsPCMigrationCheckNeeded()` | **是** |

### 哪些状态现在能被区分

| 场景 | P0-4（旧） | P0-4R（新） |
|------|-----------|------------|
| needMigration=true, slots=0, global=false | `MIGRATION_DISABLED` | `ELIGIBLE_CLIENT_CLOUD` |
| needMigration=false, slots=0, global=false | `MIGRATION_DISABLED` | `NOT_NEEDED` |
| needMigration=true, slots=2, global=false | `MIGRATION_DISABLED` | `NOT_NEEDED` |
| needMigration=false, slots=3, global=false | `MIGRATION_DISABLED` | `NOT_NEEDED` |

P0-4 中所有输入都被 `MIGRATION_GLOBALLY_ENABLED=false` 坍缩为 `MIGRATION_DISABLED`，丢失了"无需迁移"和"有旧数据但策略禁用"的区分。P0-4R 修正后，调用方可以知道"这个用户有旧数据"（ELIGIBLE），同时动作层仍然不触发迁移。

## 1. 任务定义

任务ID：`P0-1A`

任务名称：`测试门禁收口与运行环境固化`

变更类型：`缺陷修复`

目标：

- 把 `P0-1` 产出的统一 runner 收口成真正可用的门禁
- 解决默认门禁可能长期红灯的问题
- 固化一条真实可执行的测试命令和环境前置条件
- 防止“全 skip 也显示成功”的误导性结果

---

## 2. 为什么要先做这个

基于 `P0-1` 完成报告，当前还存在 3 个门禁层面的收口问题：

1. 默认全量执行结果包含 `prison_tower` 已知失败，门禁基线不是绿色。
2. 运行入口依赖 `python3 + lupa`，需要把运行环境前置条件写死并脚本化提示。
3. 某些 group 可能全部是 `skip`，但不能把这种结果当成“有覆盖的成功”。

本任务不重做 `P0-1`，只收口门禁质量。

---

## 3. 本任务的成功标准

成功标准只有 5 条：

1. 默认阻断门禁命令必须明确，并且在目标环境中真实可执行。
2. 默认阻断门禁必须是绿色基线。
3. 如果某组没有实际执行任何测试，输出必须明确标记，不能伪装成正常成功。
4. 环境缺失时必须给出清晰错误信息，而不是只返回非 0。
5. 必须明确区分：
   - 阻断门禁测试
   - 非阻断测试
   - 已知问题测试

如果 5 条里任一条不满足，本任务不算完成。

## 7. 固定执行步骤

严格按这个顺序做。

### 步骤 1：先定义测试集合语义

在注册表层明确区分至少 3 类：

- `blocking`
- `non_blocking`
- `known_red` 或等价含义

要求：

- 默认门禁只跑 `blocking`
- 其他测试必须仍可显式执行

### 步骤 2：处理 `prison_tower`

二选一，只能选一个：

#### 方案 A

如果能够证明 7 个失败只是测试期望漂移，且不需要改业务配置：

- 允许修正 `test_prison_tower.lua`
- 修完后继续作为 `blocking`

#### 方案 B

如果不能在本任务内证明只是测试漂移：

- 不改业务配置
- 不强行修测试
- 将 `prison_tower` 暂时移出默认阻断门禁
- 显式标记为 `known_red`

禁止第三种做法：

- 既不修，也不隔离，还让默认门禁一直红

### 步骤 3：收口空执行结果

必须补一条规则：

- 如果某次执行 `executed = 0`
- 且这次执行本意是跑某个 group 或门禁集
- 则输出必须明确提示这是“无有效覆盖”

可接受做法：

1. 返回特殊错误码
2. 输出 `[EMPTY]`
3. 在汇总里增加 `executed` 字段并触发失败

但不能继续默认为普通成功。

### 步骤 4：固化运行环境

对 `_run_via_lupa.py` 做最小收口：

1. 显式检查 Python 解释器可用性
2. 显式检查 `lupa` 是否可导入
3. 失败时打印明确指导信息
4. 最好补一个最小依赖说明文件

可接受新增：

- `scripts/tests/requirements.txt`
- `scripts/tests/README.md`

### 步骤 5：补 smoke 覆盖

至少补到 `test_runner_smoke.lua`：

1. 默认阻断集过滤正确
2. `known_red` 不进入默认门禁
3. 空执行结果不会伪绿
4. 桥接前置检查失败时提示清晰

## 8. 本任务的停手条件

出现以下情况，停止编码，改交付问题说明：

1. 修 `prison_tower` 需要修改业务配置或产品逻辑
2. 无法证明 `prison_tower` 失败是测试期望问题
3. 当前机器上没有任何可用的 Lua/Python 执行入口
4. 需要修改 `scripts/tests/` 之外的运行时代码才能继续

如果停手，仍然要交付：

- 门禁问题清单
- 阻塞原因
- 建议拆分方案

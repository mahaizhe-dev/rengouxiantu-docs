# BM-S4C：黑市卖出拦截模型收口任务卡

## 1. 任务定位

- **任务 ID**：`BM-S4C`
- **任务类型**：`高优先级代码收口`
- **目标**：收清黑市卖出当前 3 层拦截逻辑的职责边界，解决“看起来全部无法出售”的混乱表现。
- **前置状态**：
  - `BM-S4AR` 已实现强制分堆、锁图标、NPC 面板提示
  - `BM-S4B` 已尝试修“成功存档后解锁”和“整类禁售 UI”
  - 当前最新问题：**全局未同步门 + 同类锁门 + 服务端未锁数量门叠加，导致真实表现与产品规则不一致**

---

## 2. 本轮必须解决的核心问题

### 问题 A：卖出拦截模型职责混乱

当前存在 3 层门：

1. `BlackMarketSyncState.IsBlocked()`  
   - 全局未同步门  
   - 只要 `_dirtyConsume / _dirtyWarehouse / SaveSession.IsDirty()` 任一命中，就挡全部卖出

2. `InventorySystem.HasAnyLockedConsumable(itemId)`  
   - 同类锁门  
   - 只要某个 `itemId` 存在任意锁堆，整类禁售

3. `BackpackUtils.CountUnlockedItem(backpack, consumableId)`  
   - 服务端未锁数量门  
   - 服务端当前仍按“未锁数量是否足够”判断卖出

当前问题不是单个条件错，而是：

- **这 3 层还没有重新分工**
- 玩家表现成“全部都不能卖”
- UI 上看到的提示和真实阻断原因不一致

### 问题 B：产品规则未真正冻结到代码

用户已明确接受的产品规则是：

- **只要某个 `itemId` 存在任意锁定堆，该 `itemId` 整类暂不可黑市出售**
- **不做“部分可卖”**
- **强制新堆保持**
- **成功存档后必须正确解锁**

所以本轮必须保证：

- 同类锁门是明确、稳定、前置可感知的
- 全局未同步门不能再把大量无关商品一并挡掉，除非产品明确接受
- 服务端判定要和客户端产品规则对齐，不能一个按“整类禁售”，一个按“未锁数量足够即可卖”

---

## 3. 本轮目标

本轮只做 4 件事：

1. **明确拦截优先级与职责**
   - 定义 3 层门谁负责什么
   - 不能再让“全局未同步门”覆盖掉“整类锁门”的产品语义

2. **把 UI 前置口径改对**
   - 如果某个 `itemId` 因锁被禁售，按钮、状态文案、确认弹窗路径必须一致
   - 不能再出现“还能点出售 / 还能弹确认框 / 最后才报错”的链路

3. **把服务端卖出判定与产品规则收口**
   - 如果当前产品规则是“同类有锁即整类禁售”
   - 服务端也必须按这个规则处理
   - 不能继续按“只要还有未锁数量就放行”

4. **用测试把 3 类阻断原因分开**
   - `itemId` 锁导致禁售
   - 全局未同步导致禁售
   - 正常无锁且无未同步时可卖

---

## 4. 本轮明确不做的事

以下内容本轮禁止顺手做：

- 不做细粒度“同类部分可卖”
- 不改黑市商品配置结构
- 不改黑市买入逻辑
- 不改自动回收逻辑
- 不改仙石系统结算模型
- 不改存档版本号
- 不改 `server_main.lua`
- 不引入新的服务端重型权威库存模型

## 6. 固定实施步骤

### Step 1：先冻结 3 层门的职责

执行 AI 必须先给出一张简表，写清：

- 全局未同步门：负责什么
- 同类锁门：负责什么
- 服务端卖出门：负责什么

并明确：

- 哪一层负责“整类禁售”
- 哪一层负责“同步窗口”
- 哪一层负责“最终卖出放行”

### Step 2：收口客户端 UI 行为

至少做到：

- 同类锁命中时，商品卡直接不可卖
- 不进入“确认出售”弹窗
- 状态提示要明确区分：
  - `保护期禁售`
  - `背包数据未同步`

不能继续把不同原因都表现成同一条模糊提示。

### Step 3：收口服务端卖出判定

如果客户端按产品规则已经把某个 `itemId` 视为“整类禁售”，  
服务端 `HandleSell()` 也必须与之对齐。

本轮允许的最低要求：

- 服务端在处理 consumable 卖出时，检查该 `itemId` 是否存在任意活动锁堆
- 若存在，则整类拒绝卖出

不允许继续出现：

- 客户端整类禁售
- 服务端却只按 `CountUnlockedItem()` 放行

### Step 4：补测试，把 3 层门拆开验证

至少新增/补齐以下测试：

1. **同类锁命中**
   - 一个 `itemId` 只要存在一个锁堆
   - 整类禁售
   - 不弹确认框

2. **全局未同步命中**
   - 没有锁堆
   - 但 `_dirtyConsume / _dirtyWarehouse / SaveSession.IsDirty()` 命中
   - 黑市卖出被阻断
   - 提示应为未同步，而不是保护期

3. **正常放行**
   - 无锁堆
   - 无未同步
   - 服务端与客户端都允许卖出

4. **同类锁与未同步同时存在**
   - 要明确哪一个优先展示
   - 产品规则与代码行为一致

## 7. 通过标准

本轮通过必须同时满足：

1. 不再出现“背包只有锁定堆时，还能弹确认出售框”
2. `itemId` 锁导致禁售，与“全局未同步禁售”可被清晰区分
3. 服务端与客户端对 consumable 卖出规则一致
4. 测试中明确覆盖：
   - 锁命中
   - 未同步命中
   - 正常放行
5. 真实 runner 输出为绿：
   - `python3 scripts/tests/_run_via_lupa.py`
   - `python3 scripts/tests/_run_via_lupa.py --all`

---

## 8. 失败判据

出现以下任一项即判失败：

- 仍然可以弹出不该出现的“确认出售”框
- 同类锁规则只在客户端生效，服务端仍按未锁数量放行
- 全局未同步门继续把大量无关商品一并拦死，且无产品说明
- 测试只测纯函数，不测真实 UI / `HandleSell()` 路径

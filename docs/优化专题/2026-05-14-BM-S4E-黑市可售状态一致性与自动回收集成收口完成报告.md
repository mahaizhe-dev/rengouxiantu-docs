### 3.1 函数签名

```lua
---@param itemId string|nil  商品 ID（传 nil 则只检查 L2）
---@return string reason     "none" | "locked_item" | "global_unsync"
---@return string|nil detail 详细原因（用于日志）
function BlackMarketSyncState.GetSellBlockReason(itemId)
```

### 3.2 判定优先级

```
L1 同类锁门（优先级最高）
  ├─ 装备类豁免 → 跳过 L1
  └─ 消耗品 → HasAnyLockedConsumable(itemId) → "locked_item"
      ↓ 通过
L2 全局未同步门
  ├─ dirtyConsume   → "global_unsync"
  ├─ dirtyWarehouse → "global_unsync"
  └─ SaveSession    → "global_unsync"
      ↓ 通过
"none" — 允许卖出
```

### 3.3 调用点收敛（4 处 → 1 个入口）

| 调用位置 | 旧逻辑 | 新逻辑 |
|----------|--------|--------|
| 卡片级 `canSell` 计算 (L488) | 仅查 `HasAnyLockedConsumable` (L1) | `GetSellBlockReason` (L1+L2) |
| 按钮文本显示 (L615) | 仅显示 "🔒锁定" / "出售" | 三态: "🔒锁定" / "⏳同步中" / "出售" |

**核心改进**: 卡片级就能感知 L2，按钮不再出现"亮起但下一步拦住"的分裂。

## 4. "锁已解但仍 global_unsync"根因结论

### 根因分析

排查发现 **S4D 已修复了此问题的根因**:

1. **SaveSession._active 残留** — S4D 新增 `game_saved` 事件监听，`ClearOnSave()` 在存档成功时自动清除 `_active` 标记
2. **EventBus 同步执行模型** — `Emit("game_saved")` 使用 `pcall` 循环遍历快照数组，所有监听器在同一事件帧内执行完毕，因此:
   - `BlackMarketSyncState.ClearAll()` 清除 `_dirtyConsume` / `_dirtyWarehouse`
   - `SaveSession.ClearOnSave()` 清除 `_active` / `_opCount`
   - `Warehouse.ClearDirty()` 清除仓库脏标记
   三者**同帧完成**，不存在"清了 A 但 B 还脏"的中间状态

3. **结论**: "买入→保存成功→锁消失后仍 global_unsync"的场景**已不可复现**，无需额外修复代码

### 测试验证

`T10b` 断言验证了此行为:
```

T10b: game_saved 后所有标记清除 (got none, expected none) ✅
```

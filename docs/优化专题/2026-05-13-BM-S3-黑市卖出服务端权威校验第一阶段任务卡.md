# 2026-05-13 BM-S3 黑市卖出服务端权威校验第一阶段任务卡

## 1. 任务定位

这是黑市专项的第一阶段服务端权威收口任务。

在 `BM-S2R` 之后，正常客户端已知刷法已经被客户端门禁挡住，但当前黑市卖出服务端仍然存在核心缺口：

- `BlackMerchantHandler.HandleSell()` 仍直接依据服务端已保存的 `saveData.backpack`
- 只要本地先发生了未落盘的道具变化，旧快照就可能与真实状态分裂
- 改包客户端可以绕过 UI 门禁直接打 `C2S_BMSell`

因此，`BM-S3` 的目标不是继续补客户端名单，而是把“服务端直接信旧快照就卖”的模型收紧一层。

---

## 2. 为什么现在做

当前第五章前的几个长期问题里，黑市/仙石/自动回收仍是最核心的经济风险域。

`BM-S1/BM-S1R` 已冻结了风险矩阵，`BM-S2/BM-S2R` 已完成统一未同步门。

下一步如果还不补服务端侧防线，就仍然存在：

1. 正常客户端热修有效，但改包客户端可绕过
2. 新增黑市商品仍可能把旧快照卖出模型继续放大
3. 黑市作为核心经济系统，仍然缺“默认保守”的服务端策略

所以 `BM-S3` 应被视为第五章前的高优先级任务。

---

## 3. 本任务目标

### 3.1 把黑市卖出改成“服务端默认保守”

服务端不再对所有黑市可卖商品一律沿用“只要旧存档背包里有就允许卖”的逻辑。

必须至少区分两类：

1. **安全类商品**
   - 无已知本地先变化路径
   - 仍可沿用现有服务端快照卖出逻辑

2. **高风险类商品**
   - 在 `BM-S1` 风险矩阵中属于 `P0 / P1`
   - 或存在“本地先消耗 / 入仓 / 转资源 / 开盒 / 卖钱”路径
   - 如果服务端当前无法独立证明其状态安全，则本阶段应当**显式拒绝卖出**

### 3.2 不再按临时名单热修思路推进

本任务禁止继续做：

- 再补一个 `gold_brick`
- 再补几个附灵玉
- 再补一个盒子

必须基于 `BM-S1` 风险矩阵的“类别/分级”建立服务端规则。

### 3.3 保证失败是“显式失败”，不是“静默放行”

如果服务端无法确认高风险商品当前可安全卖出：

- 应返回明确失败
- 不得继续发放仙石
- 不得留下“本地异常但仙石到账”的窗口

---

## 4. 本任务明确不做

1. 不做全背包服务端权威化
2. 不做仓库整体服务端化
3. 不重写 `InventorySystem`
4. 不重写自动回收主逻辑
5. 不重构仙石/仙玉整套账本
6. 不改 `SaveGame` 存档结构
7. 不顺手做第五章商品接入

也就是说：

`BM-S3` 是“黑市卖出服务端侧第一阶段护栏”，不是“经济系统全面重构”。

## 8. 固定执行步骤

### 步骤 1：先交“当前服务端卖出权威口径盘点”

完成报告必须先写清：

1. `HandleSell()` 当前依赖哪些数据源
2. 哪些校验是“旧存档快照校验”
3. `BM-S1` 风险矩阵里，哪些商品属于本阶段必须收口的高风险类
4. 当前自动回收是否复用同一库存/货币路径

没有这张盘点表，不允许直接改代码。

### 步骤 2：建立“安全类 / 高风险类”服务端规则

本阶段必须形成明确规则：

1. **安全类**
   - 继续允许走现有卖出逻辑

2. **高风险类**
   - 若服务端当前无法独立证明安全，则本阶段一律显式拒绝卖出

原则：

- **宁可拒绝，不可误发仙石**
- 默认保守优先于“看起来功能全保留”

### 步骤 3：在 `HandleSell()` 落地服务端护栏

至少做到：

1. 高风险类商品不能再直接沿用“旧背包有就卖”
2. 被拒绝时必须有稳定失败返回
3. 失败时不得发仙石
4. 失败时不得写出误导性成功流水

### 步骤 4：客户端仅做必要适配

如果服务端返回了新的失败码或失败原因：

1. 允许在 `BlackMerchantUI.lua` 做最小提示适配
2. 不允许顺手改黑市交互结构

### 步骤 5：测试必须覆盖“绕过客户端门禁”的情况

至少要补：

1. 高风险商品直接请求卖出时，被服务端拒绝
2. 安全类商品卖出仍正常
3. 被拒绝时仙石余额不变
4. 被拒绝时黑市历史/流水不会写出成功记录
5. 自动回收既有行为不被本轮破坏

---

## 9. 成功标准

满足以下全部条件，才算通过：

1. 服务端不再对所有黑市商品一律信旧背包快照
2. `BM-S1` 定义的高风险类商品，本阶段已被服务端侧明确收口
3. 高风险卖出被拒绝时，不发仙石、不写成功流水
4. 安全类商品卖出不回归
5. 自动回收不被本轮破坏
6. blocking gate 继续全绿

---

## 10. 失败判据

出现以下任一情况视为失败：

1. 仍然主要靠客户端门禁，服务端无实质收口
2. 高风险类商品仍可通过直接 `C2S_BMSell` 领到仙石
3. 为了收口而误伤自动回收主链
4. 把本任务做成新的“名单热修”
5. 顺手引入协议/存档结构的大改

- “不做更大范围服务端化，就无法对高风险类建立可信校验”

- 对高风险类商品暂时服务端拒卖

但必须在报告里明确写清：

1. 拒卖范围
2. 用户可见影响
3. 为什么这比继续放行更安全

**先把黑市核心经济漏洞从服务端侧堵住。**

## 2. 迁移状态模型

### 定义的状态

| 状态常量 | 语义 |
|---------|------|
| `MIGRATION_DISABLED` | 迁移功能在产品层面已禁用（当前线上真实状态） |
| `NOT_NEEDED` | 不需要迁移（已有存档或服务端标记无需迁移） |
| `ELIGIBLE_CLIENT_CLOUD` | 具备通过 clientCloud 迁移的资格 |
| `ELIGIBLE_LOCAL_FILE` | 具备通过本地导出文件迁移的资格 |
| `ALREADY_MIGRATED` | 迁移已完成 |

### 输入 → 状态映射

| 输入 | 来源 | 说明 |
|------|------|------|
| `MIGRATION_GLOBALLY_ENABLED` | MigrationPolicy 内部常量 | 硬编码 `false`，代表产品层面迁移已废弃 |
| `needMigration` | server (`HandleFetchSlots`) | 服务端根据 `isEmpty` 判断 |
| `slotCount` | server (`HandleFetchSlots`) | 当前存档槽位数量 |
| `migration_backup_done` | local file (PC) | 仅 PC/Web 平台检查 |
| `FeatureFlags.SAVE_MIGRATION_SINGLE_ENTRY` | config | 灰度开关，控制新路径是否生效 |

状态决策树（P0-4R 修正后）：

```
flag=false → Evaluate() 返回 nil → 保留旧行为不变
flag=true →
  needMigration=false 或 slotCount>0 → NOT_NEEDED
  needMigration=true 且 slotCount=0  → ELIGIBLE_CLIENT_CLOUD
```

P0-4R 将 `MIGRATION_GLOBALLY_ENABLED` 从 Evaluate() 中移除，Evaluate() 只返回
现实状态（NOT_NEEDED / ELIGIBLE_CLIENT_CLOUD），动作决策由 ShouldAutoMigrate() /
ShouldBlockEntry() 独立负责（当前均返回 false，因迁移在产品层面已废弃）。

## 3. 入口收敛

### main.lua 改了什么

- 新增 `require("network.MigrationPolicy")`
- PC 迁移检查块（`migration_backup_done` 检测）外层增加 `MigrationPolicy.IsPCMigrationCheckNeeded()` 守卫
- flag=false 时 `IsPCMigrationCheckNeeded()` 返回 `true`，保留原有行为
- flag=true 时根据 `MIGRATION_GLOBALLY_ENABLED` 判断，当前返回 `false`，跳过整个检查块

### SaveSystemNet.lua 改了什么

- 新增 `require("network.MigrationPolicy")`
- FetchSlots handler 中原有内联 `needMigration` 判断重构为双分支：
  - flag=true 分支：调用 `MigrationPolicy.Evaluate(needMigration, slotCount)` 获取统一状态
  - flag=false 分支：保留原有 `"DEPRECATED, treating as new player"` 日志行为

### SAVE_MIGRATION_SINGLE_ENTRY=false 时行为

**与改动前完全一致**：
- `Evaluate()` 返回 `nil`，走 else 分支，执行原有逻辑
- `IsPCMigrationCheckNeeded()` 返回 `true`，PC 迁移检查块正常执行
- 旧代码中所有 `print("DEPRECATED...")`、条件判断全部原封不动保留

### SAVE_MIGRATION_SINGLE_ENTRY=true 时行为

- FetchSlots 中迁移判断走 MigrationPolicy，不再执行内联旧逻辑
- PC 迁移检查被 `IsPCMigrationCheckNeeded()` 拦截（返回 false）
- `ShouldBlockEntry()` 恒返回 false，不阻断登录流程
- `ShouldAutoMigrate()` 恒返回 false，不自动触发迁移
- 当前等效于"迁移已全局禁用，所有入口静默"

## 4. 废弃分支冻结

### 哪些旧迁移函数仍保留

| 函数 | 所在文件 | 状态 |
|------|---------|------|
| `MigrateToServer()` | SaveSystemNet.lua | **保留**，未删除 |
| `MigrateFromLocalFile()` | SaveSystemNet.lua | **保留**，未删除 |
| `HandleMigrateData()` | server_main.lua | **保留**（禁止修改文件） |
| `C2S_MigrateData` / `S2C_MigrateResult` | SaveProtocol.lua | **保留**（禁止修改文件） |

### 哪些自动触发路径被关闭或改为显式

- FetchSlots 中 `needMigration=true` 时：flag=true 下走 MigrationPolicy 统一判断，不再执行原有内联逻辑
- PC `migration_backup_done` 检查：flag=true 下被 `IsPCMigrationCheckNeeded()` 拦截
- **两条旧函数均未被自动调用**：`MigrateToServer()` 和 `MigrateFromLocalFile()` 在 flag=false 和 flag=true 下均不会被自动触发

### 为什么现在不删除

1. 任务卡 §5 明确禁止删除旧迁移函数
2. `MigrateToServer()` / `MigrateFromLocalFile()` 仍可能被未来的手动迁移流程使用
3. 服务端 `HandleMigrateData` 处理器必须保留以兼容可能的在途请求
4. 删除会造成不可回滚的破坏性变更，与"等价重构"目标不符

## 7. 风险与回滚

### 当前仍保留的歧义

1. `MigrateToServer()` / `MigrateFromLocalFile()` 函数体仍然存在，但无调用方 — 这是设计如此，后续 P0-6 服务端存档服务化时可决定是否清理
2. 服务端 `HandleFetchSlots` 仍然在 `needMigration` 为空槽位时赋值 `true` — 这在 flag=true 下被 `ELIGIBLE_CLIENT_CLOUD` 状态识别，但动作层（ShouldAutoMigrate）返回 false，不触发实际迁移

### 回滚方式

- 将 `SAVE_MIGRATION_SINGLE_ENTRY` 设为 `false`（默认值）即可完全回滚至改动前行为
- 无需改动任何代码文件，flag=false 时所有新路径不会被执行

## 3. Step 1：黑市最小闭环 + 自动回收

## 3.1 黑市闭环的最终目标

第五章前，黑市不追求“大模型正确”，只追求以下 4 条真实行为成立：

1. 黑市买入后，**只有该商品自己**进入保护锁  
2. 成功存档后，该商品能**真实解锁**  
3. 未锁的其他商品**不受影响**  
4. 商品卡、顶部提示、确认出售流程的语义**一致**

### 这意味着必须回到一个更简单的模型

#### 商品级状态

只表达这个 `itemId` 自己：

- `locked`
- `sellable`

#### 市场级状态

只表达整个黑市当前：

- 请求中
- 网络异常
- 刷新中

**市场级状态不再直接做成商品卡卖出状态。**

### 商品卡规则

- 有锁：显示 `🔒锁定`
- 无锁：显示 `出售`
- 不再显示 `⏳同步中`

### 新增商品约束

- `A 类纯库存型`：允许直接接入黑市配置
- `B 类受保护接入型`：必须同时接入保护锁、操作闸和测试
- 如果暂时判不清，默认归为 `B 类`

### 提交流程规则

- 商品卡是否可点击，只看商品级锁
- 点击确认后，再看市场级状态是否允许提交
- 如果市场级状态不允许，提示：
  - `背包数据未同步，请稍后再试`
  - 但不能把全市场所有商品都改成同一状态

## 3.2 黑市这次允许保留的既有实现

如果当前代码里这些点已经成立，则保留，不再推翻：

- 强制新堆
- 锁图标
- NPC 面板常驻说明
- 锁不落盘
- 买入确认/成功二次提示

## 3.3 黑市这次必须修正的点

### A. 商品级锁与市场级状态彻底拆开

必须撤回此前 `BM-S4E` 那类做法：

- `global_unsync` 不再驱动商品卡 `canSell`
- 商品卡不再显示“同步中”

### B. 存档后解锁闭环

必须验证真实路径：

- 买入
- `save_request`
- `game_saved`
- UI 刷新
- 该商品解锁
- 其他商品始终不被误伤

### C. 确认出售弹窗只承担确认，不承担商品级判定

商品级锁应当在进入确认框前就挡掉。  
确认框只做：

- 数量确认
- 市场级状态最终阻断

## 3.4 自动回收这次的目标

自动回收不再只看“日期计算函数是否对”，而是验证完整流程：

- `RecycleTick()` 是否在真实自然日推进时触发
- `lastRecycleDate_` 是否正确推进
- 云端 `bm_recycle_date` 是否正确推进
- `ExecuteRecycle()` 是否在正确日只执行一次
- 跨月 / 跨年 / 长时间停服后重开是否仍合理

### 这次不要求做的

- 不改黑市自动回收整体产品规则
- 不做黑市 / 回收 / 库存大重构

## 4. Step 2：N1 网络波动容错

## 4.1 目标

`N1` 这次只做长期高价值、低入侵的那部分：

- 断线判定加宽限 / 滞回
- 持续断连强提示
- 只读请求短容错
- 高风险写操作不自动补发

## 4.2 这次不做的

- 会话无感重连
- 服务端连接恢复大重构
- 写请求自动补发

## 4.3 N1 为什么放在黑市后面

不是因为 `N1` 不重要，而是因为：

- 黑市当前有正在阻断真实功能的问题
- 黑市主交易路径必须先恢复到稳定可测
- 然后再加 `N1`，能避免将网络容错噪音和黑市状态问题混在一起

---

## 5. 测试与评估方式

这次不再接受“逻辑自洽 + 纯函数测试全绿 = 通过”。  
必须把真实玩家路径写成第一验收标准。

## 5.1 黑市闭环必须覆盖的真实路径

1. 买入 `A` 后，该 `itemId` 显示锁定  
2. 同时，未锁的 `B` 仍可进入出售确认  
3. 成功存档后，`A` 解锁  
4. 重登后，`A` 仍保持已解锁状态  
5. 不再出现“整个黑市一起同步中”  
6. 商品卡显示、顶部提示、确认出售三层语义一致

## 5.2 自动回收必须覆盖的真实路径

1. 同日内只执行一次  
2. 次日会再次触发  
3. 跨月仍正常  
4. 跨年仍正常  
5. 长时间停机再开不会出现数值差误判

## 5.3 N1 必须覆盖的真实路径

1. 短暂抖动不立刻判死  
2. 持续断连进入强提示  
3. 只读请求在短时间波动内不立刻硬失败  
4. 黑市、商城、兑换等写请求在波动时不会自动重发

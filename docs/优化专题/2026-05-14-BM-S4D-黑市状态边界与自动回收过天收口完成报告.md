## 3. SaveSession → BlackMarketSyncState 问题根因说明

### 根因

`SaveSession` 模块用于批量合并高频存档操作（洗炼、锻造、炼丹等）。当 `Begin()` 被调用后 `_active = true`，仅在内部 `_flush()` 超时回调中清除。

问题在于：当游戏通过其他路径触发存档（如黑市交易后的 `game_saved` 事件），`_active` 标志不会被清除。而 `BlackMarketSyncState.IsBlocked()` 的 L2 门包含了 `SaveSession.IsDirty()` 检查：

```lua
-- BlackMarketSyncState.lua L2 门
function M.IsBlocked()
    if _dirtyConsume or _dirtyWarehouse then return true, ... end
    if SaveSession.IsDirty() then return true, "save_session_dirty" end  -- ← 陈旧阻塞
    return false
end
```

`game_saved` 事件触发时，`_dirtyConsume` 和 `_dirtyWarehouse` 已被 `ClearAll()` 正确清除，但 `SaveSession._active` 仍为 `true`，导致 L2 门持续误拦。

### 修复方案

为 `SaveSession` 新增 `ClearOnSave()` 公开 API，并订阅 `game_saved` 和 `game_loaded` 事件：

```lua
function SaveSession.ClearOnSave()
    _active    = false
    _opCount   = 0
    _startTime = 0
end

EventBus.On("game_saved", function() SaveSession.ClearOnSave() end)
EventBus.On("game_loaded", function() SaveSession.ClearOnSave() end)
```

安全性：`game_saved` 意味着数据已持久化，清除陈旧会话不会丢弃未保存数据。`game_loaded` 是全量加载后的初始态，同样安全。

### 修复后行为

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 黑市交易 → game_saved | L2 门误拦（SaveSession.IsDirty=true） | L2 门放行（ClearOnSave 清除） |
| 手动存档 → game_saved | SaveSession 陈旧残留 | 自动清除 |
| 加载存档 → game_loaded | SaveSession 可能残留旧会话 | 自动清除 |
| 正常洗炼批次内 | 不影响（_flush 正常工作） | 不影响 |

## 4. 自动回收过天问题根因说明

### 根因

`RecycleTick`（BlackMerchantHandler.lua）使用 YYYYMMDD 整数直接相减来计算日差：

```lua
local gap = todayNum - cloudDate  -- 例: 20260601 - 20260531 = 70（实际应为 1 天）
```

跨月/跨年时产生错误结果：

| 场景 | 旧算法（整数差） | 实际天数 | 后果 |
|------|-----------------|---------|------|
| 5月31日→6月1日 | 70 | 1 | `>MAX_RECYCLE_GAP(2)` → 异常分支，跳过回收 |
| 12月31日→1月1日 | 8870 | 1 | 同上 |
| 2月28日→3月1日（平年） | 73 | 1 | 同上 |

### 修复方案

新增 `BMConfig.DateToTime(dateNum)` 和 `BMConfig.RealDayGap(todayNum, cloudDate)` 两个工具函数：

```lua
function BMConfig.DateToTime(dateNum)
    -- YYYYMMDD → os.time 时间戳（hour=12 避开夏令时）
    local s = tostring(math.floor(dateNum))
    local y, m, d = s:sub(1,4), s:sub(5,6), s:sub(7,8)
    return os.time({ year=y, month=m, day=d, hour=12 })
end

function BMConfig.RealDayGap(todayNum, cloudDate)
    local t1, t2 = BMConfig.DateToTime(todayNum), BMConfig.DateToTime(cloudDate)
    if t1 and t2 then return math.floor((t1 - t2) / 86400) end
    return todayNum - cloudDate  -- fallback
end
```

`RecycleTick` 中：
- `numericGap`（整数差）仅用于 `money:Add` 增量（正确：Add 需要 YYYYMMDD 值的跳跃量）
- `realGap`（真实日差）用于所有判断分支（同天检测、异常跨度检测）

### 修复后行为

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 5月31日→6月1日 | gap=70，走异常分支 | realGap=1，正常回收 |
| 12月31日→1月1日 | gap=8870，走异常分支 | realGap=1，正常回收 |
| 同一天 | gap=0，正确跳过 | realGap=0，正确跳过 |
| 超过 MAX_RECYCLE_GAP | gap 可能不准 | realGap 精确判断 |

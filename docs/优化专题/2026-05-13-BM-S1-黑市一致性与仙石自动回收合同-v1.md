## 结论一：黑市"可卖性"的定义

### 当前现实口径

一个物品是否可在黑市卖出，**完全且仅由** `BlackMerchantConfig.ITEMS` 表决定。

```lua
-- BlackMerchantConfig.lua
BMConfig.ITEMS = {
    rake_fragment_1 = ItemFromGame("rake_fragment_1", { buy_price = 10, ... }),
    -- ... 共 79 个条目
}
```

**判定规则**:

1. 物品 ID 存在于 `BMConfig.ITEMS` 中 → 可卖
2. 物品 ID 不在 `BMConfig.ITEMS` 中 → 不可卖
3. 没有运行时开关、没有条件判断、没有玩家等级限制（解锁门槛 `JINDAN_ORDER` 仅控制黑市 UI 是否可打开，不控制单个商品的可卖性）

**物品类型区分**:

| `itemType` 值 | 含义 | 卖出链路 |
|---------------|------|---------|
| `nil`（默认） | 消耗品 | `HandleSell` → `RemoveFromBackpack` → 从 `saveData.backpack` 移除 |
| `"equipment"` | 装备 | `HandleSell` → `RemoveEquipmentFromBackpack` → 从 `saveData.equipmentSlots` 移除 |

**合同约束**: 任何新增商品必须先加入 `BMConfig.ITEMS`，此操作等价于声明该物品进入黑市风险面。

## 结论二：改变黑市可卖性的操作定义

以下操作会改变一个物品在黑市中的"可卖数量"，即影响服务端 `HandleSell` 中 `CountItemHeld(backpack, consumableId)` 的返回值：

### 2.1 增加可卖数量的操作

| 操作 | 代码入口 | 存盘时机 | 说明 |
|------|---------|---------|------|
| 怪物掉落/奖励获取 | `InventorySystem.AddConsumable()` | 取决于调用方 | 增加背包中物品数量 |
| 仓库取出 | `WarehouseSystem.RetrieveItem()` | 立即 `save_request` + dirty | 物品从仓库回到背包 |
| BM 购买 | `HandleBuy` (服务端) | 服务端直接写 | 服务端直接 `AddToBackpack` |
| GM 命令 | `GMConsole` | 取决于命令 | 调试工具直接添加 |

### 2.2 减少可卖数量的操作

| 操作 | 代码入口 | 存盘时机 | BM 门禁 |
|------|---------|---------|---------|
| 炼丹消耗 | `AlchemyUI` → `ConsumeConsumable()` | `SaveSession.MarkDirty()` | **未覆盖** |
| 附灵消耗 | `TemperUI` → `ConsumeConsumable()` | `SaveSession.MarkDirty()` | **未覆盖** |
| 龙锻消耗 | `DragonForgeUI` → 直接操作 | `save_request` 直接 | 无需门禁（无延迟窗口） |
| 开盒消耗 | `InventorySystem._UseTokenBox()` | `save_request` + dirty | **BM-01A 覆盖** |
| 宠物学书 | `PetPanel` → 消耗技能书 | 隐式 dirty | **未覆盖** |
| 仓库存入 | `WarehouseSystem.StoreItem()` | `save_request` + dirty | **BM-01 覆盖** |
| 本地卖钱 | `InventorySystem.SellItem()` | 取决于调用方 | **未覆盖** |
| BM 卖出 | `HandleSell` (服务端) | 服务端直接写 | N/A（自身操作） |

### 2.3 操作类型分类

根据存盘时机将操作分为三类：

| 类型 | 特征 | 风险 |
|------|------|------|
| **即时存盘** | 操作后立即 `emit("save_request")` | 低风险 — 服务端 saveData 几乎实时 |
| **延迟存盘** | 通过 `SaveSession.MarkDirty()` 批量延迟 | **高风险** — 最长 20 秒/5 次操作窗口 |
| **隐式存盘** | 操作后无明确存盘调用，依赖后续流程 | **高风险** — 窗口不确定 |

**合同约束**: BM-S2 必须按"操作类型"而非"物品名单"设计门禁。任何"延迟存盘"或"隐式存盘"的操作，在其生效期间必须阻断 BM 卖出。

## 结论三：必须进入"黑市未同步门"的状态变化

### 3.1 当前门禁覆盖范围

| 门禁 | 机制 | 覆盖范围 |
|------|------|---------|
| BM-01 | `WarehouseSystem.IsDirty()` 检查 | 仅仓库操作（StoreItem/RetrieveItem/UnlockRow） |
| BM-01A | `_BM_SENSITIVE_CONSUMABLES` 白名单 | 仅 3 个令牌盒 |

### 3.2 BM-S2 需要补充的门禁

以下状态变化**当前未被覆盖**，但**必须**纳入黑市未同步门：

| 状态变化 | 当前入口 | 涉及 BM 商品数 | 补充方式建议 |
|---------|---------|---------------|------------|
| `ConsumeConsumable` 全量消耗 | `InventorySystem.ConsumeConsumable()` | 53 种 | 在 `ConsumeConsumable` 统一设脏标记 |
| `SaveSession` 活跃会话期间 | `SaveSession.MarkDirty()` | 同上 | 检查 `SaveSession.IsActive()` 状态 |
| 宠物技能书学习 | `PetPanel` 技能学习 | 34 种 | 学习后触发 `BackpackDirty` |
| 本地卖钱 | `InventorySystem.SellItem()` | 理论上所有 | SellItem 后触发 `BackpackDirty` |

### 3.3 建议的统一机制

**不再使用物品白名单**，改为背包级脏标记：

```
方案 A: 在 InventorySystem 引入 BackpackDirty 标记
  - ConsumeConsumable / SellItem / 任何背包变更 → BackpackDirty = true
  - BM 卖出门禁: WarehouseDirty OR BackpackDirty → 阻断
  - save_request 确认后 → BackpackDirty = false

方案 B: 复用 SaveSession.IsActive()
  - BM 卖出门禁: SaveSession 有活跃会话 → 阻断
  - 更简单但粒度更粗（SaveSession 涵盖所有延迟存盘操作）
```

**合同约束**: BM-S2 必须选择一个方案并实现，不允许继续使用 `_BM_SENSITIVE_CONSUMABLES` 名单式补丁。

## 结论四：服务端卖出权威缺口

### 4.1 当前验证链路

```
客户端 BM UI → C2S "bm_sell" { consumableId, amount }
    ↓
HandleSell:
    1. sellActive_[connKey] 互斥锁
    2. Session.ServerGetSlotData(userId, slot) → 获取 saveData
    3. CountItemHeld(saveData.backpack, consumableId) → 校验数量
    4. RemoveFromBackpack(saveData.backpack, ...) → 本地修改 saveData
    5. WriteSaveBack(userId, slot, saveData) → 异步写回存档
    6. BatchCommit(Add xianshi, Add stock) → 异步写 serverCloud
```

### 4.2 缺口分析

| 缺口 | 说明 | 影响 |
|------|------|------|
| **快照时效性** | `saveData.backpack` 是上次 `save_request` 的快照，不反映客户端未存盘的变更 | 延迟存盘窗口内，服务端校验的是过期数据 |
| **无版本号** | 客户端 C2S 不携带存档版本号或序列号 | 服务端无法判断 saveData 是否为最新 |
| **WriteSaveBack 非原子** | `WriteSaveBack` 和 `BatchCommit` 是独立异步操作 | 中间失败时有 WAL 补偿，但 WAL 本身也是异步写 |
| **sellActive 锁粒度** | `sellActive_[connKey]` 只防同一连接并发，不防连接重建 | 理论上极端情况下快速断线重连可绕过 |

### 4.3 BM-S3 需要解决的问题

1. **存档新鲜度验证**: 在 `HandleSell` 中引入存档版本号或时间戳校验
2. **或者**: 在卖出前强制触发客户端 flush + 等待确认
3. **或者**: 将卖出操作改为服务端原子 read-modify-write

**合同约束**: BM-S3 必须选择一个方案消除 `saveData.backpack` 快照时效性缺口。客户端门禁（BM-S2）是必要的辅助手段，但不能替代服务端权威验证的补强。

## 结论五：自动回收和仙石结算的治理边界

### 5.1 自动回收与黑市的关系

| 维度 | 关系 |
|------|------|
| 库存来源 | 玩家 BM 卖出 → 系统库存增加 → 回收减少库存 |
| 仙石流向 | 玩家卖出获得仙石（`xianshi` serverCloud.money.add），回收不涉及仙石 |
| 价格发现 | 回收降低库存 → 玩家可购买的库存恢复 → 间接影响仙石流通 |

### 5.2 需要一起治理的边界

| 边界 | 说明 | 归属任务 |
|------|------|---------|
| 回收开关状态 | 设计文档写 `false`，代码实际 `true` — 需统一 | BM-S4 |
| NPC 名称 | "胤" vs "大黑无天" — 需统一 | BM-S4 |
| 回收 fire-and-forget | 每个物品独立 commit 无确认，中间失败丢失 | BM-S4 |
| 回收审计日志 | `TradeLogger.LogPublic` 与回收操作无事务保证 | BM-S4 |
| 仙石汇率 | 100 灵韵 = 1 仙石(买)，1 仙石 = 50 灵韵(卖) — 差价 50% | 当前设计意图，非缺口 |
| WAL 三路补偿 | `bm_wal`(灵韵) / `bm_wal_bp`(背包) / `bm_wal_sell`(仙石) — 已实现 | 已有保护，BM-S4 可审计 |

### 5.3 仙石结算安全性

当前仙石增发唯一路径：`HandleSell` → `serverCloud.money.add(userId, "xianshi", buyPrice)`

如果幻影卖出发生，仙石增发是真实的且不可撤回（没有反向扣减机制）。WAL `bm_wal_sell` 仅补偿"背包已扣但 xianshi 未加"的正向故障，不处理"backpack stale 导致的违规增发"。

**合同约束**: BM-S4 需要：
1. 确认 `RECYCLE_ENABLED` 预期值并统一代码与文档
2. 评估是否需要回收操作的事务性保证
3. 评估仙石增发是否需要增加事后审计/回滚机制

## 结论六：第五章新增道具接入黑市的前置检查

### 6.1 检查清单

任何新道具在加入 `BMConfig.ITEMS` 前，必须完成以下检查：

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | **本地消耗路径枚举** | 列出所有客户端可以消耗/使用该物品的 UI 和函数入口 |
| 2 | **存盘方式确认** | 每个消耗路径是 `save_request` 直接 还是 `SaveSession.MarkDirty()` 延迟 |
| 3 | **门禁覆盖确认** | 消耗后是否触发 BM 未同步门（BM-S2 实施后为 BackpackDirty 或等价机制） |
| 4 | **服务端验证确认** | `HandleSell` 能否正确校验该物品的持有数量（消耗品 vs 装备 vs 新类型） |
| 5 | **自动回收兼容** | 如果 `max_stock > 0`，该物品是否适合被自动回收（回收 = 减库存） |
| 6 | **仙石定价审计** | `buy_price` / `sell_price` 是否符合 2:1 比例，是否有套利空间 |

### 6.2 分级准入

| 级别 | 条件 | 流程 |
|------|------|------|
| **可直接准入** | 无本地消耗路径（如碎片类），或所有消耗路径直接 `save_request` | 加入 BMConfig.ITEMS 即可 |
| **需专项审查** | 存在延迟存盘消耗路径，但 BM-S2 门禁已覆盖该操作类型 | 确认门禁覆盖后加入 |
| **当前禁止准入** | 存在延迟存盘消耗路径，且 BM-S2 门禁未覆盖 | 必须先扩展门禁，再加入 |

## 结论七：黑市未来新增商品的最小准入规则

### 7.1 准入规则

```
新商品准入决策树:

1. 该商品是否有本地消耗路径？
   ├─ 否 → 安全类，可直接准入
   └─ 是 ↓

2. 所有消耗路径是否直接 emit("save_request")？
   ├─ 是 → 安全类，可直接准入
   └─ 否 ↓

3. 消耗路径是否走 SaveSession.MarkDirty()？
   ├─ 是 → 高风险类 ↓
   └─ 否（隐式/无明确存盘） → 高风险类 ↓

4. BM 未同步门（BM-S2 方案）是否已覆盖该操作类型？
   ├─ 是 → 需专项审查（确认覆盖后准入）
   └─ 否 → 当前禁止准入（先扩展门禁）
```

### 7.2 必须拒绝的准入模式

以下做法在 BM-S2 门禁升级到操作类型级别之前，**禁止使用**：

1. ❌ "在 `_BM_SENSITIVE_CONSUMABLES` 中加一个 ID" — 名单式补丁不可扩展
2. ❌ "该物品很少被消耗，风险可接受" — 概率论不适用于安全边界
3. ❌ "先上线观察，有问题再改" — 仙石增发不可逆
4. ❌ "客户端 UI 不允许同时打开两个面板" — UI 层面阻断不是安全保证

### 7.3 准入模板

新商品准入必须填写以下模板并留档：

```markdown
## 商品准入审查: {item_id}

- 商品名称: 
- 分类: 
- buy_price / sell_price: 
- 本地消耗路径:
  - [ ] 路径1: {UI/函数名} → 存盘方式: {save_request / MarkDirty / 隐式}
  - [ ] 路径2: ...
- BM 门禁覆盖: {是/否，具体机制}
- 服务端验证: {消耗品链路 / 装备链路 / 需新增}
- 自动回收: {适用 / 不适用}
- 准入结论: {可直接准入 / 需专项审查 / 当前禁止}
- 审查人: 
- 日期:
```

## 附录 B：术语表

| 术语 | 定义 |
|------|------|
| 幻影卖出 | 客户端已消耗物品但存档未同步，在延迟窗口内向黑市卖出同一物品，导致仙石凭空增发 |

| WAL | Write-Ahead Log，黑市交易补偿日志（`bm_wal` / `bm_wal_bp` / `bm_wal_sell`） |
| 操作类型级门禁 | 按"消耗/卖出/存取"等操作类型设计门禁，而非按物品 ID 名单 |

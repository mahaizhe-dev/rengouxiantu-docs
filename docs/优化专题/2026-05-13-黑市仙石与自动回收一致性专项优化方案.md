# 2026-05-13 黑市、仙石与自动回收一致性专项优化方案

## 1. 方案定位

这不是一组零散热修的汇总，而是一个单独专项。

专项范围统一覆盖：

- 万界黑市买入/卖出
- 仙石/仙玉余额与发放
- 黑市自动回收
- 黑市可卖物与本地背包/仓库/消耗/转化路径
- 黑市交易日志、补偿与回滚边界

当前必须先明确一个判断：

- 黑市问题不是“仓库有 bug”“3 个令牌盒有 bug”“金砖有 bug”
- 黑市问题的根因是：`服务端卖出校验看旧 saveData.backpack`，而项目大量系统采用`本地先改物品，再异步存档`

因此，后续优化不能再按“发现一个物品，再补一个名单”推进。

## 2. 当前系统性问题

### 2.1 物品权威口径和经济权威口径分裂

现状是：

- 仙石结算是服务端权威
- 黑市卖出校验依据服务端持久化背包快照
- 大量物品变化发生在客户端本地运行时

这会持续制造以下漏洞类型：

- 本地先使用，再黑市卖出
- 本地先入仓，再黑市卖出
- 本地先转化资源，再黑市卖出
- 本地先合成/强化/喂养消耗，再黑市卖出

### 2.2 当前热修是止血，不是模型修复

已经上线的 `HOTFIX-BM-01` 和 `HOTFIX-BM-01A` 只完成了两类止血：

- 仓库存取后的黑市禁卖
- 3 个令牌盒本地先用后的黑市禁卖

它们没有解决：

- 所有黑市可卖且本地可变更的物品类别
- 改包客户端直接发卖出请求
- 服务端卖出校验仍然只看旧背包快照

### 2.3 自动回收和仙石经济不能拆开看

自动回收曾经导致重大事故，说明以下 4 个对象现在实际上属于同一经济域：

- 黑市库存
- 仙石发放
- 自动回收
- 交易日志/补偿

后续治理必须统一考虑，不能再拆成“黑市问题”和“自动回收问题”两条线。

### 2.4 测试和审计还不够支撑“最重要系统”

当前 BM 相关测试主要覆盖局部热修，不足以证明：

- 黑市全量可卖物安全
- 本地先变更类操作安全
- 自动回收不会再次污染经济
- 仙石流水具备足够审计能力

## 3. 专项目标

本专项的目标不是“修掉当前已知 bug”，而是把黑市系统推进到“不会继续按物品名单冒洞”的状态。

目标分 4 层：

1. 冻结真实风险面  
把黑市可卖物、本地变更路径、仙石发放、自动回收放进同一张矩阵。

2. 建立统一黑市未同步门  
不再依赖 `WarehouseSystem._dirty` 和特例物品集合。

3. 补服务端第二道权威校验  
卖出请求不能再只靠客户端门禁。

4. 给第五章建立黑市接入护栏  
后续新增可交易物、可消耗物、可转化物，必须过专项检查。

## 4. 扩展性约束

黑市后续还会继续加商品，所以本专项必须同时解决“当前漏洞”和“未来扩展失控”两个问题。

必须明确：

- 黑市新增商品不是普通配置变更
- 黑市新增商品等同于新增一个经济结算入口
- 只要新增商品同时具备“可黑市卖出 + 可本地先变化”，就必须纳入专项审查

后续黑市商品应至少分成两类：

### 4.1 安全类商品

满足以下特征的商品，风险较低：

- 只在背包中静态持有
- 没有本地即时使用/转化路径
- 不进入仓库以外的复杂系统
- 不会在客户端先变成别的资源

这类商品可以作为相对低风险扩展对象，但仍需经过最小准入检查。

### 4.2 高风险商品

满足以下任一条件的商品，默认高风险：

- 可本地使用
- 可本地卖钱/卖灵韵
- 可开盒/拆包/转资源
- 可被炼丹/龙锻/宠物/灵玉系统消耗
- 可入仓后再参与交易
- 可被未来章节系统进一步加工

这类商品不允许直接通过“改配置”接入黑市。

### 4.3 新增商品准入原则

后续每次新增黑市商品，必须补齐以下信息：

- 商品分类
- 本地变更类型
- 是否会改变其他资源
- 是否进入黑市未同步门
- 是否需要服务端特殊校验
- 风险等级
- 准入结论

默认原则：

- 没做专项审查，不准入
- 无法回答本地变更路径，不准入
- 需要服务端权威但尚未落地，不准入

## 5. 后续路线

### BM-S1 风险矩阵与合同冻结

目标：

- 盘点所有黑市可卖物
- 盘点所有本地先变更路径
- 标注“当前已被热修覆盖 / 未覆盖 / 未知”
- 产出后续统一修复合同

性质：

- docs-only
- 不改运行时代码
- 不改协议
- 不改黑市配置

这是本专项第一步，也是后续所有修复的事实基线。

### BM-S2 统一黑市未同步门

目标：

- 抽离独立的黑市未同步状态，不再复用 `WarehouseSystem._dirty`
- 按“操作类型”置脏，而不是按物品名单置脏
- 覆盖：
  - 仓库存取
  - 本地卖钱类
  - 本地开盒/转资源类
  - 炼丹/龙锻/宠物书/灵玉等本地消耗路径

约束：

- 仍不改协议
- 仍不改存档结构
- 先堵正常客户端路径

### BM-S3 服务端卖出权威校验收口

目标：

- `BlackMerchantHandler.HandleSell` 不再只信任旧 `saveData.backpack`
- 增加至少一层服务端二次一致性校验

可选方向由 BM-S1 冻结结果决定：

- 背包修订号 / 同步 epoch 校验
- 关键卖出物白名单服务端阻断
- 卖出前强制要求最近一次同步确认
- 更完整的服务端权威背包口径

约束：

- 不和大规模存档重构混做
- 不顺手重写整个黑市协议

### BM-S4 仙石与自动回收审计收口

目标：

- 给自动回收、黑市卖出、黑市买入、兑换建立统一审计视图
- 明确补偿/WAL/公共流水/个人流水的职责
- 识别哪些仙石变动目前不可追责

这一阶段不一定马上改逻辑，但必须收清审计口径。

### BM-S5 第五章接入护栏

第五章开始前，新增道具如果满足以下任一条件，必须过专项审查：

- 可黑市卖出
- 可本地使用
- 可本地转化资源
- 可被炼丹/龙锻/宠物/灵玉系统消耗
- 可进仓库再参与交易

## 6. 推荐顺序

优先级建议如下：

1. `BM-S1`
2. `BM-S2`
3. `BM-S3`
4. `T1` 移动端发热缓解最小补丁
5. `BM-S4`
6. `C5-PREP`

说明：

- 黑市专项优先级高于当前其他结构性优化
- `R3` 之后，不建议先做大规模客户端结构整理
- 第五章正式主线开发前，至少完成到 `BM-S3`

## 7. 测试与验证原则

### 6.1 自动化测试

本专项后续测试不能再只按单个物品写。

必须逐步建立 3 层覆盖：

1. 配置层  
哪些物品可黑市卖。

2. 行为层  
哪些本地操作会改变这些物品的可卖性。

3. 结果层  
未同步状态下是否被正确阻断，服务端是否仍可被绕过。

### 6.2 线上验证

因为项目缺少稳定线上遥测，本专项每轮运行时代码都要遵守：

- 一次只上线一轮黑市改动
- 先自有账号验证
- 再观察玩家反馈
- 不把两个高风险黑市任务打包上线

## 8. 对执行 AI 的要求

本专项后续所有报告，必须单列以下章节：

- 黑市一致性影响评估
- 仙石余额/发放影响评估
- 自动回收影响评估
- 是否引入新的“本地先变更、服务端后结算”窗口
- 是否新增了黑市可卖且本地可变更的道具

## 9. 当前建议

当前最正确的下一步不是再开一个针对单个物品的热修卡，而是先执行：

- `BM-S1 黑市、仙石与自动回收风险矩阵冻结`

只有先把风险面冻住，后面 `BM-S2 / BM-S3` 才不会继续按名单修补。

# 仙缘宝箱系统

> **状态**：施工稿 v2.0
> **设计日期**：2026-05-10
> **参考实现**：
> - **3 秒读条交互**：复用 `FortuneFruitSystem.lua` 的 `CHANNEL_DURATION = 3.0` 逻辑
> - **3 选 1 奖励 UI**：复用 `QuestRewardUI.lua` 的 `quest_reward_pick` 事件链路
> - **交互半径**：复用 `FortuneFruitSystem.INTERACT_RANGE = 1.5`（瓦片）

---

## 1. 概述

仙缘宝箱是章节底图上的固定隐藏宝箱系统。**四章各 4 个，共 16 个**，分别对应**根骨、福源、悟性、体魄**四种仙缘属性。每个宝箱只检查自己的对应属性，达标后可进行 3 秒解封读条，成功后获得当前章节档位的 3 选 1 装备奖励。

**核心定位**：
- 让四种仙缘属性都拥有明确的探索型回报
- 每个职业在本章都天然对应 1 个"自己最容易开的宝箱"
- 其余 3 个宝箱需要额外洗练、问心、丹药、日常成长支持
- 用固定藏宝室强化地图探索感和记忆点

## 2. 核心规格

| 项目 | 规格 |
|------|------|
| 宝箱数量 | **四章各 4 个，共 16 个** |
| 属性分布 | 每章各放置 1 个根骨箱、1 个福源箱、1 个悟性箱、1 个体魄箱 |
| 刷新方式 | 底图固定点位，非随机、非每日刷新 |
| 开启次数 | 每角色每箱仅可开启 **1 次**，永久记录 |
| 开启条件 | 仅检查**对应属性**是否达到门槛 |
| 交互半径 | **1.5 瓦片**（与福源果、悟道树一致） |
| 交互方式 | 点击宝箱查看要求，达标后点击"开启"并进行 **3 秒读条** |
| 读条表现 | 复用 `FortuneFruitSystem` 采集读条，附加封印破裂表现 |
| 读条中断 | 移动、受击、离开交互半径、主动取消 |
| 断线/中断处理 | **交互失败**，不保留中间状态，需重新交互 |
| 奖励方式 | 固定 Tier/品质生成 3 件不同槽位装备，玩家 **3 选 1** |
| 奖励品质 | 第一章 T3 紫（`purple`）；第二章 T5 橙（`orange`）；第三章 T7 橙（`orange`）；第四章 T9 青（`cyan`） |
| 槽位池 | `helmet`、`armor`、`shoulder`、`belt`、`boots`、`ring1`、`ring2`、`necklace`（共 8 个，见 §3.2） |
| 宝箱占地 | 宝箱实体占 1 格 |
| 环境占地 | 宝箱周围需有完整 **3x3** 环境瓦片区，不刷怪、不放障碍、不放采集物 |
| 藏宝室入口 | 入口可达但隐蔽，最后一段通路宽度 **最多 1 格** |
| 宝箱贴图 | 四种宝箱各有独立 PNG（见 §10） |

## 4. 门槛数值设计

### 4.1 计算模型

本系统门槛的**核算基线**仍按以下模型：

> **9 件装备均洗出 1 条对应仙缘属性 + 对应职业等级成长属性**

即：

```text
章节门槛 = 9 x 单条对应仙缘副属性值 + 章节等级上限
```

其中：
- 线性仙缘副属性公式：`LinearSub = floor(tier x qualityMultiplier)`
- 对应品质倍率引用自 `GameConfig.QUALITY_MULTIPLIER`：
  - purple：`1.3`
  - orange：`1.5`
  - cyan：`1.8`
- 章节等级上限取《章节设计》中的章节最高等级：
  - 初章 `Lv.16`
  - 贰章 `Lv.36`
  - 叁章 `Lv.65`
  - 肆章 `Lv.100`

### 4.2 单条仙缘副属性值

| 章节奖励档位 | 公式 | 单条值 |
|-------------|------|-------|
| 第一章 T3 purple | `floor(3 x 1.3)` | **3** |
| 第二章 T5 orange | `floor(5 x 1.5)` | **7** |
| 第三章 T7 orange | `floor(7 x 1.5)` | **10** |
| 第四章 T9 cyan | `floor(9 x 1.8)` | **16** |

### 4.3 理论开启值核算

| 章节 | 9 件装备洗练总值 | 等级成长值 | 理论总值 |
|------|----------------|-----------|---------|
| 初章 | `9 x 3 = 27` | `16` | **43** |
| 贰章 | `9 x 7 = 63` | `36` | **99** |
| 叁章 | `9 x 10 = 90` | `65` | **155** |
| 肆章 | `9 x 16 = 144` | `100` | **244** |

### 4.4 实装门槛

仙缘宝箱是**一次性高价值隐藏奖励**，门槛采用加压设计，从第一章略高于理论值起步，逐章拉大差距，要求玩家专项堆属性才能开启。

| 章节 | 理论值 | 实装门槛 | 加压幅度 |
|------|-------|---------|---------|
| 第一章 | 43 | **50** | +16% |
| 第二章 | 99 | **100** | +1% |
| 第三章 | 155 | **200** | +29% |
| 第四章 | 244 | **400** | +64% |

**实际应用方式**：
- 第一章 4 个箱子分别要求：根骨 >= 50、福源 >= 50、悟性 >= 50、体魄 >= 50
- 第二章 4 个箱子分别要求：根骨 >= 100、福源 >= 100、悟性 >= 100、体魄 >= 100
- 第三章 4 个箱子分别要求：根骨 >= 200、福源 >= 200、悟性 >= 200、体魄 >= 200
- 第四章 4 个箱子分别要求：根骨 >= 400、福源 >= 400、悟性 >= 400、体魄 >= 400

### 4.5 设计含义

- 第一章门槛略高于理论毕业值（+16%），需少量额外堆属性但不至于太难
- 第二章与理论值基本持平，作为过渡
- 第三、四章大幅高于理论值，明确要求玩家专项投入（丹药、问心、法宝等多维度补足）
- 对应职业在本章天然最容易打开自己的属性箱
- 其他 3 个箱子不能靠顺手路过开启，必须专门堆属性
- 玩家若走问心双修、丹药成长、法宝/神器补足，可跨职业开更多箱
- 第四章 400 门槛（理论值 244 的 1.64 倍）定位为长线追求目标

## 5. 固定藏宝室设计

### 5.1 点位原则

- 每章 4 个宝箱分散放置在不同后段区域
- 每个宝箱单独配一间小型藏宝室，不共用房间
- 入口可达，但最后一段通路宽度最多 1 格
- 进入藏宝室后应立即看到宝箱，不做二次迷宫

### 5.2 藏宝室模板

推荐统一采用"**5x5 房间 + 1 格入口**"模板：

```text
#####
#===#
#=C=#
#===#
##.##
```

图例：
- `C`：宝箱，占 1 格
- `=`：宝箱周围 3x3 环境区
- `.`：入口过渡格
- `#`：墙体/不可行走装饰

**硬规则**：
- `C` 的 3x3 区域内不刷怪、不放采集物、不放机关伤害
- 5x5 房间已经是满足"宝箱 1 格 + 环境 3x3"后的最小稳定模板
- 入口走廊宽度固定 1 格，长度建议 4~6 格
- 藏宝室只保留 1 个入口

## 6. 封印状态与交互表现

### 6.1 三个状态

| 状态 | 视觉表现 | 交互反馈 |
|------|---------|---------|
| **封印中** | 宝箱被锁链、符纸、阵纹包裹；地面 3x3 区域有对应属性阵纹 | 点击后显示对应属性要求面板 |
| **解封中** | 3 秒读条，锁链逐段崩裂，阵纹向内收缩 | 复用 `FortuneFruitSystem` 采集读条 UI |
| **已解封** | 宝箱盖半开、锁链断裂、阵纹熄灭 | 提示"此箱机缘已尽" |

### 6.2 属性区分表现

| 类型 | 地面阵纹 | 封印色调 |
|------|---------|---------|
| 根骨箱 | 方印 / 山纹 | 土黄、青铜 |
| 福源箱 | 云纹 / 钱纹 | 青绿、金色 |
| 悟性箱 | 剑纹 / 星纹 | 冷白、靛蓝 |
| 体魄箱 | 血纹 / 岩纹 | 暗红、黑铁 |

### 6.3 读条规则

- 读条时长：**3 秒**（复用 `FortuneFruitSystem.CHANNEL_DURATION`）
- 读条样式：复用 `FortuneFruitSystem` 采集读条组件 + `GetChannelProgress()` 进度
- 交互半径：**1.5 瓦片**（复用 `FortuneFruitSystem.INTERACT_RANGE`）
- 中断条件：
  - 玩家移动
  - 玩家受击
  - 玩家离开 1.5 瓦片交互半径
  - 玩家主动取消
  - 断线/网络中断
- 中断后不扣次数，不写入开启状态，**需重新交互**

### 6.4 交互面板

使用 `urhox-libs/UI` 组件库实现（UI.Panel + UI.Label + UI.Button）。

### 7.4 装备生成

在 `LootSystem.lua` 中新增专用生成函数：

```lua
--- 为仙缘宝箱生成 3 件候选装备
---@param tier number 固定 tier
---@param quality string 固定品质常量（"purple"/"orange"/"cyan"）
---@param guaranteedSubStat string 保底仙缘副属性（"constitution"/"fortune"/"wisdom"/"physique"）
---@return table[] items 3 件装备
function LootSystem.GenerateXianyuanReward(tier, quality, guaranteedSubStat)
    -- 1. 从 XIANYUAN_CHEST_SLOT_POOL 中随机抽 3 个不重复槽位
    -- 2. 对每个槽位调用已有的装备生成逻辑（固定 tier + quality）
    -- 3. 在副属性列表中强制注入 1 条 guaranteedSubStat
    --    - 值按现有线性公式：floor(tier * GameConfig.QUALITY_MULTIPLIER[quality])
    --    - 若随机生成的副属性已包含该属性，则合并（不重复出现）
    -- 4. 返回 3 件装备 table
end
```

### 7.5 服务端校验

| 需求 | 校验方式 | 说明 |
|------|---------|------|
| chestId 合法 | `XianyuanChestConfig.CHESTS[chestId]` | 配置中不存在则拒绝 |
| 是否已开启 | `openedXianyuanChests[chestId]` | 已开则拒绝 |
| 属性门槛 | 按 `cfg.attr` 调用 `Player[ATTR_GETTERS[cfg.attr]]()` | 只检查该箱对应属性 |
| 位置合法性 | 玩家与宝箱距离 <= 1.5 瓦片 | 防止远程伪造开启 |
| 读条完成合法性 | 校验 `openingChestId` 一致 | 防止跳过读条 |

**`openingChestId` 生命周期**（运行时内存状态，不持久化）：

| 事件 | 操作 |
|------|------|
| `StartOpen` 校验通过 | 设置 `openingChestId = chestId` |
| `CompleteOpen` 处理完成 | 清除 `openingChestId = nil` |
| `CancelOpen` 收到 | 清除 `openingChestId = nil` |
| 断线/会话结束 | 自动清除（内存回收） |

### 7.6 协议设计

| 事件名 | 方向 | 参数 | 说明 |
|--------|------|------|------|
| `C2S_XianyuanChest_StartOpen` | C->S | `{ chestId }` | 请求开始解封 |
| `S2C_XianyuanChest_StartResult` | S->C | `{ ok, reason?, attr, req, current }` | 返回校验结果 |
| `C2S_XianyuanChest_CancelOpen` | C->S | `{ chestId }` | 取消读条 |
| `C2S_XianyuanChest_CompleteOpen` | C->S | `{ chestId }` | 客户端完成 3 秒读条 |
| `S2C_XianyuanChest_Reward` | S->C | `{ chestId, items[3] }` | 下发 3 选 1 奖励 |
| `C2S_XianyuanChest_Pick` | C->S | `{ chestId, pickIndex }` | 选择奖励（1/2/3） |
| `S2C_XianyuanChest_PickResult` | S->C | `{ ok, equipment }` | 发奖结果 |

**注**：无需断线恢复协议——断线视为交互失败，客户端重新走 `StartOpen` 流程即可。

### 7.7 服务端流程

```text
C2S_XianyuanChest_StartOpen
  +-- 检查 chestId 在 XianyuanChestConfig.CHESTS 中存在
  +-- 检查 openedXianyuanChests[chestId] ~= true
  +-- 检查玩家与宝箱距离 <= 1.5 瓦片
  +-- 读取 cfg.attr / cfg.req
  +-- 按 cfg.attr 调对应 getter，检查 current >= cfg.req
  +-- 达标 -> 标记 openingChestId = chestId -> S2C ok=true
  +-- 未达标 -> S2C ok=false, reason="属性不足", attr, req, current

客户端开始 3 秒读条（复用 FortuneFruitSystem 读条逻辑）

C2S_XianyuanChest_CancelOpen
  +-- 清除 openingChestId
  +-- 无返回（静默处理）

C2S_XianyuanChest_CompleteOpen
  +-- 验证 openingChestId == chestId
  +-- 再次验证距离 + 未开启状态
  +-- 调用 LootSystem.GenerateXianyuanReward(cfg.tier, cfg.quality, cfg.attr)
  +-- 持久化 openedXianyuanChests[chestId] = true
  +-- S2C_XianyuanChest_Reward { chestId, items }

C2S_XianyuanChest_Pick
  +-- 验证 pickIndex 在 1~3 范围内
  +-- 验证该 chestId 的奖励列表仍在内存中（断线则已丢失，返回 ok=false）
  +-- 发放选中的 1 件装备到背包
  +-- 清空内存中的奖励列表
  +-- S2C_XianyuanChest_PickResult { ok=true, equipment }
```

## 11. 结论

仙缘宝箱是"**固定点、一次性、高门槛、高价值**"的探索系统。

定稿规格：

- **四章各 4 个，共 16 个**
- **每章对应根骨 / 福源 / 悟性 / 体魄各 1 个**
- **门槛按加压设计：`50 / 100 / 200 / 400`**
- **品质常量：`purple / orange / orange / cyan`**
- **奖励档位：`T3 / T5 / T7 / T9`**
- **每件候选装备保底带 1 条对应仙缘副属性**
- **槽位池 8 个（排除 weapon + 全部 SPECIAL_SLOTS）**
- **交互半径 1.5 瓦片，读条 3 秒，断线即失败**
- **存档升级至 v21，新增 `openedXianyuanChests` 字段**
- **UI 使用 `urhox-libs/UI` 组件库，3 选 1 复用 `QuestRewardUI`**
- **藏宝室坚持 1 格入口 + 3x3 宝箱环境区 + 明确封印/解封状态**

# 玩家移动与攻击动作方案

> 初版日期: 2026-05-07
> 优化日期: 2026-05-07
> 目标: 在不依赖 AI 连续帧的前提下，为玩家实现稳定、可读、可维护的基础移动与攻击动作
> 适用范围: 仅玩家角色
> 不包含: 怪物动作、怪物受击、宠物动作、技能特效重做
> 实施策略: 分两步交付——Step 1（上下身分离 + smear），Step 2（前臂独立层）

---

## 一、背景与结论

当前项目里，玩家角色使用单张精灵贴图绘制，怪物使用圆形头像图标。玩家如果直接采用“整张人物图上下摆动”或“整个人向前冲”的方案，容易出现以下问题：

- 重心漂浮，脚底接地感不成立
- 朝向、透视、肢体体块一起失真，看起来像贴纸在滑
- 自动攻击频率较高时，整图前冲会与移动叠加，观感更差
- 同一套整图变换无法同时兼顾移动和攻击

因此，本方案的结论是：

- 不再对整张玩家精灵做动作
- 改为“分层角色动画”
- 移动只驱动下半身
- 攻击只驱动上半身与前臂武器
- 角色根节点不做攻击前冲

这是一条专门针对本项目当前结构的可实施路线，目标不是“做更复杂的动画”，而是先做出可靠的基础动作系统。

---

## 二、目标与非目标

### 2.1 目标

- 为玩家提供基础移动动作
- 为玩家提供基础普攻动作
- 允许“移动中攻击”和“站立攻击”同时成立
- 与现有自动战斗、现有刀光特效、现有左右翻转兼容
- 资源量尽可能小，不依赖 AI 连续出帧

### 2.2 非目标

- 不解决怪物动作表现
- 不重构战斗公式、命中逻辑、自动索敌逻辑
- 不做八方向角色动画
- 不在本阶段引入骨骼动画系统或第三方动画中间件
- 不追求复杂连招、取消、受击硬直体系

---

## 三、现状与约束

### 3.1 现状

当前代码已有以下基础能力：

- 玩家移动状态与朝向更新: `scripts/entities/Player.lua`
- 普攻时机与目标选择: `scripts/systems/CombatSystem.lua`
- 玩家整图绘制: `scripts/rendering/EntityRenderer.lua`
- 刀光特效与浮字: `scripts/rendering/EffectRenderer.lua`

换言之，当前缺的不是战斗逻辑，而是“玩家本体的动作表现层”。

### 3.2 约束

- 玩家目前以右朝向贴图为基础，左朝向通过水平翻转实现
- 项目现有实现是整图绘制，不支持天然的分层动画
- 自动战斗下，普攻会频繁触发，攻击动作必须足够短
- 动作系统不能破坏现有碰撞、寻路、战斗判定

## 四、总体方案

### 4.1 核心思路

玩家渲染拆成两个独立通道：

- 下半身通道: 负责 `idle / move`
- 上半身通道: 负责 `idle / attack`

这两个通道共享同一个角色根节点，但互不抢控制权。

### 4.2 根节点原则

- 角色世界坐标始终以现有 `player.x / player.y` 为准
- 动画不改碰撞盒，不改寻路结果
- 动画不推动人物根节点前移
- 所有攻击“前压感”只允许发生在上半身局部偏移中

### 4.3 通道职责

#### 下半身

- 待机时显示站姿
- 移动时只在步态帧之间切换
- 不参与攻击挥砍动作

#### 上半身

- 待机时显示静止上身
- 攻击时进入短时攻击姿态
- 可在移动中独立播放攻击动作

#### 前臂与武器

- 作为独立层处理
- 移动时保持待机
- 攻击时执行主要挥动

## 五、资源方案

### 5.1 每职业最小资源集

资源按实施步骤分批，**Step 1 只需 4 张 + 1 张共享**：

#### Step 1 资源（上下身分离 + smear）

每职业 4 张，全部为右朝向，左朝向继续镜像：

- `lower_idle.png`
- `lower_step_a.png`
- `lower_step_b.png`
- `upper_idle.png` — Step 1 中上半身仅此一张，攻击靠 rotation + offset 驱动

共享资源 1 张：

- `weapon_smear.png`

#### Step 2 追加资源（前臂独立层）

每职业追加 2 张：

- `front_arm_idle.png`
- `front_arm_attack.png`

> **Step 1 交付标准：** 上下身分离渲染 + 移动步态 + 攻击 upperRotation + smear 均可工作。
> **Step 2 交付标准：** 前臂独立层接入后，攻击挥动由前臂旋转驱动而非仅靠上身倾斜。

### 5.2 资源命名建议

建议使用按职业分目录的方式，便于扩展：

```text

Textures/player/taixu/lower_step_b.png

Textures/player/zhenyue/lower_step_a.png
Textures/player/zhenyue/lower_step_b.png
Textures/player/zhenyue/upper_idle.png
Textures/player/zhenyue/front_arm_idle.png

```

### 5.3 资源制作规则

- 所有 PNG 画布尺寸必须一致
- 所有 PNG 锚点必须一致
- 所有 PNG 脚底线必须一致
- `lower_*` 与 `upper_*` 在腰部必须有 6 到 10 像素重叠，避免出现分层缝
- `front_arm_*` 只保留前臂和武器，旋转支点按肩部对齐

### 5.4 美术制作方式

本方案不要求 AI 做连续走路帧。推荐方式：

- 从现有整张职业图手工切层
- `lower_step_a / lower_step_b` 只修改腿部与下摆，不动上身
- `front_arm_attack` 只重画前臂和武器挥动姿态
- `weapon_smear` 做成共用资源，避免每职业单独画特效

结论是：手工小改图层，远比让 AI 连续出走路和攻击帧更稳。

## 六、运行时状态设计

### 6.1 Player 视觉状态

建议在 `Player` 上新增一个仅服务于表现层的字段：

```lua
self.visual = {
    movePhase = 0,
    hitStop = 0,
    attack = nil,
    lastX = x,
    lastY = y,
}
```

字段说明：

- `movePhase`: 下半身步态相位
- `hitStop`: 视觉停顿剩余时间
- `attack`: 当前攻击动画状态，空表示未攻击
- `lastX / lastY`: 上一帧位置，用于计算真实位移

### 6.2 Attack 状态结构

建议结构如下：

```lua
attack = {
    t = 0,
    duration = 0.18,
    phase = "ready",
    angle = 0,
    facingRight = true,
}
```

说明：

- `t`: 已播放时间
- `duration`: 当前攻击总时长
- `phase`: `ready / release / recover`
- `angle`: 攻击方向角，用于 smear 或局部偏移
- `facingRight`: 攻击开始时锁定朝向

## 七、移动动作方案

### 7.1 基本原则

移动动作不再做整身摆动，而是：

- 根节点不动
- 下半身切步态
- 上半身保持稳定
- 前臂保持待机

### 7.2 触发条件

移动状态由“真实位移”决定，而不是输入值决定。

判断建议：

- 用本帧 `player.x / player.y` 与 `lastX / lastY` 的差值计算实际移动距离
- 距离大于阈值则认为在移动

原因：

- 真实位移更可靠
- 可避免撞墙、减速、停顿时动作与实际位移不同步

### 7.3 步态驱动

建议：

- `movePhase += dt * 7.0 * speedRatio`
- `speedRatio = currentSpeed / baseSpeed`
- `speedRatio` 限制在 `0.85 ~ 1.20`

步态切换规则：

- 静止: `lower_idle`
- 移动且 `movePhase < 0.5`: `lower_step_a`
- 移动且 `movePhase >= 0.5`: `lower_step_b`

`movePhase` 循环取模 `1.0`。

### 7.4 禁止项

本阶段明确禁止以下做法：

- 整张人物图上下 bob
- 整张人物图整体缩放
- 整张人物图整体前后摇摆
- 移动时整个身体显著倾斜

这些做法在单张整身 sprite 上风险很高，容易直接破坏观感。

## 八、攻击动作方案

### 8.1 基本原则

攻击动作采用“定根攻击”。

- 根节点不前冲
- 下半身继续执行当前移动或站立状态
- 上半身与前臂单独播放攻击
- 攻击感来自姿态、前臂挥动、smear、hit stop

### 8.2 事件设计

**复用现有事件，不新增事件类型。** 当前代码已有两个事件在普攻命中时同步触发（`CombatSystem.lua:727-728`）：

- `player_attack(player, monster, damage)` — 已有，携带伤害值
- `player_normal_hit(player, monster)` — 已有，SkillSystem 已监听

PlayerAnimator 只需监听这两个事件即可驱动表现层：

- 监听 `player_attack` → 启动攻击视觉状态（从 `player` 读取 `attackInterval`，从 `monster` 计算 `angle`）
- 监听 `player_normal_hit` → 触发 hit stop

**不新增 `player_attack_start` 和 `player_attack_hit`。** 原因：
1. 现有事件的触发时机已覆盖需求
2. 减少事件类型可降低 EventBus 开销和维护成本
3. `player_attack` 在伤害结算后触发，攻击方向和目标此时已确定，适合驱动表现

> 注意：如果未来需要"预备动作"（在命中前就开始挥动），可以在 CombatSystem 的 AutoAttack 判定命中前加一个 `player_attack_windup` 事件。但第一版不需要——现有攻击间隔足够短，命中后再启动视觉状态在体感上无延迟。

### 8.3 动画时长

建议总时长：

```lua
attackDur = clamp(attackInterval * 0.40, 0.14, 0.24)
```

理由：

- 攻击动画必须明显短于普攻间隔
- 高频普攻时不应出现前一个动作还没播完，下一个动作已覆盖的情况

### 8.4 三段时间轴

按总时长切分三段：

- 蓄力: 25%
- 释放: 20%
- 收招: 55%

对应含义：

- 蓄力: 上半身轻微后仰，前臂后拉
- 释放: 上半身前压，前臂挥出，显示 smear
- 收招: 平滑回到待机

### 8.5 姿态参数建议

#### Step 1 参数（upperRotation + smear）

- 蓄力:
  - `upperRotation = -4°`

- 释放:
  - `upperRotation = +6°`
  - `upperOffsetX = +2px ~ +4px`
  - `smearAlpha = 1`

- 收招:
  - 全部参数缓动回到 0

#### Step 2 追加参数（前臂层）

- 蓄力:
  - `armOffsetX = -8px ~ -10px`
  - `armRotation = -12°`

- 释放:
  - `armOffsetX = +12px ~ +16px`
  - `armRotation = +24° ~ +30°`

- 收招:
  - arm 参数缓动回到 0

注意：

- 上半身前压只允许很小的像素偏移
- 脚底绝不能因为攻击而离开当前落点
- Step 1 阶段不使用任何 arm 参数，攻击感完全由 upperRotation + smear 承担

### 8.6 移动中攻击

本方案允许移动中攻击，并采取如下规则：

- 下半身继续步态切换
- 上半身独立进入攻击状态
- 前臂独立挥动

换言之，移动与攻击的混合方式不是“整个人又走又冲”，而是“腿继续走，上身出手”。

### 8.7 朝向规则

建议规则如下：

- 移动中: 朝向仍以 `dx` 为主
- 站立攻击时: 朝向以目标 `target.x - player.x` 为主
- 攻击开始后锁定朝向到本次攻击结束

这样可避免角色站定时朝向不更新，导致攻击方向错误。

## 九、命中停顿方案

### 9.1 目标

为了增强普攻成立感，需要加入一个极短的视觉停顿。

### 9.2 规则

- `hitStop = 0.03 ~ 0.04s`
- 只冻结表现层时间推进
- 不冻结战斗逻辑
- 不改伤害计算和目标选择

### 9.3 冻结内容

建议冻结：

- 上半身攻击姿态时间推进
- 前臂挥动时间推进
- smear 保持在命中峰值

不冻结：

- 玩家世界坐标更新
- 怪物逻辑
- 战斗结算

这样能提升打击感，又不引入逻辑风险。

## 十、渲染实现方案

### 10.1 绘制顺序

#### Step 1 绘制顺序

1. 下半身
2. 上半身（含 upperRotation / upperOffset）
3. smear
4. 血条、名称等 UI

#### Step 2 绘制顺序（接入前臂层后）

1. 下半身
2. 上半身
3. 前臂/武器
4. smear
5. 血条、名称等 UI

### 10.2 左右翻转

左右翻转仍然对整组层统一处理：

- 先确定角色朝向
- 若为左朝向，则对分层绘制整体镜像
- 单层内部不单独做左右资源

### 10.3 渲染辅助函数

建议在渲染工具层新增一个带 pivot 的图片绘制函数，支持：

- 图片句柄
- 绘制位置
- 宽高
- 旋转角
- pivot 点
- 水平翻转
- 透明度

原因：

- 前臂挥动必须绕肩部转
- smear 显示也需要独立旋转和透明度控制

## 十二、实施顺序

分两步交付，Step 1 交付完成并验收后再推进 Step 2。

### Step 1：上下身分离 + upperRotation + smear

1. **切出一个职业的 4 张分层资源**（lower_idle, lower_step_a, lower_step_b, upper_idle）+ 共享 weapon_smear
2. **⚠️ 立即做腰部拼接验证**——写一个最小渲染测试：只在 EntityRenderer 中将 upper + lower 拼在一起绘制，确认腰部无缝、锚点对齐、左右翻转正确。**此步骤不通过则停下来调资源，不要继续往下走。**
3. 在 `Player` 上挂载 `visual` 状态
4. 新建 `PlayerAnimator` 模块（监听现有 `player_attack` / `player_normal_hit` 事件，不新增事件）
5. 改写 `EntityRenderer.RenderPlayer` 为分层绘制（Step 1 只绘制 lower + upper + smear，无前臂层）
6. 调通移动步态（下半身切帧）
7. 调通站立攻击（upperRotation + upperOffset + smear）
8. 调通移动中攻击混合
9. **Step 1 验收**——对照 §13 验收标准逐项检查

### Step 2：前臂独立层

10. 追加一个职业的 2 张前臂资源（front_arm_idle, front_arm_attack）
11. 在 PlayerAnimator 中增加前臂层变换输出（armRotation, armOffset）
12. 在 EntityRenderer 中增加前臂绘制层
13. 调通前臂挥动 + 攻击混合
14. **Step 2 验收**
15. 接入第二个职业资源

**关键原则：**
- 不要一开始同时改两个职业和所有细节，否则调试面过大
- 资源切出后**立即验证拼接**，不要等代码写完再发现锚点不对
- Step 1 已经可以独立上线（upperRotation + smear 的攻击表现优于整图前冲）

## 十三、验收标准

动作系统完成后，至少应满足以下标准：

- 持续移动时，玩家不再出现明显上下漂浮
- 站立普攻时，玩家不再整个人向前冲
- 攻击时脚底落点稳定，碰撞位置不变
- 移动中普攻时，腿部和上身动作可同时成立
- 高频攻击时，动作不会与下一次攻击互相覆盖导致抖动
- 左右朝向翻转后，动作表现不出错
- 腰部拼接处无明显分层缝

## 十四、风险与回退策略

### 14.1 主要风险

- 腰部拼接线明显
- 前臂 pivot 点不准，挥动不自然
- 资源锚点不一致，导致攻击时层错位
- 攻击频率较快时，表现层状态被频繁重置

### 14.2 回退策略

如果前臂独立层短期难以调顺，可采用临时回退：

- 保留分层结构
- 暂时只做下半身移动步态
- 上半身攻击先只做 `upperRotation + smear`
- 前臂攻击层后续再接入

这个回退方案仍然优于整图摆动和整图前冲。

## 十五、阶段性结论

本方案不是“继续优化整张人物图变换”，而是直接引入一个轻量、稳定、与当前项目兼容的玩家分层动作系统。

它的优点是：

- 只解决玩家，不扩大问题范围
- 不依赖 AI 连续帧
- 资源量小
- 不碰现有战斗判定核心
- 能同时解决“移动难看”和“攻击前冲难看”

对当前项目来说，这是最稳的第一步。

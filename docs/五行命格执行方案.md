## 一、执行目标

本执行方案的目标不是重新设计玩法，而是把已经定稿的“五行命格”落到当前项目代码里，并保证：

- 不改动主背包和常规装备的现有逻辑
- 命格有独立背包、独立装备位、独立存档
- 命格掉落沿用现有掉落系统和地面显示系统
- 命格属性最终真实进入玩家战斗属性
- 四象套装可按五行独立触发，并允许同名套装叠加

## 三、执行总原则

### 3.1 只做增量，不推翻现有系统

本期不做以下事情：

- 不重写 `InventorySystem`
- 不把主背包和命格背包抽成通用多背包框架
- 不重写 `LootSystem` 的整体掉落模型
- 不重写 `SaveSerializer` 的整套物品存档格式
- 不把命格硬塞进主背包或主装备栏

### 3.2 允许新增独立模块，但要贴着现有写法做

本期允许新增：

- `config/MinggeData.lua`
- `systems/MinggeSystem.lua`
- `ui/MinggePage.lua`
- `ui/MinggeTooltip.lua`

但新增模块必须遵循当前工程的写法：

- 数据管理沿用 `InventorySystem` 的单例式写法
- 背包/装备容器沿用 `UI.InventoryManager`
- UI 继续挂在 `InventoryUI` 的总入口下
- 存档继续经由 `SaveSerializer`

### 3.3 命格属性不能直接写回 equip 字段

不能让命格系统直接覆盖或混写 `player.equipAtk / equipDef / equipHp ...`，原因很直接：

- `InventorySystem.RecalcEquipStats()` 会完整重算并覆盖这些字段
- 如果命格也写进 `equip*`，就会出现先后顺序依赖和互相覆盖问题

所以本期执行口径固定为：

- 装备写 `equip*`
- 命格写 `mingge*`
- `Player` 总属性 getter 统一把两边加总

这是当前代码下最小、最稳的做法。

因为当前 [InventorySystem.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/systems/InventorySystem.lua) 的 manager 只承载：

- 独立装备栏
- 独立批量出售

所以最稳的做法是：

- 保留现有 `InventorySystem` 不动
- 新建 `MinggeSystem`
- `MinggeSystem` 内部再创建一个 `UI.InventoryManager`

这不是发明新架构，而是按当前 `InventorySystem` 的写法平移一份命格版本。

### 同命格唯一的实现口径

当前代码里，`UI.InventoryManager` 和现有装备系统只做“类型匹配”，不负责“同 ID 去重”。

所以命格的“同命格 ID 不可重复装备”必须由 `MinggeSystem` 负责。执行口径如下：

- 提供 `IsDuplicateEquipped(minggeId)`
- 在命格装备动作发生前先校验
- 如果重复，则阻止装备并弹提示

不要把这条规则寄希望于底层 `InventoryManager` 自动处理。

### 掉表接入方式

当前 `LootSystem.GenerateDrops()` 已经按 `entry.type` 分支处理多种掉落。

本期直接新增一个新分支：

```lua
entry.type == "mingge"
```

因为命格有自己独立的：

- 背包
- 售价
- 图标

### 命格掉落合同

命格掉落这次要按固定顺序生成，不允许把“套装 20%”做成全品质通用判定。执行顺序固定为：

1. 先判是否掉本命命格，概率 `5%`
2. 若掉落成功，再按品质权重判定：
   - `紫 50%`
   - `橙 30%`
   - `第3档品质 20%`
3. 只有第 3 档品质命格，才继续判定是否附带套装，概率 `20%`
4. 若命中套装，再在四象中按等权选 1 个：
   - `青龙 25%`
   - `白虎 25%`
   - `朱雀 25%`
   - `玄武 25%`

按当前口径，对单次击杀的最终结果，命格掉落分布应为：

- 紫色命格：`2.5%`
- 橙色命格：`1.5%`
- 第 3 档无套命格：`0.8%`
- 第 3 档套装命格：`0.2%`

如果要算指定某一四象套装命格，单次击杀概率应为：

- `5% × 20% × 20% × 25% = 0.05%`

实现时建议把这套合同写成 `MinggeData.DROP_RULES` 的静态配置，不要把权重散落在 `LootSystem.GenerateDrops()` 的分支里硬编码。

### GenerateDrops 返回结构扩展

当前返回：

```lua
{ items, gold, exp, materials, consumables, lingYun }
```

本期扩展为：

```lua
{ items, mingges, gold, exp, materials, consumables, lingYun }
```

### 地面掉落对象

当前 `GameEvents.lua` 会把：

- 气泡掉落
- 装备掉落

拆成两个 `GameState.AddLootDrop()`。

本期新增第三类：

```lua
GameState.AddLootDrop({
    x = monster.x,
    y = monster.y,
    mingges = dropResult.mingges,
})
```

不要把命格塞进 `drop.items`，原因有三个：

1. `drop.items` 当前默认进主背包
2. 宠物当前只对 `drop.items` 做“装备不捡”的硬过滤
3. 渲染和拾取分支都会误把命格当装备处理

### 玩家拾取

当前 `LootSystem.AutoPickup()` 已经有完整的磁吸与拾取流程。

本期不新增“手动点击地上命格”的新交互，执行口径直接定为：

- 命格沿用现有装备的靠近自动吸附与自动拾取逻辑

也就是：

- 玩家靠近命格后自动磁吸
- 靠近后自动进入命格背包

这是当前代码基线下最自然的实现。

### 背包满处理

当命格背包已满（60/60）时，拾取行为：

| 策略 | 说明 |
|------|------|
| **推荐**：地面不消失 + 飘字提示 | `MinggeSystem.AddItem()` 返回 false → 拾取逻辑保留地面对象 → 显示"命格背包已满" |
| 备选：自动出售最低品质 | 复杂度高，首发不做 |

实现要点：
- `MinggeSystem.AddItem()` 必须返回 bool 表示是否入包成功
- `LootSystem.AutoPickup()` 命格分支检查返回值，失败则不移除地面对象
- 每 5 秒限频飘字提示，避免刷屏

### 宠物拾取

当前宠物会主动跳过 `drop.items`。

本期需要同步补上：

- `drop.mingges` 一律跳过

修改点：

- `LootSystem.FindPetPickable()`

目标行为：

- 宠物不拾取命格

同时保留天诛相关命格字段：

```lua
self.minggeTianzhuChance = 0
self.minggeTianzhuDamage = 0
```

### 为什么要新增 mingge 字段

因为当前所有总属性 getter 都是基于 `equip*` 做加总的。

命格如果不走平行字段，就只能：

- 直接污染 `equip*`
- 或者在 `InventorySystem.RecalcEquipStats()` 里顺带结算命格

这两种都不稳。

所以本期必须改为：

- 装备系统只算装备
- 命格系统只算命格
- `Player` 总属性 getter 统一汇总

### 需要补的 getter / 结算点（两类修改模式）

当前 `Player.lua` 的总属性 getter 分为**两类实现方式**，修改模式不同：

#### A 类：缓存式 getter（走 `_RecalcStatsCache()`）

这些 getter 通过 `_RecalcStatsCache()` 内部计算并写入 `self._cached*` 字段，每帧只算一次。

修改方式：在 `_RecalcStatsCache()` 对应累加行末尾追加 `+ (self.mingge* or 0)`。

| getter | 累加位置 | 新增字段 |
|--------|---------|---------|
| `GetTotalAtk()` | `base = self.atk + self.equipAtk + ...` | `+ (self.minggeAtk or 0)` |
| `GetTotalDef()` | `defBase = self.def + self.equipDef + ...` | `+ (self.minggeDef or 0)` |
| `GetTotalMaxHp()` | `hpBase = self.maxHp + self.equipHp + ...` | `+ (self.minggeHp or 0)` |
| `GetTotalConstitution()` | `self._cachedTotalConstitution = ...` | `+ (self.minggeConstitution or 0)` |
| `GetTotalPhysique()` | `self._cachedTotalPhysique = ...` | `+ (self.minggePhysique or 0)` |
| `GetTotalFortune()` | `self._cachedTotalFortune = ...` | `+ (self.minggeFortune or 0)` |
| `GetTotalWisdom()` | `self._cachedTotalWisdom = ...` | `+ (self.minggeWisdom or 0)` |

⚠️ **注意依赖顺序**：`_RecalcStatsCache()` 内部有先后依赖（Constitution 先于 Def，Physique 先于 MaxHp）。命格字段只需追加在末尾，不影响现有依赖链。

#### B 类：内联式 getter（直接计算，无缓存）

这些 getter **不经过** `_RecalcStatsCache()`，而是每次调用直接 return 累加。

修改方式：在对应 getter 函数的 return 表达式末尾追加 `+ (self.mingge* or 0)`。

| getter | 当前实现 | 新增字段 |
|--------|---------|---------|
| `GetTotalCritRate()` | `return 0.05 + (self.equipCritRate or 0) + ...` | `+ (self.minggeCritRate or 0)` |
| `GetTotalCritDmg()` | `return 1.5 + (self.equipCritDmg or 0) + ...` | `+ (self.minggeCritDmg or 0)` |
| `GetTotalHeavyHit()` | `return (self.equipHeavyHit or 0) + ...` | `+ (self.minggeHeavyHit or 0)` |
| `GetTotalTianzhuChance()` | `return self.equipTianzhuChance or 0` | `+ (self.minggeTianzhuChance or 0)` |
| `GetTotalTianzhuDmg()` | `return 1.5 + (self.equipTianzhuDamage or 0)` | `+ (self.minggeTianzhuDamage or 0)` |

#### C 类：其他结算点

| 计算位置 | 新增字段 |
|---------|---------|
| 每秒生命回复（HpRegen） | `+ (self.minggeHpRegen or 0)` |
| 玩家移动速度 | `+ (self.minggeSpeed or 0)` |
| `GetFortuneSyncBonus()` | `+ (self.minggePetSyncRate or 0)` |
| 击杀回血 | `+ (self.minggeKillHeal or 0)` |

### MinggeSystem.RecalcStats() 触发时机

`MinggeSystem.RecalcStats()` 负责遍历已装备命格 + 套装效果，写回 `player.mingge*` 全量字段。必须在以下时机调用：

| 时机 | 位置 | 原因 |
|------|------|------|
| 装备/卸下命格 | `MinggeSystem.Equip()` / `Unequip()` | 装备变动 |
| 存档加载完成 | `MinggeSystem.DeserializeAndApply()` | 还原属性 |
| 副本进入 | `DungeonClient.EnterDungeon()` 末尾 | 确保服务端校验基准一致 |
| 批量出售后 | `MinggeSystem.SellByQuality()` 末尾 | 已装备的命格不卖，但防止逻辑遗漏 |

对称参考：`InventorySystem.RecalcEquipStats()` 在 `Equip/Unequip/DeserializeInventory/SortBackpack` 后均被调用。

### 缓存失效

`_RecalcStatsCache()` 使用 `_statsCacheFrame == _frameCounter` 做帧级缓存。由于 `mingge*` 字段已直接写在 Player 上，`_RecalcStatsCache()` 下一帧自动重读新值，**无需手动 invalidate**。

### 套装效果接入

四象套装效果建议直接在 `MinggeSystem.RecalcStats()` 中汇总成附加值，然后写回：

- `minggeSpeed`
- `minggeCritDmg`
- `minggeTianzhuDamage`
- `minggePetSyncRate`

不要把命格套装写进 `EquipmentData.SetBonuses`。

## 十、风险评估

### 10.1 存档风险：🟢 低

| 维度 | 评估 |
|------|------|
| 数据隔离 | `saveData.mingge` 完全独立块，不修改现有 `inventory` / `equipped` 字段 |
| 旧存档兼容 | `mingge == nil` → 补空结构，零迁移成本 |
| 存档膨胀 | 满装满包极限 75 个 item（15 装备 + 60 背包），每个约 120B，增量 ~9KB |
| 回滚安全 | 若回滚到无命格版本，`saveData.mingge` 被忽略不会报错 |

**注意项**：`SavePersistence.lua` 的 DTO 合同测试需新增 mingge 字段断言。

### 10.2 线上回归风险：🟡 中低

| 风险点 | 影响面 | 防护措施 |
|--------|--------|---------|
| `_RecalcStatsCache()` 改动 | 全局战力，影响所有玩家 | 修改只在累加链末尾 append，不改变现有字段顺序；上线前 A/B 对比验证 |
| 5 个内联 getter 改动 | CritRate/CritDmg/HeavyHit/Tianzhu | 同上，末尾 append，旧逻辑不变 |
| `LootSystem.GenerateDrops()` 新增分支 | 掉率 | 新分支仅在 `entry.type == "mingge"` 时命中，不影响现有掉落 |
| `InventoryUI` 切页 | UI 交互 | 新增 Tab 不修改现有 Tab 逻辑 |

**最大风险点**：`_RecalcStatsCache()` 是全局热路径，修改后必须做**全属性对比验证**——在不装备任何命格的情况下，前后版本的 GetTotal* 输出必须逐字段一致。

### 10.3 C/S 协议风险：🟡 中

| 维度 | 评估 |
|------|------|
| 协议透明性 | 命格属性通过 `player.mingge*` → getter 汇总 → 本地伤害计算 → `pendingDamageReport_` 上报，**不需要新增协议字段** |
| 伤害校验上限 | 当前 `MAX_DAMAGE_PER_REPORT = 50000`；命格满配后总战力提升约 10-20%，极端情况可能触碰上限 |
| 建议 | 上线初期将 `MAX_DAMAGE_PER_REPORT` 从 50000 提升到 **60000-65000**，或改为动态计算 `baseMaxDmg * (1 + minggeAtkBonus%)` |
| 作弊防护 | 客户端可伪造 `mingge*` 字段；但当前架构下装备属性同样信任客户端，风险等级一致，无需额外处理 |

**关键结论**：命格系统对 C/S 协议完全透明，唯一需要关注的是 `MAX_DAMAGE_PER_REPORT` 的硬编码上限在满配命格后是否够用。

## 十一、最终执行结论

基于当前代码，五行命格首发的最小可行落地方式是：

1. **不动现有装备系统主体**
2. **新增一套平行的 MinggeSystem**
3. **InventoryUI 只负责切页，不改成大重构**
4. **掉落系统只扩一个 `mingge` 分支，不重写**
5. **属性通过 `player.mingge*` 平行字段接入**
6. **存档通过 `saveData.mingge` 独立保存**

如果按这条执行，工作量是可控的，且不会把现有背包、装备、掉落、存档链条整体打散。

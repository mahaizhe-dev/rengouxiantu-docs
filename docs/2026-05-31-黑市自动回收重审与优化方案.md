# 2026-05-31 黑市自动回收重审与优化方案

> 2026-06-01 实施更新  
> 本文后半部分保留了 2026-05-31 重审时的“问题基线”和“改造方案”上下文；如果与当前代码口径冲突，以本次实施结果为准：
>
> 1. 每日自动回收已改为北京时间 `12:00~13:00` 主窗口调度，失败后仅按固定间隔补跑，不再按帧重试。
> 2. 自动回收与发版手动回收已拆分；新增 `bm_recycle_release`，按 `GameConfig.CODE_VERSION` 做“一版一次”幂等。
> 3. 执行器已收敛为“只处理满库存商品，且每轮每个商品只处理一次”。
> 4. 满库存商品的随机回收数量已从 `0/1/2` 改为 `1/2`，不再出现“随机全 0”。
> 5. 服务端新增了最近一次回收审计快照，后续排查以 `bm_recycle_audit` 和公共交易记录为主。
> 6. 方案补充：发版手动回收必须增加“先预占安全锁，再执行回收”的保护；否则一旦状态落库失败，就存在同版本重复回收的致命风险。

## 0.1 致命风险补充

这一条不是可选优化，而是方案里的强制安全约束。发版手动回收如果不补这层保护，就会存在“重复点一次，重复扣一次库存”的致命风险。推荐约束如下：

1. `bm_recycle_release` 不再是“回收成功后再写版本状态”，而是先写 `reserved` 安全锁，再执行实际回收。
2. 如果回收前置阶段失败，且代码能确认“尚未进入扣库存阶段”，状态会落到 `failed_safe`，允许同版本重试。
3. 如果回收已经执行，但最终确认写入失败，状态会保留在阻断态，不允许同版本再次触发。

仍需明确一个残留风险：当前代码里没有发现可用的云端 CAS / 分布式锁接口，所以“每日自动回收日期占坑”在多进程并发部署下，理论上仍存在并发推进同一天日期的结构性风险。这个问题不等于“无限回收”，但属于后续应继续治理的分布式一致性项。

## 1. 结论先行

以当前代码为准，黑市自动回收的核心执行链路没有“已经删除”或“完全失效”，但它现在有 4 个明显问题：

1. **自动回收仍然是“按自然日首个成功 tick 触发”**，不是固定时间窗口触发。  
   所以它天然可能漂到凌晨 `00:40` 以后，更不符合“中午 12 点~1 点执行”的目标。

2. **失败重试策略过于粗糙**。  
   `RecycleTick()` 在云端读取失败、日期占坑失败时会把 `lastRecycleDate_` 清空，下一帧继续重试。这样虽然提高了“最终成功率”，但也会把真正执行时刻拖到任意晚的成功时刻。

3. **自动回收和手动回收没有做“调度层分离”**。  
   现在只有一个执行器 `ExecuteRecycle()`，自动/手动都直接打到它。手动路径虽然独立于每日日期限制，但没有“版本发布后仅执行一次”的版本级幂等保护。

4. **可观测性不足**。  
   当前只持久化 `bm_recycle_date` 这一个“日期值”，没有记录“最后一次自动回收的真实执行时间、执行模式、成功/失败、满库存数、随机跳过数、失败商品数”。  
   结果就是：从日志和云端状态上很难区分“今天没触发”“触发了但随机全 0”“触发了但失败”“手动触发过”。

这也是你现在会感觉“只触发过几次，之后似乎不再生效”的核心原因之一：  
**当前实现缺少足够的状态与结果记录，外部观察到的只是库存没变。**

## 4. 当前代码的关键问题拆解

### 4.1 自动回收不是“固定时间”，而是“当天首个成功时刻”

当前 `RecycleTick()` 的判断是：

- 取 `os.date("%Y%m%d")`
- 只要“今天还没执行”
- 就立即尝试读取云端日期并占坑

这意味着它**没有中午窗口概念**，而是：

- 当天第一次成功通过所有条件时就执行

因此：

- 正常时可能在 `00:00` 后不久执行
- 如果 `money:Get` / `money:Add` 在凌晨失败，则可能拖到 `00:40`
- 如果服务端进程在凌晨后才启动，也可能在启动后才执行

这与“稳定在中午 12 点~1 点执行”是两套完全不同的策略。

### 4.2 失败后“下一帧重试”会把执行时间拖晚

当前失败路径会清掉 `lastRecycleDate_`：

- [scripts/network/BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua:1356)
  日期跳转失败时清空 `lastRecycleDate_`
- [scripts/network/BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua:1379)
  日期占坑失败时清空 `lastRecycleDate_`
- [scripts/network/BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua:1387)
  云端读取失败时清空 `lastRecycleDate_`

这样做的好处是：

- 同一天失败后还能继续重试，不会整天错过

但坏处也很明显：

- **没有重试节流**
- **没有固定窗口**
- **没有“最晚允许补跑到几点”的规则**

最终效果就是：

- 什么时候第一次成功，什么时候执行

这正是“会漂到 `00:40` 以后”的直接代码根因。

### 4.3 结果摘要不够准确，容易误判为“没有触发”

`ExecuteRecycle()` 现在只有 `tasks` 不为空才会进入执行。  
但 `tasks` 的构造是“满库存 + 随机数量大于 0”：

- [scripts/network/BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua:1231)
  遍历商品
- [scripts/network/BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua:1237)
  随机 0/1/2
- [scripts/network/BlackMerchantHandler.lua](/C:/Users/mahaizhe/Documents/New%20project/taptapmaker/rengouxiantu-docs/scripts/network/BlackMerchantHandler.lua:1246)
  `#tasks == 0` 时直接返回“无满库存商品，跳过收购”

这有一个语义问题：

- 可能确实“无满库存商品”
- 也可能“有满库存商品，但本轮这些商品随机全出 0”

这两种情况现在被混成了同一个结果文案。  
运营侧看到的就会是：

- 今天像是没触发
- 或者像是系统失效

### 4.4 手动回收虽然独立，但不是“版本发布后一次性手动回收”

当前手动回收路径已经独立于每日日期：

- `GMHandler` 直接走 `ExecuteRecycle()`
- 不经过 `RecycleTick()`

这一点是好的，已经满足“与每日回收独立运作”的基础能力。

但它还缺三样东西：

1. **版本级幂等**
   - 现在没有“本版本已经手动回收过”的持久化标记
2. **版本级审计**
   - 无法知道这次手动回收是为哪个版本发布执行的
3. **受控入口**
   - 现在只有通用 `bm_recycle`
   - 没有 `release_manual` / `daily_auto` 这样的模式区分

所以：

- 目前的手动回收更像“通用应急按钮”
- 不是“发版后一次性运营动作”

### 4.5 代码与旧文档存在口径漂移

本地旧文档里仍残留一些和现代码不一致的描述，例如：

- 旧文档写过 `RECYCLE_ENABLED = false`
- 旧文档写过 NPC 是“大黑无天”
- 旧文档还写过“回收完成后再推进日期”

而当前代码实际是：

- `RECYCLE_ENABLED = true`
- NPC 名称是 `胤`
- 先占坑，再执行回收

这类漂移会直接影响后续运营判断和问题排查。  
所以本轮方案必须把**代码、文档、运维口径**重新统一。

## 5. 优化目标

本轮建议把目标收敛成两个主目标、四个配套目标。

### 5.1 主目标 A：每日自动回收稳定在中午 12:00~13:00

要求：

- 正常情况下，只在中午窗口尝试自动回收
- 不再在 `00:00` 附近触发
- 不再因为失败重试而随机漂到凌晨 `00:40`

### 5.2 主目标 B：支持“版本发布后手动触发一次回收”

要求：

- 与每日自动回收完全独立
- 能标记“这是某个版本发布后的手动回收”
- 同一版本不会被误点多次重复执行

### 5.3 配套目标

1. 增加自动/手动两条路径的独立持久化状态
2. 增加审计记录，能还原“有没有跑、什么时候跑、跑了多少”
3. 增加重试节流，不再按帧重试
4. 更新测试与文档，避免再次口径漂移

## 6. 推荐改造方案

## 6.1 总体原则

本次不要继续在 `RecycleTick()` 上叠条件补丁。  
应该把它拆成两层：

1. **调度层**
   - 只负责“今天该不该跑、现在是不是窗口期、失败是否该重试”
2. **执行层**
   - 只负责“读取库存、筛选商品、随机数量、提交回收、返回结果摘要”

也就是：

- 保留 `ExecuteRecycle()`
- 重写自动调度规则
- 新增版本发布后的手动调度规则

### 6.1.1 致命性 BUG 的底线约束

这部分需要直接写进方案，而不是只写在讨论里：

1. **不能接受“状态写失败后再次触发，导致同版本重复回收”**  
   所以发版手动回收必须先写“预占锁/阻断态”，再执行真实回收。
2. **不能接受“回收已经发生，但系统仍提示可重试”**  
   所以凡是无法确认“商品未动”的失败，都应默认保持阻断，而不是放开重试。
3. **只有明确还没进入扣库存阶段的失败，才允许重试**  
   例如：读取库存失败、构建任务失败、预占锁写入前失败。
4. **每日自动回收和发版手动回收都必须满足‘最多一次有效生效’的安全目标**  
   前者靠日期占坑，后者靠版本锁；两条路径都不能只靠内存布尔锁。

这里的取舍要明确：  
**宁可误拦一次，也不能放出一次重复回收。**

## 6.2 方案 A：把自动回收改成“午间窗口调度”

### 6.2.1 新增配置

建议在 `BlackMerchantConfig.lua` 增加：

```lua
BMConfig.RECYCLE_TIMEZONE_OFFSET = 8 * 3600   -- 固定按 UTC+8 解释业务日期
BMConfig.RECYCLE_AUTO_START_HOUR = 12         -- 自动回收窗口开始
BMConfig.RECYCLE_AUTO_END_HOUR   = 13         -- 自动回收窗口结束（开区间）
BMConfig.RECYCLE_CHECK_INTERVAL  = 60         -- 调度检查间隔，秒
BMConfig.RECYCLE_RETRY_INTERVAL  = 300        -- 失败重试间隔，秒
BMConfig.RECYCLE_LATE_CATCHUP_END_HOUR = 18   -- 可选：错过午间后的最晚补跑时间
```

### 6.2.2 新的自动调度语义

建议按“北京时间业务日”判断，而不是直接依赖宿主机本地 `os.date()`：

1. 先把当前 `os.time()` 转成 UTC+8 业务时间
2. 算出：
   - `bizDate = YYYYMMDD`
   - `bizHour = 0~23`
3. 自动回收只在以下条件同时满足时尝试：
   - `bizDate` 今日未成功执行
   - `bizHour >= 12`
   - `bizHour < 13`

### 6.2.3 失败重试规则

不要再失败后“下一帧立刻重试”。  
改成：

- 窗口内失败：每 `5` 分钟重试一次
- 到了 `13:00` 还没成功：
  - 推荐方案：进入“补跑模式”，每 `10` 分钟重试一次，直到 `18:00`
  - 记录为 `late_auto`

推荐这样做的原因：

- 你要的是“尽量稳定在 12:00~13:00”
- 但如果云端在 12:00~13:00 整个窗口都抖动，完全跳过当天又会造成“每日回收失效”

所以推荐：

- **正常时段固定在 12:00~13:00**
- **异常时段允许当天下午补跑一次**

这样既满足“时间窗口稳定”，也满足“每日回收尽量别丢”。

## 6.3 方案 B：把手动回收升级为“版本发布后一次性手动回收”

### 6.3.1 当前能力与差距

当前已有：

- `bm_recycle` 通用手动回收

当前缺少：

- 针对“版本发布后”的版本级幂等

### 6.3.2 推荐做法

新增一条独立命令，例如：

```lua
bm_recycle_release
```

服务端执行时默认携带：

- `releaseVersion = GameConfig.CODE_VERSION`

并持久化一个新的系统状态，例如：

- `bm_recycle_release_version`
- `bm_recycle_release_time`

执行规则：

1. 如果 `bm_recycle_release_version == GameConfig.CODE_VERSION`
   - 直接返回“当前版本已执行过发布回收”
2. 否则
   - 执行一次 `ExecuteRecycle(mode="release_manual")`
   - 成功后写入当前版本号与执行时间

### 6.3.3 为什么不能继续只用 `bm_recycle`

因为 `bm_recycle` 现在是：

- 随时可点
- 无版本标识
- 无一版一次的保护

它适合：

- 临时应急
- 调试验证

它不适合：

- 发版后的标准运营动作

建议最后保留两类命令：

1. `bm_recycle`
   - 通用人工回收
   - 不带版本语义
2. `bm_recycle_release`
   - 发版后一次性回收
   - 带版本幂等与审计

## 6.4 方案 C：增加独立状态与审计记录

- `bm_recycle_date`

### 6.4.1 自动回收状态

### 6.4.2 发版手动回收状态

建议新增：

- `bm_recycle_release_version`
- `bm_recycle_release_at`

### 6.4.3 回收运行日志

建议新增一个 capped list，例如：

- `bm_recycle_runs`

每次执行都写一条：

```lua
{
    t = 1717157400,
    mode = "auto_daily" | "late_auto" | "release_manual" | "manual",

    codeVersion = 5,
```

- 是没满仓、随机全 0、还是执行失败

## 6.5 方案 D：把执行结果摘要做细

建议把 `ExecuteRecycle()` 的 summary 从纯字符串升级成结构化结果，例如：

```lua
{
    ok = true,
    mode = "auto_daily",
    fullStockKinds = 5,
    zeroRollKinds = 2,
    executedKinds = 3,
    executedItems = 4,
    failedKinds = 0,
    summary = "满仓5种，随机跳过2种，成功回收3种共4件"
}
```

重点要把以下几类区分开：

1. `skip_no_full_stock`
2. `skip_zero_roll`
3. `partial_success`
4. `success`
5. `failed_before_execute`

这能直接解决“看起来像没触发”的误判。

## 8. 测试方案

这次不能只补日期工具函数测试，必须补“窗口调度 + 模式隔离”测试。

### 8.1 自动回收窗口测试

至少覆盖：

1. `11:59`
   - 不触发
2. `12:00`
   - 首次进入窗口，允许触发
3. `12:35`
   - 若未成功执行，允许窗口内重试
4. `13:01`
   - 正常窗口结束
5. `13:30`
   - 若启用补跑，允许 `late_auto`
6. `18:01`
   - 超过补跑截止，不再执行当日自动回收

### 8.2 自动回收失败重试测试

至少覆盖：

1. `money:Get` 失败后，不是下一帧立即重试，而是等到 `RETRY_INTERVAL`
2. `money:Add` 失败同理
3. 同一自然日只会有一个“成功执行”结果

### 8.3 发版手动回收测试

至少覆盖：

1. 当前版本首次执行成功
2. 当前版本第二次执行被拒绝
3. 版本号变化后允许再次执行
4. 发版手动回收不修改 `bm_recycle_auto_date`
5. 自动回收已执行过，当天仍可执行发版手动回收

### 8.4 执行结果摘要测试

至少区分：

1. 无满仓商品
2. 有满仓商品但随机全 0
3. 部分商品成功、部分失败

## 10. 最终建议

如果只按当前代码继续运行：

- 它仍然会“按天尝试回收”
- 但**不会稳定在中午**
- 也**不具备“发版后一次性手动回收”的运维能力**

所以建议本轮不要再把问题定义成“再修一次回收 bug”。  
更准确的做法是：

1. **把自动回收从“自然日首个成功 tick”改成“午间窗口调度”**
2. **把发版手动回收从“通用手动按钮”升级成“版本级一次性动作”**
3. **把回收结果从“只看库存有没有变化”升级成“有完整审计状态”**

这样才能同时满足你的两个目标：

1. 每日回收稳定在中午 `12:00~13:00`
2. 版本发布后可以手动触发一次，并且与每日回收独立运作

## 11. 直接结论

按当前代码现状，我给出的落地建议是：

- **自动回收保留**
  但重做调度层，不再使用“过天后首个成功 tick”模型

- **手动回收保留**
  但新增 `bm_recycle_release`，作为“发版后一次性回收”专用命令

- **执行器继续复用 `ExecuteRecycle()`**
  但要让它返回结构化结果，并写审计日志

- **下一步实施优先级**
  1. 调度窗口与重试节流
  2. 自动/发版手动状态拆分
  3. 发版手动回收命令与 GM 入口
  4. 审计日志与测试补齐

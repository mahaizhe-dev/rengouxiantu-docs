> 2. 新路线是否可落地

## 1. 当前前提

以下事实视为当前路线重排的输入前提：

1. `P0-4` 及之前的优化已经发布到线上。
2. 这次发布按暗发布处理：
   - `SAVE_PIPELINE_V2 = false`
   - `SAVE_MIGRATION_SINGLE_ENTRY = false`
   - `CLOUDSTORAGE_STRICT_BATCHSET = false`
3. 发布前 blocking gate 为绿。
4. 用户自测核心链路正常。
5. 当前项目是**真实线上项目**，优先级始终服从：
   - `docs/架构文档/rules.md`
6. 用户无法稳定回传线上遥测数据。
   - 后续线上判断主要依赖：
     - 用户自有账号复测
     - 玩家反馈

这意味着：

- 后续优化不能默认依赖“线上指标会告诉我们是否安全”。
- 后续高风险改造必须更强调：
  - 旧路径保留
  - feature flag
  - 可快速回滚
  - 诊断可读性

## 2. 对原后续路线的复盘结论

原路线在 `P0-4` 之后的大致顺序为：

1. `P0-6` 服务端存档事务服务化
2. `P1-1` main 编排器拆分
3. `P1-2` UI 事务降级
4. `P1-3` SaveSystemNet 状态机
5. `P1-4` 排行/奖励事务收敛
6. `P2-1/P2-2/P2-3/P2-4`

复盘后结论：

### 2.1 原 `P0-6` 不适合直接接在 `P0-4` 后面

原因：

1. 现有 `server_main.lua` 中，`FetchSlots / LoadGame / SaveGame / rank2_* / trial_* / prison_* / account_* / MigrateData` 仍混在同一条主链上。
2. 在缺少稳定线上遥测的情况下，直接做大颗粒“服务端事务服务化”，改动面过大。
3. 一旦出现线上存档异常，很难快速判定是：
   - slot 链路
   - save 链路
   - 排行写入
   - 迁移残留
   - 账号级旁路写入

因此，原 `P0-6` 应视为**范围过大的旧任务定义**，不应直接继续。

### 2.2 原 `P1-3 SaveSystemNet 状态机` 应前移

原因：

1. `SaveSystemNet.lua` 仍使用单槽 pending 结构：
   - `_pendingFetchSlots`
   - `_pendingLoad`
   - `_pendingChar`
   - `_pendingMigrate`
   - `_queuedFetchSlots`
2. 废弃迁移路径虽然已被收口判断，但旧迁移函数和旧状态仍留在同一模块内。
3. 这比拆 `main.lua` 或降级 UI 更接近真实线上风险。

结论：`SaveSystemNet` 状态机问题优先级高于 `P1-1 / P1-2`。

### 2.3 原 `P2-3 日志分级与诊断开关治理` 应前移

原因：

1. 用户无法依赖稳定遥测。
2. 关键链路里 `print` 和诊断日志数量已经很高。
3. 如果没有统一的诊断开关与分级，后续高风险任务会越来越难定位。

结论：日志/诊断治理不该继续留在较后阶段，应提前变成护栏任务。

### 2.4 原 `P1-1 / P1-2` 仍有价值，但不再是当前最高优先级

原因：

1. 它们改善结构和可维护性。
2. 但对当前线上最危险的问题帮助不直接。
3. 现在更需要优先处理：
   - 真实基线一致性
   - 诊断能力
   - 存档/网络状态机收口

结论：后移，不取消。

### 2.5 原 `P2-1 / P2-2` 改成“证据驱动”

原因：

1. 当前确实存在全局扫描、渲染循环和掉落渲染成本。
2. 但项目里已经有 `PerfMonitor`。
3. 在没有明确卡顿证据前，性能优化不应继续抢占高风险存档/网络治理前面的位置。

结论：不删除，但改为“有证据再立项”。

## 3. 新的后续路线建议

下面不是最终执行顺序，而是新的**候选顺序**。
执行 AI 需要先评估它是否还能继续拆细。

### R0 发布基线归档与源码对齐

目标：

1. 固定当前已上线版本的真实基线。
2. 修正文档事实漂移。
3. 明确后续所有任务以哪个 Git 基线为准。

必要性：

- 当前本地工作树和远端 `origin/main` 可能不一致。
- `rules.md` 中的存档常量可能与代码事实漂移。
- 如果基线不稳定，后续所有任务评估都可能建立在错误前提上。

建议范围：

1. 记录当前上线 commit SHA
2. 记录当前线上 flag 状态
3. 回填到发布基线文档
4. 校正规则文档中的事实常量：
   - `CURRENT_SAVE_VERSION`
   - `AUTO_SAVE_INTERVAL`
   - 其他已明确漂移的常量

这一步应是 docs-only 或 docs-first。

### R1 诊断与日志治理前移

目标：

1. 给后续高风险任务提供更稳定的定位手段。
2. 统一关键链路的日志等级和开关。

建议范围：

优先覆盖：

1. `server_main.lua`
2. `network/SaveSystemNet.lua`
3. `network/CloudStorage.lua`
4. `systems/save/*`

建议只做：

1. 等级分层
   - error
   - warn
   - info
   - diag/debug
2. 统一开关入口
3. 关键链路日志字段标准化

明确不做：

1. 重写全部日志
2. 改业务逻辑
3. 改协议

### R2 SaveSystemNet 状态机收口

目标：

1. 收口单槽 pending 结构。
2. 降低并发/晚到响应/重试路径的不确定性。
3. 给后续服务端链路收口打基础。

建议范围：

1. 请求关联结构统一
2. 晚到响应忽略
3. 重试/取消语义显式化
4. 废弃迁移分支 tombstone 化
   - 保留兼容痕迹
   - 但不再让旧路径继续承担真实业务职责

明确不做：

1. 强行大改协议
2. 顺手重构 UI
3. 顺手重构 `main.lua`

### R3 `P0-6A` 服务端存档主链等价拆分

这是对原 `P0-6` 的替代，不再直接做“大而全”的服务化。

目标：

1. 先只拆服务端存档主链的壳。
2. 维持行为等价。
3. 限制改动面。

第一阶段建议只覆盖：

1. `FetchSlots`
2. `LoadGame`
3. `SaveGame`

明确排除：

1. `MigrateData`
2. `rank2_* / trial_* / prison_*`
3. `Redeem / BlackMerchant / Dungeon pending reward`
4. 账号级奖励事务整合

### R4 排行/奖励事务收口

目标：

1. 单独处理最容易出现脏数据的奖励/排行写入链路。
2. 不与主存档链路拆分混做。

建议后移到 `R3` 之后。

### R5 客户端结构治理

包含原：

1. `P1-1 main 编排器拆分`
2. `P1-2 UI 事务降级`

这两项仍然有价值，但当前不应排在 `R1 / R2 / R3` 前面。

### R6 性能项改为证据驱动

原：

1. `P2-1` 空间索引
2. `P2-2` 渲染缓存

改为：

- 只有当 `PerfMonitor`、自测或玩家反馈能证明存在真实卡顿瓶颈时，才进入正式任务卡阶段。

# 炼丹炉重构执行方案

## 1. 目标

本方案用于指导 AI 或开发者对炼丹炉系统进行分阶段重构。

本轮重构的核心目标不是修改玩法，也不是统一存档格式，而是：

1. 将丹药计数的运行时真值从 `ui.AlchemyUI` 迁移到 `systems.AlchemySystem`
2. 保持玩家可见行为不变
3. 保持存档字段、服务端校验、C2S 授权链路稳定
4. 为后续 UI 瘦身和规则沉淀建立安全基础

## 2. 当前代码事实

当前炼丹炉并不是普通的 UI 页面，而是一个同时承载以下职责的混合模块：

1. UI 渲染
2. 配方静态配置
3. 炼制前置校验
4. 材料与灵蕴扣减
5. 永久属性加成
6. 丹药计数维护
7. `player.pillCounts` 双写
8. C2S 授权回调时序管理
9. 存档系统的 getter / setter 真值来源

关键耦合点如下：

1. `scripts/systems/save/SaveSerializer.lua`
   当前直接 `require("ui.AlchemyUI")` 以 IIFE 方式读取 7 个计数 getter（第 64-95 行）。

2. `scripts/systems/save/SaveLoader.lua`
   当前 `require("ui.AlchemyUI")`（第 133 行），用于：
   - 丹药计数运行时校正逻辑（第 225-268 行）：从 ChallengeSystem 完成记录反推正确计数，与 AlchemyUI.Get* 比对并修正漂移
   - 初始化 `player.pillCounts`（基于 AlchemyUI 当前值构建快照）
   - **注意**：此处不是简单的 Set/Get 替换，校正逻辑依赖 DeserializePlayer 先完成赋值

3. `scripts/ui/CharacterTab3.lua`
   当前直接 `require("ui.AlchemyUI")`（第 10 行），读取 7 个 getter 显示丹药服用次数（第 23、28、33、38、121、127、132 行）。

4. `scripts/ui/NPCDialog.lua`
   当前延迟 `require("ui.AlchemyUI")`（第 52 行），调用 `Create`、`Show`、`Hide`、`Destroy` 管理炼丹炉面板生命周期（第 309、345、840、870 行）。此为 UI 生命周期依赖，不涉及数据真值，Phase 3 涉及。

5. `scripts/network/SaveWriteService.lua`
   当前按 `tigerPillCount`、`snakePillCount`、`diamondPillCount`、`temperingPillEaten` 等字段名做上限校验。

6. `scripts/network/PillHandler.lua`
   当前按固定 `pillId` 白名单做服务端授权。

7. `scripts/ui/AlchemyUI.lua`
   当前持有 `pendingPillBuys_`、`SendBuyPill`、授权结果处理、断线和限流清理。

## 3. 硬约束

以下规则在本轮重构中不得违反：

1. 不修改存档字段名
   包括但不限于：
   - `tigerPillCount`
   - `snakePillCount`
   - `diamondPillCount`
   - `temperingPillEaten`
   - `dragonBloodPillCount`
   - `swordIntentPillCount`
   - `abyssSealPillCount`
   - `pillCounts`

2. 不修改存档结构
   第一轮不得将扁平字段强制合并为单一嵌套结构。

3. 不修改 `SaveState.CURRENT_SAVE_VERSION`

4. 不修改 `PillHandler.VALID_PILL_IDS`

5. 不修改 `SaveWriteService` 的字段校验语义

6. 不将"真值迁移"和"异步授权迁移"放在同一阶段一起做

7. 未补测试前，不允许直接删除 `AlchemyUI.Get* / Set*`

8. `AlchemySystem` 不得 require 任何 UI 模块、SaveSerializer 或 SaveLoader（防止循环依赖）

9. 阵营丹计数（`xueshaDanCount`、`haoqiDanCount`）本轮不迁移，保留在 `ChallengeSystem` 中

## 4. 风险分级

### 4.1 P0 风险

1. `SaveSerializer` / `SaveLoader` 断链
   后果：读档或存档直接失效。

2. `SaveWriteService` 校验失配
   后果：服务端误拒或失去上限保护。

3. 运行时真值迁移不完整
   后果：计数、永久属性、显示、存档四者不一致。

### 4.2 P1 风险

1. `player.pillCounts` 双写漂移
   后果：挑战系统、角色面板、读档校正结果不一致。

2. `pendingPillBuys_` 时序问题
   后果：授权成功但未执行、重复执行、断线后残留。

3. `CharacterTab3` 读取源切换不完整
   后果：角色面板显示异常，但核心存档不一定损坏。

4. 模块加载顺序 / 循环依赖
   后果：SaveSerializer 序列化时 AlchemySystem 尚未初始化，返回默认值 0，丹药计数全部清零。
   预防：AlchemySystem 必须是纯数据模块，零外部 require。

### 4.3 P2 风险

1. 配方提取后字段设计过度理想化
   后果：后续面板复用困难，但不一定影响本轮上线。

2. 阵营丹药与普通炼丹过度统一
   后果：实现复杂度上升，测试负担变大。

3. NPCDialog 面板生命周期接口变动
   后果：Phase 3 改动 AlchemyUI 导出接口时可能断链。

## 5. 总体执行策略

本轮只允许做两类迁移：

1. 代码归属迁移
2. 同步业务逻辑迁移

本轮不做四类事情：

1. 存档格式统一
2. 服务端协议改造
3. 异步授权链重写
4. 阵营丹（xuesha/haoqi）计数迁移

推荐按 4 个阶段执行，每个阶段必须：

1. 独立提交
2. 独立测试
3. 独立验收
4. 允许单独回滚

## 6. Phase 0：测试基线

### 6.1 目标

在不改生产行为的前提下，把当前炼丹炉关键行为钉住。

### 6.2 必做项

1. 新增丹药计数的序列化 / 反序列化 round-trip 测试
2. 新增 `pillCounts` 初始化测试（缺失时从扁平字段构建）
3. 新增 `pillCounts` 漂移修正测试（扁平字段与 pillCounts 不一致时以扁平为准）
4. 新增 `CharacterTab3` 读取丹药计数的兼容测试
5. 新增 `ReturnToLogin` 后炼丹运行时状态清理测试
6. 保持并复用现有 `test_save_optimization.lua` 里的保存语义测试

### 6.3 建议测试文件

1. `scripts/tests/test_alchemy_save_roundtrip.lua`
2. `scripts/tests/test_alchemy_state_sync.lua`
3. 如有必要，新增 `scripts/tests/test_alchemy_ui_auth_flow.lua`

### 6.4 验收标准

1. 不删除任何现有 getter / setter
2. 现有相关测试全部通过
3. 新增测试稳定通过

## 7. Phase 1：迁移真值归属

### 7.1 目标

新增 `scripts/systems/AlchemySystem.lua`，接管炼丹炉 7 个核心计数的运行时真值。

### 7.2 新模块最小职责

`AlchemySystem` 第一阶段只负责：

1. 持有 7 个计数的运行时状态
2. 暴露 `Get* / Set*` 接口
3. `Set*` 内部同步更新 `player.pillCounts` 对应字段（保持双写语义不变）
4. 暴露 `ResetRuntimeState()`
5. 暴露 `GetSnapshot()` 或等价快照接口

**模块依赖约束**：AlchemySystem 只允许 require `core.GameState`（用于访问 player.pillCounts），不得 require 任何 UI 模块、Save 模块或网络模块。这确保它可以在任何加载时机被安全 require。

### 7.3 必须迁移的调用方

1. `SaveSerializer.lua`
   从 `require("ui.AlchemyUI").Get*` 改为 `require("systems.AlchemySystem").Get*`

2. `SaveLoader.lua`
   从 `require("ui.AlchemyUI")` 改为 `require("systems.AlchemySystem")`
   **时序约束**：SaveLoader 中的校正逻辑依赖 DeserializePlayer 先完成赋值。迁移后 `AlchemySystem.Set*` 必须在 `SaveSerializer.DeserializePlayer` 内被调用，校正逻辑在其后执行。不可简单做字符串查找替换，必须确认执行顺序。

3. `CharacterTab3.lua`
   从 `require("ui.AlchemyUI").Get*` 改为 `require("systems.AlchemySystem").Get*`

4. `main.lua`
   `ReturnToLogin()` 必须调用 `AlchemySystem.ResetRuntimeState()`

### 7.4 兼容要求

`AlchemyUI.lua` 中原有 getter / setter 不立刻删除，改为 thin wrapper：

1. `AlchemyUI.GetTigerPillCount()` 内部转发到 `AlchemySystem.GetTigerPillCount()`
2. `AlchemyUI.SetTigerPillCount()` 内部转发到 `AlchemySystem.SetTigerPillCount()`
3. 其他 6 组同理

这些 wrapper 保证 AlchemyUI 内部的 `DoCraft*` 回调（Phase 3 才迁移）仍能正确读写计数。

### 7.5 本阶段禁止事项

1. 不迁移 `SendBuyPill`
2. 不迁移 `pendingPillBuys_`
3. 不修改任何 `DoCraft*` 实现
4. 不修改 `SaveWriteService`
5. 不修改 `PillHandler`
6. 不修改 `NPCDialog.lua`
7. 不迁移阵营丹（xueshaDanCount / haoqiDanCount）

### 7.6 验收标准

1. 非 UI 模块不再以 `ui.AlchemyUI` 作为真值源
2. 读档、存档、角色面板显示行为不变
3. 退出角色后运行时计数能正确清空
4. `player.pillCounts` 双写行为不变

## 8. Phase 2：提取配方与同步逻辑

### 8.1 目标

将同步业务逻辑从 `AlchemyUI` 中抽离，但不改异步授权链。

### 8.2 建议新增文件

1. `scripts/config/PillRecipes.lua`
2. 扩展 `scripts/systems/AlchemySystem.lua`

### 8.3 配方模型建议字段

1. `id`
2. `chapter`
3. `recipeKind`
4. `authMode`
5. `savePolicy`
6. `currencyCost`
7. `materialId`
8. `materialCount`
9. `countKey`
10. `maxCount`
11. `bonusStat`
12. `bonusValue`
13. `delegateSystem`

### 8.4 系统层建议新增接口

1. `CanCraft(recipeId)`
2. `Craft(recipeId)`
3. `SyncPlayerPillCounts()`
4. 如有必要，提供 `ApplyPillBonus()`、`ConsumeCraftCost()` 等内部辅助函数

### 8.5 迁移原则

1. 普通炼丹和产物型炼丹的同步校验迁到 `AlchemySystem`
2. 永久丹的加属性、计数增加、`pillCounts` 同步迁到 `AlchemySystem`
3. 阵营丹继续委托 `ChallengeSystem.BrewPill()`，计数保留在 ChallengeSystem
4. 令牌盒保留独立分支，不强制按普通丹药处理
5. `xueshaDanCount` / `haoqiDanCount` 不在本轮迁移范围内

### 8.6 保存语义要求

1. 普通高频炼制继续使用会话合并
2. 限量永久丹继续即时 `save_request`
3. 不得改变 `test_save_optimization.lua` 已定义的保存策略

### 8.7 验收标准

1. `AlchemyUI` 中的同步业务逻辑显著减少
2. 保存语义、计数语义、属性增益语义保持不变
3. 所有配方仍使用原有 `pillId`

## 9. Phase 3：UI 瘦身

### 9.1 目标

将 `AlchemyUI` 收敛为"渲染 + 点击分发 + 授权结果展示"。

### 9.2 UI 层保留职责

1. 面板创建与销毁
2. 配方列表渲染
3. 点击事件分发
4. `SendBuyPill`
5. `pendingPillBuys_`
6. `S2C_BuyPillResult`
7. 断线和限流清理
8. 提示文案和刷新

### 9.3 UI 层必须迁出的职责

1. 炼制前置校验
2. 材料与灵蕴扣减
3. 永久属性加成
4. 计数增加
5. `player.pillCounts` 同步（Phase 1 已迁到 AlchemySystem.Set* 内部）
6. 真值持有（Phase 1 已完成）

### 9.4 NPCDialog 兼容

AlchemyUI 的模块导出接口（`Create`、`Show`、`Hide`、`Destroy`、`IsVisible`）在本阶段必须保持签名不变，NPCDialog 依赖这些接口管理炼丹炉面板生命周期。

### 9.5 验收标准

1. `AlchemyUI` 不再持有丹药真值
2. `AlchemyUI` 不再承担主要业务逻辑
3. 授权成功、拒绝、超时、断线行为不变
4. NPCDialog 打开/关闭炼丹炉行为不变

## 10. Phase 4：未来可选项

以下内容不属于本轮必做，必须另立任务：

1. 存档字段统一到单一 `pillCounts`
2. 提升 `CURRENT_SAVE_VERSION`
3. 增加新 migration
4. 重构 `SaveWriteService` 的丹药字段校验
5. 重构 `PillHandler` 或服务端协议
6. 迁移阵营丹计数到统一系统

只有在前 3 个阶段稳定上线并完成回归验证后，才允许进入这一阶段。

## 11. AI 执行时的明确禁止项

执行 AI 在任何阶段都不得自行做以下改动：

1. 修改 `CURRENT_SAVE_VERSION`
2. 修改 `pillId`
3. 删除扁平丹药字段
4. 删除 `pillCounts`
5. 修改 `SaveWriteService` 校验字段名
6. 修改 `PillHandler.VALID_PILL_IDS`
7. 将阵营丹与普通炼丹强行统一为同一实现
8. 在没有测试的前提下删除 `AlchemyUI.Get* / Set*`
9. 将真值迁移、同步业务迁移、异步授权迁移一次性混做
10. 让 `AlchemySystem` require 任何 UI 模块或 Save 模块
11. 迁移 `xueshaDanCount` / `haoqiDanCount` 到 AlchemySystem

## 12. 自动化测试要求

### 12.1 必须覆盖

1. 7 个核心计数的 save / load round-trip
2. `pillCounts` 缺失时的初始化（从扁平字段构建）
3. `pillCounts` 漂移时的校正（以扁平字段为准）
4. `CharacterTab3` 读取正确计数
5. 普通炼丹的会话合并保存
6. 限量永久丹的即时保存
7. 关闭炼丹炉时的收口 flush
8. `ReturnToLogin` 后运行时状态清零
9. AlchemySystem.Set* 同步更新 player.pillCounts

### 12.2 建议覆盖

1. 单机模式直接执行回调
2. 联机授权成功
3. 联机授权拒绝
4. 授权超时
5. 限流清理 pending
6. 断线清理 pending

## 13. 人工回归清单

每次完成一个阶段后，至少手工验证以下场景：

1. 普通炼丹一次
2. 永久丹炼制一次
3. 阵营丹炼制一次
4. 令牌盒兑换一次
5. 联机授权成功一次
6. 联机授权拒绝一次
7. 断线后重新尝试一次
8. 退出角色后重新登录一次
9. 读档后核对永久属性与计数一次
10. 角色面板（CharacterTab3）丹药数显示正确

## 14. 交付标准

只有同时满足以下条件，才视为本轮重构合格：

1. 存档格式未变化
2. 服务端校验未失效
3. `CharacterTab3` 显示无回归
4. 读档校正逻辑无回归
5. 会话合并与即时保存语义无回归
6. `AlchemyUI` 不再是唯一真值源
7. `player.pillCounts` 双写行为不变

## 15. 给执行 AI 的任务摘要

你要在不修改存档格式、不修改 `pillId`、不修改 `SaveWriteService` 校验规则的前提下，完成炼丹炉系统的分阶段重构。第一阶段只补测试；第二阶段只迁移运行时真值到 `AlchemySystem`，并同步改造 `SaveSerializer`、`SaveLoader`、`CharacterTab3`、`main.lua`；第三阶段提取配方与同步业务逻辑；第四阶段再瘦 `AlchemyUI`。每个阶段必须独立提交、独立测试、独立验收。若实施过程中出现需要提升存档版本、修改字段名、修改 `VALID_PILL_IDS`、或同时迁移异步授权链的情况，必须立即停止并报告，不得自行继续实现。

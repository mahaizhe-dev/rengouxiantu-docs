## 〇、总览：一句话结论

项目**运行时引擎质量不错**（数据驱动的装备/掉落/图鉴管线、章节世界内容注册表、SaveSystem 拆 11 子模块、服务端 Update 的 pcall 兜底都是正面范式），但存在**四类系统性债**，全部源于"缺少几个关键抽象，于是靠手工复制/手工对齐维持"：

1. **稳定性**：EventBus 监听器泄漏（5 个副本系统每次切角色/重建世界累加订阅）+ 十余个可选模块裸 require（单文件故障全服进不去）。
2. **可维护性**：Handler 事务样板复制 ~26 遍（~700 行纯样板）、4 套平行 ArtifactSystem fork（~3400 行可省）、错误字段 4 种混用、协议 45 个缩写别名、4 个 .pre-s1 备份污染源码树。
3. **可扩展性**：存档"四文件契约"手工对齐（漏一处静默丢数据）、属性汇总是硬编码巨型求和、新增一章要复制 ~2000 行。
4. **性能**：（存档管道/日志/瓦片已在上轮覆盖）本轮新增——EdgeDecalRenderer 每帧数千次 GetTile、怪物名每帧重复测量、TownReturnEntry 常驻每帧重排、WarehouseUI 每次开箱销毁重建 112 控件。

好消息：**每一类都有现成的正面范式在项目内**（TriggerBattleSystem 的 unsub、SaveSystem 的拆分、InventoryUI 的 SetItem 复用、ChapterConfig 的注册表），重构是"向已有的好范式对齐"，不是从零设计。

## 一、稳定性

### 1.1 P0 — EventBus monster_death 监听器泄漏（SwordWallSystem）

`SwordWallSystem.lua:45-47` 的 `Init()` 内 `EventBus.On("monster_death", 匿名闭包)` **丢弃了 unsub 返回值**，而 `Init` 在 `main.lua:959` 的进入世界/章节切换/存档加载路径上——**每次世界重建都再订阅一次**，匿名闭包无法被 `EventBus.Off` 移除。`EventBus.On` 是纯 `table.insert` 无去重（`core/EventBus.lua:16`）。后果：玩多章后单次怪物死亡触发 N 份回调 → 剑气长城计数/掉落 N 倍放大 + 单帧遍历退化。

### 1.2 P1 清单

| 编号 | 问题 | 位置 | 后果 |
|---|---|---|---|
| S-1 | 同型 EventBus 泄漏，Init 在 ReturnToLogin 每次切角色累加 | ChallengeSystem:206、SealDemonSystem:62、TrialTowerSystem:55、PrisonTowerSystem:56 | 监听器无界增长，回调重复执行（多数有 IsActive 早退削弱危害） |
| S-2 | 十余个可选玩法模块**裸 require**，HandleUpdate 无 nil 守卫硬调用（对照 rules §九，仅 WarehouseUI/DungeonClient 做了 pcall） | main.lua:64-101 顶层 + :1562/1567/1571 调用 | 任一活动/神器/副本文件语法错或加载抛异常 → main.lua 整体加载失败、GameStart 未定义、**全服进不去** |
| S-3 | 奖励发放忽略 AddConsumable/AddItem 返回值（对照 rules §十一） | SealDemonSystem:332/351/690/708、AlchemySystem:385/491、ArtifactSystem:172、TrialTowerSystem:425、SwordWallSystem:289、challenge/rewards:257 | 背包满/未注册消耗品时奖励静默蒸发，无日志 |
| S-4 | 多个系统有 Reset 但未接入 ReturnToLogin 重置清单 | EventSystem:76、FortuneFruit:470、Qinglian:219、ArtifactSystem 系列:694/655/756（tiandi 无 Reset）、AtlasSystem:1192 | 模块级 local 累积态跨角色残留（EventSystem 兑换记录、Artifact _pendingActions 风险最高） |

### 1.3 正面范式（应作为整改模板）

- **`TriggerBattleSystem.Init:110-114`**：`if M._unsubMonsterDeath then M._unsubMonsterDeath() end` 再重订阅——本项目 EventBus 的**正确幂等范式**，S-1 的 5 个系统应照抄。
- `server_main.lua HandleServerUpdate:1240-1253`：全 handler `pcall + if handler then` + 打印错误，教科书级。
- `SaveSystem.Init:54-56` 有 `EventBus.Off` 兜底。

## 二、可维护性

### 2.1 上帝对象 / 超大文件（Top20 摘要）

| 类型 | 代表 | 判定 |
|---|---|---|
| **合理的大**（数据/迁移，勿动） | SaveMigrations(1628)、EquipmentData_Forge(1061)、MonsterTypes_ch4(1050) | 纯数据表 |
| **该拆的混杂逻辑** | ChallengeUI(2144)、BlackMerchantUI(1944)、DungeonHandler(1763)、SystemMenu(1631)、DragonForgeUI(1440)、Monster(1427)、Player(1361)… UI 层占 Top20 的 13 席 | 多职责混杂 |
| **枢纽型大**（拆文件不解决，需降耦） | main.lua(1884, 118 require)、GMConsole(1743, 127 require) | 引用全世界 |

### 2.2 样板代码重复（最高 ROI 重构点）

network/ 下 Handler 的权威写入样板 `GetUserId → 标志校验 → AcquireSlotLock → ServerGetSlotData → 修改 → BatchCommit → 释放锁(成功+失败双路径) → 回包` 复制情况：`GetUserId` 35 次、`ServerGetSlotData` 26 次、`BatchCommit` 39 次、`AcquireSlotLock` 17 次、`serverCloud` 直写 save_ 的 read-modify-write 散在 10+ 文件。

**估算**：26 个事务 × ~35 行样板 ≈ 900 行纯样板；抽象成 `Session.WithSlotTransaction(userId, slot, fn)` 高阶封装（内部统一锁获取/释放/失败回滚/标准回包）可消除 **600~700 行**，并把 39 处 BatchCommit 错误处理收敛到一处——**同时收窄"锁泄漏/回滚遗漏"的 bug 面**（这正是前几轮存档审计里反复出现的问题类型）。

### 2.3 命名与约定不一致

- **协议事件双写**：`C2S_BlackMerchantBuy`↔`C2S_BMBuy`、`C2S_EventExchange`↔`C2S_EvtExchange` 等——唯一事件字符串 172 个，逻辑事件仅 ~127 个，**约 45 个是缩写别名**污染协议表。
- **错误回包字段四种混用**：`error=` 172 次、`reason=` 56、`message=` 43、`msg=` 23——客户端必须四选一兜底解析。
- **server_main 66 个 `_G` 全局包装函数**做路由，与其余模块 `local M = {}` 风格冲突。

### 2.4 模块依赖健康度

- 最热公共依赖：GameState(161)、GameConfig(153)、InventorySystem(148)、EventBus(116)、CombatSystem(114)——改动波及面极广。
- 核心层（core/）无循环 require（健康）；systems↔ui 间建议专项核查（InventorySystem 被 148 处引用，若反向 require UI 则成环）。
- GMConsole(127 require) 是加载脆弱点。

### 2.5 死代码 / 历史包袱（零风险快赢）

- **4 个 .pre-s1 备份文件残留源码树**（server_main/SaveSystem/SaveSystemNet/CloudStorage，共 ~28 万字节），污染 grep、误导维护者——应立即移出。
- ~50 处 DEPRECATED/TODO/FIXME 待清。
- **BlackMarketSellGuard 是"常量伪装成算法"**：分类逻辑恒返回 blocked，可退化为一行常量判定。

## 三、可扩展性

### 3.1 五条"新增 X"改动链摩擦量化

| 新增 X | 改动文件 | 改动点 | 最易漏的环 → 后果 | 抽象现状 |
|---|---|---|---|---|
| 特殊装备/法宝 | 3+1图标（法宝+2） | 4 点 | **掉落源**（在另一目录，配置"看着完整"却漏）→ 图鉴永久灰格不可获得；order↔entries 分裂 | 运行时全数据驱动✅，但配置侧 equipId 手工重复 4 处 |

| 属性来源系统 | ~8 核心 + UI | ~15~18 点 | **服务端 clamp** + **SaveDTO 白名单**两处独立硬编码表，漏 clamp→外挂静默通过，漏白名单→重载丢失 | **无注册表**，Player._RecalcStatsCache 是一长串内联手加 |
| 消耗品（永久效果） | ~8 | ~10 | EquipTooltip 硬编码白名单 + batch_use `elseif id==` 分支 + 存档双写 + clamp 任漏一环 | 效果侧纯手写，无 effect 数据表 |

### 3.2 共性问题（反复咬人的环节）

1. **存档"四文件契约"手工对齐**（链 2/3/4/5 全中）：SaveDTO 白名单 ↔ SavePersistence.Serialize ↔ SaveLoader.Deserialize ↔ SaveWriteService clamp，任一漏项**静默丢数据、存读一轮才暴露**。**已有活证据**：SwordPool 缺 DTO 条目且无 clamp，而同类 SeaPillar 两者都有——证明这些手工点真的会被漏。
2. **属性汇总是硬编码巨型求和不是注册表**（链 1/3/4）：新增任何属性来源都要回改 `Player._RecalcStatsCache` 中枢。
3. **服务端 clamp 是第二张独立硬编码表**（链 2/4/5）：与 Player 汇总、SaveDTO 三表不同步 → rules §12 已记三起 P0。
4. **"整系统克隆"当扩展手段**（链 3 最重）：神器每章复制 ~2000 行 + 15 处 chN 分支。

## 六、四维总评与关键认知

| 维度 | 评级 | 关键结论 |
|---|---|---|
| 稳定性 | 中，有明确高危点 | EventBus 泄漏 + 裸 require 是两个"会导致全服级故障"的结构缺陷，但都有现成范式可对齐，修复成本低 |
| 可维护性 | 中偏低 | UI 层巨石 + Handler 样板复制 + 命名/字段不一致 + 死码；SaveSystem 拆分是全项目正面标杆 |
| 可扩展性 | 偏低 | 核心债是"缺注册表所以靠手工对齐/整系统克隆"；存档四文件契约和属性中枢是两个反复咬人的点 |
| 性能 | 中偏好 | 已做过认真治理、无平方复杂度炸弹；剩余是若干每帧浪费和面板重建，均可低成本消除 |

**贯穿性认知**：这四维的债**同源**——项目在"运行时引擎"上做了数据驱动（好），但在"配置登记 / 系统扩展 / 跨文件契约"上缺了 3~4 个注册表抽象（存档字段、属性来源、活动、章节神器），于是所有扩展都退化成手工复制与手工对齐。**P0 的 D（存档注册表）+ H（属性注册表）+ K/L（去克隆/活动框架）是同一治理主线的四步**，做完这条主线，稳定性（少手工点=少漏）、可维护性（少复制=少行）、可扩展性（注册即扩展）三维一起受益。性能是相对独立的一条线，P0-F + P1-I 即可覆盖玩家可感知部分。

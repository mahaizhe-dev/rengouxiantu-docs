# 服务器掉线丢存档与黑市高峰问题完整方案框架

日期：2026-06-25  
适用基准：以最新 `origin/main` 代码为准  
关联细案：`docs/优化专题/2026-06-22-服务器掉线丢存档问题分析与第一批止血方案.md`  

## 1. 方案定位

本框架不是服务端大重构方案，也不是黑市整体模型重建方案。

目标是在当前已上线项目、现有 C/S 架构和风险边界内，把以下问题从“不可观测、不可恢复、容易放大”收敛到“可感知、可恢复、可验证、可回滚”：

1. 掉线频繁体感。
2. 弱网下断线感知滞后。
3. 存档失败、超时、锁冲突后保存意图可能丢失。
4. 排行榜/黑市读请求正常，但玩家实际存档链路不健康。
5. 黑市抢回收/抢售出时频繁出现“数据同步中，请稍后再试”。
6. 多人蹲守同一黑市商品时，服务端写压力和库存并发问题放大。

## 2. 约束边界

### 2.1 必须尊重的现有架构

1. 客户端仍是主要角色状态持有方。
2. 云端核心存档仍是 `save_{slot}` 全量快照。
3. 服务端只对部分业务具备权威能力，例如黑市库存、仙石、排行榜、部分活动结算。
4. 服务端断线回调不持有完整角色状态，不能凭空补写最后一份角色存档。
5. 现有网络状态统一走 `NetworkStatus`，第一批不新增第二套状态机。
6. 第一批不新增完整 heartbeat 协议。
7. 第一批不做事件日志型存档、不做字段级 CAS、不做服务端完整权威化。

### 2.2 必须保持的不变量

1. 保存成功 ACK 前，不清除 dirty。
2. `slot_locked`、`rate_limited`、timeout、no connection 属于临时失败，不能直接消耗保存意图。
3. 排行榜、黑市查询、历史记录等读请求成功，不能证明存档写链路健康。
4. 黑市购买、出售、兑换属于非幂等操作，断线重连后不得自动重放。
5. 服务端基于旧 `save_{slot}` 的整份写回，必须收敛为“重读最新快照 + 应用本次差量”。
6. 不能通过简单缩短自动保存间隔来掩盖问题。
7. 高峰期失败必须快速、明确、可重试，不能长时间 pending。

## 3. 问题域拆分

### 3.1 连接感知域

问题：

- 网络采样依赖 `SaveSystem.Update(dt)`。
- 多个玩法模式会跳过 `SaveSystem.Update(dt)`。
- 断线 UI、保存 timeout、重试计时可能一起暂停。

目标：

- 网络采样和保存 timeout 不再受玩法模式影响。
- 不改变原有自动保存策略。

### 3.2 保存意图域

问题：

- dirty 可能在实际保存成功前被清掉。
- 保存失败后，临时错误与真实失败没有清晰分层。
- 保存 timeout 和重试依赖同一更新入口，弱网下容易失效。

目标：

- dirty 表示“尚未确认落盘的保存意图”。
- ACK 成功才清 dirty。
- 临时失败保留 dirty 并退避重试。

### 3.3 服务端存档写入域

问题：

- `SaveWriteService` 遇到 `slot_locked` 直接失败。
- 黑市、奖励、活动等服务端写回可能基于旧快照。
- 服务端不具备完整角色内存状态，不能在断线时补最终快照。

目标：

- 第一批不做服务端 pending 队列。
- 先把客户端临时失败处理正确。
- 先修黑市已确认旧快照写回风险。

### 3.4 黑市交易域

问题：

- 买入、出售、兑换部分路径服务端已写存档，客户端成功后又触发一次全量保存。
- 出售路径的库存上限校验是普通读校验，不是原子上限保护。
- 多人同时抢同一商品时，出售链路会放大为多份 `save_{slot}` 写回和多次系统库存写入。

目标：

- 出售成功路径减少重复客户端保存。
- 同商品出售增加服务端短锁，先解决单实例并发穿透和写压力放大。
- 失败后查询刷新加短延迟，避免失败者同时打读请求。

### 3.5 黑市同步门域

问题：

- `BlackMarketSyncState` 的 `global_unsync` 会阻止黑市卖出。
- 仓库存取、消耗、SaveSession 未落盘都会触发该状态。
- 状态只在 `game_saved` 或 `game_loaded` 清除。
- 玩家连续点击会重复触发 `save_request + SaveSession.Flush()`。

目标：

- 保留同步闸，不放开旧快照卖出风险。
- 给同步触发加本地冷却。
- 保存成功后恢复出售。

### 3.6 限流与 pending 域

问题：

- 服务端统一返回 `S2C_RateLimited`。
- `client_main.lua` 只处理通用 toast 和保存中的 `SaveSystem.saving`。
- `BlackMerchantUI` 当前没有自己的限流处理，`pendingRequest_` 可能无法及时释放。

目标：

- 黑市 UI 订阅限流事件。
- 属于黑市事件时释放 pending。
- 不自动重试交易。

### 3.7 观测域

问题：

- 当前很难从日志串起一次保存或黑市交易的完整链路。
- 线上只能看到“掉线、同步中、保存失败”的体感。

目标：

- 第一批只加最小日志。
- 能区分 no connection、timeout、slot_locked、rate_limited、黑市热点失败、旧快照覆盖。

## 4. 分阶段路线

### Phase 0：基线确认

目的：

- 确认代码基准和当前行为，不做功能改动。

工作：

1. 确认当前分支已同步 `origin/main`。
2. 确认是否已有本地未提交变更。
3. 记录涉及文件：
   - `scripts/systems/SaveSystem.lua`
   - `scripts/systems/save/SavePersistence.lua`
   - `scripts/network/NetworkStatus.lua`
   - `scripts/network/SaveWriteService.lua`
   - `scripts/network/BlackMerchantHandler.lua`
   - `scripts/ui/BlackMerchantUI.lua`
   - `scripts/systems/BlackMarketSyncState.lua`
   - `scripts/systems/save/SaveSession.lua`
   - `scripts/server_main.lua`

交付：

- 基线说明。
- 不改代码。

### Phase 1：保存链路止血

目的：

- 先解决断线感知和保存意图丢失问题。

工作包：

1. `SaveSystem.UpdateCritical(dt)`：
   - 网络采样。
   - 连接事件派发。
   - 保存 timeout。
   - 重试计时。
2. `main.lua` 始终调用 critical update。
3. dirty ACK 后清除。
4. 临时失败保留 dirty 并退避。
5. 保存链路最小日志。

不做：

- 不放开所有玩法模式自动存档。
- 不新增 heartbeat。
- 不做服务端保存队列。

### Phase 2：黑市一致性止血

目的：

- 先修黑市最容易导致丢档和旧快照覆盖的路径。

工作包：

1. 黑市写回前重读最新 `save_{slot}`。
2. 在最新快照上应用买入、出售、兑换差量。
3. 出售前以最新背包为准重新校验可回售数量。
4. 最新背包不足时，整笔失败。
5. 不改变黑市买入物品不可回售规则。

不做：

- 不重构整个黑市商品模型。
- 不把所有经济系统一次性改成服务端权威。

### Phase 3：黑市高峰承载止血

目的：

- 降低抢回收/抢售出时的服务端压力和 UI pending 问题。

工作包：

1. `BlackMerchantUI` 处理 `S2C_RateLimited`。
2. `global_unsync` 触发保存增加 2-3 秒冷却。
3. `HandleSell` 增加同商品出售短锁。
4. 出售成功后不再立即触发客户端全量保存。
5. 灵韵→仙石成功后不再立即触发客户端全量保存。
6. 热点失败后的 `SendQuery()` 改成短延迟刷新。

不做：

- 不做黑市交易自动重放。
- 不做购买路径复杂队列。
- 不做分布式库存锁。

### Phase 4：观察与判定

目的：

- 用线上日志判断是否进入第二批。

观察指标：

1. 保存请求量。
2. 保存成功率。
3. 保存 timeout 次数。
4. `slot_locked` 次数。
5. `rate_limited` 次数。
6. 黑市 `global_unsync` 次数。
7. 黑市同商品短锁拒绝次数。
8. 黑市交易失败后查询次数。
9. 最后一次成功保存距当前时间。

### Phase 5：二阶段增强

只有日志证明必要时才进入。

候选项：

1. 服务端 `(userId, slot)` 最新保存合并队列。
2. 应用层 heartbeat。
3. 黑市分布式容量保护。
4. 其他服务端旧快照写回路径逐个治理。

## 5. 执行模块框架

### WP-01：关键 tick 拆分

文件：

- `scripts/systems/SaveSystem.lua`
- `scripts/main.lua`

输入：

- `dt`

输出：

- 连接状态事件。
- 保存 timeout 状态。
- retry 计时推进。

验收：

- 挑战/副本中断网仍能触发 `connection_lost`。
- 保存 timeout 不因玩法模式暂停。

### WP-02：dirty 生命周期修正

文件：

- `scripts/systems/SaveSystem.lua`
- `scripts/systems/save/SavePersistence.lua`

规则：

- dirty true：存在未确认保存意图。
- saving true：已有保存请求在路上。
- dirty in-flight：避免重复发。
- save ok：清 dirty。
- save fail/timeout/transient：保留 dirty。

验收：

- 无连接保存失败后 dirty 仍为 true。
- timeout 后 dirty 仍为 true。
- 保存成功后 dirty 才清。

### WP-03：临时失败分类

文件：

- `scripts/systems/save/SavePersistence.lua`
- `scripts/network/CloudStorage.lua`
- `scripts/network/SaveWriteService.lua`

临时失败：

- `slot_locked`
- `rate_limited`
- timeout
- no connection
- server busy

行为：

- 不累计为永久失败。
- 保留 dirty。
- 退避重试。

### WP-04：黑市写前重读最新存档

文件：

- `scripts/network/BlackMerchantHandler.lua`

模式：

1. 第一次读：只用于校验和构造操作意图。
2. 最终写前：重读最新 `save_{slot}`。
3. 在最新快照上应用本次差量。
4. 写回。

验收：

- 旧快照不会覆盖最新金币、灵韵、背包。

### WP-05：黑市高峰请求控制

文件：

- `scripts/ui/BlackMerchantUI.lua`
- `scripts/network/BlackMerchantHandler.lua`

内容：

- 限流释放 pending。
- 同步中 flush 冷却。
- 同商品出售短锁。
- 失败查询延迟。
- 减少重复客户端保存。

验收：

- 10 人抢同商品时，服务端进入完整出售链路的请求数受控。
- UI 不会长期 pending。

### WP-06：最小日志

文件：

- `scripts/systems/SaveSystem.lua`
- `scripts/systems/save/SavePersistence.lua`
- `scripts/network/SaveWriteService.lua`
- `scripts/network/BlackMerchantHandler.lua`
- `scripts/ui/BlackMerchantUI.lua`

日志字段：

- requestId。
- userId。
- slot。
- itemId。
- action。
- reason。
- duration。
- dirty 状态。
- transient 分类。

禁止：

- 不输出完整存档。
- 不输出敏感玩家数据。

## 6. 验收矩阵

| 问题 | 验收场景 | 通过标准 |
| --- | --- | --- |
| 玩法中断线不感知 | 挑战/副本中断网 | 9-15 秒内进入断线状态 |
| 保存 timeout 被暂停 | 玩法中保存无 ACK | timeout 正常推进 |
| dirty 丢失 | no connection / timeout / slot_locked | dirty 保留 |
| 排行榜正常但保存失败 | 排行榜读成功、保存写 timeout | UI 仍提示保存风险 |
| 黑市旧快照覆盖 | 黑市操作并发本地状态变化 | 最新状态不被旧快照覆盖 |
| 黑市抢回收同步中 | 仓库取出后立刻卖出 | 只触发一次保存 flush，保存成功后可卖 |
| 黑市多人出售 | 10 人同商品出售 | 同实例内完整出售链路受短锁保护 |
| 黑市限流 pending | 服务端返回 `S2C_RateLimited` | UI 释放 pending |
| 失败查询风暴 | 10 个失败回包 | 查询短延迟合并，不立即打满 |

## 7. 回滚框架

每个工作包必须独立提交，方便回滚。

| 工作包 | 回滚方式 | 风险说明 |
| --- | --- | --- |
| WP-01 critical tick | 回滚 `main.lua` 调用和新增函数 | 只影响感知和 timeout |
| WP-02 dirty 生命周期 | 回滚 dirty/in-flight 修改 | 回滚后恢复旧风险 |
| WP-03 临时失败分类 | 回滚错误分类 | 可能重新放大失败 |
| WP-04 黑市重读最新 | 回滚黑市 helper | 不影响基础保存 |
| WP-05 黑市高峰控制 | 回滚 UI/短锁/延迟刷新 | 高峰压力恢复旧状态 |
| WP-06 日志 | 可独立关闭或删除 | 不应影响业务 |

## 8. 禁止清单

执行端不得在第一批做以下事情：

1. 不新增完整 heartbeat 协议。
2. 不做服务端完整角色权威化。
3. 不自动重放黑市购买、出售、兑换。
4. 不把排行榜成功当成在线/保存健康。
5. 不把自动保存间隔简单调短作为主方案。
6. 不做全项目经济系统重构。
7. 不做全项目 UI 写入口一次性大审计。
8. 不引入不可回滚的大型抽象。

## 9. 二阶段进入条件

### 9.1 进入服务端保存合并队列的条件

满足任一：

- `slot_locked` 占保存失败比例高。
- 同用户短时间多次保存被锁拒绝。
- 保存请求数量明显超过实际成功写入数量。

### 9.2 进入 heartbeat 的条件

满足任一：

- `GetServerConnection()` 非空但保存/黑市写请求长期 timeout。
- 客户端在线状态明显滞后于业务写失败。

### 9.3 进入黑市分布式容量保护的条件

满足任一：

- 单实例短锁上线后仍出现库存超过 `maxStock`。
- 多实例部署下同商品出售并发穿透仍存在。

### 9.4 进入更多旧快照治理的条件

满足任一：

- 日志证明非黑市路径也在旧快照覆盖新状态。
- 特定活动/奖励路径持续产生丢档投诉。

## 10. 最终目标状态

第一批完成后，项目应达到：

1. 断线能被及时感知。
2. 保存失败不会直接丢保存意图。
3. 黑市卖出不再依赖旧快照写回。
4. 抢回收高峰不会把服务端放大成全量保存风暴。
5. 黑市 UI 不会因为限流或 pending 长时间卡住。
6. 线上日志能定位主要失败原因。

第一批不承诺：

1. 完全消灭网络掉线。
2. 完成服务端权威化。
3. 完成所有系统一致性治理。
4. 完成断线重连后的业务恢复。

这四项必须等第一批日志证明必要性后，再作为第二批单独评估。

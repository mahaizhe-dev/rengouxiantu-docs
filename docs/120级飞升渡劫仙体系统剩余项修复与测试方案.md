# 120级飞升渡劫与仙体系统务实剩余项与测试方案

> 基线代码：`ee4d11f update: 飞升/渡劫/仙体系统逻辑完善 + 存档DTO + Player迭代`
>
> 本文按项目当前真实架构整理，不追求理想化强一致交易。原则是：不刷资源、不重复扣费、不引入明显线上回档/卡死风险；与项目已有保存、购买、突破范式保持一致。

## 1. 已确认口径

1. 仙体切换按“切换后保存并退出登录”的既定方案执行。
   - 仙体解锁是账号级；每个角色的当前仙体、切换消耗、属性生效是角色级。
   - 二次确认后，只写切换意图 `pending` 并扣除灵韵，不立即应用新仙体属性。
   - 保存成功后退出登录/选角，重新进入角色时应用 `pending` 并按目标仙体重算属性。
   - 保存失败应回到本地点击前状态，不扣灵韵、不改 `activeBodyId`、不改属性。
   - 不接受通过“加回 1000 灵韵”的增量退款实现，必须用快照恢复，避免刷资源。

2. 剑气长城兑换不做完整 WAL 补偿。
   - 剑气积分在服务端 Money/Score，仙劫丹在角色背包存档，这是跨存储链路。
   - 项目当前其他购买/进阶并没有完整补偿系统。
   - 已确认本轮只做防刷、防重复扣、防断线/重试多扣。
   - 不做“积分已扣但本地发奖/保存失败后的自动补发”。
   - 扣积分成功后客户端尽快发奖并保存；保存失败时锁住本次购买状态，提示保持在线等待后续保存。

3. 仙阶吃丹沿用洗练式关闭/切页保存口径。
   - 不要求每吃一颗都保存。
   - 关闭界面/离开 tab 保存可以接受。
   - 本轮不接入 `SaveSession` 的 5 次/20 秒收口，除非后续重新确认。

4. 渡劫中保存退出不算失败。
   - 不触发 10 分钟 CD。
   - 保存后退出，重登应能重新渡劫。
   - 关键是保存前不能把 `tribState = "inProgress"` 写入云端。

5. 复杂可靠发货 WAL、全属性公式化重构、账号资产强一致系统、吃丹高频保存收口都不放入本轮必做。

## 3. M1：服务端识别 `asc_N`

### 问题

当前 `asc_N` 是 `AscensionConfig.EnsureRealmsRegistered()` 动态注册到 `GameConfig.REALMS` 的。客户端部分路径会调用，但服务端 `SaveWriteService.lua`、`RankHandler.lua` 等只 require `GameConfig`。

如果服务端没注册：

1. `player.realm = "asc_1"` 可能被排行榜判定为非法境界。
2. 保存校验无法按仙阶 `maxLevel` 检查。
3. 槽位摘要可能显示成“凡人”。

### 务实修法

新增或直接使用一个轻量入口：

```lua
local AscensionConfig = require("config.AscensionConfig")
AscensionConfig.EnsureRealmsRegistered()
```

必须放在这些文件的模块加载阶段：

1. `scripts/network/SaveWriteService.lua`
2. `scripts/network/RankHandler.lua`
3. `scripts/network/SlotReadService.lua`
4. `scripts/systems/save/SaveSlots.lua`
5. `scripts/systems/save/SaveMigrations.lua`
6. `scripts/network/SaveSystemNet.lua` 的导入/重建排行路径

不需要抽象过度。可以先直接 require + Ensure，后续再收口成 `AscensionRealmRegistry`。

### 验收

1. `asc_1 Lv121` 可以保存，不被服务端拒绝。
2. `asc_10 Lv141` 不被 clamp 到 140。
3. 排行榜 score 高于 `dacheng_4 Lv120`。
4. 槽位显示为 `谪仙1阶`，不是 `凡人`。

## 4. M2：仙体切换防刷与保存失败处理

### 问题

当前流程有风险：

1. 点击确认后扣灵韵。
2. 立即 `ApplyPending()` 改属性并清 pending。
3. 再保存。
4. 保存失败只提示，但本地已成功。

这会带来两个问题：

1. 保存失败后，本地状态和云端状态不一致。
2. 如果用简单 `+1000 灵韵` 回滚，可能被重复回调或并发点击刷资源。

### 执行流程

保存 `pending`，保存成功后退出登录，重登应用。

1. 点击确认：
   - 检查灵韵。
   - 记录 rollback 快照。
   - 扣灵韵。
   - 写 `pending`。
   - 不立即改 `activeBodyId`，不立即改属性。

2. 强保存：
   - 成功：退出到登录/选角，下次加载时 `ApplyPending()`。
   - 失败：按快照恢复本地状态。

3. 加载后：
   - `ApplyPending()` 根据 `fromBodyId -> targetBodyId` 计算差量。
   - 改 activeBody、属性。
   - 清 pending。
   - 发一次 `save_request`，尽快把 pending 已消费状态落盘。

### 回滚要求

不能写：

```lua
player.lingYun = player.lingYun + cost
```

必须写快照恢复：

```lua
player.lingYun = snapshot.lingYun
charState.pending = snapshot.pending
charState.activeBodyId = snapshot.activeBodyId
player.maxHp = snapshot.maxHp
player.atk = snapshot.atk
player.def = snapshot.def
player.hpRegen = snapshot.hpRegen
```

同时加：

1. `switching = true` 时禁用按钮。
2. 每次切换有 `txnId`。
3. 保存回调只处理当前 `txnId`。
4. 回滚只能执行一次。

### 验收

1. 保存失败不扣灵韵，不改仙体，不改属性。
2. 保存成功后重登才生效。
3. 反复点击不会重复扣。
4. 重复保存回调不会重复回滚。
5. 切换仙体后升级，成长使用当前仙体模板。

## 5. M3/M4：剑气长城兑换最低一致性

### 当前接受的边界

本轮不做完整自动补发。也就是说：

1. 不做 pending reward WAL。
2. 不承诺极端断线后自动补发。
3. 不把剑气长城改成完整可靠交易系统。

但必须做到：

1. 不能刷物品。
2. 不能重复扣积分。
3. 不能断线/重试多扣。
4. 扣积分和限购次数必须在服务端同事务内完成。
5. 客户端拿到奖励后要尽快保存。

### 角色级限购

当前 key 类似：

```lua
sw_shop_bought_one_turn_tribulation_pill
```

需要改为：

```lua
sw_shop_bought_slot1_one_turn_tribulation_pill
```

slot 来源优先使用服务端当前连接记录的 active slot，不能只信客户端传参。

### 防重复扣

客户端购买请求带 `clientTxnId`：

```lua
{
    itemId = "one_turn_tribulation_pill",
    clientTxnId = "slot1_时间戳_随机数"
}
```

服务端记录短期幂等结果：

```lua
sw_shop_txn_slot1_<clientTxnId>
```

处理规则：

1. 如果 txn 已成功处理，重复请求只返回上次结果，不再扣积分。
2. 如果 txn 正在处理，返回“处理中”，客户端不允许继续点。
3. 如果 txn 不存在，才执行扣积分和加限购。

这个不是完整补偿，只是防止重复点击、网络重试导致多扣。

### 客户端保存

收到成功响应后：

1. 立即入包。
2. 立即 `SaveSystem.RequestImmediateSave("sword_wall_shop")` 或 `SaveSystem.Save()`。
3. 保存未完成前，该商品按钮显示“保存中”，不允许继续购买。
4. 保存失败时提示：
   - “兑换已处理，当前保存失败，请保持在线等待自动保存”
   - 不要允许玩家反复点击同商品。

### 验收

1. 同一 txn 重发 3 次，只扣 1 次积分。
2. 同一角色买满 10 颗后不能买第 11 颗。
3. 角色 A 买满 10 颗，角色 B 仍可买自己的 10 颗。
4. 积分不足时不增加限购次数。
5. 客户端保存失败时不允许立刻再次购买导致多扣。

## 6. M5：渡劫保存退出

### 设计口径

保存退出不是失败。

结果应为：

1. 不进入 CD。
2. 不扣进度。
3. 重登后可重新渡劫。

### 当前风险

如果保存时 `tribState = "inProgress"` 被写入云端，重登时 `PostLoadCheck()` 会把它当断线失败，转成 cooldown。

### 务实修法

保存前执行专用取消：

```lua
function TribulationSystem.CancelForSaveExit()
    if state.tribState ~= "inProgress" then return false end
    state.tribState = "none"
    state.startedAt = 0
    state.runId = ""
    state.lastResult = "save_exit"
    state.cooldownUntil = 0
    return true
end
```

`TribulationScene` 增加只回滚场景、不触发失败的退出：

```lua
function TribulationScene.ExitForSaveExit()
    -- 回滚地图、玩家位置、清特效
    -- 不调用 OnFail
    TribulationSystem.CancelForSaveExit()
end
```

`SystemMenu.DoSaveAndExit()` 调 `SaveSystem.Save()` 前先执行：

```lua
if TribulationScene.IsActive() then
    TribulationScene.ExitForSaveExit()
end
```

`ReturnToLogin()` 顶部也加同样防御，但不能只依赖这里，因为它通常发生在保存之后。

### 验收

1. 渡劫中保存退出，重登后无 CD。
2. 仙阶进度仍是满的。
3. 点击渡劫台可重新进入。
4. 渡劫中死亡仍进入 10 分钟 CD。
5. 非正常断线重登仍进入 10 分钟 CD。

## 7. 吃丹保存口径

已确认本轮沿用洗练式“关闭界面/离开 tab 保存”。

执行要求：

1. 吃丹成功后更新本地仙阶进度和背包消耗。
2. 不要求每颗丹都立即保存。
3. 离开仙阶 tab 或关闭 RealmPanel 时触发保存。
4. 进度满后不能继续吃丹；只有最后一颗丹允许溢出，最大理论溢出 4 点。
5. 小境界突破、进入渡劫台前，应先确保当前本地状态不会被旧界面覆盖。

本轮不做：

1. 不接入 `SaveSession` 的 5 次/20 秒收口。
2. 不做每颗丹立即保存。
3. 不做断线后自动恢复未保存吃丹进度。

# 端午世界掉落活动（设计方案）

> 基准代码：`origin/main` 已落地的六一活动（`childrensday_2026`）
>
> 参考文档：`五一世界掉落活动.md` / `六一世界掉落活动（试行方案）.md` / `六一世界掉落活动（代码执行文档）.md`
>
> 当前口径：六一活动已结算，端午活动在六一骨架上切换；本方案按现有代码结构收口，不额外重构活动主链。

## 1. 基准结论

基于当前 `origin/main`，活动与相关系统现状如下：

1. 六一活动已经是完整的双掉落物 + 双宝箱 + 排行 + 保底骨架，不再是五一的单福袋版本。
2. 现有活动核心链路已经拆成以下结构：
   - `scripts/config/EventConfig.lua`：活动配置
   - `scripts/systems/loot/rewards.lua`：世界掉落追加
   - `scripts/network/EventHandler.lua`：服务端开箱 / 排行 / 记录 / 保底
   - `scripts/ui/EventExchangeUI.lua`：活动面板
   - `scripts/systems/EventSystem.lua`：`event_data` 本地缓存与保底进度
3. 六一当前已经处于“收尾态”：
   - `EventConfig.ACTIVE_EVENT.enabled = false`
   - `GameConfig` 里的 `childday_*` 文案已经改成“活动已结束，可以出售换金币”
4. 普通世界 BOSS 总击杀当前仍是“客户端累计 + 存档同步”：
   - `scripts/systems/GameEvents.lua` 中击杀 BOSS 时累加 `GameState.bossKills`
   - `saveData.bossKills` / `boss_kills_{slot}` 已随存档同步到云端
5. 宠物大奖皮肤系统已经存在：
   - `pet_premium_wuxing` 已在 `scripts/config/PetAppearanceConfig.lua` 中配置完成
   - 但当前 `hidden = true`，且未进入 `SkinShopConfig.SKIN_IDS`
6. 第一种“可手动食用且可上黑市”的属性丹药当前尚不存在，现有代码只有：
   - 普通批量使用型消耗品
   - 炼丹 / 挑战 / 封妖链路产出的永久属性丹
   - 黑市 `B 类` 商品保护锁

## 2. 六一收尾原则

端午活动上线前，六一按以下口径处理：

1. 六一活动保持关闭，不再继续使用 `childrensday_2026`。
2. 六一历史排行、开奖记录、保底计数保留，不清表。
3. 六一道具继续保留“活动已结束”描述，玩家库存不做强删。
4. 端午活动必须使用全新 `eventId`、全新排行 key、全新开奖记录 key，禁止复用六一 key。

## 3. 端午活动总规则

### 3.1 活动 ID

- `eventId = "dragonboat_2026"`

### 3.2 玩法结构

端午活动整体结构与六一保持一致：

1. BOSS 世界掉落 2 种开启物。
2. 开启物分别对应小宝箱 / 大宝箱。
3. 保留积分排行、全服开奖记录、双保底机制。
4. 不开启兑换系统。
5. 新增一个服务端校验的一次性 BOSS 击杀里程碑任务页。
6. 新增一种可食用、可黑市交易、可进奖池的稀有掉落物 `仙界精品粽`。

### 3.3 与六一相比的明确改动

这轮不是纯“换皮”，端午相对六一有 5 个明确变化：

1. 两种开启物改为：
   - `dragonboat_colored_rope`
   - `dragonboat_sachet`
2. 小宝箱大奖不再是“大宝箱开启物”，改为 `仙界精品粽`。
3. 大宝箱奖池中新增 `仙界精品粽`。
4. 大宝箱大奖皮肤由六一的福缘皮肤改为悟性皮肤 `pet_premium_wuxing`。
5. 新增 10 档一次性里程碑任务，每档奖励 `dragonboat_sachet ×1`。

这意味着：

1. 世界掉落的“双开启物结构”仍与六一一致。
2. 但奖池循环不再完全等同六一，小宝箱不再负责产出香囊大奖位。
3. 香囊的主要供给将回到“高境界 BOSS 掉落 + 里程碑奖励”。

## 5. 世界掉落与奖池设计

### 5.1 世界掉落

端午活动的前两段掉落逻辑，完全复用六一当前实现方式：

1. 第一段：按 `chapter + monster.category` 掉 `dragonboat_colored_rope`
2. 第二段：按 `monsterOverrides > realmDropRates` 掉 `dragonboat_sachet`

实现上直接做同构替换：

1. `childday_rattle -> dragonboat_colored_rope`
2. `childday_pinwheel -> dragonboat_sachet`

### 5.2 仙界精品粽的独立掉落

`仙界精品粽` 作为第三段独立掉落，不影响前两段判定。

判定规则：

1. 仅 BOSS 参与，不含普通怪和精英怪。
2. 仅 `realm >= yuanying_1` 的 BOSS 参与。
3. 四仙剑保留单独覆盖口径。
4. 与彩绳/香囊互不排斥，允许同一只 BOSS 在一次击杀中同时掉落开启物和精品粽。

建议概率表：

| 档位 | 掉率 |
| --- | --- |
| `yuanying_*` | `0.0005` |
| `huashen_*` | `0.0008` |
| `heti_*` | `0.0010` |
| `dacheng_*` | `0.0012` |
| `dujie_*` | `0.0015` |
| 四仙剑覆盖 | `0.0020` |

这套数值的目标是：

1. 保持明显稀有于香囊。
2. 允许高强度刷 BOSS 的玩家在活动期内看到产出。
3. 不会因为又进宝箱、又进黑市，而过快把 `福缘 +10` 上限打满。

### 5.3 彩绳宝箱调整

小宝箱这轮按你的新口径处理：

1. 小宝箱不做额外补权重调整
2. 除大奖槽位外，其余奖励项与权重保持 `origin/main` 当前六一实装版现状
3. 只替换大奖槽位：

4. `xianjie_premium_zong` 在奖池试算中按 `1000 灵韵` 估值
5. 小池大奖权重改为：
   - `weight = 10`
6. 小宝箱总权重（含辟邪香囊稀有槽 weight=20）：
   - `1000`
7. 小池保底 120 次出精品粽：

### 7.2 “同步黑市商店放出”的代码口径

这里需要明确一个基于现代码的口径：

1. 你说的“黑市商店放出”，代码上对应的不是 `BlackMerchantConfig.lua`
2. 宠物皮肤走的是独立的：
   - `scripts/config/SkinShopConfig.lua`
   - `scripts/network/SkinShopHandler.lua`
   - `scripts/ui/SkinShopUI.lua`

也就是说，悟性皮肤同步放出时，应该做的是：

1. `pet_premium_wuxing.hidden = false`
2. 把 `pet_premium_wuxing` 加入 `SkinShopConfig.SKIN_IDS`
3. 让大奖开出和皮肤商店购买同时成立

### 7.3 价格边界

当前皮肤商店代码是统一定价：

- `SkinShopConfig.SKIN_PRICE = 1000`

所以如果只是把悟性皮肤加入现有商店列表，它会自动按 `1000 仙石` 售卖。

如果你希望：

1. 悟性皮肤比现有 3 款更贵
2. 或者活动期单独定价

那就不能只改配置列表，还要把 `SkinShopConfig` 和 `SkinShopHandler` 从“全局统一价”改成“按皮肤单独价格”。

## 8. 仙界精品粽的细致准备

### 8.1 为什么它必须单独做

`仙界精品粽` 是当前代码里第一种同时满足以下条件的道具：

1. 背包中可手动食用
2. 食用后给永久属性
3. 有单角色食用上限
4. 允许上架黑市

这 4 条叠在一起后，它不再是普通活动物，而是典型的黑市 `B 类商品`。

### 8.2 存档字段建议

推荐不要只加一个 `pillFortune`，而是做成“属性字段 + 次数字段”双字段：

1. `player.pillFortune`
2. `player.premiumZongEaten`

推荐关系：

1. 每食用 1 个 `仙界精品粽`
2. `player.premiumZongEaten += 1`
3. `player.pillFortune += 1`
4. 两者上限都为 `10`

这样做的原因：

1. 现有永久丹药链路已经有“次数字段 + 属性字段”的校验惯例
2. 后续 `SaveLoader` 才能做反推校验
3. 出现异常存档时，能检查：
   - `pillFortune == premiumZongEaten`

如需继续沿用现有 `pillCounts` 体系，可额外镜像：

```lua
player.pillCounts.zong = player.premiumZongEaten
```

但 `pillCounts` 只能作为辅助镜像，不建议代替顶层计数字段。

### 8.3 为什么不能复用 `fruitFortune`

不建议把精品粽直接记到 `fruitFortune`：

1. `fruitFortune` 是既有福缘果系统专属字段
2. 当前 `SaveWriteService` 已对 `fruitFortune` 做上限校验
3. 基于 `origin/main` 当前代码，`fruitFortune` 上限已经写成 `40`
4. 强行复用会把“地图采集收益”和“活动丹药收益”混在一起

因此端午应新增独立字段，不污染旧系统。

### 8.4 使用链路建议

基于 `origin/main` 当前实现，`仙界精品粽` 必须走专用使用链路：

1. `scripts/systems/inventory/batch_use.lua` 新增 `xianjie_premium_zong` 分支
2. `scripts/ui/EquipTooltip.lua` 把它加入“批量使用按钮”名单
3. 使用时通过 `ConsumeConsumable()` 扣除
4. 按 `10 - premiumZongEaten` 自动夹紧本次可食用数量
5. 食用成功后更新：
   - `player.premiumZongEaten`
   - `player.pillFortune`
6. UI 提示要明确显示：
   - `已食用 3/10`
   - `已达食用上限`

### 8.5 为什么 `GameConfig` 里的 consumable category 不能写成 `event`

基于 `origin/main` 当前代码：

1. `batch_use.lua` 里，`cfgData.category == "event"` 会直接走 `_SellBatchConsumable`
2. 也就是它会被系统当成“活动卖钱物”
3. 而不是“可手动食用丹药”

所以：

1. `BlackMerchantConfig` 里的黑市分类可以是活动类
2. 但 `GameConfig.CONSUMABLES["xianjie_premium_zong"]` 的道具分类不能走这条老 `event` 语义

建议单独定义为普通消耗品，并走专用使用逻辑。

### 8.6 黑市接入与保护锁

`仙界精品粽` 既可使用又可黑市交易，必须按 `B 类商品` 处理。

本轮至少要满足：

1. 黑市配置显式接入，不能只写 `EventConfig.blackMarketItems` 参考表
2. `buy_price = 10`
3. `sell_price = 20`
4. `max_stock = 50`
5. 食用口必须走 `ConsumeConsumable()`，从而自动受黑市锁保护
6. 不能出现“黑市锁定中的精品粽仍可被直接食用”

补充口径：

1. `dragonboat_colored_rope` 与 `dragonboat_sachet` 不额外提高黑市库存上限
2. 两者继续沿用活动常规库存策略即可
3. 只对 `xianjie_premium_zong` 单独设置 `max_stock = 50`

### 8.7 角色信息面板接入

`仙界精品粽` 不仅要影响总福缘，还要在角色信息面板里可见，且能让玩家看清数据来源。

建议同时接入两处：

1. `CharacterUI` 属性页的福缘板块
   - 展示总福缘时，`pillFortune` 已计入 `GetTotalFortune()`
   - 额外增加一行说明：`仙界精品粽福缘 +N`
2. `CharacterUI` 丹药/成长统计区
   - 增加 `仙界精品粽`
   - 展示 `已食用 X/10`
   - 文案说明 `福缘+1`

这样做的原因：

1. 玩家能看到“总福缘已经生效”
2. 玩家也能看到“精品粽到底吃了几个”
3. 后续如果出现存档异常，运营和测试更容易核账

## 9. 服务端校验与存档设计

### 9.1 里程碑校验源

端午里程碑的服务端校验源，直接使用现有持久化链路：

1. `saveData.bossKills`
2. `boss_kills_{slot}`

服务端以 `saveData.bossKills` 为准，不信任客户端直接传入的击杀数量。

### 9.2 推荐存档结构

建议把里程碑领取状态并入现有 `event_data`，不要新开顶层存档块：

```lua
saveData.event_data["dragonboat_2026"] = {
    pity = {
        small = 0,
        big = 0,
    },
    bossMilestones = {
        claimed = {
            ["300"] = true,
            ["600"] = true,
        }
    }
}
```

### 9.3 协议建议

建议继续挂在 `EventHandler.lua`，不要新起独立 Handler。

新增协议：

1. `C2S_EventClaimBossMilestone`
2. `S2C_EventClaimBossMilestoneResult`

请求字段：

```lua
{
    milestone = 300
}
```

服务端处理顺序：

1. 校验活动激活且 `eventId == dragonboat_2026`
2. 获取 `userId + slot`
3. 加 `slot lock`
4. 读取 `saveData`
5. 读取 `saveData.bossKills`
6. 读取 `event_data.dragonboat_2026.bossMilestones.claimed`
7. 未达标则拒绝
8. 已领取则拒绝
9. 发放 `dragonboat_sachet ×1`
10. 写回 `claimed[milestone] = true`
11. `BatchCommit`

### 9.4 时效性边界

这里要明确一个基于现有代码的现实边界：

1. 普通世界 BOSS 击杀数当前不是服务端实时累加
2. 服务器能读到的是“最近一次已存档的 `bossKills`”

所以本方案虽然是“服务端校验领取”，但它的实时性依赖存档同步。

推荐领取流程：

1. 客户端点“领取”前，先触发一次 `save_request`
2. 存档返回或短延迟后，再发 `C2S_EventClaimBossMilestone`

## 12. 当前已确认结论

本轮按你已经拍板的规则，当前设计收口如下：

1. 端午双开启物定为：
   - `dragonboat_colored_rope`
   - `dragonboat_sachet`
2. 黑市价格定为：
   - 彩绳 `1 / 2`
   - 香囊 `6 / 12`
   - 精品粽 `10 / 20`
3. 精品粽黑市库存定为 `50`
4. 里程碑 10 档奖励固定 `dragonboat_sachet ×1`
5. 端午大奖皮肤定为 `pet_premium_wuxing`
6. 悟性皮肤重复补偿定为 `5000 灵韵`
7. 悟性皮肤同步进入皮肤商店放出
8. 彩绳宝箱大奖不再是香囊，改为 `仙界精品粽`
9. 香囊宝箱奖池中加入 `仙界精品粽`
10. 精品粽是第一种“可手动食用的属性丹药”，单角色最多食用 `10` 个
11. 精品粽仍保留高阶 BOSS 稀有独立掉落身份
12. 精品粽需要加入角色信息面板，既显示福缘贡献，也显示食用进度
13. 彩绳、香囊黑市库存不上调，维持常规库存策略

## 13. 交付执行默认值

为了直接交由执行 AI 开发，本方案按以下默认值收口：

1. 悟性皮肤 `pet_premium_wuxing` 同步进入皮肤商店，沿用现有统一价 `1000 仙石`
2. 悟性皮肤重复补偿固定为 `5000 灵韵`
3. `仙界精品粽` 在奖池平衡试算中按 `1000 灵韵` 处理
4. 小宝箱里的 `small_zong` 权重按 `10` 执行；除大奖槽位替换外，其余小宝箱奖励项与权重不调整，因此小宝箱总权重为 `990`
5. 大宝箱里的 `big_zong` 权重按 `20` 执行
6. 大宝箱里除皮肤外，其余单项权重不得低于 `20`
7. 为保持大宝箱总权重 `1000`，建议按以下方式调整：
   - `金币 ×20000: 170 -> 150`
   - `金币 ×30000: 90 -> 75`
   - `金砖 ×1: 15 -> 20`
   - `直接灵韵 ×400: 10 -> 20`

若后续需要改价或改权重，再按新的策划口径单独调整即可。

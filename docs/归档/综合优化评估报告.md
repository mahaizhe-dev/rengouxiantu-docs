## 二、评估维度与评分

### 评分标准
- **A (90-100)**: 生产级质量，可直接扩展
- **B (75-89)**: 良好，有小问题但不影响运营
- **C (60-74)**: 合格，存在结构性风险需关注
- **D (40-59)**: 需要重构，已影响开发效率
- **F (<40)**: 严重问题，影响线上稳定性

### 2.1 代码质量 — 评分: **B- (72)**

#### 优点
1. **模块化分层清晰**：config/systems/network/ui/rendering 五层分离，职责边界明确
2. **命名规范一致**：文件命名（XxxSystem/XxxConfig/XxxUI/ServerXxx）全局统一
3. **存档系统工程质量高**：8 子模块 Facade 模式，类型校验 + checksum + 原子写入 + 4级恢复链 + 版本迁移，是整个项目质量最高的模块
4. **配置数据驱动**：所有游戏数值集中在 config/ 层，系统层纯逻辑

#### 问题

| 严重度 | 问题 | 位置 | 影响 |
|--------|------|------|------|
| **HIGH** | 7 个文件超 1000 行（最大 1,755 行）| Server.lua, ServerCombat.lua, ServerQiankunFurnace.lua, Client.lua, QiankunFurnaceSystem.lua, QiankunFurnaceUI.lua, client_main.lua | 可维护性下降，合并冲突风险 |
| **HIGH** | client_main 和 server_main 重复 24 行相同的 require | 两个入口文件 | 新增模块时必须同步两处，易遗漏 |
| **MED** | 乾坤炉（QiankunFurnace）横跨所有层且每层都是最大文件 | System 1,192 + UI 1,087 + Server 1,353 = 3,632行 | 单一功能占项目 12.6%，修改风险集中 |
| **MED** | SaveKeys.CURRENT_VERSION = 3 vs rules.md 记载 v17 | save/SaveKeys.lua vs 架构文档 | 新版存档系统未与线上迁移链对齐 |
| **LOW** | `SaveRecovery._tryPartialRecover` 中布尔正则使用 PCRE 语法 `(true|false)` | save/SaveRecovery.lua | Lua 模式不支持 `|`，布尔字段永远无法部分恢复 |

#### 建议
1. **提取共享初始化模块**：将 24 行重复 require 提取为 `shared_init.lua`，client/server 各只需 `require "shared_init"`
2. **拆分乾坤炉**：ServerQiankunFurnace 按子功能（炼制/材料/配方/事件）拆分为 2-3 个文件
3. **修复 SaveRecovery 正则**：将 `(true|false)` 改为两次独立 `string.match`

### 2.2 章节框架扩展性 — 评分: **B+ (82)**

#### 架构分析

```
config/ChapterConfig.lua (424行)
    ↓ 数据驱动
systems/ChapterSystem.lua (407行)
    ↓ 网络协议
network/ServerChapter.lua (520行) + network/Client.lua (章节相关部分)
    ↓ UI 展示
ui/ChapterUI.lua (490行)
```

#### 优点
1. **数据驱动设计**：章节/关卡/波次/敌人配置集中在 ChapterConfig，新增章节只需添加配置
2. **系统-网络分离**：ChapterSystem 负责逻辑（解锁、奖励、进度），ServerChapter 负责验证，职责清晰
3. **配置量适中**：424行配置 + 407行逻辑，新开发者可快速理解

#### 问题

| 严重度 | 问题 | 影响 |
|--------|------|------|
| **MED** | 怪物数据分散在 MonsterData(1,975行) + MonsterTypes_ch3(1,922行) 等多个文件 | 章节引用怪物时跨文件查找，新章节需知道去哪个文件找模板 |
| **LOW** | ChapterUI 与 ChapterSystem 紧耦合（直接 require 调用） | UI 无法脱离系统独立测试 |

#### 扩展成本估算
- **新增一个章节**：ChapterConfig 添加配置 + MonsterData/MonsterTypes 添加怪物模板 = ~100-200行配置
- **新增章节机制**（如 Boss 关卡、分支路线）：ChapterSystem 添加逻辑 + ServerChapter 添加验证 + ChapterUI 添加展示 = 3 个文件修改

#### 建议
1. **统一怪物数据源**：考虑将 MonsterTypes_ch3 等合并或建立统一索引，减少查找成本
2. **章节模板化**：提供标准章节配置模板，新章节只需填写差异部分

### 2.3 战斗框架扩展性 — 评分: **B (78)**

#### 架构分析

```
config/CombatConfig.lua (215行) ← 战斗参数配置
    ↓
systems/CombatSystem.lua (917行) ← 战斗核心逻辑
    ↓
network/ServerCombat.lua (1,421行) ← 服务端战斗验证
    ↓
rendering/CombatRenderer.lua (538行) ← 战斗渲染
```

#### 优点
1. **配置-逻辑分离**：CombatConfig 定义参数，CombatSystem 执行计算，修改数值无需改逻辑
2. **公式集中管理**：伤害/暴击/闪避/元素克制等公式集中在 CombatSystem
3. **服务端权威**：所有战斗判定在 ServerCombat 执行，客户端仅展示

#### 问题

| 严重度 | 问题 | 影响 |
|--------|------|------|
| **HIGH** | ServerCombat.lua 1,421行，战斗逻辑+AI+技能+Buff全部耦合 | 新增技能类型需在单一大文件中修改，风险高 |
| **MED** | CombatSystem(917行) 同时包含回合制逻辑、伤害计算、Buff处理 | 职责过多，应拆分为 DamageCalculator + BuffProcessor + TurnManager |
| **MED** | 世界Boss (WorldBoss) 战斗逻辑独立于 CombatSystem | 两套战斗逻辑可能数值不一致 |

#### 扩展成本估算
- **新增技能类型**：CombatConfig 添加配置 + CombatSystem 添加处理分支 + ServerCombat 添加验证 = 3 文件修改
- **新增 Buff 效果**：CombatSystem 修改 Buff 处理逻辑 + ServerCombat 同步 = 2 文件修改
- **新增战斗模式**（如 PvP）：需要复制大量 ServerCombat 代码或重构为策略模式

#### 建议
1. **拆分 ServerCombat**：至少分为 `ServerCombatCore.lua`（回合调度）+ `ServerCombatSkill.lua`（技能执行）+ `ServerCombatBuff.lua`（Buff处理）
2. **统一战斗入口**：WorldBoss 战斗应复用 CombatSystem 而非独立实现
3. **Buff 数据驱动化**：将 Buff 模板从代码硬编码改为 Config 配置

### 2.4 存档系统 — 评分: **A- (88)**

#### 架构分析

```
SaveSystem.lua (门面, 185行)
    ├── SaveLoader.lua     (流程编排, 184行)
    │   ├── SaveState.lua     (运行时CRUD, 245行)
    │   ├── SaveRecovery.lua  (4级恢复, 197行)
    │   │   ├── SavePersistence.load/loadBackup
    │   │   ├── SaveSerializer.deserialize
    │   │   ├── SaveMigrations.migrate
    │   │   └── SaveKeys.getDefaults
    │   ├── SavePersistence.lua (文件I/O, 220行)
    │   │   └── SaveSerializer.lua (JSON+checksum, 143行)
    │   └── SaveSlots.lua     (多槽位, 186行)
    └── SaveMigrations.lua (版本迁移, 171行)
         └── SaveKeys.lua    (字段注册, 108行)

EventBus ← save:value_changed / save:loaded / save:saved / save:recovered
```

#### 优点
1. **工程质量最高的模块**：8 子模块清晰分层，每个模块 < 250 行
2. **4 级恢复链**：正常加载 → 修复（JSON修复） → 备份恢复（3份） → 部分提取 → 重置默认
3. **原子写入**：临时文件 → 备份轮转 → 重命名 → 验证读回
4. **类型安全**：SaveKeys 定义字段类型 + 范围，SaveState 写入时强校验
5. **脏标记优化**：只在数据变更时触发自动保存，减少 I/O
6. **Checksum 完整性**：DJB2 校验和检测数据损坏

#### 问题

| 严重度 | 问题 | 位置 | 影响 |
|--------|------|------|------|
| **HIGH** | SaveKeys.CURRENT_VERSION = 3 但 rules.md 记载线上版本 v17 | save/SaveKeys.lua | **这是新旧存档系统的版本号差异**——新系统从 v1 重新计数，与线上 17 版迁移链不兼容。需确认线上是否使用新系统 |
| **MED** | `_tryPartialRecover` 布尔正则 `(true|false)` 是 PCRE 语法 | save/SaveRecovery.lua | Lua 不支持 `|` 交替，布尔字段永远无法部分恢复 |
| **MED** | `quickSave()` 绕过 SaveLoader 直接调用 SavePersistence | SaveSystem.lua | 脏标记不清除、不触发事件、不写 `_saveTime` |
| **MED** | `SaveSlots.getAllSlots()` 每次调用完整加载所有 7 个存档文件 | save/SaveSlots.lua | 槽位选择 UI 每次打开都做 7 次文件读取+JSON解码 |
| **LOW** | SaveRecovery 硬编码路径 `"saves/" .. slotName .. ".sav"` | save/SaveRecovery.lua | 与 SavePersistence 的路径构造逻辑不同步 |
| **LOW** | `play_time` 字段在 SaveSlots.getSlotInfo 中引用但未在 SaveKeys 中定义 | save/SaveSlots.lua | 预览始终显示 playTime = 0 |
| **INFO** | 原子写入窗口：先 Delete 再 Rename，中间崩溃会丢失 | save/SavePersistence.lua | 概率极低，但可改为 Rename+Rename 双重命名模式 |

#### 建议
1. **明确新旧系统关系**：确认线上是否已迁移到新存档系统（8子模块版）。如果是，SaveKeys.CURRENT_VERSION 应与 rules.md 的 v17 保持一致
2. **修复布尔正则**：改为 `string.match(content, '"' .. key .. '"%s*:%s*(true)') or string.match(content, '"' .. key .. '"%s*:%s*(false)')`
3. **统一 quickSave 路径**：改为调用 `SaveLoader.save(quickSaveSlot)` 而非直接调用 SavePersistence
4. **添加 play_time 到 SaveKeys**：补充字段定义以修复预览

### 2.5 C/S 架构 — 评分: **B (76)**

#### 架构分析

```
main.lua
  ├── network.Standalone (160行, 单机 shim)
  ├── client_main.lua (1,080行)
  │   ├── 24 shared modules
  │   └── network.Client (1,210行)
  └── server_main.lua (1,684行)
      ├── 24 shared modules (完全相同)
      └── network.Server (1,755行)

network.Protocol (226行) ← 消息定义

Server 派生模块:
  ServerCombat (1,421), ServerQiankunFurnace (1,353),
  ServerPetFood (826), ServerShop (716), ServerDaily (593),
  ServerArtifact (590), ServerWorldBoss (527), ServerChapter (520),
  ServerDragonToken (503), ServerJob (465), ServerYaoChi (383)
```

#### 优点
1. **服务端权威模式**：所有状态变更在 Server 侧验证，客户端仅展示
2. **功能模块化的 Server Handler**：每个游戏系统有独立的 ServerXxx 处理器（11个），职责清晰
3. **共享配置**：client/server 引用同一套 config + system 模块，保证数据一致
4. **Protocol 集中管理**：消息 ID 和编解码集中在 Protocol.lua
5. **Standalone 降级**：160行 shim 实现单机模式，无需修改业务代码

#### 问题

| 严重度 | 问题 | 影响 |
|--------|------|------|
| **HIGH** | Server.lua (1,755行) 承载过多职责：连接管理 + 玩家数据 + 消息路由 + 通用逻辑 | 单点瓶颈，修改任一功能都在同一文件 |
| **HIGH** | client_main / server_main 24行重复 require 未提取 | 新增模块必须双端同步，已发生过遗漏 |
| **MED** | 网络协议使用 VectorBuffer 手动序列化，无版本号/schema | 协议变更无法向后兼容，新旧客户端混连会崩溃 |
| **MED** | Server Handler 无统一的防刷框架 | rules.md §8 已记录被利用的三重漏洞，但防护是 per-handler 手动实现 |
| **MED** | 无重连/断线恢复机制 | 客户端断线后状态丢失，需手动重连重新同步 |
| **LOW** | Client.lua 无 HandleUpdate —— 无帧循环 | 客户端无法做动画插值、倒计时等 |

#### 单机→多人迁移评估

当前架构**已完成**从单机到多人的迁移（Standalone.lua 为兼容层）。但以下方面仍需完善：

| 维度 | 现状 | 目标 |
|------|------|------|
| 协议版本化 | 无版本号 | 添加协议版本头，支持灰度升级 |
| 玩家身份 | 连接即新建 | 接入认证 + 存档加载 |
| 防刷框架 | 逐 handler 手动 | 提取 `RateLimiter` + `AtomicTransaction` 中间件 |
| 断线重连 | 无 | 添加 session token + 状态重同步 |
| 水平扩展 | 单进程全部玩家 | 考虑 room 分片或 stateless server |

#### 建议
1. **拆分 Server.lua**：至少分为 `ServerCore.lua`（连接/路由） + `ServerPlayerData.lua`（玩家数据管理）
2. **提取共享初始化**：创建 `shared_init.lua` 统一 24 个 require
3. **协议版本化**：在每条消息头添加 `PROTOCOL_VERSION` 字段
4. **防刷中间件**：提取 `ServerGuard.lua`，封装内存锁 + 原子事务 + 频率限制

### 2.6 性能 — 评分: **B (77)**

#### 评估方法
基于代码静态分析，评估 per-frame 开销、内存模式、I/O 模式。

#### 优点
1. **自动保存仅在脏标记时触发**：60秒间隔 + isDirty 检查，避免无效 I/O
2. **配置数据只读**：12 个 Config 模块在启动时加载，运行时不修改
3. **系统层无每帧分配**：CombatSystem/ChapterSystem 等系统不在 Update 中创建临时对象

#### 问题

| 严重度 | 问题 | 位置 | 影响 |
|--------|------|------|------|
| **MED** | SaveSlots.getAllSlots() 每次 7 次文件读取 | save/SaveSlots.lua | 打开槽位选择 UI 时卡顿 |
| **MED** | SaveSerializer 双重 JSON 编码（payload JSON → wrapper JSON） | save/SaveSerializer.lua | 存档文件体积膨胀 + 额外 CPU 开销 |
| **MED** | SavePersistence.save() 写入后全量回读验证 | save/SavePersistence.lua | 每次保存 I/O 翻倍 |
| **LOW** | EventBus emit 时对 listeners 做 snapshot（浅拷贝列表） | systems/EventBus.lua | 高频事件时有分配压力，但当前事件频率低 |
| **LOW** | Monster/Equipment 数据表（3,897行）全量常驻内存 | MonsterData + MonsterTypes_ch3 | ~100KB 级别，对移动端可接受 |

#### 建议
1. **SaveSlots 元数据缓存**：首次加载后缓存预览信息，后续读缓存
2. **SaveSerializer 单层编码**：将 checksum 作为同级字段而非嵌套 wrapper
3. **可选关闭写入验证**：在非关键场景（自动保存）跳过回读验证

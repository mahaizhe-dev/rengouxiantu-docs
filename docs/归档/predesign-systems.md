## 1. 青云试炼（试炼塔）

**状态**：预设计 v2.2
**设计日期**：2026-03-22

### 1.1 概述

260层爬塔系统 = 26个小境界 × 10层/境界。复用 ChallengeSystem 的"覆盖层副本"模式，扩展为多怪物同时战斗 + 每日奖励 + 排行榜。

### 1.2 核心规格

| 项目 | 规格 |
|------|------|
| 总层数 | 260层（26小境界 × 10层） |
| 竞技场尺寸 | 10×10 可行走区域（12×12 含墙） |
| 专属增幅 | 青云试炼增幅（见 §1.4） |
| 小怪层(1-3) | 8只小怪 |
| 混合层(4-6) | 6小怪 + 2精英 = 8只 |
| 精英层(7-9) | 4小怪 + 4精英 = 8只 |
| BOSS层(每10层) | 仅BOSS，2只（携带地刺） |
| 每层怪物上限 | 8只（BOSS层为2只） |
| 排行榜 | CloudStorage，分数 = 最高层数(1~260) |
| 每日奖励 | 基于最高层数档位，灵韵 + 金条（= 第10层首通×2） |
| 首通奖励 | 每层一次性灵韵 + 金条，纯线性增长，无BOSS区分 |

### 1.3 每层怪物组合规则

| 层号（localFloor） | 组合 | 数量 |
|-------------------|------|------|
| 1-3（小怪层） | 8只小怪(N) | 8 |
| 4-6（混合层） | 6小怪(N) + 2精英(E) | 8 |
| 7-9（精英层） | 4小怪(N) + 4精英(E) | 8 |
| 10（BOSS层） | 仅2只BOSS(B) | 2 |

### 1.4 青云试炼专属增幅

所有试炼塔内怪物获得一套专属增幅加成，显著强于野外同等级怪物。

| 增幅项 | 效果 | 影响范围 |
|--------|------|---------|
| 狂暴 | 攻速×1.3 / 移速×1.3 / 技能CD×0.7 | 所有怪物（小怪、精英、BOSS） |
| 强化攻击 | 攻击力 +30%（ATK × 1.3） | 所有怪物 |
| 强化生命 | 生命值 +100%（HP × 2.0） | 所有怪物 |
| 地刺技能 | 携带 trial_ground_spike | 仅BOSS（boss/king_boss/emperor_boss） |

实现伪代码：

```lua
local TRIAL_BUFF = {
    berserk = { atkSpeedMult = 1.3, speedMult = 1.3, cdrMult = 0.7 },
    atkMult = 1.3,
    hpMult = 2.0,
}

function ApplyTrialBuff(monster, category)
    monster.berserkBuff = TRIAL_BUFF.berserk
    monster.hasBerserkBuff = true
    monster.atk = monster.atk * TRIAL_BUFF.atkMult
    monster.maxHp = monster.maxHp * TRIAL_BUFF.hpMult
    monster.hp = monster.maxHp
    if category == "boss" or category == "king_boss" or category == "emperor_boss" then
        table.insert(monster.skills, "trial_ground_spike")
    end
end
```

难度对比（以练气初期 Lv.25 BOSS 为例）：

| 属性 | 野外BOSS | 试炼BOSS | 倍率 |
|------|---------|---------|------|
| 攻击力 | 100 (基准) | 169 (×1.3增幅×1.3狂暴攻速) | 1.69x |
| 生命值 | 500 (基准,5xHP) | 1000 (×2.0生命增幅) | 2.0x |
| 攻速 | 1.0x | 1.3x | 1.3x |
| 移速 | 1.0x | 1.3x | 1.3x |
| 技能CD | 1.0x | 0.7x | 0.7x |
| 地刺 | 无 | trial_ground_spike | — |

### 1.5 怪物等级计算

每个境界10层，怪物等级在 requiredLevel ~ maxLevel 之间均匀分布：

```
monsterLevel = requiredLevel + floor((maxLevel - requiredLevel) * (localFloor - 1) / 9)

- 升档时，旧档已领过，不再重复

```

### 2.1 概述

每个章节地图中放置一棵悟道树，玩家靠近交互后进行 10 秒读条参悟，获得一次经验奖励，并有 10% 概率获得 1 点悟性。每天最多与 3 棵不同的悟道树交互，每棵树每天只能交互 1 次。悟性从悟道树获取有上限。

**核心定位**：日常轻量活动，为悟性（剑修核心属性）提供装备之外的稳定获取途径。

### 2.4 经验奖励（v1.2 — 等级制方案）

悟道树经验奖励**基于玩家当前等级**，与在哪棵树交互无关。所有悟道树给予相同经验。

#### 核心规则

| 规则 | 说明 |
|------|------|
| 奖励依据 | **玩家当前等级**，不区分悟道树 |
| 开启等级 | Lv.10（练气初期） |
| 等级封顶时 | 经验**不积累**（溢出丢弃），悟性丹照常掉落 |
| 每日次数 | 仍为每棵树 1 次/日，每日最多 3 棵 |

#### 计算公式

```
reward = 0.225 × level ^ 3.824
```

锚点：Lv.10 ≈ 1,500（占升级经验 ~20%），Lv.100 = 10,000,000（占 ~0.52%）。
幂函数保证**严格单调递增**，不会出现高等级反而奖励下降的问题。

- 初期慷慨（20%/次）：凡人期极短，新手福利加速度过
- 中期适中（3~6%）：日常参悟有体感但不取代挂机刷怪
- 后期偏低（<1.5%）：高等级以悟性丹为主要收益，经验为附带
- Lv.35-45 占比出现微小平台（4.6%→4.8%→4.7%）：因 EXP_TABLE 在金丹期增速放缓所致，可忽略

> ⚠️ **配套改动**：`MonsterData.CalcBaseStats` 已增加 Lv.55+ EXP 二次成长
> `(level - 55)² × 0.5`（与 HP 的 `(level - 55)² × 5` 等比），保持高等级刷怪
> 效率不因 HP 膨胀而崩坏。此改动影响所有怪物经验，不仅限于悟道树。

### 2.5 悟性获取与上限

#### 悟性来源对比

| 来源 | 获取方式 | 理论上限 |
|------|---------|---------|
| **专属槽装备** | 装备主属性（T1=5 ~ T7=78） | 取决于装备品阶 |
| **其他槽位副属性** | 蓝3/紫7/橙12 | 取决于装备数量 |
| **宠物技能书** | 灵兽通慧书(+10) / 贰(+20) | 30 |
| **悟道树** ← 本系统 | 每日参悟，10% 概率 +1 | **50** |

#### 悟道树悟性上限设计

- **上限值**：50 点
- **累计不重置**：达到 50 后，参悟仍获得经验，但不再有概率获得悟性
- **UI 提示**：达到上限后交互面板显示"悟道树悟性已达上限（50/50）"

**为什么是 50**：
- 50 悟性 = 技能伤害 +10% + 连击概率 +1%
- 属于有感的增益但不会颠覆平衡（装备才是主要悟性来源）
- 以每日 3 次、10% 概率计算，期望 0.3 点/天 → 约 167 天（≈5.5 个月）可累满

### 2.6 服务端架构

所有日常活动基于服务端架构（C/S 模式），防止客户端篡改。

#### 使用的 serverCloud API

| 需求 | API | key | 说明 |
|------|-----|-----|------|
| 单树每日 1 次 | **Quota.Add** | `dao_tree_{treeId}` | limit=1, refresh="day" |
| 每日总计 3 次 | **Quota.Add** | `dao_tree_daily` | limit=3, refresh="day" |
| 悟性累计记录 | **Score.Get / Score.Set** | 存档内 `player.daoTreeWisdom` | 整数，累计从悟道树获得的悟性总量 |
| 经验发放 | **Score.Add** 或 **BatchCommit** | 存档内 `player.exp` | 增加经验值 |
| 悟性发放 | **BatchCommit** | Score.Set + Quota.Add 原子提交 | 悟性 +1 写入存档 |

#### 协议设计

| 事件名 | 方向 | 参数 | 说明 |
|--------|------|------|------|
| `C2S_DaoTree_StartMeditate` | C→S | `{ treeId = "dao_tree_ch1" }` | 客户端请求开始参悟 |
| `S2C_DaoTree_MeditateResult` | S→C | `{ ok, treeId, canStart, reason? }` | 服务端验证结果（次数是否足够） |
| `C2S_DaoTree_CompleteMeditate` | C→S | `{ treeId = "dao_tree_ch1" }` | 客户端完成 10 秒读条 |
| `S2C_DaoTree_Reward` | S→C | `{ exp, gotWisdom, wisdomTotal, dailyRemain }` | 服务端发放奖励并回告 |

#### 服务端处理流程

```
C2S_DaoTree_StartMeditate
  │
  ├─ Quota.Add("dao_tree_daily", 1, 3, "day")  ← 检查每日总次数
  │   └─ 失败（已满3次）→ S2C_MeditateResult { ok=false, reason="daily_limit" }
  │
  ├─ Quota.Add("dao_tree_{treeId}", 1, 1, "day")  ← 检查该树今日次数
  │   └─ 失败（该树已交互）→ S2C_MeditateResult { ok=false, reason="tree_used" }
  │          ⚠️ 此处需回滚 dao_tree_daily 的 Quota（或改为先查后加，见下方）
  │
  └─ 全部通过 → S2C_MeditateResult { ok=true, canStart=true }
       → 客户端开始 10 秒读条

C2S_DaoTree_CompleteMeditate
  │
  ├─ 服务端验证：该玩家确实处于"参悟中"状态（防伪造）
  │
  ├─ 计算经验奖励 = CalcBaseExp(playerLevel) × 5
  │
  ├─ 判定悟性：
  │   ├─ Score.Get → daoTreeWisdom >= 50 → 不判定
  │   └─ daoTreeWisdom < 50 → math.random() < 0.10 → gotWisdom = true
  │
  └─ BatchCommit:
       ├─ Score.Add("player.exp", expReward)
       ├─ if gotWisdom: Score.Set("player.wisdom" += 1, "player.daoTreeWisdom" += 1)
       └─ → S2C_DaoTree_Reward { exp=..., gotWisdom=true/false, ... }
```

**Quota 先查后加方案**（避免回滚复杂度）：

```
实际实现建议：
1. Quota.Get("dao_tree_daily") → 查当前计数
2. Quota.Get("dao_tree_{treeId}") → 查该树计数
3. 两个都满足 → Quota.Add + Quota.Add（完成读条后再扣）
   不满足 → 直接拒绝
```

但由于 Quota API 可能不支持 Get（仅支持 Add 和 Reset），实际方案需要阶段 C 验证后确定。

**备选方案**：如果 Quota 不支持先查再扣，改用 Score 域手动计数 + 服务端内存缓存每日重置状态。

#### 与装备悟性的关系

当前 `Player:GetTotalWisdom()` 仅统计装备和宠物悟性：

```lua
-- 现有
function Player:GetTotalWisdom()
    local petBonus = self.petOwnerBonuses and self.petOwnerBonuses.wisdom or 0
    return (self.equipWisdom or 0) + petBonus
end

-- 改为
function Player:GetTotalWisdom()
    local petBonus = self.petOwnerBonuses and self.petOwnerBonuses.wisdom or 0
    local daoTreeWisdom = self.daoTreeWisdom or 0
    return (self.equipWisdom or 0) + petBonus + daoTreeWisdom
end
```

## 3. 福缘宝箱

**状态**：预设计 v1.0
**设计日期**：2026-03-23

### 3.1 概述

每个章节地图中放置 3 个福缘宝箱，玩家靠近交互后进行 3 秒读条开启，获得 3 选 1 装备奖励。每个宝箱有福缘门槛要求，福缘越高可开启的宝箱品质越好。每天最多开启 3 个宝箱，每个宝箱每天只能开启 1 次。

**核心定位**：
- 为福缘属性提供**直接收益**（福缘越高 → 开启更好宝箱 → 获得更强装备）
- 与悟道树形成日常活动矩阵（悟道树→经验/悟性，福缘宝箱→装备）
- 强化阵法师职业身份（福缘核心属性的回报路径）

### 3.6 服务端架构

所有日常活动基于服务端架构（C/S 模式），与悟道树相同模式。

#### 使用的 serverCloud API

| 需求 | API | key | 说明 |
|------|-----|-----|------|
| 单箱每日 1 次 | **Quota.Add** | `fortune_chest_{chestId}` | limit=1, refresh="day" |
| 每日总计 3 次 | **Quota.Add** | `fortune_chest_daily` | limit=3, refresh="day" |
| 福缘检查 | **读取存档** | `player.GetTotalFortune()` | 校验当前总福缘 ≥ 宝箱门槛 |
| 装备生成 | **LootSystem 扩展** | — | 服务端按 tier+qualityRange 生成 3 件装备 |
| 装备发放 | **BatchCommit** | Score.Set（存档背包） | 选中的装备写入背包 |

#### 协议设计

| 事件名 | 方向 | 参数 | 说明 |
|--------|------|------|------|
| `C2S_FortuneChest_StartOpen` | C→S | `{ chestId }` | 请求开启宝箱 |
| `S2C_FortuneChest_StartResult` | S→C | `{ ok, reason?, fortuneReq? }` | 验证结果（福缘/次数是否满足） |
| `C2S_FortuneChest_CompleteOpen` | C→S | `{ chestId }` | 客户端完成 3 秒读条 |
| `S2C_FortuneChest_Reward` | S→C | `{ items[3], dailyRemain, chestId }` | 服务端生成 3 件装备并发送 |
| `C2S_FortuneChest_Pick` | C→S | `{ chestId, pickIndex }` | 玩家从 3 件中选择 1 件 |
| `S2C_FortuneChest_PickResult` | S→C | `{ ok, equipment }` | 确认选择，装备写入背包 |

#### 服务端处理流程

```
C2S_FortuneChest_StartOpen { chestId }
  │
  ├─ 查找宝箱配置 → 获取 fortuneReq, tier, qualityMin, qualityMax
  │
  ├─ 检查福缘门槛：player.GetTotalFortune() >= fortuneReq
  │   └─ 不足 → S2C { ok=false, reason="fortune_not_enough", fortuneReq=X }
  │
  ├─ Quota.Add("fortune_chest_daily", 1, 3, "day")
  │   └─ 已满 → S2C { ok=false, reason="daily_limit" }
  │
  ├─ Quota.Add("fortune_chest_{chestId}", 1, 1, "day")
  │   └─ 已用 → S2C { ok=false, reason="chest_used" }
  │       ⚠️ 需回滚 fortune_chest_daily 的 Quota
  │
  └─ 全部通过 → S2C { ok=true }
       → 客户端开始 3 秒读条
       → 服务端标记玩家进入"开箱中"状态

C2S_FortuneChest_CompleteOpen { chestId }
  │
  ├─ 验证：玩家确实处于"开箱中"状态（防伪造）
  │
  ├─ 生成 3 件装备（调用 LootSystem 扩展函数）：
  │   ├─ tier = 宝箱配置的 tier
  │   ├─ 品质 = 按 qualityRange 过滤权重后随机
  │   ├─ 槽位 = 从 REWARD_SLOTS 中随机选 3 个不同槽位
  │   └─ 按 EquipmentData 生成完整装备属性（主属性+副属性）
  │
  └─ S2C_FortuneChest_Reward { items[3], dailyRemain }
       → 客户端展示 3 选 1 UI

C2S_FortuneChest_Pick { chestId, pickIndex }
  │
  ├─ 验证：pickIndex ∈ [1, 3]，玩家确实有待选装备
  │
  ├─ 将选中装备写入背包（BatchCommit → Score.Set）
  │
  └─ S2C_FortuneChest_PickResult { ok=true, equipment }
       → 客户端收起选择 UI，显示获得提示
```

**Quota 回滚问题**（同悟道树方案）：

优先使用"先查后扣"策略（Quota.Get → 判断 → Quota.Add），读条完成后才实际消耗 Quota。如果 Quota API 不支持 Get，则使用 Score 域手动计数。

### 3.7 装备生成逻辑

需要在 LootSystem 中新增一个函数，支持**指定品质范围**的装备生成：

```lua
--- 为福缘宝箱生成装备
--- @param tier number 品阶(1-11)
--- @param qualityMin string 最低品质 ("white"/"green"/...)
--- @param qualityMax string 最高品质 ("purple"/"orange"/"cyan"/...)
--- @param excludeSlots table 已使用的槽位（确保 3 件不重复）
--- @return table 生成的装备数据
function LootSystem.GenerateChestEquipment(tier, qualityMin, qualityMax, excludeSlots)
    -- 1. 从 TIER_QUALITY_WEIGHTS[tier] 中过滤出 [qualityMin, qualityMax] 范围
    -- 2. 按过滤后的权重随机选择品质
    -- 3. 从 REWARD_SLOTS 中排除 excludeSlots，随机选择槽位
    -- 4. 调用现有装备生成逻辑（主属性+副属性）
    -- 5. 返回完整装备数据
end
```

生成 3 件装备的调用：
```lua
local items = {}
local usedSlots = {}
for i = 1, 3 do
    local item = LootSystem.GenerateChestEquipment(
        chestCfg.tier, chestCfg.qualityMin, chestCfg.qualityMax, usedSlots
    )
    table.insert(items, item)
    usedSlots[item.slot] = true
end
```

2. **章节扩展**：从 ch1-ch2（8 任务）扩展到 ch1-ch3（12 任务）
3. **日常任务机制**：每章 4 个一次性封魔任务全部完成后解锁该章日常封魔任务
4. **奖励升级**：原 BOSS 完整掉落 + 固定任务奖励（仙骨/妖兽精华 + 灵韵），日常额外固定给体魄丹
5. **魔化 BOSS 类别升级**：ch2 全部升为王级(king_boss)，ch3 最高档升为皇级(emperor_boss)
6. **魔化境界**：每个魔化 BOSS 拥有对应等级的最高可达境界
7. **成就隔离**：击杀魔化 BOSS 不触发原 BOSS 成就
8. **独立刷新**：魔化 BOSS 不影响原 BOSS 刷新，召唤位置远离原 BOSS 警戒范围
9. **全部重置**：已完成的除魔任务进度全部清零，玩家需重新完成

**核心定位**：
- 中期日常活动，为玩家提供**可控的 BOSS 战斗机会**
- 体魄（体修核心属性）的稳定获取途径之一（日常固定 1 颗体魄丹）
- 与福缘宝箱（阵法师）、悟道树（剑修）形成四职业日常活动矩阵

### 4.5 魔化 BOSS 设计

#### 魔化增幅

魔化 BOSS 在原 BOSS 基础上叠加以下增幅：

| 增幅项 | 效果 | 说明 |
|--------|------|------|
| 狂暴 buff | 攻速×1.3 / 移速×1.3 / CDR×0.7 | 与现有 berserkBuff 一致 |
| 生命强化 | HP ×1.5 | 比野外 BOSS 多 50% 血量 |
| 等级调整 | 部分 BOSS 等级提升至任务等级 | 确保与任务等级匹配 |
| **类别升级** | ch2 全部→king_boss, ch3 最后→emperor_boss | 魔化增强 BOSS 级别 |

> **与试炼塔增幅的区别**：试炼塔增幅更强（ATK×1.3 + HP×2.0 + 地刺），封魔任务增幅相对温和（仅 HP×1.5），因为封魔 BOSS 有完整掉落库作为奖励。

#### 成就隔离 🔴

**击杀封魔系统的魔化 BOSS 不会触发原 BOSS 的成就**。

通过 `isSealDemon = true` 标记区分，成就系统检查时过滤该标记：

```lua
-- AchievementSystem 中检查
if monster.isSealDemon then
    return  -- 封魔BOSS击杀不计入原BOSS成就
end
```

#### 刷新隔离

- 魔化 BOSS 是**独立召唤**的临时实体，不影响原 BOSS 的刷新机制
- 召唤位置必须**远离原 BOSS 的警戒范围**（AGGRO_RANGE），避免两个 BOSS 同时存在于玩家视野中
- 实现：召唤坐标与原 BOSS 出生点距离 ≥ `LEASH_RADIUS[category] + AGGRO_RANGE` (约 20-30 格)

#### 掉落保持原样

**掉落库评估结论：保持原 BOSS 掉落不变，不适配魔化等级品质。**

原因：
1. DeepCopy 原 BOSS 的 dropTable 后，装备掉落使用 `minQuality/maxQuality` 范围，LootSystem 会自动根据 BOSS 等级 roll 品质
2. 等级覆盖（overrideLevel）后装备品质自动跟随，无需额外代码
3. 材料掉落（灵韵、食物、技能书等）为固定 ID，与等级无关
4. 简化实现，避免引入新的掉落计算逻辑

#### 魔化 BOSS 数据结构

```lua
-- 魔化 BOSS 模板（运行时从原 BOSS 复制并叠加增幅）
function SealDemonSystem.CreateDemonBoss(sourceBossId, config)
    local source = MonsterData.Types[sourceBossId]
    local demon = DeepCopy(source)

    -- 魔化标记
    demon.isDemon = true
    demon.isSealDemon = true  -- 区分封魔系统的魔化BOSS（成就隔离）

    -- 等级覆盖
    if config.overrideLevel then
        demon.level = config.overrideLevel
    end

    -- 类别升级
    if config.overrideCategory then
        demon.category = config.overrideCategory
    end

    -- 境界覆盖
    if config.overrideRealm then
        demon.realm = config.overrideRealm
    end

    -- 魔化增幅
    demon.berserkBuff = { atkSpeedMult = 1.3, speedMult = 1.3, cdrMult = 0.7 }
    demon.hasBerserkBuff = true
    demon.sealDemonHpMult = 1.5  -- 生命×1.5

    -- 保留原 BOSS 完整掉落库（已通过 DeepCopy 复制）
    -- 掉落品质会自动跟随 overrideLevel

    -- 名称前缀
    demon.name = "魔·" .. source.name

    return demon
end
```

#### 与旧除魔怪物的对比

| 对比项 | 旧除魔怪物（魔化精英） | 新封魔怪物（魔化BOSS） |
|--------|---------------------|---------------------|
| 原型 | 精英怪（spider_elite, boar_captain 等） | BOSS（spider_queen, tiger_king 等） |
| 等级分类 | boss / king_boss | **升级**：ch2→king_boss, ch3最后→emperor_boss |
| 境界 | 无特殊 | **增幅**：当前等级最高可达境界 |
| 掉落 | **空**（dropTable = {}） | **原 BOSS 完整掉落库**（保持不变） |
| 额外掉落 | 无 | 固定仙骨/妖兽精华 + 灵韵 |
| 增幅 | berserkBuff 仅（攻速/移速/CDR） | berserkBuff + HP×1.5 + 类别升级 |
| 技能 | 继承精英技能 | 继承 BOSS 技能（poison_spit、shield_slam 等） |
| 成就 | 不触发 | **不触发原 BOSS 成就**（isSealDemon 隔离） |
| 刷新 | 不影响 | **不影响原 BOSS 刷新**，独立召唤位置 |
| 难度定位 | 稍强于精英 | 强于同级野外 BOSS |

#### 物品定义

```lua

    icon = "💪",
    quality = "purple",   -- 极品（紫色）

    attribute = "physique",

    sellPrice = 500,

```

#### 体魄丹获取上限

与悟道树悟性上限（50）对齐：

| 属性 | 来源 | 单次获取 | 累计上限 | 获取方式 |
|------|------|---------|---------|---------|
| 悟性 | 悟道树 | +1（25%概率，6次保底） | 50 | 日常参悟 |
| **体魄** | **体魄丹** | **+1** | **50** | **日常封魔固定 1 颗** |

体魄丹获取节奏：
- 每日日常封魔固定给 1 颗体魄丹
- 50 颗上限 ÷ 1 颗/天 = **50 天**即可集满
- 对比悟性：25% 概率约需 ~60 天（含保底），节奏相近

设计意图：
- 50 体魄 = HP +500 + 受伤减免 +5%（根据 class-system-v7.md 体魄效果）
- 属于有感增益但不颠覆平衡（装备才是主要体魄来源）
- 体修天然获得 +1 体魄/级，体魄丹对体修锦上添花、对其他职业雪中送炭

#### 使用流程

```
玩家获得体魄丹 → 背包中右键使用
  ├─ 检查 player.pillPhysique < 50
  │   ├─ 满足 → player.pillPhysique += 1，消耗丹药，提示 "体魄 +1"
  │   └─ 已满 → 提示 "体魄丹效果已达上限（50/50），无法继续服用"
  └─ 体魄丹可出售（500 金币）
```

#### 对 Player.lua 的影响

```lua
-- 现有
function Player:GetTotalPhysique()

-- 改为

```

### 4.9 任务状态机

#### 一次性封魔任务状态

```
locked → available → active → completed
  │         │          │
  │         │          └─ 击杀魔化BOSS → completed（永久）
  │         └─ 接取任务 → 召唤魔化BOSS → active
  └─ 满足等级要求 → available
```

| 状态 | 条件 | 说明 |
|------|------|------|
| locked | 玩家等级 < requiredLevel | 未解锁，NPC 面板中灰显 |
| available | 玩家等级 >= requiredLevel 且未完成 | 可接取 |
| active | 已接取，魔化 BOSS 已召唤 | 战斗中 |
| completed | 击杀魔化 BOSS | 永久完成，勾选标记 |

#### 日常封魔任务状态

```
locked → available → active → completed_today
  │         │          │           │
  │         │          │           └─ 次日重置 → available
  │         │          └─ 击杀魔化BOSS → completed_today
  │         └─ 接取任务 → 召唤随机魔化BOSS → active
  └─ 该章4个一次性任务全部completed → available
```

| 状态 | 条件 | 说明 |
|------|------|------|
| locked | 该章一次性任务未全部完成 | 不可见 |
| available | 一次性全完成 + 今日未接取日常 | 可接取 |
| active | 已接取，随机 BOSS 已召唤 | 战斗中 |
| completed_today | 今日已完成 | 次日自动重置为 available |

### 4.11 服务端架构

#### 使用的 serverCloud API

| 需求 | API | key | 说明 |
|------|-----|-----|------|
| 日常每日 1 次 | **Quota.Add** | `seal_demon_daily` | limit=1, refresh="day" |
| 一次性任务完成 | **Score.Set** | 存档 `sealDemon.quests` | 标记单个任务 completed |
| 体魄丹累计 | **Score.Get/Set** | 存档 `player.pillPhysique` | 检查/增加体魄丹使用量 |
| 掉落发放 | **BatchCommit** | 背包 + 各资源字段 | 原子写入所有掉落物品 |

#### 协议设计

| 事件名 | 方向 | 参数 | 说明 |
|--------|------|------|------|
| `C2S_SealDemon_Accept` | C→S | `{ questId }` 或 `{ dailyChapter }` | 接取一次性/日常任务 |
| `S2C_SealDemon_AcceptResult` | S→C | `{ ok, reason?, bossId?, spawnX?, spawnY?, warning? }` | 验证结果，返回召唤的 BOSS 信息 |
| `C2S_SealDemon_BossKilled` | C→S | `{ questId or dailyChapter, bossTypeId }` | 客户端上报击杀 |
| `S2C_SealDemon_Reward` | S→C | `{ lingYun?, drops[], questId?, dailyComplete? }` | 服务端验证并发放奖励 |
| `C2S_SealDemon_Abandon` | C→S | `{ questId or dailyChapter }` | 放弃当前任务 |
| `S2C_SealDemon_AbandonResult` | S→C | `{ ok }` | 放弃确认，魔化 BOSS 消失 |

#### 服务端处理流程

```
C2S_SealDemon_Accept { questId = "seal_ch2_30" }
  │
  ├─ 判断：一次性任务 or 日常任务？
  │
  ├─ [一次性任务]
  │   ├─ 检查等级：player.level >= quest.requiredLevel
  │   ├─ 检查状态：sealDemon.quests[questId] ~= "completed"
  │   ├─ 检查：当前无进行中的封魔任务
  │   └─ 通过 → 召唤魔化 BOSS → S2C { ok=true, bossId, spawnX, spawnY }
  │
  └─ [日常任务] { dailyChapter = 2 }
      ├─ 检查：该章 4 个一次性任务全部 completed
      ├─ Quota.Add("seal_demon_daily", 1, 1, "day")
      │   └─ 已满 → S2C { ok=false, reason="daily_limit" }
      ├─ 检查低阶警告：dailyChapter < highestUnlockedChapter
      │   └─ 是 → S2C { ok=true, warning="low_tier" }（客户端弹确认）
      ├─ 随机选择该章 BOSS → 创建最高等级魔化版本
      └─ S2C { ok=true, bossId, spawnX, spawnY }

C2S_SealDemon_BossKilled { questId = "seal_ch2_30" }
  │
  ├─ 验证：该玩家确实有进行中的封魔任务，且 BOSS 确实被击杀
  │
  ├─ [一次性任务]
  │   ├─ 标记 sealDemon.quests[questId] = "completed"
  │   ├─ 发放灵韵奖励
  │   ├─ BOSS 掉落按原 dropTable + 额外掉落判定
  │   ├─ 检查是否解锁日常：该章 4 个全部 completed？
  │   └─ BatchCommit → S2C_Reward { lingYun, drops, questId, dailyUnlocked? }
  │
  └─ [日常任务]
      ├─ 发放固定奖励（灵韵 + 仙骨/妖兽精华 + 体魄丹）→ 弹窗展示（与一次性任务一致）
      ├─ BOSS 掉落按原 dropTable 判定
      ├─ 标记今日日常已完成（Quota 扣除）
      └─ BatchCommit → S2C_Reward { lingYun, petFood, physiquePill, drops, dailyComplete=true }
```

封魔任务每日仅 1 次，但单次收益包含：完整 BOSS 掉落库 + 固定宠物食材 + 固定体魄丹。与悟道树类似，是需要每天坚持的长线养成系统。日常不额外给灵韵，灵韵仅来自 BOSS 掉落。

#### 体魄丹积累节奏

- 每日 1 枚，最多积累 50 枚
- 50 天集满（约 7 周）
- 每 10 枚一个体魄等级，共 5 级（+2/+4/+6/+8/+10 体魄）

### 4.17 未迁移存档玩家影响评估

#### 背景：存档迁移链路

存档迁移采用两阶段发布（详见 `server-save-migration.md`）：

| 阶段 | `multiplayer.enabled` | 存储引擎 | 迁移动作 |
|------|----------------------|---------|---------|
| Phase 1 | `false` | clientScore | 每次存档后导出 `save_export_X.json` 到本地 |
| Phase 2 | `true` | serverCloud | 首次连接时读本地文件 → 上传服务端 → 删文件 |

"未迁移玩家"指：Phase 2 上线后，尚未登录完成迁移的玩家。

#### 评估结论：无负面影响

| 维度 | 分析 | 结论 |
|------|------|------|
| 旧进度丢失？ | §4.10 设计意图：`exorcism = nil`，旧除魔进度**全部重置**，与迁移状态无关 | 所有玩家一视同仁 |
| 迁移时序冲突？ | 封魔用新字段 `sealDemon`，旧字段 `exorcism`。迁移"只搬运不改结构" | 数据隔离，无冲突 |
| 版本号冲突？ | 迁移保持 v9 不变；封魔上线新增 v10 迁移。MIGRATIONS 链式执行，顺序无关 | 先迁移后升级 / 先升级后迁移均正确 |
| PC 玩家？ | 本地导出文件关闭后丢失，需先在手机迁移。封魔系统无额外影响 | 已有机制兜底 |

#### 三类玩家场景

**场景 A：已迁移玩家**（serverCloud 已有 v9 存档）

```
登录 → 加载 serverCloud 存档（v9）
  → v10 迁移：exorcism = nil, 初始化 sealDemon
  → 所有封魔任务 locked，按等级刷新
  → 正常游玩
```

**场景 B：未迁移玩家**（Phase 2 上线后首次登录）

```
登录 → 迁移链触发：
  1. 读取本地 save_export_X.json（v9，含 exorcism 字段）
  2. 原样上传 serverCloud（不改结构）
  3. 加载存档 → v10 迁移：exorcism = nil, 初始化 sealDemon
  → 结果与场景 A 完全一致
```

**场景 C：无本地文件玩家**（Phase 1 期间从未登录）

```
登录 → 无本地文件，无 clientCloud 数据
  → 空存档 / 恢复槽位机制
  → 封魔任务全部 locked（与封魔无关，属迁移层面已知边界）
```

#### 实现注意事项

**1. v10 迁移必须包含在封魔上线版本中**：

```lua
-- SaveSystem.lua → MIGRATIONS
[10] = function(data)
    -- 清除旧除魔数据
    data.exorcism = nil

    -- 初始化封魔数据
    data.sealDemon = {
        quests = {},
        dailyDate = nil,
        dailyChapter = nil,
    }

    -- 体魄丹计数
    if data.player then
        data.player.pillPhysique = 0
    end

    print("[SaveSystem] v9→v10 migration: exorcism → sealDemon")
    data.version = 10
    return data
end
```

**2. 删除旧 ExorcismSystem 引用**：

SaveSystem 加载逻辑中现有代码（`SaveSystem.lua:2037`）：

```lua
-- 现有（需随 ExorcismSystem 一起删除）
if saveData.exorcism then
    local ExorcismSystem = require("systems.ExorcismSystem")
    ExorcismSystem.Deserialize(saveData.exorcism)
end

-- 替换为
if saveData.sealDemon then
    local SealDemonSystem = require("systems.SealDemonSystem")
    SealDemonSystem.Deserialize(saveData.sealDemon)
end
```

v10 迁移确保 `exorcism = nil`，即使旧加载代码残留也不会被触发，但仍建议同步清理。

#### 结论

封魔任务的"全部重置"设计（§4.10）天然规避了新旧数据兼容问题：

- 不需要映射旧 quest ID（`exorcism_ch1_5` → `seal_ch1_1`）
- 不需要继承旧进度
- 所有玩家从同一起跑线开始

迁移状态对封魔系统完全透明，**无需针对未迁移玩家做特殊处理**。
